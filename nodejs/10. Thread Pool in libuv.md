## Event Loop Tick

- 1 cycle of the event loop = **1 tick** of the event loop
- Official Node.js docs list **more than 4 phases** (including Pending Callbacks and Idle/Prepare)

### Additional Phases

|Phase|Purpose|
|---|---|
|**Pending Callbacks**|Executes I/O callbacks deferred to the next loop iteration (due to errors, retry logic, or starvation prevention)|
|**Idle, Prepare**|Used only internally by Node.js/libuv -- not relevant for application-level code|

---

## Thread Pool

- libuv includes a **built-in thread pool**: a collection of threads managed by libuv to handle heavy, non-blocking tasks asynchronously

### When is the Thread Pool Used?

When an async task is offloaded by V8 to libuv for operations that **would block** the main thread:

- **File system operations** (e.g., `fs.readFile`)
- **DNS lookups** (e.g., `dns.lookup`)
- **Crypto operations** (e.g., `crypto.pbkdf2`)
- **User-defined custom I/O** operations

```
V8 (main thread) --offloads heavy async task--> libuv Thread Pool (4 threads) --completion--> callback queued in event loop
```

---

## Thread Pool Default Behavior

- Default **`UV_THREADPOOL_SIZE` = 4**: Only 4 tasks can run concurrently -- the 5th task **waits** until a thread is free
- Execution order of the first 4 tasks is **not guaranteed** (whichever thread finishes first prints first)
- The 5th task always finishes last since it has to wait

### Changing Thread Pool Size

```js
// Set before any async operations
process.env.UV_THREADPOOL_SIZE = 8;  // now 8 threads
```

## Is Node.js Single-Threaded or Multi-Threaded?

|Workload|Threading|
|---|---|
|**Only synchronous code**|Single-threaded (V8 handles all on main thread)|
|**Async tasks involved**|Multi-threaded (offloaded to libuv's thread pool)|

It depends on the type of workload (sync vs async).

## Network Calls & Sockets

- **API calls don't use the thread pool**: networking is handled via **sockets**: All networking in Node.js runs on **sockets, not threads**

### How Networking Works

1. When a user makes a request (e.g., HTTP), a **TCP connection** is established using a socket
2. Each connection has its own **socket descriptor** (aka file descriptor)
3. Node.js doesn't spawn a thread per connection -- that's **inefficient**

### Why Not One Thread per Connection?

- Thread-per-Connection model is not scalable: 10,000 users = 10,000 threads = high memory, poor performance
- Better approach: OS-level I/O notification mechanisms

## epoll, kqueue & IOCP

libuv uses **OS-specific mechanisms** for efficient I/O:

|OS|Mechanism|
|---|---|
|**Linux**|`epoll`|
|**macOS/BSD**|`kqueue`|
|**Windows**|`IOCP`|

### How epoll Works

```
Client Connections
  |   |   |   |
  v   v   v   v
Socket Descriptors (unique ID per connection)
  |   |   |   |
  +---+---+---+
        |
        v
  epoll Descriptor (monitors all sockets)
        |
        | kernel notifies when I/O ready
        v
  libuv adds callback to Event Loop Queue
        |
        v
  Node.js executes callback
```

1. When clients connect, the OS assigns a **socket descriptor** to each connection
2. All descriptors are registered with the **epoll descriptor**, which monitors them
3. If any socket becomes ready for I/O, the **kernel notifies** the application
4. libuv adds the callback to the event loop queue, Node.js executes it
5. **Result:** Non-blocking, event-driven I/O -- handles **thousands of concurrent connections** with very few threads

- epoll operates in **O(1)** time (older system calls: O(n))
- epoll uses a **Red-Black Tree** data structure
- Timers queue uses a **min heap** data structure

- Opening a web server: create a socket and start listening
- Clients initiate connections via **system calls** (requests to the OS Kernel)

## Event-Driven Architecture

Node.js follows an **event-driven model**: the OS notifies when something happens, instead of Node.js constantly checking.

Uses:
- **Event emitters** (Observer pattern)
- **Streams**: **Buffers**: **Pipes**: **Callbacks and Promises**

This makes Node.js **lightweight, scalable, and efficient** for I/O-heavy applications.

---

## Best Practice: Don't Block the Main Thread

Avoid these on the main thread:
- Synchronous methods (`readFileSync`, etc.)
- Heavy JSON objects
- Complex Regex
- Complex calculations/loops
