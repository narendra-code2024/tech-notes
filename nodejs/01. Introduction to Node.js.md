## What is Node.js?

- **JavaScript runtime environment** that runs on the **V8 JavaScript Engine** and executes JavaScript code **outside a web browser**: Written in: JavaScript, C++, Python and C
- Lets developers use JS for **server-side scripting**: Used to generate **dynamic web page content** server-side before sending to browser
- **Event-driven architecture** capable of **asynchronous I/O**: optimizes throughput and scalability in:
    - Web apps with many I/O operations
    - Real-time web applications (e.g., real-time communication, browser games)

## History

- **Apache** had limited capability to handle 10,000+ concurrent connections (blocking)
- **Sequential programming** can block entire processes or require multiple execution stacks
- Node.js was created to solve these scalability issues by combining:

```
Google's V8 Engine  +  Event Loop  +  Low-level I/O API
```

```
2009  Ryan Dahl creates Node.js
  |     - Initially called web.js, renamed to Node.js
  |     - First tried SpiderMonkey (Firefox engine), switched to V8 after 2 days
  |
2010  npm introduced (Node Package Manager)
  |
2011  Windows support added (Joyent + Microsoft collaboration)
  |
2012  Isaac Schlueter takes over npm maintenance
  |
2014  io.js fork created (by Fedor Indutny and others) over governance concerns
  |
2015  io.js merges back --> Node.js Foundation formed (under Linux Foundation)
  |
2019  JS Foundation + Node.js Foundation --> OpenJS Foundation
```

## Overview

- Primarily used to build **network programs** such as web servers
- Modules provided for: File system I/O, Networking (DNS, HTTP, TCP, TLS/SSL, UDP), Binary data (buffers), Cryptography, Data streams
- JavaScript is the **only language Node.js supports natively**, but compile-to-JS languages work too (TypeScript, CoffeeScript, Dart, etc.)

### Node.js vs PHP

| Feature | PHP | Node.js |
| --- | --- | --- |
| **Execution** | Blocking -- commands execute only after previous ones finish | Non-blocking -- commands execute concurrently |
| **Completion Signal** | Implicit (sequential) | Callbacks to signal completion or failure |

## Threading

- Operates on a **single-thread event loop**, using non-blocking I/O calls -- supports tens of thousands of concurrent connections **without thread context switching cost**: Uses the **observer pattern**: any function performing I/O must use a callback
- Uses **libuv's fixed-sized thread pool** for some blocking operations (e.g., file I/O)

```
Main Thread (V8)
    |
    | posts tasks to shared task queue
    v
Thread Pool (libuv)
    |
    | threads pull and execute tasks
    v
On completion --> notify main thread --> execute callback
```

- **Non-blocking** system functions (networking) --> kernel-side non-blocking sockets
- **Blocking** system functions (file I/O) --> run in thread pool --> callback on completion

### Downsides

- No **vertical scaling** by increasing CPU cores without `cluster`, StrongLoop PM, or `pm2`
- **CPU-bound tasks freeze the event loop** until completion

---

## Event Loop

- Node.js **registers with the OS**: OS notifies on async I/O events (e.g., new connections)
- Events trigger **callbacks**: each connection is a small **heap allocation** (not heavyweight OS threads)
- Event loop **does not need to be called explicitly**: server enters it after callbacks are defined
- **Exits the event loop** when no further callbacks remain
