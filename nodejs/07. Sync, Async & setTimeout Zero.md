## Sync Code Example

```js
// app.js
console.log("Hello World");

var a = 100;
var b = 200;

function multiplyFn(x, y) {
    const result = x * y;
    return result;
}

var c = multiplyFn(a, b);
console.log("Multiplication result is : ", c);
```

```
> node app.js
Hello World
Multiplication result is : 20000
```

- Everything runs **line by line** on V8's call stack -- fully synchronous

## Async Code Example (Non-Blocking I/O or Asynchronous I/O)

```js
// async.js
const fs = require("fs");
const https = require("https");

console.log("Hello World");

var a = 100;
var b = 200;

https.get("https://dummyjson.com/products/1", (res) => {
    console.log("Fetched Data Successfully");
});

setTimeout(() => {
    console.log("setTimeout called after 5 seconds");
}, 5000);

fs.readFile("./file.txt", "utf-8", (err, data) => {
    console.log("File data: " + data);
});

function multiplyFn(x, y) {
    const result = x * y;
    return result;
}

var c = multiplyFn(a, b);
console.log("Multiplication result is : ", c);
```

```
> node async.js
Hello World
Multiplication result is : 20000
File data: This is file data
Fetched Data Successfully
setTimeout called after 5 seconds
```

- Sync code runs first, async callbacks execute **after** the call stack is empty

## Blocking vs Non-Blocking I/O

- `readFileSync` -- **synchronous**, JS engine **cannot offload** to libuv, **blocks the main thread**, doesn't take a callback (returns data directly)
- `readFile` -- **asynchronous**, offloaded to libuv, **non-blocking**, takes a callback

```js
const fs = require("fs");

// Blocking -- entire program waits until file is read
const data = fs.readFileSync("./file.txt", "utf-8");
console.log("Sync file data: " + data);

// Non-Blocking -- program continues, callback runs later
fs.readFile("./file.txt", "utf-8", (err, data) => {
    console.log("Async file data: " + data);
});
console.log("This runs BEFORE async file is read");
```

```
> node blocking.js
Sync file data: This is file data
This runs BEFORE async file is read
Async file data: This is file data
```

## Blocking the Main Thread (crypto example)

```js
// blocking.js
const crypto = require("node:crypto");

console.log("Start");

// Async -- offloaded to libuv thread pool, does NOT block main thread
crypto.pbkdf2("password", "salt", 5000000, 50, "sha512", (err, key) => {
    console.log("Async key is generated");
});

// Sync -- BLOCKS the main thread until complete
const key = crypto.pbkdf2Sync("password", "salt", 5000000, 50, "sha512");
console.log("Sync key is generated");

function multiplyFn(x, y) {
    const result = x * y;
    return result;
}

var c = multiplyFn(a, b);
console.log("Multiplication result is : ", c);
```

```
> node blocking.js
Start
Sync key is generated
Multiplication result is : 20000
Async key is generated
```

- `crypto.pbkdf2` (async) uses the **thread pool**: main thread stays free
- `crypto.pbkdf2Sync` (sync) **blocks the main thread**: avoid in production
- Sync code runs first, async key generation finishes last

## setTimeoutZero Example -- setTimeout(cb, 0)

```js
// setTimeoutZero.js
console.log("Start");

setTimeout(() => {
    console.log("setTimeout callback");
}, 0);

// setTimeout will only be pushed to call stack once the call stack is empty

setTimeout(() => {
    console.log("call me right now");
}, 0);

function multiplyFn(x, y) {
    const result = x * y;
    return result;
}

var c = multiplyFn(a, b);
console.log("Multiplication result is : ", c);
```

```
> node setTimeoutZero.js
Start
Multiplication result is : 20000
setTimeout callback
call me right now
```

- Even with `0` ms delay, `setTimeout` is **asynchronous**: The callback is offloaded to libuv and placed in the **timer queue**: It only executes **after the call stack is empty**: `setTimeout(cb, 0)` doesn't mean "run immediately" -- it means "run as soon as possible **after** the current synchronous code finishes"
