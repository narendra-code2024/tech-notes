## Multiple Route Handlers

- One route can have **multiple route handlers**

```js
app.get('/user', (req, res) => {}, (req, res) => {});
```

- Code executes synchronously -- 1st handler runs, sends response, and **won't move** to 2nd handler
- If you **remove** `res.send()` from 1st handler without calling `next()`, the request **hangs** (Postman shows "sending request..." forever)

---

## next() Function

- Pass `next` as the **3rd argument** in a route handler and call `next()` to move to the next handler

```js
app.get('/user',
    (req, res, next) => {
        console.log("Handler 1");
        next();  // passes control to next handler
    },
    (req, res) => {
        res.send("Hello from Handler 2");
    }
);
```

### next() Behavior

|Scenario|Result|
|---|---|
|`res.send()` then `next()`|Error -- response already sent, 2nd handler also tries to send|
|`next()` with no next handler|Error -- Express expects another handler|
|No `res.send()` and no `next()`|Request hangs indefinitely|

### Array of Route Handlers

```js
// You can pass handlers as an array, or mix arrays and functions
app.get('/user', [handler1, handler2], handler3);
app.get('/user', [(req, res, next) => { next(); }, (req, res) => { res.send('Hello'); }]);
```

### Separate Route Definitions with Same Path

```js
// These work like chained handlers -- next() in first passes to second
app.get('/user', (req, res, next) => { console.log("First"); next(); });
app.get('/user', (req, res) => { res.send('Hello'); });
```

## What is Middleware?

- Route handlers placed in the **middle** of the request chain (between starting point and final response)
- There is no formal concept of "middleware" vs "route handler" in Express -- it's **developer terminology**: They're called middlewares because they're called in the middle of the request chain

```
GET /user --> Middleware Chain --> Request Handler (sends response)

Express matches routes top to bottom:
  All previous functions that don't send a response = middlewares
  The function that sends the response = route handler
```

## Authorization Middleware Example

**Problem:** Writing token validation logic in every route

**Solution:** Create a reusable middleware

```js
// middleware/auth.js
const authMiddleware = (req, res, next) => {
    const token = req.headers.authorization;
    if (!token || !isValidToken(token)) {
        return res.status(401).send("Unauthorized");
    }
    next();  // authorized -- proceed to route handler
};

module.exports = authMiddleware;
```

```js
// app.js -- all /admin routes go through auth middleware first
app.use('/admin', authMiddleware);

app.get('/admin/dashboard', (req, res) => {
    res.send("Admin Dashboard");
});
```

- Generally write middleware using `app.use` because middleware should work with **all HTTP methods**: `app.all` also works with all HTTP methods (small difference -- Express community uses `app.use`)

## Error Handling

### Parameter Order Matters

|Parameters|Order|
|---|---|
|2 params|`req`, `res`|
|3 params|`req`, `res`, `next`|
|4 params|`err`, `req`, `res`, `next` (error handler)|

- If passing `err`, it **must be the first** parameter

### Error Handling Approaches

```js
// Basic approach
app.use((err, req, res, next) => {
    if (err) {
        res.status(500).send("Something went wrong");
    }
});

// Better approach: try-catch block
app.get('/user', async (req, res) => {
    try {
        const user = await findUser();
        res.send(user);
    } catch (err) {
        res.status(500).send("Something went wrong");
    }
});
```
