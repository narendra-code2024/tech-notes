## MongoDB Indexes

- For **unique fields**, MongoDB creates an index **by default** (e.g., `_id`, fields with `unique: true`)
- Indexes speed up **read queries** significantly

---

## Single Index vs Compound Index

|Index Type|When to Use|Example|
|---|---|---|
|**Single Index**|Querying by **one field**|Find users by `email`|
|**Compound Index**|Querying by **multiple fields together**|Find connections by `fromUserId` AND `toUserId`|

- If you search by both `fromUserId` and `toUserId`, a single index on each field separately **won't be effective**: You need a **compound index** on both fields together

---

## Creating Compound Index in Mongoose

```js
const connectionRequestSchema = new mongoose.Schema({
    fromUserId: { type: mongoose.Schema.Types.ObjectId, ref: 'User', required: true },
    toUserId: { type: mongoose.Schema.Types.ObjectId, ref: 'User', required: true },
    status: { type: String, enum: ["interested", "ignored", "accepted", "rejected"] }
});

// Compound index on both fields
connectionRequestSchema.index({ fromUserId: 1, toUserId: 1 });
```

---

## When to Use

```js
// This query benefits from a compound index on { firstName: 1, lastName: 1 }
User.find({ firstName: "Narendra", lastName: "Katta" });

// This query benefits from a single index on { email: 1 }
User.findOne({ email: "narendra@example.com" });
```

---

## Advantages & Disadvantages of Indexes

|Advantages|Disadvantages|
|---|---|
|Faster read queries|Slower write operations (insert, update, delete)|
|Efficient sorting|Uses additional storage space|
|Quick lookups on indexed fields|Too many indexes = overhead on every write|

- **Don't create unnecessary indexes**: each index adds overhead on writes
- Only index fields that are frequently queried or sorted
