## Interpreted vs Compiled Languages

| Interpreted Language | Compiled Language |
| --- | --- |
| Executes code **line by line** | Converts **entire code** before execution |
| Slow (no optimization) | Fast (optimized machine code) |
| No intermediate file generated | Generates machine code file |
| Detects errors **immediately** at runtime | Detects errors **after** full compilation |

- V8 uses **both** approaches -- this is called **JIT (Just-In-Time) Compilation**: -

## V8 Code Execution Process

```
JavaScript Source Code
        |
        v
   1. PARSING
   ├── Lexical Analysis (Tokenization)
   │   └── Code broken into tokens
   └── Syntax Analysis (Parsing)
       └── Tokens --> Abstract Syntax Tree (AST)
        |
        v
   2. COMPILATION (JIT)
   └── Ignition (Interpreter)
       └── AST --> Bytecode
        |
        v
   3. EXECUTION & OPTIMIZATION
   ├── Ignition executes Bytecode
   └── TurboFan (JIT Compiler)
       └── Optimizes hot code --> Machine Code
        |
        v
   4. GARBAGE COLLECTION
   └── Orinoco GC
       └── Mark-and-Sweep Algorithm
```

---

## Stage 1: Parsing

### Lexical Analysis (Tokenization)

- JavaScript code is broken into **tokens** (smallest meaningful units)

```js
// Code:  const x = 10;
// Tokens: [const] [x] [=] [10] [;]
```

### Syntax Analysis (Parsing)

- Tokens are converted into an **Abstract Syntax Tree (AST)**: **Identifier** = Variable name
- **Literal** = Variable's value
- **Syntax Error** occurs if the AST cannot be generated due to unexpected tokens

---

## Stage 2: Compilation (JIT -- Just-In-Time)

- V8 does **not** interpret JS directly -- it compiles JavaScript into machine code
- Uses **Ignition** (Interpreter) to generate **Bytecode** from the AST
- JIT = code is compiled at runtime, not ahead of time

---

## Stage 3: Execution & Optimization

- **Ignition** executes the Bytecode
- **TurboFan** (JIT Compiler) watches for frequently used code ("hot code") and optimizes it into **highly efficient machine code**: Optimization techniques:
  - **Inline caching**: caches property lookups
  - **Dead code elimination**: removes unused code
  - **Hidden classes**: optimizes object property access

---

## Stage 4: Garbage Collection

- V8 uses multiple garbage collectors:
  - **Orinoco**: main GC, handles most memory cleanup
  - **Oilpan**: manages C++ objects tied to JS objects
  - **MCompact**: compacts memory to reduce fragmentation
- **Mark-and-Sweep Algorithm** removes unused memory objects

```
Mark-and-Sweep:
  1. Mark   -- traverse from root, mark all reachable objects
  2. Sweep  -- free memory of unmarked (unreachable) objects
```

---

## Why is it Called a "Syntax Error"?

- A Syntax Error occurs when the V8 Engine **fails to generate the AST** due to an unexpected token in the code
- It happens during the **Parsing** phase, before any code is executed
