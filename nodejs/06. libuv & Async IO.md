## Restaurant Analogy -- Sync vs Async

```
Items:  [Coke: 0 min]   [Pizza: 10 min]   [Noodles: 5 min]


SYNCHRONOUS (Blocking -- A → B):

  A --> Coke    (0 min)
  B --> Noodles (5 min)
  C --> Pizza   (15 min)
  D --> Coke    (15 min)
  E --> Noodles (20 min)
        F, G, H ... still waiting!

  Each order starts ONLY after the previous one is done.


ASYNCHRONOUS (Non-Blocking -- A → D):

  All orders started together:        Waiting Area (cooking):
                                       +----------+----------+
    A --> Coke    (0 min)   served!    | Pizza    | Noodles  |
    B --> Pizza   (10 min)  ------>    |  (B)     |  (C)     |
    C --> Noodles (5 min)   ------>    +----------+----------+
    D --> Coke    (0 min)   served!    | Noodles  |
    E --> Noodles (10 min)  ------>    |  (E)     |
                                       +----------+

  Serve as ready:
    0 min  --> A, D (Coke -- instant)
    5 min  --> C (Noodles -- done cooking)
    10 min --> B, E (Pizza, Noodles -- done cooking)

  F, G, H ... keep taking orders, no waiting!
```

- **Synchronous** = finish one **entire order** before starting the next
- **Asynchronous** = start **all orders**, serve each as it's ready -- no blocking!

---

## JavaScript is Synchronous & Single-Threaded

- **Synchronous** = Blocking -- each line waits for the previous one to finish
- **Asynchronous** = Non-blocking -- tasks can run concurrently
- JavaScript is synchronous, but with **Node.js superpowers**, it becomes asynchronous

---

## Synchronous vs Asynchronous

|Synchronous (Blocking)|Asynchronous (Non-Blocking)|
|---|---|
|Code runs line by line|Code doesn't wait for long-running tasks|
|Each line blocks until complete|Offloads tasks and continues execution|
|Simple but slow for I/O|Efficient for I/O-heavy applications|

---

## How Synchronous Code Executes

- V8 Engine executes code **synchronously** using the **call stack**: Each function call is pushed onto the stack, executed, and popped off

```
Call Stack (V8):

  | multiplyFn() |  <-- pushed, executed, popped
  | console.log()|  <-- pushed, executed, popped
  |   main()     |  <-- program entry
  +--------------+
```

```js
var a = 10;
var b = 20;

function multiplyFn(x, y) {
    const result = x * y;
    return result;
}

var c = multiplyFn(a, b);
console.log(c);  // 200
// Everything runs line by line on the call stack
```

---

## How Asynchronous Code Executes

- **V8 Engine is only capable of executing synchronous code**: When V8 encounters async operations (I/O, timers, network requests), it **offloads them to libuv**: libuv interacts with the **OS or system APIs** to handle them in the background

```
+----------------+      offloads     +----------------+          +----------+
|  V8 JS Engine  |  ---------------> |     libuv      |  ------> |    OS    |
|  (Call Stack)  |                   | (Genie/Super   |          | - File   |
|                |  <-- callback --  |   Hero!)       |  <-----  | - Network|
+----------------+                   +----------------+          | - More.. |
                                                                 +----------+
```

- Once an async task completes, libuv notifies the **event loop**: Event loop queues the callback in the **task queue** (or microtask queue for promises)
- Event loop continuously checks if the **call stack is empty**: when it is, it dequeues and executes pending tasks

### Async Code Execution Example

```js
const fs = require("fs");
const https = require("https");

// V8 JS Engine is synchronous, but Node.js is asynchronous
// This allows Node.js to perform async I/O (which is nothing but V8 engine's call stack)

console.log("Before");

https.get("https://dummyjson.com/products/1", (res) => {
    console.log("Data fetched");
});

fs.readFile("./file.txt", "utf-8", (err, data) => {
    console.log("File data: " + data);
});

console.log("After");

// Output:
// Before
// After
// File data: ...    (async - offloaded to libuv)
// Data fetched      (async - offloaded to libuv)
```

## libuv

- **Multi-platform C library** that provides async I/O capabilities and an event loop
- Written in **C** because C is highly efficient for interacting with the Operating System
- V8 communicates with libuv, and libuv interacts with the OS

```
+--------------------------------------+
|             Node.js (C++)            |
|                                      |
|    +----------+    +----------+      |
|    |    V8    |    |  libuv   |      |
|    | (Google) |    |   (C)    |      |
|    +----------+    +----------+      |
|                                      |
|       "Async I/O made simple"        |
+--------------------------------------+
```

### What libuv Manages

- File system operations (I/O)
- Networking (TCP, UDP, DNS, etc.)
- Timers
- Thread pool for CPU-intensive tasks
- Process management and child processes

It acts as a **bridge between Node.js and the Operating System**, enabling high-performance, non-blocking execution.

---

## Why Node.js is Called Non-Blocking

- It does **not block the main thread** (where all JavaScript code runs -- V8 engine's call stack)
- V8 JavaScript Engine is synchronous, but **Node.js is asynchronous**: This allows Node.js to perform async I/O operations like reading files and making network requests
