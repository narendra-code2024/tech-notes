## The Web -- Big Picture

- Every computer connected to the internet has a **unique IP address**: These computers can connect and communicate with each other -- hence, it's called the **Web**: To communicate, computers follow certain **protocols**: defined sets of rules

---

## How a Request & Response Works

```
User (Client)                          Server
    |                                     |
    |  1. Types google.com in browser     |
    |  2. Request goes to DNS             |
    |         |                           |
    |    DNS resolves domain              |
    |    to IP address                    |
    |         |                           |
    |  3. Request reaches server  ------->|
    |                                     |  4. Processes request
    |  5. Response sent back  <-----------|
    |                                     |
    |  6. Browser displays response       |
    |     (HTML, JSON, images, etc.)      |
```

## What is a Server?

A server can refer to both:

|Type|Description|
|---|---|
|**Hardware**|A physical or virtual machine (like your PC or a cloud VM). Can run applications, store files, listen for incoming connections|
|**Software**|The program that listens to requests and serves data (e.g., an HTTP server like Node.js or Apache)|

When someone says "the server is down", they usually mean the software or hardware is not responding or crashed.

### Types of Servers

|Server Type|Purpose|
|---|---|
|Web Server|Serves HTTP content (HTML, CSS, JS, JSON)|
|HTTP Server|Specific type of web server handling HTTP protocol|
|Proxy Server|Acts as a middleman between client and server|
|DNS Server|Resolves domain names to IP addresses|
|FTP Server|Used to transfer files|
|SMTP Server|Used for sending emails|

## Client-Server Architecture

```
Client (Browser)                          Server (Node.js App)
  192.168.1.10                              52.23.145.67
       |                                         |
       |  1. Open socket connection               |
       |  --------- TCP handshake ---------->     |
       |                                         |
       |  2. Send HTTP request  ------------>     |
       |                                         |  3. Process request
       |  4. Receive response  <-------------     |
       |                                         |
       |  5. Connection closed                    |
       +------------------------------------------+
```

- **Client:** Something that wants to access data or services from a server (e.g., browser)
- **Server:** A computer that responds to client requests
- **Socket:** A communication channel between two devices

### TCP/IP

- **TCP** (Transmission Control Protocol) -- ensures **reliable, ordered** delivery of data
- **IP** (Internet Protocol) -- responsible for **addressing and routing** packets to the correct machine
- TCP/IP works together: IP finds the destination, TCP ensures data arrives correctly

### TCP 3-Way Handshake

```
Client                              Server
  |                                    |
  |  1. SYN  ---------------------->  |   (Client: "I want to connect")
  |                                    |
  |  2. SYN-ACK  <------------------  |   (Server: "OK, I acknowledge")
  |                                    |
  |  3. ACK  ---------------------->  |   (Client: "Great, connection established")
  |                                    |
  |  ===== Connection Established =====|
  |                                    |
  |  HTTP Request/Response happens     |
  |  over this TCP connection          |
  |                                    |
  |  4. FIN  ---------------------->  |   (Connection close)
  |  5. FIN-ACK  <------------------  |
```

- Every HTTP connection starts with a **TCP handshake** first
- **SYN** = Synchronize, **ACK** = Acknowledge, **FIN** = Finish

### HTTP on Top of TCP

- **HTTP** is an **application layer** protocol that runs **on top of TCP**: TCP handles the connection and delivery, HTTP defines the **format of request/response**

```
HTTP Request:
+-----------+---------------------------------------------+
| Method    | GET, POST, PUT, DELETE, PATCH                |
| URL       | /api/users                                   |
| Headers   | Content-Type, Authorization, Cookie, etc.    |
| Body      | JSON data (for POST/PUT)                     |
+-----------+---------------------------------------------+

HTTP Response:
+-----------+---------------------------------------------+
| Status    | 200 OK, 404 Not Found, 500 Server Error     |
| Headers   | Content-Type, Set-Cookie, etc.               |
| Body      | HTML, JSON, file data, etc.                  |
+-----------+---------------------------------------------+
```

### HTTP Methods

| Method | Purpose |
| --- | --- |
| **GET** | Retrieve data |
| **POST** | Create new data |
| **PUT** | Update/replace entire resource |
| **PATCH** | Update partial resource |
| **DELETE** | Delete a resource |

### HTTP Status Codes

| Range | Category | Examples |
| --- | --- | --- |
| **1xx** | Informational | 100 Continue |
| **2xx** | Success | 200 OK, 201 Created |
| **3xx** | Redirection | 301 Moved, 304 Not Modified |
| **4xx** | Client Error | 400 Bad Request, 401 Unauthorized, 404 Not Found |
| **5xx** | Server Error | 500 Internal Server Error, 503 Service Unavailable |

## Streams & Buffers

- Response data isn't sent all at once -- it's broken into smaller **packets**: Packets are sent over the internet and **reassembled** at the client's end

```
Server sends large file:

  [Full Data]
      |
      v
  [Chunk 1] [Chunk 2] [Chunk 3] [Chunk 4]
      |         |         |         |
      +-------- Stream (pipe) ------+
      |         |         |         |
      v         v         v         v
  [Buffer] --> [Buffer] --> [Buffer] --> [Buffer]
      |
      v
  [Reassembled Data at Client]
```

- **Stream:** A continuous flow of data -- like a pipe between two computers (e.g., YouTube video streaming)
- **Buffer:** Temporary storage that holds data chunks while being transferred

### Protocols

|Protocol|Layer|Purpose|
|---|---|---|
|**TCP/IP**|Transport/Network|Break data into packets, ensure safe delivery, correct addressing|
|**HTTP/HTTPS**|Application|Web browsing, APIs|
|**FTP**|Application|File transfer|
|**SMTP**|Application|Email sending|

TCP/IP ensures packets get to the destination safely. HTTP/FTP/SMTP define what the content means.

## Ports & Multiple Servers

- You **can** run multiple HTTP servers on one computer
- Servers are differentiated using **ports** (e.g., HTTP on port 80, another app on port 3000)
- Servers can also act as **clients**: one server can send requests to another server's endpoint (common in microservices)

## URL Structure & API Endpoints

```
Domain Name (remoteAddress)  :  PORT  /  PATH
         |                        |        |
   e.g., example.com           :3000   /getStudents

Full URL = http://example.com:3000/fs/getStudents
```

- **Domain Name**: human-readable address that maps to an IP (via DNS)
- **Port**: identifies which server/application on the machine
- **Path**: specific resource or route on the server
- **Domain + Port + Path = API Endpoint**: -

## Socket vs WebSocket

|Feature|Socket|WebSocket|
|---|---|---|
|**Type**|Low-level network connection|Protocol for two-way communication|
|**Lifetime**|Short-lived, connection closed after response|Persistent -- stays open until explicitly closed|
|**Use Case**|File transfer, HTTP request|Live chats, real-time games|

## Creating an HTTP Server

### Using Native `http` Module

```js
const http = require("node:http");

const server = http.createServer((req, res) => {
    if (req.url === "/") {
        res.end("Welcome to Home Page");
    } else if (req.url === "/about") {
        res.end("Welcome to About Page");
    } else {
        res.end("Page Not Found");
    }
});

server.listen(3000, () => {
    console.log("Server is running on port 3000");
});
```

- `http.createServer()` creates a server that listens for incoming requests
- **`req`** (request) -- contains information about the incoming request (URL, method, headers, body)
- **`res`** (response) -- used to send data back to the client
- `server.listen(port)` -- starts the server on the specified port

### Using Express.js

```js
const express = require("express");
const app = express();

app.get("/", (req, res) => {
    res.send("Welcome to Home Page");
});

app.get("/about", (req, res) => {
    res.send("Welcome to About Page");
});

app.listen(3000, () => {
    console.log("Server is running on port 3000");
});
```

- **Express.js** is a framework built on top of Node.js `http` module
- Simplifies routing, middleware, and request handling
- Much less boilerplate than native `http`

```
Native http module  -->  Express.js (easier server logic)
Raw MongoDB driver  -->  Mongoose (easier DB operations)
```
