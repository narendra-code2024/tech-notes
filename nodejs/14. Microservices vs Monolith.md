## How Projects Are Built in the Industry (SDLC)

```
Requirement Gathering --> System Design --> Development --> Testing --> Deployment --> Maintenance
       (PMs)            (Tech Leads)        (SDEs)         (SDETs)     (DevOps)        (All)
                                                                                         |
       <-----------------------------  Cyclical Process  <-------------------------------+
```

### Phase 1: Requirement Gathering

- **Purpose:** Understand what needs to be built
- **Who:** Product Managers (PMs), or founders/Tech Leads in smaller startups
- **Questions:** What is the product? Core features? Use-case scenarios? Target audience?
- PMs collaborate with **UI/UX Designers** to prepare design mockups or wireframes

### Phase 2: System Design

- **Purpose:** Define how the system will be built
- **Who:** Tech Leads, Senior Engineers, Engineering Managers
- **Decisions**: Monolithic vs Microservices architecture
  - Service communication (REST APIs, gRPC, message queues)
  - Tech stack (backend, frontend, database, cloud provider)
- **Documentation:** HLD (High-Level Design) + LLD (Low-Level Design)

### Phase 3: Development

- **Purpose:** Write the actual code
- **Who:** Software Development Engineers (SDE 1, SDE 2, etc.)
- Build features, follow best practices, write unit tests

### Phase 4: Testing

- **Purpose:** Ensure quality and correctness
- **Who:** SDETs (Software Development Engineers in Test)
- Manual testing, automation test cases, performance and security validation

### Phase 5: Deployment

- **Purpose:** Release the application to users
- **Who:** DevOps Engineers
- Manage CI/CD, servers, infrastructure, monitoring, incident management

### Phase 6: Maintenance

- Monitor performance, fix bugs, implement new features
- Forms a **cyclical process**: entire SDLC repeats for each new feature

## What are Microservices?

- An architectural style where an application is composed of **multiple small, independent services**: Each service is responsible for a **specific functionality**: Can be developed, deployed, and scaled **independently**: Services interact via **APIs** (HTTP or message brokers)

Examples: notification service, frontend service, auth service, analytics service

---

## Monolith vs Microservices

|Aspect|Monolith|Microservices|
|---|---|---|
|**Dev Speed**|Slower -- many devs in single codebase, merge conflicts|Faster -- smaller, isolated codebases|
|**Code Repo**|Usually a single repo|Each service has its own repo|
|**Scalability**|Difficult -- harder to scale individual parts|Excellent -- scale specific services independently|
|**Deployment**|Simpler -- single pipeline. But even small changes = full redeploy|Complex -- each service needs own pipeline. Deploy only what changed|
|**Tech Stack**|Restricted -- usually one stack|Highly flexible -- best stack per service|
|**Infra Cost**|Cheaper -- one app to host|More expensive -- infra per service|
|**Complexity**|Simple for small apps, grows messy|High initial complexity, manageable at scale|
|**Fault Isolation**|One bug can crash entire app|One failure doesn't affect others|
|**Testing**|Easier -- end-to-end tests straightforward|Harder -- contract testing and mocking needed|
|**Ownership**|Centralized -- one team|Decentralized -- different teams own different services|
|**Debugging**|Easier in one codebase|Distributed logs and tracing needed|
|**Revamps**|Touching one part may break another|Can rewrite one service without affecting rest|
