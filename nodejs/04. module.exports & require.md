## Node.js Module System

- Writing all code in a single file is possible but **not logical or scalable**: Real-world projects follow a **folder and directory structure** for better organization
- **Modularization** is an important concept in Node.js -- each file is treated as a **module**: Every Node.js application has a single **entry point** (e.g., `app.js`)

---

## require() and module.exports

### Without Export -- ReferenceError

```js
// sum.js
console.log("Sum Module Executed");
function calculateSum(a, b) {
    const sum = a + b;
    console.log(sum);
}

// app.js
require("./xyz.js");
require("./sum.js");
var a = 100;
var b = 200;
calculateSum(a, b);  // ReferenceError: calculateSum is not defined
```

- `require` **runs** the module code (prints "Sum Module Executed") but does **not expose** its variables/functions
- Each module has its **own scope**: protects variables from leaking into other modules

### Single Function Export

```js
// sum.js
function calculateSum(a, b) {
    const sum = a + b;
    console.log(sum);
}
module.exports = calculateSum;

// app.js
const calculateSum = require("./sum.js");
calculateSum(100, 200);  // 300
```

### Exporting Multiple Items (as Object)

```js
// sum.js
const x = "Hello World";
function calculateSum(a, b) {
    const result = a + b;
    console.log(result);
}
module.exports = {
    x: x,
    calculateSum: calculateSum,
};

// app.js
const obj = require("./sum.js");
obj.calculateSum(100, 200);  // 300
console.log(obj.x);          // Hello World
```

### Destructuring Imports

```js
// app.js -- cleaner than obj.property
const { x, calculateSum } = require("./sum");
calculateSum(100, 200);  // 300
console.log(x);           // Hello World
```

### ES6 Shorthand Export

```js
// When key and value names are the same
module.exports = { x, calculateSum };  // shorthand for { x: x, calculateSum: calculateSum }
```

### module.exports.varName Syntax

```js
// Initially, module.exports is an empty object {}
// Attaching properties one by one
module.exports.x = x;
module.exports.calculateSum = calculateSum;
```

## File Extensions & Importing Core Modules

- In **CommonJS**, you **don't need** the `.js` extension -- Node.js assumes `.js` automatically
- In **ESM**, the file extension is **required**

```js
// CommonJS
require("./sum");              // works same as require("./sum.js")

// ESM
import { x } from "./sum.js"; // .js is mandatory
```

### Importing Node.js Core Modules

Can import Node.js core modules in either way, both work fine:

```js
const fs = require("fs");
const https = require("https");
const crypto = require("crypto");

// OR with node: protocol prefix
const fs = require("node:fs");
const https = require("node:http");
const crypto = require("node:crypto");
```

## CommonJS vs ES Modules

| Feature | CommonJS (CJS) | ES Modules (ESM) |
| --- | --- | --- |
| **Import** | `require()` | `import` |
| **Export** | `module.exports` | `export` / `export default` |
| **File Extension** | `.js` | `.mjs` (or `.js` if `"type": "module"`) |
| **Loading** | Synchronous | Asynchronous |
| **Mode** | Non-strict by default | Strict mode by default |
| **Default Export** | Not enforced | Only one allowed per module |
| **Enable** | Default, or `"type": "commonjs"` in package.json | `"type": "module"` in package.json |

### ESM Syntax

```js
// sum.js (with "type": "module" in package.json)
export const x = "Hello World";
export function calculateSum(a, b) {
    const sum = a + b;
    console.log(sum);
}

// app.js
import { x, calculateSum } from "./sum.js";
import xyz from './file.js';           // default import
import { abc } from './file.js';       // named import
```

### ESM Strict Mode

```js
// sum.js (ESM)
z = "Hello World";  // ReferenceError: z is not defined (strict mode!)

// CJS -- this would work (non-strict by default)
x = 10;  // no error
```

- In ESM, you **cannot** use `require()` or `module.exports`

## Nested Modules & Folder as a Module

### Nested Modules (Individual Requires)

```
project/
├── app.js
└── calculate/
    ├── sum.js
    └── multiply.js
```

```js
// calculate/sum.js
function calculateSum(a, b) {
    const sum = a + b;
    console.log(sum);
}
module.exports = { x, calculateSum };

// calculate/multiply.js
function calculateMultiply(a, b) {
    const result = a * b;
    console.log(result);
}
module.exports = { calculateMultiply };

// app.js -- requiring individual files
const { x, calculateSum } = require("./calculate/sum");
const { calculateMultiply } = require("./calculate/multiply");
calculateSum(100, 200);       // 300
calculateMultiply(100, 200);  // 20000
```

### Folder as a Module (using index.js)

- When you have many files, use an `index.js` to create a **self-contained module**: Provides **abstraction**: `app.js` doesn't need to know the folder's internal structure

```
calculate/
├── index.js       <-- entry point for the folder module
├── sum.js
└── multiply.js
```

```js
// calculate/index.js
const { calculateSum } = require("./sum");
const { calculateMultiply } = require("./multiply");
module.exports = { calculateSum, calculateMultiply };

// app.js -- Node.js looks for index.js inside the folder
const { calculateSum, calculateMultiply } = require("./calculate");
calculateSum(100, 200);       // 300
calculateMultiply(100, 200);  // 20000
```

---

## Key Takeaways

- **Encapsulation:** Modules are encapsulated by default -- variables and functions are private unless explicitly exported
- **No access to private variables:** If not exported, you cannot access it in another file
- This ensures **code security and modularity**, reducing conflicts
