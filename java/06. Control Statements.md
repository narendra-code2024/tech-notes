## Conditional Statements

### `if` / `if-else` / `else-if`

```java
if (condition) {
    // runs if condition is true
}

if (score >= 90) {
    grade = "A";
} else if (score >= 80) {
    grade = "B";
} else if (score >= 70) {
    grade = "C";
} else {
    grade = "F";
}
```

- The evaluated condition must resolve to a `boolean` (or its wrapper `Boolean`).
- `else if` is not a standalone keyword; it is simply an `else` containing another `if` block.
- Braces are optional for single-line bodies but highly recommended to prevent **dangling-else** bugs.

### `switch` Statement

The classic `switch` statement uses `break` to prevent fall-through.

```java
switch (day) {
    case 1:
        System.out.println("Monday");
        break; // Correct: exits the switch
    case 2:
        System.out.println("Tuesday");
        break; // Correct: exits the switch

    case 3:
    case 4: // Intentional Grouping (Wednesday - Thursday)
        System.out.println("Midweek");
        break; // Correct: exits the switch

    case 5:
        System.out.println("Friday");
        // Missing break! (Accidental Fall-through Bug)
    case 6:
    case 7: // Intentional Grouping (Saturday - Sunday)
        System.out.println("Weekend"); // If day is 5, prints BOTH "Friday" and "Weekend"
        break;

    default:
        System.out.println("Unknown Day");
}
```

- Supports `byte`, `short`, `int`, `char` (and their wrappers), `String`, and `enum` types.
- Omitting `break` causes execution to continue into the next case (**fall-through**). While useful for multi-value grouping, it is a common source of bugs. Modern `switch` expressions eliminate this behavior.

### `switch` Expressions (Java 14+)

Introduced in Java 14, `switch` expressions return a value and eliminate fall-through using **arrow syntax**.

```java
String name = switch (day) {
    case 1 -> "Monday";
    case 2 -> "Tuesday";
    case 3, 4, 5 -> "Midweek";               // Multi-value case
    default -> {
        log.warn("Unknown day: " + day);
        yield "Unknown";                      // yield returns a value from a block arm
    }
};
```

**Key benefits**:
- Arrow syntax (`->`) eliminates fall-through, ensuring only the single matching branch runs.
- Multiple case constants can be comma-separated (`case 1, 2, 3 ->`).
- Evaluates directly to a value, allowing assignment or return.
- Compiler enforces **exhaustiveness** for `enum` types.

Arms can be single expressions or multi-statement blocks `{ ... }`. Block arms must use `yield` to return a value (analogous to `return` in methods); omitting `yield` causes a compilation error.

### `switch` Statement vs `switch` Expression

| Feature            | `switch` Statement            | `switch` Expression (Java 14+)   |
| :----------------- | :---------------------------- | :------------------------------- |
| Syntax             | `case X: ... break;`          | `case X -> ...`                  |
| Fall-through       | Supported (requires `break`)  | Not supported                    |
| Returns value      | No (statement)                | Yes (expression)                 |
| Exhaustiveness     | Not enforced                  | Enforced (enum types)            |
| Multi-value case   | Multiple labels               | Comma-separated list             |
| Primary use case   | Side-effects                  | Value production                 |

## Iterative Statements

### `for`

Executes based on three optional components: initialization, condition, and update. For example, `for (;;) { }` creates a legal infinite loop.

```java
for (int i = 0; i < 10; i++) {
    System.out.println(i);
}
```

### `while`

Checks the condition before executing the body, running zero or more times. Ideal when the total iteration count is unknown upfront.

```java
while (!queue.isEmpty()) {
    process(queue.poll());
}
```

### `do-while`

Executes the body at least once before evaluating the condition. Useful for retry logic or user input validation.

```java
int attempts = 0;
do {
    System.out.println("Try " + attempts);
    attempts++;
} while (attempts < 3);
```

### Enhanced `for-each` (Java 5+)

Iterates over arrays or `Iterable` collections without manual index management.

```java
List<String> names = List.of("Alice", "Bob", "Charlie");
for (String name : names) {
    System.out.println(name);
}

// Works on arrays and any Iterable; use entrySet() for maps
Map<String, Integer> scores = Map.of("Alice", 90, "Bob", 85);
for (Map.Entry<String, Integer> entry : scores.entrySet()) {
    System.out.println(entry.getKey() + ": " + entry.getValue());
}
```

Use a classic `for` loop instead of enhanced `for-each` when:
- Modifying the collection during iteration (otherwise throws `ConcurrentModificationException`).
- Accessing the current element's index is required.
- Iterating in reverse order.
- Iterating multiple collections in parallel.

## Branching Statements

### `break`

Exits the nearest enclosing loop or `switch` block.

```java
for (int i = 0; i < 100; i++) {
    if (i == 50) break;   // Exits loop
    System.out.println(i);
}
```

### `continue`

Skips the rest of the current iteration and moves to the next.

```java
for (int i = 0; i < 10; i++) {
    if (i % 2 == 0) continue;    // Skip even numbers
    System.out.println(i);       // Prints 1, 3, 5, 7, 9
}
```

### `return`

Exits the current method, optionally returning a value.

```java
boolean isValid(String s) {
    if (s == null) return false;
    if (s.isEmpty()) return false;
    return s.length() < 100;
}
```

### Labeled Loops

Enables control of nested loops by target labeling, overriding default single-level behavior.

```java
search:
for (int i = 0; i < matrix.length; i++) {
    for (int j = 0; j < matrix[i].length; j++) {
        if (matrix[i][j] == target) {
            System.out.println("Found at " + i + "," + j);
            break search;   // Exits both loops
        }
    }
}
```

> **Note**: Labeled loops are generally avoided. Use method extraction with early `return` instead.

## Q&A

1. **What is "fall-through" in a switch statement, and why is it dangerous?**
In a classic `switch`, omitting `break` causes execution to continue into the next `case` block regardless of matching. This fall-through behavior is dangerous because it is often accidental, introducing silent logical bugs.

2. **Why does the expression `if (a = true)` compile, and what is the likely mistake?**
The assignment expression `a = true` assigns `true` to `a` and evaluates to `true`, making the condition always pass. The compiler cannot catch this mistake for `boolean` types because the assignment itself evaluates to a boolean.

3. **What is a `ConcurrentModificationException`, and why does it occur in an enhanced `for-each` loop?**
Modifying a collection directly while iterating via an enhanced loop invalidates the underlying `Iterator` state, triggering `ConcurrentModificationException`. To modify safely during iteration, use an explicit `Iterator` and its `remove()` method.

4. **When is a `default` case mandatory in a switch expression?**
A `default` case is mandatory in a `switch` expression to guarantee compile-time exhaustiveness, unless the expression matches on all constants of an `enum`.

5. **Why could a loop like `for (int i = 0; i <= Integer.MAX_VALUE; i++)` become an infinite loop?**
When `i` reaches `Integer.MAX_VALUE`, incrementing it causes a silent numeric overflow, wrapping the value around to `Integer.MIN_VALUE`. Since `Integer.MIN_VALUE` remains less than the comparison limit, the loop condition never fails.

6. **Can you use qualified names like `case Color.RED:` in a switch statement?**
No. Because the compiler already knows the enum type from the switch variable, you must use the bare constant name (e.g., `case RED:`). Adding the enum type prefix (e.g., `case Color.RED:`) confuses the compiler and causes a compilation error.
