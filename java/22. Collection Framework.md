## Hierarchy Overview

**Collection vs Collections:** `Collection`, interface (root of List, Set, Queue hierarchy). `Collections`, utility class with static methods (sort, reverse, shuffle, unmodifiableList, etc.)

```
Iterable
 └── Collection
      ├── List  ──→ ArrayList, LinkedList, Vector (→ Stack)
      ├── Set   ──→ HashSet (→ LinkedHashSet), SortedSet (→ TreeSet)
      └── Queue ──→ PriorityQueue, Deque (→ ArrayDeque, LinkedList)

Map (separate hierarchy, NOT a Collection)
 ├── HashMap (→ LinkedHashMap)
 ├── SortedMap (→ TreeMap)
 ├── Hashtable
 └── ConcurrentHashMap
```

## List (Ordered, Duplicates Allowed, Index-based)

|Class|Backed By|Thread-Safe?|Notes|
|---|---|---|---|
|**ArrayList**|Dynamic array|No|Fast random access O(1), slow insert/delete in middle O(n). Best for read-heavy use cases. Initial capacity 10, grows by 50% when full (`newCapacity = oldCapacity + oldCapacity >> 1`)|
|**LinkedList**|Doubly linked list|No|O(1) insert/delete at head/tail only, otherwise O(n) to traverse. Also implements Deque. Rarely preferred in real-world apps, ArrayList usually performs better due to cache locality|
|**Vector**|Dynamic array|Yes (synchronized)|Legacy. Use `Collections.synchronizedList()` or `CopyOnWriteArrayList` instead|
|**Stack**|Extends Vector|Yes|Legacy. Use `ArrayDeque` as a stack instead|

list.subList(from, to);      // view [from, to)

## Set (No Duplicates)

|Class|Ordering|Null?|Notes|
|---|---|---|---|
|**HashSet**|No order|1 null allowed|O(1) add/remove/contains. Uses HashMap internally|
|**LinkedHashSet**|Insertion order|1 null allowed|Slightly slower than HashSet due to linked list overhead|
|**TreeSet**|Sorted (natural or Comparator)|No nulls|O(log n). Implements `NavigableSet`. Elements must be Comparable or supply a Comparator|

**Why no nulls in TreeSet?**: Because the comparator/compareTo needs to compare values. It can't compare `null` → throws `NullPointerException`.

**Custom classes in TreeSet (e.g. Employee):** They have NO natural ordering by default → `ClassCastException` at runtime. Fix 1: `class Employee implements Comparable<Employee>` and override `compareTo()` Fix 2: Pass a Comparator → `new TreeSet<>((a, b) -> a.name.compareTo(b.name))` If both exist, **Comparator wins** (overrides natural ordering).

**Classes that already have natural ordering (no extra work needed):** `String` (alphabetical), `Integer`, `Long`, `Double`, `Float` (numeric), `Character` (Unicode value), `BigDecimal`, `BigInteger` (numeric), `LocalDate`, `LocalTime`, `LocalDateTime` (chronological), `Enum` (declaration order in enum definition)

### Key Methods

```java
set.add(element);       // returns false if duplicate
set.remove(obj);
set.contains(obj);
set.size();

// TreeSet extras
treeSet.first();         // smallest
treeSet.last();          // largest
treeSet.headSet(e);      // elements < e
treeSet.tailSet(e);      // elements >= e
treeSet.subSet(from, to);
treeSet.ceiling(e);      // smallest element >= e
treeSet.floor(e);        // largest element <= e
```

## Queue & Deque

|Class|Notes|
|---|---|
|**PriorityQueue**|Min-heap by default. O(log n) offer/poll. Not thread-safe. No nulls|
|**ArrayDeque**|Resizable array. Faster than LinkedList as queue/stack. No nulls|
|**LinkedList**|Also implements Deque (already in List section). Allows nulls. Use ArrayDeque unless you need null support|

### Queue Methods (FIFO)

```java
// Throws exception     |  Returns null/false
queue.add(e);           |  queue.offer(e);      // insert
queue.remove();         |  queue.poll();         // remove head
queue.element();        |  queue.peek();         // examine head
```

### Deque as Stack (LIFO)

```java
deque.push(e);     // addFirst
deque.pop();       // removeFirst
deque.peek();      // peekFirst
```

### Deque Full Methods (Double-ended)

|Action|Head|Tail|
|---|---|---|
|Add|`offerFirst(e)`|`offerLast(e)`|
|Remove|`pollFirst()`|`pollLast()`|
|Examine|`peekFirst()`|`peekLast()`|

All return `false` (add) or `null` (remove/examine) on full/empty, no exceptions.

### PriorityQueue

Head is always the smallest (or highest priority) element. But it is **NOT fully sorted internally**: only heap ordering is guaranteed. Only `poll()` guarantees smallest-first; iteration does not.

```java
PriorityQueue<Integer> pq = new PriorityQueue<>();
pq.addAll(List.of(30, 10, 20));
pq.poll();  // 10 (smallest)
pq.poll();  // 20
// But iterating (for-each) does NOT guarantee 10, 20, 30 order
```

### PriorityQueue with Custom Comparator

```java
// Max-heap
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());

// Custom object, sort by salary ascending
PriorityQueue<Employee> pq = new PriorityQueue<>((a, b) -> a.salary - b.salary);

// int[] pairs, sort by second element (index 1)
// e.g. {id, age} → sorts by age
PriorityQueue<int[]> pq = new PriorityQueue<>((a, b) -> a[1] - b[1]);
```

## Map (Key-Value Pairs, Keys Unique)

|Class|Ordering|Null Key?|Thread-Safe?|
|---|---|---|---|
|**HashMap**|No order|1 null key|No|
|**LinkedHashMap**|Insertion order|1 null key|No|
|**TreeMap**|Sorted by key|No null key|No|
|**Hashtable**|No order|No nulls at all|Yes (legacy)|
|**ConcurrentHashMap**|No order|No nulls|Yes (preferred)|

**Why no nulls in ConcurrentHashMap?**: In concurrent reads, `get(key)` returning `null` becomes ambiguous, is the key absent or is the value `null`?

**Why no null key in TreeMap?**: Same as TreeSet, comparator can't compare `null`.

**Custom class as TreeMap key (e.g. Employee):** Same rule as TreeSet, key class must be Comparable or supply a Comparator. `new TreeMap<>((a, b) -> a.salary - b.salary)`, sorts by salary. This applies to **keys only**.

### HashMap Internal Working (Interview Favorite)

```
put(key, value) flow:
1. key.hashCode() → compute bucket index
2. If bucket empty → insert new node
3. If collision (bucket occupied):
   → Walk chain, call equals() on each key
   → Match found → replace value
   → No match → add to end of chain/tree

Treeify (Java 8+): chain in a bucket converts to red-black tree when:
- Chain length ≥ 8 (collisions in ONE bucket) AND capacity ≥ 64 (total buckets)
- If capacity < 64 → resizes instead (more buckets = fewer collisions)

Defaults: capacity = 16, load factor = 0.75
Resizes (doubles) when size > capacity × load factor
```

### Key Methods

```java
map.put(key, value);
map.get(key);                          // null if absent
map.getOrDefault(key, defaultVal);
map.containsKey(key);
map.containsValue(value);
map.remove(key);
map.putIfAbsent(key, value);
map.replace(key, newValue);
map.size();
map.isEmpty();
map.keySet();                          // Set<K>
map.values();                          // Collection<V>
map.entrySet();                        // Set<Map.Entry<K,V>>

// Compute helpers (very useful)
map.merge(key, val, (oldV, newV) -> oldV + newV);
map.computeIfAbsent(key, k -> new ArrayList<>());  // great for grouping

// Example: frequency count

```

## Iteration Techniques (All Collections)

### How iteration works under the hood

**Three things to know**: `Iterable`, interface that says "I can be iterated". Has one method: `iterator()`. This is what enables the for-each loop.
- `Iterator`, interface that does the actual iteration. Has `hasNext()`, `next()`, `remove()`.
- `iterator()`, method defined in `Iterable` that returns an `Iterator` object.

**The chain:** `ArrayList implements List → extends Collection → extends Iterable` So ArrayList must provide `iterator()` method → which returns an `Iterator`.

```java
// for-each is just shorthand for Iterator:
for (String s : list) { }

// is actually:
Iterator<String> it = list.iterator();  // Iterable gives you an Iterator
while (it.hasNext()) {
    String s = it.next();               // Iterator walks through elements
}

// One collection can give multiple independent iterators:
Iterator<String> it1 = list.iterator();  // starts at beginning
Iterator<String> it2 = list.iterator();  // independent, also at beginning
```

### Iterator (safe removal during iteration)

```java
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    if (s.equals("remove")) it.remove();  // SAFE
}
```

### for-each loop (shorthand for Iterator)

```java
for (String s : list) { }
```

### `forEach()` method (Java 8+, no stream needed)

Comes from `Iterable` interface, not Stream. Takes a `Consumer` functional interface. `Consumer<T>` has one method `accept(T t)`, takes an input, returns nothing (void). Just performs an action.

```java
list.forEach(x -> System.out.println(x));
list.forEach(System.out::println);               // method reference
set.forEach(System.out::println);                 // works on any Collection
map.forEach((k, v) -> System.out.println(k + "=" + v));  // Map has its own forEach
```

### ListIterator (bidirectional, List only)

```java
// set(), replace while iterating
ListIterator<String> li = names.listIterator();
while (li.hasNext()) {
    String name = li.next();
    li.set(name.toUpperCase());  // replaces current element
}
// ["alice", "bob"] → ["ALICE", "BOB"]

// add(), insert while iterating
ListIterator<String> li = list.listIterator();
while (li.hasNext()) {
    li.next();
    li.add("---");  // inserts after current
}
// ["A", "B", "C"] → ["A", "---", "B", "---", "C", "---"]

// remove(), remove while iterating
ListIterator<String> li = names.listIterator();
while (li.hasNext()) {
    if (li.next().length() < 4) {
        li.remove();  // removes current element
    }
}
// ["Alice", "Bob", "Tom", "David"] → ["Alice", "David"]

// previous(), iterate backward
ListIterator<String> li = list.listIterator(list.size());
while (li.hasPrevious()) {
    System.out.println(li.previous());
}
// prints C, B, A
```

Streams, see [24. Streams API](<24. Streams API (Java 8+).md>).

## Comparable vs Comparator

Both are **functional interfaces** (single abstract method).

||Comparable|Comparator|
|---|---|---|
|Package|`java.lang`|`java.util`|
|Method|`compareTo(T o)`|`compare(T o1, T o2)`|
|@FunctionalInterface|No (technically qualifies, but not annotated)|Yes|
|Modifies class?|Yes (implements in the class)|No (external, passed as arg)|
|Use case|Default/natural ordering|Custom/multiple orderings|
|Lambda possible?|No, used in class definition (`implements`)|Yes, passed as method argument|

**Why lambda works with Comparator but not Comparable:** Lambda replaces an **object being passed as an argument**, not a class being defined. Comparator is passed as an argument → lambda works. Comparable is implemented in a class definition → lambda doesn't apply.

```java
// Comparable – inside the class, class says "I know how to compare myself"
class Student implements Comparable<Student> {
    int marks;
    public int compareTo(Student other) {
        return this.marks - other.marks; // ascending
    }
}

// Comparator – outside, someone else defines the sorting logic
// All three below are equivalent:
Collections.sort(students, (a, b) -> b.marks - a.marks);          // lambda
Collections.sort(students, Comparator.comparingInt(Student::getMarks)); // method ref
Collections.sort(students, new Comparator<Student>() {             // anonymous class
    public int compare(Student a, Student b) {
        return b.marks - a.marks;
    }
});

// Chained comparator
students.sort(Comparator.comparing(Student::getName)
                        .thenComparingInt(Student::getMarks).reversed());
```

**Sorting order shortcut:** `a - b` / `this - other` → ascending `b - a` / `other - this` → descending Returns: negative (first comes first), zero (equal), positive (second comes first)

Warning: **Overflow bug**: `a - b` can overflow for large integers → wrong ordering. Safer alternatives: `Integer.compare(a, b)` or `Comparator.comparingInt(e -> e.salary)`

**Common method signatures that accept Comparator (where lambda fits):**

```java
// Collections.sort(), two versions
Collections.sort(list);              // natural ordering (Comparable)
Collections.sort(list, comparator);  // your Comparator ← lambda fits here

// List.sort(), always needs Comparator
list.sort(comparator);

// TreeSet / TreeMap constructors
new TreeSet<>();              // natural ordering
new TreeSet<>(comparator);    // your Comparator

new TreeMap<>();              // natural ordering
new TreeMap<>(comparator);    // your Comparator

// PriorityQueue constructors
new PriorityQueue<>();                      // min-heap, natural ordering
new PriorityQueue<>(comparator);            // your ordering
new PriorityQueue<>(capacity);              // initial capacity, natural ordering
new PriorityQueue<>(capacity, comparator);  // both

// Stream.sorted()
stream.sorted();              // natural ordering
stream.sorted(comparator);   // your Comparator
```

Pattern: version without Comparator → relies on Comparable (natural ordering). Version with Comparator → that's where lambda/method reference fits.

## Bulk Operations (Work on Any Collection)

```java
list.addAll(anotherList);       // add entire collection
list.removeAll(anotherList);    // remove everything that exists in another
list.retainAll(anotherList);    // keep ONLY what exists in another (intersection)
list.containsAll(anotherList);  // check if all elements exist, returns boolean
list.clear();                   // remove everything

// removeIf, bulk remove by condition
// Takes a Predicate functional interface, has test(T t) method, returns boolean
// The lambda IS the Predicate: n -> (n % 2 == 0) means test(n) returns true if even
list.removeIf(s -> s.startsWith("a"));
set.removeIf(n -> n % 2 == 0);
map.values().removeIf(v -> v < 100);              // remove entries by value
map.entrySet().removeIf(e -> e.getValue() == null); // remove entries with null values
```

**Set operations using bulk ops (interview favorite):**

```java
Set<Integer> a = new HashSet<>(Set.of(1, 2, 3, 4));
Set<Integer> b = Set.of(3, 4, 5, 6);

a.addAll(b);       // Union:        {1, 2, 3, 4, 5, 6}
a.retainAll(b);    // Intersection: {3, 4}
a.removeAll(b);    // Difference:   {1, 2}
a.containsAll(b);  // Subset check: false
```

These methods accept `Collection` as parameter, you can mix types: `list.addAll(set)` works.

## Fail-Fast vs Fail-Safe Iterators

**The problem:** You're iterating a collection and it gets modified during iteration.

```java
// Fail-fast, ArrayList crashes
List<String> list = new ArrayList<>(List.of("A", "B", "C"));
for (String s : list) {
    if (s.equals("B")) list.remove(s);  // ConcurrentModificationException! 💥
}

// Fail-safe, CopyOnWriteArrayList works on a snapshot
CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>(List.of("A", "B", "C"));
for (String s : list) {
    if (s.equals("B")) list.remove(s);  // No exception, iterating on a copy
}
```

**How fail-fast works:** Collections like ArrayList maintain an internal `modCount`. Every add/remove increments it. During iteration, if iterator sees `modCount` changed → throws `ConcurrentModificationException`. It's best-effort, not guaranteed to catch every case.

**How fail-safe works:** Iterates on a copy/snapshot of the data. Modifications don't affect the ongoing iteration, but iterator won't see changes made during the loop.

||Fail-Fast|Fail-Safe|
|---|---|---|
|Behavior|Crashes immediately on modification|Works on snapshot, never crashes|
|Collections|ArrayList, HashMap, HashSet, TreeMap|ConcurrentHashMap, CopyOnWriteArrayList|
|Safe removal|Use `Iterator.remove()` or `removeIf()`|Can modify freely during iteration|

**Real-world scenario:** An e-commerce app has a list of active orders. One thread iterates to send notifications, another thread adds new orders or cancels existing ones. With `ArrayList` → crash. With `CopyOnWriteArrayList` → notification thread works on a snapshot safely while other threads modify the list.

Single-threaded code: just use `Iterator.remove()` or `removeIf()`, fail-fast/fail-safe doesn't matter much.

## Quick Decision Guide

|Need|Use|Why|
|---|---|---|
|Ordered, index access, duplicates OK|`ArrayList`|Read-heavy, fast random access|
|Frequent insert/delete at both ends|`ArrayDeque`|Faster than LinkedList in practice|
|Unique elements, no order needed|`HashSet`|O(1) lookups|
|Unique elements, sorted|`TreeSet`|Auto-sorted, O(log n)|
|Unique elements, insertion order|`LinkedHashSet`|Hash speed + order preserved|
|Key-value, fast lookup|`HashMap`|Default choice for most cases|
|Key-value, sorted keys|`TreeMap`|When sorted key order required|
|Key-value, insertion order|`LinkedHashMap`|Hash speed + insertion order|
|FIFO queue|`ArrayDeque`|Faster than LinkedList as queue|
|Priority-based processing|`PriorityQueue`|Min/max heap|
|Stack (LIFO)|`ArrayDeque`|Replaces legacy Stack class|
|Thread-safe map|`ConcurrentHashMap`|Best concurrent map|
|Thread-safe list (read-heavy)|`CopyOnWriteArrayList`|Snapshot on write|

## Q&A

1. **Why is it critical to override both `equals()` and `hashCode()` when using objects in a `HashSet` or as a `HashMap` key?**
   Hash-based collections use the `hashCode()` to determine the bucket for storage and the `equals()` method to find the exact object within that bucket. If two objects are equal according to `equals()`, they **must** return the same `hashCode`. If they don't, the collection will fail to find or correctly manage the objects, leading to silent bugs like duplicate entries in a `Set` or "lost" values in a `Map`.

2. **What is the behavior of a list returned by `Arrays.asList()`, and how does it differ from a standard `ArrayList`?**
   `Arrays.asList()` returns a **fixed-size** list that is a wrapper (view) over the original array. This means you cannot `add()` or `remove()` elements (doing so throws `UnsupportedOperationException`). Furthermore, the list and the array share the same memory; changing an element in the array will change it in the list, and vice versa. To get an independent, fully mutable list, you should wrap it: `new ArrayList<>(Arrays.asList(arr))`.

3. **How do the `of()` factory methods (e.g., `List.of()`, `Set.of()`) differ from standard collection implementations?**
   The factory methods introduced in Java 9 create collections that are **fully immutable**. Any attempt to modify them will throw an `UnsupportedOperationException`. Additionally, these collections are strictly "null-hostile", they will throw a `NullPointerException` if you try to add a `null` value or even check for the existence of `null`.

4. **In a `List<Integer>`, how does Java distinguish between `remove(1)` and `remove(Integer.valueOf(1))`?**
   The `List` interface has two overloaded `remove` methods: `remove(int index)` and `remove(Object o)`. If you pass a primitive `int`, Java assumes you want to remove the element at that **index**. If you want to remove an element by its **value**, you must pass it as an object (e.g., by using `Integer.valueOf(1)` or casting the primitive).

5. **How can you implement a simple LRU (Least Recently Used) cache using a standard Java collection?**
   You can use a `LinkedHashMap` by passing `true` for the `accessOrder` parameter in its constructor. You then override the `removeEldestEntry` method to return `true` when the map exceeds your desired maximum size.

   ```java
   Map<K,V> lru = new LinkedHashMap<>(16, 0.75f, true) {
       protected boolean removeEldestEntry(Map.Entry<K,V> e) {
           return size() > MAX_SIZE;
       }
   };
   ```

6. **What is the most efficient way to iterate over all entries in a `HashMap`?**
   The most efficient way is to iterate over the `entrySet()` using an `Iterator` or a for-each loop. This allows you to access both the key and the value in a single operation. You should avoid iterating over the `keySet()` and then calling `get(key)` for each entry, as this performs a redundant and expensive hash lookup for every single element.

7. **How do `TreeSet` and `TreeMap` determine if an element is a duplicate?**
   Unlike other collections, `TreeSet` and `TreeMap` do **not** use `equals()` or `hashCode()`. They rely entirely on the `compareTo()` (or `compare()`) method. If the comparison returns `0`, the elements are considered identical, and the new element will not be added. This can lead to bugs where distinct objects (e.g., two employees with the same salary) are treated as duplicates if the comparator only considers the salary.

8. **Why does `Collection.add(E e)` take the generic type, while `Collection.remove(Object o)` takes a plain `Object`?**
   `add(E e)` is restricted to the generic type to enforce type safety at compile time, adding the wrong type is always a logic error. However, `remove(Object o)` (along with `contains` and `indexOf`) accepts any `Object` because these methods work by calling `equals()`. Searching for or trying to remove an object of the "wrong" type is not an error; it is simply a valid search that will safely return `false` because the objects are not equal.
