`Collection` is the root interface for the `List`, `Set`, and `Queue` families; `Collections` (with an `s`) is an unrelated utility class of static helpers (`sort`, `reverse`, `unmodifiableList`). `Map` is a separate hierarchy and is not a `Collection`.

```
Iterable
 └── Collection
      ├── List  ──→ ArrayList, LinkedList, Vector (→ Stack)
      ├── Set   ──→ HashSet (→ LinkedHashSet), SortedSet (→ TreeSet)
      └── Queue ──→ PriorityQueue, Deque (→ ArrayDeque, LinkedList)

Map (separate hierarchy, not a Collection)
 ├── HashMap (→ LinkedHashMap)
 ├── SortedMap (→ TreeMap)
 ├── Hashtable
 └── ConcurrentHashMap
```

## List (Ordered, Duplicates, Index-based)

| Class      | Backed by          | Thread-safe       | Notes                                                                                                                            |
| ---------- | ------------------ | ----------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| ArrayList  | Dynamic array      | No                | Random access O(1), mid insert/delete O(n). Initial capacity 10, grows 50% (`newCap = oldCap + (oldCap >> 1)`). Default choice.  |
| LinkedList | Doubly linked list | No                | O(1) insert/delete at the ends, O(n) to reach the middle. Also a `Deque`. Rarely beats `ArrayList` due to cache locality.        |
| Vector     | Dynamic array      | Yes (synchronized)| Legacy. Prefer `Collections.synchronizedList()` or `CopyOnWriteArrayList`.                                                       |
| Stack      | Extends Vector     | Yes               | Legacy. Use `ArrayDeque` as a stack.                                                                                             |

### Key Methods

```java
list.add(e);              // append
list.add(index, e);       // insert at index
list.get(index);
list.set(index, e);       // replace
list.remove(index);       // by index
list.remove(obj);         // by value, first occurrence
list.indexOf(obj);        // first index, -1 if absent
list.subList(from, to);   // view over [from, to)
list.sort(comparator);
```

## Set (No Duplicates)

| Class         | Ordering                      | Null         | Notes                                                                       |
| ------------- | ----------------------------- | ------------ | --------------------------------------------------------------------------- |
| HashSet       | None                          | 1 null       | O(1) add/remove/contains. Backed by a `HashMap`.                            |
| LinkedHashSet | Insertion order               | 1 null       | HashSet plus a linked list; slight overhead for the ordering.               |
| TreeSet       | Sorted (natural or Comparator)| No nulls     | O(log n). Implements `NavigableSet`. Elements must be ordered.              |

- **No nulls in TreeSet**: the comparator (or `compareTo`) cannot order `null`, so it throws `NullPointerException`.
- **Custom element types**: a class with no natural ordering throws `ClassCastException` in a `TreeSet`. Fix by `implements Comparable<T>` (override `compareTo`) or by passing a `Comparator` to the constructor; if both exist, the `Comparator` wins.
- **Already ordered** (no extra work): `String` (lexicographic), the numeric wrappers and `BigDecimal`/`BigInteger`, `Character` (code point), the `java.time` types, and `enum` constants (declaration order).

### Key Methods

```java
set.add(e);          // returns false if already present

// TreeSet / NavigableSet extras
ts.first();  ts.last();
ts.ceiling(e);       // smallest element >= e
ts.floor(e);         // largest element <= e
ts.headSet(e);       // elements < e
ts.tailSet(e);       // elements >= e
```

## Queue & Deque

| Class         | Notes                                                                        |
| ------------- | ---------------------------------------------------------------------------- |
| PriorityQueue | Min-heap by default. O(log n) offer/poll. Not thread-safe. No nulls.         |
| ArrayDeque    | Resizable array. Faster than `LinkedList` as a queue or stack. No nulls.     |
| LinkedList    | Also a `Deque`, and allows nulls. Prefer `ArrayDeque` unless you need nulls. |

### Queue Operations (FIFO)

Each operation has a throwing form and a special-value form; prefer the latter to branch on failure instead of catching.

| Operation    | Throws on failure | Returns special |
| ------------ | ----------------- | --------------- |
| Insert       | `add(e)`          | `offer(e)`      |
| Remove head  | `remove()`        | `poll()`        |
| Examine head | `element()`       | `peek()`        |

### Deque

`ArrayDeque` is the modern stack and double-ended queue. As a stack, `push`/`pop`/`peek` map to `addFirst`/`removeFirst`/`peekFirst`. The full double-ended API adds at either end and returns `false`/`null` on empty rather than throwing.

| Action  | Head           | Tail          |
| ------- | -------------- | ------------- |
| Add     | `offerFirst(e)`| `offerLast(e)`|
| Remove  | `pollFirst()`  | `pollLast()`  |
| Examine | `peekFirst()`  | `peekLast()`  |

### PriorityQueue

The head is always the smallest (or highest-priority) element, but the heap is not fully sorted: only `poll()` returns elements in order, iteration does not.

```java
PriorityQueue<Integer> pq = new PriorityQueue<>();   // min-heap
pq.addAll(List.of(30, 10, 20));
pq.poll();   // 10, then 20, then 30; for-each order is unspecified

PriorityQueue<Integer> max = new PriorityQueue<>(Collections.reverseOrder());
PriorityQueue<Employee> byPay = new PriorityQueue<>(Comparator.comparingInt(e -> e.salary));
PriorityQueue<int[]> byAge = new PriorityQueue<>(Comparator.comparingInt(a -> a[1]));  // {id, age}
```

## Map (Key-Value, Unique Keys)

| Class             | Ordering        | Null key | Thread-safe    |
| ----------------- | --------------- | -------- | -------------- |
| HashMap           | None            | 1 null   | No             |
| LinkedHashMap     | Insertion order | 1 null   | No             |
| TreeMap           | Sorted by key   | No       | No             |
| Hashtable         | None            | No       | Yes (legacy)   |
| ConcurrentHashMap | None            | No       | Yes (preferred)|

- **No nulls in ConcurrentHashMap**: under concurrent access a `null` from `get` is ambiguous, absent key or null value, so both are banned.
- **No null key in TreeMap**: like `TreeSet`, the comparator cannot order `null`.
- **Custom key type**: same rule as `TreeSet`, the key must be `Comparable` or the map must receive a `Comparator`; this applies to keys only.

### HashMap Internals

```
put(key, value):
1. key.hashCode(), spread, then mask to a bucket index
2. bucket empty       → insert node
3. collision (occupied) → walk the chain, compare keys with equals()
                          match → replace value
                          no match → append to chain (or tree)
```

- **Treeify** (Java 8+): a bucket's chain becomes a red-black tree when its length reaches 8 and total capacity is at least 64; below capacity 64 the map resizes instead (more buckets, fewer collisions).
- **Sizing**: default capacity 16, load factor 0.75; the table doubles once size exceeds capacity × load factor.

### Key Methods

```java
map.getOrDefault(key, def);
map.putIfAbsent(key, val);
map.keySet();   map.values();   map.entrySet();

// compute helpers
map.merge(key, val, (oldV, newV) -> oldV + newV);
map.computeIfAbsent(key, k -> new ArrayList<>());   // grouping into lists
map.computeIfPresent(key, (k, v) -> v + 1);

// frequency count
map.merge(word, 1, Integer::sum);
```

### Iterating a Map

```java
for (Map.Entry<String, Integer> e : map.entrySet())   // most efficient: key + value in one step
    System.out.println(e.getKey() + " = " + e.getValue());

map.forEach((k, v) -> System.out.println(k + " = " + v));
// avoid keySet() + get(key): a redundant hash lookup per entry
```

## Iteration

### Under the Hood

- `Iterable`: declares `iterator()`; implementing it is what enables the for-each loop.
- `Iterator`: does the walking via `hasNext()`, `next()`, `remove()`.
- `ArrayList implements List → Collection → Iterable`, so it must supply an `Iterator`.

```java
for (String s : list) { }      // desugars to:
Iterator<String> it = list.iterator();
while (it.hasNext()) { String s = it.next(); }
// each iterator() call returns an independent cursor starting at the beginning
```

### Iterator (Safe Removal)

```java
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("x")) it.remove();   // safe; list.remove() here would throw
}
```

### forEach (Java 8+)

Defined on `Iterable` (not Stream) and on `Map`; takes a `Consumer`.

```java
list.forEach(System.out::println);
map.forEach((k, v) -> System.out.println(k + "=" + v));
```

### ListIterator (Bidirectional, List Only)

Adds `set`, `add`, and backward traversal to a plain `Iterator`.

```java
ListIterator<String> li = list.listIterator();
while (li.hasNext()) {
    String s = li.next();
    li.set(s.toUpperCase());   // replace current
    li.add("---");             // insert after current
}

ListIterator<String> back = list.listIterator(list.size());
while (back.hasPrevious()) System.out.println(back.previous());   // reverse order
```

> **Reference**: stream-based traversal is covered in [24. Streams API](<24. Streams API (Java 8+).md>).

## Comparable vs Comparator

| Dimension          | Comparable                     | Comparator                  |
| ------------------ | ------------------------------ | --------------------------- |
| Package            | `java.lang`                    | `java.util`                 |
| Method             | `compareTo(T o)`               | `compare(T a, T b)`         |
| Lives in           | The class itself               | External, passed as an arg  |
| Use case           | One natural ordering           | Custom or multiple orderings|
| Lambda             | No (defined via `implements`)  | Yes (passed as an argument) |

A lambda can replace a `Comparator` because it is supplied as an argument, whereas `Comparable` is fixed in a class definition where a lambda has no place.

```java
class Student implements Comparable<Student> {   // natural ordering baked into the type
    int marks;
    public int compareTo(Student o) { return Integer.compare(this.marks, o.marks); }
}

// Comparator: supplied externally, several equivalent forms
students.sort((a, b) -> b.marks - a.marks);                  // lambda, descending
students.sort(Comparator.comparingInt(s -> s.marks));        // method-style, ascending
students.sort(Comparator.comparing(Student::getName)
                        .thenComparingInt(s -> s.marks)
                        .reversed());                        // reversed() flips the whole chain
```

The subtraction trick (`a - b` ascending, `b - a` descending) returns negative/zero/positive for first/equal/second.

> **Note**: `a - b` can overflow for large `int` values and produce the wrong order; prefer `Integer.compare(a, b)` or `Comparator.comparingInt(...)`.

## Bulk Operations

```java
list.addAll(other);        // append a whole collection
list.removeAll(other);     // remove everything present in other
list.retainAll(other);     // keep only what is present in other (intersection)
list.containsAll(other);   // boolean: all present
list.removeIf(s -> s.startsWith("a"));            // remove by predicate
map.entrySet().removeIf(e -> e.getValue() == null);
```

These accept any `Collection`, so types can mix (`list.addAll(set)`). On `Set` the same three drive the set algebra:

```java
Set<Integer> a = new HashSet<>(Set.of(1, 2, 3, 4));
a.addAll(Set.of(5, 6));        // union
a.retainAll(Set.of(3, 4, 5));  // intersection
a.removeAll(Set.of(1, 2));     // difference
```

## Fail-Fast vs Fail-Safe Iterators

Both describe what an iterator does when the collection is modified mid-iteration.

```java
// Fail-fast: structural change during the loop is detected
List<String> list = new ArrayList<>(List.of("A", "B", "C"));
for (String s : list)
    if (s.equals("B")) list.remove(s);   // ConcurrentModificationException

// Fail-safe: iterate a snapshot, modifications do not disturb the loop
var safe = new CopyOnWriteArrayList<>(List.of("A", "B", "C"));
for (String s : safe)
    if (s.equals("B")) safe.remove(s);   // no exception; loop does not see the change
```

- **Fail-fast**: collections track a `modCount`; the iterator throws `ConcurrentModificationException` if it changes. Best-effort, not guaranteed. Remove safely via `Iterator.remove()` or `removeIf()`.
- **Fail-safe**: the iterator works on a copy, so it never throws but also never sees concurrent changes.

| Property     | Fail-fast                          | Fail-safe                              |
| ------------ | ---------------------------------- | -------------------------------------- |
| On modify    | Throws immediately                 | Works on snapshot, never throws        |
| Examples     | ArrayList, HashMap, HashSet        | ConcurrentHashMap, CopyOnWriteArrayList|
| Safe removal | `Iterator.remove()` / `removeIf()` | Free to modify during iteration        |

> **Note**: in single-threaded code the distinction rarely matters; `Iterator.remove()` or `removeIf()` covers it. Fail-safe collections earn their cost only when one thread iterates while others mutate.

## Thread-Safe Collections

- **Legacy (avoid)**: `Vector`, `Hashtable`, `Stack`, all method-level synchronized.
- **Synchronized wrappers**: `Collections.synchronizedList/Map(...)`; coarse lock, manual sync still needed when iterating.
- **Concurrent (preferred)**: `ConcurrentHashMap` (CAS plus per-bucket `synchronized` since Java 8, segment locking before it), `CopyOnWriteArrayList`/`CopyOnWriteArraySet` for read-heavy data, `ConcurrentLinkedQueue`, and `BlockingQueue` implementations for producer-consumer.

## Q&A

1. **Why must `equals()` and `hashCode()` be overridden together for hash keys?**
A hash collection picks the bucket by `hashCode()` and finds the entry inside it with `equals()`. If equal objects return different hashes they land in different buckets, producing duplicate set entries or "lost" map values.

2. **How does `Arrays.asList()` differ from a real `ArrayList`?**
It returns a fixed-size view backed by the original array: `add`/`remove` throw `UnsupportedOperationException`, and writes pass through to the array and back. Wrap it in `new ArrayList<>(...)` for an independent, mutable list.

3. **How do the `of()` factories differ from normal collections?**
`List.of()`, `Set.of()`, `Map.of()` (Java 9+) are fully immutable and null-hostile: modification throws `UnsupportedOperationException`, and any `null` throws `NullPointerException`.

4. **How does Java pick between `remove(1)` and `remove(Integer.valueOf(1))` on a `List<Integer>`?**
`List` overloads `remove(int index)` and `remove(Object o)`. A primitive `int` binds to the index overload; to remove by value, pass an object via `Integer.valueOf(1)`.

5. **How do you build an LRU cache from a standard collection?**
Use `LinkedHashMap` in access-order mode and override `removeEldestEntry` to evict past a size cap.

```java
Map<K, V> lru = new LinkedHashMap<>(16, 0.75f, true) {
    protected boolean removeEldestEntry(Map.Entry<K, V> e) { return size() > MAX; }
};
```

6. **How do `TreeSet` and `TreeMap` decide equality?**
By `compareTo`/`compare` only, never `equals`/`hashCode`. A result of `0` means duplicate, so a comparator that looks at one field (e.g. salary) silently merges distinct objects that share it.

7. **Why does `add(E e)` take the generic type but `remove(Object o)` and `contains` take `Object`?**
`add` is constrained to `E` because inserting the wrong type is always a bug. `remove`/`contains` work through `equals`, so searching with any type is a valid query that simply returns `false` on no match; widening to `Object` keeps pre-generics code compiling.

8. **What does the load factor control, and why 0.75?**
It is the fullness threshold (`size > capacity × loadFactor`) that triggers a resize. 0.75 balances space against collision probability; lower wastes memory, higher slows lookups.
