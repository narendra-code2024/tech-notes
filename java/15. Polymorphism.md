Polymorphism (Greek for "many forms") lets one method name take on different behavior depending on the object or arguments it acts on. In Java this resolves in two ways:
- **Compile-time** (Static/Early Binding): resolved at compile time → Method Overloading.
- **Runtime** (Dynamic/Late Binding): resolved at runtime → Method Overriding.

---

## Method Overloading

Same method name, **different parameter list** (number, type, or order). Compiler decides which to call based on arguments.

```java
class Calculator {
    int add(int a, int b) { return a + b; }
    double add(double a, double b) { return a + b; }
    int add(int a, int b, int c) { return a + b + c; }
}

Calculator c = new Calculator();
c.add(1, 2);           // calls add(int, int)
c.add(1.5, 2.5);       // calls add(double, double)
c.add(1, 2, 3);        // calls add(int, int, int)
```

Return type, access modifier, and thrown exceptions may also differ, but they don't count toward overloading, so two methods that differ only by return type won't compile:

```java
int add(int a, int b) { }
double add(int a, int b) { }     // compile error: identical parameters
```

Applies equally to instance methods, static methods, and constructors. One special case: `main` can be overloaded, but the JVM only ever calls the `String[]` version as the entry point.

```java
public static void main(String[] args) { }     // ← JVM entry point
public static void main(int x) { }             // valid overload, never called by JVM
```

### Resolution Order

When multiple methods could match, Java resolves in this order:

```
1. Exact match
2. Widening (int → long)
3. Autoboxing (int → Integer)
4. Varargs (int... → last resort)
```

```java
void process(int x) { }        // 1. exact
void process(long x) { }       // 2. widening
void process(Integer x) { }    // 3. autoboxing
void process(int... x) { }     // 4. varargs

process(5);    // calls process(int), exact match
```

When two candidates match equally well, the call is ambiguous and won't compile:

```java
void process(int x, double y) { }
void process(double x, int y) { }
process(5, 5);    // compile error: both match via widening
```

These conversions can also chain, with one limit: an argument can box *then* widen (`int → Integer → Object`), but never widen *then* box (`int → long → Long`):

```java
void m(Long x) { }
m(5);              // compile error: can't widen int→long then box to Long

void m(Object x) { }
m(5);              // OK: boxes int→Integer, then widens Integer→Object
```

## Method Overriding

Method overriding allows a subtype to redefine a behavior inherited from its supertype. In Java, a subclass provides a specific implementation of a parent class method, which the JVM resolves dynamically at runtime based on the actual object type.

```java
class Animal {
    void sound() { System.out.println("Some sound"); }
}

class Dog extends Animal {
    @Override
    void sound() { System.out.println("Bark"); }
}

class Cat extends Animal {
    @Override
    void sound() { System.out.println("Meow"); }
}

Animal a = new Dog();    // reference: Animal, object: Dog
a.sound();               // "Bark", calls Dog's method, not Animal's (runtime decision)

Animal b = new Cat();
b.sound();               // "Meow"
```

**Rules**:
- Overriding requires matching signatures exactly (same name and parameters)
- The overriding method's return type can be covariant (subtype), its access modifier must be same or wider, and thrown checked exceptions must be same or narrower
- Private, final, static methods, and constructors cannot be overridden

```java
class Parent {
    protected Animal create() throws IOException { return new Animal(); }
}

class Child extends Parent {
    @Override
    public Dog create() throws FileNotFoundException {    // OK: all valid:
        return new Dog();                                  // public > protected (wider access)
    }                                                      // Dog is subtype of Animal (covariant)
}                                                          // FileNotFoundException is subtype of IOException (narrower)
```

> **Note**: The `@Override` annotation is optional but recommended; it documents developer intent and forces the compiler to verify the override, catching method signature typos at compile time.

## Method Hiding

Method hiding occurs when a subclass defines a static method with the same signature as an parent class static method. In Java, this resolves at compile time based on the reference type, bypassing dynamic dispatch.

```java
class Parent {
    static void greet() { System.out.println("Parent"); }
}

class Child extends Parent {
    static void greet() { System.out.println("Child"); }   // hides, not overrides
}

Parent p = new Child();
p.greet();          // "Parent", resolved by reference type (compile time)
                    // If it were overriding → would print "Child"

Child c = new Child();
c.greet();          // "Child", reference type is Child
```

Overriding resolves at runtime (based on object type); hiding resolves at compile time (based on reference type).

## Polymorphic Member Resolution

When a parent reference points to a child instance (`Parent p = new Child()`) and both classes define members with the same name/signature, access resolves differently based on the member type:

| Member Type | Resolution Basis | Binding Time | Behavior on `Parent p = new Child()` |
| :--- | :--- | :--- | :--- |
| **Instance Method** | Object Type (`Child`) | Runtime (Dynamic) | Polymorphic; executes `Child`'s override |
| **Static Method**   | Reference Type (`Parent`) | Compile time (Static) | Non-polymorphic; executes `Parent`'s hidden method |
| **Instance Field**  | Reference Type (`Parent`) | Compile time (Static) | Non-polymorphic; accesses `Parent`'s field |
| **Static Field**    | Reference Type (`Parent`) | Compile time (Static) | Non-polymorphic; accesses `Parent`'s static field |

## Q&A

1. **How does Java resolve an overloaded method call when `null` is passed as an argument?**
Java resolves overloaded methods by selecting the most specific type matching the argument. If the parameters of the overloading candidates are hierarchical (e.g., `Object` vs `String`), the most specific subtype (`String`) is chosen; if they are not hierarchical (e.g., `String` vs `Integer`), the call is ambiguous and triggers a compile-time error.

2. **Is field access polymorphic in Java?**
No. Only instance methods exhibit polymorphic behavior in Java. Field access is resolved at compile time based on the reference type of the variable rather than the actual runtime object type.

```java
class Parent { int x = 10; }
class Child extends Parent { int x = 20; }
Parent p = new Child();
System.out.println(p.x);     // 10 (resolved by reference type)
```

3. **How does Java resolve a call when a method is overloaded in a subclass and overridden from its parent?**
The compiler first binds the call to a specific method signature using the reference type at compile time. At runtime, the JVM executes the overriding implementation of that specific signature on the actual object, meaning overloaded subclass methods with different signatures are ignored.

```java
class Parent {
    void process(Object o) { System.out.println("Parent"); }
}
class Child extends Parent {
    void process(Object o) { System.out.println("Child-Override"); } // override
    void process(String s) { System.out.println("Child-Overload"); } // overload
}
Parent p = new Child();
p.process("hello");          // "Child-Override" (bound to process(Object) signature)
```

## Quick Revision Cheat Sheet

| Concept                | Remember                                                                                          |
| ---------------------- | ------------------------------------------------------------------------------------------------- |
| Overloading resolution | Priority: exact match → widening → autoboxing → varargs                                           |
| Overridden signatures   | Same name and parameters; static, final, private methods, and constructors cannot be overridden   |
| Covariance & access    | Return type can be covariant (subtype); access modifier must be same or wider                     |
| Exception constraints  | Thrown checked exceptions must be same, narrower, or omitted                                      |
| Static & field access  | Non-polymorphic; resolved at compile time based on the reference type                             |
| `null` arguments       | Resolves to the most specific type; non-hierarchical overloading targets cause compile ambiguity |
| Multi-level dispatch   | Compiler binds signature by reference type; JVM resolves override of that signature by object type|
| `@Override`            | Optional; forces compiler verification of the override signature to catch typos                   |
