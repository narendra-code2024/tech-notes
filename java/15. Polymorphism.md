Polymorphism (Greek for "many forms") lets one method name take on different behavior depending on the object or arguments it acts on. In Java this resolves in two ways:

- **Compile-time** (static / early binding): resolved at compile time → method overloading.
- **Runtime** (dynamic / late binding): resolved at runtime → method overriding.

---

## Method Overloading

Same method name, **different parameter list** (number, type, or order). The compiler picks which to call based on the arguments.

```java
class Calculator {
    int add(int a, int b) { return a + b; }
    double add(double a, double b) { return a + b; }
    int add(int a, int b, int c) { return a + b + c; }
}

Calculator c = new Calculator();
c.add(1, 2);           // add(int, int)
c.add(1.5, 2.5);       // add(double, double)
c.add(1, 2, 3);        // add(int, int, int)
```

Return type, access modifier, and thrown exceptions may differ but don't count toward overloading, so two methods differing only by return type won't compile:

```java
int add(int a, int b) { }
double add(int a, int b) { }     // compile error: identical parameters
```

It applies equally to instance methods, static methods, and constructors. One special case: `main` can be overloaded, but the JVM only ever calls the `String[]` version as the entry point.

```java
public static void main(String[] args) { }     // ← JVM entry point
public static void main(int x) { }             // valid overload, never called by the JVM
```

### Resolution Order

When several methods could match, Java resolves in this order:

```
1. Exact match
2. Widening (int → long)
3. Autoboxing (int → Integer)
4. Varargs (int... → last resort)
```

```java
void process(int x) { }        // 1. exact
void process(long x) { }       // 2. widening
void process(Integer x) { }    // 3. autoboxing
void process(int... x) { }     // 4. varargs

process(5);    // calls process(int), exact match
```

When two candidates match equally well, the call is ambiguous and won't compile:

```java
void process(int x, double y) { }
void process(double x, int y) { }
process(5, 5);    // compile error: both match via widening
```

These conversions can chain, with one limit: an argument can box *then* widen (`int → Integer → Object`), but never widen *then* box (`int → long → Long`):

```java
void m(Long x) { }
m(5);              // compile error: can't widen int→long then box to Long

void m(Object x) { }
m(5);              // OK: boxes int→Integer, then widens Integer→Object
```

## Method Overriding

Overriding lets a subtype redefine behavior inherited from its supertype. The subclass provides its own implementation of a parent method, and the JVM resolves it dynamically at runtime based on the actual object type.

```java
class Animal {
    void sound() { System.out.println("Some sound"); }
}

class Dog extends Animal {
    @Override
    void sound() { System.out.println("Bark"); }
}

class Cat extends Animal {
    @Override
    void sound() { System.out.println("Meow"); }
}

Animal a = new Dog();    // reference: Animal, object: Dog
a.sound();               // "Bark", calls Dog's method (runtime decision)

Animal b = new Cat();
b.sound();               // "Meow"
```

Overriding follows these rules:

- The signature must match exactly (same name and parameters).
- The return type may be covariant (a subtype), the access modifier must be the same or wider, and thrown checked exceptions must be the same or narrower.
- Private, final, and static methods, and constructors, cannot be overridden.

```java
class Parent {
    protected Animal create() throws IOException { return new Animal(); }
}

class Child extends Parent {
    @Override
    public Dog create() throws FileNotFoundException {    // all valid:
        return new Dog();                                  // public > protected (wider access)
    }                                                      // Dog subtype of Animal (covariant)
}                                                          // FileNotFoundException subtype of IOException (narrower)
```

> **Note**: `@Override` is optional but recommended; it documents intent and makes the compiler verify the override, catching signature typos at compile time.

## Method Hiding

Method hiding occurs when a subclass defines a static method with the same signature as a parent class static method. It resolves at compile time based on the reference type, bypassing dynamic dispatch.

```java
class Parent {
    static void greet() { System.out.println("Parent"); }
}

class Child extends Parent {
    static void greet() { System.out.println("Child"); }   // hides, not overrides
}

Parent p = new Child();
p.greet();          // "Parent", resolved by reference type (compile time)
                    // if it were overriding → would print "Child"

Child c = new Child();
c.greet();          // "Child", reference type is Child
```

Overriding resolves at runtime (by object type); hiding resolves at compile time (by reference type).

## Polymorphic Member Resolution

When a parent reference points to a child instance (`Parent p = new Child()`) and both classes define members with the same name/signature, access resolves differently by member type:

| Member Type         | Resolution Basis          | Binding Time          | Behavior on `Parent p = new Child()`               |
| :------------------ | :------------------------ | :-------------------- | :------------------------------------------------- |
| **Instance method** | Object type (`Child`)     | Runtime (dynamic)     | Polymorphic; runs `Child`'s override               |
| **Static method**   | Reference type (`Parent`) | Compile time (static) | Non-polymorphic; runs `Parent`'s hidden method     |
| **Instance field**  | Reference type (`Parent`) | Compile time (static) | Non-polymorphic; reads `Parent`'s field            |
| **Static field**    | Reference type (`Parent`) | Compile time (static) | Non-polymorphic; reads `Parent`'s static field     |

## Q&A

1. **How does Java resolve an overloaded call when `null` is passed?**
It picks the most specific matching type. If the candidates are hierarchical (`Object` vs `String`), the most specific subtype (`String`) wins; if they are unrelated (`String` vs `Integer`), the call is ambiguous and is a compile error.

2. **Is field access polymorphic?**
No. Only instance methods are polymorphic. Field access is resolved at compile time from the reference type, not the runtime object type.

```java
class Parent { int x = 10; }
class Child extends Parent { int x = 20; }
Parent p = new Child();
System.out.println(p.x);     // 10 (resolved by reference type)
```

3. **How does Java resolve a call when a method is both overloaded in a subclass and overridden from the parent?**
The compiler first binds the call to a specific signature using the reference type. At runtime the JVM runs the override of *that* signature on the actual object, so overloaded subclass methods with different signatures are not considered.

```java
class Parent {
    void process(Object o) { System.out.println("Parent"); }
}
class Child extends Parent {
    void process(Object o) { System.out.println("Child-Override"); } // override
    void process(String s) { System.out.println("Child-Overload"); } // overload
}
Parent p = new Child();
p.process("hello");          // "Child-Override" (bound to process(Object))
```

## Quick Revision Cheat Sheet

| Concept                | Remember                                                                                          |
| ---------------------- | ------------------------------------------------------------------------------------------------- |
| Overloading resolution | Priority: exact match → widening → autoboxing → varargs                                           |
| Overriding signatures  | Same name and parameters; static, final, private methods, and constructors cannot be overridden   |
| Covariance & access    | Return type may be covariant (subtype); access modifier same or wider                             |
| Exception constraints  | Thrown checked exceptions must be same, narrower, or omitted                                      |
| Static & field access  | Non-polymorphic; resolved at compile time by reference type                                       |
| `null` arguments       | Picks the most specific type; unrelated overload targets cause compile ambiguity                 |
| Multi-level dispatch   | Compiler binds the signature by reference type; JVM runs the override of that signature by object type |
| `@Override`            | Optional; forces compiler verification of the override signature                                  |
