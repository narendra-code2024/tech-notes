## `var`, Local Variable Type Inference (Java 10)

The compiler infers a local variable's static type from its initializer, so you write `var` instead of the full type. The inferred type is fixed at compile time, this is not dynamic typing.

```java
var list   = new ArrayList<String>();              // ArrayList<String>
var map    = new HashMap<String, List<Integer>>(); // verbose generic, the real win
var stream = list.stream().filter(s -> s.length() > 3);

var s = "hi";   // String
s = 123;        // compile error: still statically typed

// Illegal, nothing to infer from:
var x;                 // no initializer
var n = null;          // null has no type
var arr = {1, 2, 3};   // array initializer needs an explicit type
var f = () -> "hi";    // lambda needs a target type
```

- Local variables only: not fields, method parameters, or return types.
- Must be initialized at declaration.
- `var` is a reserved type name, not a keyword, so legacy variables named `var` still compile.

> **Note**: use `var` when the type is long and obvious from the right-hand side (generics, builders); avoid it when the initializer hides the type and readability suffers.

## Switch Expressions (Java 14)

`switch` used as an expression that yields a value. Arrow labels (`->`) carry no fall-through and accept multiple comma-separated constants; `yield` returns a value from a multi-statement block.

```java
// Statement form, fall-through risk if break is forgotten
switch (day) {
    case "MON", "TUE", "WED": type = "Weekday"; break;
    case "SAT", "SUN":        type = "Weekend"; break;
}

// Expression form, no fall-through, exhaustive via default
String type = switch (day) {
    case "MON", "TUE", "WED" -> "Weekday";
    case "SAT", "SUN"        -> "Weekend";
    default                  -> "Unknown";
};

// Multi-statement arm uses yield
String label = switch (day) {
    case "MON" -> { log(day); yield "Weekday"; }
    default    -> "Unknown";
};
```

## Text Blocks (Java 15)

Multi-line string literals delimited by `"""`, removing `\n` and quote-escaping clutter. Useful for JSON, SQL, and HTML.

```java
String json = """
        {
          "name": "Alice",
          "age": 25
        }
        """;
```

> **Note**: incidental leading whitespace is stripped relative to the least-indented line (the closing `"""` sets the margin), and trailing whitespace per line is removed. End a line with `\` to suppress its newline, or `\s` to keep a trailing space.

## Records (Java 16)

Immutable data carriers. From the header alone the compiler generates the canonical constructor, accessors (`x()`, no `get` prefix), and value-based `equals()`, `hashCode()`, and `toString()`.

Beyond killing ~40 lines of DTO ceremony, the real payoff is correctness: generated `equals()`/`hashCode()` always cover every component, eliminating the classic "added a field, forgot to update `equals`" bug that silently breaks `HashMap`/`HashSet`.

```java
record Point(int x, int y) { }   // replaces a ~30-line POJO

var p = new Point(3, 4);
p.x();          // 3
p.toString();   // "Point[x=3, y=4]"
```

- Components are `private final`; the record is implicitly `final` and extends `java.lang.Record`, so it cannot extend another class (it can implement interfaces).
- Allowed: static fields/methods, instance methods, and a compact constructor. Not allowed: instance fields beyond the header.

**Compact constructor** (validation/normalization, runs before fields are assigned):

```java
record Employee(String name, int salary) {
    Employee {
        if (salary < 0) throw new IllegalArgumentException("negative salary");
    }
}
```

**Shallow immutability** (the reference is final, the referent is not):

```java
record User(List<String> roles) { }
var u = new User(new ArrayList<>(List.of("ADMIN")));
u.roles().add("USER");   // mutates the backing list; the record can't prevent it
```

> **Note**: for true immutability, defensively copy mutable components inside the compact constructor (e.g. `roles = List.copyOf(roles);`).

Use for DTOs, API request/response models, value objects, map keys, and config holders. Avoid for JPA entities, which need a no-arg constructor and mutability that records forbid.

## Pattern Matching (Java 16, `instanceof`; Java 21, `switch`)

`instanceof` can bind the cast result to a pattern variable in the same test, removing the redundant cast. Java 21 extends type patterns to `switch`, including a `null` case and guarded `when` clauses.

```java
// instanceof: test and bind in one step, with flow scoping
if (obj instanceof String s && s.length() > 5) {   // s in scope only where the match holds
    System.out.println(s);
}

// switch patterns (Java 21)
String r = switch (obj) {
    case Integer i when i > 0 -> "positive int " + i;  // guarded pattern
    case Integer i            -> "int " + i;
    case String s             -> "string " + s;
    case null                 -> "null";              // switch can match null explicitly
    default                   -> "other";
};
```

## Sealed Classes (Java 17)

A `sealed` type uses `permits` to list exactly which classes may extend or implement it, closing the hierarchy at compile time.

Permitted subtypes must reside in the same module (same package if unnamed), and each must declare its own openness: `final` (closed), `sealed` (further restricted), or `non-sealed` (re-opened). Records are implicitly `final`, so they satisfy `permits` with no extra modifier.

```java
sealed interface Payment permits CreditCard, DebitCard, Upi { }
record CreditCard(String number, String cvv) implements Payment { }
record DebitCard(String number, String pin)  implements Payment { }
record Upi(String upiId)                       implements Payment { }

// Compiler knows the full set, so no default is needed and a missing case is a compile error
String process(Payment p) {
    return switch (p) {
        case CreditCard c -> "CC "  + c.number();
        case DebitCard d  -> "DB "  + d.number();
        case Upi u        -> "UPI " + u.upiId();
    };
}
```

Sealed types plus records plus pattern matching give algebraic data types (ADTs): a closed set of variants, each with its own shape. Prefer them over enums when the variants carry different fields (here `Upi` has `upiId`, `CreditCard` has `number`/`cvv`); enums only fit homogeneous constants.

## Virtual Threads (Java 21)

JVM-scheduled lightweight threads. A virtual thread mounts on a platform "carrier" thread only while running and unmounts on blocking I/O, so millions can coexist at a fraction of a platform thread's ~1 MB stack cost.

```java
Thread.startVirtualThread(() -> doWork());

// The practical form: one virtual thread per task
try (var exec = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++)
        exec.submit(() -> { Thread.sleep(1000); return "done"; });
}   // 100K virtual threads are fine; 100K platform threads would exhaust the machine
```

| Model           | Behavior                                      |
| --------------- | --------------------------------------------- |
| Thread pool     | Limited threads, tasks queue under load       |
| Virtual threads | Thread-per-task, no queueing, scales with I/O |

- Enables a simple thread-per-request model without pools. Do not pool virtual threads, creating them is cheap and pooling defeats the purpose.
- Best for I/O-bound work (HTTP, DB, file), and reduces the need for reactive stacks (WebFlux). For CPU-bound work use platform threads or parallel streams.
- On JDK 21, `synchronized` blocks pin the carrier thread for the duration of a blocking call, capping scalability; prefer `ReentrantLock`. JDK 24 (JEP 491) largely removes this pinning.

## Structured Concurrency (Java 21, Preview)

Treats a group of concurrent subtasks as one unit of work: fork them into a scope, and if any fails the rest are cancelled, avoiding orphaned threads.

```java
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    var user  = scope.fork(() -> fetchUser());
    var order = scope.fork(() -> fetchOrder());
    scope.join();             // wait for all forks
    scope.throwIfFailed();    // propagate the first failure
    use(user.resultNow(), order.resultNow());
}
```

> **Note**: still a preview feature, and the API is evolving across releases (e.g. the `StructuredTaskScope` factory and policy methods changed after JDK 21), so guard production use behind the preview flag.

## Q&A

1. **Why can't a record be used as a JPA entity?**
JPA requires a public/protected no-arg constructor and mutable fields it can populate via reflection after instantiation, plus a non-final class to proxy. Records are final, immutable, and expose only the canonical constructor, so they violate every one of those requirements.

2. **Is a record guaranteed to be immutable?**
No, only shallowly: its component references are `final`, but a referenced mutable object (a `List`, array, or `Date`) can still be mutated through its accessor. True immutability requires defensively copying mutable components in the compact constructor.

3. **Why do `var n = null` and `var f = () -> "hi"` fail to compile?**
`var` infers from the initializer's type, and neither supplies one: a `null` literal has no type, and a lambda is a poly expression that needs a target type supplied by the context. `var` cannot provide that target, so both are rejected.

4. **What does sealing a type give you in a `switch`?**
The compiler knows the complete set of permitted subtypes, so a pattern `switch` over a sealed type is checked for exhaustiveness: you need no `default`, and a missing variant is a compile error rather than a runtime surprise. This is what makes sealed types plus records usable as ADTs.

5. **Why shouldn't you pool virtual threads, and when do they not help?**
Virtual threads are cheap to create and meant to be one-per-task, so pooling them reintroduces the very contention they remove. They help only with blocking I/O-bound work; CPU-bound tasks gain nothing because the bottleneck is cores, not thread count.

6. **How can `synchronized` undermine virtual-thread scalability?**
On JDK 21, when a virtual thread blocks inside a `synchronized` block it pins its carrier platform thread, so under heavy contention the pool of carriers is exhausted and throughput collapses. Using `ReentrantLock` avoids pinning; JDK 24 (JEP 491) removes the `synchronized` pinning itself.
