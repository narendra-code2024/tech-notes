## What & Why

Multithreading lets a program do **multiple things at the same time**: e.g., downloading a file while updating the UI. A thread is the smallest unit of execution within a process.

```
Process (your Java app)
├── Main Thread
├── Thread-1 (downloading file)
├── Thread-2 (processing data)
└── Thread-3 (handling UI)
```

**Key terms**: **Process**: an independent running program with its own memory
- **Thread**: a lightweight sub-process within a process, shares memory with other threads
- **Concurrency**: multiple tasks making progress (may not be simultaneous, time-slicing)
- **Parallelism**: multiple tasks running truly simultaneously (needs multiple CPU cores)

## Creating Threads

### Way 1: Extend Thread class

```java
class MyThread extends Thread {
    public void run() {
        System.out.println("Running: " + Thread.currentThread().getName());
    }
}

MyThread t = new MyThread();
t.start();   // OK: starts new thread, calls run() internally
// t.run();  // invalid: does NOT start a new thread, runs in current thread
```

### Way 2: Implement Runnable (preferred)

```java
class MyRunnable implements Runnable {
    public void run() {
        System.out.println("Running: " + Thread.currentThread().getName());
    }
}

Thread t = new Thread(new MyRunnable());
t.start();

// Lambda (since Runnable is a functional interface)
Thread t = new Thread(() -> System.out.println("Running"));
t.start();
```

### Way 3: Implement Callable (returns a result)

`Runnable` runs but returns nothing. `Callable<V>` runs and returns a value, plus can throw checked exceptions. You need a wrapper to get the result back:

```java
Callable<Integer> task = () -> {
    Thread.sleep(1000);
    return 42;
};

// FutureTask wraps a Callable so a plain Thread can run it
FutureTask<Integer> futureTask = new FutureTask<>(task);
new Thread(futureTask).start();

Integer result = futureTask.get();   // blocks until the Callable finishes
System.out.println(result);          // 42
```

**What's happening**: `FutureTask<V>` is a `Runnable` that wraps your `Callable<V>`. Thread runs it like any other `Runnable`.
- `futureTask.get()` blocks the calling thread until the `Callable` returns, then hands you the value.

> In real code, `Callable` is almost always submitted to an `ExecutorService` (thread pool) instead of a raw Thread, see [ExecutorService (Thread Pools)](<#ExecutorService (Thread Pools)>) for that pattern.

### Runnable vs Callable

||Runnable|Callable|
|---|---|---|
|Method|`run()`|`call()`|
|Returns|void|V (generic return type)|
|Exceptions|Can't throw checked exceptions|Can throw checked exceptions|
|Used with|Thread, ExecutorService|ExecutorService only|

**Why Runnable is preferred over extending Thread**: Java doesn't support multiple inheritance, extending Thread blocks extending other classes
- Separates the task (what to do) from the thread (how to run it)
- Runnable can be used with ExecutorService (thread pools)

> **Why the exception behavior differs**: comes down to the SAM signatures (`run()` has no `throws`, `call() throws Exception`) and Java's override rule. Full design rationale, including *why* it was designed this way, is in [ExecutorService (Thread Pools)](<#ExecutorService (Thread Pools)>) (after `Future.get()` is introduced, the Callable side makes much more sense).

## Thread Lifecycle

```
NEW → RUNNABLE → TERMINATED
           ↓ ↑
   BLOCKED / WAITING / TIMED_WAITING
```

Java does NOT have a separate RUNNING state, `RUNNABLE` includes both ready-to-run and currently running.

|State|When|
|---|---|
|**NEW**|Thread created but `start()` not called yet|
|**RUNNABLE**|`start()` called, ready to run or currently running|
|**BLOCKED**|Waiting to acquire a lock (synchronized block)|
|**WAITING**|Waiting indefinitely (`wait()`, `join()`, `park()`)|
|**TIMED_WAITING**|Waiting with timeout (`sleep()`, `wait(ms)`, `join(ms)`)|
|**TERMINATED**|`run()` completed or exception thrown|

### Quick background, what "lock" and "waiting" mean here

Three terms above need a minimal mental model (depth in [Synchronization (Thread Safety)](<#Synchronization (Thread Safety)>) and [wait(), notify(), notifyAll()](<#wait(), notify(), notifyAll()>)):

- **"Acquire a lock"**: a *lock* is an invisible token that `synchronized` uses to let only one thread into a block at a time. If Thread B tries to enter a `synchronized` block that Thread A is already in, B is **BLOCKED** at the door until A exits and releases the lock.
    ```java
    synchronized (lock) {   // only one thread at a time inside this block
        // ...               // others wanting in are BLOCKED
    }
    ```
- **"Waiting indefinitely"**: a thread voluntarily pauses itself to wait for a *specific signal*, with **no time limit**. It'll sleep forever until something wakes it (another thread calling `notify()`, the thread you're `join()`-ing finishes, etc.). This is **WAITING**.
- **"Waiting with timeout"**: same as above but with a deadline: *"wake me on signal OR after N ms, whichever comes first."* This is **TIMED_WAITING** (e.g., `Thread.sleep(1000)`, `wait(5000)`, `join(3000)`).

| State | Why asleep? | Who wakes it? |
|---|---|---|
| **BLOCKED** | Contending for a lock someone else holds | Lock becomes free |
| **WAITING** | Paused via `wait()`, `join()`, `park()` | Only a matching signal (no timer) |
| **TIMED_WAITING** | Same as WAITING but with a deadline | Signal OR timeout |

## Essential Thread Methods

```java
// Starting
t.start();                      // starts thread, calls run() in new thread

// Sleeping, pauses current thread
Thread.sleep(1000);             // sleep 1 second (throws InterruptedException)

// Joining, wait for another thread to finish
t.join();                       // current thread waits until t finishes
t.join(5000);                   // wait max 5 seconds

// Info
Thread.currentThread();         // returns reference to currently executing thread
t.getName();                    // get thread name
t.setName("Worker-1");          // set thread name
t.getId();                      // unique thread ID
t.isAlive();                    // true if started and not terminated
t.getState();                   // returns Thread.State enum

// Priority (1-10, default 5), hint to scheduler, not guaranteed
t.setPriority(Thread.MAX_PRIORITY);  // 10
t.setPriority(Thread.MIN_PRIORITY);  // 1

// Yield, hints scheduler to pause current thread and give other threads a chance
// Not guaranteed, scheduler may ignore it
Thread.yield();

// Daemon thread, background thread, JVM exits when only daemon threads remain
t.setDaemon(true);              // must set before start()
t.isDaemon();

// Interrupt, request a thread to stop (cooperative, not forced)
t.interrupt();                  // sets interrupt flag
Thread.interrupted();           // checks AND clears interrupt flag (static)
t.isInterrupted();              // checks without clearing
```

### Daemon vs User Threads

||User Thread|Daemon Thread|
|---|---|---|
|JVM waits?|Yes, JVM waits for all user threads to finish|No, JVM exits when only daemons remain|
|Example|Main thread, worker threads|GC thread, background logging|
|Default|User thread|Must set `setDaemon(true)` before `start()`|

## Synchronization (Thread Safety)

When multiple threads access shared data, you get **race conditions**: unpredictable results because threads interleave.

```java
// Race condition example
class Counter {
    int count = 0;
    void increment() { count++; }  // NOT atomic: read → modify → write
}

// Two threads incrementing 1000 times each → result might be less than 2000
```

### synchronized keyword

```java
// Way 1: Synchronized method, locks on 'this' object
class Counter {
    int count = 0;
    synchronized void increment() { count++; }  // only one thread at a time
}

// Way 2: Synchronized block, lock on specific object (finer control)
class Counter {
    int count = 0;
    final Object lock = new Object();

    void increment() {
        synchronized (lock) {       // only this block is locked, not entire method
            count++;
        }
    }
}

// Way 3: Static synchronized, locks on Class object
class Counter {
    static int count = 0;
    static synchronized void increment() { count++; }  // locks on Counter.class
}
```

**synchronized guarantees**: **Mutual exclusion**: only one thread executes the synchronized block at a time
- **Visibility**: changes made inside synchronized block are visible to other threads
- **Happens-before**: actions before unlock happen-before actions after lock

## volatile Keyword

Makes a variable **visible to all threads**: reads/writes go directly to main memory, not thread-local cache.

```java
volatile boolean running = true;

// Thread 1
new Thread(() -> {
    while (running) {    // always reads from main memory
        // do work
    }
}).start();

// Thread 2
running = false;         // writes to main memory immediately, Thread 1 sees it
```

### volatile vs synchronized

||volatile|synchronized|
|---|---|---|
|Guarantees|Visibility only|Visibility + atomicity|
|Use for|Flags, simple reads/writes|Compound operations (check-then-act, read-modify-write)|
|Blocking|No|Yes (threads wait for lock)|
|Performance|Faster|Slower (lock overhead)|

```java
volatile int count = 0;
count++;   // invalid: NOT safe, count++ is read + modify + write (3 steps, not atomic)
// Use synchronized or AtomicInteger for count++
```

`volatile` also guarantees **happens-before** relationship and prevents instruction reordering:

```java
volatile boolean ready = false;
int data;

// Thread 1
data = 42;
ready = true;    // volatile write, everything before this is visible to other threads

// Thread 2
if (ready) {
    // guaranteed to see data = 42 (happens-before)
}
// Without volatile → Thread 2 may see stale data = 0
```

### Deep Dive: What "thread-local cache" Actually Means

The term "thread-local cache" here refers to **CPU hardware caches**: not Java's `ThreadLocal` class.

**The hardware reality**: modern CPUs don't read/write RAM for every operation (too slow). Each CPU core has private caches:

```
┌──────────────────┐   ┌──────────────────┐
│ Core 1           │   │ Core 2           │
│ ┌──────────────┐ │   │ ┌──────────────┐ │
│ │ Registers    │ │   │ │ Registers    │ │
│ │ L1 cache     │ │   │ │ L1 cache     │ │
│ │ L2 cache     │ │   │ │ L2 cache     │ │
│ └──────────────┘ │   │ └──────────────┘ │
└────────┬─────────┘   └─────────┬────────┘
         │                       │
         └───────────┬───────────┘
                     │
              ┌──────┴──────┐
              │  L3 cache   │  (shared)
              └──────┬──────┘
                     │
              ┌──────┴──────┐
              │ Main memory │  (RAM, slow, shared)
              └─────────────┘
```

- Register access: ~1 ns
- L1 cache: ~1 ns
- Main memory (RAM): ~100 ns, **~100× slower**

So when a thread reads variable `x`, the CPU keeps a local copy in its core's cache for speed. Two threads on two cores each maintain **their own copy** of the same variable.

**The visibility problem:**

```java
// Shared field (NOT volatile)
boolean running = true;

// Thread A (runs on Core 1)
while (running) { /* do work */ }   // reads from Core 1's cache

// Thread B (runs on Core 2)
running = false;                    // writes to Core 2's cache → eventually RAM
```

Thread B's write sits in Core 2's cache. Until it's flushed to RAM **AND** Core 1's cached copy is invalidated, Thread A **still sees `true`** and loops forever. This is a **visibility bug**: a correct-looking program that silently misbehaves on multi-core hardware.

**What `volatile` actually does at the hardware level**: **Every read** bypasses the core's cache, fetches directly from main memory
- **Every write** bypasses the core's cache, commits directly to main memory
- Generates CPU memory barriers (fence instructions) that prevent reordering

Trade-off: every access is ~100× slower than a cached access, but correctness is preserved.

**Do NOT confuse with `java.lang.ThreadLocal`:**

||CPU "thread-local cache"|`java.lang.ThreadLocal<T>`|
|---|---|---|
|What|Hardware, per-CPU-core caches|Java library class, per-thread storage|
|Who manages|CPU + JVM (invisible to code)|You, via `new ThreadLocal<>()`|
|What `volatile` does to it|Bypasses it|Nothing, totally unrelated|
|Example|L1/L2 cache holding a stale `int`|`ThreadLocal<SimpleDateFormat>` giving each thread its own formatter|

**Mental model:**

> `volatile` doesn't make operations atomic. It makes reads/writes **visible** across threads by forcing them through main memory, bypassing the fast per-core caches where stale copies can linger. That's the cost of correctness on multi-core hardware.

## wait(), notify(), notifyAll()

Inter-thread communication, one thread waits for a condition, another signals when it's met. Must be called inside a `synchronized` block on the same object.

```java
class SharedQueue {
    Queue<Integer> queue = new LinkedList<>();
    int capacity = 5;

    // Producer
    synchronized void produce(int item) throws InterruptedException {
        while (queue.size() == capacity) {
            wait();          // release lock and wait until consumer takes something
        }
        queue.add(item);
        notifyAll();         // wake up waiting consumers
    }

    // Consumer
    synchronized int consume() throws InterruptedException {
        while (queue.isEmpty()) {
            wait();          // release lock and wait until producer adds something
        }
        int item = queue.poll();
        notifyAll();         // wake up waiting producers
        return item;
    }
}
```

**Key rules**: Must call `wait()/notify()/notifyAll()` inside `synchronized` block, the thread must hold the **monitor (lock) of the object** on which these methods are called, else `IllegalMonitorStateException`
- Always use `while` loop with `wait()`, not `if`, protects against **spurious wakeups**: `notify()` wakes one random waiting thread, `notifyAll()` wakes all
- `wait()` releases the lock, `sleep()` does NOT release the lock

## Locks (java.util.concurrent.locks)

More flexible than `synchronized`, can try to acquire, timeout, and have separate read/write locks.

### ReentrantLock

```java
ReentrantLock lock = new ReentrantLock();

lock.lock();
try {
    // critical section
} finally {
    lock.unlock();    // ALWAYS unlock in finally
}

// Try lock, non-blocking
if (lock.tryLock()) {
    try { /* critical section */ }
    finally { lock.unlock(); }
} else {
    // couldn't acquire lock, do something else
}

// Try with timeout
if (lock.tryLock(5, TimeUnit.SECONDS)) { ... }
```

### ReadWriteLock

Multiple readers can read simultaneously, but only one writer at a time.

```java
ReadWriteLock rwLock = new ReentrantReadWriteLock();
Lock readLock = rwLock.readLock();
Lock writeLock = rwLock.writeLock();

// Multiple threads can read at the same time
readLock.lock();
try { /* read data */ }
finally { readLock.unlock(); }

// Only one thread can write (blocks all readers and writers)
writeLock.lock();
try { /* write data */ }
finally { writeLock.unlock(); }
```

### synchronized vs Lock

||synchronized|Lock (ReentrantLock)|
|---|---|---|
|Release|Automatic (end of block)|Manual (`unlock()` in finally)|
|Try acquire|No|Yes (`tryLock()`)|
|Timeout|No|Yes (`tryLock(time)`)|
|Fairness|No|Yes (optional fair ordering)|
|Read/Write|No|Yes (ReadWriteLock)|
|Interruptible|No|Yes (`lockInterruptibly()`)|

## Atomic Classes (java.util.concurrent.atomic)

Lock-free, thread-safe operations using **CAS (Compare-And-Swap)**.

```java
AtomicInteger count = new AtomicInteger(0);

count.incrementAndGet();     // ++count (returns new value)
count.getAndIncrement();     // count++ (returns old value)
count.addAndGet(5);          // count += 5
count.get();                 // read
count.set(10);               // write
count.compareAndSet(10, 20); // if current==10, set to 20 (returns boolean)

// Others: AtomicLong, AtomicBoolean, AtomicReference<T>
```

**When to use:** Simple counters, flags, single-variable thread safety. For compound operations (check-then-act on multiple variables), use `synchronized` or `Lock`.

### High-contention counters, `LongAdder`, `DoubleAdder`

For counters hammered by many threads, `AtomicLong` becomes a bottleneck, every thread fights over the same CAS cell. `LongAdder` spreads updates across multiple internal cells; threads usually hit different cells and only combine at read time.

```java
LongAdder hits = new LongAdder();

// Many writer threads
hits.increment();              // like count++
hits.add(5);                   // like count += 5

// Reader (occasional)
long total = hits.sum();              // aggregate across all cells
long snap  = hits.sumThenReset();     // read + reset in one go
```

**Trade-off:** write-heavy scenarios are much faster; reads are slower (must aggregate) and only *eventually consistent*. Not a replacement for `AtomicLong` when you need exact current-value semantics like `compareAndSet`.

**`LongAccumulator` / `DoubleAccumulator`**: generalized versions: supply any binary operation, not just sum. Useful for tracking max, min, or custom reductions under contention.

```java
LongAccumulator max = new LongAccumulator(Long::max, Long.MIN_VALUE);
max.accumulate(42);
max.accumulate(7);
max.get();                     // 42
```

**Rule of thumb**: Few writers, frequent exact reads → `AtomicLong`
- Many writers, occasional reads (metrics, stats) → `LongAdder`
- Non-sum reductions (max/min/custom) under contention → `LongAccumulator`

> For the full numeric class family (wrappers, `BigInteger`/`BigDecimal`, `Atomic*`, adders/accumulators) see [Numbers, Summary of Java numeric classes](<10. Numbers, Math & Random.md#`java.lang.Number`, the numeric root>).

## Synchronizers (java.util.concurrent)

Coordination primitives for common thread-coordination patterns. Each solves a specific problem `synchronized` alone can't.

### CountDownLatch, wait for N tasks to finish, one-shot

```java
CountDownLatch latch = new CountDownLatch(3);

for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        doWork();
        latch.countDown();             // signal done
    }).start();
}

latch.await();                         // blocks until count reaches 0
System.out.println("All 3 workers finished");
```

> **Note:** CountDownLatch is one-shot and cannot be reset. Use `CyclicBarrier` if you need a reusable synchronization point.

### CyclicBarrier, N threads sync at a common point, reusable

```java
CyclicBarrier barrier = new CyclicBarrier(3, () -> System.out.println("All arrived"));

for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        doPhase1();
        barrier.await();              // wait for other 2 to reach here
        doPhase2();                   // all 3 proceed together
    }).start();
}
```

**Real use**: multi-phase parallel computation (e.g., map-reduce phases, simulations). Barrier resets automatically after tripping, reusable.

### Semaphore, limit concurrent access to a resource

```java
Semaphore permits = new Semaphore(5);   // max 5 concurrent

for (int i = 0; i < 100; i++) {
    new Thread(() -> {
        try {
            permits.acquire();          // blocks if 5 already holding
            callExternalAPI();          // only 5 threads here at once
        } finally {
            permits.release();
        }
    }).start();
}
```

**Real use**: rate-limiting external API calls, connection pool limits, throttling expensive operations.

### Quick compare

||CountDownLatch|CyclicBarrier|Semaphore|
|---|---|---|---|
|Purpose|Wait for N to finish|Sync N threads at a point|Limit concurrent access|
|Reusable|No (one-shot)|Yes|Yes|
|Who waits|Caller of `await()`|All N threads|Threads exceeding N permits|

## ExecutorService (Thread Pools)

Don't create threads manually, use thread pools for better resource management.

**Why not `new Thread()` every time?**: Thread creation is expensive (OS resource allocation)
- No control over how many threads run simultaneously
- No task queuing, excess threads crash the app
- Thread pool reuses threads = much faster

**Real-world example:** "Used a fixed thread pool for sending notification emails, instead of creating a new thread per email, 5 threads handled thousands of emails from a queue."

```java
// Fixed pool, fixed number of threads
ExecutorService executor = Executors.newFixedThreadPool(4);

// Single thread, one thread processes tasks sequentially
ExecutorService executor = Executors.newSingleThreadExecutor();

// Cached pool, creates threads as needed, reuses idle ones
ExecutorService executor = Executors.newCachedThreadPool();

// Scheduled, for delayed/periodic tasks
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);
scheduler.schedule(() -> System.out.println("delayed"), 5, TimeUnit.SECONDS);
scheduler.scheduleAtFixedRate(() -> System.out.println("periodic"), 0, 1, TimeUnit.SECONDS);
```

### Submitting tasks

```java
// Runnable, no return value
executor.execute(() -> System.out.println("task"));

// Runnable via submit, returns Future<?> (can check completion)
Future<?> future = executor.submit(() -> System.out.println("task"));

// Callable, returns Future<T> with result
Future<Integer> future = executor.submit(() -> 42);
Integer result = future.get();           // blocks until result ready
Integer result = future.get(5, TimeUnit.SECONDS);  // blocks with timeout

// Future methods
future.isDone();        // check if completed
future.cancel(true);    // cancel task (true = interrupt if running)
future.isCancelled();   // check if cancelled
```

### Why Callable can throw checked exceptions but Runnable can't, the design rationale

Revisiting the difference from [Creating Threads](<#Creating Threads>), now that `Future.get()` is in view, the design makes sense:

```java
public interface Runnable {  void run();  }                 // no throws
public interface Callable<V> {  V call() throws Exception;  }
```

Java's override rule forbids implementations from declaring broader checked exceptions than the interface. So lambdas/classes implementing these inherit the SAM's exception contract.

- **Runnable (Java 1.0)**: `run()` executes on a separate thread, the code that called `.start()` has already moved on. There's **no caller to propagate an exception to**. An uncaught exception just kills the thread (goes to `UncaughtExceptionHandler`). Forcing `throws IOException` would be misleading, no one would ever catch it. Interface was kept minimal.
- **Callable (Java 5)**: designed to be submitted to `ExecutorService`, which returns a `Future`. When the submitter calls `future.get()`, **any exception thrown inside `call()` is wrapped in `ExecutionException` and delivered back to the submitter**. Now there's a real path to propagate failures, so allowing checked exceptions became useful.

```java
Future<List<String>> future = executor.submit(() -> Files.readAllLines(path));  // OK: IOException allowed

try {
    List<String> lines = future.get();
} catch (ExecutionException e) {
    Throwable actual = e.getCause();  // the original IOException from call()
}
```

Mental model: **Runnable was designed for fire-and-forget execution with no result channel. Callable was designed with a result channel (`Future`), so it could also carry exceptions back.**: -

### Shutting down

```java
executor.shutdown();              // no new tasks, finish existing ones
executor.shutdownNow();           // attempt to cancel all tasks
executor.awaitTermination(10, TimeUnit.SECONDS);  // wait for completion

// Best practice pattern
executor.shutdown();
try {
    if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
        executor.shutdownNow();
    }
} catch (InterruptedException e) {
    executor.shutdownNow();
}
```

### ThreadPoolExecutor (Production-grade, more control)

Default `Executors` factory methods use unbounded queues which can cause `OutOfMemoryError` under heavy load. For production, use `ThreadPoolExecutor` directly:

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    5,                      // corePoolSize, minimum threads
    10,                     // maximumPoolSize, max threads under load
    60, TimeUnit.SECONDS,   // keepAliveTime, idle thread timeout
    new ArrayBlockingQueue<>(100),  // bounded work queue
    new ThreadPoolExecutor.CallerRunsPolicy()  // rejection policy
);
```

**Rejection policies** (when queue is full AND max threads are busy):

|Policy|Behavior|
|---|---|
|**AbortPolicy** (default)|Throws `RejectedExecutionException`|
|**CallerRunsPolicy**|Calling thread runs the task itself (slows down submission)|
|**DiscardPolicy**|Silently drops the task|
|**DiscardOldestPolicy**|Drops oldest queued task, submits new one|

## CompletableFuture (Java 8+)

Asynchronous programming without blocking. Chainable, composable, non-blocking.

```java
// Run async task (no return)
CompletableFuture<Void> cf = CompletableFuture.runAsync(() -> {
    System.out.println("Running async");
});

// Supply async task (with return)
CompletableFuture<String> cf = CompletableFuture.supplyAsync(() -> {
    return "Hello";
});
```

### Chaining

```java
CompletableFuture.supplyAsync(() -> "Hello")
    .thenApply(s -> s + " World")          // transform result (Function)
    .thenApply(String::toUpperCase)
    .thenAccept(System.out::println)       // consume result (Consumer)
    .thenRun(() -> System.out.println("Done"));  // run after (Runnable)

// Result: prints "HELLO WORLD" then "Done"
```

### Combining

```java
// Combine two independent futures
CompletableFuture<String> name = CompletableFuture.supplyAsync(() -> "Alice");
CompletableFuture<Integer> age = CompletableFuture.supplyAsync(() -> 25);

CompletableFuture<String> combined = name.thenCombine(age,
    (n, a) -> n + " is " + a);   // "Alice is 25"

// Wait for all to complete
CompletableFuture.allOf(future1, future2, future3).join();

// Wait for any one to complete
CompletableFuture.anyOf(future1, future2).thenAccept(System.out::println);
```

### Exception handling

```java
CompletableFuture.supplyAsync(() -> {
    if (true) throw new RuntimeException("failed");
    return "result";
})
.exceptionally(e -> "fallback: " + e.getMessage())   // handle exception
.thenAccept(System.out::println);                     // "fallback: failed"

// handle, processes both result and exception
.handle((result, ex) -> {
    if (ex != null) return "error";
    return result;
});
```

### `join()` vs `get()`

```java
future.get();    // blocks, throws checked exceptions (ExecutionException, InterruptedException)
future.join();   // blocks, throws unchecked CompletionException
// Use join() in streams/lambdas where checked exceptions are inconvenient
```

### Async variants

```java
// Default, runs on ForkJoinPool.commonPool()
// ForkJoinPool.commonPool() is also used by parallel streams
// warning: Avoid heavy blocking operations (DB calls, I/O) in commonPool, can starve other tasks
.thenApply(s -> s.toUpperCase())

// Async, runs on a different thread from the pool
.thenApplyAsync(s -> s.toUpperCase())

// Async with custom executor
.thenApplyAsync(s -> s.toUpperCase(), myExecutor)
```

## Virtual Threads (Java 21+)

**Project Loom**: lightweight threads managed by the JVM, not the OS. Millions can exist simultaneously. Designed for I/O-heavy workloads (HTTP servers, DB calls) where platform threads would be wasted blocking.

### Why virtual threads were added (the pre-Java 21 pain)

- **Platform threads are expensive**: each one uses ~2 MB stack and a real OS thread. A 32 GB server can realistically support only ~10,000-20,000 threads. At one thread per request, that's your concurrency ceiling.
- **I/O-heavy servers waste threads blocking**: a request waiting on a DB call or external API pins an entire OS thread for the round-trip (typically 99% of the time just blocked). Expensive thread → idle.
- **Thread pools were the mandatory workaround**: since threads are scarce, you pool them and recycle across tasks. But pools introduce their own problems: context switches, queue backpressure, deadlocks when pools are undersized, tuning overhead.
- **The alternative (async/reactive) broke readable code**: `CompletableFuture`, Reactor, RxJava unlocked scale but turned linear business logic into callback chains or flatMap soup. "Simple" blocking code became fundamentally different from "scalable" non-blocking code. Two codebases, two mental models.

### How virtual threads solve it

- **Decouple threads from OS threads**: a virtual thread is a cheap JVM-level construct (~few KB). Millions can exist.
- **Auto-unmount on blocking I/O**: when a virtual thread hits `socket.read()`, JDBC call, or any blocking operation, the JVM **unmounts** it from its carrier (platform) thread so the carrier can run another virtual thread. The virtual thread resumes when I/O completes.
- **Write blocking code, get async scale**: one virtual thread per request, `connection.query(...)`, blocking calls throughout, yet the server scales to hundreds of thousands of concurrent requests. No `CompletableFuture` chains needed for throughput.
- **Kills the pooling requirement**: virtual threads are cheap enough that you don't pool them. `Executors.newVirtualThreadPerTaskExecutor()` creates one per task. Pools exist for platform threads because they're expensive; that reason is gone.
- **Backwards compatible**: a virtual thread IS a `Thread` (same API). Existing code works unchanged when the server thread is virtual (e.g., Tomcat/Jetty/Spring Boot toggle).

### Platform thread vs Virtual thread

||Platform Thread|Virtual Thread|
|---|---|---|
|Backed by|OS thread (~2 MB stack)|JVM (~KB, pooled on carrier threads)|
|Creation cost|High, pool them|Near-zero, create per task|
|Max count|Thousands|Millions|
|Blocks on I/O|Blocks the OS thread|Unmounts from carrier, carrier runs others|
|Best for|CPU-bound work|I/O-bound work (requests, DB, RPC)|

### Creating virtual threads

```java
// Way 1: one-off
Thread.ofVirtual().start(() -> handleRequest());

// Way 2: via ExecutorService (recommended)
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10_000; i++) {
        executor.submit(() -> callSlowAPI());   // fine, 10k virtual threads, cheap
    }
}   // auto-closes
```

**Key mental shift**: don't pool virtual threads. Create one per task and let the JVM manage. Pools exist for platform threads because creation is expensive, that reason disappears for virtual threads.

### When NOT to use

- **CPU-bound tasks**: virtual threads don't add throughput; use platform threads + `ForkJoinPool`.
- **Code using `synchronized` on long-running operations**: pins the carrier thread (can't unmount). Use `ReentrantLock` instead if you'll hold it across I/O.
- **`ThreadLocal` with heavy state**: one per virtual thread = huge memory blow-up. Reconsider usage.

### Real-world impact

Spring Boot 3.2+, Tomcat, Jetty all support virtual threads. Setting `spring.threads.virtual.enabled=true` can 10-100× throughput on I/O-bound endpoints with zero code change. FAANG-relevant: Netflix, Amazon have publicly migrated services.

## Concurrent Collections

Thread-safe collections without manually synchronizing.

```java
// ConcurrentHashMap, fine-grained locking + CAS (no segment-based locking in Java 8+)
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
map.put("a", 1);
map.computeIfAbsent("b", k -> 2);   // atomic compute

// CopyOnWriteArrayList, creates copy on every write, great for read-heavy
CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();

// CopyOnWriteArraySet
CopyOnWriteArraySet<String> set = new CopyOnWriteArraySet<>();

// ConcurrentLinkedQueue, non-blocking queue
ConcurrentLinkedQueue<String> queue = new ConcurrentLinkedQueue<>();

// BlockingQueue, producer-consumer (blocks on empty/full)
BlockingQueue<String> bq = new ArrayBlockingQueue<>(10);    // bounded
BlockingQueue<String> bq = new LinkedBlockingQueue<>();      // unbounded
bq.put("item");     // blocks if full
bq.take();          // blocks if empty
// BlockingQueue removes the need for manual wait()/notify(), cleaner producer-consumer
```

## Common Concurrency Problems

### Deadlock, two threads waiting for each other's locks forever

```java
class Account {
    private final Object lock = new Object();
    int balance;
    Account(int b) { this.balance = b; }
}

Account a = new Account(1000);
Account b = new Account(1000);

// Thread 1, locks a, then tries to lock b
new Thread(() -> {
    synchronized (a.lock) {
        sleep(50);
        synchronized (b.lock) {               // deadlock: waits forever
            // transfer a → b
        }
    }
}).start();

// Thread 2, locks b, then tries to lock a
new Thread(() -> {
    synchronized (b.lock) {
        sleep(50);
        synchronized (a.lock) {               // deadlock: waits forever
            // transfer b → a
        }
    }
}).start();
```

**Fix, lock ordering** (always acquire in the same global order):

```java
Object first  = System.identityHashCode(a) < System.identityHashCode(b) ? a.lock : b.lock;
Object second = first == a.lock ? b.lock : a.lock;

synchronized (first) {
    synchronized (second) {
        // safe, all threads acquire in the same order, no cycle possible
    }
}
```

**Other prevention**: `tryLock()` with timeout, avoid nested locks when possible, keep lock scope tight. **Detection**: `jstack`, VisualVM, thread dumps, look for "BLOCKED" threads waiting on locks held by each other.

### Livelock, threads keep responding to each other, making no progress

Both threads keep changing state in response to each other but never finish, like two people stepping left and right to let each other pass in a corridor.

### Starvation, a thread never gets CPU time

Low-priority thread never gets to run because high-priority threads keep taking the CPU. Fix: use fair locks (`new ReentrantLock(true)`).

### Race Condition, result depends on thread execution order

```java
// check-then-act, not atomic
if (map.containsKey(key)) {      // Thread 1 checks
    map.get(key);                 // Thread 2 removes key between check and get!
}

// Fix: use atomic operations
map.computeIfAbsent(key, k -> createValue());
```

## Q&A

1. **What is the difference between calling `start()` and `run()` on a Thread object?**
   Calling `start()` creates a new operating system thread and then invokes the `run()` method in that new thread's context. Calling `run()` directly simply executes the method in the **current** thread as a normal method call, without any parallel execution.

2. **How do `Thread.sleep()` and `Object.wait()` differ in their handling of locks?**
   This is a critical distinction: `sleep()` **does not** release any locks that the thread currently holds. In contrast, `wait()` **releases** the lock on the object it was called on, allowing other threads to enter the synchronized block. `wait()` must be called from within a synchronized context, whereas `sleep()` can be called anywhere.

3. **What happens if you attempt to call `start()` twice on the same Thread instance?**
   It will throw an `IllegalThreadStateException`. A thread lifecycle is one-shot; once a thread has started or finished execution, it cannot be restarted.

4. **Why is `HashMap` not suitable for multi-threaded environments?**
   `HashMap` is not synchronized. If two threads perform a `put()` operation simultaneously, it can lead to data corruption or, in older Java versions, an infinite loop during internal resizing. For thread-safe map operations, you should use `ConcurrentHashMap`.

5. **What does it mean that the `synchronized` keyword is "reentrant"?**
   Reentrancy means that if a thread already holds a lock on an object, it can enter other synchronized blocks or methods protected by the same lock without getting blocked. This prevents a thread from deadlocking itself when calling synchronized methods recursively.

6. **Why should you avoid using String literals or boxed types as synchronization locks?**
   String literals and boxed types (like `Integer`) are often cached and shared across the entire application (via the String Pool or Integer Cache). Synchronizing on them can cause unexpected blocking or deadlocks because unrelated parts of the system might accidentally be using the same object as a lock.

   ```java
   synchronized ("lock") { }           // warning: Dangerous, other code may use same literal
   synchronized (Integer.valueOf(1)) { }  // warning: Dangerous, shared via Integer cache
   ```

7. **How do `ThreadLocal` variables work, and what is a common cleanup requirement?**
   `ThreadLocal` provides each thread with its own independent copy of a variable. This is useful for making thread-unsafe objects (like `SimpleDateFormat`) safe to use across threads without synchronization. However, you must always call `remove()` when the thread's work is finished, especially in thread pools, to prevent memory leaks from stale references.

8. **When should you choose `volatile` vs `Atomic` types vs `synchronized`?**
   *   Use **`volatile`** for simple status flags where you only care about visibility (one thread writes, others read).
   *   Use **`Atomic`** types (like `AtomicInteger`) for simple "read-modify-write" operations like counters.
   *   Use **`synchronized`** or **`Lock`** for complex, multi-step operations that require exclusive access to multiple variables.

9. **Is a thread immediately stopped when you call `interrupt()`?**
   No. Interrupting a thread in Java is **cooperative**. Calling `t.interrupt()` simply sets an internal "interrupted" flag. The target thread must periodically check this flag (via `Thread.interrupted()`) and choose to shut down gracefully. If a thread is blocked in a method like `sleep()`, that method will throw an `InterruptedException`, at which point you should typically restore the interrupt status before exiting.

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Thread creation|Runnable (preferred), Callable (returns result), extend Thread (avoid)|
|start() vs run()|start() = new thread, run() = same thread|
|synchronized|Mutual exclusion + visibility. Method or block level|
|volatile|Visibility only. Use for flags, not compound operations|
|wait/notify|Inter-thread communication. Must be in synchronized. Use while, not if|
|sleep vs wait|sleep = holds lock, wait = releases lock|
|Lock|ReentrantLock = tryLock + timeout. ReadWriteLock = multiple readers|
|Atomic|AtomicInteger, AtomicBoolean. Lock-free CAS. For simple counters/flags|
|ExecutorService|Thread pool. Fixed, Cached, Scheduled. Always shutdown()|
|CompletableFuture|Async chaining. supplyAsync → thenApply → thenAccept. Non-blocking|
|Concurrent collections|ConcurrentHashMap, CopyOnWriteArrayList, BlockingQueue|
|Deadlock|Two threads waiting for each other's locks. Fix: same lock order|
|Race condition|Fix: synchronized, atomic, or concurrent collections|
