Multithreading lets one process run several threads at once, e.g. downloading a file while keeping the UI responsive. A thread is the smallest unit of execution within a process, and threads of a process share its memory.

```
Process (your Java app)
├── Main Thread
├── Thread-1 (downloading file)
├── Thread-2 (processing data)
└── Thread-3 (handling UI)
```

- **Process**: an independent running program with its own memory.
- **Thread**: a lightweight unit inside a process, sharing memory with sibling threads.
- **Concurrency**: multiple tasks making progress by interleaving (not necessarily at the same instant).
- **Parallelism**: multiple tasks running at the same instant on multiple cores.

## Creating Threads

### Extend Thread

```java
class MyThread extends Thread {
    public void run() { System.out.println(Thread.currentThread().getName()); }
}
new MyThread().start();   // starts a new thread, which calls run()
// new MyThread().run();  // no new thread: runs in the caller
```

### Implement Runnable (preferred)

```java
Runnable task = () -> System.out.println("Running");   // functional interface
new Thread(task).start();
```

### Implement Callable (returns a result)

`Callable<V>` returns a value and may throw checked exceptions, unlike `Runnable`. A `FutureTask` adapts it so a raw `Thread` can run it, though in practice it is submitted to an `ExecutorService`.

```java
Callable<Integer> task = () -> { Thread.sleep(1000); return 42; };

FutureTask<Integer> ft = new FutureTask<>(task);   // a Runnable wrapping the Callable
new Thread(ft).start();
Integer result = ft.get();   // blocks until the Callable returns, then yields 42
```

### Runnable vs Callable

| Aspect     | Runnable               | Callable                   |
| ---------- | ---------------------- | -------------------------- |
| Method     | `run()`                | `call()`                   |
| Returns    | void                   | `V`                        |
| Checked ex | Cannot throw           | Can throw                  |
| Used with  | Thread, ExecutorService| ExecutorService or FutureTask |

Prefer `Runnable` (or `Callable`) over extending `Thread`: it separates the task from the worker, leaves the single inheritance slot free, and works with thread pools.

## Thread Lifecycle

```
NEW → RUNNABLE → TERMINATED
           ↓ ↑
   BLOCKED / WAITING / TIMED_WAITING
```

Java has no separate RUNNING state; `RUNNABLE` covers both ready-to-run and running.

| State          | Meaning                                  | Wakes on            |
| -------------- | ---------------------------------------- | ------------------- |
| NEW            | Created, `start()` not yet called        | n/a                 |
| RUNNABLE       | Ready or running after `start()`         | n/a                 |
| BLOCKED        | Contending for a monitor lock held by another | lock becomes free |
| WAITING        | Paused via `wait()`, `join()`, `park()`  | a matching signal   |
| TIMED_WAITING  | Same, with a deadline (`sleep`, `wait(ms)`, `join(ms)`) | signal or timeout |
| TERMINATED     | `run()` finished or threw                | n/a                 |

## Essential Thread Methods

```java
t.start();                   // starts the thread; run() executes on it
Thread.sleep(1000);          // pause current thread 1s (throws InterruptedException)
t.join();                    // caller waits until t finishes; join(ms) caps the wait
t.setName("Worker-1");   t.getId();   t.isAlive();   t.getState();

t.setPriority(Thread.MAX_PRIORITY);  // 1..10, a scheduler hint only, not guaranteed
Thread.yield();              // hint to let other threads run; may be ignored

t.setDaemon(true);           // background thread; must be set before start()

t.interrupt();               // cooperative stop request: sets the interrupt flag
Thread.interrupted();        // static: checks AND clears the flag
t.isInterrupted();           // checks without clearing
```

### Daemon vs User Threads

| Dimension  | User thread                 | Daemon thread                       |
| ---------- | --------------------------- | ----------------------------------- |
| JVM waits? | Yes, until all user threads end | No, JVM exits when only daemons remain |
| Example    | Main, worker threads        | GC, background logging              |
| Default    | User                        | Must call `setDaemon(true)` first   |

## Synchronization

Without coordination, threads interleaving on shared data cause race conditions.

```java
class Counter {
    int count = 0;
    void increment() { count++; }   // read, modify, write: not atomic
}
// two threads incrementing 1000 times each may end below 2000
```

### synchronized

```java
class Counter {
    int count = 0;
    final Object lock = new Object();

    synchronized void incA() { count++; }              // locks on 'this'
    void incB() { synchronized (lock) { count++; } }   // locks on a private monitor (finer scope)
    static synchronized void incC() { /* ... */ }      // locks on Counter.class
}
```

`synchronized` provides mutual exclusion (one thread in the block at a time), visibility (changes inside are seen by the next thread to enter), and a happens-before edge (everything before the unlock is visible after the matching lock).

## volatile

A `volatile` field is visible to all threads: a write is published immediately and every read re-reads it, rather than serving a per-core cached copy.

```java
volatile boolean running = true;

new Thread(() -> { while (running) { /* work */ } }).start();
running = false;   // the worker sees this and stops; without volatile it could loop forever
```

`volatile` also establishes happens-before and prevents reordering, so a flag can safely publish other writes:

```java
int data; volatile boolean ready = false;

data = 42; ready = true;        // writer: data is published before ready flips
if (ready) { use(data); }       // reader: guaranteed to see data == 42
```

It does not make compound actions atomic:

```java
volatile int count = 0;
count++;   // still a race: read, modify, write. Use AtomicInteger or synchronized
```

| Aspect     | volatile                          | synchronized                                  |
| ---------- | --------------------------------- | --------------------------------------------- |
| Guarantees | Visibility and ordering           | Visibility, ordering, and atomicity           |
| Use for    | Flags, single reads/writes        | Compound ops (check-then-act, read-modify-write) |
| Blocking   | No                                | Yes (threads wait for the lock)               |

### Visibility and the per-core cache

Each CPU core caches shared variables in its own L1/L2 cache because main memory is roughly 100x slower. A plain write can sit in the writer's cache while another core keeps reading its own stale copy, which is why a non-volatile `while (flag)` loop can run forever after another thread clears the flag. `volatile` inserts memory barriers that force the write to be published and each read to re-read, so cache coherence delivers the latest value.

> **Note**: this per-core cache is CPU hardware and is unrelated to `java.lang.ThreadLocal<T>`, a library that gives each thread its own object (e.g. a `SimpleDateFormat`). `volatile` affects the former and has nothing to do with the latter.

## wait(), notify(), notifyAll()

Inter-thread signalling: one thread waits for a condition, another signals it. All three must be called while holding the monitor of the object they target, else `IllegalMonitorStateException`.

```java
class BoundedQueue {
    Queue<Integer> q = new LinkedList<>();
    int capacity = 5;

    synchronized void produce(int item) throws InterruptedException {
        while (q.size() == capacity) wait();   // release lock, wait for room
        q.add(item);
        notifyAll();                           // wake waiting consumers
    }

    synchronized int consume() throws InterruptedException {
        while (q.isEmpty()) wait();            // release lock, wait for an item
        int item = q.poll();
        notifyAll();
        return item;
    }
}
```

- Guard the condition with `while`, not `if`: a thread may wake spuriously or lose the condition to another waiter before it runs.
- `notify()` wakes one arbitrary waiter; `notifyAll()` wakes all.
- `wait()` releases the lock; `sleep()` keeps it.

## Locks (java.util.concurrent.locks)

More flexible than `synchronized`: try-acquire, timeouts, interruptible acquisition, fairness, and separate read/write locks.

### ReentrantLock

```java
ReentrantLock lock = new ReentrantLock();
lock.lock();
try { /* critical section */ } finally { lock.unlock(); }   // always unlock in finally

if (lock.tryLock(5, TimeUnit.SECONDS)) {   // bounded wait instead of blocking forever
    try { /* ... */ } finally { lock.unlock(); }
}
```

### ReadWriteLock

Many concurrent readers, but a writer is exclusive.

```java
ReadWriteLock rw = new ReentrantReadWriteLock();
rw.readLock().lock();   try { /* read */ }  finally { rw.readLock().unlock(); }
rw.writeLock().lock();  try { /* write */ } finally { rw.writeLock().unlock(); }
```

### synchronized vs Lock

| Capability    | synchronized        | ReentrantLock                 |
| ------------- | ------------------- | ----------------------------- |
| Release       | Automatic at block end | Manual `unlock()` in finally |
| Try / timeout | No                  | `tryLock()`, `tryLock(time)`  |
| Fairness      | No                  | Optional (`new ReentrantLock(true)`) |
| Interruptible | No                  | `lockInterruptibly()`         |
| Read/write    | No                  | via `ReadWriteLock`           |

## Atomic Classes (java.util.concurrent.atomic)

Lock-free, thread-safe updates on a single variable via CAS (compare-and-swap).

```java
AtomicInteger n = new AtomicInteger(0);
n.incrementAndGet();      // ++n, returns the new value
n.getAndIncrement();      // n++, returns the old value
n.addAndGet(5);
n.compareAndSet(10, 20);  // sets to 20 only if currently 10; returns boolean
// also AtomicLong, AtomicBoolean, AtomicReference<T>
```

Use for single-variable counters and flags. For check-then-act across several variables, use `synchronized` or a `Lock`.

### High-contention counters: LongAdder

Under many writers, `AtomicLong` is a bottleneck because every thread contends on one CAS cell. `LongAdder` spreads updates across internal cells and combines them only on read.

```java
LongAdder hits = new LongAdder();
hits.increment();   hits.add(5);     // writers rarely collide
long total = hits.sum();             // aggregates the cells; eventually consistent
```

`LongAccumulator` generalizes this to any binary operation (max, min, custom):

```java
LongAccumulator max = new LongAccumulator(Long::max, Long.MIN_VALUE);
max.accumulate(42);   max.accumulate(7);   max.get();   // 42
```

- Few writers, frequent exact reads, or need `compareAndSet`: `AtomicLong`.
- Many writers, occasional reads (metrics): `LongAdder`.
- Non-sum reductions under contention: `LongAccumulator`.

> **Reference**: the wider numeric class family is covered in [Numbers, Math & Random](<10. Numbers, Math & Random.md>).

## Synchronizers (java.util.concurrent)

Coordination primitives for patterns `synchronized` alone cannot express.

### CountDownLatch (wait for N tasks, one-shot)

```java
CountDownLatch latch = new CountDownLatch(3);
for (int i = 0; i < 3; i++) new Thread(() -> { doWork(); latch.countDown(); }).start();
latch.await();   // blocks until the count reaches 0
```

> **Note**: a `CountDownLatch` cannot be reset; use `CyclicBarrier` for a reusable rendezvous.

### CyclicBarrier (sync N threads at a point, reusable)

```java
CyclicBarrier barrier = new CyclicBarrier(3, () -> System.out.println("all arrived"));
for (int i = 0; i < 3; i++) new Thread(() -> {
    doPhase1();
    barrier.await();   // wait for the other two
    doPhase2();        // all proceed together; the barrier resets for the next round
}).start();
```

### Semaphore (cap concurrent access)

```java
Semaphore permits = new Semaphore(5);   // at most 5 concurrent
permits.acquire();                      // blocks if 5 are already held
try { callExternalApi(); } finally { permits.release(); }
```

| Dimension | CountDownLatch     | CyclicBarrier        | Semaphore                |
| --------- | ------------------ | -------------------- | ------------------------ |
| Purpose   | Wait for N to finish | Sync N at a point  | Limit concurrent access  |
| Reusable  | No                 | Yes                  | Yes                      |
| Who waits | Caller of `await()`| All N threads        | Threads beyond N permits |

## ExecutorService (Thread Pools)

Pool and reuse threads instead of creating one per task: thread creation is expensive, and unbounded creation has no backpressure and can crash the app.

```java
ExecutorService fixed     = Executors.newFixedThreadPool(4);
ExecutorService single    = Executors.newSingleThreadExecutor();
ExecutorService cached     = Executors.newCachedThreadPool();    // grows as needed, reuses idle
ScheduledExecutorService sc = Executors.newScheduledThreadPool(2);
sc.scheduleAtFixedRate(() -> tick(), 0, 1, TimeUnit.SECONDS);
```

### Submitting tasks

```java
executor.execute(() -> work());                 // Runnable, no handle
Future<?> f = executor.submit(() -> work());    // Runnable, completion handle
Future<Integer> r = executor.submit(() -> 42);  // Callable, result handle
Integer v = r.get(5, TimeUnit.SECONDS);         // blocks (optionally with timeout)
r.cancel(true);   r.isDone();   r.isCancelled();
```

A checked exception thrown inside `call()` is captured and rethrown as `ExecutionException` from `get()`; unwrap the original with `getCause()`. `Runnable.run()` declares no `throws`, so it cannot propagate checked exceptions; an uncaught one reaches the thread's `UncaughtExceptionHandler`.

```java
Future<List<String>> f = executor.submit(() -> Files.readAllLines(path));  // IOException allowed
try {
    f.get();
} catch (ExecutionException e) {
    Throwable cause = e.getCause();   // the original IOException from call()
}
```

### Shutting down

```java
executor.shutdown();                        // stop accepting tasks, finish queued ones
if (!executor.awaitTermination(60, TimeUnit.SECONDS))
    executor.shutdownNow();                 // cancel remaining tasks
```

### ThreadPoolExecutor (production control)

The `Executors` factories use unbounded queues (or unbounded threads), which can exhaust memory under load. Configure a bounded pool directly:

```java
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    5, 10,                              // core, max threads
    60, TimeUnit.SECONDS,               // idle keep-alive
    new ArrayBlockingQueue<>(100),      // bounded queue
    new ThreadPoolExecutor.CallerRunsPolicy());   // rejection policy
```

| Rejection policy    | Behavior when queue and pool are full        |
| ------------------- | -------------------------------------------- |
| AbortPolicy (default) | Throws `RejectedExecutionException`        |
| CallerRunsPolicy    | Caller runs the task itself (throttles input)|
| DiscardPolicy       | Silently drops the task                      |
| DiscardOldestPolicy | Drops the oldest queued task, then enqueues  |

## CompletableFuture (Java 8+)

Composable asynchronous pipelines without blocking.

```java
CompletableFuture.supplyAsync(() -> "Hello")   // runAsync(...) for no result
    .thenApply(s -> s + " World")              // transform (Function)
    .thenApply(String::toUpperCase)
    .thenAccept(System.out::println)           // consume (Consumer)
    .thenRun(() -> System.out.println("Done")); // run after (Runnable)
```

### Combining

```java
CompletableFuture<String> name = CompletableFuture.supplyAsync(() -> "Alice");
CompletableFuture<Integer> age  = CompletableFuture.supplyAsync(() -> 25);
name.thenCombine(age, (n, a) -> n + " is " + a);   // "Alice is 25"

CompletableFuture.allOf(f1, f2, f3).join();        // wait for all
CompletableFuture.anyOf(f1, f2);                   // complete on the first
```

### Exception handling

```java
cf.exceptionally(e -> "fallback: " + e.getMessage())   // recover on failure
  .thenAccept(System.out::println);

cf.handle((result, ex) -> ex != null ? "error" : result);   // sees both outcomes
```

### join() vs get(), and async variants

`get()` throws the checked `ExecutionException`/`InterruptedException`; `join()` throws the unchecked `CompletionException`, which is convenient inside streams and lambdas. Non-`Async` stages run on the thread that completed the previous stage; `thenApplyAsync` runs on the common `ForkJoinPool` (or a supplied executor).

> **Note**: the default pool is `ForkJoinPool.commonPool()`, shared with parallel streams; do not run blocking I/O on it, pass a dedicated executor instead.

## Virtual Threads (Java 21+)

A virtual thread (Project Loom) is a `Thread` managed by the JVM rather than the OS. They cost a few KB each, so millions can exist. When one blocks on I/O the JVM unmounts it from its carrier (platform) thread, freeing the carrier to run another, then resumes it when I/O completes. The result is one-thread-per-request blocking code that scales like async, without `CompletableFuture` chains or pooling.

| Dimension     | Platform thread        | Virtual thread                    |
| ------------- | ---------------------- | --------------------------------- |
| Backed by     | OS thread (~2 MB stack)| JVM (~KB, runs on a carrier)      |
| Creation cost | High, so pool them     | Near-zero, create one per task    |
| Max count     | Thousands              | Millions                          |
| Blocks on I/O | Blocks the OS thread   | Unmounts; the carrier runs others |
| Best for      | CPU-bound work         | I/O-bound work (requests, DB, RPC)|

```java
Thread.ofVirtual().start(() -> handleRequest());   // one-off

try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {   // recommended
    for (int i = 0; i < 10_000; i++) executor.submit(() -> callSlowApi());
}
```

Do not pool virtual threads; create one per task. Avoid them for CPU-bound work (no throughput gain, use platform threads) and for heavy `ThreadLocal` state (one copy per virtual thread can blow up memory).

> **Note**: before JDK 24, a `synchronized` block held across a blocking call pinned the carrier thread; JEP 491 (JDK 24) removed this for the common cases. On Java 21 to 23, prefer `ReentrantLock` over `synchronized` around blocking I/O.

## Concurrent Collections

```java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();  // CAS + per-bucket locks
map.computeIfAbsent("b", k -> 2);                                    // atomic

CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();    // snapshot on write, read-heavy
ConcurrentLinkedQueue<String> q = new ConcurrentLinkedQueue<>();     // non-blocking

BlockingQueue<String> bq = new ArrayBlockingQueue<>(10);             // bounded; LinkedBlockingQueue unbounded
bq.put("item");   // blocks if full
bq.take();        // blocks if empty; removes the need for manual wait()/notify()
```

## Common Concurrency Problems

### Deadlock

Two threads each hold a lock the other needs, so both wait forever.

```java
Account a = new Account(1000), b = new Account(1000);

new Thread(() -> { synchronized (a) { sleep(50); synchronized (b) { /* a then b */ } } }).start();
new Thread(() -> { synchronized (b) { sleep(50); synchronized (a) { /* b then a */ } } }).start();
```

Fix by acquiring locks in one global order for every thread, so no cycle can form:

```java
Account first  = System.identityHashCode(a) < System.identityHashCode(b) ? a : b;
Account second = first == a ? b : a;
synchronized (first) { synchronized (second) { /* safe */ } }
```

Also help: `tryLock()` with a timeout, narrower lock scope, fewer nested locks. Detect with `jstack` or a thread dump (look for mutually BLOCKED threads).

### Livelock, Starvation, Race Condition

- **Livelock**: threads keep reacting to each other and never progress, like two people sidestepping in a corridor.
- **Starvation**: a thread never gets the CPU or lock because others monopolize it; fair locks (`new ReentrantLock(true)`) help.
- **Race condition**: the result depends on interleaving, typically a non-atomic check-then-act.

```java
if (map.containsKey(key)) map.get(key);   // another thread may remove key in between
map.computeIfAbsent(key, k -> create());  // fix: one atomic operation
```

## Q&A

1. **Difference between `start()` and `run()`?**
`start()` spawns a new thread and runs `run()` on it. Calling `run()` directly just executes it on the current thread, with no parallelism.

2. **How do `sleep()` and `wait()` differ on locks?**
`sleep()` keeps every lock the thread holds; `wait()` releases the monitor of its target object so others can enter. `wait()` must run inside a synchronized context; `sleep()` can run anywhere.

3. **What happens if you call `start()` twice?**
It throws `IllegalThreadStateException`. A thread is one-shot and cannot be restarted.

4. **Why is `HashMap` unsafe under concurrency?**
It is unsynchronized; concurrent `put` during a resize can corrupt the table or (pre-Java 8) spin into an infinite loop. Use `ConcurrentHashMap`.

5. **What does "reentrant" mean for `synchronized`?**
A thread already holding a lock can re-enter any block guarded by the same lock without blocking itself, which makes recursive synchronized calls safe.

6. **Why avoid string literals or boxed values as locks?**
They are cached and shared application-wide (string pool, `Integer` cache), so unrelated code may lock on the same instance and cause surprise contention or deadlock.

7. **How does `ThreadLocal` work, and what must you clean up?**
It gives each thread its own copy of a value, making non-thread-safe objects like `SimpleDateFormat` safe without locking. In pooled threads you must call `remove()` when done, or the value leaks with the recycled thread.

8. **`volatile` vs `Atomic` vs `synchronized`?**
- **volatile**: visibility of a single field (one writer, many readers).
- **Atomic**: lock-free read-modify-write on one variable (counters).
- **synchronized / Lock**: exclusive access across multiple variables or steps.

9. **Does `interrupt()` stop a thread immediately?**
No, it is cooperative: it sets a flag the thread must check, or causes a blocking call like `sleep()` to throw `InterruptedException`. On catching it, restore the flag with `Thread.currentThread().interrupt()` before exiting.

## Quick Revision Cheat Sheet

| Concept               | Remember                                                                |
| --------------------- | ----------------------------------------------------------------------- |
| Thread creation       | Runnable/Callable preferred; Callable returns a value; avoid extending Thread |
| synchronized          | Mutual exclusion + visibility; method, block, or static (Class lock)    |
| volatile              | Visibility and ordering only, not atomicity; for flags, not `count++`   |
| wait/notify           | Inside synchronized, holding the monitor; loop with `while`             |
| sleep vs wait         | `sleep` keeps the lock; `wait` releases it                              |
| Lock vs synchronized  | ReentrantLock adds tryLock, timeout, fairness, interruptible, read/write |
| Atomic                | Lock-free CAS per variable; `LongAdder` for write-heavy counters        |
| ExecutorService       | Pool tasks; bound the queue (`ThreadPoolExecutor`); always shut down    |
| CompletableFuture     | Non-blocking chaining: supplyAsync, thenApply, thenCombine, exceptionally |
| Virtual threads       | Java 21+, cheap, one per task, for I/O-bound work; never pool           |
| Concurrent collections| ConcurrentHashMap, CopyOnWriteArrayList, BlockingQueue                  |
| Deadlock / race       | Deadlock: one global lock order; race: atomic or concurrent operations  |
