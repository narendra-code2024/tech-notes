## `java.lang.Number`

The abstract class `java.lang.Number` is the common superclass for all classes representing numeric values convertible to primitives. It defines the conversion contract:

```java
public abstract class Number implements java.io.Serializable {
    public abstract int    intValue();
    public abstract long   longValue();
    public abstract float  floatValue();
    public abstract double doubleValue();

    public byte  byteValue()  { return (byte)intValue(); }
    public short shortValue() { return (short)intValue(); }
}
```

### Subclasses of `java.lang.Number`

| Class                                                 | Purpose                                         |
| :---------------------------------------------------- | :---------------------------------------------- |
| `Byte`, `Short`, `Integer`, `Long`, `Float`, `Double` | Wrap standard numeric primitives                |
| `BigInteger`, `BigDecimal`                            | Arbitrary-precision math                        |
| `AtomicInteger`, `AtomicLong`                         | Thread-safe, lock-free numeric operations       |
| `LongAdder`, `DoubleAdder`                            | High-throughput concurrent numeric accumulation |

> **Reference**: `Atomic*`, `LongAdder`, and `LongAccumulator` in depth, see [Multithreading & Concurrency, Atomic Classes](<25. Multithreading & Concurrency.md#Atomic Classes (java.util.concurrent.atomic)>).

## `java.lang.Math`

A stateless utility class of static methods for core primitive math. It works directly on primitives (no boxing) and is auto-imported from `java.lang`.

```java
// 1. Abs, min, max
Math.abs(-5);                              // 5 (int, long, float, double)
Math.min(3, 7);                            // 3
Math.max(3, 7);                            // 7

// 2. Powers and roots
Math.pow(2, 10);                           // 1024.0 (always returns double)
Math.sqrt(16);                             // 4.0
Math.cbrt(27);                             // 3.0

// 3. Rounding
Math.round(2.5);                           // 3   (nearest integer; .5 ties round toward +∞)
Math.floor(2.7);                           // 2.0 (largest int ≤ x, as double)
Math.ceil(2.1);                            // 3.0 (smallest int ≥ x, as double)

// 4. Floor division and modulo (floor toward -∞ instead of truncating toward 0)
-7 / 2;                                     // -3 (truncates)
-7 % 2;                                     // -1 (sign matches dividend)
Math.floorDiv(-7, 2);                      // -4 (floors)
Math.floorMod(-7, 2);                      // 1  (non-negative when divisor is positive)

// 5. Overflow-checked arithmetic (throws ArithmeticException on overflow)
Math.addExact(Integer.MAX_VALUE, 1);       // throws
Math.multiplyExact(100_000, 100_000);      // throws (10^10 overflows 32-bit int)
Math.subtractExact(Integer.MIN_VALUE, 1);  // throws
Math.toIntExact(someLong);                 // throws if the long doesn't fit in an int
```

## `java.math.BigInteger`

Rarely needed in app code. Reach for it when values exceed `long` range (±9.2 × 10¹⁸), as in number theory, cryptography, or combinatorics.

```java
BigInteger a = new BigInteger("123456789012345678901234567890");
BigInteger b = BigInteger.valueOf(42);

// Arithmetic returns a new BigInteger: add, subtract, multiply, divide, mod, pow
BigInteger sum = a.add(b);
a.compareTo(b);            // -1, 0, or 1 (use compareTo for ordering)

// Constants: BigInteger.ZERO, ONE, TEN
```

**Immutability**: instances are immutable; operations return a new `BigInteger`, so you must reassign the result (`a = a.add(b)`) or the value is discarded.

## `java.math.BigDecimal`

`double` cannot represent `0.1` exactly. For money, tax, billing, and invoicing, use `BigDecimal`.

```java
// 1. Instantiation (always use the String constructor or valueOf)
BigDecimal good = new BigDecimal("0.1");        // exact 0.1
BigDecimal bad  = new BigDecimal(0.1);          // inexact: 0.1000000000000000055511151231...
BigDecimal ok   = BigDecimal.valueOf(0.1);      // exact 0.1 (uses Double.toString)

// 2. Arithmetic (returns a new BigDecimal; divide needs scale + RoundingMode)
BigDecimal a = new BigDecimal("10.00");
BigDecimal b = new BigDecimal("3.00");
BigDecimal sum  = a.add(b);                     // 13.00 (does not modify a)
BigDecimal prod = a.multiply(b);                // 30.0000 (scales add: 2 + 2 = 4)
a.divide(b);                                    // throws ArithmeticException (non-terminating)
BigDecimal quot = a.divide(b, 2, RoundingMode.HALF_UP); // 3.33 (scale 2, half-up)

// 3. Scale and rounding
new BigDecimal("3.14").scale();                // 2 (digits after the decimal point)
new BigDecimal("100").scale();                 // 0
BigDecimal val = new BigDecimal("3.14159");
BigDecimal rounded = val.setScale(2, RoundingMode.HALF_UP); // 3.14

// 4. Value vs scale (equals checks value AND scale; compareTo checks value only)
new BigDecimal("2.0").equals(new BigDecimal("2.00"));    // false (scale differs)
new BigDecimal("2.0").compareTo(new BigDecimal("2.00")); // 0 (mathematically equal)
```

Always pass a scale and a `RoundingMode` to `divide()` and `setScale()` to avoid `ArithmeticException` on non-terminating decimals. Financial code typically uses `RoundingMode.HALF_EVEN` (banker's rounding), the IEEE 754 default, which rounds ties toward the nearest even neighbor and so reduces cumulative bias over many transactions.

**Immutability**: like `BigInteger`, `BigDecimal` is immutable; operations return a new instance, so reassign the result (`a = a.add(b)`).

## Number Parsing & Formatting

Convert between strings and numbers with wrapper parsers (`parseInt`) and locale-aware formatters (`NumberFormat`, `DecimalFormat`).

```java
// Standard parsing and radix variants
Integer.parseInt("42");                       // 42
Integer.parseInt("ff", 16);                   // 255 (hex)
Integer.parseInt("1010", 2);                  // 10  (binary)

// Edge cases and exceptions
Integer.parseInt(null);                       // throws NumberFormatException ("Cannot parse null string")
Integer.parseInt("");                         // throws NumberFormatException (empty)
Integer.parseInt(" 42 ");                     // throws NumberFormatException (no trimming; call trim() first)
Integer.parseInt("1_000");                    // throws NumberFormatException (underscores are source-only)
Integer.parseInt("+42");                      // 42 (leading + valid since Java 7)
```

```java
// Locale-aware formatting (java.text.NumberFormat)
NumberFormat usCurrency = NumberFormat.getCurrencyInstance(Locale.US);
usCurrency.format(1234.56);                   // "$1,234.56"

NumberFormat percent = NumberFormat.getPercentInstance();
percent.format(0.75);                         // "75%"

// Pattern-based formatting (java.text.DecimalFormat)
DecimalFormat df = new DecimalFormat("#,##0.00");
df.format(1234.5);                            // "1,234.50"
```

> **Note**: `DecimalFormat` is not thread-safe. In multi-threaded code, create an instance per call or use a `ThreadLocal<DecimalFormat>`. The same applies to `SimpleDateFormat`.

## Random Number Generation

The JDK offers three main random APIs, differing in thread-safety and security.

```java
// 1. java.util.Random (pseudo-random, thread-safe but contended)
Random r1 = new Random();                     // seed = atomic uniquifier ^ nanoTime
Random r2 = new Random(42L);                  // fixed seed for reproducible tests

r1.nextInt();                                 // any pseudorandom 32-bit int
r1.nextInt(100);                              // 0..99 (bound exclusive, must be positive)
r1.nextInt(10, 20);                           // 10..19 (Java 17+)
r1.nextDouble();                              // [0.0, 1.0)
r1.nextBoolean();                             // true or false

// 2. java.util.concurrent.ThreadLocalRandom (per-thread, zero contention)
int num = ThreadLocalRandom.current().nextInt(100);
double val = ThreadLocalRandom.current().nextDouble();

// 3. java.security.SecureRandom (cryptographically secure, slower, unpredictable)
SecureRandom secure = new SecureRandom();
byte[] token = new byte[16];
secure.nextBytes(token);                      // fills with secure random bytes
```

Which generator to use:

| Use case                              | Class                         |
| :------------------------------------ | :---------------------------- |
| Single-threaded, need reproducibility | `Random(seed)`                |
| Multi-threaded                        | `ThreadLocalRandom.current()` |
| Security (tokens, passwords, nonces)  | `SecureRandom`                |

> **Note**: never use `Random` or `ThreadLocalRandom` for anything security-sensitive; their output is deterministic and predictable once the seed or history leaks. Use `SecureRandom` for tokens, passwords, and session IDs.

## Q&A

1. **Why is using `double`/`float` for money a bug?**
Binary floating-point cannot represent many decimal values exactly (such as `0.1`), so errors accumulate, e.g. `0.1 + 0.2` is `0.30000000000000004`. For money or any exact math, use `BigDecimal`.

2. **Does `BigDecimal.equals()` behave like `compareTo()`?**
No. `equals()` checks value *and* scale, so `new BigDecimal("2.0").equals(new BigDecimal("2.00"))` is `false`. To compare mathematical value only, use `compareTo(...) == 0`.

3. **Why avoid `new BigDecimal(double)`?**
It captures the exact binary approximation of the double, imprecision included, so `new BigDecimal(0.1)` becomes `0.1000000000000000055...`. Use the `String` constructor or `BigDecimal.valueOf(double)` for the expected value.

4. **Why does `BigDecimal.divide()` sometimes throw?**
A non-terminating quotient (like `10 / 3`) with no scale or rounding mode cannot be represented exactly, so it throws `ArithmeticException`. Always pass a scale and a `RoundingMode` when dividing.

5. **What happens on integer overflow, and how do you detect it?**
Overflow is silent: `Integer.MAX_VALUE + 1` wraps to `Integer.MIN_VALUE`. In critical code use the exact methods (`Math.addExact`, `Math.multiplyExact`, `Math.toIntExact`), which throw `ArithmeticException` instead.

6. **When prefer `ThreadLocalRandom` over `new Random()`?**
`Random` is thread-safe but contends under load because all threads share one atomic seed updated by CAS. `ThreadLocalRandom` gives each thread its own generator, removing that contention; modern `new Random()` seeds are still unique within the same millisecond via the atomic uniquifier.
