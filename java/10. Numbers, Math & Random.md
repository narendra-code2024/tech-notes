## `java.lang.Number`

The abstract class `java.lang.Number` serves as the common superclass for all platform classes representing numeric values that are convertible to primitive types. It defines the core contract for numeric conversion:

```java
public abstract class Number implements java.io.Serializable {
    public abstract int    intValue();
    public abstract long   longValue();
    public abstract float  floatValue();
    public abstract double doubleValue();

    public byte  byteValue() { return (byte)intValue(); }
    public short shortValue() { return (short)intValue(); }
}
```

### Subclasses of `java.lang.Number`

| Class                                                 | Purpose                                         |
| :---------------------------------------------------- | :---------------------------------------------- |
| `Byte`, `Short`, `Integer`, `Long`, `Float`, `Double` | Wrap standard numeric primitives                |
| `BigInteger`, `BigDecimal`                            | Arbitrary-precision math                        |
| `AtomicInteger`, `AtomicLong`                         | Thread-safe, lock-free numeric operations        |
| `LongAdder`, `DoubleAdder`                            | High-throughput concurrent numeric accumulation |

> **Reference**: detailed implementation of `Atomic*`, `LongAdder`, `LongAccumulator` → [Concurrency, Atomic Classes](<26. Multithreading & Concurrency.md#Atomic Classes (java.util.concurrent.atomic)>).

## `java.lang.Math`

Stateless utility class containing static methods for core primitive math operations. It operates directly on primitives (avoiding boxing overhead) and is automatically imported from `java.lang`.

```java
// 1. Abs, Min, Max
Math.abs(-5);                              // 5 (works on int, long, float, double)
Math.min(3, 7);                            // 3
Math.max(3, 7);                            // 7

// 2. Powers & Roots
Math.pow(2, 10);                           // 1024.0 (always returns double, even for integer inputs)
Math.sqrt(16);                             // 4.0
Math.cbrt(27);                             // 3.0

// 3. Rounding
Math.round(2.5);                           // 3      (rounds to nearest integer; .5 ties round up)
Math.floor(2.7);                           // 2.0    (largest int ≤ x, returns double)
Math.ceil(2.1);                            // 3.0    (smallest int ≥ x, returns double)

// 4. Floor Division & Modulo (floors toward -∞ instead of truncating toward 0)
-7 / 2;                                    // -3     (truncates)
-7 % 2;                                    // -1     (sign matches dividend)
Math.floorDiv(-7, 2);                      // -4     (floors)
Math.floorMod(-7, 2);                      // 1      (always non-negative when divisor is positive)

// 5. Overflow-Checked Arithmetic (throws ArithmeticException on overflow)
Math.addExact(Integer.MAX_VALUE, 1);       // throws (overflows past MAX_VALUE)
Math.multiplyExact(100_000, 100_000);      // throws (10B overflows 32-bit int)
Math.subtractExact(Integer.MIN_VALUE, 1);  // throws (underflows past MIN_VALUE)
Math.toIntExact(someLong);                 // throws if long doesn't fit in int
```

## `java.math.BigInteger`

Rarely needed in typical app code. Reach for it when values exceed `long` range (±9.2 × 10¹⁸); for example, number theory, cryptography, or combinatorics.

```java
BigInteger a = new BigInteger("123456789012345678901234567890");
BigInteger b = BigInteger.valueOf(42);

// Arithmetic returns a new BigInteger: add, subtract, multiply, divide, mod, pow
BigInteger sum = a.add(b);
a.compareTo(b);            // -1, 0, or 1 (always use compareTo for arbitrary precision)

// Constants: BigInteger.ZERO, ONE, TEN
```

**Immutability**: instances are immutable; operations return a new `BigInteger`, so you must reassign the result (`a = a.add(b)`) or the computed value is silently discarded.

## `java.math.BigDecimal`

`double` can't represent `0.1` exactly. For money, tax, billing, and invoicing, use `BigDecimal`.

```java
// 1. Instantiation & Double Pitfalls (always use String constructor or valueOf)
BigDecimal good = new BigDecimal("0.1");        // OK, exact 0.1
BigDecimal bad  = new BigDecimal(0.1);          // inaccurate: 0.1000000000000000055511151231...
BigDecimal ok   = BigDecimal.valueOf(0.1);      // OK, exact 0.1 (uses Double.toString)

// 2. Arithmetic Operations (returns a new BigDecimal; divide requires scale + RoundingMode)
BigDecimal a = new BigDecimal("10.00");
BigDecimal b = new BigDecimal("3.00");
BigDecimal sum = a.add(b);                      // 13.00 (returns new BigDecimal, does not modify a)
BigDecimal prod = a.multiply(b);                // 30.0000 (scales add: 2 + 2 = 4)
a.divide(b);                                    // throws ArithmeticException (non-terminating decimal)
BigDecimal quot = a.divide(b, 2, RoundingMode.HALF_UP); // OK, 3.33 (scale=2, half-up)

// 3. Scaling & Rounding
new BigDecimal("3.14").scale();                // 2 (digits after decimal)
new BigDecimal("100").scale();                 // 0
BigDecimal val = new BigDecimal("3.14159");
BigDecimal rounded = val.setScale(2, RoundingMode.HALF_UP); // OK, 3.14

// 4. Value vs Scale Comparisons (equals checks value AND scale; use compareTo for value equality)
new BigDecimal("2.0").equals(new BigDecimal("2.00"));     // false (scale differs)
new BigDecimal("2.0").compareTo(new BigDecimal("2.00"));  // 0 (mathematically equal)
```

**Rounding & Scaling**:
Always specify a scale and a `RoundingMode` when calling `divide()` or `setScale()` to prevent `ArithmeticException` on non-terminating decimals. Production financial applications typically use **`RoundingMode.HALF_EVEN`** (Banker's rounding). This is the IEEE 754 default and reduces cumulative rounding bias over many transactions by rounding toward the nearest even neighbor when ties occur.

**Immutability**: `BigDecimal` is immutable like `BigInteger`; operations return a new instance, so you must reassign the result (e.g., `a = a.add(b)`).

## Number Parsing & Formatting

Convert between strings and numbers with wrapper parsers (`parseInt`) and locale-aware formatters (`NumberFormat`, `DecimalFormat`).

```java
// 1. Standard Parsing & Radix Variants
Integer.parseInt("42");                       // 42
Integer.parseInt("ff", 16);                   // 255 (hex parsing)
Integer.parseInt("1010", 2);                  // 10  (binary parsing)

// 2. Parsing Edge Cases & Common Exceptions
Integer.parseInt(null);                       // throws NullPointerException
Integer.parseInt("");                         // throws NumberFormatException (empty string)
Integer.parseInt(" 42 ");                     // throws NumberFormatException (whitespace not tolerated; call trim() first)
Integer.parseInt("1_000");                    // throws NumberFormatException (underscores are source literals only)
Integer.parseInt("+42");                      // OK, 42 (leading positive sign is valid since Java 7)
```

```java
// 1. Locale-Aware Formatting (java.text.NumberFormat)
NumberFormat USCurrency = NumberFormat.getCurrencyInstance(Locale.US);
USCurrency.format(1234.56);                   // "$1,234.56"

NumberFormat percent = NumberFormat.getPercentInstance();
percent.format(0.75);                         // "75%"

// 2. Pattern-Based Formatting (java.text.DecimalFormat)
DecimalFormat df = new DecimalFormat("#,##0.00");
df.format(1234.5);                            // "1,234.50"
```

> **Note**: `DecimalFormat` is **not thread-safe**. In multi-threaded environments, create instances per-call, or use a `ThreadLocal<DecimalFormat>`. The same constraint applies to `SimpleDateFormat`.

## Random Number Generation

JDK provides three primary random number generator APIs, each tailored to specific thread-safety and security requirements.

```java
// 1. java.util.Random (Pseudo-Random, thread-safe but contended)
Random r1 = new Random();                     // Seeded using thread-safe atomic uniquifier + nanoTime
Random r2 = new Random(42L);                  // Deterministic seed (useful for reproducible unit tests)

r1.nextInt();                                 // Returns any pseudorandom 32-bit int
r1.nextInt(100);                              // Returns 0 to 99 inclusive (bound must be positive)
r1.nextInt(10, 20);                           // Returns 10 to 19 inclusive (Java 17+)
r1.nextDouble();                              // Returns value in range [0.0, 1.0)
r1.nextBoolean();                             // Returns true or false

// 2. java.util.concurrent.ThreadLocalRandom (Zero contention, multi-threaded use)
// Uses ThreadLocal to maintain a separate seed per thread, eliminating CAS contention of java.util.Random.
int num = ThreadLocalRandom.current().nextInt(100);
double val = ThreadLocalRandom.current().nextDouble();

// 3. java.security.SecureRandom (Cryptographically secure, slower but unpredictable)
SecureRandom secure = new SecureRandom();
byte[] token = new byte[16];
secure.nextBytes(token);                      // Fills byte array with cryptographically secure random bytes
```

**Choosing a Random Generator**:

| Use Case                              | Class                        |
| :------------------------------------ | :--------------------------- |
| Single-threaded, need reproducibility | `Random(seed)`               |
| Multi-threaded                        | `ThreadLocalRandom.current()` |
| Security (tokens, passwords, nonces)  | `SecureRandom`                |

> **Note**: Never use `Random` or `ThreadLocalRandom` for anything security-sensitive. Their underlying algorithms are deterministic and predictable if the seed or output history is exposed. Always use `SecureRandom` for security tokens, passwords, and session IDs.

## Q&A

1. **Why is using `double` or `float` for financial calculations considered a bug?**
Floating-point types use binary representation which cannot exactly represent many decimal values (like `0.1`). This leads to precision errors; for example, `0.1 + 0.2` results in `0.30000000000000004`. For money or any exact math, you must use `BigDecimal`.

2. **Does `BigDecimal.equals()` behave the same as `compareTo()`?**
No. `BigDecimal.equals()` checks both the **value and the scale**. This means `new BigDecimal("2.0").equals(new BigDecimal("2.00"))` returns `false`. To compare just the mathematical value, you should use `compareTo() == 0`.

3. **Why should you avoid the `new BigDecimal(double)` constructor?**
The `double` constructor captures the exact binary approximation of the double value, including its imprecision. For example, `new BigDecimal(0.1)` will result in a value like `0.1000000000000000055...`. You should always use the **`String` constructor** or `BigDecimal.valueOf(double)` to get the expected decimal value.

4. **Why does `BigDecimal.divide()` sometimes throw an `ArithmeticException`?**
If the result of a division is a non-terminating decimal (like `10 / 3`) and you haven't specified a scale or a rounding mode, Java throws an exception because it cannot represent the infinite result. You should always provide a scale and a `RoundingMode` when dividing.

5. **What happens during integer overflow in Java, and how can you detect it?**
Integer overflow is silent in Java. If you add `1` to `Integer.MAX_VALUE`, it wraps around to `Integer.MIN_VALUE`. To prevent this in critical code, use the "Exact" methods from the `Math` class (e.g., `Math.addExact()`, `Math.multiplyExact()`), which throw an `ArithmeticException` if an overflow occurs.

6. **When should you prefer `ThreadLocalRandom` over a standard `new Random()`?**
`Random` is thread-safe but suffers from high contention in multi-threaded environments because all threads share a single atomic seed value (requiring continuous CAS updates). `ThreadLocalRandom` provides a separate generator for each thread, eliminating contention. (Note: Unlike older Java versions, modern `new Random()` seeds are guaranteed unique even when created in the same millisecond via thread-safe atomic seed uniquifiers).
