Serialization converts a Java object into a byte stream so it can be written to a file, sent over a network, or stored; deserialization reverses it, rebuilding the object from the bytes.

```
Object → byte stream → file / network / DB    (Serialization)
file / network / DB → byte stream → Object    (Deserialization)
```

Common uses: persisting session state, sending objects over RMI or sockets, caching (Redis, Memcached), and deep-cloning an object graph.

---

## How to Serialize

A class must implement `java.io.Serializable`, a marker interface (no methods, just a signal that instances may be serialized).

```java
class Employee implements Serializable {
    private static final long serialVersionUID = 1L;
    String name;
    int salary;
    transient String password;   // excluded from serialization

    public Employee(String name, int salary, String password) {
        this.name = name; this.salary = salary; this.password = password;
    }
}
```

```java
Employee emp = new Employee("Alice", 50000, "secret123");

try (ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("emp.ser"))) {
    oos.writeObject(emp);
}

try (ObjectInputStream ois = new ObjectInputStream(new FileInputStream("emp.ser"))) {
    Employee e = (Employee) ois.readObject();
    System.out.println(e.name);       // "Alice"
    System.out.println(e.password);   // null: transient, not serialized
}
```

---

## serialVersionUID

A version stamp the JVM uses to confirm the serialized bytes and the loaded class are the same version.

```java
private static final long serialVersionUID = 1L;
```

Without an explicit value the JVM computes one from the class structure (fields, methods, modifiers), so any change to the class yields a different value and deserializing older bytes fails with `InvalidClassException`. Declaring it fixes the version and keeps compatibility under your control: with the UID unchanged, a removed field is ignored on read and a newly added field deserializes to its default.

```java
class Employee implements Serializable {
    private static final long serialVersionUID = 1L;
    String name;
    int age;    // added later; old byte streams restore age as 0, same UID still works
}
```

---

## transient Keyword

A `transient` field is excluded from serialization and comes back as its default (`null`, `0`, `false`) on deserialization.

```java
class User implements Serializable {
    String username;
    transient String password;     // sensitive
    transient Logger logger;       // non-serializable reference
}
```

Use it for sensitive data (passwords, tokens), derived fields that can be recomputed, non-serializable references (loggers, threads, DB connections), and temporary state.

---

## static Fields and Serialization

`static` fields are not serialized; they belong to the class, not the instance. After deserialization a static field holds whatever the currently loaded class has, not its value at serialization time.

```java
class Config implements Serializable {
    static String appName = "MyApp";  // not serialized
    String env;                       // serialized
}
```

## Inheritance and Serialization

### Serializable parent

A subclass of a `Serializable` class is itself serializable, and inherited fields are serialized.

```java
class Parent implements Serializable { int x = 10; }
class Child extends Parent { int y = 20; }   // both x and y are serialized
```

### Non-serializable parent

If only the child is `Serializable`, the parent's fields are not restored from the stream; instead the parent's no-arg constructor runs during deserialization.

```java
class Parent {                 // not Serializable
    int x = 10;
    Parent() { x = 10; }       // must exist and be reachable
}
class Child extends Parent implements Serializable { int y = 20; }

// on deserialize: y = 20 (from stream), x = 10 (from Parent's constructor, not the stream)
```

> **Note**: if the first non-serializable superclass has no accessible no-arg constructor, deserialization throws `InvalidClassException`.

## Custom Serialization (writeObject / readObject)

Private `writeObject`/`readObject` hooks override the default behavior, for encryption, validation, or restoring transient fields.

```java
class Account implements Serializable {
    private static final long serialVersionUID = 1L;
    String name;
    transient String password;

    private void writeObject(ObjectOutputStream oos) throws IOException {
        oos.defaultWriteObject();              // the normal (non-transient) fields
        oos.writeObject(encrypt(password));    // handle the transient field manually
    }

    private void readObject(ObjectInputStream ois) throws IOException, ClassNotFoundException {
        ois.defaultReadObject();
        this.password = decrypt((String) ois.readObject());
    }
}
```

## Externalizable Interface

Full manual control: you write and read every field yourself via `writeExternal`/`readExternal`, and a public no-arg constructor is mandatory (deserialization calls it before reading fields). `transient` is irrelevant because nothing is automatic.

```java
class Employee implements Externalizable {
    String name;
    int salary;

    public Employee() { }   // mandatory, public

    public void writeExternal(ObjectOutput out) throws IOException {
        out.writeUTF(name);  out.writeInt(salary);
    }
    public void readExternal(ObjectInput in) throws IOException {
        name = in.readUTF();  salary = in.readInt();
    }
}
```

| Dimension          | Serializable                 | Externalizable                       |
| ------------------ | ---------------------------- | ------------------------------------ |
| Type               | Marker interface, no methods | `writeExternal` / `readExternal`     |
| Controls the bytes | JVM (reflection)             | You                                  |
| No-arg constructor | Not required                 | Required and public                  |
| transient          | Respected                    | Irrelevant (you choose what to write)|
| Use case           | Most cases                   | Performance-critical or full control |

## Serialization and Singleton

Deserialization creates a new object, which breaks a singleton. A `readResolve` hook swaps the deserialized instance for the canonical one.

```java
class Singleton implements Serializable {
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() { }
    public static Singleton getInstance() { return INSTANCE; }

    private Object readResolve() { return INSTANCE; }   // returned instead of the new instance
}
```

An `enum` singleton avoids the problem entirely; the JVM guarantees a single instance across serialization.

```java
enum Singleton { INSTANCE; }
```

## Security

Deserializing untrusted data is a serious vulnerability. `readObject` runs code across the whole object graph, and crafted byte streams chain together library classes ("gadget chains") to reach arbitrary code execution. Never deserialize bytes you do not control. If you must, restrict the accepted classes with an `ObjectInputFilter` (JEP 290, Java 9+). For data exchange, prefer formats like JSON or Protobuf that populate plain fields instead of reconstructing arbitrary Java types.

---

## Modern Alternatives

Java's built-in serialization carries security, performance, and verbosity costs, so production systems usually exchange data in a portable format instead.

| Alternative          | Format | Use case                            |
| -------------------- | ------ | ----------------------------------- |
| JSON (Jackson, Gson) | Text   | REST APIs, config, human-readable   |
| XML (JAXB)           | Text   | Legacy systems, SOAP                |
| Protocol Buffers     | Binary | High-performance, cross-language    |
| Avro                 | Binary | Big data, Kafka                     |
| Kryo                 | Binary | Fast JVM serialization (Spark, Flink)|

```java
ObjectMapper mapper = new ObjectMapper();          // Jackson
String json = mapper.writeValueAsString(employee); // object -> JSON
Employee emp = mapper.readValue(json, Employee.class);
```

In Spring Boot, Jackson is auto-configured, and a `@RestController` serializes and deserializes JSON through it automatically.

## Q&A

1. **What causes `NotSerializableException`?**
Serializing an object whose reference graph includes a class that does not implement `Serializable`. Make every reachable object serializable, or mark the offending fields `transient`.

2. **Is a constructor called during deserialization?**
For `Serializable` classes, no, the object is rebuilt reflectively without running its constructor. The no-arg constructor of the first non-serializable superclass does run. For `Externalizable`, the public no-arg constructor is called.

3. **Why are `static` and `transient` fields skipped?**
- **static**: belongs to the class, not the instance, so it is not part of an object's state.
- **transient**: explicitly marked to skip, usually for sensitive or temporary data.

4. **Does serialization handle nested object graphs?**
Yes, recursively: serializing a root object pulls in everything it references. If any object in the chain is not serializable, the whole operation fails.

5. **What is the exact `serialVersionUID` signature?**
`private static final long serialVersionUID = ...;`. If it is not `static`, not `long`, or otherwise malformed, the JVM ignores it and generates its own, reintroducing version fragility.

6. **What happens when the class changed but the UID matches?**
Removed fields present in the bytes are ignored, and fields added since are initialized to their defaults (`null`, `0`).

7. **How does deserialization handle `final` fields?**
It sets them reflectively, one of the few cases where a `final` field is assigned after construction.

8. **Does serialization preserve aliasing within one graph?**
Yes. If two fields point to the same object, the stream records the shared reference, and deserialization restores a single shared instance rather than two copies.
