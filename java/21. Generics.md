Generics let a class, interface, or method operate over a type supplied as a parameter, moving type checks from runtime to compile time.

A parameterized type like `List<String>` enforces its element type during compilation, so wrong-type insertions become compile errors and reads need no cast.

```java
// Raw type: compiles with an unchecked warning, fails at runtime
List raw = new ArrayList();
raw.add("hello");
raw.add(123);                     // accepted silently
String s1 = (String) raw.get(1);  // runtime: ClassCastException

// Parameterized: the same error moves to compile time
List<String> safe = new ArrayList<>();
safe.add(123);                    // compile error
String s2 = safe.get(0);          // no cast needed
```

## Generic Class

```java
// T is a type parameter: a placeholder bound to a concrete type at the use site
class Box<T> {
    private T val;
    public void set(T val) { this.val = val; }
    public T get()         { return val; }
}

Box<String> b = new Box<>();
b.set("hello");
String s = b.get();   // no cast
```

**Multiple type parameters**:

```java
class Pair<K, V> {
    private final K key;
    private final V val;
    Pair(K key, V val) { this.key = key; this.val = val; }
    K key() { return key; }
    V val() { return val; }
}

Pair<String, Integer> p = new Pair<>("age", 25);
```

- **Naming**: `T` type, `E` element, `K`/`V` key/value, `N` number; `S`, `U` for further parameters.

## Generic Method

A method can declare its own type parameter, independent of any class-level one, written before the return type.

```java
// <T> before the return type declares the method's own parameter
public static <T> void printAll(T[] arr) {
    for (T e : arr) System.out.print(e + " ");
}

// compiler infers T from the argument
public static <T> T first(List<T> list) { return list.get(0); }

printAll(new Integer[]{1, 2, 3});      // T = Integer
String s = first(List.of("a", "b"));   // T = String
```

- **Generic constructor**: a constructor may declare its own `<T>` even in a non-generic class, e.g. `<T> Util(T item) { ... }`; `T` is inferred from the argument at `new Util("x")`.

---

## Generic Interface

An interface can be parameterized; an implementer either fixes the type arguments or stays generic and forwards them.

```java
interface Transformer<T, R> {
    R transform(T input);
}

// fix the type arguments
class ParseInt implements Transformer<String, Integer> {
    public Integer transform(String in) { return Integer.parseInt(in); }
}

// or remain generic and forward them
class Identity<T> implements Transformer<T, T> {
    public T transform(T in) { return in; }
}
```

## Bounded Type Parameters

A bound restricts which types may be supplied as an argument and grants access to the bound's members inside the body.

### Upper bound (`extends`)

`<T extends Number>` accepts `Number` and its subtypes, and lets you call `Number` methods on `T`.

```java
public static <T extends Number> double sum(List<T> list) {
    double total = 0;
    for (T n : list) total += n.doubleValue();   // Number method available
    return total;
}

sum(List.of(1, 2, 3));     // Integer extends Number: OK
sum(List.of("a", "b"));    // compile error: String is not a Number
```

In generics, `extends` means "is a subtype of" and covers both class inheritance and interface implementation; there is no `implements` keyword.

```java
<T extends Comparable<T>>     // OK
<T implements Comparable<T>>  // compile error: no such keyword
```

### Multiple bounds (`&`)

List several bounds with `&`; `T` must satisfy all. A class bound must come first, followed by interfaces.

```java
public static <T extends Number & Comparable<T>> T max(List<T> list) {
    return Collections.max(list);
}
```

> **Note**: there is no `super` bound for type parameters; `<T super Integer>` is invalid. `super` applies only to wildcards.

## Wildcards (`?`)

A wildcard is an anonymous type argument, used when the exact type need not be named, only constrained.

### Unbounded (`?`)

`List<?>` is a list of some unknown type; elements read back as `Object`, and only `null` can be added.

```java
public static void printList(List<?> list) {
    for (Object o : list) System.out.println(o);
}
```

### Upper bound (`? extends T`, producer)

Some unknown subtype of `T`: safe to read as `T`, but nothing can be added, since the exact type is unknown.

```java
public static double sum(List<? extends Number> list) {
    double total = 0;
    for (Number n : list) total += n.doubleValue();   // read as Number: OK
    return total;
    // list.add(1) would be a compile error: element type unknown
}

sum(List.of(1, 2, 3));     // List<Integer>: OK
sum(List.of(1.5, 2.5));    // List<Double>: OK
```

### Lower bound (`? super T`, consumer)

Some unknown supertype of `T`: safe to add a `T`, but reads yield only `Object`.

```java
public static void addInts(List<? super Integer> list) {
    list.add(1);                 // add Integer: OK
    Object o = list.get(0);      // reads are Object only
}

addInts(new ArrayList<Number>());   // Number is a supertype of Integer: OK
addInts(new ArrayList<String>());   // compile error: String is not a supertype of Integer
```

### PECS (Producer Extends, Consumer Super)

- Read from it (it produces values)? use `extends`.
- Write into it (it consumes values)? use `super`.
- Both read and write? use an exact type, no wildcard.

```java
// src is read (producer) -> extends; dest is written (consumer) -> super
public static <T> void copy(List<? extends T> src, List<? super T> dest) {
    for (T item : src) dest.add(item);
}
```

The JDK follows the same rule, e.g. `Collection.addAll(Collection<? extends E> c)`:

```java
List<Number> nums = new ArrayList<>();
nums.addAll(List.of(1, 2));        // List<Integer>: OK
nums.addAll(List.of(1.5, 2.5));    // List<Double>: OK
```

### Wildcard vs named type parameter

| Use                  | When                                                            |
| -------------------- | -------------------------------------------------------------- |
| `<T extends Number>` | `T` is referenced in several places (parameters, return, body) |
| `? extends Number`   | a single parameter needs flexibility and the type is never named |

## Type Erasure

Generics are a compile-time feature: the compiler verifies types, then erases them so the bytecode carries no type parameters. Each parameter is replaced by its leftmost bound (or `Object` if unbounded), and casts are inserted at call sites.

```java
// source
List<String> list = new ArrayList<>();
String s = list.get(0);

// after erasure (what the JVM runs)
List list = new ArrayList();
String s = (String) list.get(0);   // cast inserted by the compiler

class Box<T extends Number> { T val; }
// erases to:
class Box { Number val; }           // the bound, not Object
```

Consequences:

- No `new T()` or `new T[]`: the runtime type is unknown.
- No `instanceof List<String>`: only the reifiable `List<?>` is allowed.
- `static` members cannot use the class type parameter `T`.
- No two overloads differing only by type argument: `f(List<String>)` and `f(List<Integer>)` erase to the same signature.
- All parameterizations share one runtime class:

```java
new ArrayList<String>().getClass() == new ArrayList<Integer>().getClass();   // true
```

## Arrays vs Generics

Arrays and generics differ in variance and in whether type information survives to runtime.

- **Arrays are covariant and reified**: `String[]` is a subtype of `Object[]`, and the array checks its component type at runtime, so a bad store throws `ArrayStoreException`.
- **Generics are invariant and erased**: `List<String>` is not a subtype of `List<Object>`, so the same mistake is caught at compile time.

```java
Object[] arr = new String[3];   // compiles (covariance)
arr[0] = 123;                   // runtime: ArrayStoreException

List<Object> list = new ArrayList<String>();   // compile error (invariance)
```

### Generic array creation

Because generics are erased while arrays are reified, you cannot instantiate a generic array directly.

```java
T[] arr = new T[10];                   // compile error
@SuppressWarnings("unchecked")
T[] arr = (T[]) new Object[10];        // workaround: cast, unchecked warning
```

### Generic varargs and `@SafeVarargs`

A `T... args` parameter is backed by a non-reifiable `T[]`, so the compiler warns about possible heap pollution at the declaration and every call site.

```java
@SafeVarargs   // asserts no heap pollution; suppresses the warning at both ends
static <T> List<T> listOf(T... elements) {
    return Arrays.asList(elements);   // only reads the array
}
```

`@SafeVarargs` is restricted to non-overridable methods (`static`, `final`, `private`, or constructors), so a subclass cannot supply an unsafe override.

> **Note**: apply `@SafeVarargs` only when the method merely reads the varargs array. Storing its reference or writing a foreign element into it is real heap pollution that the annotation would wrongly hide.

## Q&A

1. **Why can't primitives be used as type arguments?**
Erasure replaces a type parameter with `Object` (or its bound), and primitives do not extend `Object`. Use the wrapper (`Integer`, `Double`); autoboxing bridges the rest, at the cost of boxing overhead.

2. **Why is `new T()` illegal inside a generic class?**
After erasure the runtime does not know `T`, so it cannot select a constructor or allocate the right type. Pass a `Supplier<T>` or `Class<T>` and call `factory.get()` or `clazz.getDeclaredConstructor().newInstance()`.

3. **Why can't you write `new T[10]`?**
Arrays are reified and need their component type at runtime, which erasure has discarded. Use a `List<T>`, or an unchecked `(T[]) new Object[10]` if an array is unavoidable.

4. **Why can't static members use the class type parameter `T`?**
`T` is bound per instance, but statics belong to the class and exist before any instance, so there is no `T` to refer to. A static method may declare its own: `static <E> E pick(E a, E b)`.

5. **What happens when a method declares a type parameter named like the class's?**
The method parameter shadows the class one inside that method, so `T` there is the method's local type. It compiles but is confusing; rename one.

6. **Is `List<Integer>` a subtype of `List<Number>`?**
No, generics are invariant: allowing it would let you insert a `Double` into a list that is really `List<Integer>`. Bridge with `List<? extends Number>` to read or `List<? super Integer>` to write.

7. **How does erasure affect overloading?**
Overloads differing only by type argument (`f(List<String>)` vs `f(List<Integer>)`) erase to the identical signature `f(List)`, which the compiler rejects as a duplicate.

8. **What are raw types and why avoid them?**
A raw type (`List` instead of `List<String>`) drops all generic checks and reverts to pre-generics behavior, admitting any element and risking a later `ClassCastException`. They exist only for backward compatibility.

9. **What is heap pollution?**
A parameterized reference pointing at an object of the wrong type, usually via raw types or unchecked casts. The mismatch surfaces as a `ClassCastException` far from its cause, often on a compiler-inserted cast.

## Quick Revision Cheat Sheet

| Concept             | Remember                                                                      |
| ------------------- | ----------------------------------------------------------------------------- |
| Type arguments      | Reference types only; `List<int>` invalid, use `List<Integer>`                |
| `<T extends X>`      | Upper bound; covers classes and interfaces (no `implements` keyword)          |
| `<T extends A & B>`  | Multiple bounds; class first, then interfaces                                 |
| No `super` bound     | `<T super X>` invalid; `super` is wildcard-only                               |
| `? extends T`        | Producer: read as `T`, cannot add                                            |
| `? super T`          | Consumer: add `T`, reads yield `Object`                                       |
| PECS                | Producer Extends, Consumer Super                                              |
| Wildcard vs `<T>`    | Wildcard for one flexible param; named `T` when referenced repeatedly         |
| Invariance          | `List<Integer>` is not a `List<Number>`; arrays are covariant (`ArrayStoreException`) |
| Type erasure        | Compile-time only; runtime sees raw type plus inserted casts; bound replaces `T` |
| Erasure limits      | No `new T()`/`new T[]`/`instanceof List<X>`; static can't use class `T`; no type-arg-only overloads |
| `@SafeVarargs`       | Suppresses `T...` heap-pollution warning; only static/final/private/constructor |
