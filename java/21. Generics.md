## What & Why

Generics allow you to write **type-safe, reusable** code that works with any type, while catching type errors at **compile time** instead of runtime.

```java
// Without generics, compiles but crashes at runtime
List list = new ArrayList();
list.add("hello");
list.add(123);
String s = (String) list.get(1);  // ClassCastException at runtime! 💥

// With generics, compile-time safety
List<String> list = new ArrayList<>();
list.add("hello");
list.add(123);          // COMPILE ERROR: caught early
String s = list.get(0); // no casting needed
```

### Why generics were added (the pre-Java 5 pain)

- **Collections held `Object`**: anything in, every read needed a manual cast (`(String) list.get(0)`), error-prone and verbose.
- **Type errors surfaced at runtime**: an `Integer` in a supposedly-`String` list wasn't caught until production threw `ClassCastException`.
- **No type-safe reusable code**: you either wrote `IntBox`, `StringBox`, `UserBox` separately or used `Object` and lost safety entirely. Javadoc "list of Strings" comments were hope, not enforcement.

### How generics solve it

- **Type parameters move the check to compile-time**: `List<String>` enforces only Strings in and out. Wrong type = compile error, not production crash.
- **No casts on reads**: `list.get(0)` returns `String`, the compiler knows.
- **Reusable and type-safe**: write `Box<T>` once, use as `Box<String>`, `Box<User>`, `Box<Order>` with no duplication.
- **Bounded parameters** (`<T extends Comparable<T>>`) and **wildcards** (`? extends`, `? super`) handle constraints and variance (PECS).

## Generic Class

```java
// T is a type parameter, placeholder for any type
class Box<T> {
    private T value;

    public void set(T value) { this.value = value; }
    public T get() { return value; }
}

// Usage, T becomes String
Box<String> strBox = new Box<>();
strBox.set("hello");
String s = strBox.get();  // no casting
```

**Multiple type parameters:**

```java
class Pair<K, V> {
    private final K key;
    private final V value;

    public Pair(K key, V value) { this.key = key; this.value = value; }
    public K getKey()   { return key; }
    public V getValue() { return value; }
}

Pair<String, Integer> p = new Pair<>("age", 25);
```

**Common naming conventions:** `T`, Type, `E`, Element, `K`, Key, `V`, Value, `N`, Number, `S, U`, 2nd, 3rd types

## Generic Method

A method can have its own type parameter, independent of the class.

```java
// <T> before return type declares the type parameter
public static <T> void printArray(T[] arr) {
    for (T element : arr) {
        System.out.print(element + " ");
    }
}

// Usage, compiler infers T from argument
printArray(new String[]{"A", "B", "C"});  // T = String
printArray(new Integer[]{1, 2, 3});       // T = Integer
```

**Generic method with return type:**

```java
public static <T> T getFirst(List<T> list) {
    return list.get(0);
}

String s = getFirst(List.of("a", "b"));   // returns String
Integer n = getFirst(List.of(1, 2, 3));   // returns Integer
```

## Bounded Type Parameters

Restrict what types can be used as type arguments.

### Upper bound, `extends` (accepts the type and its subclasses/implementations)

```java
// T must be Number or its subclass (Integer, Double, Float, etc.)
public static <T extends Number> double sum(List<T> list) {
    double total = 0;
    for (T n : list) {
        total += n.doubleValue();  // can call Number methods
    }
    return total;
}

sum(List.of(1, 2, 3));         // works, Integer extends Number
sum(List.of(1.5, 2.5));        // works, Double extends Number
sum(List.of("a", "b"));        // COMPILE ERROR, String doesn't extend Number
```

**Important:** In generics, `extends` means "is a subtype of", covers BOTH class inheritance AND interface implementation. There is NO `implements` keyword in generics.

```java
<T extends Comparable<T>>     // OK: correct
<T implements Comparable<T>>  // compile error: implements doesn't exist in generics
```

### Multiple bounds, `&` (must satisfy all)

```java
// T must extend Number AND implement Comparable
public static <T extends Number & Comparable<T>> T findMax(List<T> list) {
    return Collections.max(list);
}
```

Class must come first, then interfaces: `<T extends SomeClass & Interface1 & Interface2>`

**Note:** No `super` keyword for bounded type parameters, `<T super Integer>` is NOT valid. `super` only works with wildcards.

## Wildcards (`?`)

Used when you don't need to refer to the type by name, just need flexibility in what's accepted.

### `?`, Unbounded (any type)

```java
public static void printList(List<?> list) {
    for (Object o : list) {
        System.out.println(o);
    }
}

printList(List.of("a", "b"));    // works
printList(List.of(1, 2, 3));     // works
```

### `? extends T`, Upper bounded (Producer / read-friendly)

"Anything that IS-A T or subclass of T." You can read from it but cannot add to it (compiler doesn't know exact type).

```java
public static double sum(List<? extends Number> list) {
    double total = 0;
    for (Number n : list) {
        total += n.doubleValue();  // safe to read as Number
    }
    return total;
}

sum(List.of(1, 2, 3));       // List<Integer>, works
sum(List.of(1.5, 2.5));      // List<Double>, works
```

Same `extends` rule, works for interfaces: `List<? extends Serializable>`

### `? super T`, Lower bounded (Consumer / write-friendly)

"Anything that IS-A T or superclass of T." You can add T to it but reading gives you `Object`.

```java
public static void addNumbers(List<? super Integer> list) {
    list.add(1);    // safe to add Integer
    list.add(2);
}

addNumbers(new ArrayList<Integer>());  // works
addNumbers(new ArrayList<Number>());   // works, Number is super of Integer
addNumbers(new ArrayList<Object>());   // works, Object is super of Integer
addNumbers(new ArrayList<String>());   // COMPILE ERROR, String is not super of Integer
```

### Fruit Basket Analogy (extends vs super made simple)

```
Fruit (parent)
├── Apple
├── Banana
└── Orange
```

**`? extends Fruit`**: "a basket of some specific fruit, but I don't know which"

```java
List<? extends Fruit> basket = new ArrayList<Apple>(); // could be Apple/Banana/Orange basket

Fruit f = basket.get(0);    // OK: whatever is inside IS a Fruit
basket.add(new Apple());    // compile error: what if it's actually a Banana basket?
basket.add(new Banana());   // compile error: compiler blocks ALL adds to be safe
```

**`? super Fruit`**: "a basket that can HOLD Fruits (Fruit basket, or Object basket)"

```java
List<? super Fruit> basket = new ArrayList<Object>(); // could be Fruit/Object basket

basket.add(new Apple());    // OK: Apple is a Fruit, fits in any super-Fruit basket
Object o = basket.get(0);   // OK: only guaranteed to be Object
Fruit f = basket.get(0);    // compile error: what if basket has non-Fruit items?
```

### PECS, Producer Extends, Consumer Super

In plain English:

- Need to **take items OUT** of a collection? → `extends` (Producer gives you items)
- Need to **put items INTO** a collection? → `super` (Consumer accepts your items)
- Need both? → Use exact type, no wildcard

```java
// src = reading from it (producer) → extends
// dest = writing to it (consumer) → super
public static <T> void copy(List<? extends T> src, List<? super T> dest) {
    for (T item : src) {    // reading from src: OK
        dest.add(item);      // writing to dest: OK
    }
}
```

**Real-world PECS in Java API:**

```java
boolean addAll(Collection<? extends E> c);   // c is producer → extends

List<Number> nums = new ArrayList<>();
nums.addAll(List.of(1, 2));        // List<Integer>, Integer extends Number
nums.addAll(List.of(1.5, 2.5));    // List<Double>, Double  extends Number
```

|---|---|
|`<T extends Number>`|You need to **refer to T** in multiple places (params, return, body)|
|`<? extends Number>`|You just need **flexibility in one parameter**, don't need to name the type|

## Type Erasure (How Generics Work Internally)

Generics exist only at **compile time**. At runtime, all type info is **erased**: replaced with `Object` (or the bound if bounded).

```java
// What you write:
List<String> list = new ArrayList<>();

// What JVM sees after erasure:
List list = new ArrayList();

// With bounds, replaced with the bound, not Object
class Box<T extends Number> {
    private T value;
}
// After erasure:
class Box {
    private Number value;  // T replaced with Number, not Object
}
```

**Consequences of type erasure**: No `instanceof List<String>` (only `List<?>` allowed); no `new T()`, no `new T[]`; no primitives as type args; `static` can't use class `T`
- All parameterizations of a generic class share one runtime class:

```java
List<String> strList = new ArrayList<>();
List<Integer> intList = new ArrayList<>();
strList.getClass() == intList.getClass();  // true, both are just ArrayList
```

## Arrays vs Generics (Covariance vs Invariance)

One of the biggest differences between arrays and generics is how they handle type hierarchies (variance).

* **Arrays are Covariant**: If `S` is a subtype of `T`, then `S[]` is a subtype of `T[]` (e.g., `String[]` is a subtype of `Object[]`). This compiles successfully but carries a runtime safety risk.
* **Generics are Invariant**: `List<String>` is **not** a subtype of `List<Object>`. Type mismatch errors are caught strictly at compile time.

```java
// Array Covariance Danger
Object[] arr = new String[5];   // Compiles successfully
arr[0] = 123;                   // Throws ArrayStoreException at runtime!

// Generic Invariance Safety
List<Object> list = new ArrayList<String>(); // Compile error! Caught immediately
```

### Generic Array Creation Restriction
Because arrays are **reified** (they enforce their component type at runtime) while generics are **erased** (type parameters are discarded at runtime), you cannot create generic arrays directly:

```java
T[] arr = new T[10];           // Compile error!

// Workaround (requires casting; produces an unchecked warning)
@SuppressWarnings("unchecked")
T[] arr = (T[]) new Object[10];
```

### Generic Varargs and `@SafeVarargs`

A generic varargs parameter (`T... args`) silently creates a backing `T[]` array. Because that array is non-reifiable, the compiler warns about possible **heap pollution** at every declaration and call site.

```java
// Compiler warning: "Possible heap pollution from parameterized vararg type"
static <T> List<T> of(T... elements) {
    return Arrays.asList(elements);   // safe: only reads, never stores a foreign type
}
```

`@SafeVarargs` asserts the method does not pollute the heap, suppressing the warning at both ends. It is restricted to non-overridable methods (`static`, `final`, `private`, or constructors), so a subclass cannot supply an unsafe override.

```java
@SafeVarargs
static <T> List<T> of(T... elements) {
    return Arrays.asList(elements);
}
```

> **Note**: only apply `@SafeVarargs` when the method merely reads the varargs array. Storing the array reference or writing a foreign type into it is genuine heap pollution that the annotation would wrongly hide.

## Generic Interface

```java
// Define
interface Transformer<T, R> {
    R transform(T input);
}

// Implement with concrete types
class StringToInteger implements Transformer<String, Integer> {
    public Integer transform(String input) {
        return Integer.parseInt(input);
    }
}
```

## Generic Constructor

A constructor can have its own type parameter even if the class is not generic.

```java
class Util {
    // <T> is the constructor's own type parameter
    <T> Util(T item) {
        System.out.println(item.getClass().getName());
    }
}

new Util("hello");   // T = String
new Util(42);        // T = Integer
```

## Q&A

1. **Why can't you use primitive types like `int` as a type parameter in Generics?**
   Generics in Java were implemented using "Type Erasure" to ensure backward compatibility. At runtime, the type parameter is erased to `Object` (or its upper bound). Since primitives do not inherit from `Object`, they cannot be used. You must use the corresponding wrapper class (e.g., `Integer`), and Java's autoboxing will handle the conversion.

2. **Why is it illegal to write `new T()` inside a generic class?**
   Because of type erasure, the JVM does not know what `T` is at runtime. The type information is only available to the compiler. Since the actual type is unknown at runtime, the JVM cannot allocate memory or call a constructor for it.

3. **Can you create an array of a generic type, such as `new T[10]`?**
   No. Arrays in Java are "reified," meaning they carry their type information at runtime, whereas Generics are not. Creating a generic array would violate type safety. If you need a collection of `T`, you should use a `List<T>`. If you absolutely must have an array, you can create an `Object[]` and cast it to `T[]`, but this will generate an unchecked warning.

4. **Why can't static fields or static methods use the class-level type parameter `T`?**
   Static members belong to the class itself, not to any specific instance. Since the type parameter `T` is determined when an instance is created, it doesn't exist in a static context. However, a static method can define its **own** independent type parameter (e.g., `static <E> E process(E item)`).

5. **What happens if a method defines a type parameter with the same name as the class type parameter?**
   The method's type parameter will **shadow** the class's type parameter. Within that method, `T` refers to the method's local type, not the class's type. This is highly confusing and should be avoided.

6. **Is `List<Integer>` a subtype of `List<Number>`?**
   No. This is the concept of **Invariance**. While an `Integer` is a `Number`, a `List` of Integers is not a `List` of Numbers. If it were allowed, you could add a `Double` into a `List<Number>` that was actually a `List<Integer>`, breaking type safety. To allow this relationship safely, you must use wildcards: `List<? extends Number>`.

7. **How does type erasure affect method overloading?**
   Type erasure can cause "Erasure Ambiguity." For example, if you try to overload a method with `process(List<String>)` and `process(List<Integer>)`, both will be erased to `process(List)` at runtime. The compiler will reject this because they would have the same signature after erasure.

8. **What are "Raw Types," and why should they be avoided?**
   A raw type is a generic class used without a type parameter (e.g., `List list = new ArrayList();`). Using raw types bypasses the compiler's type checks, making the code equivalent to pre-Generics Java. It is dangerous because it allows you to insert any object type, leading to potential `ClassCastException` errors at runtime.

9. **What is "Heap Pollution"?**
   Heap pollution occurs when a variable of a parameterized type refers to an object that is not of that type. This usually happens when mixing raw types with generic types or using unchecked casts. The result is often a `ClassCastException` that happens at a completely different part of the code than where the mistake was made, making it difficult to debug.

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Generic class|`class Box<T>`, T is a placeholder for any reference type|
|Generic method|`public static <T> ...`, `<T>` before return type|
|Type parameters|Only reference types. `List<int>` invalid, use `List<Integer>`|
|`<T extends X>`|Upper bound. Works for classes AND interfaces (no `implements` keyword)|
|`<T extends A & B>`|Multiple bounds. Class first, then interfaces|
|No `super` bound|`<T super Integer>` invalid. `super` only works with wildcards|
|`?`|Unbounded wildcard, accept any type|
|`? extends T`|Producer, read from it. Can't add|
|`? super T`|Consumer, add T. Reads give Object|
|PECS|Producer Extends, Consumer Super|
|Invariance|`List<Integer>` is NOT a `List<Number>`, use wildcards to bridge|
|Type erasure|Generics are compile-time only. Runtime sees raw types|
|Erasure consequences|No `new T()`, no `new T[]`, no `instanceof List<X>`, static can't use class T|
|`@SafeVarargs`|Suppresses heap-pollution warning on `T...`. Only on static/final/private/constructor|
