Java networking is communication between processes on separate machines over the TCP/IP stack. The `java.net` package (Java 1.0+) supplies the low-level socket primitives; `java.net.http.HttpClient` (Java 11+) is the modern HTTP API.

In production, raw sockets are rare, most code sits on Spring (`RestTemplate`/`WebClient`), OkHttp, or `HttpClient`. Sockets still underpin all of it and remain standard interview material.

---

## TCP vs UDP

| Protocol | Reliability               | Ordering  | Typical use                | Java API                          |
| -------- | ------------------------- | --------- | -------------------------- | --------------------------------- |
| **TCP**  | Guaranteed delivery       | Ordered   | HTTP, DB, RPC, most apps   | `Socket` / `ServerSocket`         |
| **UDP**  | Best-effort, may drop/dup | Unordered | Streaming, DNS, VoIP, games | `DatagramSocket` / `DatagramPacket` |

TCP pays for reliability with a 3-way handshake, congestion control, and retransmission; UDP skips all of it, so the application must add any reliability it needs.

## InetAddress

Represents an IP address (v4 or v6) and performs DNS resolution. Instances are immutable.

```java
InetAddress local  = InetAddress.getLocalHost();            // this machine
InetAddress google = InetAddress.getByName("google.com");   // DNS lookup
InetAddress[] all  = InetAddress.getAllByName("google.com"); // every resolved IP

local.getHostName();      // "my-laptop"
local.getHostAddress();   // "192.168.1.5"
local.isReachable(2000);  // ICMP/echo probe, 2s timeout (often false without root)
```

> **Note**: the JVM caches DNS results. Without a security manager the default positive TTL is short (~30s), but it is tunable via `networkaddress.cache.ttl`. A stale cache can keep pointing at a decommissioned IP in autoscaled environments.

## TCP Sockets

`ServerSocket.accept()` blocks until a client connects and returns a `Socket` bound to that connection. Both ends read/write through ordinary I/O streams.

### Server

```java
try (ServerSocket server = new ServerSocket(8080)) {
    while (true) {
        Socket client = server.accept();              // blocks until a client connects
        new Thread(() -> handle(client)).start();     // one thread per connection
    }
}

void handle(Socket client) {
    try (var in  = new BufferedReader(new InputStreamReader(client.getInputStream()));
         var out = new PrintWriter(client.getOutputStream(), true)) {   // true = autoflush
        out.println("Echo: " + in.readLine());
    } catch (IOException e) { e.printStackTrace(); }
}
```

### Client

```java
try (Socket socket = new Socket("localhost", 8080);
     var out = new PrintWriter(socket.getOutputStream(), true);
     var in  = new BufferedReader(new InputStreamReader(socket.getInputStream()))) {
    out.println("Hello");
    System.out.println(in.readLine());   // "Echo: Hello"
}
```

- All sockets implement `AutoCloseable`, always wrap them in try-with-resources.
- `setSoTimeout(ms)` bounds blocking **reads** only. The `new Socket(host, port)` constructor has **no** connect timeout and can hang on an unresponsive host; bound it with `new Socket()` then `connect(new InetSocketAddress(host, port), ms)`.
- One thread per client does not scale. Use a thread pool, NIO `Selector`, or virtual threads for high connection counts.

> **Reference**: virtual threads make the simple blocking-socket-per-connection model cheap at scale, see [Java Modern Features (10-21+)](<29. Java Modern Features.md>).

## UDP Sockets

No connection and no guarantees, each `DatagramPacket` is an independent, self-addressed message.

### Sender

```java
try (DatagramSocket socket = new DatagramSocket()) {
    byte[] data = "Hello UDP".getBytes();
    InetAddress addr = InetAddress.getByName("localhost");
    socket.send(new DatagramPacket(data, data.length, addr, 9090));
}
```

### Receiver

```java
try (DatagramSocket socket = new DatagramSocket(9090)) {
    byte[] buf = new byte[1024];
    DatagramPacket packet = new DatagramPacket(buf, buf.length);
    socket.receive(packet);   // blocks until a packet arrives
    String msg = new String(packet.getData(), 0, packet.getLength());
    System.out.println("From " + packet.getAddress() + ": " + msg);
}
```

> **Note**: a single datagram is capped by the path MTU (typically 1500 bytes; ~1472 usable after IP+UDP headers). Larger payloads fragment at the IP layer, and one lost fragment drops the whole datagram. Chunk it yourself or switch to TCP.

## HTTP Clients

### HttpClient (Java 11+)

Built into the JDK, no external dependency. Fluent, immutable, HTTP/2 by default with negotiation down to HTTP/1.1, and supports both sync and async calls.

```java
HttpClient client = HttpClient.newBuilder()
    .connectTimeout(Duration.ofSeconds(5))
    .build();

HttpRequest req = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/users"))
    .header("Accept", "application/json")
    .GET()
    .build();

// Synchronous
HttpResponse<String> resp = client.send(req, HttpResponse.BodyHandlers.ofString());
System.out.println(resp.statusCode() + " " + resp.body());

// Asynchronous
client.sendAsync(req, HttpResponse.BodyHandlers.ofString())
      .thenApply(HttpResponse::body)
      .thenAccept(System.out::println);

// POST with a JSON body
HttpRequest post = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/users"))
    .header("Content-Type", "application/json")
    .POST(HttpRequest.BodyPublishers.ofString("{\"name\":\"Alice\"}"))
    .build();
```

### HttpURLConnection (legacy)

Pre-Java-11 HTTP client, still found in old code. Verbose and easy to leak.

```java
URL url = URI.create("https://api.example.com/users").toURL();  // new URL(String) deprecated (Java 20)
HttpURLConnection conn = (HttpURLConnection) url.openConnection();
conn.setRequestMethod("GET");
int status = conn.getResponseCode();   // 200
```

> **Note**: on a non-2xx response the body arrives on `getErrorStream()`, not `getInputStream()`. Failing to read and close the error stream leaks the underlying socket and eventually exhausts the connection pool.

**Picking a client**:

- Modern plain Java: `HttpClient`.
- Spring app: `RestTemplate` (sync, maintenance mode), `RestClient` (sync, fluent), or `WebClient` (reactive).
- Legacy code with no dependencies to add: `HttpURLConnection`.

## Q&A

1. **How do you serve many simultaneous clients given that `accept()` blocks?**
`accept()` blocks one thread per call, so you dispatch each returned `Socket` to its own worker: a thread, a pooled `ExecutorService` task, an NIO `Selector` loop, or a virtual thread (Java 21). Thread-per-client is simplest but caps out on OS threads; a pool or virtual threads scale far higher.

2. **Why can `new Socket(host, port)` hang, and how do you bound it?**
That constructor connects immediately with no connect timeout, so an unreachable host blocks until the OS gives up (often tens of seconds). Use `new Socket()` then `connect(new InetSocketAddress(host, port), ms)`; note `setSoTimeout` only bounds reads, not the connect.

3. **What is the classic resource leak with `HttpURLConnection`?**
On a non-2xx response the body is exposed via `getErrorStream()`, and the connection's socket is not released until that stream is fully read and closed. Ignoring the error stream leaks sockets and eventually blocks all new connections.

4. **How can JVM DNS caching break apps in dynamic environments?**
The JVM caches resolved IPs (TTL via `networkaddress.cache.ttl`), so when a load balancer or autoscaler reassigns a hostname's IP, the app may keep dialing the dead address until the cache expires. Tuning the TTL down is the fix for high-availability deployments.

5. **Why might binding to port 80 fail for a normal user?**
On Unix-like systems, ports below 1024 are privileged and require root/admin to bind. It is a security measure that stops unprivileged users from impersonating well-known services like HTTP or SSH.

6. **How do you send a UDP payload larger than the MTU?**
A datagram over ~1472 bytes fragments at the IP layer, and losing any one fragment discards the entire datagram. Either split the data into application-level chunks with your own sequencing, or use TCP, which handles segmentation transparently.

## Quick Revision Cheat Sheet

| Concept                | Remember                                                              |
| ---------------------- | -------------------------------------------------------------------- |
| TCP vs UDP             | TCP reliable + ordered; UDP best-effort, lightweight, app adds safety |
| InetAddress            | IP + DNS resolution; immutable; results cached per `cache.ttl`        |
| accept()               | Blocks per call; dispatch each Socket to a thread/pool/NIO/vthread    |
| Connect vs read timeout | `new Socket(h,p)` has no connect timeout; `setSoTimeout` = reads only |
| Sockets                | All `AutoCloseable`, always try-with-resources                        |
| UDP packet             | Independent datagram; ~1472 B usable, then IP fragmentation           |
| HttpURLConnection      | Legacy; read+close `getErrorStream()` on non-2xx or leak sockets      |
| HttpClient (Java 11+)  | Fluent, HTTP/2, `send` (sync) / `sendAsync` (async)                   |
| new URL(String)        | Deprecated (Java 20); use `URI.create(s).toURL()`                     |
| DNS TTL                | Stale cache dials dead IPs in autoscaled/LB environments              |
| Privileged ports       | Ports < 1024 need root/admin to bind                                  |
| In Spring              | `RestClient`/`WebClient`/`RestTemplate`, not raw sockets              |
