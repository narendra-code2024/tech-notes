**Inheritance** lets one class acquire the fields and methods of another, reusing its behavior and adding or overriding its own. The child (subclass) IS-A parent (superclass), a `Dog` is an `Animal`. Java declares it with `extends` and allows only one parent (single inheritance).

```java
class Animal {
    String name;
    void eat() { System.out.println(name + " is eating"); }
}

class Dog extends Animal {
    void bark() { System.out.println(name + " is barking"); }
}

Dog d = new Dog();
d.name = "Buddy";    // inherited field
d.eat();              // inherited method
d.bark();             // own method
```

- **What is inherited**:
    - public and protected members, always
    - default (package-private) members, only if same package
    - static members, inherited but resolved at compile-time (method hiding, not overriding)
    - **NOT** inherited: private members (exist in object memory but not accessible), constructors
- **What a child can do**: use inherited members directly, override inherited methods, add new fields and methods, or call the parent constructor via `super()`

Every class implicitly extends `Object`, the root of all classes, so `toString()`, `equals()`, and `hashCode()` are available on every object.

## `super` Keyword

Refers to the immediate parent. Used two ways: call a parent constructor with `super(...)`, or reach a parent member that the child has overridden or hidden with `super.x`.

### Call parent constructor

```java
class Animal {
    String name;
    Animal(String name) { this.name = name; }   // only a parameterized constructor
}

class Dog extends Animal {
    String breed;
    Dog(String name, String breed) {
        super(name);            // calls Animal(String)
        this.breed = breed;
    }
    // Dog() { }                // compile error: no Animal() for the implicit super()
}
```

- **Constructor chaining rules**:
    - `super()` is auto-inserted (calling the parent no-arg constructor) if you don't write it
    - If the parent has no no-arg constructor, the child MUST explicitly call `super(args)`
    - `super()` or `this()` must be the first statement; both can't appear

The parent is **always constructed first**: each constructor runs its `super(...)` call before its own body, so the chain climbs `Dog → Animal → Object`.

### Access parent members

```java
class Animal {
    String type = "Animal";
    void sound() { System.out.println("Some sound"); }
}

class Dog extends Animal {
    String type = "Dog";        // hides parent field (not override, fields don't override)

    void sound() {
        super.sound();          // calls parent's sound()
        System.out.println("Bark");
    }

    void printType() {
        System.out.println(this.type);    // "Dog"
        System.out.println(super.type);   // "Animal"
    }
}
```

## final in Inheritance

The `final` keyword restricts subclassing and overriding in an inheritance hierarchy.

- **final Classes**: Prevents class inheritance. No class can extend a final class (e.g., `java.lang.String` or wrapper classes).
- **final Methods**: Prevents method overriding in subclasses.

```java
final class Parent { }
// class Child extends Parent { } // compile error: cannot inherit from final Parent

class Base {
    final void show() { }
}
class Derived extends Base {
    // void show() { } // compile error: cannot override final method from Base
}
```

> **Note**: Defining a class as `final` implicitly makes all of its methods `final` as well, since the class cannot be subclassed.

## Types of Inheritance

```java
// Single, A → B
class Animal { }
class Dog extends Animal { }

// Multilevel, A → B → C
class Animal { }
class Dog extends Animal { }
class Puppy extends Dog { }

// Hierarchical, A → B, A → C
class Animal { }
class Dog extends Animal { }
class Cat extends Animal { }

// Multiple (NOT supported with classes, supported with interfaces)
class A extends B, C { }     // compile error
class A implements X, Y { }  // compiles (interfaces)

// Hybrid, combination of above (possible only through interfaces)
```

**Why Java doesn't support multiple inheritance (with classes)?**

The **diamond problem**: `D` inherits from both `B` and `C`, and if each overrides the same `A` method, the compiler can't resolve which version `D` should use.

```
     A
    / \
   B   C
    \ /
     D       ← which method()?
```

## Generalization and Specialization

```
        Vehicle            ← Generalization (moving UP, abstract common traits)
       /       \
    Car       Bike          ← Specialization (moving DOWN, adding specific traits)
   /    \
Sedan   SUV
```

- **Generalization**: extracting common behavior into a parent class (Car and Bike both move → Vehicle)
- **Specialization**: adding specific behavior in child class (SUV has offRoad(), Sedan doesn't)

## HAS-A

One object holds another as a field, a "has-a" relationship rather than the "is-a" of inheritance.

```java
class Engine { }
class Car {
    Engine engine;      // Car HAS-A Engine
}
```

Object relationships, loosest to tightest. Aggregation and composition are the whole-part HAS-A forms; both are specializations of association, which on its own is a looser "uses-a":

- **Association**: one object uses or references another; neither owns the other and each exists on its own (a `Doctor` treats `Patient`s)
- **Aggregation**: weak HAS-A, parts exist on their own and can be shared; deleting the whole leaves them intact (a `Playlist` has `Song`s, shared across playlists)
- **Composition**: strong HAS-A, the whole creates and exclusively owns its parts; they can't be shared or exist without it (an `Order` owns its `OrderItem`s)

```java
// Aggregation, Song exists on its own and is shared across playlists
class Playlist {
    List<Song> songs;    // holds references to songs created elsewhere
}

// Composition, OrderItem can't exist without its Order
class Order {
    private List<OrderItem> items;

    Order() {
        items = new ArrayList<>();
        items.add(new OrderItem());     // created inside, lifecycle tied to Order
    }
}

class OrderItem { }     // meaningless without an Order
```

> **Note**: prefer composition over inheritance, it is more flexible, avoids tight coupling, and easier to change.

## Q&A

1. **What is "field hiding," and how does it differ from method overriding?**
   Unlike methods, fields are not polymorphic. If a child class defines a field with the same name as a field in the parent, the child's field **hides** the parent's field rather than overriding it. Access to a field is determined by the **reference type** at compile-time, while access to an overridden method is determined by the **actual object type** at runtime.

   ```java
   class Parent { int x = 10; }
   class Child extends Parent { int x = 20; }

   Child c = new Child();
   Parent p = c;
   System.out.println(c.x);    // 20, Child's x
   System.out.println(p.x);    // 10, Parent's x (determined by reference type 'Parent')
   ```

2. **What is the result of `null instanceof SomeClass`?**
   The result is always `false`. The `instanceof` operator checks if an object is an instance of a type; since `null` is not an instance of any object, it is not an instance of any class.

3. **What is the difference between upcasting and downcasting in an inheritance hierarchy?**
   **Upcasting** is casting from a child type to a parent type (e.g., `Animal a = new Dog();`). It is implicit and always safe because a Dog "is-an" Animal. **Downcasting** is casting from a parent type to a child type (e.g., `Dog d = (Dog) a;`). It must be explicit and can throw a `ClassCastException` at runtime if the actual object is not of the target type.

4. **If a parent constructor calls an overridable method, which version runs during construction?**
   The subclass override runs, but on a half-built object: the subclass's fields are still at their defaults (`null`/`0`) because its constructor body hasn't executed yet. This makes calling overridable methods from a constructor a known source of bugs.

   ```java
   class Animal {
       Animal() { sound(); }                 // virtual call during construction
       void sound() { }
   }
   class Dog extends Animal {
       String name = "Buddy";
       void sound() { System.out.println(name); }   // prints null, not "Buddy"
   }
   new Dog();    // output: null, name not yet initialized
   ```

5. **What is a "covariant return type" in the context of method overriding?**
   Since Java 5, an overriding method is allowed to return a **subtype** of the return type declared in the parent's method. This makes the API more specific and easier to use for the caller.

   ```java
   class Animal {
       Animal create() { return new Animal(); }
   }
   class Dog extends Animal {
       Dog create() { return new Dog(); }    // OK: Dog is a subtype of Animal
   }
   ```

6. **What is the "Fragile Base Class" problem?**
   This refers to the architectural risk where seemingly safe changes to a parent class can unintentionally break the behavior of its subclasses. This tight coupling is one of the primary reasons why senior developers often **prefer composition over inheritance**.

## Quick Revision Cheat Sheet

| Concept           | Remember                                                                  |
| ----------------- | ------------------------------------------------------------------------- |
| `extends`         | Single inheritance only. Every class extends Object implicitly             |
| `super()`         | Calls parent constructor. Must be first line. Added implicitly if missing  |
| Inherited         | public, protected members. NOT private, NOT constructors                   |
| Fields            | Don't override, they hide. Access based on reference type, not object type |
| Diamond problem   | Why no multiple inheritance with classes. Interfaces resolve it            |
| IS-A              | Inheritance (`extends`, `implements`). Tested with `instanceof`            |
| HAS-A             | Field reference (composition/aggregation). Prefer over inheritance         |
| Aggregation       | Weak HAS-A. Parts exist on their own, shared (Playlist-Song)                |
| Composition       | Strong HAS-A. Parts owned exclusively, can't exist alone (Order-OrderItem)  |
| Casting           | Upcast (Child → Parent) implicit, safe. Downcast (Parent → Child) explicit, check `instanceof` |
| Covariant return  | Override can return subtype of parent's return type                        |
| Constructor chain | Always goes up: Child → Parent → ... → Object                              |
