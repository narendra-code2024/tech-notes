Object-Oriented Programming organizes code around **objects** (data + behavior) rather than functions and logic.

| Concept        | What it is                                                          |
| -------------- | ------------------------------------------------------------------ |
| **Class**      | Blueprint/template for creating objects. Defines fields and methods |
| **Object**     | Instance of a class. Has state (fields) and behavior (methods)      |
| **Attributes** | Fields/variables that hold object's state (name, age, salary)       |
| **Methods**    | Functions that define object's behavior (getName, calculateSalary)  |

```java
// Class = blueprint
class Employee {
    String name;          // attribute
    int salary;           // attribute

    void work() {         // method (behavior)
        System.out.println(name + " is working");
    }
}

// Object = instance
Employee emp = new Employee();   // creating object from blueprint
emp.name = "Alice";              // setting state
emp.work();                      // calling behavior
```

- A class is a **template**: no memory is allocated until an object is created with `new`.
- A class can hold fields, methods, constructors, blocks, and inner classes (each covered below).

## Object Creation

Each object is an instance with its **own copy** of instance variables; the variable holds a reference to it, not the object itself.

```java
Car myCar = new Car();    // object on Heap, reference stored on Stack
myCar.brand = "Toyota";   // accessing field
myCar.accelerate();       // calling method

Car alias = myCar;        // copies the REFERENCE, not the object (both point to the same Car)
```

**What `new Car()` does**:
1. Allocates Heap memory for the object
2. Sets fields to defaults (`0`/`null`/`false`)
3. Calls `super()`, parent constructor runs first
4. Runs instance blocks, in order of appearance
5. Runs the constructor body, sets actual values
6. Returns the reference to the Stack variable

> **Note**: Besides `new`, objects can also be created via reflection, cloning, deserialization, and factory methods (`Integer.valueOf()`, `List.of()`).

## Attributes

Attributes (also called **fields**) hold an object's state. Each instance gets its **own copy** of an instance field, while a `static` field is shared across all instances.

```java
class Employee {
    static int count = 0;     // STATIC: one copy shared across all objects
    String name;              // INSTANCE: one copy per object

    Employee(String n) {
        name = n;
        count++;              // increments the single shared counter
    }
}

Employee a = new Employee("Alice");
Employee b = new Employee("Bob");
System.out.println(Employee.count);   // 2, shared across instances, not per-object
```

Fields you leave uninitialized take their type's default (`0`/`0.0`/`false`/`null`); local variables get none.

### Compile-Time vs Runtime Constants

`static final` fields can be resolved either at compile time or runtime.

- **Compile-Time Constants**: Primitives or string literals initialized directly in their declaration. The compiler inlines these values directly into referencing bytecode.
- **Runtime Constants**: Computed values initialized via method calls or constructor execution, requiring static field lookup at runtime.

```java
class Config {
    static final String NAME = "App"; // compile-time constant (inlined)
    static final int PORT = getPort(); // runtime constant (requires field lookup)

    private static int getPort() {
        return 8080;
    }
}
```

> **Note**: Modifying a `public static final` compile-time constant requires recompiling all dependent classes; otherwise, they will continue to use the stale, inlined value.

## Methods

A method is a named block defining behavior, with a return type, parameter list, and body. Its **signature** is the method name plus parameter types (return type excluded), which is what distinguishes overloaded methods.

```java
int    add(int a, int b)   // signature: add(int, int)
double add(int a, int b)   // compile error: same signature, can't overload by return type alone
```

### Instance vs Static Methods

```java
class Calculator {
    int memory = 0;                    // instance variable

    // Instance method, needs an object, can access instance + static members
    void addToMemory(int val) {
        memory += val;
    }

    // Static method, no object needed, accesses only static members
    // Resolved at compile-time, can be hidden in a child class, no runtime polymorphism
    static int add(int a, int b) {
        return a + b;
    }
}

Calculator calc = new Calculator();
calc.addToMemory(5);           // instance method, called on the object
Calculator.add(3, 4);          // static method, called on the class
```

|            | Instance Method           | Static Method                   |
| ---------- | ------------------------- | ------------------------------- |
| Belongs to | Object                    | Class                           |
| Access     | Instance + static members | Static members only (no `this`) |
| Call       | `obj.method()`            | `ClassName.method()`            |
| `this`     | Available                 | Not available                   |
| Use for    | Object-specific behavior  | Utility/helper functions        |

## Blocks

Blocks are bare `{ }` bodies the JVM runs automatically for initialization: **instance blocks** on every object creation, **static blocks** once at class load. They differ from methods in several ways:

|            | Method           | Block                                                 |
| ---------- | ---------------- | ----------------------------------------------------- |
| Name       | Has a name       | No name                                               |
| Called     | Explicitly       | Automatically                                         |
| Parameters | Can have         | Cannot                                                |
| Return     | Can return value | Cannot                                                |
| When       | When called      | Instance: each object creation. Static: class loading |

### Instance Block

Runs every time an object is created, **before the constructor**. Useful for initialization shared across multiple constructors.

```java
class Employee {
    String name;

    // Instance block, shared initialization logic
    {
        System.out.println("Instance block");
        name = "Default";
    }

    Employee(String name) { 
	    System.out.println("Constructor");
	    this.name = name;
    }
}

new Employee("Narendra");
// Output:
// Instance block
// Constructor
```

### Static Block

Runs once when the class is loaded, before any object is created. Common for one-time setup: driver loading, config, cache warm-up.

```java
class Config {
    static String dbUrl;

    static {                                  // runs once, at class load
        dbUrl = "jdbc:mysql://localhost:3306/db";
        System.out.println("Static block");
    }
}
```

### Execution Order

```java
class Demo {
    static { System.out.println("1. Static block"); }
    { System.out.println("2. Instance block"); }
    Demo() { System.out.println("3. Constructor"); }
}

Demo d1 = new Demo();
// Output: 1. Static block → 2. Instance block → 3. Constructor

Demo d2 = new Demo();
// Output: 2. Instance block → 3. Constructor
// (Static block runs only ONCE, already loaded)
```

## Constructors

A **constructor** runs **once when an object is created** (via `new`) to initialize its state: set fields, validate inputs, acquire resources. Key rules:
- Same name as the class; **no return type**, not even `void` (adding one makes it a regular method).
- Called automatically by `new`, never invoked explicitly.
- Can be **overloaded** (multiple constructors with different parameter lists).
- If you declare no constructor, the compiler adds a no-arg default that just calls `super()`. Declaring any constructor removes that default.

```java
class Employee {
    String name;
    int salary;

    Employee(String name, int salary) {   // once defined, no default constructor exists
        this.name = name;
        this.salary = salary;
    }
}

Employee emp = new Employee("Alice", 50000);
new Employee();                            // compile error: no no-arg constructor
```

### Constructor Chaining

```java
class Employee {
    String name;
    int salary;

    Employee()                        { this("Unknown", 0); }   // this(): another constructor, must be first line
    Employee(String name)             { this(name, 0); }
    Employee(String name, int salary) { this.name = name; this.salary = salary; }
}

class Manager extends Employee {
    Manager(String name) { super(name, 80000); }                // super(): parent constructor, must be first line
}
```

`this()` and `super()` are mutually exclusive on a constructor's first line.

> **Reference**: the full parent-constructor chain and implicit `super()` → [14. Inheritance & Relationships](<14. Inheritance & Relationships.md#`super` Keyword>).

### Constructor vs Method

|                      | Constructor                   | Method                          |
| -------------------- | ----------------------------- | ------------------------------- |
| Name                 | Same as class                 | Any name                        |
| Return type          | None (not even `void`)        | Required                        |
| Called               | Automatically on `new`        | Explicitly, any number of times |
| `this()` / `super()` | Allowed (first line)          | Not allowed                     |
| Purpose              | Initialize new object's state | Define behavior                 |

> **Note**: Java has no static constructor; use a `static` block for one-time class-level setup.

## `this` Keyword

`this` refers to the **current object**, the instance on which the method or constructor was invoked.

- **Disambiguation**: Resolves name clashes between instance fields and parameters of the same name.
- **Constructor Chaining**: Invokes another constructor in the same class (must be the first line using `this(...)`).
- **Method Chaining**: Returns `this` from instance methods to enable fluent API builder patterns.
- **Reference Passing**: Passes the current instance to external methods to register or delegate actions.

```java
class Registry {
    static void add(Object obj) {}
}

class Employee {
    String name;

    Employee() {
        this("Unknown"); // constructor chaining
    }

    Employee(String name) {
        this.name = name; // field disambiguation
    }

    Employee set(String name) {
        this.name = name;
        return this; // method chaining
    }

    void register() {
        Registry.add(this); // reference passing
    }
}
```

> **Note**: Leaking `this` inside a constructor (constructor escape), such as registering the instance in a registry or starting a thread from the constructor body, exposes a partially initialized object to other threads and is a dangerous concurrency anti-pattern.

## Q&A

1. **Why can adding a constructor to a parent class break its existing subclasses?**
Defining any constructor removes the compiler-generated no-arg default. A subclass constructor that relied on the implicit `super()` then fails to compile; you must add an explicit `super(args)` or restore the parent's no-arg constructor.

2. **Can you read a field inside a constructor before it has been assigned?**
Yes, but it holds its type's default (`null`, `0`, `false`). The `this` reference is live from the moment the constructor starts, but the body's field assignments haven't run yet.

3. **Why can't a `static` method access `this` or instance variables?**
A static method belongs to the class, not an instance, so there is no "current object" for `this` to point to. To touch instance state it must receive or create an object explicitly.

4. **Does Java pass objects by reference or by value?**
Strictly **pass-by-value**: the value passed is a copy of the reference. A method can mutate the object that reference points to, but reassigning the parameter never changes which object the caller's variable points to.

```java
void modify(Employee e) {
    e.name = "Changed";        // OK: mutates the original object
}
void reassign(Employee e) {
    e = new Employee("New");   // invalid: only the local copy of the reference changes
}
```

5. **Why does changing a `public static final` primitive constant require recompiling all referencing classes?**
The compiler inlines compile-time constants (primitives and String literals) directly into the referencing bytecode. Dependent classes will continue to use the stale, inlined value unless recompiled.

6. **Why does the JVM permit invoking static methods on a null object reference without throwing a NullPointerException?**
Static methods belong to the class, not the instance. The compiler resolves the call at compile time using the declared reference type, generating an `invokestatic` instruction that bypasses runtime instance checks.

7. **When does the JVM execute static blocks, and how is their single execution guaranteed?**
Static blocks execute when the JVM loads and initializes the class on first active use. The JVM guarantees single, thread-safe execution using internal class-loader synchronization locks.

## Quick Revision Cheat Sheet

| Concept              | Remember                                                                                      |
| :------------------- | :-------------------------------------------------------------------------------------------- |
| `new`                | Heap alloc → field defaults → `super()` → instance blocks → constructor → return reference.    |
| Instance vs static   | Instance = per-object, has `this`. Static = one copy on class, no `this` (fields shared).     |
| Init order           | Static (once, class load) → instance blocks/fields → constructor (on `new`).                  |
| Constructor          | Same name as class, no return type, runs once on `new` to initialize state.                   |
| Default constructor  | Compiler-generated no-arg constructor; added only if you define no constructors.              |
| `this()` / `super()` | Call another / parent constructor; must be first line; mutually exclusive.                    |
| Pass-by-value        | Java copies reference address; can mutate object contents but cannot reassign caller's ref.   |
| Constant inlining    | Primitive/String constants inlined at compile time; updates require recompiling dependents.   |
| Constructor escape   | Passing `this` from a constructor leaks an uninitialized object, causing concurrency issues.  |
| Static dispatch      | Resolved at compile time; invoking static methods on a null reference does not throw NPE.     |
