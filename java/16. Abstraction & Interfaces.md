Abstraction is the process of hiding implementation details and exposing only essential functionality to the user.

A real-world analogy is driving a car: the driver works the steering wheel and pedals without understanding the engine internals. Java implements this through **abstract classes** and **interfaces**.

---

## Abstract Class

An abstract class is a class that cannot be directly instantiated and is designed to define abstract or concrete template behaviors for its subclasses.

```java
abstract class Shape {
    String color;

    // Abstract method, no body, child MUST implement
    abstract double area();

    // Concrete method, has a body, child inherits as-is (can override)
    void display() {
        System.out.println("Color: " + color + ", Area: " + area());
    }
}

class Circle extends Shape {
    double radius;

    Circle(double radius) { this.radius = radius; }

    @Override
    double area() { return Math.PI * radius * radius; }   // must implement
}

// Shape s = new Shape();    // compile error: can't instantiate an abstract class
Shape s = new Circle(5);     // OK: usable as a reference type
s.area();                    // calls Circle's implementation
s.display();                 // inherited concrete method
```

Abstract classes follow these rules:

- Cannot be instantiated directly, but are valid as polymorphic reference types.
- Concrete subclasses must implement all abstract methods.
- Support constructors (invoked via `super()`) to initialize inherited state.
- Can declare instance/static fields of any access modifier and define static methods.
- Restricted to single inheritance (`extends`), but can implement multiple interfaces.

An abstract class can define both **abstract methods** (no body, must be overridden) and **concrete methods** (a body, providing an inherited default).

> **Note**: abstract methods can only be instance methods; they cannot be `static`, `final`, or `private`, since each of those would prevent the overriding that an abstract method requires.

## Interface

An interface defines a contract of capabilities that implementing classes must support. It is a stateless reference type specifying what a class CAN-DO, separating behavioral specification from implementation.

```java
interface Flyable {
    void fly();       // public abstract by default
}

interface Swimmable {
    void swim();
}

// A class can implement multiple interfaces
class Duck implements Flyable, Swimmable {
    @Override
    public void fly() { System.out.println("Duck flying"); }

    @Override
    public void swim() { System.out.println("Duck swimming"); }
}
```

Interfaces follow these rules:

- Cannot declare instance fields or constructors; all variables are implicitly `public static final` constants.
- All abstract methods are implicitly `public abstract` (an implementing class must override them with `public` access).
- A class can implement multiple interfaces (multiple inheritance of capability).
- An interface can extend multiple interfaces at once (`extends A, B`).

## Concrete Methods in Interface

Since Java 8, interfaces can carry method bodies in three forms: default, static, and (since Java 9) private helpers.

### Default Methods

Introduced in Java 8, default methods let an interface supply a method body, allowing APIs to evolve without breaking existing implementations.

When `stream()` was added to `Collection` in Java 8, every existing implementation would have broken; making it a default method solved that.

```java
interface Sortable {
    void sort();

    default void bubbleSort() {
        System.out.println("Default bubble sort");
    }
}

class MyList implements Sortable {
    public void sort() { /* must implement */ }
    // bubbleSort() is inherited, can be used or overridden
}
```

### Static Methods

Introduced in Java 8, interface static methods are bound to the interface type itself and are neither inherited nor overridden by implementing classes.

```java
interface Validator {
    boolean validate(String input);

    static boolean isNotEmpty(String s) {
        return s != null && !s.isEmpty();
    }
}

Validator.isNotEmpty("hello");    // OK, called on the interface
// myValidator.isNotEmpty("x");   // compile error: not inherited
```

### Private Methods

Introduced in Java 9, private methods (instance or `static`) act as encapsulation helpers that share common logic between default or static methods without exposing it to implementers.

```java
interface Logger {
    default void logInfo(String msg) { log("INFO", msg); }
    default void logError(String msg) { log("ERROR", msg); }

    private void log(String level, String msg) {    // shared helper
        System.out.println(level + ": " + msg);
    }
}
```

## Default Method Conflicts

Java resolves default-method conflicts with three priority rules:

1. **Class wins**: a class implementation (inherited or defined) always beats any interface default.
2. **Subinterface wins**: a method from a more specific (child) interface beats a parent interface's.
3. **Explicit override**: if it is still ambiguous, the subclass must override and resolve it explicitly via `InterfaceName.super.methodName()`.

```java
interface A { default void show() { System.out.println("A"); } }
interface B { default void show() { System.out.println("B"); } }

// Explicit override resolution
class C implements A, B {
    @Override
    public void show() {
        A.super.show(); // delegates to A's implementation
    }
}

// "Class wins" resolution
class Parent {
    public void show() { System.out.println("Parent"); }
}
class Child extends Parent implements A {
    // Parent.show() takes priority over A.show()
}
```

## Marker Interfaces

A marker interface is an empty interface (no methods or fields) used purely as metadata, flagging a class for special handling by the JVM, compiler, or frameworks.

Modern Java largely replaces this pattern with **annotations** (e.g. `@Entity`, `@Component`), which carry metadata without polluting the class type hierarchy.

| Marker Interface | Purpose                                                                    |
| :--------------- | :------------------------------------------------------------------------- |
| `Serializable`   | Flags a class as serializable for JVM object serialization                 |
| `Cloneable`      | Authorizes `Object.clone()`; throws `CloneNotSupportedException` without it |
| `Remote`         | Flags an interface as available for Remote Method Invocation (RMI)          |

## Class vs Abstract Class vs Interface

|                  | Class                      | Abstract Class                                              | Interface                                          |
| ---------------- | -------------------------- | ---------------------------------------------------------- | -------------------------------------------------- |
| Instantiate      | Yes                        | No                                                         | No                                                 |
| Methods          | Concrete only              | Abstract + concrete                                        | Abstract + default + static + private              |
| Fields           | Any                        | Any                                                        | `public static final` only                         |
| Constructor      | Yes                        | Yes                                                        | No                                                 |
| Inheritance      | Single (`extends`)         | Single (`extends`)                                         | Multiple (`implements`)                            |
| Access modifiers | Any                        | Any                                                        | `public` methods, `public static final` fields     |
| Purpose          | Full implementation        | Template with partial implementation                       | Contract / capability                              |
| Use when         | Complete, stand-alone type | IS-A relation, shared state/code, or Template Method pattern | CAN-DO capability, multiple inheritance, or API evolution |

## Q&A

1. **Can you invoke an interface default method via `super` syntax from outside the implementing class?**
No. `InterfaceName.super.methodName()` is valid only inside a non-static method of a class that directly implements the interface. External callers invoke the method on an instance, which dispatches dynamically to the class's overridden version.

2. **Why can't interfaces provide default implementations of `Object` methods like `equals()` or `hashCode()`?**
Because of the "class wins" rule: every class implicitly extends `Object`, so `Object`'s implementation always takes priority, which would make an interface default dead code. Java forbids the declaration outright.

3. **Why can't an interface default method be `synchronized` or `final`?**
`final` is illegal because defaults are meant to be optionally overridden. `synchronized` is prohibited because interfaces are stateless contracts and cannot dictate the monitor lock or synchronization policy of arbitrary implementing objects.

4. **What is the defining structural difference between an abstract class and an interface?**
State and lifecycle. Abstract classes can hold instance fields (mutable state) and constructors to initialize an object, whereas interfaces are stateless (fields are implicitly `public static final` constants) and cannot define constructors.

## Quick Revision Cheat Sheet

| Concept                | Remember                                                                                       |
| :--------------------- | :--------------------------------------------------------------------------------------------- |
| Instantiation          | Abstract classes and interfaces can't be instantiated; they serve as reference types           |
| Abstract constraints   | Abstract methods can't be `private`, `static`, or `final` (they must be overridable)           |
| Interface constants    | Implicitly `public static final`; must be initialized at declaration                           |
| Interface constructors | None, and no initializer blocks; interfaces don't manage object lifecycle                      |
| Default methods        | Java 8+; instance methods with bodies; evolve APIs without breaking implementers                |
| Static methods         | Java 8+; bound to the interface type; not inherited; called via the interface name              |
| Private methods        | Java 9+; instance or static helpers for default/static methods; invisible to implementers       |
| Synchronization        | Default methods can't be `synchronized`; interfaces can't govern implementer locks             |
| Resolution priority    | Class implementation wins, then the more specific interface, then an explicit override          |
| Marker replacement     | Marker interfaces (e.g. `Serializable`) are legacy; modern APIs prefer annotations              |
