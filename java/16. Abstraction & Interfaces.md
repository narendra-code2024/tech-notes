Abstraction is the process of hiding implementation details and exposing only essential functionality to the user.

A real-world analogy is driving a car: the driver interacts with the steering wheel and pedals without needing to understand the engine internals. Java implements this concept through **abstract classes** and **interfaces**.

---

## Abstract Class

An abstract class represents a class structure that cannot be directly instantiated and is designed to define abstract or concrete template behaviors for its subclasses.

```java
abstract class Shape {
    String color;

    // Abstract method, no body, child MUST implement
    abstract double area();

    // Concrete method, has body, child inherits as-is (can override)
    void display() {
        System.out.println("Color: " + color + ", Area: " + area());
    }
}

class Circle extends Shape {
    double radius;

    Circle(double radius) { this.radius = radius; }

    @Override
    double area() { return Math.PI * radius * radius; }   // must implement
}

// Shape s = new Shape();    // compile error: can't instantiate abstract class
Shape s = new Circle(5);     // OK: can use as reference type
s.area();                     // calls Circle's implementation
s.display();                  // inherited concrete method
```

**Rules**:
- Cannot be directly instantiated, but is valid as a polymorphic reference type
- Concrete subclasses must implement all abstract methods
- Supports constructors (invoked via `super()`) to initialize inherited state
- Can declare instance/static fields of any access modifier and define static methods
- Restricted to single inheritance (`extends`), but can implement multiple interfaces

An abstract class can define both **abstract methods** (which have no body and must be overridden by subclasses) and **concrete methods** (which have a body and provide an inherited default implementation).

> **Note**: Abstract methods can only be instance methods; they cannot be declared `static`, `final`, or `private` because those modifiers prevent overriding.

## Interface

An interface defines a contract of capabilities that implementing classes must support. In Java, it is a stateless reference type specifying what a class CAN-DO, separating behavioral specification from implementation.

```java
interface Flyable {
    void fly();       // public abstract by default
}

interface Swimmable {
    void swim();
}

// A class can implement multiple interfaces
class Duck implements Flyable, Swimmable {
    @Override
    public void fly() { System.out.println("Duck flying"); }

    @Override
    public void swim() { System.out.println("Duck swimming"); }
}
```

**Rules**:
- Cannot declare instance fields or constructors; all variables are implicitly `public static final` constants
- All abstract methods are implicitly `public abstract` (subclasses overriding them must use `public` access)
- A class can implement multiple interfaces (multiple inheritance of capability)
- An interface can extend multiple interfaces simultaneously (`extends A, B`)

## Concrete Methods in Interface

### Default Methods

Introduced in Java 8, default methods allow interfaces to define method implementations with a body, providing backward compatibility to evolve APIs without breaking existing subclasses.

When `stream()` was added to `Collection` in Java 8, all existing implementations would break. Default methods solved this, and `Collection` received a default `stream()` method.

```java
interface Sortable {
    void sort();

    default void bubbleSort() {
        System.out.println("Default bubble sort");
    }
}

class MyList implements Sortable {
    public void sort() { /* must implement */ }
    // bubbleSort() is inherited, can use or override
}
```

### Static Methods

Introduced in Java 8, static methods in an interface are bound to the interface type itself and cannot be inherited or overridden by implementing subclasses.

```java
interface Validator {
    boolean validate(String input);

    static boolean isNotEmpty(String s) {
        return s != null && !s.isEmpty();
    }
}

Validator.isNotEmpty("hello");    // OK, Called on interface
// myValidator.isNotEmpty("x");   // compile error: not inherited
```

### Private Methods

Introduced in Java 9, private methods act as encapsulation helpers to share common logic between default or static methods without exposing them to implementing subclasses.

```java
interface Logger {
    default void logInfo(String msg) { log("INFO", msg); }
    default void logError(String msg) { log("ERROR", msg); }

    private void log(String level, String msg) {    // shared helper
        System.out.println(level + ": " + msg);
    }
}
```

## Default Method Conflicts

Java resolves default method conflicts using three priority rules:
1. **Class wins**: A class implementation (inherited or defined) always takes priority over any interface default method.
2. **Subinterface wins**: A method from a more specific (child) interface takes priority over a parent interface.
3. **Explicit override**: If the conflict remains ambiguous, the subclass must override the method and resolve it explicitly (using `InterfaceName.super.methodName()`).

```java
interface A { default void show() { System.out.println("A"); } }
interface B { default void show() { System.out.println("B"); } }

// Explicit override resolution
class C implements A, B {
    @Override
    public void show() {
        A.super.show(); // delegates to A's implementation
    }
}

// "Class wins" resolution
class Parent {
    public void show() { System.out.println("Parent"); }
}
class Child extends Parent implements A {
    // Parent.show() takes priority over A.show()
}
```

## Marker Interfaces

A marker interface is an empty interface with no methods or fields, acting solely as metadata to flag a class for special treatment by the JVM, compiler, or frameworks.

Modern Java largely replaces this pattern with **annotations** (e.g., `@Entity`, `@Component`), which define metadata without polluting the class type hierarchy.

| Marker Interface | Purpose                                                                    |
| :--------------- | :------------------------------------------------------------------------- |
| `Serializable`   | Flags a class as serializable for JVM object serialization                 |
| `Cloneable`      | Authorizes `Object.clone()`; throws `CloneNotSupportedException` without it |
| `Remote`         | Flags an interface as available for Remote Method Invocation (RMI)          |

## Class vs Abstract Class vs Interface

|                  | Class                        | Abstract Class                                               | Interface                                                 |
| ---------------- | ---------------------------- | ------------------------------------------------------------ | --------------------------------------------------------- |
| Instantiate      | Yes                          | No                                                           | No                                                        |
| Methods          | Concrete only                | Abstract + Concrete                                          | Abstract + Default + Static + Private                      |
| Fields           | Any                          | Any                                                          | `public static final` only                                |
| Constructor      | Yes                          | Yes                                                          | No                                                        |
| Inheritance      | Single (`extends`)           | Single (`extends`)                                           | Multiple (`implements`)                                   |
| Access modifiers | Any                          | Any                                                          | `public` (methods), `public static final` (fields)        |
| Purpose          | Full implementation          | Template with partial implementation                         | Contract / capability                                     |
| Use when         | Complete, stand-alone type   | IS-A relation, shared state/code, or Template Method pattern | CAN-DO capability, multiple inheritance, or API evolution |

## Q&A

1. **Can you invoke an interface default method using `super` syntax from outside the implementing class?**
No. The syntax `InterfaceName.super.methodName()` is only valid within non-static methods of a class that directly implements the interface. External callers must invoke the method on an instance of the implementing class, which resolves dynamically to the class's overridden version.

2. **Why are interfaces forbidden from defining default implementations for `java.lang.Object` methods like `equals()` or `hashCode()`?**
Java's conflict resolution follows the "class wins" rule. Since all classes implicitly inherit from `java.lang.Object`, any class-defined implementation (including `Object`'s default) always takes priority, which would render an interface's default implementation dead code.

3. **Why are interface default methods forbidden from being declared `synchronized` or `final`?**
Default methods are designed to be optionally overridden by implementing classes, making `final` illegal. The `synchronized` modifier is prohibited because interfaces are stateless contracts and cannot dictate the synchronization policy or monitor locks of arbitrary implementing objects.

4. **What is the defining structural difference between an abstract class and an interface in modern Java?**
The defining difference is state and lifecycle management. Abstract classes can declare instance fields (mutable state) and constructors to initialize object state, whereas interfaces are stateless (all fields are implicitly `public static final` constants) and cannot define constructors.

## Quick Revision Cheat Sheet

| Concept                 | Remember                                                                                       |
| :---------------------- | :--------------------------------------------------------------------------------------------- |
| Instantiation           | Abstract classes and interfaces cannot be instantiated directly; they serve as reference types. |
| Abstract constraints    | Abstract methods cannot be declared `private`, `static`, or `final` (since they must be overridden). |
| Interface constants     | Implicitly `public static final` constants; must be initialized at declaration.                |
| Interface constructors  | Interfaces cannot define constructors or initializer blocks; they cannot manage object lifecycle. |
| Default methods         | Java 8+; instance methods with bodies; provide backward compatibility without breaking clients.  |
| Static methods          | Java 8+; bound to the interface type; not inherited by subclasses; invoked via interface name. |
| Private methods         | Java 9+; helper methods for default/static methods; not visible to subclasses or implementers.  |
| Synchronization         | Default methods cannot be `synchronized`; interfaces cannot govern implementer monitor locks.  |
| Resolution priority     | Resolution order: class implementation wins first, then child interface wins, then override.   |
| Marker tag replacement  | Marker interfaces (e.g., `Serializable`) are legacy; modern APIs prefer annotations.           |
