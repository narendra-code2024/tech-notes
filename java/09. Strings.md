Strings in Java are **immutable** sequences of characters. They are objects, not primitives, and are instances of the `final` class `java.lang.String`, which cannot be extended.

```java
// Two ways to create String objects
String s1 = "hello";              // String literal, stored in the String Pool
String s2 = new String("hello");  // New object, stored directly on the Heap (outside the pool)
```

---

## String Immutability

String objects are **immutable**, meaning their value cannot be changed after creation. Any modification operation creates a new `String` object in memory.

```java
String s = "hello";
s.concat(" world");        // Creates a new String "hello world"; s remains "hello"
System.out.println(s);     // Prints "hello"

s = s.concat(" world");    // s now references the new "hello world" object
```

**Benefits of Immutability**:
* **String Constant Pool Safety**: Reusing string literals saves memory; if strings were mutable, changing one would affect all other shared references.
* **Thread Safety**: Immutable objects are inherently thread-safe and can be shared among multiple threads without synchronization.
* **Security**: Crucial application parameters (like database credentials, file paths, and network URLs) are protected from unauthorized runtime modifications.
* **HashCode Caching**: The hash code is computed once during creation and cached, enhancing performance in hash-based collections like `HashMap`.

---

## String Constant Pool

The **String Constant Pool** is a specialized memory area inside the JVM heap. The JVM uses this pool to store unique string literal instances, preventing redundant allocations.

```java
String a = "hello";             // Allocated in the pool
String b = "hello";             // Reuses the existing pool reference (a == b is true)
String c = new String("hello");   // Allocates 2 objects: pool literal (if absent) + new heap object; a == c is false

// intern() returns the pool reference (but does NOT modify c itself)
String d = c.intern();
System.out.println(a == d);     // true (both point to the pool reference)
System.out.println(a == c);     // false (c still points to the separate heap object!)
```

## String Comparison

```java
String a = "hello";
String c = new String("hello");

// Reference (==) checks the memory address; content (equals) checks the characters
a == c;                        // false (different objects)
a.equals(c);                   // true
a.equalsIgnoreCase("HELLO");   // true

// Lexicographical comparison (compareTo) compares ASCII values character-by-character
"apple".compareTo("banana");   // Negative value (apple < banana)
"banana".compareTo("apple");   // Positive value (banana > apple)
"apple".compareTo("apple");    // 0 (equal)
```

> **Note**: Always use `equals()` to compare string values. Never use `==` for content verification.

## `String` Methods

```java
String s = "  Hello, World!  ";

// Length and State
s.length();                  // 19 (includes leading and trailing whitespace)
s.isEmpty();                 // false (checks if length is 0)
s.isBlank();                 // false (Java 11+; checks if empty or contains only whitespace)

// Case Conversions
s.toUpperCase();             // "  HELLO, WORLD!  "
s.toLowerCase();             // "  hello, world!  "

// Whitespace Trimming
s.trim();                    // "Hello, World!" (removes standard leading/trailing spaces)
s.strip();                   // "Hello, World!" (Java 11+; Unicode-aware whitespace removal)

// Substring Search
s.contains("World");         // true
s.indexOf("World");          // 9 (first character index of match)
s.lastIndexOf("l");          // Last occurrence index
s.startsWith("  He");        // true
s.endsWith("!  ");           // true

// Character Extraction
s.charAt(2);                 // 'H'
s.substring(2, 7);           // "Hello" (extracts characters from index [2, 6])
s.toCharArray();             // char[] {' ', ' ', 'H', 'e', 'l', ...}

// String Replacement
s.replace('l', 'L');         // Replaces all occurrences of 'l' with 'L'
s.replace("World", "Java");  // Replaces substring match
s.replaceAll("[^a-zA-Z]", ""); // Uses regular expression (removes non-alphabetic chars)

// Splitting and Joining
"a,b,c".split(",");          // String[] {"a", "b", "c"}
String.join("-", "a", "b");  // "a-b"

// Formatting
String.format("Name: %s, Age: %d", "Alice", 25);  // "Name: Alice, Age: 25"

// Static Factory Conversion
Object obj = null;
String.valueOf(123);         // "123" (converts primitive to String)
String.valueOf(obj);         // Returns literal "null" safely (whereas obj.toString() throws NullPointerException)
```

## String Concatenation Internals

String concatenation in Java behaves differently depending on whether it is evaluated at compile-time or runtime:

* **Compile-Time (Constant Folding)**: The compiler automatically optimizes literal additions into a single compile-time constant. This incurs **zero** runtime cost.
* **Runtime Concatenation**: Concatenating variables using the `+` operator translates to a `StringBuilder` conversion under the hood. At runtime, each `+` statement creates a new temporary builder object, leading to severe allocation overhead.

```java
// Compile-time literal folding (optimized at compile-time)
String s = "a" + "b" + "c";    // Compiled as: String s = "abc";

// Runtime variable concatenation (uses builders under the hood)
String x = "a";
String result = x + "b" + "c"; // Handled as: new StringBuilder().append(x).append("b").append("c").toString()
```

> **Note**: Java 9+ optimizes runtime concatenation using `invokedynamic` (`StringConcatFactory`), but explicit `StringBuilder` usage remains mandatory inside loops to prevent repeated dynamic allocations.

## String vs StringBuilder vs StringBuffer

| Feature          | `String`                | `StringBuilder` | `StringBuffer`              |
| :--------------- | :---------------------- | :-------------- | :-------------------------- |
| Mutable          | No (creates new object) | Yes             | Yes                         |
| Thread-Safe      | Yes (immutable)         | No              | Yes (synchronized methods)  |
| Speed            | Slow for modifications  | Fastest         | Slower than `StringBuilder` |
| Storage Location | String Pool and Heap    | Heap            | Heap                        |

When building or modifying strings sequentially, using the `+` operator creates intermediate garbage string objects. For iterative or high-frequency manipulation, explicitly use `StringBuilder` to modify the buffer in-place:

```java
// Performance Anti-pattern: + in loops
String s = "";
for (int i = 0; i < 1000; i++) {
    s += i;                    // Creates 1000 new String objects (terrible performance)
}

// Recommended Pattern: Explicit StringBuilder
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i);              // Mutates the same object in-place (high performance)
}
String result = sb.toString();
```

### `StringBuilder` Methods

Unlike `String`, `StringBuilder` methods modify the internal character buffer in-place. Because they return a reference to the same `StringBuilder` instance, methods can be **chained**.

```java
StringBuilder sb = new StringBuilder("hello");

sb.append(" world");       // "hello world" (adds to end)
sb.insert(5, ",");         // "hello, world" (inserts at index)
sb.delete(2, 4);           // "heo, world" (deletes range of characters at indices [2, 3])
sb.reverse();              // "dlrow ,oeh" (reverses content in-place)
sb.toString();             // Converts the StringBuilder content back to a String

// Method chaining example
String chained = new StringBuilder("A").append("B").reverse().toString(); // "BA"
```

## Text Blocks

Introduced in Java 15, **Text Blocks** represent multi-line string literals, completely eliminating the need for escape sequences (`\"`) and manual concatenation (`+`).

```java
// Pre-Java 15 multiline strings
String oldJson = "{\n" +
                 "  \"name\": \"Alice\",\n" +
                 "  \"age\": 25\n" +
                 "}";

// Modern Text Block syntax
String newJson = """
                 {
                   "name": "Alice",
                   "age": 25
                 }
                 """;
```

## Q&A

1. **What does it mean that Strings are immutable, and what is a common mistake related to this?**
Strings cannot be altered after creation. Methods like `concat()`, `replace()`, and `toUpperCase()` do not modify the original object; they return a new String instance. A common bug is invoking these modification methods without assigning the returned reference back to a variable.

```java
String s = "hello";
s.concat(" world"); // Mistake: original string s remains "hello"
s = s.concat(" world"); // Correct: s now references the new object
```

2. **Where are Strings created when using literals versus the `new` keyword?**
String literals are stored in the String Constant Pool inside the heap to save memory through instant reuse. The `new` keyword bypasses the pool and allocates a unique String object on the standard heap, even if the literal value already exists in the pool.

3. **Why does `"a.b.c".split(".")` return an empty array, and how do you fix it?**
The `split()` method takes a regular expression. In regex, the dot `.` is a wildcard matching any character. Since it matches everything, the split results in an empty array. To split by a literal dot, you must escape the wildcard character with a double backslash.

```java
"a.b.c".split(".");     // Empty array (wildcard match)
"a.b.c".split("\\.");   // ["a", "b", "c"] (literal dot match)
```

4. **What happens when you concatenate a `null` value with a String using the `+` operator versus the `concat()` method?**
The `+` operator handles `null` values safely by appending the literal string `"null"`. However, invoking `s.concat("...")` or passing `null` to `concat()` throws a `NullPointerException` because `concat()` requires a non-null object target.

```java
String s = null;
System.out.println("Value: " + s);  // Prints "Value: null" (safe)
// But: s.concat("x") throws NullPointerException
```

5. **How can you safely compare a String variable with a constant to avoid a `NullPointerException`?**
Always invoke the `equals()` method on the constant string literal itself rather than the variable. If the variable is `null`, invoking `equals()` on it will throw a `NullPointerException`, but invoking it on the literal safely evaluates to `false`.

```java
"hello".equals(s);   // Safe even if s is null
s.equals("hello");   // Throws NullPointerException if s is null
```

## Quick Revision Cheat Sheet

| Concept              | Remember                                                                     |
| :------------------- | :--------------------------------------------------------------------------- |
| Immutable            | Value cannot change; modifying methods always return a new `String` object   |
| String Pool          | Literals are reused inside pool; `new String()` forces unique heap object    |
| `intern()`           | Returns the pool reference for a string, adding it to the pool if absent     |
| `==` vs `equals()`   | `==` checks reference memory addresses; `equals()` verifies character content |
| `StringBuilder`      | Mutable, high-performance, and non-thread-safe; use for loop manipulations  |
| `StringBuffer`       | Mutable, thread-safe (synchronized methods); slower than `StringBuilder`     |
| `trim()` vs `strip()` | `trim()` removes ASCII whitespace; `strip()` is Unicode-aware (Java 11+)     |
| `split("\\.")`       | Escape regex special wildcard characters like the dot using double backslash |
| `+` in loops         | Avoid; compiles to multiple allocations. Use `StringBuilder` explicitly      |
| Text blocks          | `"""multi-line string"""` (Java 15+) simplifies embedded quotes and newlines |
| `isBlank()`          | Returns true if empty or contains only whitespace characters (Java 11+)      |
