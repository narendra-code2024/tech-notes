Strings in Java are **immutable** sequences of characters. They are objects, not primitives, and are instances of the `final` class `java.lang.String`, which cannot be extended.

```java
// Two ways to create String objects
String s1 = "hello";              // literal, stored in the String Pool
String s2 = new String("hello");  // new object on the heap, outside the pool
```

---

## String Immutability

A `String`'s value cannot change after creation; any "modifying" operation returns a new `String`.

```java
String s = "hello";
s.concat(" world");        // creates a new String "hello world"; s stays "hello"
System.out.println(s);     // prints "hello"

s = s.concat(" world");    // s now references the new "hello world" object
```

Immutability brings several benefits:

- **Pool safety**: literals can be shared and reused; if they were mutable, changing one would corrupt every other reference to it.
- **Thread safety**: immutable objects are inherently thread-safe and need no synchronization to share.
- **Security**: sensitive values (credentials, file paths, URLs) cannot be altered after a check passes.
- **Hash-code caching**: the hash is computed once and cached, speeding up hash-based collections like `HashMap`.

---

## String Constant Pool

The String Constant Pool is a region of the JVM heap that holds unique string-literal instances, avoiding redundant allocations.

```java
String a = "hello";              // allocated in the pool
String b = "hello";              // reuses the pooled object, so a == b is true
String c = new String("hello");  // 2 objects: pooled literal (if absent) + new heap object; a == c is false

// intern() returns the pool reference; it does NOT modify c
String d = c.intern();
System.out.println(a == d);      // true  (both are the pooled reference)
System.out.println(a == c);      // false (c is still the separate heap object)
```

## String Comparison

```java
String a = "hello";
String c = new String("hello");

// == compares references (addresses); equals() compares characters
a == c;                        // false (different objects)
a.equals(c);                   // true
a.equalsIgnoreCase("HELLO");   // true

// compareTo: lexicographic, by Unicode code-unit values
"apple".compareTo("banana");   // negative (apple < banana)
"banana".compareTo("apple");   // positive
"apple".compareTo("apple");    // 0
```

> **Note**: always compare string content with `equals()`; never use `==`.

## `String` Methods

```java
String s = "  Hello, World!  ";

// Length and state
s.length();                  // 17 (includes the leading and trailing spaces)
s.isEmpty();                 // false (length 0?)
s.isBlank();                 // false (Java 11+; empty or whitespace-only?)

// Case
s.toUpperCase();             // "  HELLO, WORLD!  "
s.toLowerCase();             // "  hello, world!  "

// Whitespace trimming
s.trim();                    // "Hello, World!" (removes ASCII whitespace)
s.strip();                   // "Hello, World!" (Java 11+; Unicode-aware)

// Search
s.contains("World");         // true
s.indexOf("World");          // 9 (index of first match)
s.lastIndexOf("l");          // index of last occurrence
s.startsWith("  He");        // true
s.endsWith("!  ");           // true

// Character extraction
s.charAt(2);                 // 'H'
s.substring(2, 7);           // "Hello" (indices [2, 6])
s.toCharArray();             // char[] {' ', ' ', 'H', 'e', 'l', ...}

// Replacement
s.replace('l', 'L');         // replaces all 'l' with 'L'
s.replace("World", "Java");  // replaces the substring
s.replaceAll("[^a-zA-Z]", ""); // regex: removes non-letters

// Split and join
"a,b,c".split(",");          // String[] {"a", "b", "c"}
String.join("-", "a", "b");  // "a-b"

// Formatting
String.format("Name: %s, Age: %d", "Alice", 25);  // "Name: Alice, Age: 25"

// Static factory conversion
Object obj = null;
String.valueOf(123);         // "123"
String.valueOf(obj);         // "null" safely (obj.toString() would throw NPE)
```

## String Concatenation Internals

How concatenation compiles depends on whether it happens at compile time or runtime:

- **Compile-time (constant folding)**: the compiler folds literal-only additions into a single constant, at zero runtime cost.
- **Runtime**: concatenating variables with `+` compiles to `StringBuilder` operations. A single statement uses one builder, but `+` inside a loop creates a fresh builder every iteration, which is where allocation cost becomes severe.

```java
// Compile-time folding
String s = "a" + "b" + "c";    // compiled as: String s = "abc";

// Runtime concatenation
String x = "a";
String result = x + "b" + "c"; // ~ new StringBuilder().append(x).append("b").append("c").toString()
```

> **Note**: Java 9+ compiles runtime concatenation via `invokedynamic` (`StringConcatFactory`), but an explicit `StringBuilder` is still required inside loops to avoid repeated allocations.

## String vs StringBuilder vs StringBuffer

| Feature          | `String`                | `StringBuilder` | `StringBuffer`             |
| :--------------- | :---------------------- | :-------------- | :------------------------- |
| Mutable          | No (creates new object) | Yes             | Yes                        |
| Thread-safe      | Yes (immutable)         | No              | Yes (synchronized methods) |
| Speed            | Slow for modifications  | Fastest         | Slower than `StringBuilder` |
| Storage          | Pool and heap           | Heap            | Heap                       |

Using `+` to build a string in a loop creates intermediate garbage objects. For iterative or high-frequency manipulation, use `StringBuilder` to mutate the buffer in place:

```java
// Anti-pattern: + in a loop
String s = "";
for (int i = 0; i < 1000; i++) {
    s += i;                    // creates 1000 new String objects
}

// Recommended: explicit StringBuilder
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i);              // mutates one object in place
}
String result = sb.toString();
```

### `StringBuilder` Methods

`StringBuilder` methods mutate the internal buffer in place and return the same instance, so calls can be chained.

```java
StringBuilder sb = new StringBuilder("hello");

sb.append(" world");       // "hello world" (append to end)
sb.insert(5, ",");         // "hello, world" (insert at index)
sb.delete(2, 4);           // "heo, world" (delete indices [2, 3])
sb.reverse();              // reverses in place
sb.toString();             // back to a String

// Chaining
String chained = new StringBuilder("A").append("B").reverse().toString(); // "BA"
```

## Text Blocks

Introduced in Java 15, text blocks are multi-line string literals that remove the need for `\"` escapes and `+` concatenation.

```java
// Pre-Java 15
String oldJson = "{\n" +
                 "  \"name\": \"Alice\",\n" +
                 "  \"age\": 25\n" +
                 "}";

// Text block
String newJson = """
                 {
                   "name": "Alice",
                   "age": 25
                 }
                 """;
```

## Q&A

1. **What does String immutability mean, and what mistake does it cause?**
A `String` cannot be changed after creation; `concat()`, `replace()`, `toUpperCase()` and the rest return a new instance rather than mutating the original. The common bug is calling such a method without assigning the result back.

```java
String s = "hello";
s.concat(" world");     // mistake: s is still "hello"
s = s.concat(" world"); // correct: s now references the new object
```

2. **Where are Strings created with a literal versus `new`?**
A literal goes into the String Constant Pool and is reused on repeat occurrences. `new String(...)` bypasses the pool and allocates a distinct heap object even when the same literal already exists.

3. **Why does `"a.b.c".split(".")` return an empty array, and how do you fix it?**
`split` takes a regex, and `.` matches any character, so everything is treated as a delimiter. Escape it as `"\\."` to split on a literal dot.

```java
"a.b.c".split(".");     // empty array (wildcard)
"a.b.c".split("\\.");   // ["a", "b", "c"]
```

4. **How does concatenating `null` differ between `+` and `concat()`?**
The `+` operator converts `null` to the literal `"null"`, so it is safe. `concat()` calls a method on the target and requires a non-null argument, so a `null` involved in `concat` throws `NullPointerException`.

```java
String s = null;
System.out.println("Value: " + s);  // "Value: null" (safe)
// s.concat("x") would throw NullPointerException
```

5. **How do you compare a String against a constant without risking a `NullPointerException`?**
Call `equals()` on the literal, not the variable: `"hello".equals(s)` is safe even when `s` is `null`, whereas `s.equals("hello")` throws if `s` is `null`.

```java
"hello".equals(s);   // safe even if s is null
s.equals("hello");   // NPE if s is null
```

## Quick Revision Cheat Sheet

| Concept               | Remember                                                                      |
| :-------------------- | :---------------------------------------------------------------------------- |
| Immutable             | Value never changes; modifying methods return a new `String`                  |
| String pool           | Literals are reused in the pool; `new String()` forces a unique heap object    |
| `intern()`            | Returns the pooled reference, adding the string to the pool if absent          |
| `==` vs `equals()`    | `==` compares references; `equals()` compares characters                       |
| `StringBuilder`       | Mutable, fastest, not thread-safe; use for loop building                       |
| `StringBuffer`        | Mutable, thread-safe (synchronized), slower than `StringBuilder`               |
| `trim()` vs `strip()` | `trim()` removes ASCII whitespace; `strip()` is Unicode-aware (Java 11+)        |
| `split("\\.")`        | `split` is regex; escape special chars like `.`                                |
| `+` in loops          | Avoid; each iteration allocates. Use `StringBuilder`                            |
| Text blocks           | `"""..."""` (Java 15+) for multi-line literals without escapes                 |
| `isBlank()`           | True if empty or whitespace-only (Java 11+)                                    |
