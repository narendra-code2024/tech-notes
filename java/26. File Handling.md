Java I/O reads from and writes to files, the console, the network, and similar sources. Two packages cover it:

- **`java.io`**: the original stream-based, blocking I/O (Java 1.0).
- **`java.nio.file`**: the modern `Path` + `Files` API (Java 7), with richer errors, attributes, and directory operations. Preferred for new code.

---

## I/O Streams

An I/O stream is a pipeline for reading or writing data, unrelated to the Java 8 Streams API. Streams come in two families by the unit they carry.

| Aspect       | Byte streams                          | Character streams                       |
| ------------ | ------------------------------------- | --------------------------------------- |
| Unit         | 1 byte                                | Characters (encoding-dependent)         |
| Base classes | `InputStream` / `OutputStream`        | `Reader` / `Writer`                     |
| Use for      | Binary data (images, PDFs, audio)     | Text data (files, logs, CSV)            |
| Examples     | `FileInputStream`, `BufferedInputStream` | `FileReader`, `BufferedReader`        |

---

## Reading & Writing Files

### Character Streams (text)

```java
try (BufferedReader br = new BufferedReader(new FileReader("input.txt"))) {
    String line;
    while ((line = br.readLine()) != null) System.out.println(line);   // readLine() returns null at EOF
}

try (BufferedWriter bw = new BufferedWriter(new FileWriter("output.txt"))) {
    bw.write("Hello World");
    bw.newLine();                 // platform-independent line separator
}

try (FileWriter fw = new FileWriter("output.txt", true)) {   // true = append, not overwrite
    fw.write("appended\n");
}
```

### Byte Streams (binary)

```java
// copy a file in 1 KB chunks
try (FileInputStream in = new FileInputStream("source.png");
     FileOutputStream out = new FileOutputStream("dest.png")) {
    byte[] buf = new byte[1024];
    int n;
    while ((n = in.read(buf)) != -1) out.write(buf, 0, n);   // write only the bytes read
}
```

### PrintWriter (formatted text)

```java
try (PrintWriter pw = new PrintWriter(new FileWriter("output.txt"))) {
    pw.println("Line 1");                            // appends a newline
    pw.printf("Name: %s, Age: %d%n", "Alice", 25);   // formatted, %n is the line separator
}
```

## Buffered Streams

Wrapping a stream in a buffered one reads or writes in chunks instead of per byte, cutting disk access. The gain is large on big or frequent I/O.

```java
BufferedReader br = new BufferedReader(new FileReader("file.txt"));
BufferedWriter bw = new BufferedWriter(new FileWriter("file.txt"));
BufferedInputStream  bis = new BufferedInputStream(new FileInputStream("file.bin"));
BufferedOutputStream bos = new BufferedOutputStream(new FileOutputStream("file.bin"));
```

---

## Bridge: InputStreamReader / OutputStreamWriter

These bridge a byte stream to a character stream, needed when the source is bytes (a socket, `System.in`) but you want text. They are also where you pin the charset.

```java
BufferedReader console = new BufferedReader(new InputStreamReader(System.in));

BufferedReader br = new BufferedReader(
    new InputStreamReader(new FileInputStream("file.txt"), StandardCharsets.UTF_8));
BufferedWriter bw = new BufferedWriter(
    new OutputStreamWriter(new FileOutputStream("file.txt"), StandardCharsets.UTF_8));
```

> **Note**: `FileReader`/`FileWriter` use the platform default charset, which differs across machines and can corrupt text. Specify a charset explicitly, or use the `Files` methods below (which default to UTF-8).

## java.nio.file (Modern File API, Java 7+)

`Path` describes a location; `Files` operates on it. Cleaner and more capable than `java.io.File`.

### Path

```java
Path p = Path.of("data", "subdir", "file.txt");   // Java 11+; joins with the OS separator
Path q = Paths.get("data/file.txt");               // Java 7+ equivalent

p.getFileName();     // file.txt
p.getParent();       // data/subdir
p.toAbsolutePath();
p.resolve("sub");    // append a child element
```

### Files utility class

```java
String text   = Files.readString(Path.of("file.txt"));          // Java 11+
List<String> lines = Files.readAllLines(Path.of("file.txt"));   // Java 7+
byte[] bytes  = Files.readAllBytes(Path.of("file.bin"));

Files.writeString(Path.of("file.txt"), "Hello");                            // Java 11+
Files.write(Path.of("file.txt"), List.of("line1", "line2"));                // Java 7+
Files.writeString(Path.of("file.txt"), "more", StandardOpenOption.APPEND);  // append

Files.copy(src, dst, StandardCopyOption.REPLACE_EXISTING);
Files.move(src, dst, StandardCopyOption.REPLACE_EXISTING);   // can be atomic with ATOMIC_MOVE
Files.delete(path);            // throws NoSuchFileException if absent
Files.deleteIfExists(path);

Files.exists(path);   Files.isDirectory(path);   Files.size(path);
Files.createDirectories(path);   // creates missing parents, like mkdir -p
```

### Copying streams

```java
Files.copy(inputStream, Path.of("uploaded.txt"), StandardCopyOption.REPLACE_EXISTING);  // upload
Files.copy(Path.of("file.txt"), outputStream);                                          // download

try (InputStream in = new FileInputStream("src.bin");
     OutputStream out = new FileOutputStream("dest.bin")) {
    in.transferTo(out);   // Java 9+, efficient native transfer
}
```

> **Note**:
> - `Files.copy(inputStream, ...)` does not close the stream; manage it with try-with-resources.
> - `Files.copy` handles files only, not recursive directory copies; walk the tree and copy each entry.

### Stream-based reading (Java 8+)

```java
try (Stream<String> lines = Files.lines(Path.of("large.txt"))) {   // lazy, one line at a time
    lines.filter(l -> l.contains("ERROR")).forEach(System.out::println);
}

try (Stream<Path> paths = Files.walk(Path.of("src"), 2)) {   // recursive, depth-limited to 2
    paths.filter(Files::isRegularFile)
         .filter(p -> p.toString().endsWith(".java"))
         .forEach(System.out::println);
}

try (Stream<Path> entries = Files.list(Path.of("data"))) { /* one level, non-recursive */ }
```

> **Note**: streams from `Files.lines`/`walk`/`list` hold an open filesystem handle; close them with try-with-resources or they leak file descriptors. `Files.walk` can be expensive on deep trees, so filter early and bound the depth.

A `WatchService` (`FileSystems.getDefault().newWatchService()`, then `path.register(...)` for `ENTRY_CREATE`/`DELETE`/`MODIFY`) notifies on directory changes.

### Buffered reading/writing

`Files.newBufferedReader`/`newBufferedWriter` combine buffering and an explicit charset in one call, the cleanest legacy-free option:

```java
try (BufferedReader br = Files.newBufferedReader(Path.of("file.txt"), StandardCharsets.UTF_8)) {
    String line = br.readLine();
}
```

## File class (Legacy, java.io)

Still seen in older code, but `Path` + `Files` is preferred.

```java
File f = new File("data/file.txt");
f.exists();   f.isFile();   f.length();   f.delete();
f.mkdirs();        // create directory with parents
f.listFiles();     // File[] of directory contents
Path p = f.toPath();   File back = p.toFile();   // convert either way
```

---

## try-with-resources for I/O

Every I/O stream implements `AutoCloseable`, so a try-with-resources block closes it automatically (even on exception), replacing manual `finally { stream.close(); }`.

```java
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    String line = br.readLine();
}   // br.close() runs automatically
```

## java.io.File vs java.nio.file

| Dimension         | java.io.File              | java.nio.file (Path + Files)        |
| ----------------- | ------------------------- | ----------------------------------- |
| Introduced        | Java 1.0                  | Java 7                              |
| Error reporting   | boolean, silent failure   | throws `IOException` with a reason  |
| Symbolic links    | Poor                      | Full support                        |
| File attributes   | Limited                   | Rich (permissions, owner, times)    |
| Directory walking | Manual recursion          | `Files.walk()`, `Files.list()`      |
| Atomic operations | No                        | Yes (`Files.move` with ATOMIC_MOVE) |
| Watch service     | No                        | `WatchService`                      |

Use `Path` + `Files` for new code; reach for `File` only when a legacy API requires it.

## Q&A

1. **Why avoid `FileReader` and `FileWriter` in production?**
They use the platform default charset, which varies by OS and can corrupt text across environments. Use `InputStreamReader`/`OutputStreamWriter` or the `Files` methods with an explicit charset like `StandardCharsets.UTF_8`.

2. **What does `BufferedReader.readLine()` return at end of file?**
`null`, not an empty string or an exception. Loop until it returns `null`.

3. **How do you append instead of overwrite?**
Pass `true` to the `FileWriter` constructor, or use `StandardOpenOption.APPEND` with `Files.write`/`writeString`.

4. **When should you avoid `Files.readAllLines()`?**
On large files: it loads every line into memory at once and can throw `OutOfMemoryError`. Use `Files.lines()`, a lazy `Stream<String>` read one line at a time.

5. **Must you call `flush()` before closing a buffered stream?**
No, `close()` flushes first. Call `flush()` only to force data out while keeping the stream open (logging, real-time output).

6. **Why prefer `Files.delete()` over `File.delete()`?**
`File.delete()` returns a bare boolean with no reason on failure; `Files.delete()` throws a specific exception (`NoSuchFileException`, `DirectoryNotEmptyException`), which is far easier to diagnose.

7. **How do you keep paths cross-platform?**
Never hardcode `/` or `\`. Build paths with `Path.of("a", "b")`, which inserts the host separator.

8. **Must you close the stream from `Files.lines()`?**
Yes. It holds an open file handle, so without try-with-resources you leak file descriptors and can eventually exhaust them.

9. **Why might `Files.write()` with only `APPEND` fail?**
`APPEND` alone does not create a missing file and throws. Combine `CREATE` and `APPEND` to handle both cases.

```java
Files.write(path, data, StandardOpenOption.CREATE, StandardOpenOption.APPEND);
```

## Quick Revision Cheat Sheet

| Concept            | Remember                                                                |
| ------------------ | ----------------------------------------------------------------------- |
| Byte streams       | `InputStream`/`OutputStream` for binary data (images, PDFs)             |
| Character streams  | `Reader`/`Writer` for text                                              |
| Buffered           | Wrap streams to read/write in chunks; large gain on big files           |
| Files (nio)        | Modern and preferred: `readString`, `write`, `copy`, `lines`            |
| Path (nio)         | `Path.of()` / `Paths.get()` replaces `new File()`                       |
| File (io)          | Legacy; boolean returns, silent failures                               |
| try-with-resources | Always use for I/O; auto-closes even on exception                       |
| Bridge classes     | `InputStreamReader`/`OutputStreamWriter` convert bytes and characters   |
| Large files        | `Files.lines()` is lazy and memory-light; close the stream             |
| Encoding           | Specify UTF-8 explicitly; never rely on the platform default            |
| Append             | `new FileWriter(f, true)` or `StandardOpenOption.APPEND` (with `CREATE`) |
