## What & Why

Functional programming in Java (Java 8+) lets you treat **functions as values**: pass them as arguments, return them, store them in variables. This enables cleaner, more concise code.

```java
// Before Java 8, anonymous class (verbose)
Collections.sort(list, new Comparator<String>() {
    public int compare(String a, String b) {
        return a.length() - b.length();
    }
});

// Java 8+, lambda (concise)
Collections.sort(list, (a, b) -> a.length() - b.length());
```

### Why this feature was added (the pre-Java 8 pain)

- **Anonymous inner classes for single-method callbacks were syntax noise**: 5 lines of boilerplate to express "compare by length." Any callback-heavy API (Comparator, Runnable, event listeners, click handlers) forced verbose anonymous classes everywhere.
- **Data processing was stuck in imperative for-loops**: filtering, mapping, reducing a list meant declaring result collections, iterating manually, mutating state. Hard to read the *intent* under the *mechanics*.
- **No way to parallelize loops easily**: even on multi-core machines, collection processing ran single-threaded by default. Parallelizing required thread pools, locks, futures, huge code surgery.
- **Java lagged competitors**: Scala, C#, Python, JavaScript had first-class functions for years. Java needed them to stay relevant for modern data-processing and reactive workloads.

### How lambdas + functional interfaces solve it

- **Lambdas collapse the callback boilerplate**: `(a, b) -> a.length() - b.length()` replaces 5 lines of anonymous class.
- **Functional interfaces provide the "type"**: every lambda still needs a target type (Java is statically typed). `java.util.function` gives ready-made ones (`Function`, `Predicate`, `Consumer`, `Supplier`) so you don't define a new interface for every callback shape.
- **Unlocks Streams API**: once you can pass functions cheaply, you can write `list.stream().filter(...).map(...).collect(...)`, declarative data processing with trivial parallelization via `.parallelStream()`.
- **Backward compatible**: default methods on interfaces (Java 8) let `Collection` get `.stream()` without breaking every existing `Collection` implementation.

## Functional Interfaces

A functional interface has **exactly one abstract method** (SAM, Single Abstract Method). This is what makes lambdas possible, the lambda becomes the implementation of that one method.

```java
@FunctionalInterface       // optional but recommended, compiler enforces SAM rule
interface Greeting {
    void sayHello(String name);          // the one abstract method
    default void hi() {}                 // default methods don't count
    static Greeting noop() { return n -> {}; }  // static methods don't count
    @Override boolean equals(Object o);  // Object's public methods don't count
}

// Lambda implements the abstract method
Greeting g = name -> System.out.println("Hello " + name);
g.sayHello("Alice");   // Hello Alice
```

A `public abstract` redeclaration of an `Object` method (e.g. `equals`, `toString`) is ignored when counting the single abstract method, so the interface above is still functional.

### Built-in Functional Interfaces (java.util.function)

These cover most use cases, rarely need to create your own.

|Interface|Method|Takes|Returns|Use case|
|---|---|---|---|---|
|`Predicate<T>`|`test(T)`|T|boolean|Filtering, conditions|
|`Function<T,R>`|`apply(T)`|T|R|Transform/map|
|`Consumer<T>`|`accept(T)`|T|void|Perform action (forEach)|
|`Supplier<T>`|`get()`|nothing|T|Provide/generate value|
|`UnaryOperator<T>`|`apply(T)`|T|T|Transform, same type in & out|
|`BinaryOperator<T>`|`apply(T,T)`|T, T|T|Combine two values (reduce)|
|`BiFunction<T,U,R>`|`apply(T,U)`|T, U|R|Two inputs, one output|
|`BiPredicate<T,U>`|`test(T,U)`|T, U|boolean|Condition on two inputs|
|`BiConsumer<T,U>`|`accept(T,U)`|T, U|void|Action on two inputs (Map.forEach)|

### Examples of each

```java
// Predicate, is this true?
Predicate<Integer> isEven = n -> n % 2 == 0;
isEven.test(4);    // true
isEven.test(7);    // false

// Function, transform A to B
Function<String, Integer> toLength = s -> s.length();
toLength.apply("hello");   // 5

// Consumer, do something, return nothing
Consumer<String> printer = s -> System.out.println(s);
printer.accept("hello");   // prints: hello

// Supplier, give me something
Supplier<Double> randomNum = () -> Math.random();
randomNum.get();   // 0.7234...

// UnaryOperator, transform, same type
UnaryOperator<String> toUpper = s -> s.toUpperCase();
toUpper.apply("hello");   // "HELLO"

// BinaryOperator, combine two values
BinaryOperator<Integer> sum = (a, b) -> a + b;
sum.apply(3, 4);   // 7

// BiFunction, two inputs, different output
BiFunction<String, String, Integer> totalLength = (a, b) -> a.length() + b.length();
totalLength.apply("hi", "world");   // 7

// BiConsumer, used in Map.forEach
Map<String, Integer> map = Map.of("a", 1, "b", 2);
map.forEach((k, v) -> System.out.println(k + "=" + v));
```

### Chaining Functional Interfaces

```java
// Predicate, and, or, negate
Predicate<Integer> isEven = n -> n % 2 == 0;
Predicate<Integer> isPositive = n -> n > 0;
Predicate<Integer> isEvenAndPositive = isEven.and(isPositive);
Predicate<Integer> isEvenOrPositive = isEven.or(isPositive);
Predicate<Integer> isOdd = isEven.negate();

// Function, andThen, compose
Function<String, String> trim = String::trim;
Function<String, String> toUpper = String::toUpperCase;

// andThen: trim first → then toUpper
Function<String, String> cleanUp = trim.andThen(toUpper);
cleanUp.apply("  hello  ");   // "HELLO"

// compose: toUpper first → then trim (reverse order)
Function<String, String> reversed = trim.compose(toUpper);

// Comparator chaining with method references (interview favorite)
list.sort(Comparator.comparing(String::length));              // sort by length
list.sort(Comparator.comparing(String::length).reversed());   // sort by length descending
list.sort(Comparator.comparing(String::length).thenComparing(String::compareTo)); // tiebreaker

// Consumer, andThen
Consumer<String> print = System.out::println;
Consumer<String> log = s -> System.err.println("LOG: " + s);
Consumer<String> printAndLog = print.andThen(log);  // does both
```

### Functional Interface vs Regular Interface

||Functional Interface|Regular Interface|
|---|---|---|
|Abstract methods|Exactly 1 (SAM)|Any number|
|Lambda support|Yes|No|
|`@FunctionalInterface`|Optional but recommended|Not applicable|
|Default methods|Allowed (any number)|Allowed|
|Static methods|Allowed (any number)|Allowed|

### Primitive Specialized Functional Interfaces

Avoid autoboxing overhead with primitive specializations:

```java
// Instead of Predicate<Integer>, avoids boxing int → Integer
IntPredicate isEven = n -> n % 2 == 0;
isEven.test(4);   // true, no boxing

// Instead of Function<Integer, Integer>
IntUnaryOperator doubler = n -> n * 2;
doubler.applyAsInt(5);   // 10

// Instead of Supplier<Integer>
IntSupplier random = () -> (int)(Math.random() * 100);
random.getAsInt();   // 42

// Instead of Consumer<Integer>
IntConsumer printer = n -> System.out.println(n);
printer.accept(42);  // prints: 42

// Similarly: LongPredicate, DoublePredicate, IntFunction<R>,
//            IntToDoubleFunction, ToIntFunction<T>, etc.
```

## Lambda Expressions

A lambda is a short way to write an **anonymous function**: it implements the single abstract method of a functional interface.

### Syntax

```java
// Full syntax
(parameters) -> { body; return value; }

// Shortcuts
(a, b) -> a + b              // single expression, no braces, no return needed
a -> a * 2                    // single param, no parentheses needed
() -> System.out.println("hi") // no params, empty parentheses required
```

### Rules

```java
// Parameter types are optional, compiler infers them
(String a, String b) -> a + b   // explicit types
(a, b) -> a + b                 // inferred types, both are valid

// If body has multiple statements, braces + return required
(a, b) -> {
    int sum = a + b;
    return sum;
}

// Lambda can only be assigned to a functional interface type
Runnable r = () -> System.out.println("running");
Comparator<String> c = (a, b) -> a.compareTo(b);
```

### Lambda is NOT a method, key differences

```java
// Lambda has no name, no access modifier, no return type declaration
// It captures the type from the functional interface it's assigned to

// Lambda can access:
// - local variables (only if effectively final)
// - instance/static variables (freely)
int factor = 2;                        // effectively final, never reassigned
Function<Integer, Integer> doubler = n -> n * factor;  // OK
// factor = 3;                         // compile error: would make above lambda invalid
```

## Method References

Shorthand for lambdas that just call an existing method. If the lambda only passes its arguments to a method, use a method reference instead.

### Four types

|Type|Syntax|Lambda equivalent|
|---|---|---|
|Static method|`ClassName::staticMethod`|`(args) -> ClassName.staticMethod(args)`|
|Instance method (specific object)|`object::method`|`(args) -> object.method(args)`|
|Instance method (any object of type)|`ClassName::method`|`(obj, args) -> obj.method(args)`|
|Constructor|`ClassName::new`|`(args) -> new ClassName(args)`|

### Examples

```java
// 1. Static method reference
Function<String, Integer> parse = Integer::parseInt;       // s -> Integer.parseInt(s)
parse.apply("123");   // 123

// 2. Instance method, specific object
String prefix = "Hello ";
Function<String, String> greeter = prefix::concat;         // s -> prefix.concat(s)
greeter.apply("World");   // "Hello World"

// 3. Instance method, any object of the type
Function<String, String> toUpper = String::toUpperCase;    // s -> s.toUpperCase()
toUpper.apply("hello");   // "HELLO"

// This is the tricky one, first param becomes the object:
BiFunction<String, String, Boolean> checker = String::startsWith;
// equivalent to: (s1, s2) -> s1.startsWith(s2)
checker.apply("Hello", "He");   // true

// 4. Constructor reference
Supplier<ArrayList> newList = ArrayList::new;               // () -> new ArrayList()
Function<String, Integer> toInt = Integer::new;            // s -> new Integer(s)
```

### When to use method reference vs lambda

```java
// Method reference, when lambda just passes args to a method
list.forEach(System.out::println);          // cleaner
list.forEach(s -> System.out.println(s));   // same but more verbose

// Lambda, when you need extra logic
list.forEach(s -> System.out.println("Item: " + s));  // can't use method ref here
```

## Effectively Final Variables

Lambdas can capture local variables, but those variables must be **effectively final**: assigned once and never changed.

```java
int multiplier = 3;                              // effectively final
Function<Integer, Integer> fn = n -> n * multiplier;  // OK

int counter = 0;
// counter++;                                    // if uncommented, lambda below fails
// Consumer<String> c = s -> counter++;          // compile error: counter is not effectively final
```

**Why?** Lambdas capture a **copy** of the variable, not the variable itself. If the variable could change after capture, the copy and original would go out of sync.

**Workaround**: use an array or AtomicInteger:

```java
int[] counter = {0};                             // array is final, contents are not
list.forEach(s -> counter[0]++);                 // OK

AtomicInteger count = new AtomicInteger(0);
list.forEach(s -> count.incrementAndGet());      // OK: thread-safe version
```

## Q&A

1. **How does the `this` keyword behave differently in a Lambda expression versus an Anonymous Inner Class?**
   In a Lambda expression, the `this` keyword refers to the **enclosing class** instance. A Lambda does not create its own scope or its own `this` context. In contrast, an anonymous inner class creates a new scope, and `this` refers to the **anonymous class instance** itself. Furthermore, an anonymous class can have its own instance fields and state, while a Lambda cannot.

   ```java
   class Outer {
       Runnable lambda = () -> System.out.println(this);  // Refers to 'Outer' instance
       Runnable anon = new Runnable() {
           int count = 0;                                  // OK: Anonymous class can have state
           public void run() { count++; System.out.println(this); }  // Refers to 'anon' instance
       };
   }
   ```

2. **Can a functional interface inherit from another interface?**
   Yes. An interface is still considered a functional interface as long as the final result of the inheritance hierarchy contains exactly **one abstract method**.

3. **What is a common point of confusion when using method references like `String::toUpperCase`?**
   It looks like a static method call, but `toUpperCase` is actually an instance method. In this context, Java treats the first parameter passed to the functional interface as the object on which to call the method. For example, `Function<String, String> f = String::toUpperCase;` is equivalent to the lambda `s -> s.toUpperCase()`.

4. **Why can't you use a Lambda expression with `var` without a target type?**
   A Lambda has no type of its own; its type is inferred from the functional interface it is being assigned to (the "target type"). Without a declared target type, the compiler has no way to determine which functional interface the Lambda is intended to implement.

   ```java
   var fn = (a, b) -> a + b;          // compile error: No target type for inference
   Comparator<String> fn = (a, b) -> a.compareTo(b);  // OK: Target type is explicit
   ```

5. **Why can't you throw checked exceptions from within a standard Lambda like a `Function` or `Consumer`?**
   A Lambda expression is an implementation of a functional interface's Single Abstract Method (SAM). If that method's signature does not declare a `throws` clause, the Lambda cannot throw any checked exceptions. Most built-in Java functional interfaces (like `Predicate` or `Function`) do not declare any exceptions. To use code that throws a checked exception inside a lambda, you must either catch the exception and wrap it in a `RuntimeException` or define your own functional interface that declares the exception.

6. **What is a common syntax error when writing a method reference?**
   A common mistake is including parentheses at the end of the method name. A method reference is a reference to the **method itself**, not a call to it. Including `()` attempts to invoke the method immediately, which will result in a compilation error.

   ```java
   list.forEach(System.out::println);    // OK: Correct method reference
   list.forEach(System.out::println());  // compile error: parentheses not allowed
   ```

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Functional Interface|Exactly 1 abstract method (SAM), enables lambda|
|@FunctionalInterface|Optional annotation, compiler enforces SAM rule|
|Lambda|Anonymous function, implements the SAM|
|Method Reference|Shorthand for lambda, 4 types: static, instance (specific), instance (any), constructor|
|Effectively Final|Local variable used in lambda must never be reassigned|
|Built-in FIs|Predicate→boolean, Function→R, Consumer→void, Supplier→T|
|`this` in lambda|Refers to enclosing class (not like anonymous class)|
|Primitive variants|IntPredicate, LongFunction, etc., avoid autoboxing|
|Chaining|Predicate: and/or/negate, Function: andThen/compose, Consumer: andThen|
