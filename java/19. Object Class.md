`java.lang.Object` is the implicit root of every class hierarchy: every class extends it directly or indirectly.

Every class inherits the methods below from `Object`. The first three are the ones you routinely override (debugging, collections, correctness); the rest are situational and covered in their own notes.

| Method                                 | Purpose                                       | Covered in                                                   |
| :------------------------------------- | :-------------------------------------------- | :----------------------------------------------------------- |
| `toString()`, `equals()`, `hashCode()` | String form, equality, hashing                | This note                                                    |
| `clone()`                              | Shallow copy (requires `Cloneable` marker)    | [Encapsulation](<13. Encapsulation & Access Control.md>)     |
| `getClass()`                           | Runtime class (`final`, can't override)       | [Inheritance](<14. Inheritance & Relationships.md>)          |
| `wait()`, `notify()`, `notifyAll()`    | Thread coordination on the object's monitor   | [Multithreading](<25. Multithreading & Concurrency.md>)      |
| `finalize()`                           | Pre-GC hook; deprecated (Java 9), for removal | N/A                                                          |

## `toString()`

Returns a String representation of the object. The default is `ClassName@hexHashCode`, rarely useful.

```java
class Employee {
    String name;
    String email;
}

Employee e = new Employee();
System.out.println(e);    // "Employee@1a2b3c", useless
```

Override it to return something meaningful:

```java
class Employee {
    String name;
    String email;

    @Override
    public String toString() {
        return "Employee{name='" + name + "', email='" + email + "'}";
    }
}

System.out.println(e);    // "Employee{name='Alice', email='alice@corp.com'}"
```

- **Called**: directly via `obj.toString()`; implicitly by `System.out.println(obj)`.
- **Null-safe printing**: `Objects.toString(obj, defaultIfNull)` returns the fallback when the reference is null.

```java
Objects.toString(e, "N/A");       // uses e.toString()
Objects.toString(null, "N/A");    // "N/A"
```

## `equals()`

The default is reference equality (same as `==`). Override to compare objects by **logical equality** (their contents).

| Operator    | Compares                                           |
| :---------- | :------------------------------------------------- |
| `==`        | Primitives by value; objects by reference          |
| `.equals()` | Objects by content/logical equality, if overridden |

```java
String a = new String("hello");
String b = new String("hello");

a == b         // false, different objects
a.equals(b)    // true, same content (String overrides equals)

Employee e1 = new Employee("Alice", "alice@corp.com");
Employee e2 = new Employee("Alice", "alice@corp.com");
e1 == e2       // false
e1.equals(e2)  // false, Employee hasn't overridden equals, falls back to reference equality
```

Override it to compare by content:

```java
class Employee {
    String name;
    String email;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;                  // same reference, equal
        if (!(o instanceof Employee)) return false;  // type check (also handles null)
        Employee e = (Employee) o;
        return Objects.equals(email, e.email) && Objects.equals(name, e.name);
    }
}
```

> **Note**: use `Objects.equals(a, b)` for null-safe comparison; it handles null on either side (no NPE). Internally: `(a == b) || (a != null && a.equals(b))`.

> **Reference**: Java 16+ folds the type check and cast into one step (`o instanceof Employee e`); see [Java Modern Features](<29. Java Modern Features (10–21+).md>).

## `hashCode()`

Returns an `int` hash code for the object. The default derives it from the object's identity (roughly its memory address), not its field values, so two logically equal objects get different hash codes unless `hashCode()` is overridden.

Override it alongside `equals()`, using the same fields:

```java
class Employee {
    String name;
    String email;

    @Override
    public int hashCode() {
        return Objects.hash(name, email);    // Java 7+, clean and correct
    }
}
```

`Objects.hash(...)` combines the hashes of its arguments with a standard algorithm, equivalent to writing the combining code by hand.

> **Note**: if `a.equals(b)` is true, then `a.hashCode() == b.hashCode()` must also be true, so always override both using the same fields. The reverse isn't required (equal hashes for unequal objects is a normal collision).

## Q&A

1. **What breaks if you override `equals()` but not `hashCode()`?**
You break the equals/hashCode contract: two equal objects can produce different hash codes, so hash-based collections (`HashMap`, `HashSet`) probe the wrong bucket and silently fail to find, get, or remove the entry.

2. **Why does `==` sometimes pass for Strings, and why is that risky?**
`==` compares references. Interned literals (`"hello"`) share one pooled object, so `==` may accidentally return `true`; a `new String(...)` or runtime-built value is a distinct object where `==` is `false` for identical text. Always use `.equals()`.

3. **Why does `==` on `Integer` work for `127` but not `128`?**
Autoboxing reuses cached `Integer` instances for `-128..127`, so boxed values in range share one object; outside it each box is new. Never compare wrappers with `==`.

```java
Integer a = 127, b = 127;
System.out.println(a == b);   // true (cached)

Integer c = 128, d = 128;
System.out.println(c == d);   // false (new objects)
```

4. **What is the classic signature mistake when overriding `equals()`?**
Declaring `equals(Employee o)` instead of `equals(Object o)`: that is an overload, not an override, so framework code (`List.contains`, `HashMap`) calls `Object.equals` and ignores yours. `@Override` catches it at compile time.

## Quick Revision Cheat Sheet

| Concept                  | Remember                                                                                     |
| :----------------------- | :------------------------------------------------------------------------------------------- |
| `toString()`             | Default `Class@hexHash`; override for meaningful debug/log output                            |
| `==` vs `.equals()`      | `==` compares references (primitives by value); `.equals()` compares content when overridden |
| equals() contract        | Reflexive, symmetric, transitive, consistent, null-safe                                      |
| equals() signature       | Param must be `Object`, not your type, else it's an overload `@Override` won't bind          |
| `hashCode()`             | Default = identity (memory-based); override with the same fields as `equals()`               |
| equals/hashCode contract | `a.equals(b)` ⇒ `a.hashCode() == b.hashCode()` (reverse not required); else hash lookups fail |
| Integer cache            | `==` on boxed `Integer` true only for `-128..127`; use `.equals()`                           |
| String identity          | Literals are pooled (`==` may pass); `new String()` is distinct; compare with `.equals()`     |
| Null-safe helpers        | `Objects.equals(a, b)`, `Objects.hash(...)`, `Objects.toString(o, default)`                   |
