An **exception** is an unexpected event that occurs during the execution of a program and disrupts its normal flow of instructions. Technically, it is an **object** that encapsulates information about an error, such as its **type** and the **state of the program** when it occurred.

Exception handling lets you catch these events gracefully instead of crashing.

```java
// Without handling, this throws ArithmeticException and the program terminates
int result = 10 / 0;

// With handling, program continues
try {
    int result = 10 / 0;
} catch (ArithmeticException e) {
    System.out.println("Can't divide by zero");
}
System.out.println("Program continues");  // still runs
```

## Exception Hierarchy

```
Throwable
├── Error
│   ├── OutOfMemoryError
│   ├── StackOverflowError
│   └── VirtualMachineError
│
└── Exception
    ├── IOException           ──┐
    ├── SQLException            │
    ├── FileNotFoundException   │  ← checked
    ├── ClassNotFoundException  │
    ├── InterruptedException  ──┘
    │
    └── RuntimeException  ← unchecked
        ├── NullPointerException
        ├── ArrayIndexOutOfBoundsException
        ├── ClassCastException
        ├── IllegalArgumentException
        ├── ArithmeticException
        ├── NumberFormatException
        ├── ConcurrentModificationException
        ├── UnsupportedOperationException
        └── IllegalStateException
```

- **`Throwable`**: root of all exceptions and errors; catching it directly is discouraged.
- **`Exception`**: your problem, application-level and generally recoverable; this is what you handle.
- **`Error`**: the JVM's problem, system-level and unrecoverable (e.g., `OutOfMemoryError`); never catch it.

> **Reference**: this is a class-inheritance tree (each type `extends` its parent), the same IS-A mechanism custom exceptions use; inheritance is covered in [14. Inheritance & Relationships](<14. Inheritance & Relationships.md>).

## Checked vs Unchecked Exceptions

"Checked" is a category, not a class. Any subclass of `Exception` except `RuntimeException` is checked (the compiler forces you to handle or declare it); `RuntimeException` and its subclasses are unchecked (optional to handle).

| Feature                | Checked Exception                                                                                | Unchecked Exception                                                                                        |
| :--------------------- | :----------------------------------------------------------------------------------------------- | :--------------------------------------------------------------------------------------------------------- |
| **Compiler Check**     | Verified at **compile-time**. The compiler ensures they are handled.                             | Ignored at compile-time. Verified only at **runtime**.                                                     |
| **Mandatory Handling** | Yes. You **must** use a try-catch block or declare them with `throws`.                           | No. Handling is optional; the program will compile without explicit `catch`/`throws`.                      |
| **Class Hierarchy**    | Extends `Exception` (but **not** `RuntimeException`).                                            | Extends `RuntimeException` or `Error`.                                                                     |
| **Typical Cause**      | External factors outside the program's control (e.g., missing file, network failure).            | Programming logic errors (e.g., null pointers, dividing by zero, array index out of bounds).               |
| **Recovery**           | Generally considered **recoverable**.                                                            | Generally considered **unrecoverable** programming bugs.                                                   |
| **Examples**           | `IOException`, `SQLException`, `FileNotFoundException`                                           | `NullPointerException`, `ClassCastException`, `ArithmeticException`                                        |

```java
// Checked, compiler forces you to handle
public void readFile() throws IOException {     // must declare
    FileReader fr = new FileReader("file.txt"); // may throw FileNotFoundException
}

// Unchecked, compiler doesn't care
public void process(String s) {
    s.length();   // may throw NullPointerException, no handling required
}
```

## try-catch-finally

```java
try {
    // risky code
    int result = 10 / 0;
} catch (ArithmeticException e) {
    // handle specific exception
    System.out.println("Error: " + e.getMessage());
} finally {
    // ALWAYS runs (exception or not), used for cleanup: close files, connections, etc.
    System.out.println("Cleanup done");
}
```

### Multiple catch blocks

```java
try {
    // risky code
} catch (FileNotFoundException e) {
    // most specific first
} catch (IOException e) {
    // more general second
} catch (Exception e) {
    // most general last
}
// compile error: putting general before specific → unreachable code
```

### Multi-catch (Java 7+)

```java
// Multiple exceptions in one catch, separated by |
try {
    // risky code
} catch (IOException | SQLException e) {
    // handle both the same way
    System.out.println("Error: " + e.getMessage());
}
```

> **Note**: 
> - Exceptions in a multi-catch can't be parent-child, so `catch (FileNotFoundException | IOException e)` fails to compile (`FileNotFoundException` is a subclass of `IOException`).
> - `finally` always runs, even when `try`/`catch` executes a `return` or an uncaught exception propagates. It is skipped only if `System.exit()` is called or the JVM crashes.

## try-with-resources (Java 7+)

Automatically closes any resource implementing `AutoCloseable`/`Closeable` when the block exits, replacing the error-prone manual `finally { close() }` pattern. Multiple resources close in **reverse order of declaration**, and if both the body and `close()` throw, the `close()` exception is suppressed instead of masking the original.

```java
// Before Java 7, manual close in finally (verbose, easy to get wrong)
BufferedReader br = null;
try {
    br = new BufferedReader(new FileReader("file.txt"));
    String line = br.readLine();
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (br != null) {
        try { br.close(); } catch (IOException e) { }  // ugly nested try
    }
}

// Java 7+, try-with-resources, br.close() called automatically, even on exception
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    String line = br.readLine();
} catch (IOException e) {
    e.printStackTrace();
}

// Multiple resources, closed in reverse order (fos first, then fis)
try (FileInputStream fis = new FileInputStream("in.txt");
     FileOutputStream fos = new FileOutputStream("out.txt")) {
    // ...
}

// Java 9+, can use a pre-declared effectively-final variable
BufferedReader br = new BufferedReader(new FileReader("file.txt"));
try (br) {                // no need to re-declare inside try
    String line = br.readLine();
}
```

> **Note**: prefer declaring the resource inside `try(...)`. If it is constructed before the try block and that construction throws, auto-close won't apply.

### Suppressed Exceptions

```java
// If the body throws AND close() throws, close()'s exception is suppressed (not lost)
try (MyResource r = new MyResource()) {
    throw new Exception("try failed");
} catch (Exception e) {
    System.out.println(e.getMessage());        // "try failed" (body's exception wins)
    System.out.println(e.getSuppressed()[0]);  // the close() exception
}
```

### AutoCloseable vs Closeable

The resource class **must** implement one of these interfaces for try-with-resources to work.

| Interface           | Package     | `close()` throws         | Purpose                                                                   |
| :------------------ | :---------- | :----------------------- | :------------------------------------------------------------------------ |
| **`AutoCloseable`** | `java.lang` | `Exception` (broader)    | Parent, introduced in Java 7 for try-with-resources. Any resource type    |
| **`Closeable`**     | `java.io`   | `IOException` (narrower) | Extends `AutoCloseable`, specifically for I/O streams (older, since Java 5) |

- Built-in classes like `BufferedReader`, `FileInputStream`, `Connection`, `Scanner` already implement `Closeable` or `AutoCloseable`

**Writing your own**: use `AutoCloseable` for general resources (DB connections, locks); use `Closeable` only for I/O that throws `IOException`

```java
class MyResource implements AutoCloseable {
    @Override
    public void close() throws Exception {
        System.out.println("Resource closed");
    }
}

try (MyResource r = new MyResource()) {
    // use resource
}  // r.close() called automatically
```

## throw vs throws

|             | `throw`                       | `throws`                                |
| :---------- | :---------------------------- | :-------------------------------------- |
| What        | Explicitly throw an exception | Declare that method may throw           |
| Where       | Inside method body            | In method signature                     |
| Followed by | Exception object              | Exception class name(s)                 |

```java
// throw, create and throw an exception inside the method body
public void validate(int age) {
    if (age < 0) {
        throw new IllegalArgumentException("Age can't be negative");
    }
}

// throws, declare in the signature that the method may throw
public void readFile() throws IOException {
    // ...
}
```

**Exception Propagation**: an unhandled checked exception must be declared with `throws` and propagates up the call stack until someone catches it.

```java
public void methodA() throws IOException {
    methodB();
}

public void methodB() throws IOException {
    throw new IOException("failed");
}

public void caller() {
    try {
        methodA();
    } catch (IOException e) {
        // caught and handled here
    }
}
```

> **Note**: don't declare redundant subclasses in `throws`; `FileNotFoundException` is already covered by `IOException`.

## Custom Exceptions

```java
// Checked custom exception, extends Exception
class InsufficientFundsException extends Exception {
    private static final long serialVersionUID = 1L;  // needed if exception is serialized
    private double amount;

    public InsufficientFundsException(String message, double amount) {
        super(message);
        this.amount = amount;
    }

    public double getAmount() { return amount; }
}

// Unchecked custom exception, extends RuntimeException
class InvalidUserException extends RuntimeException {
    public InvalidUserException(String message) {
        super(message);
    }

    public InvalidUserException(String message, Throwable cause) {
        super(message, cause);   // chaining, wraps original exception
    }
}

// Usage
public void withdraw(double amount) throws InsufficientFundsException {
    if (amount > balance) {
        throw new InsufficientFundsException("Not enough balance", amount);
    }
}
```

**When to create**:
- Built-in exceptions don't convey the right meaning.
- You need to attach extra fields (`amount`, `errorCode`).

**Checked or unchecked**:
- Checked: caller can reasonably recover (retry, fallback, ask user).
- Unchecked: programming error, or nothing useful the caller can do.

## Exception Chaining

Wrap a low-level exception in a higher-level one while preserving the original via the cause constructor. Lower layers throw technical exceptions (SQL, IO); upper layers rethrow meaningful business exceptions without losing the root cause (retrievable via `getCause()`).

```java
try {
    // database operation
} catch (SQLException e) {
    throw new ServiceException("Failed to save user", e);  // e preserved as the cause
}
```

**Wrapping checked as unchecked**: convert a checked exception to unchecked so callers aren't forced to declare `throws` (Spring does this everywhere, e.g. `SQLException` → `DataAccessException`).

- Checked exceptions in lower layers (JDBC, IO) pollute every signature up the stack.
- Business code rarely handles them where they occur; wrapping lets them reach a centralized handler with the cause preserved.

```java
public User findUser(long id) {
    try {
        return jdbc.query("SELECT * FROM users WHERE id = ?", id);
    } catch (SQLException e) {
        throw new RuntimeException("DB lookup failed for id=" + id, e);  // wrap as unchecked
    }
}
```

> **Note**: only wrap as unchecked when the caller can't meaningfully recover. If recovery is genuine (retry, fallback, ask user), keep it checked so the compiler enforces handling.

## Exception Handling with Inheritance

```java
class Parent {
    void doWork() throws IOException { }
}

class Child extends Parent {
    // OK, same exception
    void doWork() throws IOException { }

    // OK, narrower (subclass) exception
    void doWork() throws FileNotFoundException { }

    // OK, no exception at all
    void doWork() { }

    // compile error, broader (parent) exception
    void doWork() throws Exception { }

    // compile error, new checked exception
    void doWork() throws SQLException { }

    // OK, can add unchecked exceptions freely
    void doWork() throws IOException, NullPointerException { }
}
```

> **Note**:
> - Override can throw the **same, a narrower, or no** checked exception, never broader or a new checked one; unchecked exceptions are unrestricted.
> - Callers only catch what the parent declared (`throws IOException`), so a broader child exception (`throws Exception`) would escape their catch and break the contract.

> **Reference**: this is a method-overriding rule; overriding and its full signature constraints are covered in [15. Polymorphism](<15. Polymorphism.md>).

## Q&A

1. **Does the `finally` block execute if the `try` block contains a `return` statement?**
Yes, even when `try` has a `return`. But a `return` in `finally` overrides the try's return and silently swallows any pending exception, so avoid returning from `finally`.

```java
public int test() {
    try { return 1; }
    finally { return 2; }  // Returns 2, not 1!
}

// Exception lost, finally's return swallows it
try {
    throw new IOException();
} finally {
    return;  // warning: IOException is silently discarded and the method returns normally
}
```

2. **Under what rare circumstances does a `finally` block NOT execute?**
It won't run if `System.exit()` is called, the JVM crashes, or the executing thread is killed.

3. **Is `StackOverflowError` an Exception or an Error, and should you catch it?**
It is an `Error`, not an `Exception` (typically from infinite recursion). Don't catch `Error` types; they signal unrecoverable system-level failures.

4. **What improvement was made to `NullPointerException` messages in Java 14?**
Java 14 introduced "Helpful NullPointerExceptions," which explicitly name the variable that was `null`. Previously, you only got the line number, which made debugging difficult if multiple variables on the same line could have been `null`.

```java
String s = null;
s.length();   // Java 14+: "Cannot invoke 'String.length()' because 's' is null"
```

5. **Why might a `catch` block cause a compilation error even if the syntax is correct?**
In Java, it is a compilation error to catch a **checked exception** if the code within the corresponding `try` block cannot possibly throw that exception. (Note: This rule does not apply to `Exception`, `Throwable`, or unchecked exceptions like `RuntimeException`).

```java
try {
    System.out.println("hello");
} catch (IOException e) { }   // compile error: println doesn't throw IOException
```

6. **What is the difference between `final`, `finally`, and `finalize`?**
- **`final`**: keyword that makes a variable constant, prevents method overriding, or prevents class inheritance.
- **`finally`**: block that runs unconditionally after `try`/`catch` for cleanup.
- **`finalize`**: obsolete `Object` method (deprecated in Java 9), once called by the garbage collector before an object was destroyed.

## Quick Revision Cheat Sheet

| Concept             | Remember                                                       |
| :------------------ | :------------------------------------------------------------- |
| Checked exception   | Must handle (try-catch) or declare (throws). External failures |
| Unchecked exception | Optional to handle. Programming bugs. Extends RuntimeException |
| Error               | Don't catch. JVM/system level. Unrecoverable                   |
| try-with-resources  | Auto-closes AutoCloseable/Closeable resources. Java 7+         |
| Suppressed          | body's exception wins; close()'s read via `getSuppressed()`    |
| throw vs throws     | `throw` = raise (in body); `throws` = declare (in signature)   |
| finally             | Always runs (except System.exit/JVM crash). Cleanup code       |
| Multi-catch         | `catch(A &#124; B e)`; Java 7+, can't be parent-child          |
| Custom exception    | Extend Exception (checked) or RuntimeException (unchecked)     |
| Exception chaining  | `throw new X("msg", cause)` preserves the original exception   |
| Override rule       | Same, narrower, or no checked exception; never broader         |
| Best practice       | Catch specific, don't swallow, throw early, catch late         |
