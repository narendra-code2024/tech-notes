An **exception** is an unexpected event during program execution that disrupts the normal flow of instructions. Technically it is an **object** carrying information about the error, including its **type** and the **program state** when it occurred.

Exception handling lets you respond to these events gracefully instead of crashing.

```java
// Without handling, this throws ArithmeticException and the program terminates
int result = 10 / 0;

// With handling, the program continues
try {
    int result = 10 / 0;
} catch (ArithmeticException e) {
    System.out.println("Can't divide by zero");
}
System.out.println("Program continues");  // still runs
```

## Checked vs Unchecked Exceptions

"Checked" is a category, not a class. Any subclass of `Exception` except `RuntimeException` is checked (the compiler forces you to handle or declare it); `RuntimeException` and its subclasses are unchecked (handling optional).

| Feature                | Checked Exception                                                         | Unchecked Exception                                                       |
| :--------------------- | :------------------------------------------------------------------------ | :------------------------------------------------------------------------ |
| **Compiler check**     | Verified at compile time; the compiler ensures they are handled.          | Ignored at compile time; surfaces only at runtime.                        |
| **Mandatory handling** | Yes, via try-catch or a `throws` declaration.                             | No, the program compiles without `catch`/`throws`.                        |
| **Class hierarchy**    | Extends `Exception` but not `RuntimeException`.                           | Extends `RuntimeException` or `Error`.                                    |
| **Typical cause**      | External factors outside the program (missing file, network failure).     | Programming logic errors (null pointer, divide by zero, bad index).       |
| **Recovery**           | Generally recoverable.                                                    | Generally unrecoverable programming bugs.                                 |
| **Examples**           | `IOException`, `SQLException`, `FileNotFoundException`                     | `NullPointerException`, `ClassCastException`, `ArithmeticException`        |

```java
// Checked, the compiler forces you to handle it
public void readFile() throws IOException {     // must declare
    FileReader fr = new FileReader("file.txt"); // may throw FileNotFoundException
}

// Unchecked, the compiler doesn't care
public void process(String s) {
    s.length();   // may throw NullPointerException, no handling required
}
```

## try-catch-finally

```java
try {
    int result = 10 / 0;            // risky code
} catch (ArithmeticException e) {
    System.out.println("Error: " + e.getMessage());   // handle a specific exception
} finally {
    System.out.println("Cleanup done");  // ALWAYS runs, for cleanup: close files, connections
}
```

### Multiple catch blocks

```java
try {
    // risky code
} catch (FileNotFoundException e) {
    // most specific first
} catch (IOException e) {
    // more general second
} catch (Exception e) {
    // most general last
}
// compile error: a general type before a specific one makes the specific catch unreachable
```

### Multi-catch (Java 7+)

```java
// Multiple exception types in one catch, separated by |
try {
    // risky code
} catch (IOException | SQLException e) {
    System.out.println("Error: " + e.getMessage());   // handle both the same way
}
```

> **Note**: the alternatives in a multi-catch cannot be parent-child, so `catch (FileNotFoundException | IOException e)` fails to compile because `FileNotFoundException` already is an `IOException`.

## try-with-resources (Java 7+)

Automatically closes any resource implementing `AutoCloseable`/`Closeable` when the block exits, replacing the error-prone manual `finally { close() }`. Multiple resources close in **reverse order of declaration**, and if both the body and `close()` throw, the `close()` exception is suppressed rather than masking the original.

```java
// Before Java 7, manual close in finally (verbose, easy to get wrong)
BufferedReader br = null;
try {
    br = new BufferedReader(new FileReader("file.txt"));
    String line = br.readLine();
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (br != null) {
        try { br.close(); } catch (IOException e) { }  // nested try, ugly
    }
}

// Java 7+, try-with-resources, br.close() called automatically, even on exception
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    String line = br.readLine();
} catch (IOException e) {
    e.printStackTrace();
}

// Multiple resources close in reverse order (fos first, then fis)
try (FileInputStream fis = new FileInputStream("in.txt");
     FileOutputStream fos = new FileOutputStream("out.txt")) {
    // ...
}

// Java 9+, an effectively-final variable can be used directly
BufferedReader br = new BufferedReader(new FileReader("file.txt"));
try (br) {                // no need to re-declare inside try
    String line = br.readLine();
}
```

> **Note**: prefer declaring the resource inside `try(...)`. If it is constructed before the try block and that construction throws, auto-close does not apply.

### Suppressed Exceptions

```java
// If the body throws AND close() throws, close()'s exception is suppressed (not lost)
try (MyResource r = new MyResource()) {
    throw new Exception("try failed");
} catch (Exception e) {
    System.out.println(e.getMessage());        // "try failed" (the body's exception wins)
    System.out.println(e.getSuppressed()[0]);  // the close() exception
}
```

### AutoCloseable vs Closeable

The resource class **must** implement one of these for try-with-resources to work.

| Interface           | Package     | `close()` throws         | Purpose                                                               |
| :------------------ | :---------- | :----------------------- | :------------------------------------------------------------------- |
| **`AutoCloseable`** | `java.lang` | `Exception` (broader)    | Parent, added in Java 7 for try-with-resources; any resource type    |
| **`Closeable`**     | `java.io`   | `IOException` (narrower) | Extends `AutoCloseable`; for I/O streams (older, since Java 5)        |

Built-in classes like `BufferedReader`, `FileInputStream`, `Connection`, and `Scanner` already implement `Closeable` or `AutoCloseable`. For your own resources, implement `AutoCloseable` for general use (DB connections, locks) and `Closeable` only for I/O that throws `IOException`:

```java
class MyResource implements AutoCloseable {
    @Override
    public void close() throws Exception {
        System.out.println("Resource closed");
    }
}

try (MyResource r = new MyResource()) {
    // use resource
}  // r.close() called automatically
```

## throw vs throws

|             | `throw`                       | `throws`                      |
| :---------- | :---------------------------- | :---------------------------- |
| What        | Explicitly throw an exception | Declare a method may throw    |
| Where       | Inside the method body        | In the method signature       |
| Followed by | An exception object           | Exception class name(s)       |

```java
// throw: create and raise an exception in the body
public void validate(int age) {
    if (age < 0) {
        throw new IllegalArgumentException("Age can't be negative");
    }
}

// throws: declare in the signature that the method may throw
public void readFile() throws IOException {
    // ...
}
```

An unhandled checked exception must be declared with `throws` and propagates up the call stack until something catches it.

```java
public void methodA() throws IOException {
    methodB();
}

public void methodB() throws IOException {
    throw new IOException("failed");
}

public void caller() {
    try {
        methodA();
    } catch (IOException e) {
        // caught and handled here
    }
}
```

> **Note**: don't declare redundant subclasses in `throws`; `FileNotFoundException` is already covered by `IOException`.

## Custom Exceptions

```java
// Checked custom exception, extends Exception
class InsufficientFundsException extends Exception {
    private static final long serialVersionUID = 1L;  // needed if the exception is serialized
    private double amount;

    public InsufficientFundsException(String message, double amount) {
        super(message);
        this.amount = amount;
    }

    public double getAmount() { return amount; }
}

// Unchecked custom exception, extends RuntimeException
class InvalidUserException extends RuntimeException {
    public InvalidUserException(String message) {
        super(message);
    }

    public InvalidUserException(String message, Throwable cause) {
        super(message, cause);   // chaining, wraps the original exception
    }
}

// Usage
public void withdraw(double amount) throws InsufficientFundsException {
    if (amount > balance) {
        throw new InsufficientFundsException("Not enough balance", amount);
    }
}
```

Create a custom exception when a built-in one doesn't convey the right meaning, or when you need to attach extra fields (`amount`, `errorCode`). Choose the category by recoverability: make it checked when the caller can reasonably recover (retry, fallback, ask the user), and unchecked when it represents a programming error the caller can do nothing about.

## Exception Chaining

Wrap a low-level exception in a higher-level one while preserving the original through the cause constructor. Lower layers throw technical exceptions (SQL, IO); upper layers rethrow meaningful business exceptions without losing the root cause (retrievable via `getCause()`).

```java
try {
    // database operation
} catch (SQLException e) {
    throw new ServiceException("Failed to save user", e);  // e preserved as the cause
}
```

Wrapping a checked exception as unchecked lets callers avoid declaring `throws` (Spring does this everywhere, e.g. `SQLException` ŌåÆ `DataAccessException`):

- Checked exceptions in lower layers (JDBC, IO) pollute every signature up the stack.
- Business code rarely handles them where they occur; wrapping carries them to a centralized handler with the cause preserved.

```java
public User findUser(long id) {
    try {
        return jdbc.query("SELECT * FROM users WHERE id = ?", id);
    } catch (SQLException e) {
        throw new RuntimeException("DB lookup failed for id=" + id, e);  // wrap as unchecked
    }
}
```

> **Note**: only wrap as unchecked when the caller can't meaningfully recover. If recovery is genuine (retry, fallback, ask the user), keep it checked so the compiler enforces handling.

## Exception Handling with Inheritance

```java
class Parent {
    void doWork() throws IOException { }
}

class Child extends Parent {
    void doWork() throws IOException { }            // OK, same exception
    void doWork() throws FileNotFoundException { }  // OK, narrower (subclass)
    void doWork() { }                               // OK, no exception
    void doWork() throws Exception { }              // compile error, broader
    void doWork() throws SQLException { }           // compile error, new checked exception
    void doWork() throws IOException, NullPointerException { } // OK, unchecked can be added freely
}
```

An override may declare the **same, a narrower, or no** checked exception, never a broader or new checked one (unchecked exceptions are unrestricted). The reason: callers only catch what the parent declared (`throws IOException`), so a broader child exception (`throws Exception`) would escape their handler and break the contract.

> **Reference**: this is a method-overriding rule; overriding and its full signature constraints are covered in [Polymorphism](<15. Polymorphism.md>).

## Q&A

1. **Does `finally` run if `try` contains a `return`?**
Yes, even when `try` returns. But a `return` inside `finally` overrides the try's return and silently swallows any pending exception, so never return from `finally`.

```java
public int test() {
    try { return 1; }
    finally { return 2; }  // returns 2, not 1
}

try {
    throw new IOException();
} finally {
    return;  // the IOException is silently discarded and the method returns normally
}
```

2. **When does `finally` not execute?**
Only if `System.exit()` is called, the JVM crashes, or the executing thread is killed.

3. **Is `StackOverflowError` an Exception or an Error, and should you catch it?**
It is an `Error` (typically from infinite recursion), not an `Exception`. Don't catch `Error` types; they signal unrecoverable system-level failures.

4. **What changed about `NullPointerException` messages in Java 14?**
Java 14 added Helpful NullPointerExceptions, which name the exact variable that was `null`. Previously you got only the line number, which was ambiguous when several variables on a line could be `null`.

```java
String s = null;
s.length();   // Java 14+: "Cannot invoke 'String.length()' because 's' is null"
```

5. **Why can a syntactically valid `catch` cause a compile error?**
It is a compile error to catch a checked exception that the `try` body cannot possibly throw. (This does not apply to `Exception`, `Throwable`, or unchecked types.)

```java
try {
    System.out.println("hello");
} catch (IOException e) { }   // compile error: println doesn't throw IOException
```

6. **What is the difference between `final`, `finally`, and `finalize`?**

- **`final`**: a keyword that makes a variable constant, prevents method overriding, or prevents class inheritance.
- **`finally`**: a block that runs after `try`/`catch` for cleanup.
- **`finalize`**: an obsolete `Object` method (deprecated in Java 9) the GC once called before reclaiming an object.

## Quick Revision Cheat Sheet

| Concept             | Remember                                                       |
| :------------------ | :------------------------------------------------------------- |
| Checked exception   | Must handle (try-catch) or declare (`throws`); external failures |
| Unchecked exception | Optional to handle; programming bugs; extends `RuntimeException` |
| Error               | Don't catch; JVM/system level; unrecoverable                   |
| try-with-resources  | Auto-closes `AutoCloseable`/`Closeable`; Java 7+               |
| Suppressed          | Body's exception wins; read `close()`'s via `getSuppressed()`  |
| throw vs throws     | `throw` raises (in body); `throws` declares (in signature)     |
| finally             | Always runs (except `System.exit`/JVM crash); cleanup code     |
| Multi-catch         | `catch (A &#124; B e)`; Java 7+; cannot be parent-child        |
| Custom exception    | Extend `Exception` (checked) or `RuntimeException` (unchecked) |
| Exception chaining  | `throw new X("msg", cause)` preserves the original             |
| Override rule       | Same, narrower, or no checked exception; never broader         |
| Best practice       | Catch specific, don't swallow, throw early, catch late         |
