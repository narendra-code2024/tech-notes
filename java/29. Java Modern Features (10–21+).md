## `var`, Local Variable Type Inference (Java 10)

Compiler infers the type, you write `var` instead of the full type. Only for **local variables**.

```java
// Before
List<String> list = new ArrayList<>();
Map<String, List<Integer>> map = new HashMap<>();

// With var, cleaner
var list = new ArrayList<String>();
var map = new HashMap<String, List<Integer>>();
var stream = list.stream().filter(s -> s.length() > 3);
```

**Rules**: Local variables only, NOT for fields, method params, or return types
- Must be initialized at declaration, `var x;` is invalid
- Not a keyword, it's a reserved type name (can't be used as a class name, but can be used as a variable name in older code)
- Type is fixed at compile time, it's NOT dynamic typing

```java
var x = "hello";     // x is String, fixed at compile time
x = 123;             // compile error: x is String, not int

// Cannot use var for
var x;               // no initializer
var x = null;        // can't infer type from null
var arr = {1, 2, 3}; // can't infer array type
var lambda = () -> "hello";  // can't infer, lambda needs target type
```

**When to use:** Long generic types. **When to avoid:** When type isn't obvious from the right side, readability > brevity.

## Records (Java 16)

Immutable data carriers, replaces boilerplate POJOs. Compiler generates constructor, getters, `equals()`, `hashCode()`, `toString()` automatically.

### Why records were added (the pre-Java 16 pain)

- **A 4-field DTO required ~40-50 lines of boilerplate**: private final fields, constructor, getter per field, `equals()`, `hashCode()`, `toString()`. For a simple *"holder of data"* class, 90% of code was mechanical ceremony.
- **Equals/hashCode were easy to get wrong**: hand-writing consistent `equals()` and `hashCode()` that match all fields is error-prone. Add a field, forget to update them → silent bugs in `HashMap` / `HashSet`.
- **Workarounds were partial**: Lombok (`@Data`, `@Value`) generated boilerplate via annotation processing. Useful but introduced build-time dependency, IDE plugin requirements, and sometimes surprising behavior at runtime.
- **No semantic signal for "pure data class"**: a reader looking at a 50-line DTO couldn't immediately tell it was meant to be immutable data vs a regular class with behavior. Intent was hidden under ceremony.

### How records solve it

- **Declarative syntax**: `record Point(int x, int y) {}` replaces the full 30-line class. Compiler generates constructor, accessors (`x()`, `y()`), `equals()`, `hashCode()`, `toString()` from the header.
- **Immutability by default**: fields are `private final`. No setters possible. Encourages value-type semantics.
- **Correct equals/hashCode for free**: based on all components, always in sync with the record's shape. Impossible to forget a field when fields change.
- **Intent is visible**: `record` is a different keyword from `class`. A reader instantly knows this is a data carrier, not a behavioral type.
- **Interop with pattern matching**: records + sealed classes + switch expressions combine into algebraic data types (ADTs). Powerful for modeling domain types exhaustively.

```java
// Before, 30+ lines of boilerplate
class Point {
    private final int x;
    private final int y;
    Point(int x, int y) { this.x = x; this.y = y; }
    int x() { return x; }
    int y() { return y; }
    // equals, hashCode, toString...
}

// With Record, one line
record Point(int x, int y) { }

// Usage
var p = new Point(3, 4);
p.x();          // 3 (getter, no "get" prefix)
p.y();          // 4
p.toString();   // "Point[x=3, y=4]"
```

**Key characteristics**: Fields are `private final`, immutable
- Implicitly extends `java.lang.Record`, can't extend other classes
- Can implement interfaces
- Can have custom constructors, methods, and static fields
- Can't add instance fields beyond what's declared in the header

```java
record Employee(String name, int salary) {
    // Compact constructor, validation
    Employee {
        if (salary < 0) throw new IllegalArgumentException("Negative salary");
    }

    // Custom method
    String nameUpperCase() { return name.toUpperCase(); }
}
```

**Use for:** DTOs, API responses, value objects, configuration holders. **Not for:** JPA entities (due to immutability and constructor constraints).

**Records are shallowly immutable**: fields are final, but referenced objects can still be mutated:

```java
record User(List<String> roles) { }
var user = new User(new ArrayList<>(List.of("ADMIN")));
user.roles().add("USER");  // OK: list is mutated, record didn't protect it
```

## Pattern Matching for `instanceof` (Java 16)

Eliminates redundant casting after `instanceof` check.

```java
// Before, check + cast separately
if (obj instanceof String) {
    String s = (String) obj;     // redundant cast
    System.out.println(s.length());
}

// After, check + cast in one step
if (obj instanceof String s) {   // s is already cast
    System.out.println(s.length());
}

// Works with else too, s is scoped
if (obj instanceof String s && s.length() > 5) {
    System.out.println(s);
}
```

**Also works with switch (Java 21, finalized, preview in earlier versions):**

```java
// Pattern matching in switch
String result = switch (obj) {
    case Integer i -> "Integer: " + i;
    case String s  -> "String: " + s;
    case null      -> "null";
    default        -> "Unknown";
};
```

## Sealed Classes (Java 17)

Restrict which classes can extend/implement a class/interface. Gives controlled inheritance.

### Why sealed classes were added (the pre-Java 17 pain)

- **No way to close an inheritance hierarchy**: an `abstract class Shape` could be extended by ANY class, anywhere, at any time. Library authors had zero control over who implements their types.
- **`final` was all-or-nothing**: either no one can extend (`final`) or anyone can (non-final). No middle ground for *"only these specific subtypes."*
- **Domain modeling couldn't express "closed set"**: a `Payment` that can *only* be CreditCard, DebitCard, or UPI had no way to enforce that at compile time. Adding a rogue subclass compiled fine.
- **`switch` couldn't be exhaustive**: on an abstract/interface type, the compiler can't know all implementations, so it couldn't warn when a `switch` missed a case. You had to add a `default` branch that was either dead code or a silent bug catcher.
- **Package-private was the workaround**: hide the class behind package access and hope nobody put their subclass in the same package. Fragile and limiting.

### How sealed classes solve it

- **Closed hierarchy**: `sealed class Shape permits Circle, Rectangle, Triangle` declares the complete set of permitted subtypes at compile time. No one else can extend.
- **Each subclass declares its own openness**: must be one of: `final` (no further extension), `sealed` (further-restricted), or `non-sealed` (explicitly re-opened). Intent is visible.
- **Exhaustive `switch`**: compiler knows all subtypes, so a `switch` over a sealed type without a `default` branch is checked for completeness. Missing a case = compile error.
- **Enables algebraic data types (ADTs)**: sealed + records + pattern matching = expressive domain modeling. *"A Payment is exactly one of: CreditCard, DebitCard, UPI, each with its own fields."*
- **Better than enums when variants need different shape**: enums work when all cases are homogeneous constants. Sealed types shine when variants carry different fields (e.g., UPI has `upiId`, CreditCard has `number + cvv`).

```java
// Only Circle, Rectangle, Triangle can extend Shape
sealed class Shape permits Circle, Rectangle, Triangle { }

final class Circle extends Shape { }        // final, no further extension
non-sealed class Rectangle extends Shape { } // open for extension
sealed class Triangle extends Shape permits EquilateralTriangle { }

final class EquilateralTriangle extends Triangle { }
```

**Subclass must be one of**: `final`, no further extension
- `sealed`, further restricted
- `non-sealed`, open again

**Why use**: Exhaustive switch, compiler knows all subtypes, warns if you miss one
- Domain modeling, "a Payment is ONLY CreditCard, DebitCard, or UPI"
- Works especially well with pattern matching for exhaustive, type-safe domain modeling
- Better than enum when subtypes need different fields/behavior

```java
sealed interface Payment permits CreditCard, DebitCard, UPI { }
record CreditCard(String number, String cvv) implements Payment { }
record DebitCard(String number, String pin) implements Payment { }
record UPI(String upiId) implements Payment { }

// Compiler knows all cases, warns if incomplete
String process(Payment p) {
    return switch (p) {
        case CreditCard cc -> "CC: " + cc.number();
        case DebitCard db  -> "DB: " + db.number();
        case UPI upi       -> "UPI: " + upi.upiId();
    };
}
```

## Virtual Threads (Java 21)

Lightweight threads managed by JVM, not OS. Can create millions of them without OS thread overhead.

```java
// Traditional thread, OS-managed, expensive (1-2 MB per thread)
Thread t = new Thread(() -> doWork());

// Virtual thread, JVM-managed, lightweight (dynamically managed stack, much smaller than platform threads)
Thread vt = Thread.startVirtualThread(() -> doWork());

// With executor, most practical
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            // Each task gets its own virtual thread
            // Handles blocking I/O efficiently
            Thread.sleep(1000);
            return "done";
        });
    }
}
// 100K virtual threads, no problem. 100K platform threads, severe resource exhaustion.
```

|Model|Behavior|
|---|---|
|Thread pool|Limited threads → tasks queue up under load|
|Virtual threads|Thread-per-task → no queueing, scales with I/O|

**Why it matters**: Traditional threads are expensive (OS resource, ~1-2 MB stack each)
- Thread pools limit concurrency (e.g., 200 threads for 10K requests = queueing)
- Virtual threads allow **one thread per request** model without resource exhaustion

**Best for:** I/O-bound tasks (HTTP calls, DB queries, file I/O), reduces the need for reactive programming (WebFlux) in many cases. **Not for:** CPU-bound tasks (use parallel streams or ForkJoinPool instead).

**Key points**: Drop-in replacement, mostly same `Thread` API, but some blocking operations (`synchronized`, native calls) can limit scalability
- Carrier threads, virtual threads are mounted on platform (OS) threads by JVM
- Don't pool virtual threads, create new ones freely, that's the point
- `synchronized` can pin virtual threads to carrier threads during blocking, reducing scalability, prefer `ReentrantLock` for better control in highly concurrent scenarios

## Structured Concurrency (Java 21, Preview)

Manage multiple virtual threads as a unit, if one subtask fails, cancel the rest automatically.

```java
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    var user = scope.fork(() -> fetchUser());
    var order = scope.fork(() -> fetchOrder());
    scope.join();            // wait for both
    scope.throwIfFailed();   // propagate exception if any failed
    var userResult = user.resultNow();
    var orderResult = order.resultNow();
}
```

Avoids the common problem of orphaned threads when one task fails but others keep running.

## Text Blocks (Java 15)

Multi-line string literals, no more `\n` concatenation.

```java
// Before
String json = "{\n" +
              "  \"name\": \"Alice\",\n" +
              "  \"age\": 25\n" +
              "}";

// Text block, cleaner
String json = """
              {
                "name": "Alice",
                "age": 25
              }
              """;
```

Preserves formatting, handles indentation intelligently. Great for JSON, SQL, HTML templates.

## Switch Expressions (Java 14)

Switch as an expression that returns a value. Arrow syntax eliminates fall-through.

```java
// Old switch, statement, fall-through risk
switch (day) {
    case "MON": case "TUE": case "WED":
        type = "Weekday";
        break;                    // forget break = bug
    case "SAT": case "SUN":
        type = "Weekend";
        break;
}

// New switch, expression, no fall-through
String type = switch (day) {
    case "MON", "TUE", "WED" -> "Weekday";
    case "SAT", "SUN"         -> "Weekend";
    default                    -> "Unknown";
};

// Multi-line with yield
String type = switch (day) {
    case "MON" -> {
        System.out.println("Monday");
        yield "Weekday";          // yield returns value from block
    }
    default -> "Unknown";
};
```
