## JDK vs JRE vs JVM

```
JDK (Java Development Kit)
├── JRE (Java Runtime Environment)
│   ├── JVM (Java Virtual Machine)
│   └── Core Libraries (java.lang, java.util, java.io, etc.)
├── Compiler (javac)
├── Debugger (jdb)
├── Other tools (jar, javadoc, jstack, jmap, etc.)
```

||JDK|JRE|JVM|
|---|---|---|---|
|What|Development kit|Runtime environment|Virtual machine|
|Purpose|Develop + Run|Run only|Execute bytecode|
|Contains|JRE + compiler + tools|JVM + libraries|Execution engine|
|Who needs it|Developers|End users|Both|

```bash
# Flow: Write → Compile → Run
HelloWorld.java  -->  javac HelloWorld.java  -->  HelloWorld.class  -->  java HelloWorld  -->  Output
    (Source)               (Compiler)                (Bytecode)            (JVM/Runtime)
```

**"Write Once, Run Anywhere"**: `javac` compiles to platform-independent bytecode (`.class`). JVMs on Windows, Mac, Linux each interpret/compile that same bytecode to their own native code.

> **Note:** Oracle no longer ships the JRE as a separate download (since Java 11). Developers and end users both install the JDK.

## JAR Files

**JAR** (Java ARchive) is a zip file with `.jar` extension that bundles compiled `.class` files plus metadata (`META-INF/MANIFEST.MF`). It's how Java libraries and applications get distributed as a single file.

- **Library JAR**: packaged code you depend on (e.g., `jackson-databind.jar`, `spring-web.jar`)
- **Executable JAR**: has a `Main-Class` entry in the manifest; runs with `java -jar app.jar`
- **Fat / uber JAR**: an executable JAR that bundles all dependencies and an embedded server

Behind the scenes a JAR is just a ZIP — rename `.jar` to `.zip` and you can extract the `.class` files.

## Classpath

List of locations (directories and JAR files) where the Java compiler (`javac`) and JVM look for `.class` files. It typically includes:

- Your compiled application classes (`target/classes` in Maven projects)
- All dependency JARs pulled in by your build tool

Set via the `-cp` / `--class-path` command-line flag or the `CLASSPATH` environment variable (legacy, rarely used today). 

Managed automatically by build tools like Maven / Gradle from your declared dependencies.

**Why it matters:** a missing classpath entry throws `ClassNotFoundException` at runtime.

## JVM Architecture

```
ClassLoader Subsystem
        ↓
Runtime Data Areas (Memory)
  ├── Method Area (Metaspace)   ← class metadata, static vars
  ├── Heap                      ← objects, instance vars, String Constant Pool
  ├── Stack (per thread)        ← local vars, method frames
  ├── PC Register (per thread)  ← current instruction pointer
  └── Native Method Stack (per thread)
        ↓
Execution Engine
  ├── Interpreter
  ├── JIT Compiler
  └── Garbage Collector
        ↓
Java Native Interface (JNI) → Native Libraries
```

## Q&A

1. **What is the practical difference between JDK and JRE for a modern Java developer?**
   JRE is just the subset of the JDK that contains the JVM and standard libraries, while the JDK adds the compiler (`javac`) and diagnostic tools. Prior to Java 11, the JRE was a separate, smaller download for end users. Today, Oracle only distributes the JDK, which includes everything.

2. **How do the Interpreter and the JIT compiler work together within the JVM?**
   To ensure a fast startup, the JVM initially interprets bytecode line-by-line. As the application runs, the JVM identifies "hot" code paths (methods called frequently) and uses the **JIT (Just-In-Time) compiler** to compile them into native machine code. This balances quick startup with near-native performance after a "warm-up" period.

3. **What is a Fat JAR?**
   A standard JAR contains only the application's compiled classes. A Fat JAR bundles the application code **plus all its dependencies** and an embedded web server into a single executable file. This makes deployment significantly easier, as the app can run on any machine with a JVM without needing manual classpath management.

## Quick Revision Cheat Sheet

| Concept     | Remember                                                |
| ----------- | ------------------------------------------------------- |
| JDK         | JRE + Development tools (`javac`, `jdb`, `jstack`)      |
| JRE         | JVM + Core Libraries (Runtime only)                     |
| JVM         | Virtual machine that executes bytecode (`.class` files) |
| JIT         | Compiles frequent bytecode to native code for speed     |
| Metaspace   | Replaced PermGen in Java 8                              |
| String Pool | In the **Heap** since Java 7 (previously in PermGen)    |
| G1 GC       | Default Garbage Collector since **Java 9**              |
| Classpath   | Search path used by JVM to find classes                 |
| Fat JAR     | Single file containing App + Dependencies + Server      |
