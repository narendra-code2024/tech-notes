The JDK, JRE, and JVM are nested layers of the Java platform: the JDK contains the JRE, which contains the JVM.

```
JDK (Java Development Kit)
├── JRE (Java Runtime Environment)
│   ├── JVM (Java Virtual Machine)
│   └── Core Libraries (java.lang, java.util, java.io, etc.)
├── Compiler (javac)
├── Debugger (jdb)
├── Other tools (jar, javadoc, jstack, jmap, etc.)
```

|              | JDK                    | JRE                 | JVM              |
| ------------ | ---------------------- | ------------------- | ---------------- |
| What         | Development kit        | Runtime environment | Virtual machine  |
| Purpose      | Develop + run          | Run only            | Execute bytecode |
| Contains     | JRE + compiler + tools | JVM + libraries     | Execution engine |
| Who needs it | Developers             | End users           | Both             |

```
# Flow: Write → Compile → Run
HelloWorld.java  -->  javac HelloWorld.java  -->  HelloWorld.class  -->  java HelloWorld  -->  Output
    (Source)               (Compiler)                (Bytecode)            (JVM/Runtime)
```

**"Write Once, Run Anywhere"**: `javac` compiles to platform-independent bytecode (`.class`); the JVM on each OS interprets and JIT-compiles that same bytecode to its own native code.

> **Note**: Oracle no longer ships the JRE as a separate download (since Java 11). Both developers and end users install the JDK.

## JAR Files

A JAR (Java ARchive) is a ZIP file with a `.jar` extension bundling compiled `.class` files plus metadata (`META-INF/MANIFEST.MF`). It is how Java libraries and applications ship as a single file; rename `.jar` to `.zip` and the classes extract like any archive.

- **Library JAR**: code you depend on (e.g. `jackson-databind.jar`, `spring-web.jar`).
- **Executable JAR**: has a `Main-Class` manifest entry; runs with `java -jar app.jar`.
- **Fat / uber JAR**: an executable JAR that bundles all dependencies (and often an embedded server) into one runnable file.

---

## Classpath

The list of locations (directories and JAR files) where `javac` and the JVM look for `.class` files. It typically includes:

- Your compiled application classes (`target/classes` in Maven projects).
- Every dependency JAR pulled in by your build tool.

Set it with the `-cp` / `--class-path` flag or the legacy `CLASSPATH` environment variable; Maven and Gradle manage it automatically from declared dependencies. Since Java 9, modular applications use the **module path** (`--module-path` / `-p`) instead, though most apps still run on the classpath.

> **Note**: a class that is absent at runtime surfaces as `NoClassDefFoundError`, or as `ClassNotFoundException` when loaded by name via `Class.forName`.

## JVM Architecture

```
ClassLoader Subsystem
  ├── Loading          ← reads .class bytecode into memory
  ├── Linking          ← verify, prepare, resolve
  └── Initialization   ← runs static initializers & blocks
        ↓
Runtime Data Areas (Memory)
  ├── Method Area (Metaspace)   ← class metadata, static vars
  ├── Heap                      ← objects, instance vars, String Constant Pool
  ├── Stack (per thread)        ← local vars, method frames
  ├── PC Register (per thread)  ← current instruction pointer
  └── Native Method Stack (per thread)
        ↓
Execution Engine
  ├── Interpreter
  ├── JIT Compiler
  └── Garbage Collector
        ↓
Java Native Interface (JNI) → Native Libraries
```

> **Reference**: JVM internals (ClassLoader phases, runtime data areas, execution engine, object creation, JVM flags), see [JVM Architecture & Memory Model](<30. JVM Architecture & Memory Model.md>).

## Q&A

1. **What is the practical difference between the JDK and JRE for a modern developer?**
The JRE is the subset of the JDK that runs Java (JVM plus standard libraries); the JDK adds the compiler and diagnostic tools. Since Java 11 Oracle distributes only the JDK, so in practice everyone installs the JDK.

2. **How do the interpreter and JIT compiler work together?**
The JVM interprets bytecode first for fast startup, then detects hot methods and JIT-compiles them to native machine code. This pairing gives quick startup with near-native throughput after a short warm-up.

3. **What is a fat JAR and why use one?**
A standard JAR holds only the application's own classes, whereas a fat JAR also bundles every dependency (and often an embedded server) into one executable. It simplifies deployment: the app runs on any JVM with no manual classpath assembly.
