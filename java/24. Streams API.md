A stream is a pipeline for processing a sequence of elements from a source (collection, array, file, generator) through a chain of operations. It stores nothing and does not modify the source; it produces a new result.

```java
List<String> result = names.stream()           // source
    .filter(n -> n.length() > 3)               // intermediate (lazy)
    .map(String::toUpperCase)                  // intermediate (lazy)
    .sorted()                                  // intermediate (lazy)
    .toList();                                 // terminal (eager, triggers the pipeline)
```

- **Lazy**: intermediate operations run only when a terminal operation fires, and the framework fuses them into a single pass (no intermediate collections).
- **One-time use**: a stream cannot be reused after its terminal operation.
- **Sequential or parallel**: the same pipeline runs on one thread or across cores via `parallelStream()`.

---

## Creating Streams

```java
list.stream();                                 // from a Collection
Arrays.stream(arr);                            // from an array
Stream.of("a", "b", "c");                      // from values
Stream.empty();

Stream.iterate(0, n -> n + 2);                 // infinite: 0, 2, 4, ... (needs limit())
Stream.generate(Math::random);                 // infinite: random, random, ...
Stream.iterate(0, n -> n < 10, n -> n + 2);    // bounded (Java 9+): 0, 2, 4, 6, 8

IntStream.range(1, 5);                          // 1..4 (end exclusive)
IntStream.rangeClosed(1, 5);                    // 1..5 (end inclusive)
"hello".chars();                               // IntStream of char values

map.entrySet().stream();                        // keySet()/values() also stream
```

## Intermediate Operations

Lazy and chainable: each returns a new stream, and nothing executes until a terminal operation runs.

```java
List<String> out = names.stream()
    .filter(n -> n.length() > 3)     // Predicate: keep matches
    .distinct()                      // drop duplicates (by equals)
    .map(String::toUpperCase)        // Function: 1-to-1 transform
    .sorted()                        // stable; natural order, or pass a Comparator
    .skip(1).limit(5)                // slice: drop first 1, take next 5
    .toList();
```

`flatMap` turns each element into a stream and merges them into one, the 1-to-many counterpart of `map`:

```java
sentences.stream()
    .flatMap(s -> Arrays.stream(s.split(" ")))   // ["hello","world","foo","bar"]
    .toList();
```

`mapToInt`/`mapToLong`/`mapToDouble` switch to a primitive stream, unlocking numeric terminals without boxing:

```java
int total = names.stream().mapToInt(String::length).sum();   // IntStream: sum/average/min/max
```

> **Note**: `peek` passes each element through unchanged and exists for debugging only; it may not run for every element when a downstream operation short-circuits.

## Terminal Operations

Eager: they consume the stream, produce a result or side effect, and close it.

### collect

```java
List<String> l   = s.collect(Collectors.toList());   // or s.toList() (Java 16+, unmodifiable)
Set<String> set  = s.collect(Collectors.toSet());
TreeSet<String> t = s.collect(Collectors.toCollection(TreeSet::new));

// toMap: the 3rd arg merges duplicate keys, without it a clash throws IllegalStateException
Map<String, Integer> m = s.collect(Collectors.toMap(k -> k, String::length, (a, b) -> a));

String csv = s.collect(Collectors.joining(", ", "[", "]"));   // "[a, b, c]"

// groupingBy: classifier to List by default, or a downstream collector
Map<Integer, List<String>> byLen  = s.collect(Collectors.groupingBy(String::length));
Map<Integer, Long> countByLen     = s.collect(Collectors.groupingBy(String::length, Collectors.counting()));
Map<String, Integer> payByDept    = emps.collect(Collectors.groupingBy(Employee::getDept,
                                        Collectors.summingInt(Employee::getSalary)));
Map<String, List<String>> byDept  = emps.collect(Collectors.groupingBy(Employee::getDept,
                                        Collectors.mapping(Employee::getName, Collectors.toList())));

// partitioningBy: groupingBy on a boolean
Map<Boolean, List<Integer>> evenOdd = nums.collect(Collectors.partitioningBy(n -> n % 2 == 0));
```

### reduce

Combine all elements into one value. Prefer a built-in (`sum`, `max`, `collect`) when one exists; use `reduce` for custom accumulation.

```java
int sum = nums.reduce(0, Integer::sum);            // identity + accumulator
Optional<Integer> r = nums.reduce(Integer::sum);   // no identity, returns Optional (empty-safe)
```

### Others

```java
long n = s.count();
Optional<String> first = s.findFirst();            // findAny() is cheaper in parallel
Optional<String> max   = s.max(Comparator.comparing(String::length));
boolean any = s.anyMatch(x -> x.length() > 5);     // allMatch / noneMatch likewise
String[] arr = s.toArray(String[]::new);
s.forEach(System.out::println);                    // Iterable.forEach needs no stream
```

## Optional

`Optional<T>` is a container that may or may not hold a value, making absence explicit in a return type instead of returning `null`. Stream terminals like `findFirst`, `reduce`, `min`, and `max` return one because the result may not exist.

```java
Optional.of("hello");          // value must be non-null, else NPE
Optional.ofNullable(value);    // null becomes an empty Optional
Optional.empty();
```

```java
opt.isPresent();   opt.isEmpty();              // isEmpty since Java 11
opt.get();                                     // throws NoSuchElementException if empty

opt.orElse("default");                          // value or a default (always evaluated)
opt.orElseGet(() -> compute());                 // value or a lazily computed default
opt.orElseThrow();                              // NoSuchElementException if empty (Java 10+)
opt.orElseThrow(() -> new IllegalStateException("missing"));

opt.map(String::toUpperCase);                   // transform if present
opt.flatMap(this::findById);                    // when the mapper itself returns Optional
opt.filter(s -> s.length() > 3);                // empty if the test fails
opt.ifPresent(System.out::println);
opt.ifPresentOrElse(System.out::println, () -> log.warn("empty"));   // Java 9+
```

Use `Optional` only as a return type. As a field, method parameter, or collection element it adds cost and a second "absent" state without benefit, and `get()` without a presence check defeats the purpose.

## Primitive Streams

`IntStream`, `LongStream`, and `DoubleStream` avoid autoboxing and add numeric terminals.

```java
IntStream.of(1, 2, 3, 4, 5).sum();              // 15; also average(), min(), max()
IntStream.of(1, 2, 3).summaryStatistics();      // count, sum, min, average, max in one pass

IntStream.of(1, 2, 3).boxed();                  // IntStream -> Stream<Integer>
list.stream().mapToInt(Integer::intValue);      // Stream<Integer> -> IntStream
```

## Parallel Streams

`parallelStream()` (or `stream().parallel()`) splits the pipeline across the `ForkJoinPool.commonPool()`.

```java
list.parallelStream().filter(s -> s.length() > 3).toList();
```

Parallelism is not automatically faster; splitting and merging have overhead, and gains depend on data size and per-element cost. Measure first.

- **Use when**: large datasets, stateless CPU-bound operations, and order does not matter.
- **Avoid when**: small collections (overhead wins), order-dependent logic, shared mutable state (race conditions, collect instead of mutating an external list), or blocking I/O (threads stall and starve the shared pool).

---

## Common Patterns

```java
// duplicates: count occurrences, keep those seen more than once
Set<Integer> dups = list.stream()
    .collect(Collectors.groupingBy(n -> n, Collectors.counting()))
    .entrySet().stream().filter(e -> e.getValue() > 1).map(Map.Entry::getKey)
    .collect(Collectors.toSet());

// second highest: distinct, descending, skip one
Optional<Integer> second = list.stream().distinct()
    .sorted(Comparator.reverseOrder()).skip(1).findFirst();

// top N
List<Integer> top3 = list.stream().sorted(Comparator.reverseOrder()).limit(3).toList();

// max by a field
Optional<Employee> highest = emps.stream().max(Comparator.comparingInt(Employee::getSalary));

// first match or a fallback
String hit = list.stream().filter(s -> s.startsWith("A")).findFirst().orElse("none");
```

## Short-Circuiting Operations

These can finish without traversing the whole stream: `findFirst`, `findAny`, `anyMatch`, `allMatch`, `noneMatch`, and `limit`. They let infinite streams (`iterate`/`generate`) terminate.

---

## Stateless vs Stateful Operations

| Stateless (parallel-friendly)       | Stateful                                          |
| ----------------------------------- | ------------------------------------------------- |
| `filter`, `map`, `flatMap`, `peek`  | `sorted`, `distinct`, `limit`, `skip`             |
| Each element handled independently  | Need element history or ordering; costlier in parallel |

---

## Best Practices

- **Filter early**: shrink the stream with `filter` before expensive `sorted`/`map` steps.
- **Avoid side effects**: do not mutate external or shared state inside a pipeline; keep it declarative and collect the result.
- **collect vs reduce**: `collect` is a mutable reduction for building collections; `reduce` combines elements into a single immutable value.
- **Streams vs loops**: streams win on readability; a plain loop can be faster for trivial or hot, performance-critical logic.

## Q&A

1. **Can a stream be reused after a terminal operation?**
No. Once consumed, any further operation throws `IllegalStateException: stream has already been operated upon or closed`. Create a fresh stream from the source instead.

2. **How does `collect(Collectors.toList())` differ from `.toList()` (Java 16+)?**
`collect(toList())` returns a mutable `ArrayList`; `.toList()` returns an unmodifiable list. Prefer `.toList()` unless you need to mutate the result.

3. **What does `Collectors.toMap()` do on duplicate keys?**
The two-arg form throws `IllegalStateException`. Supply a third merge function to resolve clashes, for example `(v1, v2) -> v1` to keep the first.

4. **Is element order guaranteed for `forEach` on a parallel stream?**
No, it runs in arbitrary order. Use `forEachOrdered` to preserve encounter order, at the cost of some parallel benefit.

5. **Why avoid `peek` for production side effects?**
It is meant for debugging and is not guaranteed to run for every element when a downstream operation short-circuits (`findFirst`, `limit`). Use a terminal `forEach` for intentional side effects.

6. **`orElse` vs `orElseGet`?**
`orElse(x)` always evaluates `x`, even when the value is present; `orElseGet(supplier)` runs only when empty. Use `orElseGet` whenever the default is expensive to build.

7. **`map` vs `flatMap`?**
`map` is one-to-one: each element becomes exactly one result. `flatMap` is one-to-many: each element becomes a stream, and the streams are flattened into one.

## Quick Revision Cheat Sheet

| Concept              | Remember                                                                |
| -------------------- | ----------------------------------------------------------------------- |
| Lazy evaluation      | Intermediate ops run only when a terminal op fires; fused into one pass  |
| One-time use         | Reusing after a terminal op throws `IllegalStateException`              |
| collect vs toList()  | `toList()` (Java 16+) is unmodifiable; `collect(toList())` is a mutable ArrayList |
| toMap duplicate keys | Throws `IllegalStateException`; pass a merge function                   |
| Optional             | Return-type only; prefer `orElseGet`/`map` over `get()`                 |
| orElse vs orElseGet  | `orElse` always evaluates its arg; `orElseGet` is lazy                  |
| map vs flatMap       | `map` is 1-to-1; `flatMap` is 1-to-many then flatten                    |
| Primitive streams    | `IntStream`/`LongStream`/`DoubleStream` avoid boxing; add sum/average   |
| Short-circuiting     | `findFirst`/`findAny`/`anyMatch`/`limit` stop early                     |
| Stateful ops         | `sorted`/`distinct` buffer, `limit`/`skip` slice; weaker in parallel    |
| Order matters        | Filter early to shrink the stream before sort/map                       |
| Parallel             | Large CPU-bound data only; no shared mutable state, no blocking I/O     |
