## What & Why

A Stream is a **pipeline for processing data**: it doesn't store data, it processes it from a source (Collection, array, file, etc.) through a chain of operations.

```java
List<String> names = List.of("Alice", "Bob", "Charlie", "David");

List<String> result = names.stream()       // 1. create stream
    .filter(n -> n.length() > 3)           // 2. intermediate operation
    .map(String::toUpperCase)              // 2. intermediate operation
    .sorted()                              // 2. intermediate operation
    .collect(Collectors.toList());         // 3. terminal operation

// result: ["ALICE", "CHARLIE", "DAVID"]
```

**Key characteristics**: Not a data structure, doesn't store elements
- Doesn't modify the source, creates a new result
- Lazy, intermediate operations run only when terminal operation is called
- One-time use, can't reuse a stream after terminal operation
- Can be sequential or parallel

### Why Streams were added (the pre-Java 8 pain)

- **For-loops mixed "what" with "how"**: to get "all users over 18, their names, sorted, as a list," you'd write 10+ lines of `for`, `if`, `add`, `Collections.sort`. The *intent* was buried under the *iteration mechanics*.
- **Intermediate collections everywhere**: chaining filter → map → sort meant creating temporary `List`s at each step, wasting memory and slowing things down.
- **Parallelization required major surgery**: to process a million items across 8 cores, you'd hand-roll thread pools, partitioning logic, and reduce-step code. Hard and error-prone.
- **Different code for different sources**: iterating a `List`, an `InputStream`, a file line-by-line, a database result set all looked different. No unified abstraction for "sequence of values to process."

### How Streams solve it

- **Declarative pipeline**: `.filter().map().sorted().collect()` reads left-to-right like a sentence: *filter these, transform those, sort the rest, collect to a list*. Mechanics become implicit.
- **Lazy evaluation = fused pipeline**: no intermediate collections. The framework fuses all operations into a single pass over the source. Huge memory and throughput win.
- **One-switch parallelization**: `.parallelStream()` runs the pipeline across all CPU cores using `ForkJoinPool`. No manual threading.
- **Uniform over any source**: Collections, arrays, files, generators, I/O all expose a `Stream<T>`. Same pipeline code works everywhere.
- **Primitive specializations** (`IntStream`, `LongStream`, `DoubleStream`) avoid autoboxing overhead for number-heavy pipelines.

## Creating Streams

```java
// From Collection
List<String> list = List.of("a", "b", "c");
Stream<String> s1 = list.stream();

// From array
String[] arr = {"a", "b", "c"};
Stream<String> s2 = Arrays.stream(arr);

// From values directly
Stream<String> s3 = Stream.of("a", "b", "c");

// Empty stream
Stream<String> s4 = Stream.empty();

// Infinite streams
Stream<Integer> s5 = Stream.iterate(0, n -> n + 2);    // 0, 2, 4, 6, ...
Stream<Double> s6 = Stream.generate(Math::random);      // random, random, ...
// Must use limit() with infinite streams or they run forever

// Bounded iterate (Java 9+)
Stream<Integer> s7 = Stream.iterate(0, n -> n < 10, n -> n + 2);  // 0, 2, 4, 6, 8

// Primitive streams
IntStream is1 = IntStream.range(1, 5);        // 1, 2, 3, 4 (exclusive end)
IntStream is2 = IntStream.rangeClosed(1, 5);  // 1, 2, 3, 4, 5 (inclusive end)
IntStream is3 = IntStream.of(10, 20, 30);

// String to stream
IntStream chars = "hello".chars();             // stream of char values

// From Map
map.entrySet().stream();
map.keySet().stream();
map.values().stream();
```

## Intermediate Operations (Lazy, return a Stream)

These don't execute until a terminal operation is called. They transform the stream and return a new stream.

### filter, keep elements matching condition

```java
// Predicate<T> → keeps elements where test() returns true
list.stream().filter(s -> s.length() > 3)    // keep strings longer than 3
```

### map, transform each element

```java
// Function<T,R> → transforms each element
list.stream().map(String::toUpperCase)        // "hello" → "HELLO"
list.stream().map(s -> s.length())            // "hello" → 5
```

### flatMap, flatten nested structures

```java
// When each element maps to a stream, flatMap merges them into one stream
List<List<Integer>> nested = List.of(List.of(1, 2), List.of(3, 4), List.of(5));
nested.stream()
    .flatMap(Collection::stream)              // flattens to: 1, 2, 3, 4, 5
    .collect(Collectors.toList());

// Common: split words from sentences
List<String> sentences = List.of("hello world", "foo bar");
sentences.stream()
    .flatMap(s -> Arrays.stream(s.split(" ")))  // ["hello", "world", "foo", "bar"]
    .collect(Collectors.toList());
```

### sorted, sort elements

```java
list.stream().sorted()                          // natural order
list.stream().sorted(Comparator.reverseOrder()) // reverse order
list.stream().sorted(Comparator.comparing(String::length))  // by length
// sorted() is stable, maintains relative order of equal elements
```

### distinct, remove duplicates

```java
// Uses equals() to check duplicates
List.of(1, 2, 2, 3, 3).stream().distinct()    // 1, 2, 3
```

### peek, debug/inspect without modifying

```java
// Consumer<T> → performs action on each element, passes it through unchanged
list.stream()
    .filter(s -> s.length() > 3)
    .peek(s -> System.out.println("Filtered: " + s))  // debug what passed filter
    .map(String::toUpperCase)
    .collect(Collectors.toList());
```

### limit & skip, control stream size

```java
Stream.iterate(1, n -> n + 1)
    .skip(5)       // skip first 5 elements
    .limit(3)      // take next 3 elements → 6, 7, 8
    .collect(Collectors.toList());
```

### mapToInt / mapToLong / mapToDouble, convert to primitive stream

```java
// Avoids autoboxing, better performance
list.stream()
    .mapToInt(String::length)     // returns IntStream, not Stream<Integer>
    .sum();                        // IntStream has sum(), average(), max(), min()
```

## Terminal Operations (Eager, trigger execution)

These consume the stream and produce a result. After a terminal operation, the stream is closed.

### collect, gather results into a collection

```java
// To List
List<String> result = stream.collect(Collectors.toList());

// To Set (removes duplicates)
Set<String> result = stream.collect(Collectors.toSet());

// To specific collection
TreeSet<String> result = stream.collect(Collectors.toCollection(TreeSet::new));

// To Map
Map<String, Integer> result = list.stream()
    .collect(Collectors.toMap(s -> s, String::length));   // key, value
// warning: Throws IllegalStateException on duplicate keys, use merge function:
Map<String, Integer> result = list.stream()
    .collect(Collectors.toMap(s -> s, String::length, (v1, v2) -> v1));

// Joining strings
String result = list.stream().collect(Collectors.joining(", "));  // "a, b, c"
String result = list.stream().collect(Collectors.joining(", ", "[", "]"));  // "[a, b, c]"

// Grouping
Map<Integer, List<String>> byLength = list.stream()
    .collect(Collectors.groupingBy(String::length));
// {3=[Bob], 5=[Alice, David], 7=[Charlie]}

// Partitioning (groupBy boolean)
Map<Boolean, List<Integer>> evenOdd = numbers.stream()
    .collect(Collectors.partitioningBy(n -> n % 2 == 0));
// {true=[2, 4], false=[1, 3, 5]}

// Counting per group
Map<Integer, Long> countByLength = list.stream()
    .collect(Collectors.groupingBy(String::length, Collectors.counting()));

// Summing per group
Map<String, Integer> salaryByDept = employees.stream()
    .collect(Collectors.groupingBy(Employee::getDept, Collectors.summingInt(Employee::getSalary)));

// toList() shortcut (Java 16+)
List<String> result = stream.toList();  // returns unmodifiable list
```

### forEach, perform action on each element

```java
list.stream().forEach(System.out::println);
// Shortcut (from Iterable, no stream needed):
list.forEach(System.out::println);
```

### reduce, combine all elements into one value

Prefer built-in methods (`sum()`, `max()`, `collect()`) over `reduce()` for readability and performance. Use `reduce()` for custom accumulations.

```java
// With identity (starting value)
int sum = numbers.stream().reduce(0, Integer::sum);         // 0 + 1 + 2 + 3...
String concat = list.stream().reduce("", String::concat);   // "" + "a" + "b"...

// Without identity, returns Optional (stream could be empty)
Optional<Integer> sum = numbers.stream().reduce(Integer::sum);

// How reduce works step by step:
// reduce(0, (a, b) -> a + b)
// Step 1: 0 + 1 = 1
// Step 2: 1 + 2 = 3
// Step 3: 3 + 3 = 6
```

### count, min, max

```java
long count = stream.count();

Optional<String> min = stream.min(Comparator.naturalOrder());
Optional<String> max = stream.max(Comparator.comparing(String::length));
```

### findFirst, findAny

```java
Optional<String> first = stream.findFirst();   // first element
Optional<String> any = stream.findAny();       // any element (useful in parallel)
```

### anyMatch, allMatch, noneMatch

```java
boolean hasLong = list.stream().anyMatch(s -> s.length() > 5);   // at least one matches
boolean allLong = list.stream().allMatch(s -> s.length() > 5);   // all match
boolean noneLong = list.stream().noneMatch(s -> s.length() > 5); // none match
```

### toArray

```java
String[] arr = stream.toArray(String[]::new);
```

## Optional (Companion to Streams)

`Optional<T>` is a container that may or may not hold a value. Used to avoid `NullPointerException`. Many stream terminal operations return `Optional`.

### Why Optional exists

- **`NullPointerException` was Java's #1 bug** (Tony Hoare's "billion-dollar mistake"). Null carried no intent, a method signature couldn't signal "might be absent." `Optional<User>` makes absence explicit at compile time.
- **Defensive `if (x != null)` sprawled**: `Optional` forces the caller to unwrap via safe APIs (`map`, `orElse`, `orElseGet`, `ifPresent`); `.get()` is frowned on.
- **Chained calls could NPE at any dot**: `opt.map(User::getAddress).map(Address::getCity).orElse("unknown")` replaces nested null checks.
- **Stream terminal ops return Optional**: `findFirst`, `findAny`, `reduce`, `min`, `max` all return `Optional<T>` because the result may not exist.

**What Optional is NOT for**: fields on entities, method parameters, collection elements. Only use as a return type. (See Anti-patterns below.)

### Creating Optional

```java
Optional<String> opt1 = Optional.of("hello");        // value must be non-null
Optional<String> opt2 = Optional.ofNullable(null);    // allows null → empty Optional
Optional<String> opt3 = Optional.empty();              // explicitly empty
```

### Using Optional

```java
// Check and get
opt.isPresent();          // true if value exists
opt.isEmpty();            // true if no value (Java 11+)
opt.get();                // returns value, throws NoSuchElementException if empty!

// Safe access (preferred)
opt.orElse("default");                      // value or default
opt.orElseGet(() -> computeDefault());      // value or lazily computed default
opt.orElseThrow();                          // throws NoSuchElementException if empty (Java 10+, no-arg version)
opt.orElseThrow(() -> new RuntimeException("not found"));  // value or custom exception

// Transform
opt.map(String::toUpperCase);               // Optional<String> → Optional<String>
opt.flatMap(this::findById);                // when mapper returns Optional, avoids nesting

// Conditional action
opt.ifPresent(System.out::println);         // run only if value present
opt.ifPresentOrElse(                        // Java 9+
    System.out::println,
    () -> System.out.println("empty")
);

// Filter
opt.filter(s -> s.length() > 3);           // returns empty Optional if condition fails

// Stream (Java 9+)
opt.stream();                               // Optional to Stream (0 or 1 element)
```

### Optional Anti-patterns

```java
// Don't use Optional as method parameter
void process(Optional<String> name) { }     // bad practice

// Don't use Optional for fields
class User { Optional<String> email; }      // bad practice

// Don't use get() without isPresent()
opt.get();                                   // can throw, use orElse/orElseThrow

// OK: Optional is meant for return types
Optional<User> findById(int id) { }         // correct usage
```

## Primitive Streams

Specialized streams that avoid autoboxing overhead. Have extra utility methods.

```java
// IntStream, LongStream, DoubleStream
IntStream intStream = IntStream.of(1, 2, 3, 4, 5);

// Extra methods not available on Stream<T>
intStream.sum();          // 15
intStream.average();      // OptionalDouble(3.0)
intStream.min();          // OptionalInt(1)
intStream.max();          // OptionalInt(5)
intStream.summaryStatistics();  // count=5, sum=15, min=1, avg=3.0, max=5

// Boxing, primitive stream → object stream
Stream<Integer> boxed = IntStream.of(1, 2, 3).boxed();

// Unboxing, object stream → primitive stream
IntStream unboxed = list.stream().mapToInt(Integer::intValue);
```

## Parallel Streams

Process elements in parallel using multiple threads (ForkJoinPool).

```java
// Create parallel stream
list.parallelStream()
    .filter(s -> s.length() > 3)
    .collect(Collectors.toList());

// Convert existing stream to parallel
list.stream().parallel()
    .map(String::toUpperCase)
    .collect(Collectors.toList());
```

Parallel streams use `ForkJoinPool.commonPool()` by default. Avoid heavy blocking operations (DB calls, HTTP calls), can exhaust the pool. **Not always faster**: performance depends on data size, CPU-bound vs I/O-bound, and the cost of splitting & merging. Measure before committing.

**When to use:** Large datasets + stateless operations + CPU-intensive transformations.

**When NOT to use**: Small collections, overhead > benefit
- Order-dependent operations, parallel doesn't guarantee order
- Shared mutable state, causes race conditions (use `collect()` instead of mutating external lists)
- I/O operations, threads block on I/O, wastes ForkJoinPool

## Common Stream Patterns

```java
// Frequency count
Map<String, Long> freq = words.stream()
    .collect(Collectors.groupingBy(w -> w, Collectors.counting()));

// Find duplicates (functional, parallel-safe)
Set<Integer> duplicates = list.stream()
    .collect(Collectors.groupingBy(n -> n, Collectors.counting()))
    .entrySet().stream()
    .filter(e -> e.getValue() > 1)
    .map(Map.Entry::getKey)
    .collect(Collectors.toSet());

// Flatten and collect unique
List<String> allTags = articles.stream()
    .flatMap(a -> a.getTags().stream())
    .distinct()
    .collect(Collectors.toList());

// Sum of specific field
int totalSalary = employees.stream()
    .mapToInt(Employee::getSalary)
    .sum();

// Max by field
Optional<Employee> highest = employees.stream()
    .max(Comparator.comparingInt(Employee::getSalary));

// Group and sort within groups
Map<String, List<Employee>> grouped = employees.stream()
    .sorted(Comparator.comparing(Employee::getName))
    .collect(Collectors.groupingBy(Employee::getDept));

// Convert List to Map
Map<Integer, String> idToName = employees.stream()
    .collect(Collectors.toMap(Employee::getId, Employee::getName));

// Comma-separated string
String csv = list.stream().collect(Collectors.joining(", "));

// First match with default
String found = list.stream()
    .filter(s -> s.startsWith("A"))
    .findFirst()
    .orElse("not found");

// Check condition on all elements
boolean allAdults = users.stream().allMatch(u -> u.getAge() >= 18);

// Mapping downstream collector (group + transform)
Map<String, List<String>> namesByDept = employees.stream()
    .collect(Collectors.groupingBy(
        Employee::getDept,
        Collectors.mapping(Employee::getName, Collectors.toList())
    ));

// Nth highest element (interview classic)
// Second highest:
Optional<Integer> secondHighest = list.stream()
    .distinct()
    .sorted(Comparator.reverseOrder())
    .skip(1)
    .findFirst();

// Top N elements
List<Integer> top3 = list.stream()
    .sorted(Comparator.reverseOrder())
    .limit(3)
    .toList();
```

## Stream Pipeline Order Matters

```java
// Inefficient: sorts everything, then filters
list.stream()
    .sorted()
    .filter(s -> s.length() > 3)
    .collect(Collectors.toList());

// OK: Efficient, filter first (fewer elements to sort)
list.stream()
    .filter(s -> s.length() > 3)
    .sorted()
    .collect(Collectors.toList());
```

Rule of thumb: **filter early, sort/map late**. Reduce elements first, then do expensive operations on fewer elements.

## Short-Circuiting Operations

These stop processing early, don't need to traverse the entire stream:

- `findFirst()`, `findAny()`, return as soon as a match is found
- `anyMatch()`, `allMatch()`, `noneMatch()`, return as soon as result is determined
- `limit()`, takes only first N elements

## Stateless vs Stateful Operations

|Stateless|Stateful|
|---|---|
|`filter`, `map`, `flatMap`, `peek`|`sorted`, `distinct`|
|Process each element independently|`limit`, `skip` (stateful in parallel / slicing ops)|
|Parallel-friendly|Expensive, less parallel-friendly|

## Stream Best Practices

- **Avoid side effects**: don't modify external variables or shared mutable state inside streams. Streams should be declarative and functional.
- **`collect()` vs `reduce()`**: `collect()` is mutable reduction (preferred for building collections). `reduce()` is general-purpose reduction (for combining into single value).
- **Streams vs loops**: Streams improve readability and declarative style. Loops may be faster for simple or performance-critical logic.

## Q&A

1. **Can you reuse a Java Stream once a terminal operation has been called?**
   No. A Stream is designed for one-time use. Once a terminal operation (like `forEach`, `collect`, or `reduce`) has been executed, the Stream is considered consumed. Attempting to call another operation on the same Stream instance will throw an `IllegalStateException`.

   ```java
   Stream<String> s = list.stream();
   s.forEach(System.out::println);
   s.forEach(System.out::println);  // throws IllegalStateException: stream already operated upon
   ```

2. **What is the difference between `collect(Collectors.toList())` and the `.toList()` method introduced in Java 16?**
   `collect(Collectors.toList())` returns a **mutable** `ArrayList`. The newer `.toList()` method returns an **unmodifiable** list. In modern codebases (Java 16+), `.toList()` is preferred unless you specifically need to modify the resulting list afterward.

3. **What happens if `Collectors.toMap()` encounters duplicate keys, and how do you handle it?**
   By default, `toMap(keyMapper, valueMapper)` will throw an `IllegalStateException` if it encounters duplicate keys. To handle potential collisions, you must provide a third argument, a **merge function** that specifies how to resolve the conflict (e.g., by keeping the first value, the last value, or combining them).

   ```java
   // OK: Handle duplicates, keep the first value encountered
   stream.collect(Collectors.toMap(e -> e.getDept(), e -> e, (v1, v2) -> v1));
   ```

4. **Is the order of execution guaranteed when using `forEach()` on a parallel stream?**
   No. On a parallel stream, `forEach()` executes in an arbitrary order to maximize performance. If the original order of the elements is important, you must use `forEachOrdered()`, although this may reduce the performance benefits of parallelism.

5. **Why is using `peek()` for production side effects discouraged?**
   `peek()` is intended strictly for debugging (e.g., logging elements as they flow through the pipeline). It is not guaranteed to execute for every element if the stream is short-circuited by a downstream operation (like `findFirst()` or `limit()`). For intentional side effects, you should use the terminal `forEach()` method.

6. **What is the performance difference between `Optional.orElse()` and `Optional.orElseGet()`?**
   `orElse(defaultValue)` evaluates the default value **every time**, regardless of whether the Optional is empty or not. `orElseGet(() -> defaultValue)` uses a supplier and is only executed if the Optional is actually empty. Therefore, you should always use `orElseGet()` for expensive operations (like a database call or complex object creation).

7. **How do you distinguish between the `map()` and `flatMap()` operations?**
   `map()` is a **one-to-one** transformation where each element in the stream is transformed into exactly one result. `flatMap()` is a **one-to-many** transformation; it takes each element, transforms it into a *stream* of results, and then "flattens" those nested streams into a single, flat output stream.

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Stream|Pipeline for processing data, not a data structure|
|Lazy|Intermediate ops don't run until terminal op is called|
|One-time use|Can't reuse after terminal operation|
|Intermediate ops|filter, map, flatMap, sorted, distinct, peek, limit, skip|
|Terminal ops|collect, forEach, reduce, count, min, max, findFirst, anyMatch|
|Optional|Container for nullable return, use orElse, not get()|
|Primitive streams|IntStream, LongStream, DoubleStream, avoid autoboxing|
|Parallel|Use for large data + stateless ops, avoid shared mutable state|
|Order matters|Filter early, sort/map late|
|collect vs toList()|toList() (Java 16+) = unmodifiable, collect = mutable|
|Short-circuiting|findFirst, anyMatch, limit, stop early, don't traverse all|
|Stateless vs Stateful|filter/map = stateless (parallel-friendly), sorted/distinct = stateful|
|Side effects|Avoid modifying external state inside streams|
|collect vs reduce|collect = build collections, reduce = combine into single value|
|map vs flatMap|map = 1-to-1, flatMap = 1-to-many then flatten|
|Collectors|toList, toSet, toMap, joining, groupingBy, partitioningBy|
