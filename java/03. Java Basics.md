## Program Structure

### Hello World

```java
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}
```

Structural rules:

- One public class per file
- Filename must match the class name `class HelloWorld` â†’ `HelloWorld.java`
- `main` method is the program entry point, JVM looks for this method
- Code lives inside a class; no free-floating functions or statements

### `main` Method Signature

```java
public static void main(String[] args)
```

Each piece is mandatory for the JVM to recognize this as the entry point:

- **`public`**: JVM needs to access it from outside
- **`static`**: JVM can call it without creating an object for class
- **`void`**: `main` doesn't return a value to the OS
- **`main`**: JVM looks for this exact name
- **`String[] args`**: array of command-line arguments

Alternate valid signatures:

```java
public static void main(String... args)                    // varargs form
public static final synchronized void main(String[] args)  // modifiers can combine
```

### Command-Line Arguments

While executing the `.class` file whatever you type after the class name becomes the `args` array:

```bash
java MyApp foo bar baz
```

```java
public class MyApp {
    public static void main(String[] args) {
		System.out.println(args.length);    // 3
        System.out.println(args[0]);        // "foo"
        System.out.println(args[1]);        // "bar"
    }
}
```

- `args` is never `null` â€” it's an empty array if no arguments passed
- Elements are always `String` â€” parse to numbers yourself via `Integer.parseInt(args[0])` etc.

### Parameters vs Arguments

```java
// PARAMETER is a placeholder in the method declaration
void greet(String name) {
    System.out.println("Hi " + name);
}

greet("Alice");   // ARGUMENT is the actual value passed at call time
```

## Statements, Expressions & Blocks

**Statements**: instructions that do something; ends with `;`

```java
int x = 5;                        // assignment
System.out.println("hello");      // expression statement
return x;                         // return
```

**Expressions**: produce a value.

```java
5 + 3                             // â†’ 8
x > 0                             // â†’ true/false
name.toUpperCase()                // â†’ String
new Employee("Alice")             // â†’ Employee instance
```

>**Note:** Only **assignments, increment/decrement, method calls, and object creation** can stand alone as statements. Pure arithmetic (`5 + 3;`) and comparisons (`x > 0;`) are compile errors.

**Blocks**: code grouped with `{ }` (method bodies, loops, conditionals). Variables declared in a block are only visible within it.

```java
if (condition) {
    int x = 10;                   // block-scoped
    System.out.println(x);
}                                  // x goes out of scope here
```

## Identifiers, Keywords & Naming Conventions

### Identifiers

Identifiers are names you give to classes, methods, variables, etc. and must follow these rules:

- Can contain letters, digits, `$`, `_`
- Must start with a letter, `$`, or `_`
- **Case-sensitive**: `name` and `Name` are different
- **No special characters** (other than `$` and `_`)
- **Cannot use a reserved keyword as identifier** (`int`, `class`, `public`, etc.)

### Keywords

Words the compiler reserves, these can never be used as identifiers.

```
abstract    continue    for           new         switch
assert      default     if            package     synchronized
boolean     do          goto*         private     this
break       double      implements    protected   throw
byte        else        import        public      throws
case        enum        instanceof    return      transient
catch       extends     int           short       try
char        final       interface     static      void
class       finally     long          strictfp    volatile
const*      float       native        super       while
```

`goto` and `const` are unused

**Literals** (reserved but not keywords): `true`, `false`, `null`.

**Contextual Keywords** (Restricted Identifiers): These are only reserved in specific contexts (like within a specific file type or code block). You can technically use them as identifiers elsewhere, though it is discouraged.

- **Module-related** (Java 9+): `module`, `requires`, `exports`, `opens`, `uses`, `provides`, `to`, `with` â€” only reserved within `module-info.java`
- **Local Variable Type Inference** (Java 10+): `var`
- **Switch Expressions** (Java 14+): `yield`
- **Records** (Java 16+): `record`
- **Sealed Classes** (Java 17+): `sealed`, `non-sealed`, `permits`

### Naming Conventions

Not enforced by the compiler, but universal in the Java ecosystem:

| Entity                     | Convention               | Example                          |
| -------------------------- | ------------------------ | -------------------------------- |
| Packages                   | lowercase, dot-separated | `com.example.order`              |
| Classes / Interfaces       | `PascalCase`             | `StringBuilder`, `Runnable`      |
| Methods                    | `camelCase` â€” verbs      | `getName()`, `calculateSalary()` |
| Variables / Fields         | `camelCase` â€” nouns      | `userName`, `totalPrice`         |
| Constants (`static final`) | `UPPER_SNAKE_CASE`       | `MAX_RETRIES`, `DEFAULT_TIMEOUT` |
| Generic type parameters    | Single uppercase letter  | `T`, `E`, `K`, `V`               |
| Enum constants**           | `UPPER_SNAKE_CASE`       | `PENDING`, `MAX_RETRIES`         |

## Comments

### Single-line `//`

```java
// TODO: validate input
int x = 5;  // inline comment at end of statement
```

### Multi-line `/* */`

```java
/*
 * Calculates the daily interest accrued on this account.
 * Formula: (principal * rate) / 365
 */
double dailyInterest() { ... }
```

### Javadoc `/** */`

Special multi-line comment that the `javadoc` tool extracts to generate API docs.

```java
/**
 * Transfers funds from this account to the target.
 *
 * @param target the destination account â€” must not be null
 * @param amount amount to transfer in smallest currency unit (e.g., cents)
 * @return true if transfer succeeded, false if insufficient funds
 * @throws IllegalArgumentException if amount is negative
 * @since 1.2
 */
public boolean transfer(Account target, long amount) { ... }
```

Common Javadoc tags:

|Tag|Use|
|---|---|
|`@param`|Describe each parameter|
|`@return`|Describe the return value|
|`@throws`|Exceptions the method may throw|
|`@since`|Version this was introduced|
|`@deprecated`|Mark as deprecated, say what to use instead|
|`@see`|Link to related class / method|
|`{@link ClassName}`|Inline link to another API element|

Javadoc comments must appear immediately before the class, method, or field they describe.

## `package` and `import`

### `package` Statement

First non-comment line in a file. Groups related classes into a namespace (and maps to the directory structure on disk):

```java
package com.example.orders;   // â†’ file must live in com/example/orders/ folder

public class OrderService { }
```

- Only one `package` per file
- Package name is **lowercase, dot-separated**
- Universal convention: reverse your organization's domain (`com.company.project.module`)

### `import` Statement

Appears after `package`, before class declaration. Lets you use a class by its short name instead of fully qualified name:

```java
package com.example.orders;

import java.util.List;
import java.util.ArrayList;

public class OrderService {
    private List<Order> orders = new ArrayList<>();   // instead of java.util.List
}
```

Wildcard import (brings in ALL classes from a package):

```java
import java.util.*;      // imports List, ArrayList, Map, HashMap, etc.
```

Wildcard imports are concise but can cause name conflicts. Most codebases use explicit imports.

**Classes that don't need import**: `java.lang` classes (`String`, `Integer`, `Object`, `System`, `Math`, etc.) are always available, and classes in the same package can see each other directly.

### `import static`

Import static members (fields, methods) so you can use them **without the class name prefix**:

```java
// Without static import
double result = Math.sqrt(Math.pow(x, 2) + Math.pow(y, 2));

// With static import
import static java.lang.Math.sqrt;
import static java.lang.Math.pow;

double result = sqrt(pow(x, 2) + pow(y, 2));   // cleaner
```

Common uses:

```java
import static java.util.Arrays.asList;
import static org.junit.jupiter.api.Assertions.*;   // junit assertions
```

Use sparingly â€” overuse reduces readability (reader can't tell where `sqrt` comes from).

The standard library is organized into packages. Common ones:

|Package|Contents|
|---|---|
|`java.lang`|`String`, `Integer`, `Object`, `System`, `Math`, `Thread`, `Throwable` â€” auto-imported|
|`java.util`|`List`, `Map`, `Set`, `ArrayList`, `HashMap`, `Collections`, `Random`, `Scanner`|
|`java.io`|Classic I/O â€” `File`, `InputStream`, `OutputStream`, `Reader`, `Writer`|

|`java.net`|Networking â€” `URL`, `Socket`, `HttpURLConnection`|
|`java.sql`|JDBC â€” `Connection`, `Statement`, `ResultSet`|

## Q&A

1. **Can the `String[] args` parameter in the `main` method ever be `null`?**
   No. Even if no command-line arguments are passed, the JVM initializes `args` as an empty array (`new String[0]`). This allows developers to safely check `args.length` without a null-pointer check.

2. **Why is the use of the dollar sign (`$`) in identifiers strongly discouraged?**
   While technically legal, `$` is conventionally reserved for machine-generated code. The Java compiler uses it to name nested classes (e.g., `Outer$Inner.class`). Using it in hand-written code can lead to name collisions or confusion between developer-authored code and tool-generated artifacts.

3. **Does a wildcard import like `import java.util.*` recursively include sub-packages?**
   No. Wildcard imports only bring in the classes directly within the specified package. For example, `import java.util.*` provides access to `List` and `Map`, but not to classes in `java.util.concurrent`. Sub-packages must be imported explicitly to avoid namespace pollution and keep the compilation unit focused.

4. **Why must the `main` method be declared as `static`?**
   The `static` modifier allows the JVM to invoke the entry point without first instantiating the class. If `main` were an instance method, the JVM would face a "chicken-and-egg" problem: it would need to call a constructor to create an object, but constructors themselves might contain logic that relies on the environment being fully initialized by the entry point.

## Quick Revision Cheat Sheet

| Concept                 | Remember                                                                      |
| ----------------------- | ----------------------------------------------------------------------------- |
| Filename                | Must match the public class name exactly (case-sensitive)                     |
| `main` signature        | `public static void main(String[] args)` â€” every word mandatory               |
| `args`                  | String array â€” never null, possibly empty                                     |
| Parameter vs Argument   | Parameter = declaration placeholder; Argument = actual value passed           |
| Identifier rules        | Letters/digits/`$`/`_`; can't start with digit; case-sensitive; not a keyword |
| Comments                | `//`, `/* */`, Javadoc `/** */`                                               |
| `package`               | First non-comment line; one per file                                          |
| `java.lang`             | Auto-imported â€” no import needed for `String`, `Math`, etc.                   |
| Static import           | `import static java.lang.Math.sqrt;` â€” skip class prefix                      |
