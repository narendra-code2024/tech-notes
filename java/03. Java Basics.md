## Program Structure

### Hello World

```java
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}
```

Structural & JVM entry rules:
- **One public class per file**: Filename must match the class name exactly (e.g., `HelloWorld.java`). All code must live inside classes; no free-floating statements.
- **Entry Point Signature**: The JVM specifically looks for this mandatory method signature:
  - `public`: Accessible outside the class package.
  - `static`: Executable without creating an object instance.
  - `void`: Returns nothing to the operating system.
  - `main`: The exact entry point name recognized by the JVM.
  - `String[] args` (or `String... args`): Command-line arguments accepted as a `String` array.
- **Alternative Forms**: Standard modifiers like `final` and `synchronized` can combine (e.g., `public static final synchronized void main(String[] args)`).

### Command-Line Arguments

While executing the `.class` file whatever you type after the class name becomes the `args` array:

```bash
java MyApp foo bar baz
```

```java
public class MyApp {
    public static void main(String[] args) {
        System.out.println(args.length);    // 3
        System.out.println(args[0]);        // "foo"
        System.out.println(args[1]);        // "bar"
    }
}
```

- `args` is never `null`; it's an empty array if no arguments passed
- Elements are always `String`; parse to numbers yourself via `Integer.parseInt(args[0])` etc.
- **Parameter vs Argument**: A *parameter* is the placeholder variable in the method signature (e.g., `String[] args`), whereas an *argument* is the actual value passed during invocation (e.g., `"foo"`).

## Statements, Expressions & Blocks

**Statements**: instructions that do something; ends with `;`

```java
int x = 5;                        // assignment
System.out.println("hello");      // expression statement
return x;                         // return
```

**Expressions**: produce a value.

```java
5 + 3                             // → 8
x > 0                             // → true/false
name.toUpperCase()                // → String
new Employee("Alice")             // → Employee instance
```

> **Note**: Only assignments, increment/decrement, method calls, and object creation can stand alone as statements. Pure arithmetic (`5 + 3;`) and comparisons (`x > 0;`) are compile errors.

**Blocks**: code grouped with `{ }` (method bodies, loops, conditionals). Variables declared in a block are only visible within it.

```java
if (condition) {
    int x = 10;                   // block-scoped
    System.out.println(x);
}                                  // x goes out of scope here
```

## Identifiers & Keywords

### Identifiers

Identifiers are names you give to classes, methods, variables, etc. and must follow these rules:

- Can contain letters, digits, `$`, `_`
- Must start with a letter, `$`, or `_`
- **Case-sensitive**: `name` and `Name` are different
- **No special characters** (other than `$` and `_`)
- **Cannot use a reserved keyword as identifier** (`int`, `class`, `public`, etc.)

**Naming Conventions**: not enforced by the compiler, but universal in the Java ecosystem.

| Entity                     | Convention               | Example                          |
| -------------------------- | ------------------------ | -------------------------------- |
| Packages                   | lowercase, dot-separated | `com.example.order`              |
| Classes / Interfaces       | `PascalCase`             | `StringBuilder`, `Runnable`      |
| Methods                    | `camelCase` (verbs)      | `getName()`, `calculateSalary()` |
| Variables / Fields         | `camelCase` (nouns)      | `userName`, `totalPrice`         |
| Constants (`static final`) | `UPPER_SNAKE_CASE`       | `MAX_RETRIES`, `DEFAULT_TIMEOUT` |
| Generic type parameters    | Single uppercase letter  | `T`, `E`, `K`, `V`               |
| Enum constants             | `UPPER_SNAKE_CASE`       | `PENDING`, `MAX_RETRIES`         |

### Keywords

Words reserved by the compiler that can never be used as identifiers.

```
abstract    continue    for           new         switch
assert      default     if            package     synchronized
boolean     do          goto*         private     this
break       double      implements    protected   throw
byte        else        import        public      throws
case        enum        instanceof    return      transient
catch       extends     int           short       try
char        final       interface     static      void
class       finally     long          strictfp    volatile
const*      float       native        super       while
```

* `goto` and `const` (marked with `*`) are reserved but currently unused in Java.
* **Literals** (reserved values, not keywords): `true`, `false`, `null`.

**Contextual Keywords** (Restricted Identifiers): Only reserved within specific language contexts; usable as standard identifiers elsewhere (though discouraged for clarity).

- **Module-related** (Java 9+): `module`, `requires`, `exports`, `opens`, `uses`, `provides`, `to`, `with` (reserved only inside `module-info.java`).
- **Local Variable Type Inference** (Java 10+): `var`.
- **Switch Expressions** (Java 14+): `yield`.
- **Records** (Java 16+): `record`.
- **Sealed Classes** (Java 17+): `sealed`, `non-sealed`, `permits`.

## Comments

Java supports three types of comments for annotations and documentation:

**Single-line Comments (`//`)**: Used for short notes or inline explanations.

```java
// TODO: validate input before processing
int x = 5; // inline comment at end of statement
```

**Multi-line Comments (`/* ... */`)**: Used for block notes, multi-line formulas, or disabling code blocks.

```java
/*
 * Calculates the daily interest accrued on this account.
 * Formula: (principal * rate) / 365
 */
double calculateDailyInterest(double principal, double rate) {
    return (principal * rate) / 365.0;
}
```

**Javadoc Comments (`/** ... */`)**: Special multi-line comments processed by the `javadoc` tool to generate HTML-based API documentation. These must appear immediately before the class, method, or field they describe.

```java
/**
 * Transfers funds from this account to the destination target.
 *
 * @param target the destination account (must not be null)
 * @param amount the transfer amount in cents
 * @return true if the transfer succeeded, false otherwise
 */
public boolean transfer(Account target, long amount) {
    return false; // stub implementation
}
```

Common Javadoc tags:

| Tag           | Use                                                     |
| :------------ | :------------------------------------------------------ |
| `@param`      | Describes a method parameter                            |
| `@return`     | Describes the return value                              |
| `@throws`     | Documents exceptions the method may throw               |
| `@since`      | Specifies the version when this was introduced          |
| `@deprecated` | Marks as deprecated and suggests an alternative         |

## Packages & Imports

Java uses packages for namespaces and directory grouping, and imports to reference classes outside the active package.

```java
package com.example.orders;   // 1. Package: must live in com/example/orders/ folder

import java.util.List;        // 2. Imports: allows using List instead of java.util.List
import java.util.ArrayList;
import static java.lang.Math.sqrt; // 3. Static Import: skips ClassName prefix for static members
import static java.lang.Math.pow;

public class OrderService {
    private List<Order> orders = new ArrayList<>();
    
    double hypotenuse(double a, double b) {
        return sqrt(pow(a, 2) + pow(b, 2)); // Used directly without "Math."
    }
}
```

Key Rules:
- **Package Convention**: Dot-separated and lowercase, using the reversed domain format (e.g., `com.company.project`). Only one package statement is allowed per file.
- **Implicit Imports**: Classes in `java.lang` (`String`, `System`, `Math`, etc.) and classes in the same package are **automatically imported** and available without declaration.
- **Wildcard Imports (`*`)**: Brings in all classes within a single package (e.g., `import java.util.*;`). Note that wildcards are **not recursive** (they do not import sub-packages like `java.util.concurrent`).

### Common Standard Packages

| Package                | Contents                                                                                     |
| :--------------------- | :------------------------------------------------------------------------------------------- |
| `java.lang`            | Core fundamentals (`String`, `Integer`, `Object`, `System`, `Math`, `Thread`); auto-imported |
| `java.util`            | Collections, date/time, and utilities (`List`, `Map`, `ArrayList`, `HashMap`, `Scanner`)      |
| `java.io` / `java.nio` | Input/Output, paths, and dynamic file streaming (`File`, `InputStream`, `Path`, `Files`)      |
| `java.time`            | Modern thread-safe date and time API (`LocalDate`, `LocalDateTime`, `Duration`)               |
| `java.util.concurrent` | High-level concurrent utilities (`ExecutorService`, `Future`, `ConcurrentHashMap`)            |
| `java.util.stream`     | Functional stream operations and pipelines (`Stream`, `Collectors`)                           |

## Q&A

1. **Can the `String[] args` parameter in the `main` method ever be `null`?**
No. Even if no command-line arguments are passed, the JVM initializes `args` as an empty array (`new String[0]`). This allows developers to safely check `args.length` without a null-pointer check.

2. **Why is the use of the dollar sign (`$`) in identifiers strongly discouraged?**
While technically legal, `$` is conventionally reserved for machine-generated code. The Java compiler uses it to name nested classes (e.g., `Outer$Inner.class`). Using it in hand-written code can lead to name collisions or confusion between developer-authored code and tool-generated artifacts.

3. **Does a wildcard import like `import java.util.*` recursively include sub-packages?**
No. Wildcard imports only bring in the classes directly within the specified package. For example, `import java.util.*` provides access to `List` and `Map`, but not to classes in `java.util.concurrent`. Sub-packages must be imported explicitly to avoid namespace pollution and keep the compilation unit focused.

4. **Why must the `main` method be declared as `static`?**
The `static` modifier allows the JVM to invoke the entry point without first instantiating the class. If `main` were an instance method, the JVM would face a "chicken-and-egg" problem: it would need to call a constructor to create an object, but constructors themselves might contain logic that relies on the environment being fully initialized by the entry point.

## Quick Revision Cheat Sheet

| Concept                 | Remember                                                                      |
| ----------------------- | ----------------------------------------------------------------------------- |
| Filename                | Must match the public class name exactly (case-sensitive)                     |
| `main` signature        | `public static void main(String[] args)`; every word mandatory                |
| `args`                  | String array; never null, possibly empty                                      |
| Parameter vs Argument   | Parameter = declaration placeholder; Argument = actual value passed           |
| Identifier rules        | Letters/digits/`$`/`_`; can't start with digit; case-sensitive; not a keyword |
| Comments                | `//`, `/* */`, Javadoc `/** */`                                               |
| `package`               | First non-comment line; one per file                                          |
| `java.lang`             | Auto-imported; no import needed for `String`, `Math`, etc.                    |
| Static import           | `import static java.lang.Math.sqrt;`, skips the class prefix                   |
