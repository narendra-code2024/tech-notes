An enum defines a fixed, finite set of named constants as a type-safe class; each constant is a singleton instance of that class, not a bare `int` or `String`.

Unlike `public static final` constants, enums are compiler-verified (invalid values are rejected), enumerable via `values()`, and unlock `EnumSet`/`EnumMap` optimizations.

---

## Syntax & Built-in Methods

Constants are singletons, so `==` is safe and idiomatic (no `equals()` needed).

```java
enum Color { RED, GREEN, BLUE }

Color c = Color.RED;
c == Color.RED;          // true
Color.values();          // Color[] in declaration order (compiler-generated)
Color.valueOf("RED");    // Color.RED; throws IllegalArgumentException on no match
Enum.valueOf(Color.class, "RED"); // generic form; useful when type is a parameter

Color g = Color.GREEN;
g.name();                // "GREEN"
g.ordinal();             // 1
g.toString();            // "GREEN" unless overridden
g.compareTo(Color.RED);  // positive int: GREEN declared after RED
```

> **Note**: never persist `ordinal()`; inserting or reordering constants silently corrupts stored values. Persist `name()` instead.

## Fields, Constructors, and Methods

Each constant invokes the constructor with its own arguments. Constructors are implicitly private.

```java
enum Planet {
    MERCURY(3.303e+23, 2.4397e6),
    VENUS  (4.869e+24, 6.0518e6),
    EARTH  (5.976e+24, 6.37814e6);

    private final double mass;
    private final double radius;

    Planet(double mass, double radius) {
        this.mass   = mass;
        this.radius = radius;
    }

    public double surfaceGravity() {
        return 6.67300E-11 * mass / (radius * radius);
    }
}

Planet.EARTH.surfaceGravity(); // 9.80...
```

## Constant-Specific Behavior

Each constant can override an abstract method declared on the enum, or a method from an implemented interface, encoding per-constant dispatch without a `switch`.

**Abstract method:**

```java
enum Operation {
    ADD      { public int apply(int a, int b) { return a + b; } },
    SUBTRACT { public int apply(int a, int b) { return a - b; } },
    MULTIPLY { public int apply(int a, int b) { return a * b; } };

    public abstract int apply(int a, int b);
}

Operation.ADD.apply(2, 3); // 5
```

**Interface implementation:**

```java
interface Describable { String describe(); }

enum Status implements Describable {
    ACTIVE   { public String describe() { return "running"; } },
    INACTIVE { public String describe() { return "stopped"; } };
}

Describable d = Status.ACTIVE;
d.describe(); // "running"
```

> **Note**: sealed interfaces (Java 17+) are a competing pattern for per-type dispatch; prefer them when the variants are not a fixed domain concept or when the set may grow.

## Enums in `switch`

Constants are referenced unqualified inside `case` (`PAID`, not `OrderStatus.PAID`).

```java
// Classic: silently ignores a newly added constant if no default is present
switch (status) {
    case PENDING   -> notify("waiting");
    case PAID      -> shipOrder();
    case CANCELLED -> refund();
}

// Exhaustive expression (Java 14+): compiler error until every constant is handled
String label = switch (status) {
    case PENDING, PAID -> "In Progress";
    case SHIPPED       -> "On the way";
    case DELIVERED     -> "Done";
    case CANCELLED     -> "Refunded";
};
```

Prefer exhaustive switch expressions; they make newly added constants a compile error, not a silent logic bug.

## `EnumSet` and `EnumMap`

`EnumSet` is backed by a bit-vector; `EnumMap` by an array indexed on `ordinal()`. Both are faster and smaller than `HashSet`/`HashMap` over enums and preserve declaration order.

```java
EnumSet<OrderStatus> active = EnumSet.of(OrderStatus.PENDING, OrderStatus.PAID);
EnumSet<OrderStatus> all    = EnumSet.allOf(OrderStatus.class);
EnumSet<OrderStatus> none   = EnumSet.noneOf(OrderStatus.class);
EnumSet<OrderStatus> range  = EnumSet.range(OrderStatus.PENDING, OrderStatus.SHIPPED);
EnumSet<OrderStatus> rest   = EnumSet.complementOf(active); // all constants not in active

active.contains(OrderStatus.PAID); // O(1) bitwise check

EnumMap<OrderStatus, String> labels = new EnumMap<>(OrderStatus.class);
labels.put(OrderStatus.PENDING, "Waiting for payment");
labels.put(OrderStatus.PAID,    "Payment received");
```

## Singleton Pattern

A single-constant enum is the safest Singleton (Effective Java, Bloch).

```java
enum Database {
    INSTANCE;

    private final Connection conn;

    Database() { this.conn = connect(); }

    public void query(String sql) { /* use conn */ }
}

Database.INSTANCE.query("SELECT * FROM users");
```

Advantages over double-checked locking:

- **Serialization-safe**: deserialization resolves the name to the existing singleton; the constructor does not re-run.
- **Reflection-safe**: the JVM forbids constructing enum constants via reflection.
- **Thread-safe**: class-loading guarantees one-time initialization with no explicit synchronization.

## Real-World Patterns

### State Machine

```java
enum OrderStatus {
    PENDING, PAID, SHIPPED, DELIVERED, CANCELLED;

    public boolean isTerminal() {
        return this == DELIVERED || this == CANCELLED;
    }

    public boolean canTransitionTo(OrderStatus next) {
        return switch (this) {
            case PENDING   -> next == PAID      || next == CANCELLED;
            case PAID      -> next == SHIPPED   || next == CANCELLED;
            case SHIPPED   -> next == DELIVERED;
            case DELIVERED, CANCELLED -> false;
        };
    }
}
```

### Roles / Permissions

```java
enum UserRole {
    ADMIN  (Set.of("read", "write", "delete", "manage")),
    EDITOR (Set.of("read", "write")),
    VIEWER (Set.of("read"));

    private final Set<String> permissions;
    UserRole(Set<String> p) { this.permissions = p; }

    public boolean can(String action) { return permissions.contains(action); }
}

if (currentUser.getRole().can("delete")) { /* ... */ }
```

## Q&A

1. **Why must `ordinal()` never be persisted?**
`ordinal()` is a zero-based declaration index; inserting or reordering any constant silently shifts every subsequent value, corrupting stored data. Persist `name()` instead, and use a named field (e.g. `int code`) when a stable numeric identity is required.

2. **What happens when a new constant is added to an enum used in a `switch`?**
A classic `switch` without `default` silently skips the new constant. An exhaustive switch expression (Java 14+) fails to compile until the new constant is handled, which is why it is preferred.

3. **How does enum serialization work, and what breaks it?**
Only `name()` is written to the byte stream; on deserialization the JVM resolves that name to the existing singleton without re-running the constructor. Renaming a constant breaks compatibility with persisted data. Jackson serializes to `name()` by default, so renaming also silently breaks JSON APIs.

4. **What is the natural ordering of enum constants?**
Enums implement `Comparable` ordered by declaration (`ordinal()`). Supply a `Comparator` for any other order.

5. **When should an enum be avoided?**
When the set of values is not known at compile time (e.g., categories driven by a database or editable config). Enums are closed at compile time and cannot grow at runtime.

## Quick Revision Cheat Sheet

| Concept               | Remember                                                                         |
| :-------------------- | :------------------------------------------------------------------------------- |
| Singleton guarantee   | Constants are singletons; `==` is safe and idiomatic                             |
| `ordinal()`           | Declaration index only; never persist, never use for business logic              |
| `valueOf()` forms     | `Color.valueOf("RED")` (class-specific) vs `Enum.valueOf(Color.class, "RED")`    |
| `EnumSet` / `EnumMap` | Bit-vector / array-backed; faster than `HashSet`/`HashMap`; prefer always        |
| `complementOf()`      | Returns all constants not in a given `EnumSet`                                   |
| Switch safety         | Exhaustive switch expression (Java 14+) enforces compile-time completeness       |
| Constant-specific behavior | Abstract method or interface override per constant; alternative: sealed interfaces (Java 17+) |
| Singleton pattern     | `enum X { INSTANCE; }`: serialization-, reflection-, and thread-safe             |
| Serialization         | Only `name()` persisted; constructor does not re-run on deserialization          |
| Jackson               | Serializes as `name()` by default; renaming a constant silently breaks JSON APIs |
| Can extend?           | No classes (implicitly extends `Enum`); interfaces only                          |
| Avoid when            | Values are runtime-dynamic (DB, config, API); enums are compile-time-closed      |
