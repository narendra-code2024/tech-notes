The JVM is the managed runtime that loads, verifies, and executes Java bytecode. Its architecture rests on three pillars: the ClassLoader subsystem, the runtime data areas (memory), and the execution engine (interpreter, JIT, and garbage collector).

> **Reference**: for JDK vs JRE vs JVM and the high-level picture, see [JDK, JRE & JVM](<02. JDK, JRE & JVM.md>).

---

## ClassLoader Subsystem

Loads `.class` files into memory in three phases: loading, linking, and initialization.

### Loading

| ClassLoader            | Loads                                                                              |
| ---------------------- | --------------------------------------------------------------------------------- |
| Bootstrap              | Core classes (`java.lang.*`, `java.util.*`) from the runtime image (modules in Java 9+, `rt.jar` before) |
| Platform (Extension)   | Other platform/JDK modules. Pre-Java-9 this was the Extension loader reading `jre/lib/ext`, removed in Java 9 |
| Application (System)   | Your application classes from the classpath                                       |

- **Delegation (parent-first)**: a loader asks its parent (Application → Platform → Bootstrap) before loading itself. This prevents duplicate loading and stops application code from shadowing core classes.

### Linking

1. **Verification**: confirms the bytecode is well-formed and obeys JVM rules.
2. **Preparation**: allocates static fields and sets them to default values (`0`, `null`, `false`).
3. **Resolution**: replaces symbolic references with direct references, often performed lazily on first use.

### Initialization

Runs static initializers and assigns the real values to static fields, top-down in source order. It happens once, lazily, on the first active use of the class (first instance, static field access, or static method call).

```java
class Example {
    static int x = 10;   // Preparation sets x = 0, Initialization sets x = 10
    static {
        System.out.println("static block");   // runs during initialization
    }
}
```

## Runtime Data Areas

### Method Area (Metaspace, Java 8+)

- Stores class metadata: class structure, method and field info, and the runtime constant pool. Shared across all threads.
- Metaspace (Java 8+) replaced PermGen and lives in native memory, growing dynamically. PermGen had a fixed size and routinely threw `OutOfMemoryError: PermGen space`; Metaspace can still be capped with `-XX:MaxMetaspaceSize`.
- Static variables are conceptually part of this area but in HotSpot (Java 8+) actually reside on the heap, attached to the class's `Class` object.

### Heap

Shared across all threads and managed by the garbage collector. Holds all objects, their instance variables, and the String Constant Pool (moved here from PermGen in Java 7).

```
Heap
├── Young Generation
│   ├── Eden       (new objects start here)
│   ├── Survivor S0
│   └── Survivor S1
└── Old Generation (long-lived objects promoted here)
```

- New objects start in Eden; survivors are copied between S0/S1, and past an age threshold are promoted to Old Gen.
- Minor GC collects the Young gen (fast, frequent); Full GC collects the whole heap including Old Gen (slower, rarer). Most collections involve Stop-The-World pauses; G1 and ZGC are designed to minimize them.

The String Constant Pool interns string literals, so equal literals share one object:

```java
String s1 = "hello";              // interned in the pool
String s2 = "hello";              // reuses the same pooled object
String s3 = new String("hello");  // new heap object, "hello" still interned in the pool

s1 == s2;            // true, same interned object
s1 == s3;            // false, different objects
s1 == s3.intern();   // true, intern() returns the pooled reference
```

### Stack (per thread)

- Each method call pushes a frame holding its local variables, operand stack, and return address; the frame pops on return (LIFO).
- Holds primitives and object references, never the objects themselves, so stack data is inherently thread-safe. Size is set with `-Xss`.
- `StackOverflowError` is thrown when frames exhaust the stack, usually from unbounded recursion.

```java
void methodA() {
    int x = 10;          // primitive on methodA's frame
    String s = "hello";  // reference on the frame, object in the pool/heap
    methodB();           // pushes methodB's frame
}                        // methodA returns, frame popped
```

### PC Register and Native Method Stack (per thread)

- **PC Register**: holds the address of the JVM instruction the thread is currently executing; undefined while running a native method.
- **Native Method Stack**: supports native (JNI, C/C++) calls, kept separate from the Java stack.

### Memory summary

| Area         | Stores                                      | Scope               |
| ------------ | ------------------------------------------- | ------------------- |
| Method Area  | Class metadata, static vars                 | Shared, all threads |
| Heap         | Objects, instance vars, String Pool         | Shared, all threads |
| Stack        | Local vars, method frames                   | Per thread          |
| PC Register  | Current instruction address                 | Per thread          |
| Native Stack | Native method calls                         | Per thread          |

> **Note**: references live on the stack and objects on the heap, but escape analysis may scalar-replace or stack-allocate objects that never leave a method. Per-thread areas (stack, PC, native stack) are inherently thread-safe; the shared method area and heap require synchronization.

## Execution Engine

### Interpreter

Executes bytecode instruction by instruction: quick to start, but slow under repetition because hot code is re-interpreted every pass.

### JIT Compiler

Detects hot methods and loops, compiles them to cached native code at runtime, and reuses it. It uses tiered compilation: C1 compiles fast with light optimization (good startup), C2 compiles slower with aggressive optimization (peak throughput).

| Aspect   | Interpreter             | JIT                         |
| -------- | ----------------------- | --------------------------- |
| Speed    | Slow, re-interprets     | Fast, native after compile  |
| Startup  | Fast                    | Slower, compilation cost    |
| Memory   | Low                     | Higher, stores native code  |
| Used for | Cold code (runs rarely) | Hot code (runs frequently)  |

> **Note**: the JVM uses both, interpreting cold code and JIT-compiling hot paths. This is why Java apps "warm up", the first requests are slower and then speed up.

### Garbage Collector

Reclaims unreachable heap objects. Reachability is computed from GC Roots (stack references, static fields, JNI handles), not reference counting, which cannot reclaim reference cycles. Any object not reachable from a root is eligible for collection.

```java
Employee e = new Employee();   // object on heap, reference on stack
e = null;                       // unreachable, now eligible for GC
```

| GC                      | Behavior                          | Use case                  |
| ----------------------- | --------------------------------- | ------------------------- |
| Serial                  | Single-threaded, stop-the-world   | Small or client JVMs      |
| Parallel                | Multi-threaded, throughput-first  | Batch, throughput-bound   |
| G1 (default, Java 9+)   | Region-based, predictable pauses  | General purpose, balanced |
| ZGC (Java 15+)          | Concurrent, sub-millisecond pauses (generational since JDK 21) | Large heaps, low latency |

- `System.gc()` is only a hint; the JVM decides when to collect.
- `finalize()` is deprecated; use try-with-resources or `Cleaner` for cleanup.

## Object Creation Flow

What `new Employee("Alice", 50000)` triggers:

```
1. Class loading      ClassLoader loads Employee.class if not already loaded
2. Memory allocation  JVM reserves heap space for the object
3. Default init       all fields set to defaults (0, null, false)
4. Constructor        constructor runs, fields set to actual values
5. Reference          a stack reference is bound to the heap object
```

The class metadata and static fields sit in the Method Area, the object and its instance fields on the Heap, and the reference variable on the Stack.

## Common JVM Flags

```bash
-Xms256m                          # initial heap size
-Xmx1024m                         # max heap size
-Xss512k                          # stack size per thread
-XX:MaxMetaspaceSize=256m         # cap Metaspace
-XX:+UseG1GC                      # select G1 (default Java 9+)
-XX:+UseZGC                       # select ZGC (Java 15+)
-Xlog:gc                          # unified GC logging (Java 9+)
-XX:+HeapDumpOnOutOfMemoryError   # dump heap on OOM
```

## Q&A

1. **Are objects always allocated on the heap?**
Not necessarily. When escape analysis proves an object never escapes its method, the JIT can scalar-replace it (keep its fields in registers/stack) or stack-allocate it, so it may never touch the heap. Conceptually, though, references live on the stack and objects on the heap.

2. **What is the difference between PermGen and Metaspace?**
PermGen (Java 7 and earlier) stored class metadata in a fixed-size region inside the heap, which frequently caused `OutOfMemoryError: PermGen space`. Java 8 replaced it with Metaspace, which uses native memory and grows dynamically, though it can still be bounded with `-XX:MaxMetaspaceSize`.

3. **Why was the String Constant Pool moved from PermGen to the heap in Java 7?**
In PermGen, interned strings were rarely collected and added constant memory pressure. On the heap they participate in generational GC and benefit from the larger, better-managed space.

4. **How do you distinguish the common `OutOfMemoryError` variants?**

- **Java heap space**: too many live objects for the configured heap.
- **Metaspace**: class metadata exceeds native memory, common with heavy reflection or dynamic proxy generation.
- **Unable to create new native thread**: thread count hits an OS or memory limit.
- **GC overhead limit exceeded**: the JVM spends over 98% of time in GC while reclaiming under 2% of the heap.

5. **Can a garbage-collected language like Java still leak memory?**
Yes. If objects remain reachable but are no longer needed, the GC cannot reclaim them. Classic causes are ever-growing static collections, unclosed resources, and forgotten listeners/callbacks held by long-lived objects.

6. **What is the difference between `ClassNotFoundException` and `NoClassDefFoundError`?**
`ClassNotFoundException` is a checked exception thrown when code explicitly loads a class by name (`Class.forName`, `loadClass`) and it isn't on the classpath. `NoClassDefFoundError` is an `Error` thrown by the JVM when a class that was present at compile time is missing at runtime, or when its static initialization previously failed.

## Quick Revision Cheat Sheet

| Concept             | Remember                                                            |
| ------------------- | ------------------------------------------------------------------ |
| ClassLoader         | Parent-first delegation: Bootstrap → Platform → Application         |
| Linking             | Verify → Prepare (defaults) → Resolve (symbolic refs, often lazy)   |
| Method Area         | Class metadata; Metaspace (native, Java 8+) replaced PermGen        |
| Heap                | Objects + instance vars; Young (Eden/S0/S1) + Old; GC-managed       |
| Stack               | Per-thread frames: locals, operand stack, return address; LIFO      |
| String pool         | In heap since Java 7; `new String()` skips the pool, `intern()` joins it |
| Interpreter vs JIT  | Interpret cold code, C1/C2-compile hot code to native (warm-up)     |
| GC roots            | Reachability from roots, not reference counting; `System.gc()` is a hint |
| G1 / ZGC            | G1 default (Java 9+); ZGC sub-ms pauses for large, low-latency heaps |
| StackOverflowError  | Stack frames exhausted, usually infinite recursion                  |
| OutOfMemoryError    | Heap / Metaspace / native-thread / GC-overhead variants             |
| finalize()          | Deprecated; use try-with-resources or `Cleaner`                     |
