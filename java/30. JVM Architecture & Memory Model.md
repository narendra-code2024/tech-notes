## ClassLoader Subsystem

Loads `.class` files into memory in three phases:

### Loading, who loads what

|ClassLoader|What it loads|
|---|---|
|**Bootstrap**|Core Java classes (`java.lang.*`, `java.util.*`) from runtime image (modules in Java 9+, `rt.jar` in older versions)|
|**Extension/Platform**|Extension classes from `jre/lib/ext`|
|**Application/System**|Your application classes from classpath|

**Delegation model (Parent-first):** Application → Extension → Bootstrap. Child asks parent first. Only loads itself if parent can't find it. Prevents duplicate class loading and ensures core classes can't be overridden.

### Linking

1. **Verification**: checks bytecode is valid and doesn't violate JVM rules
2. **Preparation**: allocates memory for static variables, assigns default values
3. **Resolution**: replaces symbolic references with actual memory references

### Initialization

- Executes static blocks and assigns actual values to static variables
- Happens top-down in order of appearance

```java
class Example {
    static int x = 10;          // Preparation: x = 0, Initialization: x = 10
    static {
        System.out.println("Static block");  // runs during initialization
    }
}
```

## Runtime Data Areas (Memory)

### Method Area (Metaspace in Java 8+)

- Stores **class metadata**: class structure, method info, field info, constant pool
- Shared across all threads
- **Metaspace** (Java 8+), replaced PermGen. Uses native memory, grows dynamically
    - PermGen had fixed size → `OutOfMemoryError: PermGen space` (common pre-Java 8)
    - Metaspace grows as needed → less OOM issues (but can still be limited with `-XX:MaxMetaspaceSize`)
- **Static variables**: conceptually part of Method Area, but in HotSpot JVM (Java 8+) they are stored in the **Heap**

### Heap

- Stores **all objects** and **instance variables**: Shared across all threads
- Managed by Garbage Collector
- **String Constant Pool** lives here (since Java 7; was in PermGen before)

```
Heap
├── Young Generation
│   ├── Eden Space (new objects created here)
│   ├── Survivor S0
│   └── Survivor S1
└── Old Generation (long-lived objects promoted here)
```

- **New objects** → Eden → if survive GC → Survivor → if survive multiple GCs (age threshold) → promoted to Old Gen
- **Minor GC**: cleans Young Generation (fast, frequent)
- **Major GC / Full GC**: cleans entire heap including Old Gen (slow, infrequent)
- Most GC operations involve **Stop-The-World (STW)** pauses, all application threads are paused. Modern GCs (G1, ZGC) aim to minimize pause time.

### Stack (per thread)

- Stores **local variables**, **method calls** (frames), and **references** to heap objects
- Each method call creates a **stack frame** (local vars + operand stack + return address)
- **Per-thread**: each thread has its own stack
- LIFO order, method finishes → frame popped
- `StackOverflowError`, stack is full (usually infinite recursion)

```java
void methodA() {
    int x = 10;         // x on stack
    String s = "hello"; // reference s on stack, object "hello" on heap
    methodB();           // new frame pushed
}
void methodB() {
    int y = 20;          // y on stack (methodB's frame)
}
// methodB returns → its frame popped
// methodA returns → its frame popped
```

### PC Register (per thread)

- Holds address of the **currently executing JVM instruction** for each thread (undefined for native methods)

### Native Method Stack (per thread)

- For **native methods** (written in C/C++ via JNI)
- Separate from Java stack

### Memory summary

|Area|Stores|Shared?|Thread-safe?|
|---|---|---|---|
|**Method Area**|Class metadata, static vars|Yes (all threads)|Needs synchronization|
|**Heap**|Objects, instance vars, String Constant Pool|Yes (all threads)|Needs synchronization|
|**Stack**|Local vars, method frames|No (per thread)|Inherently safe|
|**PC Register**|Current instruction address|No (per thread)|Inherently safe|
|**Native Stack**|Native method calls|No (per thread)|Inherently safe|

**Punchline:** Typically, references on stack, objects on heap. However, JVM optimizations (escape analysis) may allocate some objects on stack or eliminate them entirely. Static stuff conceptually in Method Area, actually in Heap (HotSpot Java 8+).

## Execution Engine

### Interpreter

- Reads and executes bytecode **line by line**: Fast startup but slow overall (re-interprets repeated code)

### JIT (Just-In-Time) Compiler

- Identifies **hot code** (frequently executed methods/loops)
- Compiles hot bytecode to **native machine code** at runtime
- First few calls = interpreted, then JIT kicks in = much faster
- Compiled native code is cached and reused
- Uses **tiered compilation** (C1 + C2 compilers):
    - C1, fast compilation, less optimized (quick startup)
    - C2, slower compilation, highly optimized (peak performance)

### Interpreter vs JIT

||Interpreter|JIT|
|---|---|---|
|Speed|Slow (re-interprets each time)|Fast (compiles to native once)|
|Startup|Fast|Slow (compilation overhead)|
|Memory|Low|Higher (stores compiled code)|
|Used for|Cold code (runs once)|Hot code (runs frequently)|

**Java uses both:** Interpreter for quick startup, JIT for performance-critical paths. This is why Java apps "warm up", first few requests are slower, then get faster.

### Garbage Collector

Automatically reclaims memory from **unreachable objects** in heap.

**Object is unreachable when**: No reference points to it
- Set to `null`
- Object goes out of scope

JVM uses **reachability analysis from GC Roots** (not simple reference counting). GC Roots include: stack references, static fields, JNI references. Any object not reachable from a GC Root is eligible for collection.

```java
Employee e = new Employee();   // object on heap, ref on stack
e = null;                       // object is now unreachable → eligible for GC
// OR
void method() {
    Employee e = new Employee();
}   // method returns → e out of scope → object eligible for GC
```

**Key GC types:**

|GC|Behavior|Use case|
|---|---|---|
|**Serial GC**|Single-threaded, stop-the-world|Small apps, client JVMs|
|**Parallel GC**|Multi-threaded young gen|Throughput-focused|
|**G1 GC** (default Java 9+)|Region-based, predictable pauses|General purpose, balanced|
|**ZGC** (Java 15+)|Ultra-low pause (< 10ms)|Large heaps, low-latency|

**You can't force GC**: `System.gc()` is a suggestion, not a command. JVM decides when to run GC.

**`finalize()`**: deprecated (Java 9+). Don't use it. Use try-with-resources or `Cleaner` instead.

## Object Creation Flow

What happens when you write `new Employee("Alice", 50000)`?

```
1. Class Loading     → ClassLoader loads Employee.class (if not already loaded)
2. Memory Allocation → JVM allocates memory on Heap for the object
3. Default Init      → All fields set to defaults (0, null, false)
4. Constructor       → Constructor runs, sets actual values
5. Reference         → Reference variable on Stack points to Heap object
```

```java
Employee emp = new Employee("Alice", 50000);

// Stack: emp (reference) → points to Heap
// Heap: Employee object { name="Alice", salary=50000 }
// Method Area: Employee class metadata, static fields
```

## String Memory

```java
String s1 = "hello";              // String Pool (in Heap)
String s2 = "hello";              // reuses same pool reference
String s3 = new String("hello");  // new object in Heap (outside pool)

// s1, s2 → same pool object
// s3 → different object in heap
// "hello" literal also exists in pool (total: 2 objects for s3 line)
```

## JVM Flags (Commonly Used)

```bash
# Heap size
-Xms256m         # initial heap size
-Xmx1024m        # max heap size

# Stack size
-Xss512k         # stack size per thread

# Metaspace
-XX:MaxMetaspaceSize=256m

# GC selection
-XX:+UseG1GC              # use G1 (default Java 9+)
-XX:+UseZGC               # use ZGC (Java 15+)

# GC logging
-verbose:gc
-Xlog:gc                  # Java 9+ unified logging

# Debugging
-XX:+HeapDumpOnOutOfMemoryError    # dump heap on OOM
```

## Q&A

1. **How does Java decide whether to store data on the Stack or the Heap?**
   Java follows a simple rule: **Objects and their instance variables** always go on the Heap, while **method frames, local variables, and primitive values** are stored on the Stack. For example, in `int x = 5;`, both the variable name and its value are on the Stack. In `String s = "hello";`, the reference `s` is on the Stack, but the actual String object resides on the Heap.

2. **What is the difference between PermGen and Metaspace?**
   PermGen (Permanent Generation) was used in Java 7 and earlier to store class metadata. It had a fixed maximum size, which frequently led to `OutOfMemoryError: PermGen space`. In Java 8, it was replaced by **Metaspace**, which uses **native memory** (outside the JVM heap). Metaspace grows dynamically by default, making it much more robust, although it can still run out of memory if an application loads an excessive number of classes.

3. **Why was the String Constant Pool moved from PermGen to the Heap in Java 7?**
   Moving the pool to the main Heap allows the String literals to be **garbage collected** more efficiently. In PermGen, Strings were rarely collected, leading to memory pressure. In the Heap, they benefit from the sophisticated generational garbage collection algorithms and the generally larger space available.

4. **How can you distinguish between the different types of `OutOfMemoryError`?**
   *   **`Java heap space`**: The most common type; occurs when the application creates more objects than the allocated heap can hold.
   *   **`Metaspace`**: Occurs when class metadata exceeds the available native memory (common in apps using heavy reflection or dynamic proxy generation).
   *   **`Unable to create new native thread`**: Occurs when the application tries to start more threads than the operating system's limits allow.
   *   **`GC overhead limit exceeded`**: A safety check where the JVM throws an error because it is spending over 98% of its time doing GC and recovering less than 2% of the heap.

5. **Is it possible to have a memory leak in a garbage-collected language like Java?**
   Yes. A memory leak in Java occurs when objects are **still referenced** by the application but are **no longer needed**. Because they are reachable, the Garbage Collector cannot reclaim them. Common causes include static collections that grow indefinitely, unclosed database connections or file handles, and forgotten event listeners.

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|ClassLoader|Bootstrap → Extension → Application (parent-first delegation)|
|Method Area|Class metadata, static vars. Metaspace in Java 8+ (native memory)|
|Heap|Objects + instance vars. Shared. GC-managed|
|Stack|Local vars + method frames. Per-thread. LIFO|
|Interpreter|Line-by-line. Fast startup, slow execution|
|JIT|Compiles hot code to native. Slow startup, fast execution|
|GC|Auto-reclaims unreachable objects. Can't be forced|
|G1 GC|Default Java 9+. Region-based, balanced|
|StackOverflowError|Stack full, usually infinite recursion|
|OutOfMemoryError|Heap full, too many objects or memory leak|
|String Pool|In Heap since Java 7. Was in PermGen before|
|Pass-by-value|Java always passes value. For objects, value = reference|
