**Encapsulation** bundles an object's data with the methods that operate on it, hiding the internal state and exposing it only through public methods. The object decides what is visible and how it can be modified.

```java
// Without encapsulation, anyone can set invalid data
class Employee {
    public int salary;
}
Employee e = new Employee();
e.salary = -5000;    // no validation, invalid state

// With encapsulation, controlled access
class Employee {
    private int salary;

    public void setSalary(int salary) {
        if (salary < 0) throw new IllegalArgumentException("Salary can't be negative");
        this.salary = salary;
    }

    public int getSalary() { return salary; }
}
```

Encapsulation is data hiding + controlled access, not just "private fields + getters/setters". The point is **validation and control**.

## Access Modifiers

| Modifier               | Same Class | Same Package | Subclass (diff package) | Other Package |
| ---------------------- | ---------- | ------------ | ----------------------- | ------------- |
| `private`              | Yes        | No           | No                      | No            |
| `default` (no keyword) | Yes        | Yes          | No                      | No            |
| `protected`            | Yes        | Yes          | Yes                     | No            |
| `public`               | Yes        | Yes          | Yes                     | Yes           |

All four modifiers apply to fields, methods, constructors, and nested classes; only `public` and `default` are valid on a **top-level class** (`private`/`protected` would make it inaccessible).

### private

```java
class Account {
    private double balance;          // only this class can access
    private Account() { }            // private constructor prevents external instantiation

    public double getBalance() {     // controlled access
        return balance;
    }
}

// Private members are accessible within the same class, even across instances
class Account {
    private double balance;

    boolean hasMoreThan(Account other) {
        return this.balance > other.balance;   // OK: can read another instance's private field
    }
}
```

### protected

```java
// Accessible in: same class + same package + subclass (even in a different package)
// In a different package: only via inheritance (this), NOT via a parent reference
package zoo;

class Animal {
    protected String name;
    protected void makeSound() { }
}
```

```java
package pets;            // different package
import zoo.Animal;

class Dog extends Animal {
    void test() {
        this.name = "Buddy";    // OK: via inheritance (child reference)
        makeSound();            // OK: inherited method

        Animal a = new Animal();
        a.name = "X";           // compile error: parent reference in a different package
    }
}
```

## Getters and Setters

Controlled read/write access to private fields. The JavaBeans naming convention, `getX()` / `setX()` with `isX()` for booleans, is what frameworks like Jackson, JPA, and Spring use to bind properties.

```java
class Employee {
    private String name;
    private int age;
    private boolean active;

    public String getName() { return name; }              // read-only: getter, no setter

    public void setAge(int age) {                         // setter validates before assigning
        if (age < 18 || age > 65) throw new IllegalArgumentException("Invalid age");
        this.age = age;
    }
    public int getAge() { return age; }                   // read-write (paired with setAge)

    public boolean isActive() { return active; }          // boolean getter uses 'is', not 'get'
}
```

Not every field needs both accessors:

- Read-only → getter only.
- Write-only → setter only (rare, e.g. passwords).
- Internal-only → neither.
- Computed → getter with logic, no backing field.

```java
// Computed getter, no backing field
public String getFullName() {
    return firstName + " " + lastName;
}
```

## final Keyword

The `final` keyword restricts modification once a variable is initialized.

- **Instance/local variables**: prevent reassigning the primitive value or the reference.
- **Method parameters**: prevent reassigning the parameter inside the body.
- **Blank final**: an instance field declared without an initializer that must be assigned exactly once in every constructor.

```java
import java.util.List;
import java.util.ArrayList;

class FinalDemo {
    final int x; // blank final

    FinalDemo(int val) {
        this.x = val; // assigned once in the constructor
    }

    void test(final int param) {
        final int local = 10;
        // local = 20; // compile error
        // param = 30; // compile error

        final List<String> list = new ArrayList<>();
        list.add("Java"); // OK: mutates internal state
        // list = new ArrayList<>(); // compile error: reference reassignment forbidden
    }
}
```

> **Note**: local final variables and parameters are compile-time constraints; they have no bytecode representation and zero runtime cost.

## Mutable vs Immutable

Whether an object's state can change after construction determines how safely it can be shared.

### Mutable

State can change after creation.

```java
class Employee {
    private String name;
    public void setName(String name) { this.name = name; }  // state changes
}

Employee e = new Employee();
e.setName("Alice");
e.setName("Bob");    // same reference, different state
```

### Immutable

State cannot change after creation.

```java
// String is immutable
String s = "hello";
s.toUpperCase();           // returns a new String, original unchanged
System.out.println(s);     // still "hello"

// Integer, LocalDate, etc. are also immutable
```

Immutable objects bring several benefits:

- Thread-safe with no synchronization needed.
- Predictable, with no side effects.
- Safe as `HashMap` keys, since the hash never changes.
- Cacheable and freely shareable.

## Creating an Immutable Class

Combine five guarantees so the state can never change after construction:

```java
// 1. Class is final, can't be extended (stops overriding methods to break immutability)
public final class Employee {

    // 2. Fields are private final, can't be reassigned
    private final String name;
    private final int salary;
    private final List<String> skills;

    // 3. Initialize via the constructor
    public Employee(String name, int salary, List<String> skills) {
        this.name = name;
        this.salary = salary;
        // 5. Defensive copy of mutable input, so later external edits don't leak in
        this.skills = new ArrayList<>(skills);
    }

    // 4. No setters, only getters
    public String getName() { return name; }
    public int getSalary() { return salary; }

    // 5. Defensive copy on the way out, so the caller can't mutate the internal list
    public List<String> getSkills() {
        return new ArrayList<>(skills);    // copy, not the original
        // NOT: return skills;             // would let the caller modify internal state
    }
}
```

> **Note**: only mutable fields (`List`, `Date`, arrays) need defensive copies; already-immutable fields (`String`, `int`, `Integer`) can be assigned directly.

```java
// Without defensive copies, immutability breaks
List<String> skills = new ArrayList<>(List.of("Java", "SQL"));
Employee emp = new Employee("Alice", 50000, skills);
skills.add("Python");              // mutates internal state if the constructor didn't copy
emp.getSkills().add("Python");     // mutates internal state if the getter didn't copy

// With defensive copies, truly immutable:
// constructor copies input  → external list changes don't affect the object
// getter copies output      → the caller can't modify the internal list
```

## Singleton

Only **one instance** of the class can exist. Common for configs, connection pools, loggers.

**Eager**: instance created at class load.

```java
public class Config {
    private static final Config INSTANCE = new Config();   // created at class loading
    private Config() { }                                   // private constructor
    public static Config getInstance() { return INSTANCE; }
}
```

**Lazy (Bill Pugh)**: created on first access via a static holder class.

```java
public class Config {
    private Config() { }

    private static class Holder {
        private static final Config INSTANCE = new Config();  // created only when accessed
    }

    public static Config getInstance() {
        return Holder.INSTANCE;    // holder loaded on first call
    }
}
```

**Double-checked locking**: lazy, using a `volatile` field and a double null-check around a `synchronized` block.

```java
public class Config {
    private static volatile Config instance;
    private Config() { }

    public static Config getInstance() {
        if (instance == null) {                    // first check (no lock)
            synchronized (Config.class) {
                if (instance == null) {            // second check (with lock)
                    instance = new Config();
                }
            }
        }
        return instance;
    }
}
```

**Enum** (recommended): the JVM guarantees a single instance.

```java
public enum Config {
    INSTANCE;

    private String dbUrl;

    public String getDbUrl() { return dbUrl; }
    public void setDbUrl(String url) { this.dbUrl = url; }
}

Config.INSTANCE.getDbUrl();
```

All four are thread-safe; they differ in laziness and resistance to reflection/serialization attacks:

| Approach       | Lazy       | Reflection/serialization-safe |
| -------------- | ---------- | ----------------------------- |
| Eager          | No         | No                            |
| Bill Pugh      | Yes        | No                            |
| Double-checked | Yes        | No                            |
| Enum           | No (eager) | Yes                           |

Reflection can call the private constructor via `setAccessible(true)` to create a second instance, and serialization and cloning can also produce extra instances. Enum is immune to all three; for the others, implement `readResolve()`, covered in [Serialization](<27. Serialization.md>).

## Q&A

1. **Does `private` actually prevent access at runtime?**
No. Access modifiers are enforced by the compiler and the JVM's normal checks, but reflection's `setAccessible(true)` bypasses them, the same loophole that breaks singletons. Treat encapsulation as a design contract, not a security boundary.

2. **What is a `protected` constructor for?**
It lets subclasses, even in other packages, instantiate the class through a `super()` call, while preventing a non-subclass in a different package from constructing it directly with `new`.

3. **Why is Singleton sometimes called an anti-pattern?**
Overusing it introduces hidden global state and tight coupling, which makes unit testing hard. Modern code usually prefers dependency injection (e.g. Spring-managed beans) to share instances, keeping dependencies explicit and easy to mock.

## Quick Revision Cheat Sheet

| Concept             | Remember                                                                       |
| ------------------- | ------------------------------------------------------------------------------ |
| Encapsulation       | Data hiding + controlled access (not just getters/setters)                     |
| Access modifiers    | `private` (class) < `default` (package) < `protected` (package + subclass) < `public` (everywhere) |
| private scope       | Accessible across instances of the **same class**, not just `this`             |
| Top-level class     | Only `public` or `default`                                                     |
| Immutable class     | final class + private final fields + no setters + defensive copies             |
| Defensive copy      | Copy mutable fields in the constructor AND the getter                          |
| `final` ≠ immutable | `final` = can't reassign the reference; the object's contents can still change  |
| Singleton           | One instance; enum is best; Bill Pugh for lazy                                 |
| Singleton breakers  | Reflection, serialization, cloning (enum is immune)                            |
