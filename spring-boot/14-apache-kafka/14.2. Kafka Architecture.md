## Cluster вҶ’ Broker вҶ’ Node

**Topic is a logical name; partition is the physical storage unit.** Brokers store partitions on disk, not topics вҖ” a topic is just a namespace its partitions belong to. Naming convention: `<topic>/P<n>` (e.g., `orders/P0`, `payments/P1`) вҖ” the topic prefix is part of the partition's identity.

```
Kafka Cluster (3 brokers, 2 topics)
в”ңв”Җв”Җ Node 1 (Machine/Server)
в”Ӯ     в””в”Җв”Җ Broker 1 (Kafka process running on it)
в”Ӯ           в”ңв”Җв”Җ orders/P0   вҶ’ Leader
в”Ӯ           в”ңв”Җв”Җ orders/P1   вҶ’ Replica
в”Ӯ           в””в”Җв”Җ payments/P0 вҶ’ Leader
в”Ӯ
в”ңв”Җв”Җ Node 2 (Machine/Server)
в”Ӯ     в””в”Җв”Җ Broker 2 (Kafka process running on it)
в”Ӯ           в”ңв”Җв”Җ orders/P0   вҶ’ Replica
в”Ӯ           в”ңв”Җв”Җ orders/P1   вҶ’ Leader
в”Ӯ           в””в”Җв”Җ payments/P0 вҶ’ Replica
в”Ӯ
в””в”Җв”Җ Node 3 (Machine/Server)
      в””в”Җв”Җ Broker 3 (Kafka process running on it)
            в”ңв”Җв”Җ orders/P0   вҶ’ Replica
            в”ңв”Җв”Җ orders/P1   вҶ’ Replica
            в””в”Җв”Җ payments/P0 вҶ’ Replica
```

- **Node**: the machine/server
- **Broker**: Kafka process running on a node (typically 1 node = 1 broker)
- **Cluster**: group of brokers coordinated by KRaft (or ZooKeeper in older versions)
- A single broker hosts partitions from **multiple topics** (Broker 1 above holds partitions from both `orders` and `payments`) вҖ” but a partition belongs to **exactly one topic**.

## Partitioning Logic

When a producer sends an event, it goes to a specific partition based on:

**With a partition key**: Producer assigns a key (e.g., `customerId: 123`) to the event. Kafka runs it through a hash function вҶ’ same key always goes to the same partition.

<p align="center"> <img src="partition-logic.png" width="500"> </p>

> Good partition keys: user ID, order ID, device ID вҖ” anything that groups related events together.

**Without a partition key**: Kafka uses **round-robin** assignment, spreading events evenly across partitions.

**Message ordering:** guaranteed **only within a partition**. To preserve order for related events (e.g., all events for the same order ID), use the order ID as the partition key вҖ” they all land on the same partition. Cross-partition ordering is never guaranteed.

## Partition Replication

Replication = copying a partition to other brokers for fault tolerance. Each partition has one **leader** and one or more **followers** (replicas).

<p align="center"> <img src="https://cs-prod-assets-bucket.s3.ap-south-1.amazonaws.com/image_7_2f6417027d.png" width="600"> </p>

- Producer always writes to the **leader** partition
- Replicas don't receive from the producer directly вҖ” they **pull from the leader** continuously
- If a leader goes down, a follower is **automatically elected** as the new leader
- Data durability depends on replication factor + acknowledgement config

> `replication.factor=3` means 3 copies of each partition across 3 different brokers. Requires at least 3 brokers.

### Does every broker hold all partitions?

No вҖ” each broker only holds the partitions assigned to it, either as leader or replica. Only when `replication.factor = total brokers` does every broker hold all partitions.

```
replication.factor=2, 3 partitions, 3 brokers:
Broker 1 вҶ’ P1 (Leader), P3 (Replica)     вҶҗ no P2 at all
Broker 2 вҶ’ P2 (Leader), P1 (Replica)     вҶҗ no P3 at all
Broker 3 вҶ’ P3 (Leader), P2 (Replica)     вҶҗ no P1 at all

replication.factor=3, 3 partitions, 3 brokers:
Broker 1 вҶ’ P1 (Leader), P2 (Replica), P3 (Replica)  вҶҗ has all 3
Broker 2 вҶ’ P2 (Leader), P1 (Replica), P3 (Replica)  вҶҗ has all 3
Broker 3 вҶ’ P3 (Leader), P1 (Replica), P2 (Replica)  вҶҗ has all 3
```

| Replication Factor | Fault Tolerance |
|---|---|
| 1 | No fault tolerance |
| 2 | Can tolerate 1 failure |
| 3 | Can tolerate 1 failure safely (2 only when all replicas are in ISR) |

### Same partition number on multiple brokers вҖ” is that duplicate data?

No вҖ” that's **replication**:

- Same partition number across brokers = replication (same data, different roles вҖ” leader vs replica)
- Different partition numbers across brokers = distribution (different data)

```
Broker 1: P1 Leader  в”җ same partition,
Broker 2: P1 Replica в”ҳ different role вҶ’ replication

Broker 1: P1 Leader  в”җ different partitions,
Broker 2: P3 Leader  в”ҳ different data  вҶ’ distribution
```

### Leadership is per-partition, not per-broker

Every broker is a leader for some partitions and a replica for others вҖ” Kafka intentionally distributes partition leaders across brokers for load balancing, fault tolerance, and parallel throughput.

```
Cluster: 3 brokers, 2 topics
  вҖў orders   вҖ” 3 partitions (P0, P1, P2)
  вҖў payments вҖ” 2 partitions (P0, P1)
  вҖў replication.factor = 3 (every partition has 3 copies)

                 Broker 1          Broker 2          Broker 3
               в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ      в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ      в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
 orders/P0     в”Ӯ вҳ… LEADER в”Ӯ      в”Ӯ follower в”Ӯ      в”Ӯ follower в”Ӯ
 orders/P1     в”Ӯ follower в”Ӯ      в”Ӯ вҳ… LEADER в”Ӯ      в”Ӯ follower в”Ӯ
 orders/P2     в”Ӯ follower в”Ӯ      в”Ӯ follower в”Ӯ      в”Ӯ вҳ… LEADER в”Ӯ
 payments/P0   в”Ӯ follower в”Ӯ      в”Ӯ вҳ… LEADER в”Ӯ      в”Ӯ follower в”Ӯ
 payments/P1   в”Ӯ вҳ… LEADER в”Ӯ      в”Ӯ follower в”Ӯ      в”Ӯ follower в”Ӯ
               в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ      в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ      в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                  2 leads           2 leads           1 leads
```

- вҳ… LEADER handles all reads/writes for that partition; followers replicate silently
- Leadership is **balanced** across brokers (2-2-1) so writes spread, no hotspot
- If a broker dies, one of the followers for each partition it led gets promoted automatically

> If all leaders were on one broker вҶ’ all writes go there вҶ’ **hotspot**. Replicas on other brokers would sit idle, defeating the purpose of multiple brokers.

This is also why Kafka scales well вҖ” multiple producers can write to the same topic simultaneously, each hitting a different broker's leader partition.

## In-Sync Replicas (ISR)

Not all replicas are always fully up-to-date:

- **ISR** = replicas that are fully synced with the leader
- Only ISR replicas acknowledge writes when `acks=all`
- Only ISR replicas are eligible to become the new leader if the current leader goes down
- If a replica falls behind, it's **removed from the ISR** until it catches up

### Unclean Leader Election

If no in-sync replica is available when a leader fails, Kafka can optionally elect an **out-of-sync replica** as the new leader:

- Avoids downtime
- May cause **data loss** (out-of-sync replica may be missing recent messages)

Controlled by: `unclean.leader.election.enable` (default: `false`)

## Reading Records & Offsets

Unlike RabbitMQ (push-based), **Kafka uses a pull-based model**: consumers fetch data from brokers at their own pace. They decide when to read, how fast, and from which offset.

- Consumer connects to a partition leader
- Reads messages in the order they were written
- Remembers the **last consumed offset** for each partition
- On next pull, continues from where it left off

**Can consumers read messages after a few days?** Yes вҖ” Kafka does not delete messages after consumption. Messages stay until the retention policy removes them. A consumer can even go back to an earlier offset and reprocess old messages (within the retention window).

### Offsets are local to a partition

The same offset number in different partitions refers to completely different messages.

```
Partition 1 вҶ’ offset 5 = "Order #100 created"
Partition 2 вҶ’ offset 5 = "Payment #200 processed"   вҶҗ different message, same offset number
```

To uniquely identify a message: `Topic + Partition + Offset`

### Consumer offset tracking

Committed offsets are stored in Kafka itself in an internal topic called `__consumer_offsets`. The app controls when to commit; Kafka persists it.

```
__consumer_offsets topic
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ   Group ID      в”Ӯ  Partition  в”Ӯ Offset в”Ӯ
в”ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө
в”Ӯ consumer-group1 в”Ӯ      1      в”Ӯ   20   в”Ӯ
в”Ӯ consumer-group1 в”Ӯ      2      в”Ӯ   18   в”Ӯ
в”Ӯ consumer-group2 в”Ӯ      1      в”Ӯ   15   в”Ӯ  вҶҗ different group, independent offset
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

What's stored: `Group ID + Partition + Offset` вҖ” multiple consumer groups read the same topic independently, each at their own pace.

### Consumer lag

Consumer lag = difference between the latest offset in a partition and the consumer's current offset. High lag means consumers are falling behind producers вҖ” a critical metric to monitor in production.

**Consumer crash recovery**: on restart, consumers resume from the last committed offset. Note: offsets are not automatically updated after every message вҖ” depends on commit strategy.

### Offset commit strategies

- **Auto commit**: Kafka periodically commits offsets at a configured interval
- **Manual commit**: application explicitly commits offsets after processing

> Manual commit gives better control and prevents data loss on crashes, but may lead to duplicate processing if a message is processed but the commit fails before the next restart.

## Consumer Groups

Without consumer groups, multiple instances of the same service would all read the same event, causing duplicate processing (e.g., 4 instances of Inventory Service all reducing stock for the same order).

**Consumer Groups** solve this. A group ID ties multiple consumer instances together вҖ” Kafka ensures each partition is assigned to **exactly one consumer** within the group at a time.

<p align="center"> <img src="https://cs-prod-assets-bucket.s3.ap-south-1.amazonaws.com/image_6_1d6f851de7.png" width="500"> </p>

```
Consumer Group A (4 consumers, 4 partitions):
в”ңв”Җв”Җ Consumer 1 вҶ’ Partition 1
в”ңв”Җв”Җ Consumer 2 вҶ’ Partition 2
в”ңв”Җв”Җ Consumer 3 вҶ’ Partition 3
в””в”Җв”Җ Consumer 4 вҶ’ Partition 4
```

Two consumers from the same group can never read the same partition simultaneously вҖ” this prevents duplicate processing.

| Consumers | Partitions | Result |
|---|---|---|
| 4 | 4 | Perfect вҖ” 1 each |
| 2 | 4 | Each consumer reads 2 partitions |
| 6 | 4 | 4 active, 2 sit idle |

- **Within a group**: each partition goes to only one consumer
- **Across groups**: both Group A and Group B receive all events independently (e.g., Inventory Service group and Notification Service group both get the `order-placed` event)

**Max parallelism = number of partitions**, not number of consumers. To increase parallelism, increase partitions.

> Key rule: **active consumers вүӨ partitions**.

### Consumer routing вҖ” bootstrap broker

Consumers don't need to know upfront which broker holds the leader partition:

1. Consumer connects to **any** broker initially вҖ” called a **bootstrap broker**
2. That broker returns **cluster metadata**: which broker holds which partition leader
3. Consumer connects directly to the correct broker
4. If a leader changes (broker goes down, rebalancing), Kafka updates the metadata and consumer retries automatically

By default, consumers read from the **leader partition** (followers are not used for reads in standard setups; follower-fetch is possible since Kafka 2.4 but not default).

> Cross-partition ordering is never guaranteed вҖ” `poll()` retrieves batches in round-robin from assigned partitions.

---

## Why Multiple Partitions?

1. **Parallelism**: multiple consumers in a group read simultaneously, one per partition. More partitions = higher throughput.
2. **Storage distribution**: partitions are spread across brokers, so no single broker holds all the data.
3. **Fault tolerance**: each partition is replicated across brokers. If one broker goes down, another already has the replica ready to become leader.

```
Single partition:    Consumer A reads вҶ’ others wait       (bottleneck)
Multiple partitions: Consumer A вҶ’ P1, Consumer B вҶ’ P2     (parallel)
                     Consumer C вҶ’ P3, Consumer D вҶ’ P4
```

Partition IDs are **independent per topic**: `orders P3` and `notifications P3` are completely different partitions that just share the same number.

```
Topic: "orders"        вҶ’ Broker 1: P1, P2  |  Broker 2: P3, P4
Topic: "payments"      вҶ’ Broker 1: P1, P3  |  Broker 2: P2, P4
Topic: "notifications" вҶ’ Broker 1: P2, P4  |  Broker 2: P1, P3
```

## Rebalancing

Rebalancing = re-assignment of partition ownership among consumers in a group so every consumer gets one or more partitions.

Triggered when:

- A new consumer joins the group
- An existing consumer goes down
- New partitions are added
- A consumer is considered dead (hasn't polled within the configured timeout)

> During rebalancing, message consumption **pauses temporarily**, which can impact throughput.

> One consumer in the group is elected as the **Group Leader** (responsible for partition assignment), but this is managed automatically by Kafka.

### Stop-the-world vs cooperative rebalancing

The classic `RangeAssignor`/`RoundRobinAssignor` strategies use **eager** rebalancing вҖ” every consumer revokes all its partitions, then a new assignment is computed and handed out. The whole group is paused during this window.

Modern Kafka (2.4+) supports the **cooperative-sticky** assignor, which only revokes the partitions that actually need to move and lets unaffected consumers keep processing:

```yaml
spring:
  kafka:
    consumer:
      properties:
        partition.assignment.strategy: org.apache.kafka.clients.consumer.CooperativeStickyAssignor
```

> Strongly preferred for production вҖ” dramatically reduces rebalance pause time when consumers scale in/out or roll-restart.

## Kafka Brokers Management (KRaft)

Someone needs to track which broker is alive, which partition has which leader, etc.

- **Previously:** ZooKeeper handled this as a separate system
- **Now:** Kafka uses **KRaft (Kafka Raft consensus protocol)**: brokers coordinate directly with each other to elect a controller and manage cluster state. No separate ZooKeeper needed.

## Log Retention

Controls how long messages are kept in a topic. Configured per topic:

| Config | Description |
|---|---|
| `retention.ms` | Keep messages for this many milliseconds (e.g., 7 days) |
| `retention.bytes` | Keep messages up to this size per partition |
| `retention.ms=-1` | Keep messages indefinitely (watch storage costs!) |

> Whichever limit is hit first triggers deletion. Old messages are deleted in the background.

## Acks and Retries

**Acks** = how many acknowledgements the producer needs from brokers before considering a message successfully sent.

| acks value | Behaviour | Speed | Reliability |
|---|---|---|---|
| `0` | Producer sends and forgets вҖ” no ack waited | Fastest | No guarantee |
| `1` | Leader broker acknowledges receipt | Medium | Data safe on leader only |
| `all` / `-1` | All in-sync replicas (ISRs) acknowledge | Slowest | Highest guarantee |

> `acks=all` is the safest вҖ” message is confirmed only after all in-sync replicas (ISR) acknowledge it.

> **Pair `acks=all` with `min.insync.replicas`**: without it, `acks=all` is misleading. If the ISR shrinks to 1, `acks=all` only waits for that one replica, defeating the durability promise. Set `min.insync.replicas=2` (with `replication.factor=3`) on the broker/topic so writes fail loudly when fewer than 2 replicas are in sync, instead of silently weakening durability.

**Retries**: if a send request fails (network issue, broker down), producer automatically retries up to the configured `retries` count.

- Useful for transient failures
- Combined with `acks=all` вҖ” provides strong durability guarantees as long as in-sync replicas are available
- Configurable: `retries`, `retry.backoff.ms`

> Because retries can lead to duplicate messages, consumers should be **idempotent**: designed to safely handle the same message being processed more than once.

> **Idempotent producer** (`enable.idempotence=true`) вҖ” eliminates duplicates from the producer side: Kafka assigns a producer ID + sequence number per partition and dedupes retries on the broker. Strongly recommended in production. For end-to-end exactly-once across multiple topics, use **transactional producers** (`transactional.id`) вҖ” but that's an advanced setup, see Spring Kafka's `KafkaTransactionManager`.


## Q&A

1. **How does Kafka guarantee message ordering?** Only **within a partition**. To preserve ordering for related events (e.g., per order ID), use the order ID as the **partition key**: Kafka hashes it so all events for that key land on the same partition. Cross-partition ordering is never guaranteed.

2. **What is a consumer group and why does it matter?** A group ID ties multiple consumer instances together вҖ” within a group, each partition is owned by exactly one consumer at a time, preventing duplicate processing. Different groups consume the same topic independently. Max parallelism = number of partitions.

3. **What's the difference between leader and replica? Is leadership at the broker or partition level?** Leadership is **per-partition**. Each partition has one leader (handles reads/writes) and N followers (replicas that pull from leader). Every broker is leader for some partitions and replica for others вҖ” Kafka distributes leadership for load balancing.

4. **What happens during rebalancing and why is it expensive?** When a consumer joins/leaves the group (or partitions change), Kafka redistributes partition ownership. Consumption **pauses** during rebalancing вҖ” frequent rebalances hurt throughput.

5. **What does `acks=all` actually guarantee?** Producer waits for **all in-sync replicas (ISR)** to acknowledge the write before considering it successful. Strongest durability, slowest. With `acks=1` only the leader acks; with `acks=0` the producer fires and forgets.

6. **What is unclean leader election and why is it risky?** If no ISR is available when a leader fails, Kafka can promote an out-of-sync replica to leader (default off). Avoids downtime but **may lose recently written messages** that the new leader never received.

7. **Auto-commit vs manual commit вҖ” which is safer?** **Manual commit** is safer: the app commits only after successfully processing a message. Auto-commit can advance the offset before processing completes, risking message loss on crash. Either way, consumers should be **idempotent** (at-least-once delivery).

8. **Same partition number on two brokers вҖ” is that duplicate data?** No, it's **replication**. Same partition number across brokers = same data, different roles (leader vs replica). Different partition numbers across brokers = different data (distribution).

9. **How does a consumer find the right broker for a partition?** It connects to any **bootstrap broker**, fetches **cluster metadata** (which broker holds which leader), then connects directly to the correct broker. On leader changes, metadata updates and consumer retries automatically.

10. **What happens if you create a topic with `replication.factor=3` but only have 2 brokers?** Topic creation fails. Each replica of a partition must live on a **different broker**, so the replication factor cannot exceed the number of available brokers. To run RF=3 you need at least 3 brokers.

11. **If you delete a topic, what happens to its partitions?** All partitions of that topic are deleted along with it. Partitions have no independent existence вҖ” they are owned by a topic. This is the strongest evidence that partitions belong to topics, not the other way around.
