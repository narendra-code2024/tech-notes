A topic is a logical name; the partition is the physical storage unit. Brokers store partitions on disk, not topics, and a partition belongs to exactly one topic (naming convention `<topic>/P<n>`, e.g. `orders/P0`). A single broker hosts partitions from many topics, and partitions of one topic spread across brokers, which is how Kafka scales.

```
Kafka Cluster (3 brokers)
  Broker 1: orders/P0 (Leader),  orders/P1 (Replica), payments/P0 (Leader)
  Broker 2: orders/P0 (Replica), orders/P1 (Leader),  payments/P0 (Replica)
  Broker 3: orders/P0 (Replica), orders/P1 (Replica), payments/P0 (Replica)
```

- **Node**: the machine. **Broker**: the Kafka process on it (typically 1:1).
- **Cluster**: brokers coordinated by KRaft.

## Partitioning Logic

A producer's event lands on a partition by one of two rules:

- **With a key** (e.g. `customerId: 123`): Kafka hashes the key, so the same key always maps to the same partition. Good keys group related events: user ID, order ID, device ID.
- **Without a key**: the default partitioner batches records to a partition for throughput (you cannot predict which).

Because ordering is guaranteed only within a partition, keying by order ID keeps all events for one order on the same partition and therefore in order. Cross-partition ordering is never guaranteed.

## Replication, Leaders, and ISR

Replication copies each partition to other brokers for fault tolerance. Each partition has one **leader** and one or more **follower** replicas:

- The producer always writes to the leader; followers pull from the leader continuously.
- If the leader dies, a follower is automatically elected leader.
- `replication.factor=3` means 3 copies across 3 different brokers, so it requires at least 3 brokers (creating a topic with RF greater than the broker count fails).

Leadership is per-partition, not per-broker: every broker leads some partitions and follows others, and Kafka balances leaders across brokers so writes spread instead of hitting one hotspot.

```
3 brokers, RF=3, orders has P0-P2, payments has P0-P1:
              Broker 1    Broker 2    Broker 3
 orders/P0    LEADER      follower    follower
 orders/P1    follower    LEADER      follower
 orders/P2    follower    follower    LEADER
 payments/P0  follower    LEADER      follower
 payments/P1  LEADER      follower    follower
              (2 leads)   (2 leads)   (1 lead)
```

The same partition number on two brokers is replication (same data, different roles); different partition numbers on two brokers is distribution (different data).

| Replication factor | Fault tolerance                                              |
| ------------------ | ----------------------------------------------------------- |
| 1                  | None                                                        |
| 2                  | Tolerates 1 broker failure                                  |
| 3                  | Tolerates 1 failure safely (2 only when all are in ISR)     |

**In-sync replicas (ISR)** are the replicas fully caught up with the leader. Only ISR members acknowledge writes under `acks=all` and only they are eligible for leader election; a replica that falls behind is dropped from the ISR until it catches up.

> **Note**: if no ISR is available when a leader fails, **unclean leader election** (`unclean.leader.election.enable`, default `false`) can promote an out-of-sync replica, avoiding downtime at the cost of losing recent messages. Leave it off unless availability strictly outranks durability.

## Reading: Offsets, Commits, and Lag

Kafka is pull-based (unlike RabbitMQ's push): a consumer fetches from the partition leader at its own pace, remembers the last consumed offset, and continues from there on the next poll. Messages are not deleted on consumption, so a consumer can re-read from an earlier offset within the retention window.

Offsets are local to a partition (offset 5 in P0 and P1 are different messages), so a message is identified by `Topic + Partition + Offset`. Committed offsets are stored in the internal `__consumer_offsets` topic, keyed by `Group ID + Partition`, so multiple groups read the same topic independently:

```
__consumer_offsets
  consumer-group1 | P1 | 20
  consumer-group1 | P2 | 18
  consumer-group2 | P1 | 15   (different group, independent offset)
```

**Consumer lag** is the gap between the partition's latest offset and the consumer's committed offset; rising lag means consumers are falling behind producers, a key production metric. On restart, a consumer resumes from its last committed offset, so when it commits matters:

- **Auto commit**: Kafka commits periodically on a timer; the offset can advance before processing finishes, risking message loss on a crash.
- **Manual commit**: the app commits after processing; safer, but a crash after processing and before the commit causes redelivery.

Either way delivery is at-least-once, so consumers must be **idempotent**.

## Consumer Groups

A consumer group ties instances together under one group ID; within a group, each partition is owned by exactly one consumer at a time, which prevents duplicate processing when a service runs multiple instances. Different groups each receive all events independently.

| Consumers | Partitions | Result                          |
| --------- | ---------- | ------------------------------- |
| 4         | 4          | One partition each              |
| 2         | 4          | Each consumer reads 2 partitions |
| 6         | 4          | 4 active, 2 idle                |

Maximum parallelism equals the partition count, not the consumer count, so `active consumers <= partitions`. To raise parallelism, add partitions. Consumers find the right broker by connecting to any **bootstrap broker**, fetching cluster metadata (which broker leads which partition), then connecting directly; on a leader change the metadata updates and the consumer retries.

---

## Rebalancing

Rebalancing reassigns partition ownership across a group's consumers. It triggers when a consumer joins, leaves, or is declared dead (no poll within the timeout), or when partitions are added. Consumption pauses during a rebalance, so frequent rebalances hurt throughput.

- **Eager** (classic `RangeAssignor` / `RoundRobinAssignor`): every consumer revokes all partitions, then a new assignment is computed; the whole group pauses.
- **Cooperative-sticky** (`CooperativeStickyAssignor`): revokes only the partitions that must move, so unaffected consumers keep processing. Strongly preferred for production.

```yaml
spring:
  kafka:
    consumer:
      properties:
        partition.assignment.strategy: org.apache.kafka.clients.consumer.CooperativeStickyAssignor
```

> **Note**: Kafka 4.0 makes the new consumer group protocol (KIP-848) generally available, which moves rebalance computation to the broker-side coordinator and largely removes client-driven stop-the-world pauses. The cooperative-sticky assignor remains the right default on the classic protocol.

## Durability: Acks, Retries, Idempotence

`acks` controls how many acknowledgements the producer waits for:

| `acks`     | Behaviour                              | Speed   | Reliability             |
| ---------- | -------------------------------------- | ------- | ----------------------- |
| `0`        | Fire and forget, no ack                | Fastest | No guarantee            |
| `1`        | Leader acknowledges                    | Medium  | Safe on leader only     |
| `all`/`-1` | All in-sync replicas acknowledge       | Slowest | Highest                 |

On a failed send the producer retries (`retries`, `retry.backoff.ms`). Since retries can duplicate messages, the **idempotent producer** assigns a producer ID + per-partition sequence number so the broker dedupes retries.

> **Note**: since Kafka 3.0 the producer defaults to `enable.idempotence=true` with `acks=all` and effectively unlimited retries, so these strong guarantees are on unless you weaken them. Do not set `acks=0/1` or disable idempotence without a specific reason. For durability also set broker/topic `min.insync.replicas=2` with `replication.factor=3`, so writes fail loudly when fewer than 2 replicas are in sync instead of silently weakening `acks=all`. End-to-end exactly-once across topics needs transactional producers (`transactional.id`, Spring's `KafkaTransactionManager`).

## Log Retention and KRaft

Retention controls how long messages are kept per topic; whichever limit is hit first triggers background deletion:

| Config            | Meaning                                          |
| ----------------- | ------------------------------------------------ |
| `retention.ms`    | Keep messages for this many milliseconds         |
| `retention.bytes` | Keep up to this size per partition               |
| `retention.ms=-1` | Keep indefinitely (watch storage cost)           |

Cluster coordination (which broker is alive, who leads each partition, controller election) runs on **KRaft**, Kafka's built-in Raft consensus among brokers. The external ZooKeeper dependency was deprecated through the 3.x line and fully removed in Kafka 4.0, so KRaft is the only mode on current versions.

## Q&A

1. **How does Kafka guarantee ordering?**
Within a partition only. Key related events (e.g. by order ID) so they hash to the same partition; cross-partition ordering is never guaranteed.

2. **What is a consumer group and why does it matter?**
A group ID under which each partition is owned by exactly one consumer at a time, preventing duplicate processing across instances. Different groups consume independently; max parallelism equals the partition count.

3. **Leader vs replica, and is leadership per broker or per partition?**
Per partition. Each partition has one leader (handles reads/writes) and followers that replicate; every broker leads some partitions and follows others so load spreads.

4. **What does `acks=all` guarantee, and what is the default now?**
The producer waits for all in-sync replicas before counting a write successful. Since Kafka 3.0, `acks=all` with idempotence is the default.

5. **What is unclean leader election?**
Promoting an out-of-sync replica when no ISR is available (default off). It avoids downtime but can lose recently written messages.

6. **Auto-commit vs manual commit?**
Manual is safer: commit after processing. Auto-commit can advance the offset before processing finishes, risking loss on a crash. Delivery is at-least-once either way, so consumers must be idempotent.

7. **How does a consumer find the right broker for a partition?**
It connects to any bootstrap broker, fetches metadata about partition leaders, then connects directly; on leader change the metadata updates and it retries.

8. **`replication.factor=3` with only 2 brokers?**
Topic creation fails. Each replica must live on a different broker, so RF cannot exceed the broker count.

9. **Delete a topic, what happens to its partitions?**
They are deleted with it. Partitions have no independent existence; they belong to a topic.
