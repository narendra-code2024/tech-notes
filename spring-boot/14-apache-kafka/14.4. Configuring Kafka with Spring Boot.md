Two microservices communicating via Kafka: `user-service` publishes (producer), `notification-service` consumes (consumer). This is a Kafka-focused demo, so the usual gateway/config-server/discovery pieces are omitted, though a real system would still need them. Both services depend on `spring-kafka` (plus web, JPA, the Postgres driver, and Lombok).

---

## Producer: user-service

Point the producer at the broker and define the topic name once:

```yaml
spring:
  application:
    name: user-service
  kafka:
    bootstrap-servers: localhost:9092
server:
  port: 9050
kafka:
  topic:
    user-random: user-random-topic
```

A `NewTopic` bean auto-creates the topic on startup (skipped if it exists); the producer sends via `KafkaTemplate`:

```java
@Configuration
public class KafkaTopicConfig {
    @Bean
    public NewTopic userRandomTopic(@Value("${kafka.topic.user-random}") String name) {
        return new NewTopic(name, 3, (short) 1);   // name, partitions, replication factor
    }
}

@RestController
@RequestMapping("/users")
@RequiredArgsConstructor
public class UserController {

    private final KafkaTemplate<String, String> kafkaTemplate;

    @Value("${kafka.topic.user-random}")
    private String userRandomTopic;

    @PostMapping("/{message}")
    public ResponseEntity<String> sendMessage(@PathVariable String message) {
        kafkaTemplate.send(userRandomTopic, message);    // async; returns CompletableFuture
        return ResponseEntity.ok("Message queued");
    }
}
```

> **Note**: `KafkaTemplate.send()` is asynchronous; it returns a `CompletableFuture` and does not wait for broker acknowledgement by default, so attach callbacks to handle success/failure. `send(topic, value)` sends without a key (the default partitioner batches to a partition); `send(topic, key, value)` hashes the key so the same key always lands on the same partition, giving per-key ordering.

> **Note**: in production, topics are pre-created by infrastructure (CLI, Terraform, managed services like AWS MSK or Confluent Cloud) with intended partitioning, replication, and retention, and apps often lack permission to create them. Remove the `NewTopic` bean there and just connect to existing topics.

## Consumer: notification-service

Every consumer must belong to a group; `${spring.application.name}` is the conventional group ID:

```yaml
spring:
  application:
    name: notification-service
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: ${spring.application.name}
server:
  port: 9060
kafka:
  topic:
    user-random: user-random-topic
```

`@KafkaListener` marks a method as a listener; Spring runs the container and invokes it on each message:

```java
@Service
@Slf4j
public class UserKafkaConsumer {

    @KafkaListener(topics = "${kafka.topic.user-random}")
    public void handleUserRandomTopic(String message) {
        log.info("Message received: {}", message);
    }
}
```

> **Note**: delivery is at-least-once by default, so if a consumer processes a message but crashes before committing, it is redelivered on restart. Consumers must be idempotent. In production, failures are routed through retries and a Dead Letter Topic (see [14.5. Advanced Kafka Configuration with Spring Boot](<14.5. Advanced Kafka Configuration with Spring Boot.md>)).

---

## Scaling Consumers

Multiple `@KafkaListener` methods in the same group, even within one app instance, act as independent consumers; method names are just identifiers and have no bearing on partition assignment. Within one instance, `concurrency` spawns parallel listener threads:

```yaml
spring:
  kafka:
    listener:
      concurrency: 3   # concurrent listener threads per @KafkaListener
```

Effective parallelism is still capped by partition count (`active consumers <= partitions`, see [14.2. Kafka Architecture](<14.2. Kafka Architecture.md>)). Production combines both levers: multiple instances, each with multiple listener threads, all in one group.

## Q&A

1. **Is `KafkaTemplate.send()` synchronous or asynchronous?**
Asynchronous; it returns a `CompletableFuture` and does not wait for broker acknowledgement by default. Attach callbacks for success/failure.

2. **Why is `group-id` required on the consumer?**
Kafka tracks offsets per `(group-id, partition)` in `__consumer_offsets`, and partition assignment is coordinated within a group. Without a group ID, neither is possible.

3. **In production, who creates topics?**
Infrastructure/DevOps via CLI, Terraform, or managed services; apps usually lack create permission. Drop the `NewTopic` bean and connect to existing topics.

4. **What controls listener parallelism within one instance?**
`spring.kafka.listener.concurrency`, the number of listener threads per `@KafkaListener`, still capped by partition count.

5. **Why must consumers be idempotent?**
Delivery is at-least-once; a crash after processing and before commit causes redelivery, so reprocessing must be safe.
