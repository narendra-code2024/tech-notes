## Project Setup

Two microservices communicating via Kafka. This is a simplified demo project — no API Gateway, Config Server, or Eureka is set up here since the focus is purely on Kafka. In a real production system, all of these would still be needed alongside Kafka as they serve different purposes.

```
user-service          →  publishes messages to Kafka  (Producer)
notification-service  →  consumes messages from Kafka  (Consumer)
```

## User Service (Producer)

### `application.yml`

```yaml
spring:
  application:
    name: user-service
  kafka:
    bootstrap-servers: localhost:9092
  datasource:
    url: jdbc:postgresql://localhost:5432/userdb
    username: postgres
    password: postgres
  jpa:
    hibernate:
      ddl-auto: update

server:
  port: 9050

kafka:
  topic:
    user-random: user-random-topic
```

### Topic Config — `KafkaTopicConfig.java`

Creates the topic automatically on startup. If the topic already exists, Kafka skips creation.

```java
@Configuration
public class KafkaTopicConfig {

    @Value("${kafka.topic.user-random}")
    private String userRandomTopic;

    @Bean
    public NewTopic userRandomTopic() {
        return new NewTopic(userRandomTopic, 3, (short) 1);
        //                  topic name    partitions  replication factor
    }
}
```

> **Production note:** Topics are typically pre-created by DevOps/infrastructure teams (via Kafka CLI, Terraform, Ansible) — not auto-created by the application. This gives full control over partition count, replication factor, retention policy, and prevents accidental topic creation from typos. Applications may not even have permission to create topics in production.
> 
> In production, **remove the `NewTopic` bean**: the application just connects to an already-running Kafka cluster and already-existing topics using `KafkaTemplate` and `@KafkaListener`. Kafka itself is typically managed externally via:
> 
> - **Docker / Docker Compose**: self-hosted or local dev
> - **Cloud managed services**: AWS MSK, Confluent Cloud, Azure Event Hubs — topics configured via their UI, CLI, or IaC tools like Terraform

### Controller — `UserController.java`

```java
@RestController
@RequestMapping("/users")
@RequiredArgsConstructor
@Slf4j
public class UserController {

    private final KafkaTemplate<String, String> kafkaTemplate;

    @Value("${kafka.topic.user-random}")
    private String userRandomTopic;

    @PostMapping("/{message}")
    public ResponseEntity<String> sendMessage(@PathVariable String message) {
        kafkaTemplate.send(userRandomTopic, message);
        return ResponseEntity.ok("Message queued");
    }
}
```

> **`KafkaTemplate<K, V>`**: Spring's main class for sending messages to Kafka.  
> `send(topic, value)` — sends a message without a key (sticky partition assignment).  
> `send(topic, key, value)` — sends with a key (same key always goes to same partition).

> **`KafkaTemplate.send()`** is **asynchronous**: it returns a `CompletableFuture` and does not wait for Kafka to acknowledge by default. You can attach callbacks to handle success or failure.

### Sending with a Key (Partition Control)

```java
// Key = i % 2 → values 0 or 1 → messages split across 2 partitions
for (int i = 0; i < 1000; i++) {
    kafkaTemplate.send(userRandomTopic, String.valueOf(i % 2), message + i);
}
```

> Without a key → Kafka uses **sticky partitioning by default** (since 2.4+): it sends batches of messages to one partition for better throughput, then switches. You cannot predict or control which partition a message lands on without a key.  
> With a key → Kafka **hashes the key** → same key always goes to same partition → ordering guaranteed within that partition.

## Notification Service (Consumer)

### `application.yml`

```yaml
spring:
  application:
    name: notification-service
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: ${spring.application.name}   # consumer group ID
  datasource:
    url: jdbc:postgresql://localhost:5432/notificationdb
    username: postgres
    password: postgres
  jpa:
    hibernate:
      ddl-auto: update

server:
  port: 9060

kafka:
  topic:
    user-random: user-random-topic
```

> **`group-id`** is required — every consumer must belong to a consumer group.  
> Using `${spring.application.name}` as the group ID is a common convention.

### Consumer — `UserKafkaConsumer.java`

```java
@Service
@Slf4j
public class UserKafkaConsumer {

    @KafkaListener(topics = "${kafka.topic.user-random}")
    public void handleUserRandomTopic(String message) {
        log.info("Message received: {}", message);
    }
}
```

> **`@KafkaListener(topics = "...")`**: marks a method as a Kafka message listener. Spring manages the listener container internally and invokes this method when messages arrive.  
> Multiple topics can be specified: `@KafkaListener(topics = {"topic1", "topic2"})`.

> **At-least-once delivery**: Kafka guarantees at-least-once delivery by default. If a consumer processes a message but crashes before committing the offset, the same message will be read again on restart — leading to duplicates. Consumers must be **idempotent** to safely handle this.

> **Error handling**: in production, failures are handled using retry mechanisms and **Dead Letter Topics (DLT)**, where failed messages are redirected for later analysis instead of being lost.

### Multiple Consumers

```java
@Service
@Slf4j
public class UserKafkaConsumer {

    @KafkaListener(topics = "${kafka.topic.user-random}")
    public void handleUserRandomTopic(String message) {
        log.info("handleUserRandomTopic - Message received: {}", message);
    }

    @KafkaListener(topics = "${kafka.topic.user-random}")
    public void handleUserRandomTopicOne(String message) {
        log.info("handleUserRandomTopicOne - Message received: {}", message);
    }

    @KafkaListener(topics = "${kafka.topic.user-random}")
    public void handleUserRandomTopicTwo(String message) {
        log.info("handleUserRandomTopicTwo - Message received: {}", message);
    }
}
```

> Method names are just identifiers — they have **no relation to which partition** the consumer reads from. Kafka automatically assigns partitions to consumers within the group — one partition per consumer at a time. See [14.2 Kafka Architecture](<14.2. Kafka Architecture.md>) for full consumer group details.

> Multiple consumers in the same group don't have to be on separate servers — multiple `@KafkaListener` methods within the **same application instance** each act as independent consumers in the group. Actual parallelism depends on both partition count and the `concurrency` configuration:

```yaml
spring:
  kafka:
    listener:
      concurrency: 3   # number of concurrent listener threads per @KafkaListener
```

In production, both are typically combined — multiple instances, each with multiple listeners — for maximum parallelism.

```
Same App Instance:                    Multiple Instances:
─────────────────                     ──────────────────
notification-service                  notification-service    notification-service
  ├── Consumer 1 (@KafkaListener)       ├── Consumer 1          ├── Consumer 3
  ├── Consumer 2 (@KafkaListener)       └── Consumer 2          └── Consumer 4
  └── Consumer 3 (@KafkaListener)
  all in same group                     same group across instances
```

## How It Works End to End

```
POST /users/hello
        │
        ▼
UserController
        │  kafkaTemplate.send("user-random-topic", "hello")
        ▼
Kafka Broker (localhost:9092)
        │  topic: user-random-topic
        │  partitions: 0, 1, 2
        ▼
UserKafkaConsumer (notification-service)
        │  @KafkaListener picks up the message
        ▼
log: "Message received: hello"
```

## Q&A

1. **Is `KafkaTemplate.send()` synchronous or asynchronous?** **Asynchronous**: it returns a `CompletableFuture` and does not wait for broker acknowledgement by default. Attach callbacks for success/failure handling.
2. **Why is `group-id` required on the consumer?** Kafka tracks offsets per `(group-id, partition)` in the internal `__consumer_offsets` topic. Without a group ID, Kafka cannot persist where the consumer left off, and partition assignment within a group cannot be coordinated.
3. **In production, who creates Kafka topics?** Topics are typically pre-created by DevOps/infra (Kafka CLI, Terraform, managed services like AWS MSK / Confluent Cloud). Apps usually don't have permission to create topics — drop the `NewTopic` bean and just connect to existing topics.
4. **What controls listener parallelism within one app instance?** `spring.kafka.listener.concurrency` — number of concurrent listener threads per `@KafkaListener`. Effective parallelism is still capped by partition count: `active consumers ≤ partitions`.
5. **Why is consumer idempotency important?** Kafka guarantees **at-least-once** delivery by default — if a consumer processes a message but crashes before committing, the message is redelivered on restart. Idempotent consumers safely handle duplicates.

## Quick Reference

|Concept|Remember|
|---|---|
|`KafkaTemplate<K, V>`|Producer-side send class. `send(topic, value)` → no key (sticky); `send(topic, key, value)` → keyed (deterministic partition)|
|`@KafkaListener(topics = "...")`|Marks consumer method; Spring runs the listener container|
|`NewTopic` bean|Auto-creates topic on startup — remove in production|
|`group-id`|Required on consumer; convention: `${spring.application.name}`|
|`spring.kafka.listener.concurrency`|Concurrent listener threads per `@KafkaListener`|
|Delivery|At-least-once by default — design consumers to be **idempotent**|
