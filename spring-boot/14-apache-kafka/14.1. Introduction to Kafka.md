## The Problem

In a distributed system, data flows from **sources** (services that produce data) to **targets** (services that consume that data).

> **Source** = any service that generates data — e.g., Order Service, Payment Service  
> **Target** = any service that needs that data — e.g., Inventory Service, Notification Service, Analytics

Without a messaging system, services call each other directly using synchronous calls (e.g., HTTP APIs).

<p align="center"> <img src="https://estafet.com/wp-content/uploads/2024/02/image-21.png" width="500"> </p>

This **tight coupling** creates serious problems as you scale:

**If a source goes down, cascading failures occur**: in direct synchronous communication, targets depend on source availability. If Order Service crashes, any service trying to fetch data from it will fail or timeout, causing cascading failures.

**Adding a new target requires changes in all sources**: if you add a new Reporting Service that needs order data, you have to go into every source (Order Service, Payment Service, etc.) and add code to also send data to it. Every new consumer = code changes in all sources.

**Services cannot scale independently**: Target is directly waiting for Source to respond. If Order Service is slow, everything waiting on it is slow too.

**If a target goes down temporarily, data is lost**: Order Service sends data → Inventory Service is down for 2 minutes → those events are lost unless the system implements retries or buffering — there is no built-in replay mechanism. Critical for payments, inventory, auditing.

---

## What is a Messaging System?

Think of it like a **post office**. Instead of you directly delivering a letter to someone, you drop it at the post office and the receiver picks it up when they're ready. The sender and receiver don't need to be available at the same time.

```
Without messaging:

  Order Service ──direct call──> Inventory Service

  Inventory down? → data lost, error returned

With messaging:

  Order Service ──> [Kafka] ──> Inventory Service

  Inventory down? → message waits safely in Kafka, delivered when it's back
```

The sender **drops a message and moves on** without waiting for the receiver. This is called **asynchronous communication**.

> This is different from **REST-based synchronous communication** (e.g., HTTP APIs, OpenFeign) where the sender waits for a response, and **WebSockets** which are real-time two-way connections between a browser/app and a server. Kafka is primarily used for **backend service-to-service async communication**.

---

## What is Kafka?

Apache Kafka is a **horizontally scalable, fault-tolerant, fast messaging system** based on the **pub-sub (publish-subscribe) model**. Producers publish events to Kafka, consumers subscribe independently — fully decoupled from each other.

Kafka is often categorized as a messaging system, but it is more accurately a **distributed event streaming platform**.

Kafka is also a **distributed append-only log system**: this is an important concept and a common interview keyword.

<p align="center"> <img src="https://estafet.com/wp-content/uploads/2024/02/image-22.png" width="500"> </p>

- Developed at **LinkedIn** in Scala, open-sourced in **2011**, now part of Apache Foundation
- Stores data on **disk** so data survives crashes — but uses **OS disk cache for hot data** so it's still very fast

> **Hot data** = recently written or frequently accessed messages. Your OS automatically keeps recently used disk data in RAM (called page cache). Kafka takes advantage of this — recent messages are served at RAM speed without Kafka needing to manage it manually. That's why Kafka is fast despite writing to disk.

> **Fault Tolerant** = Kafka is highly durable and copies data across multiple brokers. However, guarantees depend on configuration — replication factor and acknowledgements (acks). Covered in detail in [14.2 Kafka Architecture](<14.2. Kafka Architecture.md>).

> **LinkedIn processes 1 billion messages per day using Kafka**: every page view, profile visit, job click, connection request = one event. Analytics, feed recommendations, notifications all independently consume from Kafka.

### Key Performance Numbers

| Feature             | What it means                                                                 | Kafka                                        |
|---------------------|-------------------------------------------------------------------------------|----------------------------------------------|
| **Latency**         | Time between a message being sent and received — like ping, lower is better   | As low as 2ms                                |
| **Throughput**      | How many messages processed per second — like bandwidth, higher is better     | 100s MB/s, 100Ks msg/s                       |
| **Fault Tolerance** | System keeps working even when parts fail                                     | High — depends on replication + acks config  |

> **Node** = one server/machine. "Scales to 100s of nodes" = you can keep adding machines and Kafka handles the rest.

## Kafka is a Log, Not a Queue

This is an important mindset shift from traditional message queues:

- **Traditional queue**: message is removed after it is consumed
- **Kafka**: message stays after consumption, until the retention policy removes it

This means multiple consumers can read the same message independently at their own pace.

> **Pull vs push:** unlike traditional queues (e.g., RabbitMQ) that **push** messages to consumers, Kafka uses a **pull model**: consumers fetch data at their own pace. Covered in detail in [14.2 Kafka Architecture](<14.2. Kafka Architecture.md>).

## Use Cases

**Use Kafka for**: **Real-time data streaming**: processing continuous streams of events as they arrive
- **Log aggregation**: collect logs from many services into one place
- **Event sourcing**: `order-placed` event → Inventory Service reduces stock + Notification Service sends email, both independently
- **Async messaging between microservices**: services communicate without waiting for each other
- **Stream processing**: Kafka is fundamentally a streaming platform, but also supports batch-style consumption (reading large chunks at once)

**Do NOT use Kafka for**: **Simple request-response**: just need data from another service? Use REST/OpenFeign
- **Small-scale projects**: Kafka adds significant infrastructure overhead, not worth it for small apps
- **High latency tolerance**: if a 1-2 second delay is acceptable, Kafka is overkill
- **Monolithic applications**: Kafka is built for distributed architectures with multiple independent services

## Key Terminologies

<p align="center"> <img src="https://cs-prod-assets-bucket.s3.ap-south-1.amazonaws.com/image_1_95ebbd7440.png" width="500"> </p>

> **ZooKeeper** (used to manage broker metadata — which broker is alive, who is the leader) is now **deprecated**. Kafka is replacing it with **KRaft (Kafka Raft)**: a built-in protocol that lets brokers manage themselves without any third-party tool. Full architecture in [14.2 Kafka Architecture](<14.2. Kafka Architecture.md>).

**Producer**: any service that sends events to Kafka. Example: Order Service publishing `order-placed`.

**Consumer**: any service that reads events from Kafka. Example: Inventory Service listening to `order-placed`.

**Event**: a single message with data, timestamp, and metadata. Belongs to a topic.

**Broker**: the Kafka software/process running on a node. Stores data, serves producers and consumers.

**Cluster**: a group of brokers working together as one system, distributing load among themselves.

> **Node vs Broker:** Node = a physical/virtual machine (server) — just hardware, could be an EC2 instance or a VM. Broker = the Kafka application running ON that node. Typically **1 node = 1 broker**. People use them interchangeably in casual conversation but they are different concepts.

## Kafka Topic

A **topic** is a named stream of data — like a table in a NoSQL database. Different types of events go to different topics.

```
Order events    ──> orders topic
Shipment events ──> shipments topic
Payment events  ──> payments topic
```

Example topic names for e-commerce: `order-placed`, `order-shipped`, `inventory-updated`, `payment-processed`

> To **uniquely identify one event** in Kafka: `Topic Name + Partition ID + Offset`

---

## Topic Partitions

Each topic is split into **partitions**: the basic unit of parallelism in Kafka. Think of a topic as a big database table that's been **sharded** into smaller pieces.

<p align="center"> <img src="https://cs-prod-assets-bucket.s3.ap-south-1.amazonaws.com/image_3_2475f0bb4e.png" width="500"> </p>

- Partitions of the **same topic** can live on **different brokers** → this is how Kafka scales horizontally
- More partitions = more consumers can read simultaneously in parallel
- Number of partitions should be chosen carefully — too many can impact performance and cluster stability

**How is an event assigned to a partition?**: **With a message key** → Kafka hashes the key → same key always goes to same partition. Message keys are critical when order matters — e.g., all events for the same order ID must go to the same partition. Covered in detail in [14.2 Kafka Architecture](<14.2. Kafka Architecture.md>).
- **Without a key** → Kafka uses round-robin → events spread evenly across partitions

**Ordering guarantee**: Kafka guarantees ordering **only within a partition**, not across partitions. Covered in detail in [14.2 Kafka Architecture](<14.2. Kafka Architecture.md>).

## Consuming Topics

- Kafka does **not delete messages after consumption**: messages are kept based on a **retention policy** (time-based or size-based). Covered in detail in [14.2 Kafka Architecture](<14.2. Kafka Architecture.md>).
- Consumers can **reprocess** old messages by reading from an earlier offset — as long as the retention policy hasn't deleted them yet. This is critical for debugging, rebuilding state, and event sourcing.

## Key Guarantees of Kafka

|Guarantee|Detail|
|---|---|
|**Ordering**|Only within a partition, not across partitions|
|**Durability**|Depends on replication factor + acks configuration|
|**Scalability**|Horizontal scaling via adding partitions and brokers|
|**Replayability**|Consumers control their own offsets — can re-read old messages|
|**Decoupling**|Producers and consumers are fully independent of each other|

## Q&A

1. **What is Kafka and how is it different from a traditional message queue?** Kafka is a **distributed append-only log** / event streaming platform. Unlike traditional queues that delete messages on consumption, Kafka keeps messages until the retention policy expires — multiple consumers can independently re-read at their own pace.
2. **Why is Kafka fast despite writing to disk?** Kafka writes sequentially to disk and relies on the OS **page cache** to serve hot data at RAM speed. Sequential I/O + zero-copy transfers make it faster than typical random-access disk usage.
3. **When should you NOT use Kafka?** For simple request-response, small-scale projects, monolithic apps, or anywhere a 1-2 second latency is acceptable. Kafka adds significant infrastructure overhead — use REST/OpenFeign instead for synchronous calls.
4. **What replaces ZooKeeper in modern Kafka?** **KRaft (Kafka Raft)**: a built-in consensus protocol where brokers coordinate among themselves to manage cluster metadata, eliminating the external ZooKeeper dependency.
5. **How is a single event uniquely identified in Kafka?** By the triple `Topic Name + Partition ID + Offset`. Offsets are local to a partition, so the partition ID is required.
