In a distributed system, services that produce data (sources) need to reach services that consume it (targets). Wiring them with direct synchronous calls (HTTP) couples them tightly: if a source is down its callers fail and cascade, every new consumer means editing every producer, a slow source slows everyone waiting on it, and an event is lost if a target is briefly down. A messaging system decouples the two: the producer drops a message and moves on (asynchronous), and the consumer reads it when ready, even if the two are never up at the same time.

This differs from REST/OpenFeign (synchronous request-response) and WebSockets (real-time two-way); Kafka is for backend service-to-service async communication.

---

## What Kafka Is

Apache Kafka is a horizontally scalable, fault-tolerant **distributed event streaming platform** based on the publish-subscribe model: producers publish events, consumers subscribe independently, fully decoupled. The mental model that matters in interviews is that Kafka is a **distributed, append-only log**, not a queue:

- A traditional queue (RabbitMQ, SQS) deletes a message once consumed.
- Kafka keeps messages until a retention policy expires them, so multiple consumers can read the same messages independently and at their own pace, and can replay old ones.

It is durable because it writes sequentially to disk, yet fast because the OS page cache serves recently-written ("hot") data from RAM. Developed at LinkedIn in Scala and open-sourced in 2011, it delivers single-digit-millisecond latency and hundreds of MB/s throughput, scaling to hundreds of nodes. Actual durability depends on configuration (replication factor and `acks`), covered in [14.2. Kafka Architecture](<14.2. Kafka Architecture.md>).

## Core Terminology

- **Producer**: a service that sends events (Order Service publishing `order-placed`).
- **Consumer**: a service that reads events (Inventory Service listening to `order-placed`).
- **Event**: a single message (key, value, timestamp, metadata) belonging to a topic.
- **Broker**: the Kafka process running on a node; stores data and serves clients.
- **Node vs broker**: a node is the machine, a broker is the Kafka process on it; typically one node runs one broker.
- **Cluster**: a group of brokers acting as one system, coordinated by **KRaft** (see below).
- **Topic**: a named stream of events, like a table; different event types go to different topics (`order-placed`, `payment-processed`).
- **Partition**: a topic is split into partitions, the unit of parallelism and the physical storage unit (a topic is just a namespace over them).
- **Offset**: a per-partition sequential ID (0, 1, 2, ...) for each record. A single event is uniquely identified by `Topic + Partition + Offset`, since offsets are local to a partition.

Partition mechanics (key-hash vs round-robin assignment, per-partition ordering, replication, consumer groups, offset commits) are the subject of [14.2. Kafka Architecture](<14.2. Kafka Architecture.md>).

> **Note**: cluster metadata (which broker is alive, who leads each partition) was historically managed by an external **ZooKeeper**. Kafka now uses **KRaft** (Kafka Raft), a built-in consensus protocol where brokers coordinate among themselves. ZooKeeper mode was fully removed in Kafka 4.0, so KRaft is the only option on current versions.

## When To Use Kafka, and When Not

Use Kafka for real-time data streaming, log aggregation, event sourcing (one `order-placed` event independently drives inventory and notifications), async microservice messaging, and stream processing.

Avoid it for simple request-response (use REST/OpenFeign), small-scale or monolithic apps where the infrastructure overhead is not justified, and cases where a 1-2 second delay is perfectly acceptable.

| System         | Language | Storage             | Note                                                        |
| -------------- | -------- | ------------------- | ----------------------------------------------------------- |
| RabbitMQ       | Erlang   | Memory or disk      | Simple, lightweight queue; push-based                       |
| ActiveMQ       | Java     | Memory or disk      | Apache, JMS-oriented                                         |
| Apache Kafka   | Scala    | Disk + OS page cache | Higher throughput via sequential writes, batching, page cache; pull-based |
| Amazon SQS     | -        | Managed by AWS      | Fully managed, not open source                              |

## Key Guarantees

| Guarantee     | Detail                                                       |
| ------------- | ----------------------------------------------------------- |
| Ordering      | Within a partition only, never across partitions            |
| Durability    | Depends on replication factor + `acks` config               |
| Scalability   | Horizontal, via more partitions and brokers                 |
| Replayability | Consumers control their offsets and can re-read old messages |
| Decoupling    | Producers and consumers are fully independent               |

## Q&A

1. **What is Kafka and how does it differ from a traditional message queue?**
A distributed append-only log / event streaming platform. Unlike queues that delete on consumption, Kafka retains messages until the retention policy expires them, so multiple consumers can independently re-read at their own pace.

2. **Why is Kafka fast despite writing to disk?**
It writes sequentially (not random I/O) and lets the OS page cache serve hot data at RAM speed, with zero-copy transfers to consumers.

3. **When should you not use Kafka?**
Simple request-response, small or monolithic apps, or where a 1-2 second delay is fine. The operational overhead is not worth it; use REST/OpenFeign for synchronous calls.

4. **What replaces ZooKeeper in modern Kafka?**
KRaft, a built-in Raft-based consensus protocol where brokers elect a controller and manage metadata themselves. ZooKeeper mode was removed in Kafka 4.0.

5. **How is a single event uniquely identified?**
By `Topic + Partition + Offset`; offsets are local to a partition, so the partition is required.
