In [14.5. Advanced Kafka Configuration with Spring Boot](<14.5. Advanced Kafka Configuration with Spring Boot.md>) the event class had to be duplicated in every service with a matching package. That does not scale: 10 services means 10 copies, a field change means editing all of them, and there is no central record of what events look like or how they evolved. A **Schema Registry** is a centralized server that stores and versions event schemas; producers and consumers reference schemas from it instead of hand-maintaining copies, which enforces compliance (producers cannot send malformed messages) and underpins data governance.

---

## How It Works

1. The producer registers its schema with the registry and gets a numeric **schema ID**.
2. The producer serializes the message and prepends the schema ID to the payload (a magic byte + 4-byte ID, then the Avro-encoded data).
3. The consumer reads the schema ID from the payload, fetches that schema from the registry (cached after first use), and deserializes.

Because the ID travels in the payload and consumers cache schemas, the registry is hit only on an unseen ID, not per message.

---

## Setup

Run Confluent Schema Registry via the official KRaft Docker Compose ([cp-all-in-one-kraft](https://github.com/confluentinc/cp-all-in-one)); keep the `broker` and `schema-registry` (port 8081) services, optionally `control-center` (UI, port 9021). Stop any local Kafka first, since both use port 9092.

> **Note**: if Schema Registry is unavailable, producers cannot register new schemas and may fail to send, but consumers with a cached schema keep working.

Define the event as an Avro schema in `src/main/resources/avro/user-created-event.avsc` in both services:

```json
{
  "name": "UserCreatedEvent",
  "namespace": "com.learn.kafka.event",
  "type": "record",
  "fields": [
    { "name": "id",    "type": "long",   "default": 0  },
    { "name": "name",  "type": "string", "default": "" },
    { "name": "email", "type": "string", "default": "" }
  ]
}
```

> **Note**: `namespace` sets the Java package of the generated class. Default values are required when adding new fields so the change stays backward-compatible (the default compatibility mode is `BACKWARD`); without them the registry rejects the new schema.

Add the Confluent Avro serializer, the schema-registry client, and Avro itself (from the Confluent Maven repo, since these are not on Maven Central), plus the `avro-maven-plugin` to generate Java classes from `.avsc` at build time. Versions below are examples; align them with your Confluent Platform and Avro versions:

```xml
<dependency>
    <groupId>io.confluent</groupId>
    <artifactId>kafka-avro-serializer</artifactId>
    <version>7.7.1</version>
</dependency>
<dependency>
    <groupId>org.apache.avro</groupId>
    <artifactId>avro</artifactId>
    <version>1.11.3</version>
</dependency>
<repositories>
    <repository>
        <id>confluent</id>
        <url>https://packages.confluent.io/maven/</url>
    </repository>
</repositories>
```

Building (`./mvnw install`) runs the plugin and generates `UserCreatedEvent.java` under the namespace package, replacing the hand-written class. Repeat for both services.

> **Note**: keeping the `.avsc` in both services is a build-time concern (the plugin needs it to generate the class); the registry is a separate runtime concern (validation and versioning). In production the `.avsc` files live in a shared Git repo so every service builds from one source.

Switch the serializers to Avro and point at the registry:

```yaml
# user-service
spring:
  kafka:
    producer:
      value-serializer: io.confluent.kafka.serializers.KafkaAvroSerializer
      properties:
        schema.registry.url: http://localhost:8081

# notification-service
spring:
  kafka:
    consumer:
      value-deserializer: io.confluent.kafka.serializers.KafkaAvroDeserializer
      properties:
        schema.registry.url: http://localhost:8081
        specific.avro.reader: true   # map to the generated class, not GenericRecord
```

> **Note**: `spring.json.trusted.packages` is no longer needed; Avro + Schema Registry handle type safety through the schema ID in each payload.

## Versioning and Compatibility

Each rebuild with a changed `.avsc` registers a new version under a **subject**, named `<topic>-value` by default (so topic `user-created-topic` uses subject `user-created-topic-value`). The registry enforces a compatibility mode between versions:

| Mode              | Rule                                                  |
| ----------------- | ----------------------------------------------------- |
| `BACKWARD` (default) | New consumers can read data written by old producers |
| `FORWARD`         | Old consumers can read data written by new producers  |
| `FULL`            | Both backward and forward compatible                  |
| `NONE`            | No validation                                         |

Because the default is `BACKWARD`, new fields need defaults, otherwise the new schema is rejected. Choose the mode by which side you roll out first.

## JSON vs Avro

|                   | JSON serializer (14.5)    | Avro + Schema Registry (14.6)        |
| ----------------- | ------------------------- | ------------------------------------ |
| Setup             | Simple                    | Needs a Schema Registry server       |
| Schema management | Manual (duplicate classes) | Centralized `.avsc`                  |
| Validation        | None                      | Registry enforces the schema         |
| Versioning        | None                      | Full version history                 |
| Payload size      | Larger (verbose JSON)     | Smaller (binary)                     |
| Best for          | Small projects, quick setup | Production, multiple teams/services |

## Q&A

1. **What problem does Schema Registry solve?**
It removes manual class duplication across services. Producers and consumers register and fetch schemas from a central server keyed by schema ID, giving one source of truth for event shape.

2. **Where is the schema ID stored, header or payload?**
Payload. Confluent's wire format prepends a magic byte plus a 4-byte schema ID, then the Avro-encoded data.

3. **Is the registry called for every message?**
No. Consumers cache schemas after the first fetch and only call the registry on an unseen ID. Cached consumers survive a brief registry outage; producers may fail to register new schemas.

4. **Why are default values required when adding fields?**
The default compatibility mode is `BACKWARD` (new consumers must read old data); a new field without a default breaks that, so the registry rejects the schema.

5. **`BACKWARD` vs `FORWARD` vs `FULL`?**
`BACKWARD`: new consumers read old data. `FORWARD`: old consumers read new data. `FULL`: both. `NONE`: no validation. Pick by which side deploys first.

6. **Why keep `.avsc` in both services in this demo?**
It is a build-time input for the Avro plugin to generate the class; the registry is a separate runtime concern. In production the `.avsc` lives in a shared repo.

7. **Avro vs JSON, when to pick which?**
Avro + registry for production with multiple teams (smaller binary payloads, enforced schema, version history); JSON for small projects where the registry overhead is not justified.
