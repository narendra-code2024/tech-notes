Extends the basic setup in [14.4. Configuring Kafka with Spring Boot](<14.4. Configuring Kafka with Spring Boot.md>) to send complex objects as JSON: serializer/deserializer config, trusted packages, the package-coupling problem, and production-grade producer/consumer options. The scenario: on user creation, publish a `UserCreatedEvent` that interested services (notification, wallet) consume independently instead of being called directly.

---

## Event Class

Use a dedicated event class, separate from the entity and the request DTO:

```java
// com.learn.kafka.event.UserCreatedEvent
@Data @AllArgsConstructor @NoArgsConstructor
public class UserCreatedEvent {
    private Long id;
    private String email;
}
```

> **Note**: the entity is a database model (should not leak over the wire), the request DTO is controller input, and the event carries only what consumers need. Keeping them separate prevents schema coupling between the database, the API, and the event bus.

---

## Producer and Consumer Serialization

The producer serializes the key as `Long` and the value as JSON; the consumer mirrors it and must declare trusted packages:

```yaml
# user-service
spring:
  kafka:
    producer:
      key-serializer: org.apache.kafka.common.serialization.LongSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer

# notification-service
spring:
  kafka:
    consumer:
      group-id: ${spring.application.name}
      key-deserializer: org.apache.kafka.common.serialization.LongDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
      properties:
        spring.json.trusted.packages: "com.learn.kafka.*"
```

```java
// producer: publish after saving
UserCreatedEvent event = modelMapper.map(savedUser, UserCreatedEvent.class);
kafkaTemplate.send(userCreatedTopic, savedUser.getId(), event);

// consumer
@KafkaListener(topics = "${kafka.topic.user-created}")
public void handleUserCreated(UserCreatedEvent event) {
    log.info("event received: {}", event);
}
```

> **Note**: `spring.json.trusted.packages` is a safelist of packages the `JsonDeserializer` is allowed to instantiate. Without it the deserializer rejects unknown classes, which defends against deserialization attacks (malicious payloads loading dangerous classes).

## The Package-Coupling Problem

`JsonSerializer` embeds the producer's fully-qualified class name in a `__TypeId__` header:

```
__TypeId__ = com.learn.kafka.event.UserCreatedEvent
```

The consumer's deserializer instantiates that exact class path, so the consumer must define `UserCreatedEvent` in the identical package, which means duplicating the class in every consuming service.

> **Reference**: a cleaner solution is a central schema store; see [14.6. Kafka Schema Registry with Confluent](<14.6. Kafka Schema Registry with Confluent.md>), which removes the class-copying entirely.

## Production Producer and Consumer Options

```yaml
spring:
  kafka:
    producer:
      acks: all                      # default since Kafka 3.0
      retries: 3
      properties:
        enable.idempotence: true     # default since Kafka 3.0; do not disable
    consumer:
      auto-offset-reset: earliest    # only applies on first connect (no committed offset)
      enable-auto-commit: false      # manual commit is safer for at-least-once
      max-poll-records: 500          # max records per poll() (default 500)
```

`auto-offset-reset` decides where a brand-new group starts when it has no committed offset: `earliest` reads from the oldest retained message (safe, misses nothing), `latest` jumps to the next produced message. Once a committed offset exists, the consumer always resumes from there regardless of this setting.

> **Note**: since Kafka 3.0 the producer already defaults to `acks=all` + `enable.idempotence=true` (the broker dedupes retries via a producer ID and per-partition sequence numbers), so these are guarantees you keep rather than add. For durability, pair them with broker-side `min.insync.replicas=2` and `replication.factor=3`.

> **Note**: prefer `enable-auto-commit: false` in production. Auto-commit can advance the offset before processing finishes, silently losing a message on a crash. Manual commit (`AckMode.MANUAL_IMMEDIATE` + `Acknowledgment.acknowledge()`) commits only after successful processing for at-least-once delivery; consumers must still be idempotent since redelivery is possible.

## Dead Letter Topics

A poison message can never be processed (malformed payload, deserialization error, business-rule violation). Without handling, Spring redelivers it forever and blocks the partition. The standard pattern retries a few times, then forwards the record to a Dead Letter Topic:

```java
@Bean
public DefaultErrorHandler errorHandler(KafkaTemplate<Object, Object> template) {
    DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(template);
    return new DefaultErrorHandler(recoverer, new FixedBackOff(1000L, 2L)); // 1s backoff, 2 retries
}
```

Spring publishes the failed record to `<topic>.DLT` with headers describing the failure; operators inspect it, fix the bug or data, and replay. Pair this with lag and DLT-depth alerts so poison messages do not go unnoticed.

## Q&A

1. **Why a separate event class instead of the entity or DTO?**
The entity is a DB model, the DTO is controller input, and the event carries only what consumers need. Separating them avoids coupling the database, API, and event bus schemas.

2. **What is `spring.json.trusted.packages` and why required?**
A safelist of packages the `JsonDeserializer` may instantiate. Without it, it rejects unknown classes to defend against deserialization attacks.

3. **Why must the consumer's event class share the producer's package?**
`JsonSerializer` writes the FQCN into `__TypeId__`, and the deserializer uses that exact path. Schema Registry (14.6) removes this coupling.

4. **`auto-offset-reset=earliest` vs `latest`?**
Applies only on first connect. `earliest` starts from the partition's oldest retained message (misses nothing); `latest` skips to the next produced message. With a committed offset, the consumer resumes from there regardless.

5. **Why disable auto-commit in production?**
It can advance the offset before processing finishes, losing messages on a crash. Manual commit after processing gives at-least-once delivery; consumers still need idempotency.

6. **What does `enable.idempotence=true` give you, and is it default?**
The broker dedupes producer retries via a producer ID + per-partition sequence number, eliminating producer-side duplicates. It is the default since Kafka 3.0; end-to-end exactly-once still needs consumer idempotency or transactions.
