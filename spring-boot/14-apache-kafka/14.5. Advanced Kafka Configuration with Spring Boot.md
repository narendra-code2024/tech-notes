## Use Case — User Created Event

> Extends the basic setup from [14.4 Configuring Kafka with Spring Boot](<14.4. Configuring Kafka with Spring Boot.md>) to cover sending complex objects (JSON), serializer/deserializer configuration, trusted packages, and advanced producer/consumer options.

When a user is created, instead of calling multiple microservices directly (tight coupling), publish an event to Kafka. Any interested service can consume it independently.

```
POST /users
      │
      ▼
UserService → saves user → publishes UserCreatedEvent to Kafka
                                        │
                              ┌─────────┴──────────┐
                              ▼                    ▼
                    NotificationService      WalletService
                    (sends welcome email)   (creates wallet)
```

---

## Event Class

Create a dedicated event class — separate from the entity and request DTO:

```java
// com.learn.kafka.user_service.event.UserCreatedEvent
@Data
@AllArgsConstructor
@NoArgsConstructor
public class UserCreatedEvent {
    private Long id;
    private String email;
}
```

> **Why a separate event class?**
> 
> - Entity = database model, should not be exposed over the network
> - Request DTO = input from controller, may contain fields not relevant to other services
> - Event = only what other services need to know

> **Package naming matters**: the consumer needs to resolve the same class. The package name in the producer's event class must match what the consumer expects. More on this below.

## Producer Configuration — `user-service`

### `application.yml`

```yaml
spring:
  application:
    name: user-service
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.LongSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer

kafka:
  topic:
    user-random: user-random-topic
    user-created: user-created-topic
```

> **Serializer** = converts a Java object → bytes for transmission over the network.  
> `LongSerializer` — for Long keys (user ID).  
> `JsonSerializer` — converts the event object to JSON bytes automatically.

### Topic Config — add new topic

```java
@Bean
public NewTopic userCreatedTopic(
    @Value("${kafka.topic.user-created}") String topicName) {
    return new NewTopic(topicName, 3, (short) 1);
}
```

### Service — publish event

```java
@Service
@RequiredArgsConstructor
public class UserService {

    private final UserRepository userRepository;
    private final KafkaTemplate<Long, UserCreatedEvent> kafkaTemplate;

    @Value("${kafka.topic.user-created}")
    private String userCreatedTopic;

    public User createUser(CreateUserRequest request) {
        User savedUser = userRepository.save(
            User.builder()
                .name(request.getName())
                .email(request.getEmail())
                .build()
        );

        UserCreatedEvent event = modelMapper.map(savedUser, UserCreatedEvent.class);
        kafkaTemplate.send(userCreatedTopic, savedUser.getId(), event);

        return savedUser;
    }
}
```

## Consumer Configuration — `notification-service`

### `application.yml`

```yaml
spring:
  application:
    name: notification-service
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: ${spring.application.name}
      key-deserializer: org.apache.kafka.common.serialization.LongDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
      properties:
        spring.json.trusted.packages: "com.learn.kafka.*"

kafka:
  topic:
    user-created: user-created-topic
```

> **Deserializer** = converts bytes received from Kafka → back into a Java object.  
> `LongDeserializer` — for Long keys.  
> `JsonDeserializer` — converts JSON bytes back to the event object.

> **`spring.json.trusted.packages`**: required for security. By default, the JSON deserializer rejects unknown packages to prevent **deserialization attacks** (malicious code disguised as serialized objects). You must explicitly whitelist trusted packages.  
> Use `"com.learn.kafka.*"` to trust all packages under that namespace, or specify the exact package.

### Event Class — must match producer's package

The consumer needs the same event class with the **exact same package name** as the producer:

```java
// com.learn.kafka.user_service.event.UserCreatedEvent  ← must match producer
@Data
@AllArgsConstructor
@NoArgsConstructor
public class UserCreatedEvent {
    private Long id;
    private String email;
}
```

### Consumer

```java
@Service
@Slf4j
public class UserKafkaConsumer {

    @KafkaListener(topics = "${kafka.topic.user-created}")
    public void handleUserCreated(UserCreatedEvent event) {
        log.info("handleUserCreated - event received: {}", event);
        // send welcome email, trigger onboarding, etc.
    }
}
```

## The Package Problem & Solution

**Problem:** The `JsonSerializer` embeds the producer's full class path in the message header:

```
__TypeId__ = com.learn.kafka.user_service.event.UserCreatedEvent
```

If the consumer doesn't have the exact same package structure, deserialization fails.

**Solution:** Create the event class in the consumer with the **exact same package name** as in the producer:

```
user-service/
  └── com.learn.kafka.user_service.event.UserCreatedEvent  ← producer

notification-service/
  └── com.learn.kafka.user_service.event.UserCreatedEvent  ← consumer (same package!)
```

> This works but requires duplicating the class in every consuming service. A cleaner solution is **Kafka Schema Registry** (covered in [14.6 Kafka Schema Registry with Confluent](<14.6. Kafka Schema Registry with Confluent.md>)) — a central place to manage event schemas, eliminating the need to copy classes.

## Advanced Producer & Consumer Configuration

```yaml
spring:
  kafka:
    producer:
      key-serializer: org.apache.kafka.common.serialization.LongSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
      acks: all           # 0 = no ack, 1 = leader ack, all = all ISR replicas ack
      retries: 3          # retry on transient failures (network issues, broker down)
      properties:
        enable.idempotence: true   # safe retries — no duplicates from the producer side

    consumer:
      key-deserializer: org.apache.kafka.common.serialization.LongDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
      auto-offset-reset: earliest   # earliest or latest — see below
      enable-auto-commit: false     # manual commit is safer for at-least-once (see below)
      max-poll-records: 500         # max messages fetched per poll call (default: 500)
      properties:
        spring.json.trusted.packages: "com.learn.kafka.*"
```

> **`enable.idempotence=true`**: pairs with `acks=all` + `retries>0` to give exactly-once semantics on the producer side. Kafka assigns each producer a sequence number per partition and dedupes retries on the broker. Strongly recommended for any non-trivial producer.

### `auto-offset-reset`

|Value|Behaviour|
|---|---|
|`earliest`|If no committed offset exists for the group, start from the **oldest** retained message in the partition|
|`latest`|If no committed offset exists, start from the **next** message produced after the consumer connects|

> Applies only on **first connect** (or when the saved offset has been deleted by retention) — when a committed offset exists, the consumer always resumes from there regardless of this setting.  
> `earliest` is safer for most use cases — ensures no historical messages are missed when a brand-new consumer group starts.

### `enable-auto-commit`

- `true` (Spring/Kafka default) — Kafka automatically commits the offset after the message is read
- `false` (production-safe) — application manually commits after processing

> **Production warning:** prefer `false` in production. With auto-commit, the offset can advance before processing finishes — if the consumer crashes, the unprocessed message is silently lost. Manual commit (e.g., `AckMode.MANUAL_IMMEDIATE` + `Acknowledgment.acknowledge()`) commits only after successful processing, guaranteeing at-least-once delivery. Consumers must still be **idempotent** since redelivery is possible. Offset commit strategies covered in detail in [14.2 Kafka Architecture](<14.2. Kafka Architecture.md>) → Offset commit strategies.

### `max-poll-records`

Maximum number of messages fetched in a single `poll()` call. Default is `500`. Increase for high-throughput servers that can spawn many threads to process records in parallel.

### `acks`

Covered in detail in [14.2 Kafka Architecture](<14.2. Kafka Architecture.md>). Configure via `spring.kafka.producer.acks` — values: `0`, `1`, `all`. For durability, pair `acks=all` with broker-side `min.insync.replicas=2` (and `replication.factor=3`) so writes fail loudly when the cluster cannot provide the promised durability.

## Dead Letter Topics (Poison Messages)

A **poison message** is one that the consumer can never successfully process — malformed payload, deserialization error, business-rule violation. Without DLT handling, Spring keeps redelivering it forever, blocking the entire partition.

The standard pattern is to forward failed messages to a **Dead Letter Topic (DLT)** after a few retries:

```java
@Bean
public DefaultErrorHandler errorHandler(KafkaTemplate<Object, Object> template) {
    DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(template);
    // 3 attempts, 1s backoff, then publish to <topic>.DLT
    return new DefaultErrorHandler(recoverer, new FixedBackOff(1000L, 2L));
}
```

- Spring auto-creates `<original-topic>.DLT` (configurable) — the failed record is published there with headers describing the failure
- Operators inspect the DLT, fix the bug or the data, and replay manually
- Pair with **lag monitoring + DLT depth alerts** so poison messages don't go unnoticed

## Q&A

1. **Why use a separate event class instead of the entity or DTO?** The **entity** is a database model (shouldn't leak over the network), the **request DTO** carries fields specific to the controller input, and the **event** carries only what consumers need. Decoupling them prevents schema coupling between database, API, and event bus.
2. **What is `spring.json.trusted.packages` and why is it required?** A safelist of packages the `JsonDeserializer` will instantiate. Without it, the deserializer rejects unknown classes to defend against **deserialization attacks** (malicious payloads loading dangerous classes).
3. **Why does the consumer's event class need the same package name as the producer's?** `JsonSerializer` embeds the producer's fully-qualified class name in the `__TypeId__` header. The consumer's deserializer uses that exact class path to instantiate the object — mismatched packages fail. **Schema Registry** ([14.6 Kafka Schema Registry with Confluent](<14.6. Kafka Schema Registry with Confluent.md>)) eliminates this coupling.
4. **`auto-offset-reset=earliest` vs `latest` — which to pick?** Only applies on **first connect** (no committed offset exists) — once a committed offset exists, the consumer always resumes from there. `earliest` reads from the start of the partition (safer — no messages missed); `latest` skips to the next produced message (useful only when historical data is irrelevant).
5. **Why turn `enable-auto-commit` off in production?** Auto-commit can advance the offset before processing finishes, silently losing messages if the consumer crashes mid-processing. Manual commit (`AckMode.MANUAL_IMMEDIATE` + `Acknowledgment.acknowledge()`) commits only after successful processing — guarantees at-least-once delivery. Consumers must still be idempotent since redelivery is possible.
6. **What does `enable.idempotence=true` give you?** Combined with `acks=all` and `retries>0`, the producer assigns a sequence number per partition so the broker dedupes retries — eliminating duplicates from the producer side. Effectively exactly-once on the producer; consumers still need idempotency for end-to-end exactly-once.
7. **What does `max-poll-records` control?** Maximum messages returned per `poll()` call (default 500). Larger values improve throughput on multi-threaded consumers; smaller values reduce per-poll processing time and rebalance risk.
