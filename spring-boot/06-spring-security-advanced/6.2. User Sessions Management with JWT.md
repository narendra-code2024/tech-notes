DB-backed refresh + access token implementation. Access token in response body (sent via Bearer header), refresh token in HttpOnly cookie (sent automatically).

---

## Endpoints

|Endpoint|Method|Auth Required|Purpose|
|---|---|---|---|
|`/api/auth/register`|POST|No|Create account|
|`/api/auth/login`|POST|No|Returns access token in body + refresh cookie|
|`/api/auth/refresh`|POST|No (refresh cookie needed)|Returns new access token in body + new refresh cookie|
|`/api/auth/logout`|POST|Yes|Clears cookie + deletes session from DB|

## End-to-End Flow

```
POST /api/auth/login { email, password }
       │
       ▼
Server validates credentials
       │
       ▼
Server returns:
  Access token  → JSON body (15 min)
  Refresh token → HttpOnly cookie (7 days, stored in DB)
       │
       ▼
Client stores access token:
  SPA     → memory (React state / JS variable)
  Postman → environment variable (via script)
       │
       ▼
All API calls:
  Authorization: Bearer <access_token>
       │
       ▼
Access token expired (401)?
       │
       ▼
POST /api/auth/refresh
  Refresh cookie sent automatically
  Server validates against DB
  Returns new access token in body + new refresh cookie
       │
       ▼
Refresh also fails (401)?
  → Redirect to login (session fully expired)
```

> SPA refreshes silently on 401; Postman requires a manual `POST /refresh` call (or a script).

## Server-Side Flows

**Login:**

```
Client: POST /api/auth/login { email, password }
       │
       ▼
Server validates credentials (AuthenticationManager)
       │
       ▼
Checks session limit (max from properties)
  → If at limit, oldest session deleted
       │
       ▼
Creates access token JWT (15 min)
       │
       ▼
Creates refresh token JWT (7 days)
       │
       ▼
Saves refresh token in Session table (DB)
       │
       ▼
Returns:
  Body: { "token": "eyJ..." }
  Cookie: refresh_token=eyJ... (HttpOnly, Secure, SameSite=Lax)
```

**Refresh:**

```
Client: POST /api/auth/refresh
  Cookie: refresh_token=eyJ... (sent automatically)
       │
       ▼
Server extracts refresh token from cookie
       │
       ▼
Validates JWT signature + expiry
       │
       ▼
Checks if token exists in Session table (DB)
  ├── Not found → 401 (token was revoked)
  └── Found ↓
       │
       ▼
Deletes old session from DB
       │
       ▼
Creates new access token + new refresh token
       │
       ▼
Saves new refresh token in Session table
       │
       ▼
Returns:
  Body: { "token": "eyJ..." }
  Cookie: refresh_token=eyJ... (new, HttpOnly, Secure, SameSite=Lax)
```

**Logout:**

```
Client: POST /api/auth/logout
       │
       ▼
Deletes session from DB (by refresh token)
       │
       ▼
Clears refresh cookie (maxAge = 0)
       │
       ▼
Access token expires on its own (15 min max)
```

## Implementation

All code below extends the implementation in [5.7 JWT Implementation](<../05-spring-security-fundamentals/5.7. JWT Implementation.md>); deltas are explained inline.

**Application properties**: separate secrets and expiry per token type, plus the per-user session cap:

```properties
jwt.access-token.secret=your-access-token-secret-key-make-it-long-256-bit
jwt.access-token.expiration=900000
# 900000ms = 15 minutes

jwt.refresh-token.secret=your-refresh-token-secret-key-different-from-access
jwt.refresh-token.expiration=604800000
# 604800000ms = 7 days

jwt.session.max-limit=3
# Max active sessions per user (e.g., 3 devices)
```

> Use different secrets for access and refresh tokens. If the access token secret is compromised, the refresh tokens are still safe.

> **Session limit behavior**: when a user already has `max-limit` active sessions and logs in from a new device, the **oldest session is evicted** to make room for the new one. Same pattern as Netflix / Spotify. Enforced by the `while` loop in `AuthService.login` below.

**Session entity**: one row per active refresh token; `provider` records which auth path created it:

```java
@Entity
@Table(name = "sessions")
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class Session {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, unique = true)
    private String refreshToken;

    @Column(nullable = false)
    private Instant expiresAt;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    private AuthProvider provider; // LOCAL, GOOGLE, GITHUB

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", nullable = false)
    private User user;
}

public enum AuthProvider {
    LOCAL, GOOGLE, GITHUB
}
```

**SessionRepository**: lookups for refresh validation, count-by-user for the session cap, and oldest-first ordering for eviction:

```java
public interface SessionRepository extends JpaRepository<Session, Long> {

    Optional<Session> findByRefreshToken(String refreshToken);

    void deleteByUser(User user);

    void deleteByRefreshToken(String refreshToken);

    long countByUser(User user);

    // Delete oldest session(s) when limit is reached
    Optional<Session> findFirstByUserOrderByIdAsc(User user);
}
```

**JwtService**: separate secrets and expiry for access vs refresh tokens:

```java
@Service
public class JwtService {

    @Value("${jwt.access-token.secret}")
    private String accessSecret;

    @Value("${jwt.access-token.expiration}")
    private long accessExpiration;

    @Value("${jwt.refresh-token.secret}")
    private String refreshSecret;

    @Value("${jwt.refresh-token.expiration}")
    private long refreshExpiration;

    public String generateAccessToken(String email) {
        return buildToken(email, accessExpiration, getSigningKey(accessSecret));
    }

    public String generateRefreshToken(String email) {
        return buildToken(email, refreshExpiration, getSigningKey(refreshSecret));
    }

    public String extractEmail(String token, boolean isRefreshToken) {
        SecretKey key = isRefreshToken
            ? getSigningKey(refreshSecret)
            : getSigningKey(accessSecret);
        return extractClaims(token, key).getSubject();
    }

    public boolean isTokenValid(String token, UserDetails userDetails,
            boolean isRefreshToken) {
        String email = extractEmail(token, isRefreshToken);
        SecretKey key = isRefreshToken
            ? getSigningKey(refreshSecret)
            : getSigningKey(accessSecret);
        return email.equals(userDetails.getUsername())
            && !isTokenExpired(token, key);
    }

    public Instant getRefreshTokenExpiry() {
        return Instant.now().plusMillis(refreshExpiration);
    }

    public long getRefreshExpirationSeconds() {
        return refreshExpiration / 1000;
    }

    private String buildToken(String email, long expiration, SecretKey key) {
        return Jwts.builder()
            .subject(email)
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + expiration))
            .signWith(key)
            .compact();
    }

    private boolean isTokenExpired(String token, SecretKey key) {
        return extractClaims(token, key).getExpiration().before(new Date());
    }

    private Claims extractClaims(String token, SecretKey key) {
        return Jwts.parser()
            .verifyWith(key)
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }

    private SecretKey getSigningKey(String secret) {
        return Keys.hmacShaKeyFor(secret.getBytes());
    }
}
```

**AuthService**: access token in body only, refresh token in HttpOnly cookie only, session stored in DB, session limit enforced:

```java
@Service
@Slf4j
@RequiredArgsConstructor
public class AuthServiceImpl implements AuthService {

    private final UserRepository userRepository;
    private final SessionRepository sessionRepository;
    private final PasswordEncoder passwordEncoder;
    private final AuthenticationManager authenticationManager;
    private final JwtService jwtService;
    private final UserMapper userMapper;
    private final CustomUserDetailsService userDetailsService;

    @Value("${jwt.session.max-limit}")
    private int maxSessionLimit;

    @Override
    public AuthResponse login(LoginRequest request,
            HttpServletResponse response) {

        // Throws AuthenticationException if credentials are invalid
        authenticationManager.authenticate(
            new UsernamePasswordAuthenticationToken(
                request.getEmail(), request.getPassword()));

        User user = userRepository.findByEmail(request.getEmail())
            .orElseThrow();

        // Enforce session limit — remove oldest if at max
        while (sessionRepository.countByUser(user) >= maxSessionLimit) {
            sessionRepository.findFirstByUserOrderByIdAsc(user)
                .ifPresent(sessionRepository::delete);
        }

        String accessToken = jwtService.generateAccessToken(user.getEmail());
        String refreshToken = jwtService.generateRefreshToken(user.getEmail());

        // Save refresh token in DB
        Session session = Session.builder()
            .refreshToken(refreshToken)
            .expiresAt(jwtService.getRefreshTokenExpiry())
            .provider(AuthProvider.LOCAL)
            .user(user)
            .build();
        sessionRepository.save(session);

        // Set refresh token as HttpOnly cookie
        setRefreshCookie(response, refreshToken);

        // Return access token in body
        return AuthResponse.builder().token(accessToken).build();
    }

    @Override
    @Transactional
    public AuthResponse refresh(HttpServletRequest request,
            HttpServletResponse response) {

        String refreshToken = extractCookie(request, "refresh_token");
        if (refreshToken == null) {
            throw new AuthenticationCredentialsNotFoundException(
                "Refresh token not found");
        }

        // First — validate the JWT signature + expiry. If signature is bad
        // or token expired, extractEmail / isTokenValid will throw and we
        // bail out before any DB lookup.
        String email = jwtService.extractEmail(refreshToken, true);
        UserDetails userDetails = userDetailsService.loadUserByUsername(email);

        if (!jwtService.isTokenValid(refreshToken, userDetails, true)) {
            throw new BadCredentialsException("Invalid refresh token");
        }

        // Signature was valid. Now check if the token row exists in the DB.
        // If it doesn't, this is a legitimately-issued token that has
        // already been rotated (or never stored) — almost certainly a
        // **stolen token being replayed**. Revoke EVERY session for this
        // user, forcing re-login on every device.
        Optional<Session> maybeSession =
            sessionRepository.findByRefreshToken(refreshToken);

        if (maybeSession.isEmpty()) {
            User user = userRepository.findByEmail(email).orElseThrow();
            log.warn("Refresh token reuse detected for userId={} — revoking all sessions",
                user.getId());
            sessionRepository.deleteByUser(user);
            throw new BadCredentialsException(
                "Refresh token reuse detected — all sessions revoked");
        }

        Session session = maybeSession.get();

        // Delete old session
        sessionRepository.delete(session);

        // Create new tokens
        String newAccessToken = jwtService.generateAccessToken(email);
        String newRefreshToken = jwtService.generateRefreshToken(email);

        // Save new session (carry over provider from old session)
        User user = session.getUser();
        Session newSession = Session.builder()
            .refreshToken(newRefreshToken)
            .expiresAt(jwtService.getRefreshTokenExpiry())
            .provider(session.getProvider())
            .user(user)
            .build();
        sessionRepository.save(newSession);

        // Set new refresh cookie
        setRefreshCookie(response, newRefreshToken);

        // Return new access token in body
        return AuthResponse.builder().token(newAccessToken).build();
    }

    @Override
    @Transactional
    public void logout(HttpServletRequest request,
            HttpServletResponse response) {

        String refreshToken = extractCookie(request, "refresh_token");
        if (refreshToken != null) {
            sessionRepository.deleteByRefreshToken(refreshToken);
        }

        // Clear refresh cookie
        clearRefreshCookie(response);
    }

    private void setRefreshCookie(HttpServletResponse response, String token) {
        ResponseCookie cookie = ResponseCookie.from("refresh_token", token)
            .httpOnly(true)
            .secure(true)
            .path("/")
            .maxAge(jwtService.getRefreshExpirationSeconds())
            .sameSite("Lax")
            .build();
        response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());
    }

    private void clearRefreshCookie(HttpServletResponse response) {
        ResponseCookie cookie = ResponseCookie.from("refresh_token", "")
            .httpOnly(true)
            .secure(true)
            .path("/")
            .maxAge(0)
            .sameSite("Lax")
            .build();
        response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());
    }

    private String extractCookie(HttpServletRequest request, String name) {
        Cookie[] cookies = request.getCookies();
        if (cookies != null) {
            for (Cookie cookie : cookies) {
                if (name.equals(cookie.getName())) {
                    return cookie.getValue();
                }
            }
        }
        return null;
    }
}
```

> **Refresh token reuse detection (security-critical).** The "signature valid but no DB row" branch is the stolen-token detection mechanism described in [6.1 Access and Refresh Tokens](<6.1. Access and Refresh Tokens.md>). Because every successful `/refresh` deletes the old session row and saves a new one, a refresh token whose signature checks out but whose row is missing has already been rotated — the only ways that happens are (a) the legitimate user already used it and the attacker is replaying a stolen copy, or (b) the attacker used it first and the legitimate user is now replaying. Either way, both parties should be kicked out: cascade-delete every session for that user and force re-authentication on every device. The wide-blast response is intentional — it's how you punish reuse.


**JwtAuthFilter**: only `extractToken()` changes; the filter now checks the `Authorization` header only (no cookie fallback — that holds the *refresh* token now, handled separately in `AuthService.refresh()`):

```java
private String extractToken(HttpServletRequest request) {
    String authHeader = request.getHeader("Authorization");
    if (authHeader != null && authHeader.startsWith("Bearer ")) {
        return authHeader.substring(7);
    }
    return null;
}
```

## Q&A

1. **Why store the refresh token in a `Session` row instead of leaving it stateless?**
   A DB row makes the refresh token revocable — you can implement logout, logout-everywhere, admin force-logout, and per-device session listing. With a stateless refresh **JWT**, none of that is possible until expiry.

2. **How is the "max 3 sessions per user" limit enforced and why a `while` loop?**
   On login, `countByUser` is checked; if at limit, the oldest session is deleted via `findFirstByUserOrderByIdAsc`. A `while` loop (not `if`) safely handles drift if the limit was lowered after rows already existed — drains down to `max - 1` before saving the new one.

3. **Why use different secrets and different expiries for access vs refresh tokens?**
   Different secrets contain blast radius — leaking one doesn't compromise the other. Different expiries reflect their roles: access is sent on every request (short), refresh only on `/refresh` (long).

4. **Why does the updated `JwtAuthFilter` only check the `Authorization` header, no cookie fallback?**
   The cookie now holds the **refresh** token, not the access token. Letting the filter accept it would let a refresh token authenticate a normal API call, defeating the short-lived-access-token model.

5. **In `refresh()`, why delete the old session before saving the new one inside `@Transactional`?**
   Token rotation — the old refresh token must be unusable the moment the new one is issued. Wrapping in `@Transactional` ensures atomicity: either both happen or neither, preventing the user from being left tokenless on a partial failure.

6. **What does the `provider` column on `Session` capture and why is it on the session, not the user?**
   It records which auth path produced this session (`LOCAL`, `GOOGLE`, `GITHUB`). It lives on the session because one user may sign in through multiple providers across devices — each session is independent.
