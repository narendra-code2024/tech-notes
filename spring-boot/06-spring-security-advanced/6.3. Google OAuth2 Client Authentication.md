OAuth2 lets users **login with their existing accounts** (Google, GitHub, etc.) instead of creating a new username/password for your app. Your app never sees the user's Google password — Google handles authentication and gives your app the user's profile info.

---

## OAuth2 Flow

```
Client                          Your Server                  Google (Auth Server)
  │                                  │                                │
  │ 1. Click "Login with Google"     │                                │
  │  GET /oauth2/authorization/google│                                │
  ├─────────────────────────────────►│                                │
  │                                  │ 2. Connect with client_id      │
  │                                  ├───────────────────────────────►│
  │                                  │                                │
  │ 3. Redirect to Google login page │                                │
  │◄─────────────────────────────────┤                                │
  │                                  │                                │
  │ 4. User enters Google credentials + consent                       │
  ├──────────────────────────────────────────────────────────────────►│
  │                                  │                                │
  │                                  │ 5. Redirect back with          │
  │                                  │    state + auth_code           │
  │                                  │◄───────────────────────────────┤
  │                                  │                                │
  │                                  │ 6. Exchange auth_code          │
  │                                  │    for access token            │
  │                                  ├───────────────────────────────►│
  │                                  │                                │
  │                                  │ 7. Returns Google              │
  │                                  │    access token                │
  │                                  │◄───────────────────────────────┤
  │                                  │                                │
  │                                  │ 8. OAuth2SuccessHandler        │
  │                                  │    - Get user info from token  │
  │                                  │    - Create/find user in DB    │
  │                                  │    - Generate our AT + RT      │
  │                                  │    - Save session in DB        │
  │                                  │                                │
  │ 9. Redirect to frontend          │                                │
  │    with access token             │                                │
  │    + refresh cookie set          │                                │
  │◄─────────────────────────────────┤                                │
```

## Key Concepts

|Term|Meaning|
|---|---|
|**Client ID**|Your app's public identifier (registered with Google)|
|**Client Secret**|Your app's private key (never exposed to frontend)|
|**Auth Code**|Temporary code Google gives after user consents — exchanged for tokens server-side|
|**State**|Random value to prevent CSRF during OAuth flow|
|**Google Access Token**|Token from Google to access user's profile — your server uses this once, then discards|
|**Our Access Token**|JWT your server creates for the user — used for all API calls going forward|

> After the OAuth2 flow completes, the user is treated exactly like a regular user — same access token in body, same refresh token in HttpOnly cookie, same session in DB.

## Google Cloud Setup

1. Go to [Google Cloud Console](https://console.cloud.google.com/)
2. Create a new project (or select existing)
3. Navigate to **APIs & Services → Credentials**
4. Click **Create Credentials → OAuth Client ID**
5. Application type: **Web application**
6. Add Authorized redirect URI: `http://localhost:8080/login/oauth2/code/google`
7. Copy the **Client ID** and **Client Secret**

> The redirect URI must match exactly — Google will reject requests with mismatched URIs.

## Implementation

All code below extends the implementation in [6.2 User Sessions Management with JWT](<6.2. User Sessions Management with JWT.md>); deltas are explained inline.

**Dependencies:**

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-oauth2-client</artifactId>
</dependency>
```

**Application properties**: Spring Boot auto-configures the Google provider; you just supply `client-id` / `client-secret` / `scope`:

```yaml
# Google OAuth2
spring:
  security:
    oauth2:
      client:
        registration:
          google:
            client-id: your-client-id
            client-secret: your-client-secret
            scope: email,profile
```

**User entity**: only change is making `password` nullable for OAuth2-only users (they don't have one initially). Provider is tracked per session, not per user, so a single user can login via both email/password and Google:

```java
@Column(nullable = true)
private String password; // null for OAuth2-only users, BCrypt encoded for local users
```

**OAuth2SuccessHandler**: runs after Google authentication succeeds. Creates or finds the user in your DB, generates your access + refresh tokens, and redirects to the frontend:

```java
@Component
@RequiredArgsConstructor
public class OAuth2SuccessHandler extends
        SimpleUrlAuthenticationSuccessHandler {

    private final UserRepository userRepository;
    private final SessionRepository sessionRepository;
    private final JwtService jwtService;

    @Value("${jwt.session.max-limit}")
    private int maxSessionLimit;

    @Value("${app.frontend-url}")
    private String frontendUrl;

    @Override
    public void onAuthenticationSuccess(HttpServletRequest request,
            HttpServletResponse response,
            Authentication authentication) throws IOException {

        OAuth2User oAuth2User = (OAuth2User) authentication.getPrincipal();

        String email = oAuth2User.getAttribute("email");
        String firstName = oAuth2User.getAttribute("given_name");
        String lastName = oAuth2User.getAttribute("family_name");

        // Detect provider
        String registrationId = ((OAuth2AuthenticationToken) authentication)
            .getAuthorizedClientRegistrationId(); // "google" or "github"
        AuthProvider provider = AuthProvider.valueOf(
            registrationId.toUpperCase());

        // Create user if first time, or find existing
        User user = userRepository.findByEmail(email)
            .orElseGet(() -> {
                User newUser = User.builder()
                    .email(email)
                    .firstName(firstName)
                    .lastName(lastName)
                    .password(null) // No password for OAuth2 users
                    .roles(Set.of("ROLE_USER"))
                    .build();
                return userRepository.save(newUser);
            });

        // Enforce session limit
        while (sessionRepository.countByUser(user) >= maxSessionLimit) {
            sessionRepository.findFirstByUserOrderByIdAsc(user)
                .ifPresent(sessionRepository::delete);
        }

        // Generate our tokens
        String accessToken = jwtService.generateAccessToken(email);
        String refreshToken = jwtService.generateRefreshToken(email);

        // Save session in DB with provider
        Session session = Session.builder()
            .refreshToken(refreshToken)
            .expiresAt(jwtService.getRefreshTokenExpiry())
            .provider(provider)
            .user(user)
            .build();
        sessionRepository.save(session);

        // Set refresh token as HttpOnly cookie
        ResponseCookie cookie = ResponseCookie.from("refresh_token", refreshToken)
            .httpOnly(true).secure(true).path("/")
            .maxAge(jwtService.getRefreshExpirationSeconds())
            .sameSite("Lax").build();
        response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());

        // Redirect to frontend with access token
        String redirectUrl = frontendUrl + "/oauth2/callback?token=" + accessToken;
        getRedirectStrategy().sendRedirect(request, response, redirectUrl);
    }
}
```

> The refresh-cookie + session-row pattern above is the same one used in [6.2 User Sessions Management with JWT](<6.2. User Sessions Management with JWT.md>) → AuthService — only the user creation (auto-provision on first OAuth2 login) and the redirect step are OAuth2-specific.

> **Why put the access token in the URL?** After OAuth, the browser is on your backend's URL — the access token has to ride a redirect back to the frontend, and a URL query param is the only channel. The frontend reads it, stores it in memory, and immediately scrubs the URL. The brief exposure is acceptable because access tokens are short-lived (15 min). The refresh token rides the same redirect response as an `HttpOnly` cookie, never JS-visible.

**SecurityConfig**: add `.oauth2Login()` with the success handler. Other config from 6.2 is unchanged. `.oauth2Login()` tells Spring to handle the entire OAuth2 flow (redirects, auth-code exchange, token fetch) — you just configure what happens on success:

```java
http
    // ... csrf, sessionManagement, authorizeHttpRequests from 6.2 unchanged ...

    .oauth2Login(oauth2 -> oauth2
        .failureUrl("/login?error=true")
        .successHandler(oAuth2SuccessHandler)
    )

    .addFilterBefore(jwtAuthFilter,
        UsernamePasswordAuthenticationFilter.class);
```

**Frontend trigger**: link to Spring's auto-configured OAuth entry URL (no controller needed; Spring redirects the user to Google's login page):

```html
<a href="http://localhost:8080/oauth2/authorization/google">
    Login with Google
</a>
```

**Frontend callback handling**: on the `/oauth2/callback` page, read the access token from the URL, store it in memory, then scrub the URL:

```javascript
const params = new URLSearchParams(window.location.search);
const token = params.get("token");

if (token) {
    // Store access token in memory
    setAccessToken(token);

    // Clean URL (remove token from address bar)
    window.history.replaceState({}, "", "/oauth2/callback");

    // Redirect to home/dashboard
    navigate("/dashboard");
}
```

## Login Scenarios

| Scenario                                    | What Happens                                                     |
|---------------------------------------------|------------------------------------------------------------------|
| New Google user                             | User created with `provider = GOOGLE`, no password               |
| Existing Google user                         | Found by email, new session created                              |
| Email/password user tries Google login       | Found by email, session created — works (Yes)                    |
| Google-only user tries email/password login | BCrypt fails naturally (no valid hash stored) — returns 401 (OK) |

> The `provider` field is **informational**: it tracks how the user first signed up. It doesn't restrict login methods. A user who signed up with email/password can also login with Google, and vice versa (if they set a password later via a "Set Password" feature).

Just add their properties — Spring auto-configures them:

```yaml
# GitHub
spring:
  security:
    oauth2:
      client:
        registration:
          github:
            client-id: your-github-id
            client-secret: your-github-secret
            scope: user:email,read:user
```

```html
<a href="http://localhost:8080/oauth2/authorization/github">
    Login with GitHub
</a>
```

Then handle different attribute names in `OAuth2SuccessHandler` — Google uses `given_name` / `family_name`, GitHub uses `name` / `login`. Use a switch or strategy based on `registrationId` (already extracted in the handler).

## Q&A

1. **What problem does **OAuth2** solve compared to plain email/password?**
   It removes the need for your app to store user passwords — Google (or another **IdP**) authenticates the user and gives your app verified profile info. Better security (no password DB to leak), better UX (one-click login), and trust transfer to a known provider.

2. **What's the role of the `state` parameter in the OAuth2 flow?**
   It's a random per-request value sent on the redirect to Google and validated when Google redirects back. It prevents CSRF on the authorization endpoint — an attacker can't trick the user into completing an OAuth flow they didn't initiate.

3. **Why exchange the auth code for a token server-side instead of giving the token directly?**
   The auth code is single-use and short-lived; the actual token exchange uses your `client_secret`, which never leaves the server. If a token were sent to the browser directly, the secret would have to be exposed or omitted (less secure).

4. **What does `OAuth2SuccessHandler` do that `AuthService.login` doesn't?**
   It runs after Google authenticates — it has no password to validate. It extracts profile attributes from `OAuth2User`, creates the local user if first-time login, sets `provider` on the session, then issues your app's own access + refresh tokens.

5. **Why does the access token go in the redirect URL but the refresh token go in a cookie?**
   The browser is on your backend after OAuth — you must hand control to the frontend. URL is the only channel for that brief redirect; the frontend reads it, stores it in memory, and immediately scrubs the URL. The refresh token rides the redirect response as an `HttpOnly` cookie — never JS-visible.

6. **Why is `password` nullable on the `User` entity now?**
   OAuth2-only users never set a local password. A null hash also fails BCrypt verification cleanly if someone tries email/password login on a Google-only account — naturally returns 401 without special-casing.

7. **Where is `provider` (LOCAL/GOOGLE/GITHUB) stored and why there?**
   On `Session`, not `User`. One user can sign in via different providers across devices — each session records its own origin, useful for audit logs and per-session UI ("logged in via Google from Chrome").
