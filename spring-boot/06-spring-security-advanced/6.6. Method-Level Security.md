## Method-Level vs URL-Level Security

URL-level rules in `SecurityConfig` gate the request before it reaches the controller; method annotations run on the controller (or service) method itself, with access to method parameters and the authenticated principal.

||URL-Level (Request Matchers)|Method-Level (@PreAuthorize, @Secured)|
|---|---|---|
|Where|SecurityConfig|On any method (controller, service)|
|Granularity|Broad ‚Äî URL patterns|Fine-grained ‚Äî per method|
|Checks|Roles, authentication|Roles, permissions, custom logic, method parameters|
|Use for|"Only authenticated users can access `/api/**`"|"Only the wallet owner can view this wallet"|

> Use **request matchers** to secure endpoints broadly (authentication, role-based URL patterns). Use **method annotations** for specific business rules inside controllers or services.

---

## Enabling Method Security

Method annotations are **disabled by default**. Add this to your config:

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity(securedEnabled = true)
public class SecurityConfig {
    // ...
}
```

|Parameter|What It Enables|Default|
|---|---|---|
|`prePostEnabled`|`@PreAuthorize`, `@PostAuthorize`|`true` (enabled by default in `@EnableMethodSecurity`)|
|`securedEnabled`|`@Secured`|`false` (must enable explicitly)|

> `@PreAuthorize` works as soon as you add `@EnableMethodSecurity`. You only need `securedEnabled = true` if you also want to use `@Secured`.

## @Secured

Simple role-based check at method level. Only supports role names ‚Äî no expressions.

```java
@Secured("ROLE_USER")
@GetMapping("/profile")
public ResponseEntity<UserResponse> getProfile() {
    // only users with ROLE_USER can access
}

@Secured({"ROLE_ADMIN", "ROLE_MANAGER"})
@DeleteMapping("/posts/{id}")
public ResponseEntity<Void> deletePost(@PathVariable Long id) {
    // ADMIN or MANAGER can access
}
```

|Pros|Cons|
|---|---|
|Simple|Roles only ‚Äî no permissions, no expressions|
|Easy to read|Can't access method parameters|
||Can't use custom logic|

> Must pass the full `ROLE_` prefix ‚Äî `@Secured("ROLE_ADMIN")`, not `@Secured("ADMIN")`.

## @PreAuthorize

Expression-based check ‚Äî much more powerful than `@Secured`. Runs **before** the method executes.

**Check role:**

```java
@PreAuthorize("hasRole('ADMIN')")
@GetMapping("/admin/users")
public ResponseEntity<List<UserResponse>> getAllUsers() {
    // hasRole adds ROLE_ prefix automatically
}
```

**Check permission (authority):**

```java
@PreAuthorize("hasAuthority('POST_DELETE')")
@DeleteMapping("/posts/{id}")
public ResponseEntity<Void> deletePost(@PathVariable Long id) {
    // checks for POST_DELETE in authorities list
}
```

**Multiple conditions:**

```java
@PreAuthorize("hasRole('ADMIN') or hasRole('MANAGER')")

@PreAuthorize("hasAuthority('POST_CREATE') and hasAuthority('POST_UPDATE')")

@PreAuthorize("hasAnyRole('ADMIN', 'MANAGER')")

@PreAuthorize("hasAnyAuthority('POST_CREATE', 'POST_UPDATE')")
```

**Access method parameters**: `#userId` refers to the method parameter; `authentication.principal` is the current logged-in user:

```java
@PreAuthorize("#userId == authentication.principal.id")
@GetMapping("/users/{userId}")
public ResponseEntity<UserResponse> getUser(@PathVariable Long userId) {
    // user can only access their own profile
}
```

**Custom bean method**: for complex business logic, call a method on a Spring bean. `@postSecurity` resolves the bean by name; if `isOwner(#postId)` returns `false`, access is denied with 403:

```java
@PreAuthorize("@postSecurity.isOwner(#postId)")
@PutMapping("/posts/{postId}")
public ResponseEntity<PostResponse> updatePost(@PathVariable Long postId,
        @RequestBody PostRequest request) {
    // only the post owner can update
}
```

```java
@Component("postSecurity")
@RequiredArgsConstructor
public class PostSecurity {

    private final PostRepository postRepository;

    public boolean isOwner(Long postId) {
        Authentication auth = SecurityContextHolder.getContext()
            .getAuthentication();
        User user = (User) auth.getPrincipal();

        return postRepository.findById(postId)
            .map(post -> post.getUser().getId().equals(user.getId()))
            .orElse(false);
    }
}
```

## @Secured vs @PreAuthorize

| Feature                      | @Secured              | @PreAuthorize             |
|------------------------------|-----------------------|---------------------------|
| Role check                   | Yes                   | Yes                       |
| Permission check             | No                    | Yes                       |
| SpEL expressions             | No                    | Yes                       |
| Access method params         | No                    | Yes (`#paramName`)        |
| Custom bean method           | No                    | Yes (`@beanName.method()`)|
| Multiple conditions (AND/OR) | No                    | Yes                       |
| Must include `ROLE_` prefix  | Yes                   | No (`hasRole` adds it)    |
| Needs explicit enable        | `securedEnabled = true`| Enabled by default        |

> **Use `@PreAuthorize` in most cases**: it does everything `@Secured` does and more. `@Secured` exists mainly for legacy compatibility.

## 401 Unauthorized vs 403 Forbidden

||401 Unauthorized|403 Forbidden|
|---|---|---|
|Means|"I don't know who you are"|"I know who you are, but you can't do this"|
|Cause|Missing or invalid token|Valid token but insufficient permissions|
|When|No token, expired token, bad signature|User lacks required role or permission|
|Fix|Login again, refresh token|Need different role/permission|

## Layered Security ‚Äî Putting It All Together

```java
// SecurityConfig ‚Äî URL level
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/auth/**").permitAll()
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .anyRequest().authenticated()   // must be logged in
)
```

```java
// Controller ‚Äî method level
@PreAuthorize("hasAuthority('USER_VIEW')")
@GetMapping("/admin/users")
public ResponseEntity<List<UserResponse>> getAllUsers() {
    return ResponseEntity.ok(userService.getAllUsers());
}
```

```java
// Service ‚Äî business logic level
@PreAuthorize("@postSecurity.isOwner(#postId)")
public PostResponse updatePost(Long postId, PostRequest request) {
    // ...
}
```

> Each layer narrows the check: URL gating ("authenticated, right role for this URL?") ‚Üí method security ("has this specific permission?") ‚Üí business logic ("owner of this resource?"). A request must pass **all layers** to succeed.

## Q&A

1. **When should you prefer method annotations over `requestMatchers`?**
   When the rule depends on method parameters (`#userId == authentication.principal.id`), domain ownership (`@postSecurity.isOwner(#postId)`), or applies to a service-layer method that may be called from multiple controllers. URL matchers can't see inside the method.

2. **`@PreAuthorize` vs `@PostAuthorize` ‚Äî what's the difference and when use each?**
   `@PreAuthorize` runs *before* the method, so it can't inspect the return value. `@PostAuthorize` runs *after* and can reference `returnObject` ‚Äî useful for "fetch then check ownership" patterns where the resource ID isn't a method parameter.

3. **Why is `@Secured` considered legacy compared to `@PreAuthorize`?**
   `@Secured` only accepts role names (with `ROLE_` prefix), no expressions, no parameter access, no SpEL. `@PreAuthorize` supports all of that plus custom bean calls ‚Äî a strict superset, so `@Secured` mostly survives for backward compatibility.

4. **How does `@PreAuthorize("@postSecurity.isOwner(#postId)")` actually work?**
   Spring resolves `@postSecurity` as a bean by name and invokes `isOwner` with the value bound to the `postId` parameter. The expression must return boolean ‚Äî `false` triggers `AccessDeniedException` (403).

5. **Do `@PreAuthorize` annotations work on `private` methods or self-invocations?**
   No. Spring Security uses **AOP** proxies ‚Äî only externally-called public methods on Spring-managed beans are intercepted. Calling a `@PreAuthorize` method from within the same class bypasses the proxy and the check.

6. **Is method security enabled by default in Spring Boot?**
   No. You must add `@EnableMethodSecurity`. `@PreAuthorize` is enabled implicitly with that annotation; `@Secured` requires `@EnableMethodSecurity(securedEnabled = true)` explicitly.

7. **A request fails with 403 from `@PreAuthorize` ‚Äî should the user re-login?**
   No. 403 means authenticated but lacking permission ‚Äî re-login won't add roles they don't have. 401 (missing/invalid token) is the case where re-login helps.

## Quick Reference

|Concept|Remember|
|---|---|
| Enable | `@EnableMethodSecurity` (PreAuthorize on by default; `securedEnabled=true` for `@Secured`) |
| `@Secured("ROLE_X")` | Role-only, must include `ROLE_` prefix |
| `@PreAuthorize("hasRole('X')")` | SpEL; auto-prefixes `ROLE_` |
| `@PreAuthorize("hasAuthority('X')")` | Permission/authority, exact match |
| `#paramName` in SpEL | References method parameter |
| `@beanName.method()` | Calls custom Spring bean |
| `authentication.principal` | Current logged-in user |
| 401 vs 403 | 401 = no/invalid token; 403 = valid token, wrong role/permission |
| AOP caveat | Self-invocation skips the proxy ‚Äî annotations have no effect |
| When to use method security | Param-aware checks, service-layer rules, ownership |
