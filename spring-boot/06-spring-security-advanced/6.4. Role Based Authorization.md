## Roles vs Authorities

In Spring Security, **roles and authorities are the same thing**: both are strings stored in the `GrantedAuthority` list. The only difference is a naming convention:

||Authority|Role|
|---|---|---|
|Stored as|`ROLE_USER`, `ROLE_ADMIN`|Same â€” `ROLE_USER`, `ROLE_ADMIN`|
|Checked with|`hasAuthority("ROLE_USER")`|`hasRole("USER")`|
|Prefix|You include `ROLE_` yourself|Spring adds `ROLE_` automatically|

> `hasRole("ADMIN")` is just shorthand for `hasAuthority("ROLE_ADMIN")`. Spring automatically prepends `ROLE_` when using `hasRole()`.

```java
// These two are identical:
.hasAuthority("ROLE_ADMIN")
.hasRole("ADMIN")          // Spring checks for "ROLE_ADMIN" internally
```

**Important:** When storing roles in the database, always include the `ROLE_` prefix. Spring expects it.

```
DB stores: ROLE_USER, ROLE_ADMIN
hasRole("USER")      â†’ checks for ROLE_USER (valid)
hasRole("ROLE_USER") â†’ checks for ROLE_ROLE_USER (invalid - double prefix)
hasAuthority("ROLE_USER") â†’ checks for ROLE_USER (valid)
```

## Implementation

All code below extends the implementation in [6.3 Google OAuth2 Client Authentication](<6.3. Google OAuth2 Client Authentication.md>); deltas are explained inline.

**Role enum**: stored as `USER` / `ADMIN` in DB; the `ROLE_` prefix is added in `getAuthorities()` (Spring expects it but we don't pollute our enum):

```java
public enum Role {
    USER, ADMIN, MANAGER
}
```

**User entity**: instead of converting `User` to `UserDetails` in `CustomUserDetailsService`, `User` implements `UserDetails` directly. `getAuthorities()` maps `Role.ADMIN` â†’ `"ROLE_ADMIN"` so the enum stays clean and Spring gets the prefix it expects:

```java
@Entity
@Table(name = "users")
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class User implements UserDetails {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String firstName;

    @Column(nullable = false)
    private String lastName;

    @Column(nullable = false, unique = true)
    private String email;

    @Column(nullable = true)
    private String password;

    @ElementCollection(fetch = FetchType.EAGER)
    @CollectionTable(name = "user_role",
        joinColumns = @JoinColumn(name = "user_id"))
    @Enumerated(EnumType.STRING)
    @Column(name = "role")
    private Set<Role> roles;

    @Override
    public Collection<? extends GrantedAuthority> getAuthorities() {
        return roles.stream()
            .map(role -> new SimpleGrantedAuthority("ROLE_" + role.name()))
            .collect(Collectors.toList());
    }

    @Override
    public String getUsername() {
        return email;
    }
}
```

> **Migration note from 6.3:** `OAuth2SuccessHandler` in [6.3 Google OAuth2 Client Authentication](<6.3. Google OAuth2 Client Authentication.md>) uses `Set.of("ROLE_USER")` because `User.roles` was `Set<String>` there. With this `Role` enum migration, change those references to `Set.of(Role.USER)`. The `ROLE_` prefix is now added by `getAuthorities()`, not stored.

**CustomUserDetailsService**: simplified; `User` is already a `UserDetails`, no manual conversion needed:

```java
@Service
@RequiredArgsConstructor
public class CustomUserDetailsService implements UserDetailsService {

    private final UserRepository userRepository;

    @Override
    public UserDetails loadUserByUsername(String email)
            throws UsernameNotFoundException {
        return userRepository.findByEmail(email)
            .orElseThrow(() ->
                new UsernameNotFoundException("Invalid credentials"));
    }
}
```

## SecurityConfig â€” Authorization Rules

**Common methods:**

|Method|Purpose|Example|
|---|---|---|
|`permitAll()`|Anyone (no auth needed)|Public endpoints, login, register|
|`authenticated()`|Any logged-in user|General protected endpoints|
|`hasRole("ADMIN")`|User must have `ROLE_ADMIN`|Admin panel|
|`hasAnyRole("ADMIN", "MANAGER")`|User must have any of these roles|Management endpoints|
|`hasAuthority("ROLE_ADMIN")`|Same as `hasRole("ADMIN")`|Explicit prefix version|
|`hasAnyAuthority(...)`|Same as `hasAnyRole()` with prefix|Explicit prefix version|
|`denyAll()`|Block everyone|Disabled endpoints|

**URL patterns:**

|Pattern|Matches|
|---|---|
|`/api/admin`|Exactly `/api/admin`|
|`/api/admin/*`|One level: `/api/admin/users`, `/api/admin/settings`|
|`/api/admin/**`|All levels: `/api/admin/users`, `/api/admin/users/1/roles`|

**Updated config**: replace the `authorizeHttpRequests` block with role-based rules. Spring evaluates rules top-to-bottom (first match wins), so put specific rules before `.anyRequest().authenticated()`:

```java
.authorizeHttpRequests(auth -> auth
    // Public
    .requestMatchers("/api/auth/register").permitAll()
    .requestMatchers("/api/auth/login").permitAll()
    .requestMatchers("/api/auth/refresh").permitAll()

    // Role-based
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/management/**")
        .hasAnyRole("ADMIN", "MANAGER")

    // Method-specific
    .requestMatchers(HttpMethod.GET, "/api/posts/**").permitAll()
    .requestMatchers(HttpMethod.POST, "/api/posts/**")
        .hasRole("USER")
    .requestMatchers(HttpMethod.DELETE, "/api/posts/**")
        .hasRole("ADMIN")

    // Catch-all
    .anyRequest().authenticated()
)
```

## Assigning Roles

**During registration**: set the default role on the new user:

```java
User user = User.builder()
    .email(request.getEmail())
    .password(passwordEncoder.encode(request.getPassword()))
    .roles(Set.of(Role.USER))  // Default role
    .build();
```

**Admin promoting a user**: add the role to an existing user:

```java
public void assignRole(Long userId, Role role) {
    User user = userRepository.findById(userId).orElseThrow();
    user.getRoles().add(role); // e.g., Role.ADMIN
    userRepository.save(user);
}
```

> Role changes take effect on the **next token refresh**: the current access token still has the old roles encoded. When the user refreshes, `loadUserByUsername()` fetches updated roles from DB.

## Q&A

1. **Are roles and authorities different in Spring Security?**
   No â€” both are `GrantedAuthority` strings. The convention is roles carry the `ROLE_` prefix; `hasRole("ADMIN")` is just shorthand for `hasAuthority("ROLE_ADMIN")`. Spring auto-prepends `ROLE_` only inside `hasRole`/`hasAnyRole`.

2. **Why does `hasRole("ROLE_USER")` fail (or look for `ROLE_ROLE_USER`)?**
   `hasRole` adds the prefix automatically. Passing it pre-prefixed produces a double prefix. Either use `hasRole("USER")` or switch to `hasAuthority("ROLE_USER")`.

3. **What's the difference between 401 and 403 in this flow?**
   401 means unauthenticated (no/invalid token); 403 means authenticated but lacking the required role/authority. **RBAC** rule failures produce 403, not 401.

4. **Why have `User implements UserDetails` instead of a separate adapter?**
   Removes the conversion step in `CustomUserDetailsService` and lets `@AuthenticationPrincipal User user` inject your domain entity directly into controllers â€” cleaner, fewer DTOs to maintain.

5. **A user's role was just changed in DB â€” do their existing requests see the new role?**
   Not until the access token is reissued. Each request authenticates via the token; only on the next refresh does `loadUserByUsername` re-fetch fresh roles. Short access-token lifetimes (15 min) limit the staleness window.

6. **Why does request-matcher order matter in `authorizeHttpRequests`?**
   First match wins. `.anyRequest().authenticated()` must come last; a permissive rule placed above a restrictive one will silently shadow it.

## Quick Reference

|Concept|Remember|
|---|---|
| Role storage | Stored as `USER`/`ADMIN` enum; `ROLE_` prefix added in `getAuthorities()` |
