## The Single-Token Problem

Current setup — login creates a single **JWT** with 24h expiry that the client sends on every request. This works, but has two problems:

- **Long-lived token = bigger risk.** A token with 24h expiry is sent with every request. If stolen (via network interception, logs, XSS), the attacker has 24 hours to use it freely. The server can't revoke it — JWT is stateless, there's no "invalidate" button.
- **Short-lived token = bad UX.** If you shorten expiry to 15 minutes for safety, the user gets logged out every 15 minutes and has to re-enter credentials. Terrible experience.

```
Long expiry  → Good UX, Bad security
Short expiry → Good security, Bad UX
```

You can't have both with a single token.

---

## The Two-Token Solution

Instead of one token trying to do everything, split the responsibility:

|Token|Purpose|Lifetime|Sent With|
|---|---|---|---|
|**Access Token**|Authenticate API requests|Short (15–30 min)|Every request (`Authorization: Bearer`)|
|**Refresh Token**|Get a new access token|Long (7–30 days)|Only sent to `/api/auth/refresh`|

```
Access token stolen?
  → Attacker has 15 minutes, not 24 hours
  → Damage is limited

User's access token expired?
  → Frontend silently calls /refresh
  → Gets a new access token
  → User never notices
```

**Short-lived access token = security. Long-lived refresh token = UX. Best of both.**: -

## The Two-Token Lifecycle

**Login:**

```
Client: POST /api/auth/login { email, password }
         │
         ▼
Server validates credentials
         │
         ▼
Server creates two tokens:
  ├── Access Token  (JWT, 15 min, returned in JSON body)
  └── Refresh Token (JWT, 7 days, HttpOnly cookie)
         │
         ▼
Client stores access token in memory (SPA) or env variable (Postman)
Browser stores refresh cookie automatically
```

**API calls:**

```
Client: GET /api/posts
        Authorization: Bearer <access_token>
         │
         ▼
JwtAuthFilter validates access token (from header only)
         │
         ▼
Valid → Request proceeds to controller
```

**Token refresh:**

```
Client: POST /api/auth/refresh
        Cookie: refresh_token=eyJ... (sent automatically by browser)
         │
         ▼
Server validates refresh token against DB
         │
         ▼
Valid → Returns new access token in JSON body
        Sets new refresh cookie
         │
         ▼
Client stores new access token in memory
Client continues making API calls
```

**Logout:**

```
Client: POST /api/auth/logout
         │
         ▼
Server invalidates refresh token
Server clears refresh cookie
         │
         ▼
Access token will expire on its own (15 min max)
```

## Refresh Token Storage

### Stateless

```
Refresh token = another JWT with longer expiry
Stored only in HttpOnly cookie
Server doesn't store anything
```

|Pros|Cons|
|---|---|
|Simple — no DB table needed|Can't revoke a specific refresh token|
|No DB lookup on refresh|Can't do "logout from all devices"|
|Truly stateless|If stolen, valid until it expires|
|Fast — no DB hit|No way to track active sessions|

> **This is almost the same as a single JWT**: you've just split one long-lived token into two. The access token window is shorter (15 min vs 24h), so a stolen access token is less dangerous. But the refresh token itself has the same weakness as a single long-lived JWT — if stolen, it's valid until expiry and you can't revoke it. The real security jump comes with DB-backed approaches below.

### DB-Backed

```
Refresh token = random string or JWT
Stored in a Session/RefreshToken table in DB
Server validates by checking DB
```

|Pros|Cons|
|---|---|
|Can revoke individual tokens|DB lookup on every refresh|
|Can invalidate all sessions (logout everywhere)|Not purely stateless|
|Can detect token reuse (security)|Extra table and cleanup logic|
|Track active sessions per user|Slightly more complex|

### DB-Backed + Rotation

```
Same as DB-Backed, but:
  → Every /refresh call returns a NEW refresh token
  → Old refresh token is invalidated immediately
  → If old token is reused → possible theft → invalidate ALL tokens for that user
```

|Pros|Cons|
|---|---|
|All benefits of DB-Backed|Most complex to implement|
|Detects stolen refresh tokens|Requires careful concurrency handling|
|Most secure approach||
|Industry standard (OAuth2 spec recommends this)||

**How token reuse detection works:**

```
Normal flow:
  Refresh Token A → used → returns new Access Token + Refresh Token B
  Refresh Token B → used → returns new Access Token + Refresh Token C
  (Token A and B are now invalid)

Theft scenario:
  Attacker stole Token A
  Legitimate user uses Token A → gets Token B (A invalidated)
  Attacker tries Token A → already used!
  Server detects reuse → invalidates ALL tokens for this user
  Both attacker and user must re-login
```

---

## Isn't This Just Sessions?

Similar idea (server stores something), but very different performance:

|                              | Session-Based Auth   | DB-Backed Refresh Token       |
|------------------------------|----------------------|-------------------------------|
| DB hit on **every API call** | Yes                  | No (JWT signature only)       |
| DB hit on **refresh**        | Every request        | Only every 15-30 min          |
| What's stored in DB          | Full session data    | Just the refresh token string |
| Stateless API calls?        | No (Never)           | Yes (99% of requests)         |

> With sessions, **every single request** hits the DB. With refresh tokens, API calls are fully stateless — the DB is only touched during login, refresh (every 15-30 min), and logout. **99% stateless, 1% stateful.**: -

## Which Approach to Pick?

| Feature                   | Single JWT         | Stateless Refresh                    | DB-Backed Refresh | DB + Rotation       |
|---------------------------|--------------------|--------------------------------------|-------------------|---------------------|
| Implementation complexity | Simple             | Simple (marginal change)             | Moderate          | Complex             |
| DB required for auth      | No                 | No                                   | Yes               | Yes                 |
| Revoke specific token     | No                 | No                                   | Yes               | Yes                 |
| Logout from all devices   | No                 | No                                   | Yes               | Yes                 |
| Detect token theft        | No                 | No                                   | No                | Yes                 |
| Stolen token risk         | Full expiry window | Shorter for access, same for refresh | Can revoke        | Can revoke + detect |
| Scalability               | Best               | Best                                 | Good              | Good                |
| Security level            | Low                | Low–Medium                           | High              | Highest             |

```
Side project / learning → Single JWT or Stateless Refresh
Production app → DB-Backed Refresh
Sensitive app (banking, health) → DB-Backed + Token Rotation
```

What most production apps actually use (industry standard):

- **Rotation on refresh**: old refresh token invalidated, new one issued (per DB-Backed + Rotation above)
- **Refresh cookie attributes: `HttpOnly + Secure + SameSite=Lax`**: XSS-safe, CSRF-safe, HTTPS-only
- **Access token stays stateless**: signature-validated only, never DB-checked
- **No separate CSRF token needed**: access in header (not cookie), refresh cookie protected by SameSite

> For detailed comparison of all storage options (localStorage, HttpOnly cookie, in-memory, Bearer header) and their XSS/CSRF trade-offs, see [5.1 Security Concepts and Attacks](<../05-spring-security-fundamentals/5.1. Security Concepts and Attacks.md>) (CSRF section) and [5.6 Understanding JWT](<../05-spring-security-fundamentals/5.6. Understanding JWT.md>) (JWT Storage).

## SPA Integration — Silent Refresh

The "silent refresh" pattern — full flow:

```
App loads (first visit or browser refresh F5)
       │
       ▼
Access token in memory? ── No ──→ Call /api/auth/refresh
       │                                    │
      Yes                          Cookie sent automatically
       │                                    │
       ▼                                    ▼
Make API call                    Server validates refresh token
       │                                    │
       ▼                                    ▼
401 Unauthorized?                Returns new access token
       │                                    │
      Yes                           Store in memory
       │                                    │
       ▼                                    ▼
Call /api/auth/refresh           Retry original API call
       │
       ▼
Refresh also fails (401)?
       │
      Yes → Redirect to login page (session fully expired)
```

> The user never sees a login screen unless the refresh token itself has expired (after 7-30 days of inactivity).

## Q&A

1. **Why use refresh tokens at all instead of just a long-lived access token?**
   A long-lived **JWT** is stateless and can't be revoked — if stolen it's valid until expiry. Splitting into a short-lived access token (15 min) + long-lived refresh token gives you a small attack window for the token sent on every request, while the refresh token (sent only to `/refresh`) can be DB-backed and revocable.

2. **What's the lifetime trade-off between access and refresh tokens?**
   Access token: minutes (15–30) — short enough to limit damage if stolen, long enough to avoid hitting `/refresh` constantly. Refresh token: days–weeks (7–30) — long enough that users don't re-login often, but inactivity-based so dormant accounts auto-expire.

3. **Why store the refresh token in an `HttpOnly` cookie but the access token in memory?**
   `HttpOnly` cookies are immune to XSS exfiltration (JS can't read them) and the browser auto-attaches them only to your origin — perfect for the rarely-used refresh token. Access tokens go in memory because they're sent via the `Authorization` header on every request, which JS must construct.

4. **What is refresh token rotation and what attack does it detect?**
   Each `/refresh` call invalidates the old refresh token and issues a new one. If the old token is ever presented again, you have proof of theft (legitimate client already moved on) — server invalidates all tokens for that user, forcing both attacker and user to re-login.

5. **Is DB-backed refresh-token auth the same as session-based auth?**
   No. Sessions hit the DB on **every** API call; DB-backed refresh tokens hit the DB only on login, refresh (every 15–30 min), and logout. API calls remain stateless via signature verification — roughly 99% stateless, 1% stateful.

6. **Why must `/api/auth/refresh` be `permitAll()`?**
   By definition the access token is expired when the client calls `/refresh` — the request can't authenticate via the normal filter. The refresh token in the cookie is validated manually inside the service.

7. **Why use different signing secrets for access and refresh tokens?**
   Defense in depth — if the access-token secret leaks (e.g., from a misconfigured service), attackers can mint access tokens but can't forge refresh tokens to maintain long-term access.

## Quick Reference

|Concept|Remember|
|---|---|
| Access token | Short-lived (15–30 min), JSON body, memory, `Authorization: Bearer` |
| Refresh token | Long-lived (7–30 days), `HttpOnly + Secure + SameSite=Lax` cookie, sent only to `/refresh` |
| Stateless approach | Refresh token is just a JWT — simple but unrevocable |
| DB-backed approach | Refresh token row in `Session` table — revocable, supports logout-everywhere |
| Rotation | Each `/refresh` invalidates old token, issues new one — detects theft via reuse |
| `/api/auth/refresh` | Must be `permitAll()` — access token is expired by definition |
| Storage rule | Never put access token in `localStorage` (XSS leak); never put refresh token in JS-readable storage |
| Different secrets | Access vs refresh — limits blast radius of secret leak |
