**Spring Boot Actuator** exposes production-grade introspection endpoints for a running app — health checks, metrics, bean list, environment properties, thread dumps, and live log-level control — without you writing any controllers. Load balancers, Kubernetes, Prometheus, and ops dashboards typically scrape these endpoints.

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

Default base path is `/actuator`; only `/actuator/health` is exposed over the web layer (HTTP/HTTPS) by default — everything else must be opted in. Whether that web layer is HTTP or HTTPS depends on your TLS config (`server.ssl.*`, or `management.server.ssl.*` if you run a separate management port).

---

## Key Endpoints

| Endpoint | URL | Description |
|---|---|---|
| `health` | `/actuator/health` | Overall status aggregated from `HealthIndicator` beans (with `/health/liveness` + `/health/readiness` groups for k8s probes) |
| `info` | `/actuator/info` | App metadata (version, git, build info) — configure via `info.*` properties |
| `metrics` | `/actuator/metrics` | Counters/timers/gauges exposed through Micrometer |
| `loggers` | `/actuator/loggers` | View and change log levels at runtime (POST to change) — see [4.4 Logging](<4.4. Logging.md>) |
| `env` | `/actuator/env` | Spring `Environment` property sources |
| `beans` | `/actuator/beans` | Every bean in the `ApplicationContext` |
| `mappings` | `/actuator/mappings` | All `@RequestMapping` routes |
| `threaddump` | `/actuator/threaddump` | Full thread dump |
| `heapdump` | `/actuator/heapdump` | Downloads a heap dump (`.hprof`) |

## Exposing Endpoints

```yaml
management:
  endpoints:
    web:
      exposure:
        # Expose specific endpoints (recommended)
        include: health,info,metrics
        # Expose everything (dev only — leaks internals)
        # include: "*"
      # Change base path from /actuator to /management
      base-path: /management
  endpoint:
    health:
      # Show full health details (auth required in production)
      show-details: when-authorized
  server:
    # Run management endpoints on a separate port (firewall off public access)
    port: 9090
    address: 127.0.0.1
```

> **Enabled vs exposed — two different switches.** `management.endpoint.<name>.enabled` controls whether the endpoint *exists in the application context*; `management.endpoints.web.exposure.include` controls whether HTTP can reach an existing endpoint. Disabling beats hiding — a disabled endpoint isn't created at all and can't leak via JMX either.

> **Production warning:** never expose `*` in production — `env`, `beans`, `configprops`, and `heapdump` leak configuration, credentials (if not masked), and full memory state. Always lock down Actuator before shipping (see `Securing Actuator` below) and expose only what your infra needs.

## Liveness and Readiness

Spring Boot 2.3+ splits health into two probe groups for Kubernetes / cloud orchestrators — same machinery as `/actuator/health`, but with different semantics:

| Probe | Endpoint | Means | Failure action |
|---|---|---|---|
| **Liveness** | `/actuator/health/liveness` | Is the JVM alive and not deadlocked? | k8s **restarts** the pod |
| **Readiness** | `/actuator/health/readiness` | Can the app accept traffic? (DB up, caches warm, dependencies reachable) | k8s **removes** the pod from service |

Auto-enabled when running on Kubernetes (detected via env vars). Outside k8s, opt in:

```yaml
management:
  endpoint:
    health:
      probes:
        enabled: true
```

Route specific indicators into a group:

```yaml
management:
  endpoint:
    health:
      group:
        readiness:
          include: readinessState,db,paymentGateway
```

> **Why split?** Liveness fail tells the orchestrator "this JVM is broken — kill and replace." Readiness fail says "JVM's fine, just don't send me traffic right now (warming up, draining, dependency down)." Restarting on a readiness blip would amplify outages — that's the whole reason the two are separate.

## Securing Actuator

### Network layer — separate management port

Run Actuator on a non-public port bound to an internal interface. The public load balancer can't reach it; Prometheus and k8s probes (inside the cluster) can:

```yaml
management:
  server:
    port: 9090
    address: 10.0.0.0     # internal interface only
```

### Application layer — `SecurityFilterChain` on `/actuator/**`

Custom `SecurityFilterChain` matched to `/actuator/**` so business endpoints stay unaffected. Expose liveness/readiness probes anonymously — k8s can't authenticate:

```java
@Bean
public SecurityFilterChain actuatorChain(HttpSecurity http) throws Exception {
    return http.securityMatcher("/actuator/**")
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/actuator/health/liveness",
                             "/actuator/health/readiness").permitAll()
            .requestMatchers("/actuator/info").permitAll()
            .anyRequest().hasRole("ACTUATOR_ADMIN"))
        .httpBasic(Customizer.withDefaults())
        .csrf(csrf -> csrf.disable())
        .build();
}
```

Pair with credentials in `application.yml` (or a real `UserDetailsService` for multi-user setups):

```yaml
spring:
  security:
    user:
      name: actuator
      password: ${ACTUATOR_PASSWORD}
      roles: ACTUATOR_ADMIN
```

> Always source the password from an env var or secret manager (`${ACTUATOR_PASSWORD}`) — never commit it.

## Custom Health Indicator

Plug business-specific health checks into `/actuator/health` by implementing `HealthIndicator`:

```java
@Component
public class PaymentGatewayHealthIndicator implements HealthIndicator {

    private final PaymentClient paymentClient;

    public PaymentGatewayHealthIndicator(PaymentClient paymentClient) {
        this.paymentClient = paymentClient;
    }

    @Override
    public Health health() {
        try {
            paymentClient.ping();
            return Health.up().withDetail("gateway", "reachable").build();
        } catch (Exception e) {
            return Health.down(e).withDetail("gateway", "unreachable").build();
        }
    }
}
```

The overall `/actuator/health` status is `DOWN` if any indicator reports `DOWN`.

## Custom Info Contributor

Plug app-specific data into `/actuator/info` by implementing `InfoContributor` (parallel to `HealthIndicator`):

```java
@Component
public class GitCommitInfoContributor implements InfoContributor {
    @Override
    public void contribute(Info.Builder builder) {
        builder.withDetail("commit", System.getenv("GIT_SHA"));
    }
}
```

Auto-contributors when on the classpath:

- **`info.app.*` properties** — anything under `info.*` in `application.yml` is exposed (e.g., `info.app.name: Order Service`)
- **`git-commit-id-plugin`**: populates `info.git.*` (commit hash, branch, tag) at build time
- **`spring-boot-maven-plugin` `build-info` goal**: populates `info.build.*` (artifact, version, build time)

## Custom Metrics with Micrometer

Actuator wires in **Micrometer** as the metrics facade — inject a `MeterRegistry` and record counters/timers:

```java
@Service
public class OrderService {

    private final Counter ordersPlaced;

    public OrderService(MeterRegistry registry) {
        this.ordersPlaced = registry.counter("orders.placed");
    }

    public void placeOrder(Order order) {
        // ... business logic ...
        ordersPlaced.increment();
    }
}
```

Scraped by Prometheus via `/actuator/prometheus` when `micrometer-registry-prometheus` is on the classpath.

## Q&A

1. **What does Actuator give you for free?** Production-ready endpoints — `health`, `metrics`, `info`, `loggers`, `env`, `beans`, `threaddump`, `heapdump` — without writing controllers. Plus Micrometer-based metrics that integrate with Prometheus/Datadog/etc.

2. **Why is exposing `*` dangerous in production?** Endpoints like `env`, `configprops`, `beans`, and `heapdump` reveal configuration, credentials, and full JVM memory. Always expose only the minimum set and secure them with Spring Security.

3. **How do you add a custom health check?** Implement `HealthIndicator` and register it as a bean. It's aggregated automatically into `/actuator/health` and influences the overall status.

4. **How can you change a log level without restarting?** POST to `/actuator/loggers/{logger-name}` with `{"configuredLevel": "DEBUG"}`. The change is live and persists until the next restart.

5. **What does `management.endpoint.health.show-details` do?** Controls whether `/actuator/health` returns per-indicator details. Options are `never`, `when-authorized`, or `always` — use `when-authorized` in production so only authenticated ops users see sub-check details.

6. **How does Actuator integrate with Prometheus?** Add `micrometer-registry-prometheus`; Actuator exposes `/actuator/prometheus` in Prometheus text format, which the Prometheus server scrapes on an interval.

7. **Liveness vs Readiness — what's the difference and how does k8s use them?** **Liveness** answers "is the JVM alive?" — failure causes the orchestrator to **restart** the pod. **Readiness** answers "can it serve traffic right now?" — failure causes the pod to be **removed from the service** (no restart). Splitting them avoids restarting a healthy JVM that's just temporarily not ready (warming up, dependency blip).

8. **What's the difference between *enabling* and *exposing* an endpoint?** `management.endpoint.<name>.enabled` controls whether the endpoint bean is created at all. `management.endpoints.web.exposure.include` controls whether HTTP traffic can reach an enabled endpoint. Disabled endpoints can't leak — even via JMX — so disable rather than hide whenever you don't need them.

9. **How do you secure Actuator endpoints in production?** Two complementary layers, typically combined as defense in depth: (1) **Network layer**: bind `management.server.port` to an internal interface so the public load balancer can't reach it. (2) **Application layer**: custom `SecurityFilterChain` matched to `/actuator/**` with Basic auth, exposing `/health/liveness` + `/health/readiness` anonymously for k8s probes. Most public-API deployments use both; either alone is sufficient in some topologies. (Adding `spring-boot-starter-security` with default user credentials secures the whole app's HTTP surface — not Actuator-specific.)
