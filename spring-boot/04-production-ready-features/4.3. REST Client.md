`RestClient` is a **synchronous**, blocking HTTP client introduced in Spring Framework 6.1 / Spring Boot 3.2. It exposes a modern fluent API, abstracts over underlying HTTP libraries (Apache HttpClient, JDK HttpClient, Reactor Netty, etc.), and converts Java objects to/from HTTP messages.

### Evolution of HTTP clients in Spring

| Client | Type | Status |
|---|---|---|
| `RestTemplate` | Synchronous, blocking | Legacy — still works but no new features |
| `WebClient` | Asynchronous, non-blocking (Project Reactor) | Use for reactive apps or non-blocking I/O |
| `RestClient` | Synchronous, blocking | **Recommended**: modern replacement for `RestTemplate` |

Pick `RestClient` for standard synchronous REST calls; pick `WebClient` only when the app is reactive or genuinely needs non-blocking behaviour.

## Configuring the Bean

Create one `RestClient` per external service so base URL, auth, and default headers are kept separate.

```properties
employee-service.base-url=http://localhost:8081/api
employee-service.username=admin
employee-service.password=secret
```

```java
@ConfigurationProperties(prefix = "employee-service")
public class EmployeeServiceProperties {
    private String baseUrl;
    private String username;
    private String password;
    // getters and setters
}
```

```java
@Configuration
@EnableConfigurationProperties(EmployeeServiceProperties.class)
public class RestClientConfig {

    @Bean
    public RestClient employeeRestClient(EmployeeServiceProperties props) {
        return RestClient.builder()
            .baseUrl(props.getBaseUrl())
            .defaultHeader(HttpHeaders.AUTHORIZATION, basicAuth(props))
            .build();
    }

    private String basicAuth(EmployeeServiceProperties props) {
        String creds = props.getUsername() + ":" + props.getPassword();
        return "Basic " + Base64.getEncoder().encodeToString(creds.getBytes());
    }
}
```

## Calling an External Service

Expose an interface for the service you're calling, then wrap `RestClient` in an implementation. Keep the `client` package separate from `service` — services hold business logic, clients call external APIs.

```java
public interface EmployeeClient {
    EmployeeResponse getEmployee(Long id);
    List<EmployeeResponse> getAllEmployees();
    EmployeeResponse createEmployee(EmployeeRequest request);
    void deleteEmployee(Long id);
}
```

```java
@Component
public class EmployeeClientImpl implements EmployeeClient {

    private final RestClient restClient;

    public EmployeeClientImpl(@Qualifier("employeeRestClient") RestClient restClient) {
        this.restClient = restClient;
    }

    @Override
    public EmployeeResponse getEmployee(Long id) {
        return restClient.get()
            .uri("/employees/{id}", id)
            .accept(MediaType.APPLICATION_JSON)
            .retrieve()
            .body(EmployeeResponse.class);
    }

    @Override
    public List<EmployeeResponse> getAllEmployees() {
        return restClient.get()
            .uri("/employees")
            .accept(MediaType.APPLICATION_JSON)
            .retrieve()
            .body(new ParameterizedTypeReference<>() {});
    }

    @Override
    public EmployeeResponse createEmployee(EmployeeRequest request) {
        return restClient.post()
            .uri("/employees")
            .contentType(MediaType.APPLICATION_JSON)
            .body(request)
            .retrieve()
            .body(EmployeeResponse.class);
    }

    @Override
    public void deleteEmployee(Long id) {
        restClient.delete()
            .uri("/employees/{id}", id)
            .retrieve()
            .toBodilessEntity();
    }
}
```

- **HTTP methods**: `get()`, `post()`, `put()`, `patch()`, `delete()` return a fluent builder for URI, headers, body; finish with `retrieve()` to execute
- **Generic collections**: use `ParameterizedTypeReference` for `List<T>`/`Map<K,V>` since type erasure prevents `List<EmployeeResponse>.class`

> **Production note:** never commit credentials to `application.properties`. Use environment variables or a secrets manager and reference them via `${EMPLOYEE_SERVICE_PASSWORD}` placeholders, or layer on `application-prod.properties` excluded from VCS.

## Error Handling

```java
restClient.get()
    .uri("/employees/{id}", id)
    .retrieve()
    .onStatus(HttpStatusCode::is4xxClientError, (req, res) -> {
        throw new EmployeeNotFoundException("Not found: " + res.getStatusText());
    })
    .onStatus(HttpStatusCode::is5xxServerError, (req, res) -> {
        throw new ServiceUnavailableException("Employee service down");
    })
    .body(EmployeeResponse.class);
```

- Chain multiple `onStatus()` for different error classes
- Without `onStatus()`, non-2xx responses throw `RestClientResponseException` by default

## Multiple Services — Inject with `@Qualifier`

When the app calls several external services, define one `RestClient` bean per service and inject by qualifier:

```java
@Configuration
public class RestClientConfig {

    @Bean
    public RestClient employeeRestClient(EmployeeServiceProperties props) {
        return RestClient.builder()
            .baseUrl(props.getBaseUrl())
            .defaultHeader(HttpHeaders.AUTHORIZATION, basicAuth(props))
            .build();
    }

    @Bean
    public RestClient paymentRestClient(PaymentServiceProperties props) {
        return RestClient.builder()
            .baseUrl(props.getBaseUrl())
            .defaultHeader("X-API-KEY", props.getApiKey())
            .build();
    }
}
```

Each client constructor picks its bean with `@Qualifier("employeeRestClient")` / `@Qualifier("paymentRestClient")`.

## Q&A

1. **`RestClient` vs `RestTemplate` vs `WebClient`?** `RestTemplate` is legacy synchronous (maintained, no new features). `WebClient` is reactive, non-blocking. `RestClient` is the modern synchronous replacement — fluent API like `WebClient` but blocking.

2. **Why inject with `@Qualifier` when you have multiple `RestClient` beans?** Spring can't autowire by type when several beans share a type — `@Qualifier` disambiguates by bean name.

3. **Why use `ParameterizedTypeReference` for lists?** Java generics are erased at runtime, so `List<EmployeeResponse>.class` isn't expressible. `ParameterizedTypeReference` captures the generic type at compile time so the message converter knows what to deserialize into.

4. **What happens on a non-2xx response by default?** `retrieve()` throws `RestClientResponseException` unless you register an `onStatus()` handler for that status class.

5. **Where should you configure base URL and auth — per call or on the bean?** On the bean, via `RestClient.builder().baseUrl(...).defaultHeader(...)`. Keeps call sites clean and makes it easy to swap credentials via `@ConfigurationProperties`.
