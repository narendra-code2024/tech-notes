- **OpenAPI**: a language-agnostic specification for describing REST APIs (paths, methods, parameters, request/response schemas, auth)
- **Swagger UI**: a browser-based tool that renders an OpenAPI spec as interactive, testable documentation
- **Springdoc OpenAPI**: the library that auto-generates an OpenAPI 3 spec from your Spring MVC controllers and serves Swagger UI out of the box

The pre-Springdoc workflow was hand-writing YAML/JSON specs and keeping them in sync with the code. Springdoc inspects your `@RestController` classes at startup and produces the spec for you.

Add the dependency and you're done — no further config required:

```xml
<dependency>
    <groupId>org.springdoc</groupId>
    <artifactId>springdoc-openapi-starter-webmvc-ui</artifactId>
    <version>2.5.0</version>
</dependency>
```

On startup, two URLs are exposed:

| URL | Purpose |
|---|---|
| `/swagger-ui.html` | Interactive Swagger UI |
| `/v3/api-docs` | Raw OpenAPI JSON spec |

---

## Configuration

```properties
# Custom paths
springdoc.api-docs.path=/api-docs
springdoc.swagger-ui.path=/docs

# Disable in production (recommended)
springdoc.api-docs.enabled=false
springdoc.swagger-ui.enabled=false
```

> **Production warning:** leaving Swagger UI exposed in production publishes your full API surface and request/response schemas to anyone who can hit the URL. Either disable via the flags above or restrict access through Spring Security.

## Enriching the Generated Docs

Annotations enrich the auto-generated spec with summaries, response codes, and parameter descriptions:

```java
@RestController
@RequestMapping("/employees")
@Tag(name = "Employees", description = "Employee management APIs")
public class EmployeeController {

    @Operation(summary = "Get employee by ID")
    @ApiResponse(responseCode = "200", description = "Employee found")
    @ApiResponse(responseCode = "404", description = "Employee not found")
    @GetMapping("/{id}")
    public EmployeeResponse getEmployee(
            @Parameter(description = "Employee ID") @PathVariable Long id) {
        return service.getEmployee(id);
    }
}
```

> **Keeping controllers clean — the maximalist example above is illustrative.** Stacking `@Tag` + `@Operation` + multiple `@ApiResponse` + `@Parameter` on every method clutters controllers fast. In practice most codebases use one of three patterns:
>
> - **Annotate sparingly**: let Springdoc work from `@GetMapping` + return types + DTO annotations alone; add `@Operation(summary)` only where intent isn't obvious
> - **Extract to a documented interface**: `@Tag`/`@Operation`/`@ApiResponse` on an interface; controller implements it. Spec lives with the contract, controller stays clean
> - **`OperationCustomizer` bean**: programmatically attach common docs (e.g., default `401`/`403` responses for all authenticated endpoints, default error schema) without touching method signatures

DTO fields can be documented with `@Schema`:

```java
public class EmployeeResponse {

    @Schema(description = "Employee unique identifier", example = "42")
    private Long id;

    @Schema(description = "Full legal name", example = "Alice Smith")
    private String name;
}
```

| Annotation | Where it goes | Purpose |
|---|---|---|
| `@Tag` | Controller class | Groups endpoints under a label in Swagger UI |
| `@Operation` | Method | Sets endpoint summary/description |
| `@Parameter` | Method parameter | Documents path/query/header parameter |
| `@ApiResponse` | Method | Documents a possible HTTP response |
| `@Schema` | DTO field or class | Documents a model property with examples |

> **Bean Validation flows into the spec for free.** Standard JSR-380 annotations on DTOs (`@NotNull`, `@NotBlank`, `@Size`, `@Min`, `@Max`, `@Pattern`, `@Email`) are picked up by Springdoc and translated into OpenAPI schema constraints — `required: true`, `minLength`, `maximum`, `pattern`, `format: email`. No `@Schema` duplication needed; the validation annotations already on your DTOs do double duty.

## Global API Metadata

Per-endpoint annotations cover individual operations. The `OpenAPI` bean configures whole-spec metadata — needed for any production API:

```java
@Configuration
public class OpenApiConfig {

    @Bean
    public OpenAPI customOpenAPI() {
        return new OpenAPI()
            .info(new Info()
                .title("Employee Service API")
                .version("v1")
                .description("Employee management for the Acme HR platform")
                .contact(new Contact().name("Platform Team").email("platform@acme.com")))
            .servers(List.of(
                new Server().url("https://api.acme.com").description("Production"),
                new Server().url("https://api-staging.acme.com").description("Staging")
            ))
            .components(new Components()
                .addSecuritySchemes("bearerAuth", new SecurityScheme()
                    .type(SecurityScheme.Type.HTTP)
                    .scheme("bearer")
                    .bearerFormat("JWT")))
            .addSecurityItem(new SecurityRequirement().addList("bearerAuth"));
    }
}
```

- **`Info`**: what shows in the Swagger UI header (title, version, description, contact)
- **`servers`**: populates the Servers dropdown so consumers can switch environments without re-importing the spec
- **`SecurityScheme` + `SecurityRequirement`**: declares JWT/OAuth2 globally so every endpoint gets the auth lock icon. Override per-endpoint with `@SecurityRequirement(name = "bearerAuth")` or open a public endpoint with `@SecurityRequirements({})`.

## Q&A

1. **What's the difference between OpenAPI and Swagger?** **OpenAPI** is the specification (originally called Swagger Spec, donated to the OpenAPI Initiative in 2015). **Swagger** is now the toolset around it — Swagger UI, Swagger Editor, Swagger Codegen. Springdoc generates an OpenAPI spec and renders it with Swagger UI.

2. **How does Springdoc generate the spec without you writing YAML?** It scans the `ApplicationContext` for `@RestController` beans at startup, reads their request mappings, parameters, and return types via reflection, and builds the OpenAPI JSON exposed at `/v3/api-docs`.

3. **Should Swagger UI be enabled in production?** No — it publishes your entire API surface to anyone who can reach the URL. Disable with `springdoc.swagger-ui.enabled=false` or gate it behind Spring Security.

4. **Are the `@Operation` / `@Schema` annotations required?** No. Springdoc generates a working spec from raw controllers and DTOs. The annotations only enrich the output with summaries, examples, and per-response descriptions.

5. **How do you group related endpoints in Swagger UI?** Put `@Tag(name = "Employees")` on the controller class. All its endpoints render under that heading in the UI.

6. **How do you document JWT/OAuth2 auth in the generated spec?** Define an `OpenAPI` bean and register a `SecurityScheme` (e.g., `Type.HTTP`, scheme `bearer`, bearer format `JWT`), then `addSecurityItem(...)` for global protection. Per-endpoint overrides with `@SecurityRequirement(name = "bearerAuth")`, or open a public endpoint with `@SecurityRequirements({})`.

7. **How do Bean Validation annotations affect the generated spec?** Springdoc reads JSR-380 annotations (`@NotNull`, `@Size`, `@Min`, `@Pattern`, `@Email`, etc.) on DTOs and translates them into OpenAPI schema constraints — `required`, `minLength`, `maximum`, `pattern`, `format`. Validation rules show up in Swagger UI without `@Schema` duplication.
