**Auditing** automatically populates "who created this row and when, who last modified it and when" fields on JPA entities — so you don't hand-write `entity.setCreatedAt(Instant.now())` on every save. Spring Data JPA provides this out of the box via four annotations: `@CreatedBy`, `@CreatedDate`, `@LastModifiedBy`, `@LastModifiedDate`.

> Builds on JPA entity basics — see [3.1 Hibernate and JPA](<../03-hibernate-and-spring-data-jpa/3.1. Hibernate and JPA.md>).

---

## Setup

### 1. Create the `Auditable` base and extend your entities

Define a `@MappedSuperclass` base with the audit fields, then have entities extend it:

```java
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class Auditable {

    @CreatedBy
    @Column(updatable = false)
    private String createdBy;

    @CreatedDate
    @Column(updatable = false)
    private Instant createdAt;

    @LastModifiedBy
    private String lastModifiedBy;

    @LastModifiedDate
    private Instant updatedAt;

    // getters and setters
}

@Entity
public class Employee extends Auditable {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
}
```

- `@MappedSuperclass` — the fields are inherited into subclass tables, no separate `auditable` table
- `@EntityListeners(AuditingEntityListener.class)` — hooks JPA lifecycle callbacks to populate the fields
- `updatable = false` on created-* columns — locks the value after insert
- `Employee` now has `createdBy`, `createdAt`, `lastModifiedBy`, `updatedAt` columns automatically

> **Why `Instant` and not `LocalDateTime`?** `LocalDateTime` has no timezone — it's just wall-clock numbers. A server in IST writing `LocalDateTime.now()` produces `10:30`; the same code on a UTC server at the same moment produces `05:00`. Both stored as just digits. Sorting, comparing, and migrating servers across regions all break. `Instant` is epoch-based — every server, every region produces the same value for "now". For audit timestamps, lifecycle hooks, and event logs (anything system-recorded) — always `Instant`. Reserve `ZonedDateTime` for user-facing events tied to a specific zone ("3 PM IST meeting"); `LocalDate` for date-only fields (birthdays). Convert to user's local zone at the presentation layer, never at storage.

### 2. Configure auditing

Implement `AuditorAware` (tells Spring who the "current user" is) and enable auditing:

```java
@Component
public class AuditorAwareImpl implements AuditorAware<String> {

    @Override
    public Optional<String> getCurrentAuditor() {
        return Optional.ofNullable(SecurityContextHolder.getContext())
            .map(SecurityContext::getAuthentication)
            .filter(Authentication::isAuthenticated)
            .map(Authentication::getName);
    }
}

@Configuration
@EnableJpaAuditing(auditorAwareRef = "auditorAwareImpl")
public class AuditConfig { }
```

- `AuditorAware.getCurrentAuditor()` is called on each persist/update to fill `@CreatedBy` / `@LastModifiedBy` — typically pulled from Spring Security
- For local dev or prototypes (no auth yet), return a fixed value: `return Optional.of("system");`. Replace with the real auth/identity source before production
- `@EnableJpaAuditing(auditorAwareRef = "auditorAwareImpl")` activates auditing and points to the bean name above

> Internally, `AuditingEntityListener` hooks JPA lifecycle events: `@PrePersist` fills all four fields on first save; `@PreUpdate` fills only `@LastModifiedBy` and `@LastModifiedDate`. `AuditorAware.getCurrentAuditor()` is called on each persist/update to populate the `*By` fields.

## Hibernate Envers (Full History)

Spring Data auditing tracks **who/when**, but not **what changed**. For a complete version history per entity, use **Hibernate Envers** — every insert, update, and delete is recorded to a shadow audit table.

```xml
<dependency>
    <groupId>org.hibernate.orm</groupId>
    <artifactId>hibernate-envers</artifactId>
</dependency>
```

Add `@Audited` to any entity to track it:

```java
@Entity
@Audited
public class Employee {

    @Id
    private Long id;

    private String name;
    private Double salary;
}
```

Envers auto-creates an `employee_aud` table plus a global `revinfo` table.

### What gets stored

Updating salary from `50000` to `60000` leaves this trail in `employee_aud`:

| id | name | salary | rev | revtype |
|----|------|--------|-----|---------|
| 1 | John | 50000 | 1 | 0 (INSERT) |
| 1 | John | 60000 | 2 | 1 (UPDATE) |

- `rev` — revision number
- `revtype` — `0` = INSERT, `1` = UPDATE, `2` = DELETE

### Use cases

- **Compliance** — financial or healthcare systems must track every data change
- **Undo/revert** — roll back an entity to a previous version
- **Dispute resolution** — "who changed this order, and when?"
- **Data recovery** — restore accidentally modified or deleted records

## Spring Data Auditing vs Hibernate Envers

| Feature | Spring Data Auditing | Hibernate Envers |
|---|---|---|
| Tracks who & when | Yes | Yes |
| Tracks what changed (history) | No | Yes |
| Stores previous values | No | Yes |
| Separate audit tables | No | Yes |
| Setup complexity | Simple | Moderate |
| Use case | Basic audit fields | Full audit trail |

## Important Interview Questions

1. **What does `@EnableJpaAuditing` do?** It registers `AuditingEntityListener` as a JPA lifecycle listener so entities annotated (directly or via `@MappedSuperclass`) get their `@CreatedDate`/`@CreatedBy`/`@LastModifiedDate`/`@LastModifiedBy` fields populated automatically.

2. **Why `@MappedSuperclass` instead of `@Entity` for the `Auditable` base?** `@MappedSuperclass` means "inherit these columns into the subclass table" — there's no `auditable` table. `@Entity` would create a separate table and force a join.

3. **How does Spring know the current user for `@CreatedBy`?** Through an `AuditorAware<T>` bean. Its `getCurrentAuditor()` is called on each persist/update — typically pulling the username from `SecurityContextHolder`.

4. **Spring Data auditing vs Hibernate Envers — when to use which?** Use Spring Data auditing for simple "created/modified" stamps. Use Envers when you need full row-level history (previous values, revert, compliance trails).

5. **Why is `@Column(updatable = false)` used on `createdAt`?** It tells Hibernate not to include that column in UPDATE statements, so the creation timestamp is locked once set.
