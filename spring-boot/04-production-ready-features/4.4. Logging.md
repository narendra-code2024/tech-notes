**Logging** tracks events at runtime — what the app did, when, and what went wrong — so you can debug without attaching a debugger.

Spring Boot's logging stack:

- **Logback**: the default implementation; included with any Spring Boot starter and writes the actual output (console / file / network).
- **SLF4J** (**Simple Logging Facade for Java**) — the facade your code calls. It lets you switch the backend (to Log4j2, JUL, etc.) without changing application code.
- **Swap to Log4j2**: exclude `spring-boot-starter-logging` and add `spring-boot-starter-log4j2`.

Either backend has the same three moving parts:

| Element | Purpose |
|---|---|
| **Logger** | Captures log events from your code |
| **Formatter** (encoder) | Formats each log event into a string |
| **Appender** (handler) | Dispatches formatted output — console, file, SMTP, etc. |

---

## Log Levels

From most severe to most verbose:

| Level | Use for |
|---|---|
| `FATAL` | Fatal error crashing the system |
| `ERROR` | Runtime errors that need attention |
| `WARN` | Suspicious-but-recoverable situations |
| `INFO` | High-level application events |
| `DEBUG` | Internal flow, useful while diagnosing |
| `TRACE` | Fine-grained trace detail |

Enabling a level logs that level **and everything above it**. Setting the root to `WARN` emits `WARN`, `ERROR`, `FATAL` — but not `INFO`, `DEBUG`, `TRACE`.

## Using the Logger in Code

Two ways to get a `log` reference — declare it manually, or have Lombok generate it.

**Manual**: import SLF4J's `Logger` + `LoggerFactory` and declare a `static final` field:

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

@Service
public class EmployeeService {

    private static final Logger log = LoggerFactory.getLogger(EmployeeService.class);

    public Employee getEmployee(Long id) {
        log.info("Fetching employee with id: {}", id);
        log.debug("Querying database for employee: {}", id);
        return repository.findById(id)
            .orElseThrow(() -> {
                log.error("Employee not found with id: {}", id);
                return new EmployeeNotFoundException(id);
            });
    }
}
```

**Lombok `@Slf4j`**: annotate the class and Lombok injects the same `log` field at compile time, so you skip the imports + declaration:

```java
import lombok.extern.slf4j.Slf4j;

@Service
@Slf4j
public class EmployeeService {
    public Employee getEmployee(Long id) {
        log.info("Fetching employee with id: {}", id);
        // ...
    }
}
```

- The `log` name is a Lombok convention — same field, same SLF4J `Logger` type, no runtime cost
- Use `{}` placeholders — SLF4J only formats the string if the level is enabled, so string concatenation is wasted work and risks `NullPointerException` from a `null.toString()`
- **Preferred**: `@Slf4j` if Lombok is already on the project (most Spring Boot codebases); manual declaration only when avoiding Lombok

## `application.yml` (simple config)

Good for basic needs; can't do rolling files or multiple appenders.

```yaml
logging:
  # Levels
  level:
    root: INFO
    com.example: DEBUG

  # Console format
  pattern:
    console: "%d [%level] %c{1.} [%t] %m%n"
    file: "%d{yyyy-MM-dd HH:mm:ss} [%level] %c{1.} [%t] %m%n"

  # File output
  file:
    name: logs/app.log
```

- `logging.file.name` must be set for a file to be created — `logging.pattern.file` alone does nothing

---

## Pattern Placeholders

| Pattern | Meaning | Example |
|---|---|---|
| `%d{yyyy-MM-dd HH:mm:ss}` | Timestamp | `2026-02-22 14:30:00` |
| `%-5level` | Level, left-padded to 5 chars | `INFO `, `DEBUG` |
| `%level` | Level, unpadded | `INFO` |
| `%c{1}` | Class name (short) | `EmployeeService` |
| `%c` | Full class path | `com.example.service.EmployeeService` |
| `%t` | Thread name | `http-nio-8080-exec-1` |
| `%m` / `%msg` | The message | `Fetching employee with id: 1` |
| `%n` | Newline | — |

Pattern `%d{yyyy-MM-dd HH:mm:ss} %-5level %c{1} - %msg%n` produces:

```
2026-02-22 14:30:00 INFO  EmployeeService - Fetching employee with id: 1
```

## `logback-spring.xml` (production)

Place at `src/main/resources/logback-spring.xml`. When present, it **overrides** `application.yml` logging config. Use `logback-spring.xml` (not `logback.xml`) to get Spring Boot profile support (`<springProfile>`).

```xml
<configuration>

    <property name="LOG_PATTERN"
              value="%d{yyyy-MM-dd HH:mm:ss} %-5level %c{1} - %msg%n"/>
    <property name="LOG_PATH" value="logs"/>

    <!-- Rolling file appender: where logs go and when to rotate -->
    <appender name="ROLLING_FILE"
              class="ch.qos.logback.core.rolling.RollingFileAppender">

        <rollingPolicy
            class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">

            <fileNamePattern>
                ${LOG_PATH}/application.%d{yyyy-MM-dd}.%i.log
            </fileNamePattern>

            <maxHistory>30</maxHistory>

            <timeBasedFileNamingAndTriggeringPolicy
                class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
                <maxFileSize>10MB</maxFileSize>
            </timeBasedFileNamingAndTriggeringPolicy>

        </rollingPolicy>

        <encoder>
            <pattern>${LOG_PATTERN}</pattern>
        </encoder>
    </appender>

    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>${LOG_PATTERN}</pattern>
        </encoder>
    </appender>

    <root level="info">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="ROLLING_FILE"/>
    </root>

    <!-- Package-level overrides -->
    <logger name="com.example.service" level="DEBUG"/>
    <logger name="org.hibernate.SQL" level="DEBUG"/>

</configuration>
```

With `fileNamePattern = ${LOG_PATH}/application.%d{yyyy-MM-dd}.%i.log`:

- `logs/application.2026-02-22.0.log` — first file of the day
- `logs/application.2026-02-22.1.log` — second file (first hit 10MB)
- `logs/application.2026-02-23.0.log` — next day, starts fresh

After 30 days the oldest files are auto-deleted.

## Production Considerations

- **Structured (JSON) logs**: log aggregators (ELK, Loki, Datadog) parse JSON far better than free-text. Spring Boot 3.4+ ships first-class structured logging via `logging.structured.format.console: ecs` (or `logstash`/`gelf`). Pre-3.4, add `logstash-logback-encoder` and use `LogstashEncoder` in `logback-spring.xml`
- **Log levels in prod**: root `INFO`, your packages `INFO`, `org.hibernate.SQL` and `org.springframework.web` at `WARN`. `DEBUG` in production produces gigabytes per hour and slows the JVM
- **Never log secrets or PII**: passwords, tokens, full credit-card numbers, or unmasked emails. Mask in the producer, not the aggregator
- **Correlation IDs**: put a request/trace ID into MDC so every log line for one request shares an ID. Spring Boot 3 + Micrometer Tracing wires this automatically; pattern with `%X{traceId}`/`%X{spanId}`
- **Async appenders for hot paths**: wrap a file/network appender in `AsyncAppender` so logging doesn't block request threads
- **Profile-specific config**: `<springProfile name="prod">` blocks in `logback-spring.xml` keep dev verbose and prod terse without separate files

## Q&A

1. **What is SLF4J and why does Spring Boot use it?** **SLF4J** is a logging facade — application code depends only on the SLF4J API, and the real backend (Logback, Log4j2, JUL) is chosen at deployment. Spring Boot uses Logback by default.

2. **Why prefer `log.info("msg: {}", id)` over `log.info("msg: " + id)`?** The placeholder form defers string formatting until SLF4J confirms the level is enabled — at `WARN` level, the concatenation work is skipped entirely. It also avoids accidental `toString()` calls on null.

3. **`logback.xml` vs `logback-spring.xml` — why use the latter?** `logback-spring.xml` is read by Spring Boot (not raw Logback), which enables `<springProfile>` and `<springProperty>` tags so you can switch config per profile.

4. **How do you override the log level for one package only?** `logging.level.com.example: DEBUG` in `application.yml`, or `<logger name="com.example" level="DEBUG"/>` in `logback-spring.xml`. It overrides the root level for that package.

5. **What does setting the root logger to `WARN` emit?** `WARN`, `ERROR`, `FATAL` — the level and everything more severe. `INFO`/`DEBUG`/`TRACE` are suppressed.

6. **How do rolling files prevent disk exhaustion?** `TimeBasedRollingPolicy` rotates on a time boundary (daily), `maxFileSize` rotates within a day when a file grows too large, and `maxHistory` caps how many old files are retained.

## Quick Reference

| Concept | Remember |
|---|---|
| Default backend / facade | Logback / SLF4J |
| Logger field | `LoggerFactory.getLogger(X.class)` or Lombok `@Slf4j` |
| Placeholder form | `log.info("id={}", id)` (not concatenation) |
| Level ordering | `TRACE < DEBUG < INFO < WARN < ERROR < FATAL` |
| Config file | `logging.level.*`, `logging.file.name`, `logging.pattern.*` in `application.yml` |
| Advanced config | `logback-spring.xml` in `src/main/resources` |
| Rolling strategy | `RollingFileAppender` + `SizeAndTimeBasedFNATP` + `maxHistory` |
