**Spring Boot DevTools** speeds up the dev loop by automatically restarting the application when classpath files change, and triggering a browser refresh via **LiveReload** when static resources change. It also applies development-friendly defaults (e.g., disables template caching) so you see changes without restarting manually.

- Only active in development — excluded from the final **fat JAR** when run as a packaged app
- Uses two classloaders: a **base** loader for unchanged dependencies and a **restart** loader for your code, so restarts are much faster than a full cold start

> **Fat JAR**: a single executable archive containing the application's compiled classes and all external dependencies (Spring, Jackson, JDBC drivers, etc.), runnable with `java -jar app.jar` — no separate classpath needed. Spring Boot builds one with `mvn package`. Also called an **uber JAR**, or **JAR with dependencies** (the Maven Shade plugin's naming).

## Setup

### Add the Dependency
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-devtools</artifactId>
    <scope>runtime</scope>
    <optional>true</optional>
</dependency>
```

> **Note:** `<optional>true</optional>` ensures DevTools stays scoped to your project and isn't pulled into other projects that depend on yours (**Transitive dependency** management).

### Enable in IntelliJ IDEA
DevTools triggers a restart only when the classpath changes. You must configure IntelliJ to recompile on save:

- `Settings → Build, Execution, Deployment → Compiler` → Select **Build project automatically**: `Settings → Advanced Settings` → Enable **Allow auto-make to start even if developed application is currently running**: -

## Key Properties

| Property | Default | Description |
|---|---|---|
| `spring.devtools.restart.enabled` | `true` | Enables automatic restart when classpath files change |
| `spring.devtools.restart.exclude` | `static/**, public/**` | Paths that don't trigger a restart |
| `spring.devtools.restart.additional-paths` | — | Extra directories to monitor |
| `spring.devtools.restart.trigger-file` | — | Restart only when this specific file is touched |
| `spring.devtools.restart.poll-interval` | `1000ms` | How often DevTools checks for file changes |
| `spring.devtools.restart.quiet-period` | `400ms` | Wait time after last change before restarting |
| `spring.devtools.livereload.enabled` | `true` | Enables LiveReload server for browser refresh |
| `spring.devtools.livereload.port` | `35729` | Port used by the LiveReload server |
| `spring.devtools.add-properties` | `true` | Applies dev-friendly defaults (disables template caching, etc.) |

## Q&A

1. **What does Spring Boot DevTools actually do?** It watches the classpath and restarts the application when compiled classes change, enables LiveReload on the frontend, and applies development defaults like disabling template caching.

2. **Why is DevTools restart faster than a normal restart?** It uses two classloaders — a base loader for third-party JARs (unchanged) and a restart loader for your application classes. Only the restart loader is thrown away and recreated, so the JVM stays warm.

3. **Will DevTools run in production?** No. When the app runs from a packaged JAR, Spring Boot disables DevTools automatically. Marking the dependency `optional` also prevents it from being pulled into downstream artifacts.

4. **Why doesn't auto-restart trigger in IntelliJ on save?** IntelliJ doesn't recompile on save by default. You must enable "Build project automatically" and "Allow auto-make to start even if developed application is currently running".

5. **How do you prevent static-resource changes from triggering a full restart?** They're excluded by default (`static/**, public/**`). Use LiveReload for those — it refreshes the browser without restarting the JVM.
