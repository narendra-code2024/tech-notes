## Project Scaffolding

- **IDE**: IntelliJ IDEA (recommended), Eclipse, VS Code
- **Project generator**: [Spring Initializr](https://start.spring.io) — scaffolds a Spring Boot project with your chosen Java version, build tool, and starter dependencies
- **Starter dependencies**: one declaration in `pom.xml` pulls in a curated bundle of related libraries with compatible versions (e.g., `spring-boot-starter-web` brings Spring MVC + Jackson + Tomcat for REST APIs)

## Build Tools

Spring Boot supports two mainstream build tools:

- **Maven** (most common) — declarative XML (`pom.xml`) for dependency management and orchestrating the build lifecycle.
- **Gradle**: Groovy/Kotlin DSL (`build.gradle`), significantly faster for large builds due to its build cache.

---

## `pom.xml`

Configuration file for Maven-based projects — **P**roject **O**bject **M**odel. **Configures the build process**, plugins, profiles, project metadata, and dependencies.

Three key pieces you'll see in any Spring Boot `pom.xml`:

- **Parent POM**: like Java class inheritance, a `pom.xml` can declare a `<parent>` to inherit shared config (dependency versions, plugins, properties) from an upstream POM. Spring Boot ships with `spring-boot-starter-parent` as that upstream POM:
    - Supplies sensible defaults (encoding, compiler plugin, resource filtering)
    - Manages dependency versions, plugin versions, and the Java version
    - Internally uses `spring-boot-dependencies` for the version BOM

- **Dependencies**: declared in `<dependencies>`. Common starters:
    - `spring-boot-starter-web` → Tomcat, Spring MVC, Jackson
    - `spring-boot-starter-data-jpa` → Hibernate, JPA, JDBC
    - `spring-boot-starter-security` → Spring Security + auto-configured filter chain
    - `spring-boot-starter-test` → JUnit, Mockito, AssertJ, Spring Test

- **Plugins**: Build tools (Maven/Gradle) are modular; they use plugins to extend their core capabilities. Plugins handle tasks like compiling code, running tests, static analysis, or packaging. In any Spring Boot project, the most critical one is the **Spring Boot Build Plugin**:
    - **Maven:** `spring-boot-maven-plugin`
    - **Gradle:** `org.springframework.boot`
    - **What it does:** Packages your code into a **"fat JAR"** (containing all dependencies + the embedded server) so it can run anywhere with just `java -jar`. It also provides the `run` command for development.
    - **Note:** You must explicitly declare this plugin in your `pom.xml` (or `build.gradle`). **Spring Initializr includes this by default in every generated project.**: -

## Application Configuration Files (YAML vs Properties)

Spring Boot reads runtime configuration from `src/main/resources/application.{properties|yml}` (and profile-specific variants like `application-dev.yml`). Both formats are functionally equivalent — Spring loads either — but choose one and stick with it.

**`application.properties`**: older, flat key-value format:
```properties
spring.application.name=myapp
spring.datasource.url=jdbc:postgresql://localhost:5432/myapp
spring.datasource.username=admin
spring.datasource.hikari.maximum-pool-size=10
server.port=8080
```

**`application.yml`**: same config, hierarchical:
```yaml
spring:
  application:
    name: myapp
  datasource:
    url: jdbc:postgresql://localhost:5432/myapp
    username: admin
    hikari:
      maximum-pool-size: 10
server:
  port: 8080
```

### Why Prefer YAML?

- **DRY (Don't Repeat Yourself)**: Hierarchical structure avoids repeating long prefixes like `spring.datasource.hikari.`.
- **Readability**: Visual structure mirrors the configuration tree.
- **Power**: Native support for lists and profile separators (`---`).
- **Standard**: Favored in modern projects and official Spring documentation.

> **Note:** Use **spaces, never tabs.** A single misaligned line can silently map a property to the wrong parent, often without a startup error.

### When `.properties` Still Makes Sense

- **Legacy**: Maintaining consistency in older projects already using it.
- **Greppability**: Flat keys can be easier to search/filter in large codebases.
- **Compatibility**: When external config tools or older environments require it.

## Q&A

1. **What is a starter dependency?**
   A curated bundle of related libraries with compatible versions — e.g., `spring-boot-starter-web` pulls in Tomcat + Spring MVC + Jackson with a single dependency entry.

2. **What does `spring-boot-starter-parent` do?**
   It's a parent POM that manages dependency versions, plugin versions, and the Java version so your own `pom.xml` stays minimal.
