DI is a design pattern where a class receives its **dependencies** from the outside instead of creating them itself. In Spring, the **IoC container** resolves and injects them based on **type** (and optionally **name**).

---

## Types of DI

| Type | Annotation | Recommended? |
|---|---|---|
| **Constructor** | `@Autowired` optional if single constructor (Spring 4.3+); required when multiple constructors exist | Yes — best practice |
| **Setter**      | `@Autowired` on setter method                                                                        | OK for optional dependencies |
| **Field**       | `@Autowired` on field                                                                                | No — Avoid — hard to test, hides dependencies |

```java
// Constructor injection (preferred) — @Autowired optional for a single constructor
@Service
public class OrderService {
    private final PaymentService paymentService;

    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}

// Setter injection — for optional dependencies
@Service
public class OrderService {
    private PaymentService paymentService;

    @Autowired
    public void setPaymentService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}

// Field injection (discouraged)
@Service
public class OrderService {
    @Autowired
    private PaymentService paymentService;
}
```

## Multiple Beans of Same Type

### Using `@Qualifier`

Pick a specific bean by name at the injection site.

```java
@Service("s3Storage")
public class S3Storage implements StorageService { }

@Service("localStorage")
public class LocalStorage implements StorageService { }

// Consumer — picks the specific bean by name
@Service
public class MyService {
    private final StorageService storage;

    public MyService(@Qualifier("s3Storage") StorageService storage) {
        this.storage = storage;
    }
}
```

- Bean name can be set via the stereotype annotation: `@Service("s3Storage")`
- If not set, bean name defaults to the class name in camelCase (`s3Storage`)
- For `@Bean` methods, the bean name defaults to the method name

### Using `@Primary`

Mark one bean as `@Primary` — it will be chosen by default when no `@Qualifier` is specified.

```java
@Service
@Primary
public class S3Storage implements StorageService { }

@Service
public class LocalStorage implements StorageService { }

// Consumer — no @Qualifier needed
@Service
public class MyService {
    private final StorageService storage;

    public MyService(StorageService storage) {
        // S3Storage is injected (because @Primary)
        this.storage = storage;
    }
}
```

## Injecting a Collection of Beans

### Using `List` (all beans of that type)

```java
@Service
public class StorageManager {
    private final List<StorageService> services;

    public StorageManager(List<StorageService> services) {
        this.services = services;   // all beans of type StorageService, in order
    }
}
```

### Using `Map` (bean name → bean instance)

```java
@Service
public class MappedStorageManager {
    private final Map<String, StorageService> serviceMap;

    public MappedStorageManager(Map<String, StorageService> serviceMap) {
        this.serviceMap = serviceMap;   // e.g., "s3Storage" → S3Storage instance
    }
}
```

## Q&A

1. **What is IoC, and how does DI relate to it?**
   **IoC** (Inversion of Control) = the framework decides object creation and wiring instead of your code. **DI** (Dependency Injection) is the mechanism — constructor, setter, or field injection — the framework uses to hand dependencies to objects. Together they make code loosely coupled and testable.

2. **Constructor vs setter vs field injection — which is best?**
   Constructor is best: dependencies are explicit, the object is never in an invalid state, and it works cleanly with `final` fields and unit tests. Setter is OK for optional deps. Field injection is discouraged — hard to test, hides dependencies.

3. **What's the difference between `@Qualifier` and `@Primary`?**
   `@Primary` sets a default bean when multiple candidates exist — injection without `@Qualifier` picks it. `@Qualifier` explicitly picks a named bean at the injection site, overriding `@Primary`.

4. **Can you inject a collection of beans?**
   `List<StorageService>` injects all beans of that type in order. `Map<String, StorageService>` injects bean-name → bean-instance.

5. **Why is field injection discouraged?**
   Dependencies aren't visible from the constructor, can't use `final` fields, requires reflection to test (no easy constructor with mocks), and encourages too many dependencies per class.

## Quick Reference

|Concept|Remember|
|---|---|
| DI | Dependencies injected by container, not created by class |
| Constructor injection | Preferred — explicit, `final`-friendly, easy to test |
| `@Qualifier("name")` | Pick a specific bean when multiple match |
| `@Primary` | Default bean when no `@Qualifier` is used |
| `List<T>` injection | All beans of type T |
| `Map<String, T>` | Bean-name → instance map |
