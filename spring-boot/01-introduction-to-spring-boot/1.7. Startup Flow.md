## Startup Flow

### 1. `main()` Method

- Standard Java entry point — JVM invokes `public static void main(String[] args)`
- Calls `SpringApplication.run(App.class, args)`, the static bootstrap helper that wires everything that follows

### 2. Spring Boot Pre-Setup

- Detects application type: **Servlet** (Tomcat) / **Reactive** (Netty) / **Non-web**: Loads `ApplicationContextInitializer`s and `ApplicationListener`s
- Prepares Environment:
  - `application.properties` / `application.yml`
  - Profile-specific configs (`application-dev.properties`)
  - OS environment variables
  - JVM arguments
  - Profiles resolved (`spring.profiles.active`)

### 3. Banner Printed

- Default Spring banner, custom `banner.txt`, or disabled via config

### 4. ApplicationContext Created

Depending on app type:

- `AnnotationConfigServletWebServerApplicationContext` (most common — web apps)
- `ReactiveWebServerApplicationContext` (reactive apps)
- `AnnotationConfigApplicationContext` (non-web apps)

> **Created vs refreshed:** "Created" = the container object exists but is empty. "Refreshed" = the phase (steps 5–9) where bean definitions are processed, singletons instantiated, and the context becomes usable. `ContextRefreshedEvent` fires at the end of this phase.

### 5. Configuration Processing

`@SpringBootApplication` internally enables:

- `@Configuration` — marks the class as a source of bean definitions
- `@EnableAutoConfiguration` — triggers auto-configuration
- `@ComponentScan` — scans for `@Component`, `@Service`, `@Repository`, `@Controller`, `@RestController`

### 6. Auto-Configuration

- Reads `AutoConfiguration.imports` (current) or `spring.factories` (legacy)
- Applies conditional annotations (`@ConditionalOnClass`, `@ConditionalOnMissingBean`, `@ConditionalOnProperty`)

### 7. Bean Definitions Created

- Spring registers `BeanDefinition` metadata only: bean name, scope, dependencies, init methods
- Beans are NOT fully instantiated yet at this stage
- Two-phase design: definitions-first lets Spring resolve dependency order, apply `BeanPostProcessor`s, and honor user overrides (`@ConditionalOnMissingBean`) before any object is constructed

### 8. Bean Instantiation (IoC in Action)

- Creates singleton beans
- Resolves and injects dependencies (constructor → setter → field)
- Full bean lifecycle executes (see [1.3 Beans](<1.3. Beans.md>))

### 9. Embedded Web Server Starts

For web apps:

- Tomcat / Jetty / Undertow is created and started
- Port binding (e.g., 8080)
- Servlet context initialized

> **Note:** The server is created during context refresh. Bean instantiation and server startup are interleaved during this phase — not strictly sequential steps.

### 10. Startup Runners Execute

After context is fully ready:

```java
@Component
class StartupRunner implements CommandLineRunner {
    public void run(String... args) {
        // executes here
    }
}
```

Use for: DB seeding, cache warm-up, startup validation checks.

### 11. Application Lifecycle Events Published

Key events in order:

- `ApplicationStartingEvent`
- `ApplicationEnvironmentPreparedEvent`
- `ApplicationPreparedEvent`
- `ContextRefreshedEvent`
- `ApplicationStartedEvent`
- `ApplicationReadyEvent` — app is fully ready

Once the context is refreshed and listeners have fired, the app is serving requests:

```
Client → DispatcherServlet → Controller → Service → Repository → DB
```

## Property Precedence

Property precedence at startup (highest wins):

1. Command-line arguments
2. JVM system properties (`-D`)
3. OS environment variables
4. Profile-specific properties (`application-{profile}.properties`)
5. `application.properties` / `application.yml`
6. Default values in code

> Full profile activation, `@Profile` usage, and per-environment config layouts live in [8.1 Spring Profiles](<../08-spring-boot-deployment-cicd/8.1. Spring Profiles.md>).

## Q&A

1. **Walk me through what happens when a Spring Boot app starts.**
   `SpringApplication.run()` → detect app type → prepare Environment (properties + profiles) → create ApplicationContext → process `@SpringBootApplication` (auto-config + component scan) → register bean definitions → instantiate singletons + inject dependencies → start embedded server → run `CommandLineRunner`s → publish `ApplicationReadyEvent`.

2. **Where can I put startup logic that runs after the context is ready?**
   Implement `CommandLineRunner` or `ApplicationRunner` — they execute after all beans are initialized and the server has started. Common uses: DB seeding, cache warm-up, health-check pings.

3. **What's the property precedence order?**
   Command-line args → JVM system properties (`-D`) → OS env vars → profile-specific `.properties` → `application.properties` → code defaults. Higher in the list wins.

4. **Difference between `ContextRefreshedEvent` and `ApplicationReadyEvent`?**
   `ContextRefreshedEvent` fires when the ApplicationContext finishes initializing but the server may not be serving yet. `ApplicationReadyEvent` fires last — app is fully started and accepting requests.

## Quick Reference

|Concept|Remember|
|---|---|
| Entry point | `SpringApplication.run(App.class, args)` |
| `@SpringBootApplication` | `@Configuration` + `@EnableAutoConfiguration` + `@ComponentScan` |
| Startup order | Env → Context → Auto-config → Beans → Server → Runners → Ready |
| `CommandLineRunner` | Runs once after context is ready |
| `ApplicationReadyEvent` | Last event — app is fully up |
| Property precedence | CLI > JVM `-D` > env > profile `.properties` > base `.properties` > code |
