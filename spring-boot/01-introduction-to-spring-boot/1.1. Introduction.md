## Why Spring Boot Exists

**Traditional Java EE pain (pre-Spring)**: **Heavy application servers**: required deploying to external servers like WebLogic, JBoss, GlassFish
- **Complex configuration**: verbose XML-based wiring of components
- **Slow development cycle**: Build → Package → Deploy → Test loop was time-consuming
- **Inefficient JDBC and ORM integration**: lots of manual setup and boilerplate
- **Boilerplate code**: verbose, repetitive setup across projects
- **Tight coupling without DI**: before DI adoption, classes instantiated their own dependencies using `new`, making code hard to test

**Spring Framework's answer:** introduced **Inversion of Control (IoC)** and **Dependency Injection (DI)** to solve coupling and testability — but still required significant manual configuration (XML or Java config).

**Spring Boot's improvement**: **Auto-configuration**: convention over configuration
- **Starter dependencies**: curated dependency bundles with managed versions
- **Embedded servers**: Tomcat, Jetty, Undertow — no external server needed
- **Production-ready features**: health checks, metrics, monitoring via Actuator
- **In-memory databases**: H2, HSQLDB for development and testing

## Q&A

1. **What is Spring Boot?**
   An opinionated framework on top of Spring. With a few annotations and one starter dependency you get a production-ready REST API — auto-configuration wires common beans (Tomcat, Jackson, HikariCP), starter bundles manage versions, and an embedded server replaces WAR deployment.

2. **What problems does Spring Boot solve?**
   Verbose XML configuration, external server management, slow Build→Deploy cycles, boilerplate for common setups, tight coupling in pre-DI code.
