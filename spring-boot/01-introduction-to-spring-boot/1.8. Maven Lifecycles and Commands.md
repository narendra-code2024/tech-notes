## Maven Fundamentals

### The Maven Wrapper (mvnw)

Every modern Spring Boot project includes two files: `mvnw` (Linux/macOS) and `mvnw.cmd` (Windows).

- **What it is:** A bundled script that downloads and runs a specific version of Maven.
- **Why use it:** It ensures **environmental consistency**. Anyone can build the project without manually installing Maven, and everyone on the team uses the exact same Maven version.
- **How to use:** Instead of `mvn clean install`, use `./mvnw clean install` (on Windows: `mvnw clean install`).

### How Maven Works

1. Reads `pom.xml`
2. Resolves dependencies from Maven Central Repository
3. Downloads and caches them in your local repository (`~/.m2/repository`)
4. Adds all resolved dependencies to the project classpath

Downloaded dependencies appear as **External Libraries** in the IDE. The key one to know is `spring-boot-autoconfigure` — it drives auto-configuration (see [1.6 Auto-Configuration](<1.6. Auto-Configuration.md>)).

### Maven Lifecycles

Maven has three independent lifecycles:

1. **Clean lifecycle**: `clean` deletes the `target/` directory
2. **Default lifecycle**: `validate → compile → test → package → verify → install → deploy`
3. **Site lifecycle**: `site` generates project documentation

> `clean` is a separate lifecycle. Running `mvn clean install` invokes two lifecycles together.

## Common Maven Commands

| Command | Purpose |
|---|---|
| `mvn clean` | Deletes `target/` directory (separate lifecycle) |
| `mvn validate` | Validates project structure and info |
| `mvn compile` | Compiles source code → `target/classes` |
| `mvn test` | Runs unit tests (JUnit / TestNG) |
| `mvn package` | Creates JAR/WAR in `target/` |
| `mvn verify` | Runs checks on results of integration tests |
| `mvn install` | Installs JAR/WAR into local repo (`~/.m2/repository`) |
| `mvn deploy` | Builds, tests, packages, and uploads artifact to remote repo (CI/CD) |
| `mvn spring-boot:run` | Runs the app directly without packaging a JAR/WAR — ideal for local dev. |
| `mvn spring-boot:build-image` | Builds a Docker image for the app (no Dockerfile required) |

## Q&A

1. **What are Maven's lifecycles?**
   Three lifecycles: `clean` (deletes `target/`), `default` (validate → compile → test → package → verify → install → deploy), and `site` (docs). Running a phase runs every phase before it in the same lifecycle.

2. **Difference between `mvn package` and `mvn install`?**
   `package` produces the JAR/WAR in `target/`. `install` does everything `package` does plus copies the artifact into your local Maven repo (`~/.m2/repository`) so other local projects can depend on it.

3. **Difference between `mvn install` and `mvn deploy`?**
   `install` writes to the local repo only. `deploy` uploads to a remote repo (Nexus, Artifactory, Maven Central) — used in CI/CD pipelines.

4. **What does `mvn spring-boot:run` do differently from `java -jar`?**
   `spring-boot:run` is a Maven plugin goal — it builds the classpath from `pom.xml` and runs the main class directly, without packaging a JAR. `java -jar app.jar` runs the already-packaged executable JAR. The plugin is for dev; the JAR is for prod.

5. **What's in `~/.m2/repository`?**
   Your local Maven cache — every dependency JAR you've ever pulled, organized by `groupId/artifactId/version/`. Maven checks here before hitting the remote repo.
