Spring Security verifies identity through a small set of collaborating components: a context that holds the authenticated user, a service that loads users, an encoder that checks passwords, and a manager that orchestrates the flow.

---

## SecurityContext and Authentication

The per-thread security state model: `SecurityContext` is the holder, `Authentication` is the object it holds.

### `SecurityContext`

Holds the currently authenticated user, stored in a `ThreadLocal` so each request thread has its own copy. Read it anywhere downstream:

```java
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String email = auth.getName();
```

### `Authentication` object

Represents the authenticated user and carries:

- **Principal**: who (email/username).
- **Credentials**: password, usually erased after authentication.
- **Authorities**: roles/permissions (`ROLE_USER`, `ROLE_ADMIN`).

> **Reference**: authentication is triggered from inside the filter chain (by `UsernamePasswordAuthenticationFilter` or a custom filter like `JwtAuthFilter`); for where those filters sit, see [5.2. Spring Security Filter Chain.md](<5.2. Spring Security Filter Chain.md>).

## UserDetails and UserDetailsService

- **`UserDetails`**: Spring Security's contract for a user (`getUsername()`, `getPassword()`, `getAuthorities()`, plus account-state flags). Your JPA `User` entity is not a `UserDetails`; you convert it.
- **`UserDetailsService`**: a single-method bridge (`loadUserByUsername(String)`) between your data store and Spring Security. You implement it, fetch from the DB, and return a `UserDetails`. Spring calls it during authentication.
- **Built-ins**: `InMemoryUserDetailsManager` for tests/demos, or a custom implementation backed by the database.

```java
@Service
@RequiredArgsConstructor
public class CustomUserDetailsService implements UserDetailsService {

    private final UserRepository userRepository;

    @Override
    public UserDetails loadUserByUsername(String email)
            throws UsernameNotFoundException {

        User user = userRepository.findByEmail(email)
            .orElseThrow(() ->
                new UsernameNotFoundException("Invalid credentials"));

        return org.springframework.security.core.userdetails.User.builder()
            .username(user.getEmail())
            .password(user.getPassword())
            .authorities(
                user.getRoles().stream()
                    .map(SimpleGrantedAuthority::new)
                    .toList())
            .build();
    }
}
```

> **Note**:
> - The method is named `loadUserByUsername` but receives whatever the login form submits (email, username, phone); Spring does not inspect the value's meaning.
> - Authorities must carry the `ROLE_` prefix for `hasRole("ADMIN")` to match, i.e. store `ROLE_ADMIN`. Without it, only `hasAuthority("ADMIN")` matches.

## PasswordEncoder

Hashes a raw password (`encode()`) and verifies a raw password against a stored hash (`matches()`). Never store plaintext. Use `BCryptPasswordEncoder`.

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12);
}
```

> **Note**: cost factor 12 is a common production choice (default 10); each increment doubles the work. Tune for roughly 250ms per hash on the auth server.

> **Reference**: hashing internals (salting, bcrypt vs alternatives) are covered in [Encoding & Password Hashing.md](<Encoding & Password Hashing.md>).

- **`AuthenticationManager`**: the entry point, one method `authenticate`.
- **`ProviderManager`**: the default implementation; loops its `AuthenticationProvider` list and the first that `supports()` the auth type wins.
- **`DaoAuthenticationProvider`**: the username/password provider; calls `UserDetailsService` to load the user and `PasswordEncoder` to verify the password.

> **Note**: `InMemoryUserDetailsManager` is a `UserDetailsService`, not an `AuthenticationProvider`. It plugs into `DaoAuthenticationProvider` like any other user store; the provider does not care whether users come from a DB or memory.

Expose the manager as a bean so controllers and services can inject and call it directly:

```java
@Bean
public AuthenticationManager authenticationManager(
        AuthenticationConfiguration config) throws Exception {
    return config.getAuthenticationManager();
}
```

## Q&A

1. **Why is `SecurityContext` stored in a `ThreadLocal`?**
Each HTTP request runs on its own thread, so a `ThreadLocal` gives a per-request slot any downstream component (controller, service, `@PreAuthorize`) can read without passing it through method arguments. It is cleared at the end of the request.

2. **Why is your `User` entity not a `UserDetails`?**
`UserDetails` is Spring Security's contract; your entity is a JPA row. Implementing the interface on the entity couples the persistence model to the framework, so you convert inside `UserDetailsService` instead.

3. **What is the role of `AuthenticationManager` vs `AuthenticationProvider`?**
`AuthenticationManager` is the entry point (`authenticate`). Its default `ProviderManager` delegates to a list of `AuthenticationProvider`s, each handling one auth type (DAO, LDAP, OAuth2); the first that `supports()` the request wins.

4. **What does `DaoAuthenticationProvider` actually do?**
It calls `UserDetailsService.loadUserByUsername()` to load the user, then `PasswordEncoder.matches()` to compare the submitted raw password against the stored hash, returning an authenticated `Authentication` on success or throwing `BadCredentialsException`.

5. **Why `BCryptPasswordEncoder` over a fast hash like SHA-256?**
bcrypt is deliberately slow, salts each password automatically, and exposes a tunable cost factor. SHA-256 is built to be fast, which is exactly what makes brute-forcing stolen hashes cheap.
