> Authentication is triggered from inside the security filter chain — typically by `UsernamePasswordAuthenticationFilter` (form login) or a custom auth filter like `JwtAuthFilter`. For where these filters sit in the request pipeline, see [5.2 Spring Security Filter Chain](<5.2. Spring Security Filter Chain.md>).

## SecurityContext and Authentication

The per-thread security state model — `SecurityContext` is the holder, `Authentication` is the object it holds.

### `SecurityContext`

- Holds the currently authenticated user
- Stored in `ThreadLocal` — each request thread has its own
- Access anywhere:

```java
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String email = auth.getName();
```

### `Authentication` object

Represents the authenticated user. Contains:

- **Principal**: who (email/username)
- **Credentials**: password (usually erased after authentication)
- **Authorities**: roles/permissions (`ROLE_USER`, `ROLE_ADMIN`)

## UserDetails and UserDetailsService

- **`UserDetails`**: Spring Security's interface for representing a user. Methods: `getUsername()`, `getPassword()`, `getAuthorities()`. Your database `User` entity is NOT a `UserDetails` — you must convert it.
- **`UserDetailsService`**: interface with one method: `loadUserByUsername(String username)`. The bridge between your database and Spring Security; Spring calls this during authentication. You implement it, fetch from DB, return `UserDetails`.
- Built-in implementations: `InMemoryUserDetailsManager` (testing/demos) or your custom impl (`CustomUserDetailsService`) fetching from the database.

```java
@Service
@RequiredArgsConstructor
public class CustomUserDetailsService implements UserDetailsService {

    private final UserRepository userRepository;

    @Override
    public UserDetails loadUserByUsername(String email)
            throws UsernameNotFoundException {

        User user = userRepository.findByEmail(email)
            .orElseThrow(() ->
                new UsernameNotFoundException("Invalid credentials"));

        return org.springframework.security.core.userdetails.User.builder()
            .username(user.getEmail())
            .password(user.getPassword())
            .authorities(
                user.getRoles().stream()
                    .map(SimpleGrantedAuthority::new)
                    .collect(Collectors.toList())
            )
            .build();
    }
}
```

> Method is `loadUserByUsername` but we pass email — Spring doesn't care what the value represents, it just passes whatever the user provides during login.

## PasswordEncoder

- Encodes and verifies passwords
- **NEVER store plain text passwords**: Methods: `encode()` — hash a raw password, `matches()` — compare raw vs encoded
- Always use `BCryptPasswordEncoder`

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12);
}
```

> Cost factor 12 is recommended for production (default is 10). Higher = slower hashing, more brute-force resistant. Aim for ~250ms per hash on your auth server.

## AuthenticationManager and AuthenticationProvider

```
AuthenticationManager
  │  Authentication authenticate(Authentication obj);
  │
  ▼  default impl
ProviderManager
  │  holds: List<AuthenticationProvider> providers;
  │  loops the list, calls the first one that supports the auth type
  │
  ▼  delegates to
AuthenticationProvider
  │  boolean supports();
  │  Authentication authenticate(Authentication obj);
  │
  ├──► DaoAuthenticationProvider (username/password — works with any UserDetailsService)
  ├──► OAuth2LoginAuthenticationProvider (OAuth2 login flow)
  ├──► JwtAuthenticationProvider (JWT-bearer / resource server)
  └──► ...
```

- **AuthenticationManager**: interface that orchestrates authentication
- **ProviderManager**: the default implementation, holds a list of `AuthenticationProvider`s and loops through them
- **AuthenticationProvider**: does the actual authentication. Each type supports a different authentication mechanism
- **DaoAuthenticationProvider**: the one we use. It calls `UserDetailsService` to fetch user details and `PasswordEncoder` to verify the password

> `InMemoryUserDetailsManager` is a `UserDetailsService` (in-memory user store), **not** an `AuthenticationProvider`. It plugs into `DaoAuthenticationProvider` like any other `UserDetailsService` — the provider doesn't care whether users come from a DB or memory.

Expose the manager as a bean so controllers and services can inject and call it directly:

```java
@Bean
public AuthenticationManager authenticationManager(
        AuthenticationConfiguration config) throws Exception {
    return config.getAuthenticationManager();
}
```

### Full Authentication Flow (`DaoAuthenticationProvider`)

```
authenticationManager.authenticate(email, password)
       │
       ▼
ProviderManager loops through providers
       │
       ▼
DaoAuthenticationProvider selected (supports username/password)
       │
       ├──► UserDetailsService.loadUserByUsername(email)
       │           │
       │           ▼
       │    UserRepository.findByEmail(email)
       │           │
       │           ▼
       │    User Entity → converted to → UserDetails
       │
       ├──► PasswordEncoder.matches(rawPassword, encodedPassword)
       │
       ▼
 Match?
  ├── Yes → returns Authentication object (with principal + authorities)
  └── No  → throws AuthenticationException
```

## Q&A

1. **Why is `SecurityContext` stored in a `ThreadLocal`?**
   Each HTTP request runs on its own thread, so `ThreadLocal` gives a per-request slot that any downstream component (controller, service, `@PreAuthorize` check) can read without threading it through method arguments. Cleared at the end of the request.

2. **Why is your `User` entity not a `UserDetails`?**
   `UserDetails` is Spring Security's contract (`getUsername`, `getPassword`, `getAuthorities`, plus account-state flags). Your entity is a JPA row. Mixing them couples the persistence model to the security framework — you convert instead, usually inside `UserDetailsService`.

3. **What's the role of `AuthenticationManager` vs `AuthenticationProvider`?**
   `AuthenticationManager` is the entry point (one method: `authenticate`). `ProviderManager` — the default — delegates to a list of `AuthenticationProvider`s, each handling a specific auth type (DAO, LDAP, OAuth2). First provider that `supports()` the request wins.

4. **What does `DaoAuthenticationProvider` actually do?**
   Calls `UserDetailsService.loadUserByUsername()` to fetch the user, then uses `PasswordEncoder.matches()` to compare the submitted raw password against the stored hash. Returns an authenticated `Authentication` on success, throws `BadCredentialsException` otherwise.

5. **Why use `BCryptPasswordEncoder` over plain hashing (SHA-256)?**
   bcrypt is intentionally slow, adds a per-password salt automatically, and has a tunable cost factor. SHA-256 is designed to be fast — exactly what you don't want for password storage, because it makes brute-force cheap.

6. **Why does `UserDetailsService.loadUserByUsername` take the user-provided value even if we use email?**
   The method parameter is a string — Spring doesn't inspect its meaning. Whatever your login DTO submits (email, username, phone) is what Spring passes to your implementation.

| `SecurityContextHolder` / `SecurityContext` | Per-thread holder of authenticated user |
