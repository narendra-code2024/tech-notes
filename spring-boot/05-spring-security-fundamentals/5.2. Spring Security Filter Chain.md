## Adding Spring Security

Just add the dependency — Spring Boot auto-configures security with sensible defaults:

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>
```

With no further configuration, Spring Security applies a strong-default lockdown:

- Every endpoint is protected — no anonymous access
- An auto-generated login page is served at `/login`, and a default user `user` is created with a random password printed to the startup console
- Common web attacks (CSRF, session hijacking) are blocked by default
- Login, logout, and HTTP Basic auth all work out of the box

All of this is enforced by a chain of filters Spring inserts in front of every request.

---

## Filter Chain Architecture

Spring plugs into the Servlet container as a single filter, which delegates to a Spring-managed chain of ~12–15 security filters — each handling one concern (CSRF, authentication, authorization, etc.). If any filter rejects, the chain stops; `ExceptionTranslationFilter` converts the rejection into what the client sees — a 401, a 403, or a redirect to the login page.

```
Client (HTTP Request)
       │
       ▼
Servlet Container (Tomcat)
       │
       ├── Filter-0
       ├── Filter-1
       │
       ├── DelegatingFilterProxy
       │        │
       │        ▼
       │   FilterChainProxy
       │        │
       │        ▼
       │   SecurityFilterChain
       │        ├── DisableEncodeUrlFilter
       │        ├── SecurityContextHolderFilter
       │        ├── CorsFilter
       │        ├── CsrfFilter
       │        ├── LogoutFilter
       │        ├── JwtAuthFilter (custom)
       │        ├── UsernamePasswordAuthenticationFilter
       │        ├── ExceptionTranslationFilter
       │        └── AuthorizationFilter
       │
       ├── Filter-2
       ▼
DispatcherServlet → Controller
```

**Startup wiring (once at boot)**: `SecurityFilterAutoConfiguration` is a Spring Boot auto-config class (triggered by `spring-boot-starter-security` on the classpath) that registers `DelegatingFilterProxy` with the Servlet container and wires it to delegate to `FilterChainProxy` in the Spring context. This wiring runs once at startup, so it isn't in the diagram above.

**Runtime path (per request, top-down through the diagram)**: `DelegatingFilterProxy` — a regular Servlet filter that **delegates** to `FilterChainProxy` (a Spring bean). Bridges the Servlet spec (which creates filters) and the Spring context (which owns the security beans).
- `FilterChainProxy` — picks the matching `SecurityFilterChain` (you can have multiple — e.g., `/api/**` vs `/admin/**`) and runs its filters in order.
- `SecurityFilterChain` — the ordered list of security filters for a path pattern. Each filter handles one concern; if any rejects, downstream filters don't run.
- After security filters pass, the request continues through remaining Servlet filters to `DispatcherServlet`.

> The custom `JwtAuthFilter` shown above is implemented in [5.7 JWT Implementation](<5.7. JWT Implementation.md>). This file focuses on the chain mechanics.

### Key Filters (in order)

|Filter|Purpose|
|---|---|
|`DisableEncodeUrlFilter`|Prevents session IDs from leaking into URLs|
|`SecurityContextHolderFilter`|Loads `SecurityContext` from the repository (session) at request start (replaces the deprecated `SecurityContextPersistenceFilter` in Spring Security 6+)|
|`CorsFilter`|Handles CORS preflight + adds `Access-Control-*` headers (only present when CORS is configured)|
|`CsrfFilter`|Validates CSRF tokens on POST/PUT/DELETE|
|`LogoutFilter`|Matches `POST /logout`, invalidates session, clears cookies|
|`JwtAuthFilter` (custom)|Typical custom slot — extracts and validates the bearer token, populates `SecurityContext`|
|`UsernamePasswordAuthenticationFilter`|Handles form login (POST /login)|
|`ExceptionTranslationFilter`|Catches security exceptions, redirects to login or returns 403|
|`AuthorizationFilter`|Final gate — checks if user has permission|

## Q&A

1. **What is `DelegatingFilterProxy` and why is it needed?**
   It's a regular Servlet filter registered with the container that *delegates* to a Spring-managed bean (`FilterChainProxy`). This bridges the gap between the Servlet spec (which creates filters) and the Spring context (which owns the security beans).

2. **How many filters does Spring Security add?**
   12–15 by default, varying with config (CORS, OAuth2, headers, method security can add or remove filters). Each handles one concern — CSRF, session, auth, exception translation, authorization. They run in a fixed order for every request hitting a matching path.

3. **What's the difference between `FilterChainProxy` and `SecurityFilterChain`?**
   `FilterChainProxy` is the single Servlet-level entry point that picks the matching `SecurityFilterChain` (you can have multiple — e.g., one for `/api/**`, another for `/admin/**`) and runs its filters in order.

4. **Why extend `OncePerRequestFilter` for custom filters?**
   It guarantees the filter runs exactly once per request, even through internal forwards/includes. Without it, a dispatch can re-trigger your auth logic.

5. **What happens if you add the `spring-boot-starter-security` dependency with no config?**
   Every endpoint becomes protected, a default `/login` form is shown, a user `user` is created with a console-printed random password, CSRF is on, and HTTP Basic is enabled.

## Quick Reference

| Component | Role |
|---|---|
| `DelegatingFilterProxy` | Servlet filter registered with the container; delegates to a Spring bean |
| `FilterChainProxy` | Spring bean — picks the matching `SecurityFilterChain` and runs its filters |
| `SecurityFilterChain` | Ordered list of security filters for a path pattern (multiple chains supported) |
| `SecurityContextHolderFilter` | Loads `SecurityContext` from the session at request start (Spring Security 6+) |
| `OncePerRequestFilter` | Base class for custom filters — guarantees once-per-request execution |
| Add custom filter | `addFilterBefore(...)` / `addFilterAfter(...)` in `SecurityFilterChain` config |
