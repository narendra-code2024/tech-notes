Spring Security secures an application through an ordered chain of servlet filters inserted in front of every request. Each filter owns one concern (CSRF, authentication, authorization), and the first filter to reject a request stops the chain.

---

## Adding Spring Security

Adding the starter is enough; Spring Boot auto-configures security with a strong-default lockdown.

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>
```

With no further configuration:

- Every endpoint is protected; no anonymous access.
- An auto-generated login page is served at `/login`, and a default user `user` is created with a random password printed to the startup console.
- Common web attacks (CSRF, session fixation) are blocked by default.
- Login, logout, and HTTP Basic all work out of the box.

> **Note**: `WebSecurityConfigurerAdapter` was removed in Spring Security 6. Configuration is now component-based: publish a `SecurityFilterChain` `@Bean`. See [5.4. Filter Chain Configuration.md](<5.4. Filter Chain Configuration.md>).

---

## Filter Chain Architecture

Spring plugs into the servlet container as a single filter that delegates to a Spring-managed chain of roughly 12 to 15 security filters. If any filter rejects, the chain stops and `ExceptionTranslationFilter` converts the rejection into what the client sees: a 401, a 403, or a redirect to the login page.

At startup, `SecurityFilterAutoConfiguration` (triggered by the starter on the classpath) registers `DelegatingFilterProxy` with the servlet container and wires it to `FilterChainProxy` in the Spring context. This runs once at boot, so it is not on the per-request path below.

Per request, top-down through the diagram:

- **`DelegatingFilterProxy`**: a regular servlet filter that delegates to `FilterChainProxy` (a Spring bean), bridging the servlet spec (which creates filters) and the Spring context (which owns the security beans).
- **`FilterChainProxy`**: picks the matching `SecurityFilterChain` (you can have several, e.g. `/api/**` vs `/admin/**`) and runs its filters in order.
- **`SecurityFilterChain`**: the ordered filter list for a path pattern; if any filter rejects, downstream filters do not run.

After the security filters pass, the request continues through remaining servlet filters to `DispatcherServlet`.

### Key Filters (in order)

| Filter                                 | Purpose                                                                                                  |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| `DisableEncodeUrlFilter`               | Prevents session IDs from leaking into URLs.                                                             |
| `SecurityContextHolderFilter`          | Loads `SecurityContext` from the session at request start (replaced `SecurityContextPersistenceFilter` in SS6). |
| `CorsFilter`                           | Handles CORS preflight and adds `Access-Control-*` headers (present only when CORS is configured).       |
| `CsrfFilter`                           | Validates CSRF tokens on state-changing requests (POST/PUT/DELETE).                                      |
| `LogoutFilter`                         | Matches `POST /logout`, invalidates the session, clears cookies.                                         |
| `JwtAuthFilter` (custom)               | Typical custom slot; extracts and validates the bearer token, populates `SecurityContext`.              |
| `UsernamePasswordAuthenticationFilter` | Handles form login (`POST /login`).                                                                      |
| `ExceptionTranslationFilter`           | Catches security exceptions, redirects to login or returns 403.                                          |
| `AuthorizationFilter`                  | Final gate; checks authority (replaced `FilterSecurityInterceptor` in SS6).                             |

Insert a custom filter with `addFilterBefore(...)` / `addFilterAfter(...)` relative to an existing filter, and extend `OncePerRequestFilter` so it runs exactly once even through internal forwards.

> **Reference**: the custom `JwtAuthFilter` shown above is implemented in [5.7. JWT Implementation.md](<5.7. JWT Implementation.md>).

## Q&A

1. **Why is `DelegatingFilterProxy` needed instead of registering security filters directly?**
The servlet container instantiates filters and knows nothing of the Spring context. `DelegatingFilterProxy` is a container-registered filter that delegates to the Spring-managed `FilterChainProxy`, so the security filters can be Spring beans with full dependency injection.

2. **What is the difference between `FilterChainProxy` and `SecurityFilterChain`?**
`FilterChainProxy` is the single servlet-level entry point that selects the first matching `SecurityFilterChain` for the request path; each `SecurityFilterChain` is one ordered filter list scoped to a path pattern, and an app can define several.

3. **Why extend `OncePerRequestFilter` for a custom filter?**
It guarantees the filter body runs once per request even across internal forwards or includes, preventing auth logic from re-triggering on a dispatch.

4. **What does adding the starter with zero config give you?**
Every endpoint protected, a `/login` form, a generated `user` with a console-printed password, CSRF enabled, and HTTP Basic.
