`HttpSecurity` is Spring Security's fluent builder for the `SecurityFilterChain` bean; every option (CSRF, session policy, URL authorization, custom filters, exception handling, headers, CORS) flows through it. This note is a reference for the available options, not a tutorial for one auth pattern. Concrete patterns live in their own files:

- Session-based auth (form login, API-style with `JSESSIONID`): [5.5. Session-Based Authentication.md](https://claude.ai/chat/5.5.%20Session-Based%20Authentication.md)
- JWT auth (stateless, token in header/cookie): [5.7. JWT Implementation.md](https://claude.ai/chat/5.7.%20JWT%20Implementation.md)

## SecurityFilterChain Bean Setup

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http)
            throws Exception {

        http
            // ... configure here ...
            ;

        return http.build();
    }
}
```

- **`@EnableWebSecurity`**: activates Spring Security's web configuration. Implicit when the starter is on the classpath; declaring it makes intent explicit.
- **`@Bean SecurityFilterChain`**: each `HttpSecurity` config produces one chain. Multiple beans produce multiple chains (see Multiple SecurityFilterChain Beans).
- **`HttpSecurity` is mutable and fluent**: each call (`csrf(...)`, `sessionManagement(...)`) configures one part; ordering between calls does not matter for most options.

## CSRF Configuration

Keep CSRF on for browser clients that authenticate with session cookies; disable it when an auth header (not a cookie) carries identity.

```java
.csrf(Customizer.withDefaults())                                   // default: keep on (session cookie auth)
.csrf(csrf -> csrf.disable())                                      // JWT in Authorization header, no cookie
.csrf(csrf -> csrf
    .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())) // SPA on session auth
```

With the cookie repository, the token lives in an `XSRF-TOKEN` cookie that JS reads and echoes as the `X-XSRF-TOKEN` header on each state-changing request.

> **Note**: in Spring Security 6 the default `XorCsrfTokenRequestAttributeHandler` (BREACH protection) encodes the token, so the raw cookie value no longer matches the submitted header. SPAs using `CookieCsrfTokenRepository` must set `csrf.csrfTokenRequestHandler(new CsrfTokenRequestAttributeHandler())` so the cookie and header values align.

> **Reference**: CSRF concepts and prevention methods are in [5.1. Security Concepts and Attacks.md](https://claude.ai/chat/5.1.%20Security%20Concepts%20and%20Attacks.md).

## Session Management

Controls whether Spring Security creates or reuses an `HttpSession` to store the `SecurityContext`.

|Policy|Behavior|When to use|
|---|---|---|
|`IF_REQUIRED`|Creates a session when needed, typically on first authentication.|Form login, API-style session auth (default).|
|`ALWAYS`|Forces session creation on every request, even when anonymous.|Server-rendered apps needing a session early (rare).|
|`NEVER`|Will not create a session but reuses one that already exists.|Mixed setups where a session may come from another source.|
|`STATELESS`|No session ever; `SecurityContext` is per-request only.|JWT auth, fully stateless APIs.|

```java
.sessionManagement(session ->
    session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
```

> **Reference**: session fixation, concurrent-session limits, and timeout are covered in [5.5. Session-Based Authentication.md](https://claude.ai/chat/5.5.%20Session-Based%20Authentication.md).

## URL Authorization (`authorizeHttpRequests`)

Per-path authorization rules, evaluated top-down with first match wins, so order from most-specific to most-general and end with `anyRequest()`.

```java
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/", "/login", "/register").permitAll()
    .requestMatchers(HttpMethod.GET, "/api/posts/**").permitAll()
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/users/**").hasAnyRole("USER", "ADMIN")
    .requestMatchers("/api/internal/**").hasAuthority("INTERNAL_SERVICE")
    .anyRequest().authenticated()
)
```

### Matchers and rules

|`requestMatchers` form|Match|
|---|---|
|`requestMatchers("/path")`|Exact path.|
|`requestMatchers("/api/**")`|Path prefix (Ant pattern).|
|`requestMatchers(HttpMethod.GET, "/api/posts/**")`|Method + path.|
|`requestMatchers(new RegexRequestMatcher(...))`|Regex.|
|`requestMatchers("/a", "/b", "/c")`|Multiple paths sharing one rule.|

|Rule|Means|
|---|---|
|`.permitAll()`|Anyone, authenticated or not.|
|`.denyAll()`|Nobody; endpoint exists but is locked.|
|`.authenticated()`|Any authenticated user.|
|`.hasRole("ADMIN")`|Has `ROLE_ADMIN` (Spring auto-prefixes `ROLE_`).|
|`.hasAnyRole("USER", "ADMIN")`|Has any listed role.|
|`.hasAuthority("PERM_X")`|Has the literal authority string (no prefix added).|
|`.access(authzManager)`|Custom `AuthorizationManager` for full SpEL/programmatic logic.|

## Exception Handling

Customize what happens when authentication is missing (401) or insufficient (403). This matters for APIs because Spring's browser default redirects to a login page, which is wrong for JSON clients.

```java
.exceptionHandling(ex -> ex
    .authenticationEntryPoint((req, res, authEx) -> {          // no authentication present
        res.setStatus(HttpStatus.UNAUTHORIZED.value());
        res.setContentType("application/json");
        res.getWriter().write("{\"error\":\"Unauthorized\"}");
    })
    .accessDeniedHandler((req, res, deniedEx) -> {             // authenticated but lacks authority
        res.setStatus(HttpStatus.FORBIDDEN.value());
        res.setContentType("application/json");
        res.getWriter().write("{\"error\":\"Forbidden\"}");
    })
)
```

## Built-in Login Mechanisms

Spring offers two built-in flows; both fit narrow contexts and are usually skipped in apps with a frontend.

- **`.formLogin()`**: Spring serves an HTML login page. Use only for server-rendered apps (Thymeleaf, JSP).
- **`.httpBasic()`**: browser credential popup. Use for internal admin endpoints (Actuator, debug consoles) and machine-to-machine over trusted networks. Avoid for end users: no logout UX, and the password is retransmitted on every request.

> **Reference**: SPA and mobile apps skip both; see [5.5. Session-Based Authentication.md](https://claude.ai/chat/5.5.%20Session-Based%20Authentication.md) for the cookie path and [5.7. JWT Implementation.md](https://claude.ai/chat/5.7.%20JWT%20Implementation.md) for the token path.

## Custom Filter Insertion

`HttpSecurity` splices your own filter into the chain at a position relative to a Spring default.

|Method|Behavior|
|---|---|
|`addFilterBefore(F, ClassRef)`|Insert `F` immediately before `ClassRef` runs.|
|`addFilterAfter(F, ClassRef)`|Insert `F` immediately after `ClassRef` runs.|
|`addFilterAt(F, ClassRef)`|Place `F` at the same position (both run; use carefully).|

The common case is a JWT filter inserted before form login:

```java
.addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)
```

The JWT filter populates `SecurityContext` from a valid token before the form-login filter runs, so protected endpoints see an authenticated user and form login never engages.

## Multiple SecurityFilterChain Beans

Different URL groups often need different policies (public pages, API, admin, Actuator). Define several `@Bean SecurityFilterChain` methods scoped by `securityMatcher`:

```java
@Bean
@Order(1)
SecurityFilterChain apiChain(HttpSecurity http) throws Exception {
    return http.securityMatcher("/api/**")
        .csrf(csrf -> csrf.disable())
        .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        .authorizeHttpRequests(a -> a.anyRequest().authenticated())
        .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)
        .build();
}

@Bean
@Order(2)
SecurityFilterChain webChain(HttpSecurity http) throws Exception {
    return http.securityMatcher("/**")
        .authorizeHttpRequests(a -> a
            .requestMatchers("/", "/login").permitAll()
            .anyRequest().authenticated())
        .formLogin(Customizer.withDefaults())
        .build();
}
```

- **`@Order`**: lower number wins; Spring tries each chain in order and the first whose `securityMatcher` matches handles the request.
- **`securityMatcher("/api/**")`**: scopes the chain to a path; without it the chain matches every request.
- **Common split**: `/api/**` (JWT, stateless) + `/admin/**` (form login, session) + `/actuator/**` (Basic auth on an internal port).

## Security Headers (`.headers`)

Spring sets a sensible default set automatically (`X-Content-Type-Options: nosniff`, `X-Frame-Options`, cache-control). Customize via `.headers(...)`:

```java
.headers(headers -> headers
    .frameOptions(frame -> frame.sameOrigin())
    .contentSecurityPolicy(csp -> csp
        .policyDirectives("default-src 'self'; script-src 'self' 'unsafe-inline'"))
    .httpStrictTransportSecurity(hsts -> hsts
        .includeSubDomains(true)
        .maxAgeInSeconds(31536000))
    .referrerPolicy(rp -> rp
        .policy(ReferrerPolicy.STRICT_ORIGIN_WHEN_CROSS_ORIGIN))
)
```

|Header|Purpose|Common setting|
|---|---|---|
|HSTS|Force HTTPS for repeat visits.|`max-age=31536000; includeSubDomains` (1 year).|
|CSP|Restrict what scripts/styles can load.|`default-src 'self'` baseline, then tighten.|
|X-Frame-Options|Prevent clickjacking via iframe.|`SAMEORIGIN` or `DENY`.|
|X-Content-Type-Options|Prevent MIME sniffing.|`nosniff` (Spring default).|
|Referrer-Policy|Limit URL info shared cross-site.|`strict-origin-when-cross-origin`.|

## CORS Configuration

The backend explicitly allows specific origins to read its responses. Prefer the bean-based global config: it is auditable in one place, consistent across controllers, and integrates with the security filters.

```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.cors(cors -> cors.configurationSource(corsConfigurationSource()));
    // ... rest of config ...
    return http.build();
}

@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(List.of("https://app.example.com"));
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE"));
    config.setAllowedHeaders(List.of("Authorization", "Content-Type"));
    config.setAllowCredentials(true);   // required for cookie-based auth across origins

    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", config);
    return source;
}
```

Per-controller `@CrossOrigin(origins = "https://app.example.com")` on a `@RestController` works for one-off cases only.

> **Note**: with `allowCredentials(true)` you cannot use `*` for `allowedOrigins` (browsers reject it); use `setAllowedOriginPatterns(List.of("https://*.example.com"))` for subdomain wildcards.

> **Reference**: CORS concepts (SOP, preflight) are in [5.1. Security Concepts and Attacks.md](https://claude.ai/chat/5.1.%20Security%20Concepts%20and%20Attacks.md).

## Q&A

1. **What does `@EnableWebSecurity` actually do?** It imports Spring Security's web configuration (`WebSecurityConfiguration`, `HttpSecurityConfiguration`), wiring up `FilterChainProxy`, the default `SecurityFilterChain`, and supporting beans. With the starter on the classpath this is implicit; declaring it makes activation explicit.
    
2. **Why does rule order matter in `authorizeHttpRequests`?** Matching is top-down, first match wins. Putting `.anyRequest().authenticated()` first swallows everything and your `permitAll()` rules are never reached, so order specific-to-general.
    
3. **What is the difference between `hasRole("ADMIN")` and `hasAuthority("ROLE_ADMIN")`?** `hasRole` auto-prefixes `ROLE_`, so the two are equivalent. Use `hasRole` for role-based access and `hasAuthority` for fine-grained permissions (`READ_USER`, `WRITE_REPORT`).
    
4. **When would you use multiple `SecurityFilterChain` beans?** When URL groups need different policies, e.g. `/api/**` JWT-stateless, `/admin/**` form-login session, `/actuator/**` Basic auth on an internal port. Each gets its own chain via `securityMatcher`, ordered by `@Order`.
    
5. **What does `addFilterBefore(F, X.class)` do that `addFilterAt` does not?** `addFilterBefore` runs `F` immediately before `X`'s position; `addFilterAt` runs it at the same position, so both execute and can double-process. `Before` is what you almost always want.
    
6. **Why does Spring Security disable browser caching by default?** Default headers (`Cache-Control: no-cache, no-store, must-revalidate`) stop the browser caching authenticated responses, so a logged-out user cannot hit "back" and see another user's page. Override with `.headers(h -> h.cacheControl(c -> c.disable()))` only for specific caching needs.
    
7. **What is the practical difference between `NEVER` and `STATELESS`?** `NEVER` will not create a session but uses one that already exists (e.g. from a JSP). `STATELESS` ignores sessions entirely, even an existing one. Use `STATELESS` for JWT APIs and `NEVER` for mixed setups.

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Config style|`@EnableWebSecurity` + `@Bean SecurityFilterChain` returning `http.build()`.|
|CSRF|On for cookie auth; `.disable()` for header JWT; cookie repo + handler for SPAs.|
|Session policy|`STATELESS` for JWT; `IF_REQUIRED` default for session auth.|
|Authorization order|Top-down, first match wins; end with `anyRequest()`.|
|Role vs authority|`hasRole("X")` adds `ROLE_`; `hasAuthority("X")` is literal.|
|API exceptions|Custom `authenticationEntryPoint` (401) + `accessDeniedHandler` (403) as JSON.|
|Custom filter|`addFilterBefore(F, UsernamePasswordAuthenticationFilter.class)`.|
|Multiple chains|`@Bean` + `@Order` + `securityMatcher("/api/**")`.|
|CORS|`http.cors()` + `CorsConfigurationSource` bean; patterns for credentialed wildcards.|
|Headers|HSTS, CSP, X-Frame-Options, nosniff via `.headers(...)`.|
