`HttpSecurity` is Spring Security's fluent builder for configuring the `SecurityFilterChain` bean — every option you'd want to set (CSRF, session policy, URL authorization, custom filters, exception handling, headers, CORS) flows through it. This file is a **reference** for the options available, not a tutorial for any one auth pattern. Concrete implementations of each pattern live in their own files:

- **Session-based auth** (form login, API-style with `JSESSIONID`) → [5.5 Session-Based Authentication](<5.5. Session-Based Authentication.md>)
- **JWT auth** (stateless, token in header / cookie) → [5.7 JWT Implementation](<5.7. JWT Implementation.md>) → SecurityConfig for JWT

---

## SecurityFilterChain Bean Setup

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http)
            throws Exception {

        http
            // ... configure here ...
            ;

        return http.build();
    }
}
```

- **`@EnableWebSecurity`**: activates Spring Security's web configuration. Implicit when `spring-boot-starter-security` is on the classpath, but explicit makes intent clear.
- **`@Bean SecurityFilterChain`**: every `HttpSecurity` config produces one chain. Multiple beans → multiple chains (see Multiple SecurityFilterChain Beans below).
- **`HttpSecurity` is mutable + fluent**: each call (`csrf(...)`, `sessionManagement(...)`, etc.) configures one part. Order between calls doesn't matter for most options.

## CSRF Configuration

> Conceptual deep-dive in [5.1 Security Concepts and Attacks](<5.1. Security Concepts and Attacks.md>) → CSRF Prevention Methods. This section covers `HttpSecurity` config only.

### Default — keep CSRF on

```java
.csrf(Customizer.withDefaults())
```

Use when the client is browser-based and uses session cookies for auth.

### Disable

```java
.csrf(csrf -> csrf.disable())
```

Use when **headers carry the auth** (e.g., JWT in `Authorization: Bearer ...`) and no session cookie is involved.

### Cookie-based token (for SPAs on session auth)

```java
.csrf(csrf -> csrf
    .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse()))
```

For session cookie auth + SPA frontend. The token is stored in an `XSRF-TOKEN` cookie that JS reads and echoes back as `X-XSRF-TOKEN` header on each state-changing request.

> **Custom request handler**: `csrfTokenRequestHandler(new CsrfTokenRequestAttributeHandler())` overrides how the token is read from the request (e.g., a non-default header name). Edge case; rarely needed.

## Session Management

Controls whether Spring Security creates / reuses an `HttpSession` to store the `SecurityContext`.

| Policy | Behavior | When to use |
|---|---|---|
| **`IF_REQUIRED`** (default) | Creates a session when needed — typically on first authentication | Form login, API-style session auth, anywhere `SecurityContext` lives in `HttpSession` |
| **`ALWAYS`** | Forces session creation on every request, even if the user is anonymous | Server-rendered apps that need session early (rare; usually IF_REQUIRED is enough) |
| **`NEVER`** | Won't create a session, but reuses one if it already exists | Mixed setups where session may exist from another mechanism |
| **`STATELESS`** | No session ever — `SecurityContext` is per-request only, never persisted | JWT auth, fully stateless APIs |

```java
.sessionManagement(session ->
    session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
```

For session-aware setups, you also configure session fixation, concurrent session limits, and timeout via `.sessionManagement(...)`. See [5.5 Session-Based Authentication](<5.5. Session-Based Authentication.md>) → Production Considerations.

## URL Authorization (`authorizeHttpRequests`)

Per-path authorization rules. **First match wins**: order rules from most-specific to most-general, ending with `anyRequest()` as the catch-all.

```java
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/", "/login", "/register").permitAll()
    .requestMatchers(HttpMethod.GET, "/api/posts/**").permitAll()
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/users/**").hasAnyRole("USER", "ADMIN")
    .requestMatchers("/api/internal/**").hasAuthority("INTERNAL_SERVICE")
    .anyRequest().authenticated()
)
```

### `requestMatchers` patterns

| Form | Match |
|---|---|
| `requestMatchers("/path")` | Exact path |
| `requestMatchers("/api/**")` | Path prefix (Ant pattern) |
| `requestMatchers(HttpMethod.GET, "/api/posts/**")` | Method + path |
| `requestMatchers(new RegexRequestMatcher(...))` | Regex |
| `requestMatchers("/api/a", "/api/b", "/api/c")` | Multiple paths sharing the same rule |

### Authorization rules

| Rule | Means |
|---|---|
| `.permitAll()` | Anyone, authenticated or not |
| `.denyAll()` | Nobody — endpoint exists but is locked down |
| `.authenticated()` | Any authenticated user (any role) |
| `.hasRole("ADMIN")` | Has `ROLE_ADMIN` (Spring auto-prefixes) |
| `.hasAnyRole("USER", "ADMIN")` | Has any of the listed roles |
| `.hasAuthority("PERM_X")` | Has the literal authority string (no auto-prefix) |
| `.hasAnyAuthority("A", "B")` | Has any of the listed authorities |
| `.access(authzManager)` | Custom `AuthorizationManager` — full SpEL or programmatic logic |

## Exception Handling

Customize what happens when authentication is missing (401) or insufficient (403). Critical for APIs — Spring's default redirects to a login page, which is wrong for JSON clients.

```java
.exceptionHandling(ex -> ex
    .authenticationEntryPoint((req, res, authEx) -> {
        res.setStatus(HttpStatus.UNAUTHORIZED.value());
        res.setContentType("application/json");
        res.getWriter().write("{\"error\":\"Unauthorized\"}");
    })
    .accessDeniedHandler((req, res, deniedEx) -> {
        res.setStatus(HttpStatus.FORBIDDEN.value());
        res.setContentType("application/json");
        res.getWriter().write("{\"error\":\"Forbidden\"}");
    })
)
```

- **`authenticationEntryPoint`**: fires when there's NO authentication. Browser default redirects to `/login`; APIs should return JSON 401.
- **`accessDeniedHandler`**: fires when authentication exists but the user lacks the required role/authority. Default returns 403 HTML; APIs should return JSON 403.

## Built-in Login Mechanisms

Spring offers two built-in login flows. Both fit narrow contexts; both are skipped in most production apps with a frontend.

- **`.formLogin()`**: Spring serves an HTML login page. Use only for server-rendered apps (Thymeleaf, JSP).
- **`.httpBasic()`**: browser credential popup. Use for internal admin endpoints (Actuator, debug consoles) and machine-to-machine over trusted networks. Avoid for end-user-facing apps — no logout UX, password retransmitted on every request.

> SPA / mobile apps skip both — see [5.5 Session-Based Authentication](<5.5. Session-Based Authentication.md>) (cookie path) and [5.7 JWT Implementation](<5.7. JWT Implementation.md>) (token path).

## Custom Filter Insertion

`HttpSecurity` lets you splice your own filter into the chain at a specific position relative to a Spring default.

| Method | Behavior |
|---|---|
| `addFilterBefore(F, ClassRef)` | Insert `F` immediately before `ClassRef` runs |
| `addFilterAfter(F, ClassRef)` | Insert `F` immediately after `ClassRef` runs |
| `addFilterAt(F, ClassRef)` | Place `F` at the same position as `ClassRef` (Spring runs both — use carefully) |

Most common use — a JWT filter inserted before form-login:

```java
.addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)
```

The JWT filter populates `SecurityContext` from a valid token before the default form-login filter runs, so protected endpoints see an authenticated user and form login never engages.

## Multiple SecurityFilterChain Beans

Apps often need different security policies for different URL groups — public web pages, API endpoints, admin panels, Actuator. Define multiple `@Bean SecurityFilterChain` methods scoped via `securityMatcher`:

```java
@Bean
@Order(1)
SecurityFilterChain apiChain(HttpSecurity http) throws Exception {
    return http.securityMatcher("/api/**")
        .csrf(csrf -> csrf.disable())
        .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        .authorizeHttpRequests(a -> a.anyRequest().authenticated())
        .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)
        .build();
}

@Bean
@Order(2)
SecurityFilterChain webChain(HttpSecurity http) throws Exception {
    return http.securityMatcher("/**")
        .authorizeHttpRequests(a -> a
            .requestMatchers("/", "/login").permitAll()
            .anyRequest().authenticated())
        .formLogin(Customizer.withDefaults())
        .build();
}
```

- **`@Order`**: lower number wins. Spring tries each chain in order; first one whose `securityMatcher` matches handles the request.
- **`securityMatcher("/api/**")`**: scope the chain to a path. Without it, the chain matches all requests.
- **Common splits**: `/api/**` (JWT, stateless) + `/admin/**` (form login, session) + `/actuator/**` (Basic auth on internal port).

## Security Headers (`.headers`)

Spring sets a sensible default set of security headers automatically (`X-Content-Type-Options`, `X-Frame-Options`, etc.). Customize via `.headers(...)`:

```java
.headers(headers -> headers
    .frameOptions(frame -> frame.sameOrigin())
    .contentSecurityPolicy(csp -> csp
        .policyDirectives("default-src 'self'; script-src 'self' 'unsafe-inline'"))
    .httpStrictTransportSecurity(hsts -> hsts
        .includeSubDomains(true)
        .maxAgeInSeconds(31536000))
    .referrerPolicy(rp -> rp
        .policy(ReferrerPolicy.STRICT_ORIGIN_WHEN_CROSS_ORIGIN))
)
```

| Header | Purpose | Common setting |
|---|---|---|
| **HSTS** | Force HTTPS for repeat visits | `max-age=31536000; includeSubDomains` (1 year) |
| **CSP** | Restrict what scripts/styles can load | `default-src 'self'` baseline; tighten as you go |
| **X-Frame-Options** | Prevent clickjacking via iframe | `SAMEORIGIN` (or `DENY`) |
| **X-Content-Type-Options** | Prevent MIME sniffing | `nosniff` (Spring default) |
| **Referrer-Policy** | Control how much URL info is shared cross-site | `strict-origin-when-cross-origin` |

## CORS Configuration

Cross-Origin Resource Sharing — backend explicitly allows specific origins to read its responses. Conceptual breakdown in [5.1 Security Concepts and Attacks](<5.1. Security Concepts and Attacks.md>) → CORS.

### Global CORS via `CorsConfigurationSource`

```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.cors(cors -> cors.configurationSource(corsConfigurationSource()));
    // ... rest of config ...
    return http.build();
}

@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(List.of("https://app.example.com"));
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE"));
    config.setAllowedHeaders(List.of("Authorization", "Content-Type"));
    config.setAllowCredentials(true);   // required for cookie-based auth across origins

    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", config);
    return source;
}
```

### Per-controller via `@CrossOrigin`

```java
@RestController
@CrossOrigin(origins = "https://app.example.com")
public class EmployeeController { ... }
```

**Prefer the bean-based global config**: easier to audit (one place), consistent across controllers, and works correctly with security filters. Use `@CrossOrigin` only for one-off cases.

> **Production warning — `setAllowedOriginPatterns` for credentials + wildcards.** When `allowCredentials(true)` is set, you can't use `*` for `allowedOrigins` (browsers reject it). Use `setAllowedOriginPatterns(List.of("https://*.example.com"))` for subdomain wildcards.

## Q&A

1. **What does `@EnableWebSecurity` actually do?**
   It imports Spring Security's web configuration (`WebSecurityConfiguration`, `HttpSecurityConfiguration`, etc.), which wire up the `FilterChainProxy`, default `SecurityFilterChain`, and supporting beans. With `spring-boot-starter-security` on the classpath, this is implicit — adding `@EnableWebSecurity` makes the activation explicit.

2. **Why does rule order matter in `authorizeHttpRequests`?**
   Spring matches rules **top-down, first match wins**. If you put `.anyRequest().authenticated()` first, it swallows everything and your `.requestMatchers(...).permitAll()` rules are never reached. Order specific-to-general.

3. **What's the difference between `hasRole("ADMIN")` and `hasAuthority("ROLE_ADMIN")`?**
   `hasRole` auto-prefixes with `ROLE_`, so `hasRole("ADMIN")` is equivalent to `hasAuthority("ROLE_ADMIN")`. `hasAuthority` takes the literal string. Use `hasRole` for role-based access; `hasAuthority` for fine-grained permissions (`READ_USER`, `WRITE_REPORT`, etc.).

4. **When would you use multiple `SecurityFilterChain` beans?**
   When different URL groups need different policies — e.g., `/api/**` is JWT-stateless, `/admin/**` is form-login session-based, `/actuator/**` is Basic auth on an internal port. Each gets its own chain via `securityMatcher`, ordered by `@Order`.

5. **What does `addFilterBefore(F, X.class)` do that `addFilterAt` doesn't?**
   `addFilterBefore` places your filter **immediately before** the position where Spring would run `X`. `addFilterAt` places it at the *same* position — both filters run, which can cause double-processing. `Before` is what you almost always want.

6. **Why does Spring Security disable browser caching by default?**
   Default response headers (`Cache-Control: no-cache, no-store, must-revalidate`) prevent the browser from caching authenticated responses — otherwise a logged-out user could hit "back" and see another user's cached page. Override with `.headers(h -> h.cacheControl(c -> c.disable()))` only if you have specific caching requirements.

7. **What's the practical difference between `SessionCreationPolicy.NEVER` and `STATELESS`?**
   `NEVER` won't create a session but will use one if some other mechanism already created it (e.g., JSP `<%@ page session="true" %>`). `STATELESS` won't touch sessions at all — `SecurityContext` is per-request only, ignored even if a session exists. Use `STATELESS` for fully stateless APIs (JWT); `NEVER` for mixed setups.

## Quick Reference

| Config | Purpose |
|---|---|
| `@EnableWebSecurity` + `@Bean SecurityFilterChain` | The bean pattern for declaring a security policy |
| `.csrf(...)` | `Customizer.withDefaults()` (session) / `.disable()` (JWT header) / `CookieCsrfTokenRepository` (SPA + session) |
| `SessionCreationPolicy` | `STATELESS` (JWT) / `IF_REQUIRED` default (session auth) |
| `.authorizeHttpRequests(...)` | Per-path rules; first match wins; end with `.anyRequest()` |
| Authorization rules | `.permitAll()` / `.denyAll()` / `.authenticated()` / `.hasRole("X")` / `.hasAuthority("X")` |
| `.exceptionHandling(...)` | Custom 401 entry point + 403 access-denied handler |
| `.addFilterBefore(F, Class.class)` | Insert custom filter ahead of a Spring default |
| Multiple chains | `@Bean` + `@Order` + `securityMatcher("/api/**")` |
| `.headers(...)` | HSTS, CSP, X-Frame-Options, etc. |
| `.cors(c -> c.configurationSource(...))` | CORS via `CorsConfigurationSource` bean |
