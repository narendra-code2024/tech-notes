Session-based auth is Spring Security's **default**: server creates an `HttpSession`, browser holds a `JSESSIONID` cookie, every subsequent request carries it. No token in the response body; the cookie *is* the auth. Two flavors of session-based auth in Spring Boot:

- **Form login** (`.formLogin()`) — Spring Boot serves the HTML (Thymeleaf, JSP). Built-in or custom login page; Spring intercepts the form POST.
- **API-style login** (custom `/api/auth/login` controller) — Spring Boot is the backend; SPA / mobile owns the login UI and POSTs JSON. Your controller calls `AuthenticationManager` directly.

Both paths use the same `SecurityContext` + `JSESSIONID` machinery — the difference is who handles the login form.

---

## When to Choose Session Auth

Pick session-based auth when:

- **Single server or sticky-session topology**: server-side session is simpler than coordinating JWT secrets across nodes
- **Server-rendered web app** (Thymeleaf, JSP) — the browser's `JSESSIONID` cookie just works
- **Easy server-side invalidation matters**: kicking a user out is a session-store delete; JWT can't be revoked without a blocklist
- **Internal admin tools**: small user base, low coordination overhead, simpler ops

## `UserDetailsService` — Where the User Comes From

`UserDetailsService` is the bridge between your user store and Spring Security — see [5.3 Authentication Core Components](<5.3. Authentication Core Components.md>) → `UserDetails` and `UserDetailsService` for the canonical coverage and the database-backed `CustomUserDetailsService` example. Two common choices: in-memory (for testing / early dev — shown below) or DB-backed.

### `InMemoryUserDetailsManager` — for testing / early dev

```java
@Bean
public UserDetailsService userDetailsService(PasswordEncoder encoder) {
    UserDetails admin = User.builder()
        .username("admin")
        .password(encoder.encode("admin123"))
        .roles("ADMIN")
        .build();

    UserDetails user = User.builder()
        .username("user")
        .password(encoder.encode("user123"))
        .roles("USER")
        .build();

    return new InMemoryUserDetailsManager(admin, user);
}
```

Good for early development, unit tests, internal tools with a fixed user list. Lost on restart — never use for real users.

## Form Login Configuration

The session-based `SecurityFilterChain` is the inverse of the JWT one — CSRF stays **on**, sessions are created **as needed**, no custom auth filter:

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http)
            throws Exception {

        http
            .csrf(Customizer.withDefaults())
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/", "/login", "/register", "/css/**", "/js/**").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .formLogin(form -> form
                .loginPage("/login")                        // GET /login — your Thymeleaf page
                .loginProcessingUrl("/login")               // POST endpoint Spring intercepts (defaults to loginPage)
                .defaultSuccessUrl("/dashboard", true)
                .failureUrl("/login?error=true")
                .usernameParameter("email")                 // form input name (default: "username")
                .passwordParameter("password")              // form input name (default: "password")
                .permitAll()
            )
            .logout(logout -> logout
                .logoutUrl("/logout")                       // POST only — GET is CSRF-vulnerable (<img src="/logout">)
                .logoutSuccessUrl("/login?logout=true")     // redirect after success
                .invalidateHttpSession(true)                // destroy server-side session
                .deleteCookies("JSESSIONID")                // clear browser cookie
                .clearAuthentication(true)                  // clear SecurityContext
                .permitAll()
            );

        return http.build();
    }
}
```

**Default page (zero config) — `.formLogin(Customizer.withDefaults())`** serves a basic HTML form at `/login` (also responds to `POST /login`). Useful for early development and admin tools — no styling, plain Spring-generated.

### Success / failure handlers

For finer control (return JSON for AJAX login, log audit events, set custom cookies):

```java
.formLogin(form -> form
    .successHandler((request, response, auth) -> {
        // e.g., JSON response for AJAX form
        response.setContentType("application/json");
        response.getWriter().write("{\"status\":\"ok\"}");
    })
    .failureHandler((request, response, ex) -> {
        response.setStatus(HttpStatus.UNAUTHORIZED.value());
        response.getWriter().write("{\"error\":\"" + ex.getMessage() + "\"}");
    })
)
```

### Remember-Me

Keep the user logged in across browser restarts. By default `JSESSIONID` is a **session cookie** (no `Max-Age` set), so the browser drops it on close — the user would normally have to re-authenticate. Remember-Me adds a **separate persistent cookie** with explicit `Max-Age` that survives browser close and re-authenticates automatically on the next visit:

```java
.rememberMe(rm -> rm
    .key("a-secret-string-used-to-sign-the-token")
    .tokenValiditySeconds(14 * 24 * 60 * 60)   // 2 weeks
    .userDetailsService(userDetailsService)
)
```

On successful login (with the "remember me" checkbox), Spring sets a signed `remember-me` cookie. On a later request with no session, `RememberMeAuthenticationFilter` validates the cookie and re-authenticates without prompting. Logout clears it.

## API-Style Session Login (no `.formLogin()`)

When Spring Boot is the backend for a separate SPA / mobile frontend, skip `.formLogin()` and write your own `/api/auth/login`. **You don't issue a token**: successful login creates a session and the browser receives a `JSESSIONID` cookie automatically. That cookie *is* the auth.

### `SecurityConfig` differences

```java
@Bean
public SecurityFilterChain api(HttpSecurity http) throws Exception {
    http
        // CSRF cookie readable by JS (so SPA can echo it as a header)
        .csrf(csrf -> csrf
            .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse()))
        .sessionManagement(session ->
            session.sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED))
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/auth/login", "/api/auth/register").permitAll()
            .anyRequest().authenticated()
        )
        // No .formLogin() — custom controller handles login.
        // Return JSON 401 instead of redirecting to a login page.
        .exceptionHandling(ex -> ex
            .authenticationEntryPoint((req, res, authEx) -> {
                res.setStatus(HttpStatus.UNAUTHORIZED.value());
                res.setContentType("application/json");
                res.getWriter().write("{\"error\":\"Unauthorized\"}");
            })
        )
        .logout(logout -> logout
            .logoutUrl("/api/auth/logout")              // POST endpoint
            .logoutSuccessHandler((req, res, auth) ->   // JSON 204 instead of redirect
                res.setStatus(HttpStatus.NO_CONTENT.value()))
            .invalidateHttpSession(true)
            .deleteCookies("JSESSIONID")
        );

    return http.build();
}
```

### `AuthController` — login, logout, me

```java
@RestController
@RequestMapping("/api/auth")
@RequiredArgsConstructor
public class AuthController {

    private final AuthenticationManager authenticationManager;
    private final SecurityContextRepository securityContextRepository =
        new HttpSessionSecurityContextRepository();

    @PostMapping("/login")
    public ResponseEntity<UserResponse> login(
            @Valid @RequestBody LoginRequest request,
            HttpServletRequest httpRequest,
            HttpServletResponse httpResponse) {

        // 1. Authenticate — throws AuthenticationException on bad credentials
        Authentication auth = authenticationManager.authenticate(
            new UsernamePasswordAuthenticationToken(
                request.getEmail(), request.getPassword()));

        // 2. Store in SecurityContext and persist to the session
        SecurityContext context = SecurityContextHolder.createEmptyContext();
        context.setAuthentication(auth);
        SecurityContextHolder.setContext(context);
        securityContextRepository.saveContext(context, httpRequest, httpResponse);
        // ↑ This step is what triggers the JSESSIONID cookie if not already present

        // 3. Return user info — NO token, the cookie carries the auth
        return ResponseEntity.ok(toUserResponse(auth));
    }

    // POST /api/auth/logout is handled by Spring's .logout(...) config above — no controller method needed

    @GetMapping("/me")
    public ResponseEntity<UserResponse> me(@AuthenticationPrincipal UserDetails user) {
        // Frontend calls this on page load to check if user is still logged in
        return ResponseEntity.ok(toUserResponse(user));
    }
}
```

### How the SPA Talks to It

```javascript
// 1. Login
await fetch('/api/auth/login', {
    method: 'POST',
    credentials: 'include',                        // browser accepts + sends cookies
    headers: {
        'Content-Type': 'application/json',
        'X-XSRF-TOKEN': getCookie('XSRF-TOKEN'),   // CSRF token from cookie
    },
    body: JSON.stringify({ email, password })
});

// 2. Subsequent calls — JSESSIONID cookie auto-sent by the browser
const me = await fetch('/api/auth/me', { credentials: 'include' });

// 3. Logout
await fetch('/api/auth/logout', {
    method: 'POST',
    credentials: 'include',
    headers: { 'X-XSRF-TOKEN': getCookie('XSRF-TOKEN') }
});
```

## Production Considerations

- **Session store**: default `HttpSession` lives in JVM memory. In a multi-instance deployment you need either **sticky sessions** (load balancer pins each user to one node) or a **shared session store** (Redis via `spring-session-data-redis`, JDBC via `spring-session-jdbc`). Sticky is simpler; shared is more elastic.
- **Session fixation protection**: Spring rotates the `JSESSIONID` on login by default (`session.sessionFixation(SessionFixationConfigurer::migrateSession)` is the default). Don't disable.
- **Concurrent session limits**: `.sessionConcurrency(c -> c.maximumSessions(1).expiredUrl("/login?expired"))` limits a user to one active session, kicking older logins out. Useful for sensitive admin apps.
- **Cookie attributes**: set `Secure` (HTTPS only) and `SameSite=Lax` on the session cookie via `server.servlet.session.cookie.same-site=lax` and `secure=true`. Without these, the cookie travels over plain HTTP and is exposed to CSRF on cross-site POSTs.
- **Session timeout**: `server.servlet.session.timeout=30m`. Shorter for sensitive apps; pair with sliding-expiration if you want activity-based extension.

## Q&A

1. **What's the default `SessionCreationPolicy` and what does it do?**
   `IF_REQUIRED` — Spring creates a session on first authentication (or whenever something stores into the security context), then reuses it. `ALWAYS` forces creation; `NEVER` reuses but doesn't create; `STATELESS` (used in JWT) skips session entirely.

2. **Why is CSRF protection on by default for session auth but disabled for JWT?**
   Session auth uses a `JSESSIONID` cookie that the browser auto-attaches to every same-origin request — including ones triggered by malicious sites. CSRF token verification is the defense. JWT in `Authorization: Bearer` header isn't auto-attached, so CSRF isn't possible on the header path.

3. **Form login vs HTTP Basic — when do you pick which?**
   Form login: end-user-facing apps with a real login UX (custom page, success/failure handling, remember-me, logout button). HTTP Basic: internal endpoints, machine-to-machine, quick API testing — popup credentials, no logout UX, password retransmitted on every request.

4. **What does session fixation protection actually do?**
   On successful login, Spring **rotates** the session ID (assigns a new `JSESSIONID`, copies attributes over, invalidates the old one). Prevents an attacker from setting a known `JSESSIONID` on the victim's browser before login (e.g., via a planted cookie) and then reusing that session post-authentication.

5. **How does `.rememberMe()` re-authenticate without storing a session?**
   The cookie holds a signed token (`hash(username + password + expiry + key)`). On a request with no active session but a valid `remember-me` cookie, `RememberMeAuthenticationFilter` validates the signature, re-loads the user via `UserDetailsService`, and creates a fresh session. The user never sees the login form.

6. **What's `InMemoryUserDetailsManager` and when should you use it?**
   It's a `UserDetailsService` implementation that holds users in-memory (Java collection). Useful for unit tests, demos, internal admin tools with hardcoded users — anywhere persistence isn't needed. Never for real user accounts: lost on restart.

7. **Sticky sessions vs Redis-backed session store — what's the trade-off?**
   Sticky: load balancer pins each client to one app instance via cookie/IP — simple, no extra infra, but a node failure logs out everyone on that node. Redis-backed: any instance can serve any request (`spring-session-data-redis`) — resilient and elastic, but adds Redis as critical infra and a network round-trip per session read.

## Quick Reference

| Concept | Remember |
|---|---|
| Default session policy | `IF_REQUIRED` — session created on first auth, reused after |
| CSRF default | On (session cookie auto-attached → token check needed) |
| Built-in login | `.formLogin()` (HTML page); HTTP Basic covered in [5.4 Filter Chain Configuration](<5.4. Filter Chain Configuration.md>) |
| Backend-API login | Custom `/api/auth/login` controller calling `AuthenticationManager` + `SecurityContextRepository.saveContext()` to set `JSESSIONID` |
| In-memory users | `InMemoryUserDetailsManager` — testing/demos only |
| DB-backed users | Custom `UserDetailsService` impl (see [5.3 Authentication Core Components](<5.3. Authentication Core Components.md>)) |
| Remember-me | `.rememberMe().key(...)` — signed token in cookie; or `PersistentTokenBasedRememberMeServices` for DB-backed |
| Logout | `.logout().invalidateHttpSession(true).deleteCookies("JSESSIONID")` |
| Session fixation | Default `migrateSession` — rotates `JSESSIONID` on login (don't disable) |
| Multi-instance store | Sticky sessions OR `spring-session-data-redis` (production: prefer Redis) |
| Cookie attributes | `Secure` + `SameSite=Lax` via `server.servlet.session.cookie.*` properties |
