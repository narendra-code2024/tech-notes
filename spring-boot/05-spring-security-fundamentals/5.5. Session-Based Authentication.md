Session-based auth is Spring Security's default: the server creates an `HttpSession`, the browser holds a `JSESSIONID` cookie, and every subsequent request carries it. There is no token in the response body; the cookie is the auth. Spring Boot supports two flavors, differing only in who handles the login form:

- **Form login** (`.formLogin()`): Spring Boot serves the HTML (Thymeleaf, JSP) and intercepts the form POST, using a built-in or custom login page.
- **API-style login** (custom `/api/auth/login` controller): Spring Boot is the backend, the SPA or mobile client owns the login UI and POSTs JSON, and your controller calls `AuthenticationManager` directly.

Both paths share the same `SecurityContext` and `JSESSIONID` machinery.

---

## When to Choose Session Auth

- **Single-server or sticky-session topology**: a server-side session is simpler than coordinating JWT secrets across nodes.
- **Server-rendered web app** (Thymeleaf, JSP): the `JSESSIONID` cookie just works.
- **Easy server-side invalidation**: kicking a user out is a session-store delete; a JWT cannot be revoked without a blocklist.
- **Internal admin tools**: small user base, low coordination overhead, simpler ops.

## UserDetailsService: Where the User Comes From

`UserDetailsService` bridges your user store and Spring Security. The two common backings are in-memory (testing/early dev, below) and database-backed.

### `InMemoryUserDetailsManager`

```java
@Bean
public UserDetailsService userDetailsService(PasswordEncoder encoder) {
    UserDetails admin = User.builder()
        .username("admin")
        .password(encoder.encode("admin123"))
        .roles("ADMIN")
        .build();

    UserDetails user = User.builder()
        .username("user")
        .password(encoder.encode("user123"))
        .roles("USER")
        .build();

    return new InMemoryUserDetailsManager(admin, user);
}
```

Good for early development, unit tests, and internal tools with a fixed user list. The store is lost on restart, so never use it for real users.

> **Reference**: the canonical `UserDetailsService` coverage and the database-backed `CustomUserDetailsService` example are in [5.3. Authentication Core Components.md](<5.3. Authentication Core Components.md>).

## Form Login Configuration

The session-based chain is the inverse of the JWT one: CSRF stays on, sessions are created as needed, and there is no custom auth filter.

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http)
            throws Exception {

        http
            .csrf(Customizer.withDefaults())
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/", "/login", "/register", "/css/**", "/js/**").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .formLogin(form -> form
                .loginPage("/login")                        // GET /login, your Thymeleaf page
                .loginProcessingUrl("/login")               // POST endpoint Spring intercepts (defaults to loginPage)
                .defaultSuccessUrl("/dashboard", true)
                .failureUrl("/login?error=true")
                .usernameParameter("email")                 // form input name (default "username")
                .passwordParameter("password")
                .permitAll()
            )
            .logout(logout -> logout
                .logoutUrl("/logout")                       // POST only; GET is CSRF-vulnerable via <img src="/logout">
                .logoutSuccessUrl("/login?logout=true")
                .invalidateHttpSession(true)                // destroy server-side session
                .deleteCookies("JSESSIONID")                // clear browser cookie
                .clearAuthentication(true)                  // clear SecurityContext
                .permitAll()
            );

        return http.build();
    }
}
```

Zero-config `.formLogin(Customizer.withDefaults())` serves a plain Spring-generated form at `/login` (and handles `POST /login`), useful for early development and admin tools.

### Success / failure handlers

For finer control (JSON for AJAX login, audit logging, custom cookies):

```java
.formLogin(form -> form
    .successHandler((request, response, auth) -> {
        response.setContentType("application/json");
        response.getWriter().write("{\"status\":\"ok\"}");
    })
    .failureHandler((request, response, ex) -> {
        response.setStatus(HttpStatus.UNAUTHORIZED.value());
        response.getWriter().write("{\"error\":\"" + ex.getMessage() + "\"}");
    })
)
```

### Remember-Me

By default `JSESSIONID` is a session cookie (no `Max-Age`), so the browser drops it on close and the user must re-authenticate. Remember-Me adds a separate signed persistent cookie with an explicit `Max-Age` that survives browser close:

```java
.rememberMe(rm -> rm
    .key("a-secret-string-used-to-sign-the-token")
    .tokenValiditySeconds(14 * 24 * 60 * 60)   // 2 weeks
    .userDetailsService(userDetailsService)
)
```

On a later request with no session, `RememberMeAuthenticationFilter` validates the cookie and re-authenticates without prompting; logout clears it.

## API-Style Session Login (no `.formLogin()`)

When Spring Boot backs a separate SPA or mobile frontend, skip `.formLogin()` and write your own `/api/auth/login`. You do not issue a token: a successful login creates a session and the browser receives a `JSESSIONID` cookie automatically, and that cookie is the auth.

### SecurityConfig differences

```java
@Bean
public SecurityFilterChain api(HttpSecurity http) throws Exception {
    http
        // CSRF cookie readable by JS so the SPA can echo it as a header
        .csrf(csrf -> csrf
            .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse()))
        .sessionManagement(session ->
            session.sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED))
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/auth/login", "/api/auth/register").permitAll()
            .anyRequest().authenticated()
        )
        // No .formLogin(); a custom controller handles login. Return JSON 401, not a redirect.
        .exceptionHandling(ex -> ex
            .authenticationEntryPoint((req, res, authEx) -> {
                res.setStatus(HttpStatus.UNAUTHORIZED.value());
                res.setContentType("application/json");
                res.getWriter().write("{\"error\":\"Unauthorized\"}");
            })
        )
        .logout(logout -> logout
            .logoutUrl("/api/auth/logout")
            .logoutSuccessHandler((req, res, auth) ->   // JSON 204 instead of redirect
                res.setStatus(HttpStatus.NO_CONTENT.value()))
            .invalidateHttpSession(true)
            .deleteCookies("JSESSIONID")
        );

    return http.build();
}
```

### AuthController: login, me

```java
@RestController
@RequestMapping("/api/auth")
@RequiredArgsConstructor
public class AuthController {

    private final AuthenticationManager authenticationManager;
    private final SecurityContextRepository securityContextRepository =
        new HttpSessionSecurityContextRepository();

    @PostMapping("/login")
    public ResponseEntity<UserResponse> login(
            @Valid @RequestBody LoginRequest request,
            HttpServletRequest httpRequest,
            HttpServletResponse httpResponse) {

        // 1. Authenticate; throws AuthenticationException on bad credentials
        Authentication auth = authenticationManager.authenticate(
            new UsernamePasswordAuthenticationToken(
                request.getEmail(), request.getPassword()));

        // 2. Store in SecurityContext and persist to the session.
        //    The explicit saveContext is what triggers the JSESSIONID cookie (required since SS6).
        SecurityContext context = SecurityContextHolder.createEmptyContext();
        context.setAuthentication(auth);
        SecurityContextHolder.setContext(context);
        securityContextRepository.saveContext(context, httpRequest, httpResponse);

        // 3. Return user info; no token, the cookie carries the auth
        return ResponseEntity.ok(toUserResponse(auth));
    }

    // POST /api/auth/logout is handled by Spring's .logout(...) config; no controller method needed.

    @GetMapping("/me")
    public ResponseEntity<UserResponse> me(@AuthenticationPrincipal UserDetails user) {
        return ResponseEntity.ok(toUserResponse(user));   // SPA calls this on page load
    }
}
```

### How the SPA Talks to It

```javascript
// 1. Login
await fetch('/api/auth/login', {
    method: 'POST',
    credentials: 'include',                        // browser accepts and sends cookies
    headers: {
        'Content-Type': 'application/json',
        'X-XSRF-TOKEN': getCookie('XSRF-TOKEN'),   // CSRF token from cookie
    },
    body: JSON.stringify({ email, password })
});

// 2. Subsequent calls: JSESSIONID auto-sent by the browser
const me = await fetch('/api/auth/me', { credentials: 'include' });

// 3. Logout
await fetch('/api/auth/logout', {
    method: 'POST',
    credentials: 'include',
    headers: { 'X-XSRF-TOKEN': getCookie('XSRF-TOKEN') }
});
```

## Production Considerations

- **Session store**: the default `HttpSession` lives in JVM memory. A multi-instance deployment needs either sticky sessions (the load balancer pins each user to one node) or a shared store (Redis via `spring-session-data-redis`, JDBC via `spring-session-jdbc`). Sticky is simpler; shared is more elastic.
- **Session fixation**: Spring rotates the `JSESSIONID` on login by default (`migrateSession`); do not disable it.
- **Concurrent session limits**: `.sessionManagement(s -> s.sessionConcurrency(c -> c.maximumSessions(1).expiredUrl("/login?expired")))` caps a user to one active session, useful for sensitive admin apps.
- **Cookie attributes**: set `server.servlet.session.cookie.same-site=lax` and `secure=true`; without them the cookie travels over plain HTTP and is exposed to cross-site POST CSRF.
- **Session timeout**: `server.servlet.session.timeout=30m`, shorter for sensitive apps.

## Q&A

1. **What is the default `SessionCreationPolicy` and what does it do?**
`IF_REQUIRED`: Spring creates a session on first authentication (or whenever something stores into the security context), then reuses it. `ALWAYS` forces creation, `NEVER` reuses but does not create, and `STATELESS` (JWT) skips sessions entirely.

2. **Why is CSRF on by default for session auth but disabled for JWT?**
Session auth rides a `JSESSIONID` cookie the browser auto-attaches to every same-origin request, including ones a malicious site triggers; the CSRF token is the defense. A JWT in the `Authorization` header is not auto-attached, so the header path is not CSRF-exposed.

3. **Form login vs HTTP Basic, when do you pick which?**
Form login suits end-user apps with a real login UX (custom page, success/failure handling, remember-me, logout). HTTP Basic suits internal endpoints, machine-to-machine, and quick API testing; it has no logout UX and retransmits the password on every request.

4. **What does session fixation protection actually do?**
On successful login Spring rotates the session ID (new `JSESSIONID`, attributes copied over, old one invalidated). This stops an attacker who planted a known `JSESSIONID` on the victim before login from reusing that session afterward.

5. **How does `.rememberMe()` re-authenticate without a session?**
The cookie holds a signed token (`hash(username + password + expiry + key)`). On a request with no session but a valid cookie, `RememberMeAuthenticationFilter` validates the signature, reloads the user via `UserDetailsService`, and creates a fresh session, so the user never sees the login form.

6. **Sticky sessions vs a Redis-backed store, what is the trade-off?**
Sticky pins each client to one instance: simple, no extra infra, but a node failure logs out everyone on it. Redis-backed lets any instance serve any request: resilient and elastic, but adds Redis as critical infra and a network round-trip per session read.

## Quick Revision Cheat Sheet

| Concept                | Remember                                                                    |
| ---------------------- | --------------------------------------------------------------------------- |
| Default session policy | `IF_REQUIRED`; session created on first auth, reused after.                 |
| CSRF default           | On; the `JSESSIONID` cookie is auto-attached, so a token check is needed.    |
| Built-in login         | `.formLogin()` (HTML page); HTTP Basic for internal endpoints.              |
| Backend-API login      | Custom controller calling `AuthenticationManager` + `saveContext()`.        |
| User store             | `InMemoryUserDetailsManager` (tests) or custom DB-backed `UserDetailsService`. |
| Remember-me            | `.rememberMe().key(...)`; signed cookie, or persistent token for DB-backed.  |
| Logout                 | `.logout().invalidateHttpSession(true).deleteCookies("JSESSIONID")`.        |
| Session fixation       | Default `migrateSession` rotates `JSESSIONID` on login; do not disable.     |
| Multi-instance store   | Sticky sessions or `spring-session-data-redis` (prefer Redis in prod).      |
| Cookie attributes      | `Secure` + `SameSite=Lax` via `server.servlet.session.cookie.*`.            |
