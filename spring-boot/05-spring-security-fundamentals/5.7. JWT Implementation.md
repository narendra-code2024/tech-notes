End-to-end JWT auth, both sides of the flow:

- **Issue side**: `/api/auth/{register, login, logout}` endpoints that authenticate credentials and emit a token (`AuthService` + `AuthController`).
- **Validate side**: a `JwtAuthFilter` that runs before every protected request, extracts the token from header or cookie, and populates `SecurityContext`.

The conceptual half (structure, storage tradeoffs) is in [5.6. Understanding JWT.md](<5.6. Understanding JWT.md>); generic `HttpSecurity` options are in [5.4. Filter Chain Configuration.md](<5.4. Filter Chain Configuration.md>). This note is the code walkthrough.

## JwtService

Used by both sides: the issue side calls `generateToken`; the validate side calls `extractEmail` + `isTokenValid`.

```java
@Service
public class JwtService {

    @Value("${jwt.secret}")
    private String secret;

    @Value("${jwt.expiration}")
    private long expiration;

    public String generateToken(String email) {
        return Jwts.builder()
            .subject(email)
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + expiration))
            .signWith(getSigningKey())
            .compact();
    }

    public String extractEmail(String token) {
        return extractClaims(token).getSubject();
    }

    public boolean isTokenValid(String token, UserDetails userDetails) {
        String email = extractEmail(token);
        return email.equals(userDetails.getUsername()) && !isTokenExpired(token);
    }

    private boolean isTokenExpired(String token) {
        return extractClaims(token).getExpiration().before(new Date());
    }

    private Claims extractClaims(String token) {
        return Jwts.parser()
            .verifyWith(getSigningKey())
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }

    private SecretKey getSigningKey() {   // imports: io.jsonwebtoken.io.Decoders, io.jsonwebtoken.security.Keys
        return Keys.hmacShaKeyFor(Decoders.BASE64.decode(secret));
    }
}
```

> **Note**:
> - `jwt.secret` must be Base64-encoded with at least 256 bits of entropy (HS256 minimum); generate with `openssl rand -base64 32`.
> - Never commit the secret; load it from an env var or secret store.

## AuthService

```java
@Service
@RequiredArgsConstructor
public class AuthServiceImpl implements AuthService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;
    private final AuthenticationManager authenticationManager;
    private final JwtService jwtService;
    private final UserMapper userMapper;

    @Value("${jwt.expiration}")
    private long expiration;

    @Override
    public UserResponse register(RegisterRequest request) {
        if (userRepository.existsByEmail(request.getEmail())) {
            throw new IllegalArgumentException("Email already taken");
        }

        User user = User.builder()
            .firstName(request.getFirstName())
            .lastName(request.getLastName())
            .email(request.getEmail())
            .password(passwordEncoder.encode(request.getPassword()))
            .roles(Set.of("ROLE_USER"))
            .build();

        User saved = userRepository.save(user);
        return userMapper.toDto(saved);
    }

    @Override
    public AuthResponse login(LoginRequest request,
            HttpServletResponse response) {

        // Throws AuthenticationException if credentials are invalid
        authenticationManager.authenticate(
            new UsernamePasswordAuthenticationToken(
                request.getEmail(), request.getPassword()));

        String token = jwtService.generateToken(request.getEmail());

        // Cookie path (for SPA)
        ResponseCookie cookie = ResponseCookie.from("jwt", token)
            .httpOnly(true)      // XSS
            .secure(true)        // transport
            .path("/")
            .maxAge(expiration / 1000)
            .sameSite("Lax")     // CSRF
            .build();
        response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());

        // Body path (for microservices / Postman)
        return AuthResponse.builder().token(token).build();
    }

    @Override
    public void logout(HttpServletResponse response) {
        ResponseCookie cookie = ResponseCookie.from("jwt", "")
            .httpOnly(true)
            .secure(true)
            .path("/")
            .maxAge(0)   // delete the cookie
            .sameSite("Lax")
            .build();
        response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());
    }
}
```

> **Note**: returning the token in both body and `HttpOnly` cookie defeats `HttpOnly` (JS can read the body); it is done here only to show both paths. In production, split by client type: SPAs get a cookie-only endpoint, microservices/mobile a body-only endpoint, over the same `AuthService.login`.

> **Reference**: cookie attribute tradeoffs are in [5.6. Understanding JWT.md](<5.6. Understanding JWT.md>).

    @PostMapping("/login")
    public ResponseEntity<AuthResponse> login(
            @Valid @RequestBody LoginRequest request,
            HttpServletResponse response) {
        return ResponseEntity.ok(authService.login(request, response));
    }

    @PostMapping("/logout")
    public ResponseEntity<Void> logout(HttpServletResponse response) {
        authService.logout(response);
        return ResponseEntity.noContent().build();
    }
}

## SecurityConfig for JWT

The validate-side filter (`JwtAuthFilter`, next section) is injected and inserted before form login.

```java
@Configuration
@EnableWebSecurity
@RequiredArgsConstructor
public class SecurityConfig {

    private final JwtAuthFilter jwtAuthFilter;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http)
            throws Exception {

        http
            .csrf(csrf -> csrf.disable())
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/register", "/api/auth/login").permitAll()
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtAuthFilter,
                UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }
}
```

Three options make this JWT-specific: `csrf.disable()` (header-based JWT cannot be victimized by CSRF), `STATELESS` (no server-side session), and `addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)` (the JWT filter runs before form login).

## JwtAuthFilter

The validate side, extending `OncePerRequestFilter` so it runs once per request:

- Extracts the token from the `Authorization: Bearer <token>` header (microservices/Postman) or the `jwt` cookie (SPA), header first.
- If valid, sets `SecurityContext` and the request proceeds.
- If no token, skips through; public endpoints work and protected ones get 401 from `AuthorizationFilter` later.
- If a JWT exception (expired, malformed, bad signature) occurs, emits 401 JSON directly via `sendError()`.

```java
@Component
@Slf4j
@RequiredArgsConstructor
public class JwtAuthFilter extends OncePerRequestFilter {

    private final JwtService jwtService;
    private final CustomUserDetailsService userDetailsService;
    private final ObjectMapper objectMapper;

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain) throws ServletException, IOException {

        String token = extractToken(request);

        if (token == null) {
            filterChain.doFilter(request, response);
            return;
        }

        try {
            String email = jwtService.extractEmail(token);

            // guard prevents overwriting an auth set by an earlier filter, and makes the filter idempotent
            if (email != null
                && SecurityContextHolder.getContext().getAuthentication() == null) {

                UserDetails userDetails = userDetailsService.loadUserByUsername(email);

                if (jwtService.isTokenValid(token, userDetails)) {
                    UsernamePasswordAuthenticationToken authToken =
                        new UsernamePasswordAuthenticationToken(
                            userDetails, null, userDetails.getAuthorities());
                    authToken.setDetails(
                        new WebAuthenticationDetailsSource().buildDetails(request));
                    SecurityContextHolder.getContext().setAuthentication(authToken);
                }
            }

            filterChain.doFilter(request, response);

        } catch (ExpiredJwtException e) {
            sendError(response, HttpStatus.UNAUTHORIZED, "Token has expired");
        } catch (SignatureException e) {
            log.warn("JWT signature mismatch from {}", request.getRemoteAddr());
            sendError(response, HttpStatus.UNAUTHORIZED, "Invalid token signature");
        } catch (MalformedJwtException e) {
            sendError(response, HttpStatus.UNAUTHORIZED, "Malformed token");
        } catch (Exception e) {
            log.warn("JWT processing failed", e);
            sendError(response, HttpStatus.UNAUTHORIZED, "Invalid token");
        }
    }

    private String extractToken(HttpServletRequest request) {
        String authHeader = request.getHeader("Authorization");   // header first
        if (authHeader != null && authHeader.startsWith("Bearer ")) {
            return authHeader.substring(7);
        }
        Cookie[] cookies = request.getCookies();                  // then cookie (SPA)
        if (cookies != null) {
            for (Cookie cookie : cookies) {
                if ("jwt".equals(cookie.getName())) {
                    return cookie.getValue();
                }
            }
        }
        return null;
    }

    private void sendError(HttpServletResponse response,
            HttpStatus status, String message) throws IOException {
        response.setStatus(status.value());
        response.setContentType(MediaType.APPLICATION_JSON_VALUE);
        objectMapper.writeValue(response.getWriter(), new ApiError(message));
    }
}
```

The per-request `loadUserByUsername` is a deliberate DB hit: it catches users disabled or re-roled since the token was issued, at the cost of one query. Pass `null` for credentials because the token is already validated and storing the password in the context would leak it.

> **Reference**: the `CustomUserDetailsService` is in [5.3. Authentication Core Components.md](<5.3. Authentication Core Components.md>); the unwrapped error envelope is in [2.8. API Response Patterns.md](<../02-spring-mvc-and-rest-apis/2.8. API Response Patterns.md>).

## Exception Handling

JWT exceptions happen inside `JwtAuthFilter`, before `DispatcherServlet`. `@RestControllerAdvice` only sees exceptions thrown inside the dispatcher, so filter-thrown exceptions never reach it and must be handled locally.

```
Request
   │
   ▼
Filters (JWT exceptions happen here; no @RestControllerAdvice)
   │
   ▼
DispatcherServlet
   │
   ▼
Controller (auth exceptions happen here; @RestControllerAdvice works)
```

### Filter-level exceptions (in `JwtAuthFilter`)

Each `catch` calls `sendError(...)`, which uses `ObjectMapper` to write the same `ApiError` JSON shape that `@RestControllerAdvice` returns.

| Exception                | Cause                   | HTTP Status |
| ------------------------ | ----------------------- | ----------- |
| `ExpiredJwtException`    | Token past expiry.      | 401         |
| `SignatureException`     | Token tampered with.    | 401         |
| `MalformedJwtException`  | Not a valid JWT format. | 401         |
| `UnsupportedJwtException`| Unsupported JWT type.   | 401         |
| Generic `Exception`      | Any other issue.        | 401         |

### Controller-level exceptions (`@RestControllerAdvice`)

These are thrown inside `DispatcherServlet`, so the advice catches them.

| Exception                    | Cause                              | HTTP Status |
| ---------------------------- | ---------------------------------- | ----------- |
| `AuthenticationException`    | Invalid email or password (login). | 401         |
| `BadCredentialsException`    | Wrong password.                    | 401         |
| `IllegalArgumentException`   | Email already taken (register).    | 400         |

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(AuthenticationException.class)
    public ResponseEntity<ApiError> handleAuthException(AuthenticationException ex) {
        ApiError error = new ApiError("Invalid email or password");
        log.warn("Authentication failed [id={}]", error.getId());
        return ResponseEntity.status(HttpStatus.UNAUTHORIZED).body(error);
    }

    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<ApiError> handleBadRequest(IllegalArgumentException ex) {
        return ResponseEntity.badRequest().body(new ApiError(ex.getMessage()));
    }
}
```

Both layers return the same unwrapped `ApiError`, and its trace `id` (UUID) appears in the response and the server log so support can correlate a report to the exact failure:

```json
{
    "id": "5f8a3c9b-2e1f-4a8c-9b1a-d4e7f8a9b1c2",
    "message": "Token has expired",
    "timestamp": "2026-03-01T15:30:00Z"
}
```

## Production Considerations

- **Secret management**: load `jwt.secret` from env vars or a secrets manager (AWS Secrets Manager, Vault, GCP Secret Manager), never source control. Rotate on a schedule and on suspected leak; pre-stage the new secret as a fallback validator before cutover so in-flight tokens keep validating.
- **Secret strength**: Base64-encoded, at least 256 bits (32 bytes) for HS256; generate via `openssl rand -base64 64`, never a memorable string.
- **HTTPS only**: JWTs over plain HTTP can be sniffed and replayed; enforce TLS at the ingress and set `secure=true` on the cookie.
- **Filter-level observability**: filter exceptions bypass any `@RestControllerAdvice` logging, so log inside the filter's `catch` blocks before writing the response.
- **Token expiry**: keep access-token expiry short (15 to 30 min) and pair with refresh tokens; long-lived JWTs cannot be revoked without a server-side blocklist.
- **Clock skew**: `Jwts.parser().clockSkewSeconds(60)` tolerates small time drift between issuing and validating nodes.

## Q&A

1. **Why is the JWT secret read from `@Value("${jwt.secret}")` rather than hard-coded?**
The signing key must come from environment-specific config (env vars, vault), never source control; leaking it lets anyone forge valid tokens.

2. **Why does the filter check both the `Authorization` header and a cookie?**
To serve two client types from one backend: microservices/Postman send a `Bearer` header, browser SPAs rely on the auto-sent `HttpOnly` cookie. The filter accepts whichever is present, header first.

3. **Why guard with `getAuthentication() == null` before setting the context?**
To avoid overwriting an authentication set by an earlier filter, and to keep the filter idempotent if it ever runs twice.

4. **Why pass `null` for credentials to `UsernamePasswordAuthenticationToken`?**
The token is already validated, so the password is not needed downstream; storing it in the context would be a leak.

5. **Why call `loadUserByUsername` per request instead of trusting the token claims?**
Claims can be stale (the user may have been disabled or re-roled since issuance); the DB lookup catches that, at the cost of one query per request.

6. **Why doesn't `@RestControllerAdvice` catch JWT exceptions?**
The advice is wired into `DispatcherServlet`. Filters run before the dispatcher, so a filter-thrown exception bubbles to the servlet container and never reaches the advice; handle it inside the filter.

7. **What is the difference between `AuthenticationException` and JWT exceptions like `ExpiredJwtException`?**
`AuthenticationException` (and `BadCredentialsException`) is thrown by Spring Security during the controller-driven `authenticationManager.authenticate()` call, caught by `@RestControllerAdvice`. JWT exceptions come from the jjwt library inside the filter and must be caught locally.

8. **Why is "Email already taken" a 400 but "Bad credentials" a 401?**
400 signals invalid input the client can fix (choose another email); 401 signals the credentials are wrong, i.e. the client is not who they claim to be.
