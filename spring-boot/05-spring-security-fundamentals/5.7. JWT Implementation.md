End-to-end JWT auth implementation — both sides of the flow:

- **Issue side**: `/api/auth/{register, login, logout}` endpoints that authenticate credentials and emit a token (`AuthService` + `AuthController`)
- **Validate side**: a `JwtAuthFilter` that runs before every protected request, extracts the token from header or cookie, and populates `SecurityContext`

The conceptual half (what JWT is, structure, storage trade-offs) lives in [5.6 Understanding JWT](<5.6. Understanding JWT.md>). Generic `HttpSecurity` options are in [5.4 Filter Chain Configuration](<5.4. Filter Chain Configuration.md>). This file is the code walkthrough.

## JwtService

Used by both sides — the issue side calls `generateToken`; the validate side calls `extractEmail` + `isTokenValid`.

```java
@Service
public class JwtService {

    @Value("${jwt.secret}")
    private String secret;

    @Value("${jwt.expiration}")
    private long expiration;

    public String generateToken(String email) {
        return Jwts.builder()
            .subject(email)
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + expiration))
            .signWith(getSigningKey())
            .compact();
    }

    public String extractEmail(String token) {
        return extractClaims(token).getSubject();
    }

    public boolean isTokenValid(String token, UserDetails userDetails) {
        String email = extractEmail(token);
        return email.equals(userDetails.getUsername()) && !isTokenExpired(token);
    }

    private boolean isTokenExpired(String token) {
        return extractClaims(token).getExpiration().before(new Date());
    }

    private Claims extractClaims(String token) {
        return Jwts.parser()
            .verifyWith(getSigningKey())
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }

    private SecretKey getSigningKey() {
        // The secret is configured as a Base64-encoded string with ≥256 bits of entropy
        return Keys.hmacShaKeyFor(Decoders.BASE64.decode(secret));
    }
}
```

> **Secret configuration:** `jwt.secret` must be a **Base64-encoded** string carrying at least 256 bits of entropy (HS256 minimum). Generate one with `openssl rand -base64 32`. Never commit the secret — load it from an env var or a secret store. Required imports for the helper above: `io.jsonwebtoken.io.Decoders` and `io.jsonwebtoken.security.Keys`.

## AuthService

```java
@Service
@RequiredArgsConstructor
public class AuthServiceImpl implements AuthService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;
    private final AuthenticationManager authenticationManager;
    private final JwtService jwtService;
    private final UserMapper userMapper;

    @Value("${jwt.expiration}")
    private long expiration;

    @Override
    public UserResponse register(RegisterRequest request) {
        if (userRepository.existsByEmail(request.getEmail())) {
            throw new IllegalArgumentException("Email already taken");
        }

        User user = User.builder()
            .firstName(request.getFirstName())
            .lastName(request.getLastName())
            .email(request.getEmail())
            .password(passwordEncoder.encode(request.getPassword()))
            .roles(Set.of("ROLE_USER"))
            .build();

        User saved = userRepository.save(user);
        return userMapper.toDto(saved);
    }

    @Override
    public AuthResponse login(LoginRequest request,
            HttpServletResponse response) {

        // Throws AuthenticationException if credentials are invalid
        authenticationManager.authenticate(
            new UsernamePasswordAuthenticationToken(
                request.getEmail(), request.getPassword()));

        String token = jwtService.generateToken(request.getEmail());

        // Cookie path (for SPA)
        ResponseCookie cookie = ResponseCookie.from("jwt", token)
            .httpOnly(true)
            .secure(true)
            .path("/")
            .maxAge(expiration / 1000)
            .sameSite("Lax")
            .build();
        response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());

        // Body path (for microservices / Postman)
        return AuthResponse.builder().token(token).build();
    }

    @Override
    public void logout(HttpServletResponse response) {
        ResponseCookie cookie = ResponseCookie.from("jwt", "")
            .httpOnly(true)
            .secure(true)
            .path("/")
            .maxAge(0) // delete the cookie
            .sameSite("Lax")
            .build();
        response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());
    }
}
```

Cookie attrs: `httpOnly` (XSS), `secure` (transport), `sameSite="Lax"` (CSRF) — full breakdown in [5.6 Understanding JWT](<5.6. Understanding JWT.md>) → JWT Storage — Security Summary.

> **Dual-delivery trade-off.** Returning the token in **both** body and `HttpOnly` cookie defeats `HttpOnly` — JS can read the body. Done here intentionally to show both paths side-by-side. **In production, split by client type:** SPAs → cookie-only endpoint, microservices/mobile → body-only endpoint. Same `AuthService.login` underneath.

## SecurityConfig for JWT

Wire the issue side together. The validate-side filter (`JwtAuthFilter`, defined in the next section) is referenced here:

```java
@Configuration
@EnableWebSecurity
@RequiredArgsConstructor
public class SecurityConfig {

    private final JwtAuthFilter jwtAuthFilter;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http)
            throws Exception {

        http
            .csrf(csrf -> csrf.disable())

            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))

            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/register").permitAll()
                .requestMatchers("/api/auth/login").permitAll()
                .anyRequest().authenticated()
            )

            .addFilterBefore(jwtAuthFilter,
                UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }
}
```

Three options make this configuration JWT-specific (each explained in [5.4 Filter Chain Configuration](<5.4. Filter Chain Configuration.md>)):

- **`.csrf(csrf -> csrf.disable())`**: header-based JWT can't be victimized by CSRF
- **`SessionCreationPolicy.STATELESS`**: no server-side session
- **`.addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)`**: JWT filter runs before Spring's form-login filter (filter implementation in the next section)

## JwtAuthFilter

The validate side. Runs **before every request** (extends `OncePerRequestFilter`); checks for a JWT in two places, validates it via `JwtService`, populates `SecurityContext` so protected endpoints see an authenticated user.

- Extracts token from `Authorization: Bearer <token>` header (microservices/Postman) **or** `jwt` HttpOnly cookie (SPA) — header checked first
- If valid → sets `SecurityContext` → request proceeds
- If no token → skips, public endpoints work, protected ones get 401 from `AuthorizationFilter` later
- If JWT-specific exception (expired, malformed, bad signature) → emits 401 JSON directly via `sendError()`

```java
@Component
@Slf4j
@RequiredArgsConstructor
public class JwtAuthFilter extends OncePerRequestFilter {

    private final JwtService jwtService;
    private final CustomUserDetailsService userDetailsService;
    private final ObjectMapper objectMapper;

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain) throws ServletException, IOException {

        String token = extractToken(request);

        if (token == null) {
            filterChain.doFilter(request, response);
            return;
        }

        try {
            String email = jwtService.extractEmail(token);

            if (email != null
                && SecurityContextHolder.getContext()
                    .getAuthentication() == null) {

                UserDetails userDetails =
                    userDetailsService.loadUserByUsername(email);

                if (jwtService.isTokenValid(token, userDetails)) {

                    UsernamePasswordAuthenticationToken authToken =
                        new UsernamePasswordAuthenticationToken(
                            userDetails, null,
                            userDetails.getAuthorities());

                    authToken.setDetails(
                        new WebAuthenticationDetailsSource()
                            .buildDetails(request));

                    SecurityContextHolder.getContext()
                        .setAuthentication(authToken);
                }
            }

            filterChain.doFilter(request, response);

        } catch (ExpiredJwtException e) {
            log.debug("JWT expired");
            sendError(response, HttpStatus.UNAUTHORIZED,
                "Token has expired");
        } catch (SignatureException e) {
            log.warn("JWT signature mismatch from {}", request.getRemoteAddr());
            sendError(response, HttpStatus.UNAUTHORIZED,
                "Invalid token signature");
        } catch (MalformedJwtException e) {
            log.warn("Malformed JWT from {}", request.getRemoteAddr());
            sendError(response, HttpStatus.UNAUTHORIZED,
                "Malformed token");
        } catch (Exception e) {
            log.warn("JWT processing failed", e);
            sendError(response, HttpStatus.UNAUTHORIZED,
                "Invalid token");
        }
    }

    private String extractToken(HttpServletRequest request) {
        // Check header first (microservice/Postman)
        String authHeader = request.getHeader("Authorization");
        if (authHeader != null && authHeader.startsWith("Bearer ")) {
            return authHeader.substring(7);
        }
        // Then check cookie (SPA)
        Cookie[] cookies = request.getCookies();
        if (cookies != null) {
            for (Cookie cookie : cookies) {
                if ("jwt".equals(cookie.getName())) {
                    return cookie.getValue();
                }
            }
        }
        return null;
    }

    private void sendError(HttpServletResponse response,
            HttpStatus status, String message) throws IOException {
        response.setStatus(status.value());
        response.setContentType(MediaType.APPLICATION_JSON_VALUE);
        ApiError error = new ApiError(message);
        objectMapper.writeValue(response.getWriter(), error);
    }
}
```

## How the Filter Validates Each Request

1. Request arrives → `JwtAuthFilter.doFilterInternal()`
2. `extractToken()` — checks `Authorization` header, then cookie
3. No token? → `filterChain.doFilter()` → public endpoints work, protected get 401
4. Token found → `jwtService.extractEmail(token)`
5. `userDetailsService.loadUserByUsername(email)` → fetch from DB (`CustomUserDetailsService` in [5.3 Authentication Core Components](<5.3. Authentication Core Components.md>))
6. `jwtService.isTokenValid()` → check signature + expiry
7. Valid → set `SecurityContext` → request proceeds to controller
8. Invalid → `sendError()` → 401 with `ApiError` body (errors flow direct, never wrapped — see [2.8 API Response Patterns](<../02-spring-mvc-and-rest-apis/2.8. API Response Patterns.md>))

## Exception Handling

JWT exceptions (expired, malformed, bad signature) happen inside `JwtAuthFilter` — **before** `DispatcherServlet`. `@RestControllerAdvice` only sees exceptions thrown *inside* the dispatcher, so filter-thrown exceptions never reach it.

```
Request
   │
   ▼
Filters (JWT exceptions happen here)
   │  ✗ Can't use @RestControllerAdvice
   │
   ▼
DispatcherServlet
   │
   ▼
Controller (auth exceptions happen here)
      ✓ @RestControllerAdvice works here
```

### Filter-Level Exceptions (handled in `JwtAuthFilter`)

The `JwtAuthFilter` above already wires this in via its `sendError(...)` helper — each `catch` block calls it with the appropriate status and message. Use `ObjectMapper` to manually write the JSON response in the same `ApiError` shape `@RestControllerAdvice` returns.

| Exception | Cause | HTTP Status |
|---|---|---|
| `ExpiredJwtException` | Token past expiry time | 401 |
| `SignatureException` | Token tampered with | 401 |
| `MalformedJwtException` | Not a valid JWT format | 401 |
| `UnsupportedJwtException` | Unsupported JWT type | 401 |
| Generic `Exception` | Any other issue | 401 |

### Controller-Level Exceptions (`GlobalExceptionHandler`)

These ARE caught by `@RestControllerAdvice` since they happen inside `DispatcherServlet`:

| Exception | Cause | HTTP Status |
|---|---|---|
| `AuthenticationException` | Invalid email or password (login) | 401 |
| `BadCredentialsException` | Wrong password | 401 |
| `AccountExpiredException` | Account expired | 401 |
| `CredentialsExpiredException` | Password expired | 401 |
| `IllegalArgumentException` | Email already taken (register) | 400 |

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(AuthenticationException.class)
    public ResponseEntity<ApiError> handleAuthException(
            AuthenticationException ex) {
        ApiError error = new ApiError("Invalid email or password");
        log.warn("Authentication failed [id={}]", error.getId());
        return ResponseEntity.status(HttpStatus.UNAUTHORIZED).body(error);
    }

    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<ApiError> handleBadRequest(
            IllegalArgumentException ex) {
        ApiError error = new ApiError(ex.getMessage());
        return ResponseEntity.badRequest().body(error);
    }
}
```

**Consistent error response**: both filter-level and controller-level errors return the same `ApiError` shape, unwrapped (errors flow direct, never inside the success envelope). The trace `id` (UUID) appears in both the response and server logs so support can correlate a client report to the exact failure:

```json
{
    "id": "5f8a3c9b-2e1f-4a8c-9b1a-d4e7f8a9b1c2",
    "message": "Token has expired",
    "timestamp": "2026-03-01T15:30:00Z"
}
```

### Postman — Auto Token Setup

Skip retyping the token on every request: in the Login request's **Scripts** tab, save it to an environment variable; then set the collection's auth to use it.

```javascript
// Login request → Scripts tab
const response = pm.response.json();
pm.environment.set("authToken", response.data.token);
// Adjust the path to match your response shape (response.token, response.access_token, etc.)
```

In the collection / folder → **Authorization** tab, set `Type: Bearer Token`, `Token: {{authToken}}`. Every request inside the collection now reuses the saved token automatically.

## Production Considerations

- **Secret management**: load `jwt.secret` from env vars or a secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager) — never `application.properties` in source control. Rotate on a schedule and on suspected leak; pre-stage the new secret as a fallback validator before rolling so in-flight tokens keep validating during the cutover.
- **Secret strength**: Base64-encoded, ≥256 bits (32 bytes) of entropy for HS256. Generate via `openssl rand -base64 64`, never a memorable string.
- **HTTPS only**: JWTs over plain HTTP can be sniffed and replayed. Enforce TLS at the load balancer / ingress and set `secure=true` on the cookie.
- **Filter-level observability**: filter exceptions never reach `@RestControllerAdvice`, so they bypass any cross-cutting logging advice attached there. Log inside the filter's `catch` blocks before writing the response.
- **Token expiry**: keep access-token expiry short (15–30 min) and pair with refresh tokens. Long-lived JWTs can't be revoked without a server-side blocklist.
- **Clock skew**: `Jwts.parser().setAllowedClockSkewSeconds(60)` tolerates small time drift between issuing and validating nodes in a multi-server deployment.

## Q&A

1. **Why is the JWT secret read from `@Value("${jwt.secret}")` instead of being hard-coded?**
   Secrets must come from environment-specific config (env vars, vault, AWS Parameter Store) — never source control. Hard-coding leaks the signing key, which lets anyone forge valid tokens.

2. **Why does the filter check both the `Authorization` header and a cookie?**
   To support two client types from one backend — microservices/Postman send `Authorization: Bearer ...`, browser SPAs rely on the auto-sent `HttpOnly` cookie. The filter accepts whichever is present, header first.

3. **Why guard with `SecurityContextHolder.getContext().getAuthentication() == null` before setting the context?**
   To avoid overwriting an authentication already set by an earlier filter (e.g., a different auth mechanism running before JWT). Also makes the filter idempotent if it ever runs twice.

4. **Why pass `null` for credentials to `UsernamePasswordAuthenticationToken`?**
   At this point we've already validated the token — the password isn't needed downstream. Storing it in the security context would be a leak.

5. **Why call `userDetailsService.loadUserByUsername` on every request instead of trusting the token claims?**
   Token claims can be stale — the user may have been disabled or had their roles changed since issuance. The DB lookup catches that, at the cost of one query per request.

6. **Why doesn't `@RestControllerAdvice` catch JWT exceptions?**
   `@RestControllerAdvice` is wired into Spring MVC's `DispatcherServlet`. Filters run *before* the dispatcher, so any exception thrown in a filter bubbles up to the Servlet container — never reaching the advice. You must handle filter exceptions inside the filter itself.

7. **What's the difference between `AuthenticationException` and JWT exceptions like `ExpiredJwtException`?**
   `AuthenticationException` (and subclasses like `BadCredentialsException`) is thrown by Spring Security during the controller-driven `authenticationManager.authenticate()` call — caught by `@RestControllerAdvice`. JWT exceptions come from the jjwt library inside your custom filter and must be caught locally.

8. **Why use `ObjectMapper` directly in the filter to write the error?**
   The filter must produce a JSON body in the same `ApiError` shape the rest of the API uses. Without the dispatcher, you don't have the message-converter chain — `ObjectMapper.writeValue(response.getWriter(), ...)` is the manual equivalent.

9. **Why is "Email already taken" a 400, but "Bad credentials" a 401?**
   400 (Bad Request) signals the *input* is invalid in a way the client can fix (pick a different email). 401 (Unauthorized) signals the *credentials* are wrong — the client is not who they claim to be.
