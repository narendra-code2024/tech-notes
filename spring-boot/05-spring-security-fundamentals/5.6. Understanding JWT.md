A JSON Web Token is a compact, self-contained token for authentication. The server creates it, the client stores it and sends it with every request, and the server validates it without storing anything. This is what makes JWT stateless.

---

## Token Structure

```
eyJhbGciOi.eyJzdWIiOi.SflKxwRJSM
|_header__|__payload_|_signature_|
```

Three Base64URL parts separated by dots:

- **Header**: the signing algorithm (e.g. HS256) and token type.
- **Payload**: the claims (email, roles, expiry). Anyone can decode this, so never put passwords or secrets here.
- **Signature**: computed over `header.payload` with a secret key only the server knows; proves the token was not tampered with.

`HS256` (symmetric: one secret signs and verifies) suits a single service. `RS256` (asymmetric: a private key signs, a public key verifies) suits distributed setups where many services must verify but only one issues.

---

## Session vs JWT

JWT fits stateless, distributed setups (microservices, multi-server, cross-domain). Sessions are simpler on a single server (or sticky-session load balancer) and give easy server-side invalidation.

|Aspect|Session|JWT|
|---|---|---|
|State|Server stores sessions.|Server stores nothing.|
|Storage|`JSESSIONID` cookie.|Token in header or `HttpOnly` cookie.|
|Scalability|Hard across servers.|Any server can validate.|
|Logout|Destroy session on server.|Not truly revocable (use refresh tokens).|
|DB hit|Every request (session lookup).|Only at login and refresh.|

> **Note**: a common implementation does a DB lookup per request in `JwtAuthFilter` to reload `UserDetails`, trading performance for accuracy (e.g. picking up role changes); others trust the token claims alone.

> **Reference**: the `JwtAuthFilter` code is in [5.7. JWT Implementation.md](https://claude.ai/chat/5.7.%20JWT%20Implementation.md).

## How JWT Flows in Practice

1. **Login**: `POST /api/auth/login`; the server validates credentials, creates a JWT, and returns it as a JSON body or an `HttpOnly` cookie depending on client type.
2. **API call**: a request carries the token (a `Authorization: Bearer eyJ...` header or an auto-sent `jwt` cookie); `JwtAuthFilter` extracts it, validates signature and expiry, loads the user, and sets `SecurityContext` before the controller runs.
3. **Logout**: the client deletes the token or the server clears the cookie.

Delivery depends on client type:

|Approach|How it works|Used by|
|---|---|---|
|`HttpOnly` cookie|Server sets `Set-Cookie: jwt=...; HttpOnly; Secure`; browser sends it on its own.|Browser SPAs.|
|JSON body|Server returns `{ token: "..." }`; client stores it and sends a `Bearer` header.|Microservices, mobile, Postman.|

## JWT Storage: XSS / CSRF Tradeoffs

An `HttpOnly` cookie protects against XSS (JS cannot read it) but introduces CSRF: the browser auto-attaches the cookie, and since the JWT is itself the credential, a forged cross-site request authenticates with no separate token to stop it. A `Bearer` header avoids CSRF because browsers never auto-attach headers, but the token must live somewhere JS can reach to set it.

|Storage|XSS-safe?|CSRF-safe?|What you need|
|---|---|---|---|
|`Authorization: Bearer` (memory)|Yes|Yes|Nothing; headers are not auto-sent.|
|`HttpOnly` cookie|Yes|No|Add `SameSite=Lax/Strict` or a CSRF token.|
|`HttpOnly` cookie + `SameSite=Lax`|Yes|Yes|Recommended for SPAs.|
|`localStorage`|No|Yes|Avoid; XSS can steal it.|
|Non-`HttpOnly` cookie|No|No|Never use.|

> **Reference**: CSRF defenses (SameSite, double-submit token) are detailed in [5.1. Security Concepts and Attacks.md](https://claude.ai/chat/5.1.%20Security%20Concepts%20and%20Attacks.md).

## Q&A

1. **Is the JWT payload encrypted?** No, it is Base64URL-encoded JSON; anyone with the token can read the claims. The signature only proves the token was not tampered with, so never put passwords or secrets in the payload.
    
2. **How does the server validate a JWT without storing anything?** It recomputes the signature over `header.payload` with its secret key and compares it to the token's signature. A match means the token is genuine and unchanged, with no DB lookup needed for validation itself.
    
3. **Why is JWT logout hard?** The token is self-validating, so the server keeps no record of issued tokens to invalidate. The usual workarounds are short expiry plus refresh tokens, a server-side blocklist, or rotating signing keys.
    
4. **Stateless JWT vs server-side session, when do you pick which?** JWT for distributed or microservice systems where any node should validate without shared state; sessions when you need easy invalidation, smaller tokens, or already run a single-server or sticky-session setup.
    
5. **Why is storing a JWT in `localStorage` discouraged?** `localStorage` is readable by any script on the page, so a single XSS flaw leaks every user's token. An `HttpOnly` cookie or an in-memory token with silent refresh avoids that.

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|Nature|Stateless, self-contained; server stores nothing.|
|Structure|`header.payload.signature`, dot-separated Base64URL.|
|Payload|Readable by anyone; never store secrets.|
|Signature|`HMAC(header.payload, secret)`; proves integrity, not confidentiality.|
|HS256 vs RS256|Symmetric secret vs asymmetric (sign once, many verify).|
|Validation|Recompute signature + check expiry; no DB needed.|
|Logout|Not truly revocable; short expiry + refresh, or blocklist.|
|Header path|XSS- and CSRF-safe; nothing auto-sent.|
|Cookie path|XSS-safe; needs `SameSite` or a CSRF token.|
|localStorage|Avoid; XSS-readable.|
