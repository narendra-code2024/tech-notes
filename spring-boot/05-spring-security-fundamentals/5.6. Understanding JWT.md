**JSON Web Token**: a compact, self-contained token for authentication. Server creates it, client stores it, client sends it with every request. Server validates it without storing anything — **stateless**.

---

## Token Structure

```
eyJhbGciOi.eyJzdWIiOi.SflKxwRJSM
|_header__|__payload_|_signature_|
```

Three parts separated by dots:

- **Header**: algorithm used (e.g., HS256)
- **Payload**: data (email, roles, expiry). Anyone can decode this — **never put passwords here**: **Signature**: created using a secret key only your server knows. Proves token wasn't tampered with

---

## Session vs JWT

JWT is the choice for stateless, distributed setups — microservices, multi-server deployments, cross-domain authentication. Sessions are simpler when you have a single server (or sticky-session load balancer) and want easy server-side invalidation.

|Aspect|Session|JWT|
|---|---|---|
|State|Server stores sessions (stateful)|Server stores nothing (stateless)|
|Storage|JSESSIONID cookie|Token in header or HttpOnly cookie|
|Scalability|Hard across multiple servers|Easy — any server can validate|
|Logout|Destroy session on server|Can't truly invalidate (use refresh tokens)|
|DB hit|Every request (session lookup)|Only during login and refresh|

> **Note:** Our implementation does a DB lookup on every request inside `JwtAuthFilter` to load `UserDetails`. This is a design choice for accuracy (e.g., if roles change). Some implementations skip this and trust the token claims alone for better performance. See [5.7 JWT Implementation](<5.7. JWT Implementation.md>) → JwtAuthFilter for the code.

## How JWT Flows in Practice

1. **Login**: `POST /api/auth/login` → server validates credentials → creates JWT → returns token (JSON body **or** HttpOnly cookie, depending on client type)
2. **API call**: `GET /api/posts` carrying the token (either `Authorization: Bearer eyJ...` header **or** auto-sent `jwt` cookie, depending on client type) → `JwtAuthFilter` extracts the token from whichever is present → validates signature + expiry → extracts email → loads user from DB → sets `SecurityContext` → request reaches controller
3. **Logout**: client deletes token / server clears cookie

The token can be delivered to the client (and back) in two ways:

|Approach|How It Works|Used By|
|---|---|---|
|**HttpOnly cookie**|Server sets `Set-Cookie: jwt=eyJ...; HttpOnly; Secure`, browser sends it automatically|Browser SPAs|
|**JSON body**|Server returns `{ token: "eyJ..." }`, client stores it and sends via `Authorization: Bearer` header|Microservices, Postman, mobile apps|

How the client sends it:

- **SPA (cookie path):** `fetch('/api/posts', { credentials: 'include' });` — browser attaches the cookie automatically
- **Microservice / Postman (header path):** `Authorization: Bearer eyJhbGciOi...` — sent manually on each request

## JWT Storage — XSS / CSRF Tradeoffs

Storing the JWT in an HttpOnly cookie protects against **XSS** (JavaScript can't read it), but introduces a **CSRF risk**: the browser auto-attaches the cookie (which *is* the auth credential) on every request to that domain, including requests triggered by malicious sites. Because the JWT is the credential itself, there's no separate CSRF token in this flow — the cookie alone authenticates the request.

```
Attacker's site makes a request to your API
       │
       ▼
Browser automatically attaches your jwt cookie
       │
       ▼
Server sees valid JWT → executes the request
       │
       ▼
CSRF attack succeeds — attacker didn't need to read the cookie
```

Each storage option trades XSS and CSRF protection differently:

|Storage|XSS Safe?|CSRF Safe?|What You Need|
|---|---|---|---|
|`Authorization: Bearer` header (from memory)|Yes|Yes|Nothing extra — browser doesn't auto-send headers|
|`HttpOnly` cookie|Yes|No|Add `SameSite=Lax/Strict` or CSRF token|
|`HttpOnly` cookie + `SameSite=Lax`|Yes|Yes|Recommended for SPAs|
|`localStorage`|No|Yes|Avoid — XSS can steal it|
|Non-HttpOnly cookie|No|No|Never use|

> Conceptual deep-dive on CSRF defenses (SameSite, double-submit token) lives in [5.1 Security Concepts and Attacks](<5.1. Security Concepts and Attacks.md>) → CSRF Prevention Methods.

## Q&A

1. **Is the JWT payload encrypted?**
   No — it's Base64URL-encoded JSON. Anyone with the token can decode and read claims. The signature only proves the token wasn't *tampered with*. Never put passwords or secrets in the payload.

2. **How does the server validate a JWT without storing anything?**
   It re-computes the signature over `header.payload` using its secret key and compares. If the recomputed signature matches the one in the token, the token is genuine and unchanged. No DB lookup needed for validation itself.

3. **Why is JWT logout hard?**
   The token is self-validating — the server has no record of issued tokens to invalidate. Common workarounds: short expiry + refresh tokens, server-side blocklist, or rotating signing keys.

4. **Stateless JWT vs server-side session — when do you pick which?**
   JWT for distributed/microservice systems where you want any node to validate without shared state. Sessions when you need easy invalidation, smaller tokens, or already have a sticky-session/single-server setup.

5. **Why is storing JWT in `localStorage` discouraged?**
   `localStorage` is readable by any JavaScript on the page. A single XSS vulnerability leaks every user's token. `HttpOnly` cookies (or in-memory + silent refresh) prevent this.
