## How @Transactional Works

When you annotate a method with `@Transactional`, Spring wraps it using **AOP proxies** (covered in AOP chapter):

```
Method call
  â”‚
  â–Ľ
Proxy intercepts
  â”‚
  â–Ľ
Begin transaction (open DB connection)
  â”‚
  â–Ľ
Execute actual method
  â”‚
  â”śâ”€â”€ Success (no exception)
  â”‚     â†’ COMMIT (changes saved)
  â”‚
  â””â”€â”€ RuntimeException / Error thrown
        â†’ ROLLBACK (changes undone)
```

```java
@Service
public class AccountService {

    @Transactional
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        accountRepository.debit(fromId, amount);   // Step 1
        accountRepository.credit(toId, amount);     // Step 2
        // If Step 2 throws â†’ Step 1 is rolled back automatically
    }
}
```

> **Only unchecked exceptions** (RuntimeException, Error) trigger rollback by default. Checked exceptions do **not** trigger rollback unless you configure it:

```java
@Transactional(rollbackFor = Exception.class)  // rollback on ALL exceptions
```

## Transaction Propagation

Controls what happens when a `@Transactional` method calls **another** `@Transactional` method â€” should they share the same transaction or get separate ones?

```java
@Service
public class OrderService {

    @Transactional  // outer method
    public void placeOrder() {
        // save order
        paymentService.processPayment(); // inner method â€” what transaction?
    }
}

@Service
public class PaymentService {

    @Transactional  // inner method
    public void processPayment() {
        // charge payment
    }
}
```

The question propagation answers: when `placeOrder` (outer) calls `processPayment` (inner), should they share the same transaction or should the inner method get its own?

### REQUIRED (default)

Inner method **joins** the outer transaction. Both succeed or fail together.

```
placeOrder() starts Transaction 1
  â”‚
  â–Ľ
processPayment() joins Transaction 1 (same transaction)
  â”‚
  â–Ľ
If processPayment fails â†’ entire Transaction 1 rolls back
  (both order and payment undone)
```

If there's no outer transaction, inner creates its own.

### REQUIRES_NEW

Inner method gets its **own new transaction**. Outer is paused until inner finishes.

```
placeOrder() starts Transaction 1
  â”‚
  â–Ľ
processPayment() starts Transaction 2 (independent)
  Transaction 1 is paused
  â”‚
  â–Ľ
If processPayment fails â†’ only Transaction 2 rolls back
  Transaction 1 can still commit
  (order saved, payment failed)
```

> Useful for audit logging â€” save the log entry even if the main operation fails.

### MANDATORY

Inner method **must** be called within an existing transaction. Throws exception if there's none.

```
placeOrder() has Transaction 1
  â”‚
  â–Ľ
processPayment() joins Transaction 1 (joined)

But if processPayment() called directly (no outer transaction):
  â†’ Exception! "No existing transaction found"
```

> Forces the method to never run standalone â€” must always be part of a larger transaction.

### All Propagation Types

|Propagation|No outer TX exists|Outer TX exists|
|---|---|---|
|**REQUIRED**|Create new|Join outer|
|**REQUIRES_NEW**|Create new|Create new (suspend outer)|
|**MANDATORY**|Exception|Join outer|
|**SUPPORTS**|Run without TX|Join outer|
|**NOT_SUPPORTED**|Run without TX|Suspend outer, run without TX|
|**NEVER**|Run without TX|Exception|
|**NESTED**|Create new|Create nested (savepoint within outer)|

> **REQUIRED** covers most cases. Use **REQUIRES_NEW** when inner operation must commit independently. Others are rare.

## Setting Isolation Level

```java
@Transactional(isolation = Isolation.SERIALIZABLE)
public void incrementBalance(Long id, BigDecimal amount) {
    Account account = accountRepository.findById(id).orElseThrow();
    account.setAmount(account.getAmount().add(amount));
    accountRepository.save(account);
}
```

|Value|Maps To|
|---|---|
|`Isolation.DEFAULT`|Use the database's default (Read Committed for PG, Repeatable Read for MySQL)|
|`Isolation.READ_UNCOMMITTED`|Lowest â€” dirty reads possible|
|`Isolation.READ_COMMITTED`|No dirty reads|
|`Isolation.REPEATABLE_READ`|Consistent reads within transaction|
|`Isolation.SERIALIZABLE`|Highest â€” transactions behave as if sequential|

> For concurrent updates to the same data (like incrementing a balance), use `Isolation.SERIALIZABLE` to prevent lost updates.

## Common @Transactional Attributes

```java
@Transactional(
    isolation = Isolation.READ_COMMITTED,
    propagation = Propagation.REQUIRED,
    rollbackFor = Exception.class,
    readOnly = true,
    timeout = 30
)
```

|Attribute|Purpose|Default|
|---|---|---|
|`isolation`|Isolation level|`DEFAULT` (database's default)|
|`propagation`|Transaction propagation|`REQUIRED`|
|`rollbackFor`|Which exceptions trigger rollback|RuntimeException, Error|
|`readOnly`|Hint to optimize read-only transactions (no write locks)|`false`|
|`timeout`|Max seconds before transaction is rolled back|Database default|

> `readOnly = true` is useful for GET operations â€” tells the database it can optimize since no writes will happen.

## Self-Invocation & Proxy Behavior

`@Transactional` uses AOP proxies, so calling a `@Transactional` method **from within the same class** bypasses the proxy â€” the transaction won't apply.

```java
@Service
public class OrderService {

    public void placeOrder() {
        this.audit();           // invalid - direct call â€” bypasses proxy, no transaction
    }

    @Transactional
    public void audit() { ... }
}
```

**Fixes**: Move `audit()` to a separate bean and inject it â€” calls now go through the proxy.
- Inject the bean into itself (self-injection) and call `self.audit()` instead of `this.audit()`.
- Enable `@EnableAspectJAutoProxy(exposeProxy = true)` and call `((OrderService) AopContext.currentProxy()).audit()`.

> Same behavior applies to private and `final` methods â€” proxies can't intercept them.
 Proxy mechanics covered in [10.4 Spring Proxy and Internal Working of AOP](<../10-aspect-oriented-programming/10.4. Spring Proxy and Internal Working of AOP.md>).

## Q&A

1. **How does `@Transactional` actually work?**
   Spring creates an **AOP proxy** around the bean. The proxy intercepts the call, opens a transaction (via `PlatformTransactionManager`), runs the method, and commits on success or rolls back on a `RuntimeException`/`Error`.

2. **Why doesn't `@Transactional` roll back on checked exceptions by default?**
   Spring inherited EJB's convention: only unchecked exceptions trigger rollback. Override with `@Transactional(rollbackFor = Exception.class)` to roll back on checked exceptions too.

3. **Why doesn't `@Transactional` work on self-invocation or private methods?**
   AOP proxies only intercept external calls through the proxy reference. A `this.method()` call goes directly to the target object, skipping the proxy. Private methods aren't visible to the proxy at all. Move the transactional method to another bean, or inject self via a proxy.

4. **`REQUIRED` vs `REQUIRES_NEW`?**
   `REQUIRED` (default) joins the existing transaction if one exists, else starts one. `REQUIRES_NEW` always starts a fresh transaction, suspending the outer â€” useful when an inner operation must commit independently of the outer outcome (e.g., audit logging).

5. **What does `readOnly = true` do?**
   It's a hint to the persistence layer: skip dirty-checking, optionally route to a read replica, and let the DB optimize (no write locks). Doesn't enforce read-only at the SQL level on most DBs.

6. **How does `@Transactional` interact with isolation levels?**
   `@Transactional(isolation = Isolation.SERIALIZABLE)` sets the JDBC isolation for the transaction's connection. Default is `Isolation.DEFAULT` (the DB's own default â€” see [11.5 Concurrent Transaction Management and Isolation Levels](<11.5. Concurrent Transaction Management and Isolation Levels.md>)).

7. **What's the proxy mode for handling self-invocation?**
   Use `AopContext.currentProxy()` (with `@EnableAspectJAutoProxy(exposeProxy = true)`) or inject the bean into itself via constructor â€” both let you re-enter through the proxy. Cleaner: split into separate beans.
