Caching stores frequently accessed data in a **fast storage layer** so future requests are served quicker than fetching from the original source every time.

```
Without caching:
  Request → DB query (slow, every time)

With caching:
  First request  → DB query → store result in cache
  Next requests  → return from cache (fast, no DB hit)
```

---

## Why Cache?

- Data access from **memory is always faster** than fetching from database
- Reduces costly backend/database requests
- Improves application performance and response time
- Helps applications **scale** under high traffic

---

## Caching at Different Layers

```
User Request
  │
  ▼
DNS Cache (browser/OS)
  │  Domain → IP resolved once, cached for future requests
  │
  ▼
Web Cache (CDN)
  │  Static content (images, CSS, JS) served from edge servers
  │  closer to the user, not from origin server
  │
  ▼
Application Cache (Redis, Memcached)
  │  Frequently accessed data and computation results
  │  stored in memory, avoids repeated DB queries
  │
  ▼
Database Cache (query cache, buffer pool)
     DB's own internal caching layer — low level,
     managed by the database engine itself
```

|Layer|What's Cached|Implemented By|
|---|---|---|
|**DNS**|Domain → IP mapping|Browser, OS (first lookup only, then cached)|
|**Web**|Static content (pages, images, assets)|CDN (Cloudflare, AWS CloudFront)|
|**Application**|Query results, computed data, API responses|Redis, Memcached, Spring Cache|
|**Database**|Query plans, frequently read rows|Database engine internally|

> Our focus in this file is **application-level caching**: caching data inside your Spring Boot app using Spring Cache and Redis.

## Caching Terminologies

### Cache Hit and Cache Miss

```
Request for data
  │
  ▼
Is it in the cache?
  ├── Yes → Cache Hit (return from cache, fast)
  └── No  → Cache Miss (fetch from DB, store in cache for next time)
```

### Cache Eviction Policies

When the cache is full and a new entry needs to be added, which existing entry gets removed?

|Policy|How It Decides|
|---|---|
|**LRU** (Least Recently Used)|Remove the entry that hasn't been accessed for the longest time|
|**LFU** (Least Frequently Used)|Remove the entry that's been accessed the fewest times|
|**FIFO** (First In First Out)|Remove the oldest entry|
|**TTL-based**|Remove entries that have expired (based on time-to-live)|

> LRU is the most commonly used eviction policy.

### Cache Expiry and TTL

**TTL (Time-To-Live)**: how long a cached entry stays valid before it's automatically removed.

```
Cache entry created at 10:00 AM with TTL = 5 minutes
  → Valid until 10:05 AM
  → After 10:05 AM → expired → next request triggers a cache miss → fresh data fetched
```

> Without TTL, stale data could live in cache forever while the actual data in DB has changed.

### Cache Loading Strategies

**Lazy Loading**: cache starts empty. Data loaded only when first requested.

```
App starts → cache is empty
  │
  ▼
First request: GET /api/categories
  → Cache miss → fetch from DB → store in cache → return (slow)
  │
  ▼
Second request: GET /api/categories
  → Cache hit → return from cache (fast)
```

> First user pays the price (cache miss), everyone after gets it fast. Most common strategy.

**Eager Loading / Cache Warming**: pre-load frequently needed data into cache during app startup, before any requests come in.

```
App starts → load categories, config, countries into cache
  │
  ▼
First request: GET /api/categories
  → Cache hit → fast (already loaded, no cold cache)
```

> Used for data you know will be requested frequently — categories, config, country lists, etc. No user ever hits a cold cache for that data.

### Cache Size and Capacity

Cache memory is limited — you can't cache everything. Configure max entries or max memory to prevent the cache from consuming too much resources. Eviction policies kick in when the limit is reached.

## Q&A

1. **What is caching and why is it used?**
   Caching stores frequently accessed data in a fast storage layer (typically memory) so repeat requests skip the expensive source (DB, remote API). It reduces latency, lowers backend load, and helps the app scale.

2. **Cache hit vs cache miss?**
   A **hit** means the requested data was found in cache (fast path). A **miss** means it wasn't, so the app fetches from the source and typically populates the cache for next time.

3. **What is TTL and why does it matter?**
   **TTL** (time-to-live) is how long a cached entry is considered valid. Without TTL, stale data can live in cache indefinitely even after the underlying source changes.

4. **Compare common eviction policies.**
   **LRU** evicts the entry not accessed for the longest time (most common). **LFU** evicts the least frequently accessed. **FIFO** evicts the oldest by insertion. **TTL-based** evicts expired entries regardless of access pattern.

5. **Lazy loading vs cache warming?**
   **Lazy loading** populates the cache on first request (first user pays the miss cost). **Cache warming** pre-loads known hot data at startup so no user hits a cold cache — used for catalogs, config, reference lists.

6. **Where does application-level caching sit relative to other cache layers?**
   Between web/CDN caches (static assets) above and the DB's own buffer pool below. Application cache holds processed query results and computed data in memory (Redis, Memcached, Spring Cache).
