## Why Redis Over Default Cache?

The default Spring cache (`ConcurrentHashMap`) lives **inside your application's memory**. When the app restarts, the cache is gone. It also can't be shared across multiple instances.

```
Default (ConcurrentHashMap):
  App Instance 1 ‚Üí own cache (in memory)
  App Instance 2 ‚Üí own cache (different data)
  App restarts ‚Üí cache lost

Redis (dedicated server):
  App Instance 1 ‚îÄ‚îÄ‚îê
                   ‚îú‚îÄ‚îÄ‚Üí Redis Server (shared cache)
  App Instance 2 ‚îÄ‚îÄ‚îò
  App restarts ‚Üí cache survives (Redis is separate)
```

|                         | Default Cache             | Redis                                |
|-------------------------|---------------------------|--------------------------------------|
| Storage                 | App memory                | Dedicated server                     |
| Survives restart        | No                        | Yes                                  |
| Shared across instances | No                        | Yes                                  |
| Distributed             | No                        | Yes                                  |
| Speed                   | Very fast (same process)  | Fast (network call, still in-memory) |

```
App on AWS us-east-1 (Virginia) ‚Üí Redis on AWS us-east-1
```

> Same provider + same region as your app = fastest connection.

## Serialization ‚Äî The Common Error

When you switch from default cache to Redis, you'll likely get:

```
SerializationException: Cannot serialize
```

**Why?** Default cache stores Java objects in memory ‚Äî no conversion needed. Redis stores data **over the network** as bytes ‚Äî objects must be converted (serialized) to bytes and back.

**Fix:** Your DTOs must implement `Serializable`:

```java
@Data
@Builder
public class EmployeeDto implements Serializable {

    private static final long serialVersionUID = 1L;

    private Long id;
    private String firstName;
    private String lastName;
    private String email;
}
```

|Field|Purpose|
|---|---|
|`implements Serializable`|Tells Java this object can be converted to bytes|
|`serialVersionUID`|Version identifier ‚Äî ensures cached data is compatible after class changes. If you change the DTO structure, update this number to invalidate old cached data.|

> If you don't set `serialVersionUID` and change the class (add/remove fields), deserialization of old cached data will fail with `InvalidClassException`.

## RedisCacheManager Configuration

Customize TTL, key prefix, and serialization:

```java
@Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public RedisCacheManager cacheManager(
            RedisConnectionFactory connectionFactory) {

        RedisCacheConfiguration config = RedisCacheConfiguration
            .defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(30))
            .keySerializerPair(
                RedisSerializationContext.SerializerPair
                    .fromSerializer(new StringRedisSerializer()))
            .valueSerializerPair(
                RedisSerializationContext.SerializerPair
                    .fromSerializer(new GenericJackson2JsonRedisSerializer()))
            .prefixCacheNameWith("myapp:")
            .disableCachingNullValues();

        return RedisCacheManager.builder(connectionFactory)
            .cacheDefaults(config)
            .enableStatistics()
            .build();
    }
}
```

|Method|Purpose|
|---|---|
|`entryTtl(Duration)`|How long cached entries live before expiring|
|`keySerializerPair(...)`|How keys are serialized ‚Äî `StringRedisSerializer` stores keys as readable strings|
|`valueSerializerPair(...)`|How values are serialized ‚Äî `GenericJackson2JsonRedisSerializer` stores values as JSON (human-readable in Redis)|
|`prefixCacheNameWith("myapp:")`|Adds prefix to all cache keys ‚Äî useful when multiple apps share the same Redis|
|`disableCachingNullValues()`|Don't cache null results|
|`enableStatistics()`|Track cache hit/miss stats|

### With JSON Serializer ‚Äî No Need for Serializable

If you use `GenericJackson2JsonRedisSerializer` (shown above), objects are stored as **JSON** in Redis instead of Java byte serialization. This means:

- DTOs **don't need** `implements Serializable`
- Cached data is human-readable in Redis (`GET` shows JSON)
- Works across different app versions better than Java serialization

> JSON serialization is the recommended approach. Java `Serializable` works but is harder to debug and version.

## No Code Changes for Caching Logic

The `@Cacheable`, `@CachePut`, `@CacheEvict` annotations from [11.2 Spring Boot Default Caching](<11.2. Spring Boot Default Caching.md>) stay exactly the same ‚Äî they don't care whether the underlying cache is `ConcurrentHashMap` or Redis. You only change the config and dependency.

```java
// Same annotation ‚Äî works with default cache AND Redis
@Cacheable(cacheNames = "employees", key = "#id")
public EmployeeDto getEmployeeById(Long id) {
    return employeeRepository.findById(id)
        .map(employeeMapper::toDto)
        .orElseThrow();
}
```

> This is the power of Spring's Cache Abstraction ‚Äî swap providers without touching business code.

## Production Considerations

- **TTL is mandatory.** Always set `entryTtl(...)` on the `RedisCacheConfiguration` ‚Äî without it, stale entries can outlive their source-of-truth indefinitely. Pick a value tied to your data freshness SLA (e.g., 5‚Äì30 min for catalog data, seconds for hot inventory).
- **Per-cache TTLs.** `RedisCacheManager.builder(cf).withCacheConfiguration("shortLived", shortConfig).withCacheConfiguration("longLived", longConfig)` lets each cache name have its own TTL.
- **Manual invalidation alongside TTL.** Time-based eviction is a safety net; combine with explicit `@CacheEvict` on writes so updates show up immediately rather than after expiry.
- **Connection pooling.** Spring Boot ships **Lettuce** by default ‚Äî non-blocking, thread-safe, single connection multiplexed. Switch to **Jedis** with a connection pool (`spring.data.redis.client-type: jedis` + `lettuce.pool.*` / `jedis.pool.*` props) only if you have a specific reason; Lettuce handles most workloads.
- **Serializer choice.** Use `GenericJackson2JsonRedisSerializer` over Java native (`JdkSerializationRedisSerializer`) ‚Äî readable in `redis-cli`, cross-language safe, tolerant to non-breaking class changes.
- **Failover.** For high availability use Redis Sentinel or Cluster ‚Äî configure via `spring.data.redis.sentinel.*` or `spring.data.redis.cluster.*`. Treat cache as a soft dependency: a Redis outage should degrade latency, not break the app (consider `@CacheEvict(beforeInvocation = true)` patterns and circuit-breaker the cache layer if needed).

## Q&A

1. **Why use Redis over the default in-memory cache?**
   Redis survives app restarts and is shared across instances ‚Äî essential for multi-node deployments. The default `ConcurrentHashMap` cache is per-JVM and wiped on restart.

2. **What causes `SerializationException` when switching to Redis?**
   Redis stores bytes over the network, so cached objects must be serializable. Either implement `java.io.Serializable` on the DTO or configure a JSON serializer (`GenericJackson2JsonRedisSerializer`) on the `RedisCacheManager`.

3. **Why is JSON serialization preferred over Java `Serializable`?**
   JSON values are human-readable in Redis (easy to debug with `GET`), version-tolerant across class changes, and don't require DTOs to implement `Serializable`. Java byte serialization is brittle when fields change without bumping `serialVersionUID`.

4. **What is `serialVersionUID` and why does it matter?**
   It's a version identifier on `Serializable` classes. If you change the class structure and the old value remains, deserializing cached data throws `InvalidClassException`. Bump the UID to invalidate old cached data after breaking changes.

5. **Do the `@Cacheable` / `@CachePut` / `@CacheEvict` annotations change when moving to Redis?**
   No ‚Äî Spring's Cache Abstraction keeps the annotations identical. Only the dependency, `spring.cache.type`, and optionally the `RedisCacheManager` bean change.


6. **What does `prefixCacheNameWith("myapp:")` solve?**
   It namespaces keys so multiple apps sharing one Redis instance don't collide. Keys become `myapp:employees::1` instead of `employees::1`.
