## The Problem — Concurrent Transactions

Multiple transactions running at the same time can interfere with each other. Open **two terminal windows** connected to the same database to see this in action.

```
Terminal 1: Transaction A (your transfer)
Terminal 2: Transaction B (someone else's query)
Both hitting the same data at the same time
```

---

## Three Problems That Can Occur

### 1. Dirty Read

Transaction B reads data that Transaction A **hasn't committed yet**. If A rolls back, B has read data that never actually existed.

```
Transaction A                    Transaction B
     │                                │
T0:  UPDATE accounts                  │
     SET amount = 900                 │
     WHERE id = 1;                    │
     (not committed yet)              │
     │                                │
T1:  │                           SELECT amount FROM accounts
     │                           WHERE id = 1;
     │                           → sees 900 (uncommitted!)
     │                                │
T2:  ROLLBACK;                        │
     (amount is back to 1000)    B now has 900 — data that
                                 never actually existed
```

> B read "dirty" data — a value that was never committed.

### 2. Non-Repeatable Read

Transaction A reads the same row twice and gets **different values** because Transaction B committed a change in between.

```
Transaction A                    Transaction B
     │                                │
T0:  SELECT amount FROM accounts      │
     WHERE id = 1;                    │
     → sees 1000                      │
     │                                │
T1:  │                           UPDATE accounts
     │                           SET amount = 900
     │                           WHERE id = 1;
     │                           COMMIT;
     │                                │
T2:  SELECT amount FROM accounts      │
     WHERE id = 1;                    │
     → sees 900 (different!)          │
```

> Same query, same transaction, different result. The read is not "repeatable."

### 3. Phantom Read

Transaction A runs the same query twice and gets **different rows** because Transaction B inserted/deleted rows in between.

```
Transaction A                    Transaction B
     │                                │
T0:  SELECT * FROM accounts           │
     WHERE amount < 1000;             │
     → returns 3 rows                 │
     │                                │
T1:  │                           INSERT INTO accounts
     │                           VALUES ('New User', 500);
     │                           COMMIT;
     │                                │
T2:  SELECT * FROM accounts           │
     WHERE amount < 1000;             │
     → returns 4 rows (new row!)      │
```

> Same query, same transaction, but a "phantom" row appeared.

---

## Lost Update Problem

Two transactions read the same value and both update it — one update gets overwritten.

```
Account X = 100

Transaction A (friend credits ₹15)    Transaction B (you pay bill ₹25)
     │                                      │
     Read(X) → X = 100                      │
     X = 100 + 15 = 115                     │
     │                                      Read(X) → X = 100
     │                                      X = 100 - 25 = 75
     │                                      WRITE(X) → X = 75
     │                                      │
     WRITE(X) → X = 115                     │

Final X = 115 - incorrect (should be 90 = 100 + 15 - 25)
Transaction B's update is lost!
```

> This is why isolation levels exist — to prevent these problems.

---

## Isolation Levels

Isolation levels control **how much one transaction can see another's uncommitted/committed changes**. Higher isolation = fewer problems but slower performance.

### Read Uncommitted (Lowest)

Transactions can see **uncommitted** changes from other transactions.

```sql
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
```

> All problems possible: dirty reads, non-repeatable reads, phantom reads. Rarely used.

### Read Committed (Default in PostgreSQL)

Transactions only see **committed** changes. No dirty reads.

```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- or
SHOW transaction_isolation; -- check current level
```

```
T1: UPDATE amount = 900 WHERE id = 1  (not committed)
T2: SELECT amount WHERE id = 1        → still sees 1000 (old committed value)
T1: COMMIT
T2: SELECT amount WHERE id = 1        → now sees 900 (committed)
```

> Default in PostgreSQL. Prevents dirty reads, but non-repeatable reads and phantom reads can still happen.

### Repeatable Read (Default in MySQL)

Once a transaction reads a value, **it always sees the same value** throughout the transaction — even if another transaction commits a change.

```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

```
T1: SELECT amount WHERE id = 1  → sees 1000
T2: UPDATE amount = 900 WHERE id = 1; COMMIT;
T1: SELECT amount WHERE id = 1  → still sees 1000 (snapshot from start)
T1: UPDATE amount WHERE id = 1  → ERROR: could not serialize access
                                   due to concurrent update
```

> T1 sees a consistent snapshot from the start of its transaction. If T1 tries to update a row that T2 already changed, it gets an error instead of causing a lost update.

### Serializable (Highest)

Transactions behave as if they ran **one after another**, not concurrently. Maximum safety, lowest performance.

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

```
T1: credits ₹15 (starts first)
T2: debits ₹25 (starts second, finishes first)

T2 commits → X = 75
T1 tries to commit → CONFLICT detected → T1 is rolled back

T1 must retry → reads fresh value → calculates correctly
```

> Even if T1 started first, if T2 finished first and conflicts are detected, T1 is aborted. This prevents inconsistent states — no successful transaction should leave data in a wrong state.

---

## Isolation Levels Summary

| Level                | Dirty Read   | Non-Repeatable Read  | Phantom Read   | Performance |
|----------------------|--------------|----------------------|----------------|-------------|
| **Read Uncommitted** | Possible     | Possible             | Possible       | Fastest     |
| **Read Committed**   | Prevented    | Possible             | Possible       | Fast        |
| **Repeatable Read**  | Prevented    | Prevented            | Possible*      | Moderate    |
| **Serializable**     | Prevented    | Prevented            | Prevented      | Slowest     |

---

## PostgreSQL vs MySQL

||PostgreSQL|MySQL (InnoDB)|
|---|---|---|
|Default level|Read Committed|Repeatable Read|
|How Serializable works|**Snapshot Isolation (SSI)**: doesn't lock, detects conflicts and aborts transactions|**Row/Range Locks**: locks rows, transactions wait for locks|
|Dirty Read at Read Uncommitted|Not possible (PG treats it as Read Committed)|Possible|
|Phantom Read at Repeatable Read|Not possible (PG prevents it)|Possible*|
|Conflict handling|Detects and aborts conflicting transactions|Transactions wait (can deadlock)|

> PostgreSQL is more optimistic — it lets transactions run concurrently and aborts if conflicts are detected. MySQL is more pessimistic — it locks rows and makes transactions wait. PostgreSQL's approach generally gives **better concurrency** because transactions don't block each other.

## Checking and Setting Isolation Level

```sql
-- PostgreSQL: check current level
SHOW transaction_isolation;

-- PostgreSQL: set for current transaction
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- your queries here
COMMIT;

-- MySQL: check current level
SELECT @@transaction_isolation;

-- MySQL: set for current session
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

## What to Use?

```
Most applications → Read Committed (PostgreSQL default) — good balance
Financial / critical data → Serializable — maximum safety
High-read, low-write → Repeatable Read — consistent reads
```

> You don't usually set isolation levels manually in Spring Boot — `@Transactional` handles it. See [11.6 Transactional Annotation in Spring Boot](<11.6. Transactional Annotation in Spring Boot.md>).

## Q&A

1. **Dirty read vs non-repeatable read vs phantom read?**
   **Dirty read**: reading another transaction's uncommitted changes. **Non-repeatable read**: same row, same query, different value due to a committed update mid-transaction. **Phantom read**: same range query returns different rows because of a committed insert/delete.

2. **What is the lost update problem?**
   Two transactions read the same value, both compute updates from it, and the second write overwrites the first — losing the first transaction's effect. Solved by isolation level (Serializable), optimistic locking (`@Version`), or pessimistic locking (`SELECT ... FOR UPDATE`).

3. **What's the default isolation level in PostgreSQL vs MySQL?**
   PostgreSQL: **Read Committed**. MySQL (InnoDB): **Repeatable Read**.

4. **How does PostgreSQL implement Serializable differently from MySQL?**
   PostgreSQL uses **Snapshot Serializable Isolation (SSI)**: transactions don't block; conflicts are detected and the conflicting transaction is aborted. MySQL uses **row/range locks**: transactions wait, with deadlock potential. PG is optimistic; MySQL is pessimistic.

5. **Which isolation level prevents phantom reads?**
   **Serializable** in standard SQL. PostgreSQL's Repeatable Read also prevents phantoms (stricter than the SQL spec); MySQL's Repeatable Read does not by spec but typically does in practice via gap locks.

6. **Trade-off of higher isolation?**
   Stronger guarantees but lower throughput — more locking, more conflict aborts, more retries. Most apps run on Read Committed; reach for Serializable only when correctness on concurrent writes demands it.

## Quick Reference

| Concept | Remember |
|---|---|
| Read Uncommitted | Dirty + non-repeatable + phantom all possible — rarely used |
| Read Committed | No dirty reads; non-repeatable + phantom still possible — PG default, most apps |
| Repeatable Read | No dirty + no non-repeatable; phantom possible per spec — MySQL default |
| Serializable | All three prevented — financial / critical, slowest |
| PostgreSQL Serializable | **SSI**: optimistic, detects conflicts and aborts |
| MySQL Serializable | Row/range locks — pessimistic, can deadlock |
| Lost update fix | Higher isolation, optimistic `@Version`, or pessimistic `SELECT ... FOR UPDATE` |
