## Why Centralized Logging?

Zipkin handles distributed tracing (timing + request paths) but doesn't offer deep log analysis. In a microservice system, when something goes wrong you'd have to check logs of each service individually ‚Äî tedious with 10+ services.

ELK Stack solves this by providing:

- Centralized view of all microservice logs at one location
- Deep log analysis and powerful search (e.g., search by Trace ID across all services)
- Long-term log storage
- Real-time monitoring, alerting, and dashboards

---

## Alternatives to ELK

- **Grafana Loki + Promtail + Grafana**: Loki indexes only labels (not full text), so storage cost is dramatically lower; great fit if you already run Prometheus/Grafana for metrics and Tempo for traces.
- **OpenSearch + OpenSearch Dashboards**: AWS-led fork of Elasticsearch/Kibana under the Apache 2.0 licence; drop-in replacement after Elastic moved to SSPL.
- **Filebeat instead of Logstash**: lighter agent for shipping logs to Elasticsearch directly when you don't need Logstash's transformation pipeline.
- **Cloud-managed:** AWS CloudWatch Logs + Insights, GCP Cloud Logging, Datadog Logs ‚Äî less operational overhead, vendor lock-in trade-off.

The ELK setup below applies to OpenSearch with minor URL/dashboard differences.

---

## What is the ELK Stack?

Three open-source tools from Elastic, used together:

```
Spring Boot Apps ‚Üí logs ‚Üí Logstash ‚Üí Elasticsearch ‚Üê Kibana (visualize)
```

- **Elasticsearch**: NoSQL database (based on Lucene) that stores and indexes log data. Excels at indexing streams of data and fast aggregation queries. In Elasticsearch, **indexes** are equivalent to tables in SQL.
- **Logstash**: data pipeline tool that **collects, parses, transforms, and ships** log data from microservices to Elasticsearch. Configured via a `.conf` file with input (log files) and output (Elasticsearch).
- **Kibana**: analytics and visualization layer on top of Elasticsearch. Used to search, view, filter, and build dashboards from the indexed log data ‚Äî no need to write raw queries.


### Step 4 ‚Äî Configure Logback in Each Microservice

Create `logback-spring.xml` in `src/main/resources` of each microservice:

```xml
<configuration>
  <!-- Console appender -->
  <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
    <encoder>
      <pattern>%d{yyyy-MM-dd HH:mm:ss} [%thread] [%X{traceId},%X{spanId}] %-5level %logger{36} - %msg%n</pattern>
    </encoder>
  </appender>

  <!-- Rolling file appender ‚Äî same pattern as STDOUT -->
  <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>logs/${spring.application.name}/application.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
      <!-- New file per day, max 10MB per file -->
      <fileNamePattern>logs/${spring.application.name}/application-%d{yyyy-MM-dd}.%i.log</fileNamePattern>
      <timeBasedFileNamingAndTriggeringPolicy
        class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
        <maxFileSize>10MB</maxFileSize>
      </timeBasedFileNamingAndTriggeringPolicy>
    </rollingPolicy>
    <encoder>
      <pattern>${same pattern as STDOUT above}</pattern>
    </encoder>
  </appender>

  <root level="INFO">
    <appender-ref ref="STDOUT"/>
    <appender-ref ref="FILE"/>
  </root>
</configuration>
```

> Log pattern includes **Trace ID and Span ID** from Micrometer ‚Äî enabling cross-service log search by Trace ID in Kibana.  
> Each microservice gets its own log folder: `logs/order-service/`, `logs/inventory-service/`, etc.

> **Prefer structured JSON logs in production.** Plain-text patterns force Logstash (or the ingest pipeline) to grok-parse every line ‚Äî fragile and CPU-heavy. With `logstash-logback-encoder`'s `LogstashEncoder`, Logback writes one JSON object per event with `timestamp`, `level`, `message`, `traceId`, `spanId`, `service`, MDC fields, and stack traces already split into fields ‚Äî Logstash just forwards. This makes Kibana filters on individual fields trivial (`level: ERROR AND service: order-service`).

### Step 5 ‚Äî Configure Logstash

Create `logstash.conf` inside the Logstash installation folder (`logstash-<version>/`):

```ruby
input {
  file {
    type  => "log"
    # Absolute path to your project's root logs folder
    # Replace <your-project-root> with the absolute path to your project
    # e.g. /Users/john/documents/ecommerce
    path  => "<your-project-root>/logs/**/*.log"
    start_position => "beginning"
  }
}

output {
  elasticsearch {
    hosts    => ["https://localhost:9200"]
    index    => "spring-boot-logs-%{+YYYY.MM.dd}"
    ssl_certificate_verification => false
    user     => "elastic"
    password => "<your-elasticsearch-password>"
  }
}
```

> To get the absolute path of your project root: right-click the project root folder in IntelliJ ‚Üí **Copy Path ‚Üí Absolute Path**.

### Step 6 ‚Äî Run Logstash

```bash
cd logstash-<version>
bin/logstash -f logstash.conf
```

Logstash reads all log files, converts each line into a structured event, and ships them to Elasticsearch.

## Searching Logs in Kibana

The real power ‚Äî search all microservice logs by **Trace ID**:

```
Search: <traceId>   ‚Üí  shows logs from ALL services for that one request
                        sorted by timestamp
```

Example output:

```
[order-service]     Creating order method called    traceId=abc123
[inventory-service] Reducing stock for product      traceId=abc123
```

> One Trace ID spans across all microservices ‚Äî so one search gives the full picture of a request across the entire system.

## Tracing vs Logging ‚Äî Summary

||Zipkin (tracing)|ELK Stack (logging)|
|---|---|---|
|Purpose|Request flow & timing|Centralized log storage & search|
|Shows|Which services called & how long|What happened inside each service|
|Search by|Trace ID (visual timeline)|Trace ID, service name, log level, time|
|Best for|Finding bottlenecks|Debugging errors & exceptions|

## Q&A

1. **What does each piece of ELK do?** **Elasticsearch** stores and indexes log events; **Logstash** ingests/parses/ships logs from files into Elasticsearch; **Kibana** is the UI for search, dashboards, and alerts.
2. **How do logs from a Spring Boot app reach Elasticsearch?** Logback writes structured log lines (with Trace ID + Span ID) to rolling files. Logstash tails those files via its `input { file ... }` config and ships each event to Elasticsearch.
3. **Why include Trace ID and Span ID in the log pattern?** It links logs to Zipkin traces and lets you pivot from a slow trace into the actual log lines from every service for that request.
4. **What's an Elasticsearch index?** Roughly equivalent to a SQL table ‚Äî a named collection of documents (here, log events) with a defined schema. The `index` setting in Logstash routes events into a specific index (e.g., `spring-boot-logs-2026.04.25`).
5. **Why use ELK if you already have Zipkin?** Zipkin shows **where** time is spent across services; ELK shows **what** the application logged (errors, stack traces, business events). Together they give complete visibility ‚Äî search by Trace ID in either tool.
6. **What's the role of `logback-spring.xml` over plain `logback.xml`?** `logback-spring.xml` is loaded by Spring Boot after the environment is initialized ‚Äî so you can use Spring properties like `${spring.application.name}` in patterns and file paths.
