## The Problem

When a config value is changed in GitHub, running microservices still hold the **old value** in their application context — they must restart to get the new value. This causes downtime, which is not acceptable in production.

> Config Server does **not** automatically detect GitHub changes. It does a fresh `git pull` only when a microservice requests config — triggered by `/actuator/refresh`.

> Builds on [13.3 Centralized Configuration Server](<13.3. Centralized Configuration Server.md>).

---

## What is `@RefreshScope`?

A Spring Cloud annotation that enables **dynamic refreshing of Spring-managed beans**: allowing configuration to be updated at runtime without restarting the application.

- Only beans explicitly marked with `@RefreshScope` are refreshed
- Works in conjunction with Spring Cloud Config
- Requires **Spring Boot Actuator** to expose the `/actuator/refresh` endpoint

---

## How It Works

```
1. Config value changes in GitHub
         │
         ▼
2. Call POST /actuator/refresh on the microservice
         │
         ▼
3. Microservice asks Config Server for latest config
         │
         ▼
4. Config Server does a fresh git pull from GitHub
         │
         ▼
5. Returns latest values to Microservice
         │
         ▼
6. Spring replaces @RefreshScope bean instances with new ones
   using updated config values (via AOP proxy)
         │
         ▼
7. Next request gets the new value — no restart needed
```

> Spring creates an **AOP proxy** for every `@RefreshScope` bean. Only these beans are reloaded — the rest of the application context stays untouched, so there is **no downtime**.

## Setup

### Step 1 — Add Actuator Dependency

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

Add this to **every microservice** that needs refresh support.

### Step 2 — Expose the Refresh Endpoint

In the global `application.yml` in the GitHub config repo:

```yaml
management:
  endpoints:
    web:
      exposure:
        include: "*"    # exposes all actuator endpoints (dev only)
```

> In production, expose only what you need: `include: refresh,health,info`

### Step 3 — Annotate Beans with `@RefreshScope`

```java
@RestController
@RefreshScope                           // ← marks this bean as refreshable
public class OrderController {

    @Value("${my.variable}")
    private String myVariable;

    @GetMapping("/hello")
    public String hello() {
        return "my.variable = " + myVariable;
    }
}
```

### Step 4 — Trigger the Refresh

After changing a value in GitHub, call:

```
POST http://localhost:8081/actuator/refresh
```

> This is a **POST** request with no body. The response lists all beans that were refreshed.

## Feature Flags Use Case

A practical use case — toggling features on/off at runtime without restarting:

### Config class

```java
@Configuration
@RefreshScope
public class FeaturesEnabledConfig {

    @Value("${features.user-tracking.enabled}")
    private boolean isUserTrackingEnabled;

    public boolean isUserTrackingEnabled() {
        return isUserTrackingEnabled;
    }
}
```

### Controller

```java
@RestController
public class OrderController {

    private final FeaturesEnabledConfig featuresEnabledConfig;

    @GetMapping("/hello")
    public String hello() {
        if (featuresEnabledConfig.isUserTrackingEnabled()) {
            return "User tracking enabled";
        }
        return "User tracking disabled";
    }
}
```

### In GitHub config repo (`order-service-dev.properties`)

```properties
features.user-tracking.enabled=true
```

To disable at runtime — change to `false` in GitHub, then call `POST /actuator/refresh`. No restart needed.

## Don't Abuse `@RefreshScope`

- Do **not** put `@RefreshScope` on every bean
- Each refresh re-initializes all marked beans — unnecessary overhead
- Only use it on beans whose config **needs to change dynamically**

> **What won't refresh:** values injected into non-`@RefreshScope` singletons stay at their startup value. `DataSource`, connection pools, Kafka producers/consumers, and `@ConfigurationProperties` beans not annotated with `@RefreshScope` need a restart (or explicit re-binding) to pick up changes. Server `port`, JPA/Hibernate config, and most auto-configured infrastructure are also not refreshable.

> **Secure the endpoint:** `/actuator/refresh` mutates application state — never expose it publicly. Bind it to an internal management port (`management.server.port`) or protect it with Spring Security. For fan-out across many instances, pair with **Spring Cloud Bus** + a broker (RabbitMQ/Kafka) so one `/busrefresh` call refreshes all replicas.

### Good candidates for `@RefreshScope`

|Use Case|Example|
|---|---|
|Feature flags|`features.user-tracking.enabled`|
|Dynamic DB config|connection pool size, timeout|
|API rate limits|`api.rate-limit.requests-per-second`|
|External service URLs|third-party API endpoints|

## Q&A

1. **What does `@RefreshScope` do?** Marks a bean for runtime reloading. Spring wraps it in an AOP proxy; when `/actuator/refresh` is called, the underlying instance is recreated with the latest config — without restarting the app.
2. **Why isn't `@RefreshScope` on by default for every bean?** Reloading every bean would be expensive and risky (tearing down singletons, reconnecting pools). Spring only refreshes beans you explicitly opt in.
3. **What endpoint triggers a refresh and what HTTP method?** `POST /actuator/refresh` — no body. The response lists which beans were refreshed.
4. **Does Config Server auto-detect Git changes?** No. It does a fresh `git pull` only when a client requests config (typically triggered by `/actuator/refresh`). For push-based fan-out you'd add Spring Cloud Bus + a message broker.
5. **What dependency is required besides Spring Cloud Config?** `spring-boot-starter-actuator` — without it the `/refresh` endpoint doesn't exist.
6. **Common candidates for `@RefreshScope`?** Feature flags, rate limits, third-party endpoint URLs, dynamic DB tuning — values that change occasionally and shouldn't trigger a redeploy.
