Filters intercept, modify, and enhance requests and responses passing through the API Gateway. They handle cross-cutting concerns (authentication, logging, rate limiting) at a centralized entry point before requests reach microservices.

> Assumes the gateway project and routes from [12.4 Spring Cloud API Gateway](<../12-microservices/12.4. Spring Cloud API Gateway.md>) are already set up. This file adds filters on top of that.

---

## How Filters Work

```
      ┌────────────────────┐
      │   Gateway Client   │
      └────────┬─────▲─────┘
               │     │
      ┌────────▼─────┤─────┐
      │  Gateway Handler   │
      └────────┬─────▲─────┘
               │     │
      ┌────────▼─────┤─────┐
      │ Filter (order = -1)│
      ├────────────────────┤
      │ Filter (order = 0) │
      ├────────────────────┤
      │ Filter (order = n) │
      └────────┬─────▲─────┘
               │     │
      ┌────────▼─────┤─────┐
      │ Proxied Service    │
      │ (Order/Inventory)  │
      └────────────────────┘
```

Request flows **down** through filters (pre). Response flows **up** through filters in reverse (post).

---

## Two Types of Filters

- **Global filter**: applies to every route. Implement the `GlobalFilter` interface.
- **Route-specific filter**: applies only to routes that opt in via YAML. Extend `AbstractGatewayFilterFactory`.

---

## Global Filter

Applied to **every route** in the gateway. Good for logging, metrics, common headers.

```java
@Component
@Slf4j
public class GlobalLoggingFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
            GatewayFilterChain chain) {

        // Pre filter — runs before request reaches service
        log.info("Global Pre: {}", exchange.getRequest().getURI());

        return chain.filter(exchange)
            .then(Mono.fromRunnable(() -> {
                // Post filter — runs after response comes back
                log.info("Global Post: {}",
                    exchange.getResponse().getStatusCode());
            }));
    }

    @Override
    public int getOrder() {
        return 5; // lower number = runs first in pre, last in post
    }
}
```

|Part|What It Does|
|---|---|
|`GlobalFilter`|Makes this a global filter — applies to all routes|
|`Ordered`|Defines where in the filter chain this runs|
|`exchange`|Contains the request and response|
|`chain`|The filter chain — call `chain.filter(exchange)` to continue to next filter|
|`Mono<Void>`|Reactive/non-blocking return type — filter handles concurrent requests|
|`chain.filter(exchange)`|Pass to next filter (pre part done)|
|`.then(Mono.fromRunnable(...))`|Post filter — runs after response comes back|

```
Output:
  Global Pre: http://localhost:8080/api/v1/orders/core/helloOrders
  Global Post: 200 OK
```

## Route-Specific Filter (Gateway Filter)

Applied to **specific routes only**. Good for route-level authentication, transformations, validations.

```java
@Component
@Slf4j
public class LoggingOrdersFilter extends
        AbstractGatewayFilterFactory<LoggingOrdersFilter.Config> {

    public LoggingOrdersFilter() {
        super(Config.class);
    }

    @Override
    public GatewayFilter apply(Config config) {
        return (exchange, chain) -> {
            // Pre filter
            log.info("Order Filter Pre: {}",
                exchange.getRequest().getURI());

            return chain.filter(exchange);
        };
    }

    public static class Config {
        // Add config properties here if needed
    }
}
```

|Part|What It Does|
|---|---|
|`AbstractGatewayFilterFactory`|Base class for route-specific filters|
|`Config`|Inner class for filter configuration — pass custom properties if needed|
|`apply(Config config)`|Returns the actual filter logic|
|`(exchange, chain) -> { }`|Lambda implementing the `GatewayFilter` interface|

### Register in Route Config

Add the filter name to the existing route configuration from [12.4 Spring Cloud API Gateway](<../12-microservices/12.4. Spring Cloud API Gateway.md>):

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://ORDER-SERVICE
          predicates:
            - Path=/api/v1/orders/**
          filters:
            - StripPrefix=2
            - name: LoggingOrdersFilter    # must match class name
```

> The filter name in YAML must match the class name exactly. Only this route gets the filter — inventory route is unaffected.

## Global vs Route-Specific

```
Request to /api/v1/orders/**:
  → Global Filter (all routes)
  → LoggingOrdersFilter (order route only)
  → Order Service

Request to /api/v1/inventory/**:
  → Global Filter (all routes)
  → Inventory Service
  (no LoggingOrdersFilter — it's not on this route)
```

## What You Can Do in Filters

From the `exchange` object:

```java
// Request info
exchange.getRequest().getURI();
exchange.getRequest().getPath();
exchange.getRequest().getHeaders();
exchange.getRequest().getQueryParams();
exchange.getRequest().getMethod();

// Modify request (add headers, change path)
exchange.getRequest().mutate()
    .header("X-Custom-Header", "value")
    .build();

// Response info (in post filter)
exchange.getResponse().getStatusCode();
exchange.getResponse().getHeaders();
```

## Q&A

1. **What is a Gateway Filter and why use one?** A filter intercepts requests/responses at the gateway to handle cross-cutting concerns (auth, logging, rate limiting) in one place instead of duplicating them across every microservice.
2. **Difference between `GlobalFilter` and `GatewayFilterFactory`?** `GlobalFilter` applies to every route automatically; `AbstractGatewayFilterFactory` creates a named filter you opt into per route via YAML — useful when behavior should vary by route.
3. **What do pre and post filters mean?** "Pre" runs before the request reaches the downstream service (on the way down the chain); "post" runs after the response returns (on the way up). In Spring Cloud Gateway, post logic is attached via `chain.filter(exchange).then(...)`.
4. **How is filter ordering controlled?** Implement `Ordered` and return an `int` from `getOrder()`. Lower numbers run first in the pre phase and last in the post phase.
5. **Why is the return type `Mono<Void>`?** Spring Cloud Gateway is built on Spring WebFlux (reactive, non-blocking) — filters return reactive types so the gateway can handle many concurrent requests without blocking threads.

## Quick Reference

|Concept|Remember|
|---|---|
|Global filter|`implements GlobalFilter, Ordered` — auto-applied to all routes|
|Route-specific filter|`extends AbstractGatewayFilterFactory<Config>` — opt-in per route|
|Pre vs post|pre = before downstream; post = `chain.filter(...).then(...)`|
|Order|lower first (pre), last (post)|
|YAML name|must match class name; strip `GatewayFilterFactory` suffix|
