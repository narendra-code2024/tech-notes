## The Problem

Each microservice has its own `application.yml` — to change a config, you must go into each service, modify the file, and restart. With 10+ services, this is slow and error-prone.

```
Without Config Server:
  Order Service     → application.yml (local)
  Inventory Service → application.yml (local)
  API Gateway       → application.yml (local)
  Change DB URL? → edit each file → restart each service

With Config Server:
  All services → pull config from → Config Server → pulls from → GitHub
  Change DB URL? → edit one file in GitHub
```

---

## How It Works

A **Spring Cloud Config Server** is a dedicated service acting as a centralized repository, allowing all microservices to retrieve their configurations dynamically at runtime. It manages different environments — `dev`, `test`, `prod` — via **Spring profiles**, so each environment can have its own DB URLs, API keys, etc.

```
┌──────────────────┐        ┌───────────────────────┐        ┌─────────────┐
│  Order Service   │        │  Spring Cloud         │        │   GitHub    │
│  (Config Client) │◄───────│  Config Server        │◄───────│   Repo      │
├──────────────────┤        │  port: 8888           │        │             │
│  Inventory Svc   │◄───────│                       │        │ application │
│  (Config Client) │        │  Pulls configs from   │        │ .yml        │
├──────────────────┤        │  GitHub on startup    │        │ order-svc   │
│  API Gateway     │◄───────│                       │        │ .yml        │
│  (Config Client) │        │                       │        │ inv-svc     │
└──────────────────┘        └───────────────────────┘        │ .yml        │
                                                             └─────────────┘
```

1. **Config Server starts** → clones the GitHub repo
2. **Microservices start** → ask Config Server for their config (by service name + profile)
3. **Config Server returns** the matching properties
4. **Config changes in GitHub** → picked up on next restart

## GitHub Repo Structure

```
ecommerce-config-repo/
  ├── application.yml                 ← global/shared config (all services inherit)
  ├── order-service.yml               ← order service, default profile
  ├── order-service-dev.yml           ← order service, dev profile
  ├── inventory-service.yml           ← inventory service, default profile
  └── api-gateway.yml                 ← API gateway config
```

> **Important:** File names must exactly match `spring.application.name` of each service.  
> The global `application.yml` is inherited by **all** services — put shared config here (e.g., Eureka URL).

### Config Loading Order (Profile Hierarchy)

```
1. application.yml                    (global — shared by all services)
         │
         ▼
2. order-service.yml                  (service default profile)
         │
         ▼
3. order-service-dev.yml              (dev profile — overrides defaults)
```

## Setting Up the Config Server

### Step 1 — Create Project

From [start.spring.io](https://start.spring.io/):

- **Artifact:** `config-server`
- **Dependencies:** `Config Server`, `Eureka Discovery Client`

### Step 2 — Enable Config Server

```java
@SpringBootApplication
@EnableConfigServer
public class ConfigServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(ConfigServerApplication.class, args);
    }
}
```

### Step 3 — Configure `application.yml`

```yaml
spring:
  application:
    name: config-server
  cloud:
    config:
      server:
        git:
          uri: https://github.com/your-username/ecommerce-config-repo
          username: your-github-username
          password: your-github-personal-access-token
          default-label: master                         # or main
          clone-on-start: true

server:
  port: 8888

eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka
```

> **Private Repo?** Generate a **Personal Access Token** in GitHub:  
> `Settings → Developer Settings → Personal Access Tokens → Fine-grained tokens`  
> Grant **Contents: Read** and **Administration: Read** permissions for the config repo. Copy the token immediately — it won't be shown again.

### Step 4 — Verify Config Server

```
GET http://localhost:8888/order-service/default
GET http://localhost:8888/order-service/dev
GET http://localhost:8888/inventory-service/default
```

## Setting Up Config Clients

Add to **every microservice** that should pull config from the server.

### Step 1 — Add Dependency

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-config</artifactId>
</dependency>
```

### Step 2 — Minimal `application.yml`

```yaml
spring:
  application:
    name: order-service
  config:
    import: configserver:http://localhost:8888
```

> The local `application.yml` only needs the **service name** and **config server location**. Everything else comes from the GitHub repo.

## Shared / Global Config — `application.yml`

Properties common across all services go in the root `application.yml` in the GitHub repo:

```yaml
# inherited by ALL services
eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka
```

> If a specific service needs a different value, it can override in its own file.

## Profile-Specific Config

```
order-service.yml                  ← default profile
order-service-dev.yml              ← dev profile
order-service-prod.yml             ← prod profile
```

```yaml
# order-service.yml (default)
my:
  variable: order-github-default
```

```yaml
# order-service-dev.yml (dev) — overrides the default value
my:
  variable: order-github-dev
```

### Accessing in Code

```java
@Value("${my.variable}")
private String myVariable;

@GetMapping("/hello")
public String hello() {
    return "Hello from Order Service. my.variable = " + myVariable;
}
```

### Activating a Profile

```yaml
# application.yml (or via environment variable / JVM flag)
spring:
  profiles:
    active: dev
```

## Securing the Config Server

The config server hands out every service's secrets (DB passwords, API keys) — anyone who can reach it can read them. In production:

- Put the config server on an internal network only (not exposed via the API gateway)
- Add HTTP Basic auth on the server (`spring.security.user.name` / `password`) and have clients send credentials via `spring.cloud.config.username` / `password`
- Store sensitive values **encrypted** in the Git repo using `{cipher}` placeholders — the config server decrypts on read with a configured key
- Restrict the GitHub PAT to read-only on the single config repo
- Use a dedicated branch per environment (`dev`, `prod`) instead of a single `master` for stricter PR-gated promotion

## Q&A

1. **What problem does Spring Cloud Config Server solve?** It centralizes configuration for all microservices in one Git repo so a single edit propagates everywhere — no per-service `application.yml` edits and redeploys.
2. **How does a microservice locate its config?** By matching `spring.application.name` (and active profile) to file names in the Git repo. The client points at the server via `spring.config.import=configserver:http://...`.
3. **What's the role of `application.yml` in the config repo?** It holds global properties inherited by every service (e.g., shared Eureka URL). Service-specific files override it.
4. **What's the profile precedence?** `application.yml` (global) → `<service>.yml` (default) → `<service>-<profile>.yml` (active profile overrides defaults).
5. **What's the startup order with Eureka, Config Server, and microservices?** Eureka first, Config Server second (registers with Eureka, clones the repo), then microservices (fetch config + register).
6. **What's the limitation of Config Server alone?** Services don't see config changes until restart. `@RefreshScope` + Actuator's `/actuator/refresh` solves this — see [13.4 Refresh Configuration without Restart](<13.4. Refresh Configuration without Restart.md>).
