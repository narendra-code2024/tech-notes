## Why Authenticate at the Gateway?

Without gateway authentication, every microservice needs its own Spring Security setup — duplicated across all services.

```
Without gateway auth:
  Order Service     → Spring Security + JWT validation
  Inventory Service → Spring Security + JWT validation
  Payment Service   → Spring Security + JWT validation
  (same code repeated everywhere)

With gateway auth:
  API Gateway → validates JWT once
    │
    ├──→ Order Service     (trusts gateway, no auth code needed)
    ├──→ Inventory Service (trusts gateway, no auth code needed)
    └──→ Payment Service   (trusts gateway, no auth code needed)
```

> Authenticate once at the gateway. Pass the user info (userId, roles) as headers to downstream services. Services trust the gateway.

> Builds on the gateway and routes from [12.4 Spring Cloud API Gateway](<../12-microservices/12.4. Spring Cloud API Gateway.md>) and the filter patterns from [13.1 Introduction to API Gateway Filters](<13.1. Introduction to API Gateway Filters.md>).

## How It Works

```
Client sends: GET /api/v1/orders/core/helloOrders
              Authorization: Bearer eyJ...
       │
       ▼
API Gateway — Authentication Filter
       │
       ├── No Authorization header? → 401 Unauthorized (stop here)
       │
       ├── Invalid/expired token? → 401 Unauthorized (stop here)
       │
       └── Valid token?
             │
             ▼
           Extract userId from token
             │
             ▼
           Add X-User-Id header to request (mutate)
             │
             ▼
           Forward to Order Service
             │
             ▼
Order Service reads X-User-Id from header — knows who the user is
```

### JWT Service (in API Gateway)

```java
@Service
public class JwtService {

    @Value("${jwt.secretKey}")
    private String secretKey;

    public Long getUserIdFromToken(String token) {
        Claims claims = Jwts.parser()
            .verifyWith(getSigningKey())
            .build()
            .parseSignedClaims(token)
            .getPayload();

        return Long.valueOf(claims.getSubject());
    }

    private SecretKey getSigningKey() {
        return Keys.hmacShaKeyFor(secretKey.getBytes());
    }
}
```

```properties
# API Gateway application.properties
jwt.secretKey=your-256-bit-secret-key-must-match-user-service
```

> The secret key must be the **same** across the user service (which generates tokens) and the API Gateway (which validates them).

### Authentication Filter

Using the `GatewayFilterFactory` naming convention — Spring auto-detects the name without the suffix, so `AuthenticationGatewayFilterFactory` becomes just `Authentication` in config.

```java
@Component
@Slf4j
public class AuthenticationGatewayFilterFactory extends
        AbstractGatewayFilterFactory<AuthenticationGatewayFilterFactory.Config> {

    private final JwtService jwtService;

    public AuthenticationGatewayFilterFactory(JwtService jwtService) {
        super(Config.class);
        this.jwtService = jwtService;
    }

    @Override
    public GatewayFilter apply(Config config) {
        return (exchange, chain) -> {

            // Skip auth if disabled via config
            if (!config.isEnabled()) {
                return chain.filter(exchange);
            }

            // Get Authorization header
            String authHeader = exchange.getRequest().getHeaders()
                .getFirst("Authorization");

            // No header → 401
            if (authHeader == null || !authHeader.startsWith("Bearer ")) {
                exchange.getResponse()
                    .setStatusCode(HttpStatusCode.valueOf(401));
                return exchange.getResponse().setComplete();
            }

            // Extract token
            String token = authHeader.split("Bearer ")[1];

            try {
                // Validate and extract userId
                Long userId = jwtService.getUserIdFromToken(token);

                // Mutate request — add userId as header for downstream
                exchange.getRequest().mutate()
                    .header("X-User-Id", userId.toString())
                    .build();

                return chain.filter(exchange);

            } catch (Exception e) {
                log.error("Token validation failed: {}", e.getMessage());
                exchange.getResponse()
                    .setStatusCode(HttpStatusCode.valueOf(401));
                return exchange.getResponse().setComplete();
            }
        };
    }

    @Data
    public static class Config {
        private boolean enabled = true;
    }
}
```

## Route Configuration

Apply the filter to specific routes. Use `args.enabled` to enable/disable per route:

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://ORDER-SERVICE
          predicates:
            - Path=/api/v1/orders/**
          filters:
            - StripPrefix=2
            - name: Authentication
              args:
                enabled: true       # auth required

        - id: inventory-service
          uri: lb://INVENTORY-SERVICE
          predicates:
            - Path=/api/v1/inventory/**
          filters:
            - StripPrefix=2
            - name: Authentication
              args:
                enabled: false      # public route, no auth
```

> `enabled: false` skips authentication for that route. The `Config` class receives this value.

> **Important:** Don't prefix boolean fields with `is` inside the `Config` class (e.g., use `enabled` not `isEnabled`). Lombok's `@Data` generates `isEnabled()` getter for a field named `enabled`, but if the field itself is named `isEnabled`, the YAML property binding breaks.

## What About Login? Does the Client Call User Service Directly?

No — the client **always** goes through the API Gateway, including login. The login/register routes just have authentication disabled since the user doesn't have a token yet.

```
Client knows only one URL: http://api-gateway:8080

Login/Register:
  Client → API Gateway → User Service (auth disabled)
                            → validates credentials
                            → returns JWT token

All other calls:
  Client → API Gateway (auth filter validates JWT)
              → forwards to Order/Inventory/etc.
```

```yaml
routes:
  # Public — login, register (no auth)
  - id: user-service-auth
    uri: lb://USER-SERVICE
    predicates:
      - Path=/api/v1/auth/**
    filters:
      - StripPrefix=2
      - name: Authentication
        args:
          enabled: false

  # Protected — user profile, settings (auth required)
  - id: user-service
    uri: lb://USER-SERVICE
    predicates:
      - Path=/api/v1/users/**
    filters:
      - StripPrefix=2
      - name: Authentication
        args:
          enabled: true

  # Protected — orders (auth required)
  - id: order-service
    uri: lb://ORDER-SERVICE
    predicates:
      - Path=/api/v1/orders/**
    filters:
      - StripPrefix=2
      - name: Authentication
        args:
          enabled: true
```

> Same service can have **multiple routes** with different predicates and auth settings. The gateway routes by **path**, not by service.

```
/api/v1/auth/login      → User Service (no auth)
/api/v1/auth/register   → User Service (no auth)
/api/v1/users/profile   → User Service (auth required)
/api/v1/users/settings  → User Service (auth required)
/api/v1/orders/**       → Order Service (auth required)
```

> The client never needs to know individual service addresses. Everything goes through the gateway — it decides which routes are public and which need authentication.

## Reading User Info in Downstream Services

The gateway adds `X-User-Id` to the request headers. Downstream services read it:

```java
@GetMapping("/core/helloOrders")
public String helloOrders(
        @RequestHeader("X-User-Id") Long userId) {
    return "Hello from order service, userId: " + userId;
}
```

> Services don't validate the token themselves — they trust the gateway already did. They just read the userId from the header the gateway injected.

> **Trust boundary caveat:** This model assumes downstream services are only reachable via the gateway (network policies, internal-only ports). If a service is also exposed directly — or you need zero-trust between services — each service must still validate the JWT (or a signed gateway-issued header) itself. A common pattern is to also strip any client-supplied `X-User-Id` at the gateway so callers can't spoof identity by setting the header upstream of the auth filter.

## Authorization (Extending This)

Authentication verifies **who** the user is. Authorization checks **what** they can do. You can add a second filter:

```yaml
filters:
  - name: Authentication
    args:
      enabled: true
  - name: Authorization
    args:
      roles: ADMIN, SUPER_ADMIN
```

The authorization filter would:

1. Extract roles from the JWT token (stored as claims)
2. Compare with allowed roles from config
3. If role matches → pass through
4. If not → 403 Forbidden

## Mutating Requests and Responses

### Mutate Request (Pre Filter)

```java
// Add headers
exchange.getRequest().mutate()
    .header("X-User-Id", userId.toString())
    .header("X-User-Role", role)
    .build();

// Change path
exchange.getRequest().mutate()
    .path("/new-path")
    .build();
```

### Mutate Response (Post Filter)

```java
return chain.filter(exchange)
    .then(Mono.fromRunnable(() -> {
        exchange.getResponse().getHeaders()
            .add("X-Response-Time", Instant.now().toString());
    }));
```

## Q&A

1. **Why authenticate at the API Gateway instead of each microservice?** Centralizing JWT validation at the gateway avoids duplicating Spring Security setup in every service. Downstream services trust the gateway and just read `X-User-Id` from a header.
2. **How does the gateway pass identity to downstream services?** The filter mutates the request and injects headers like `X-User-Id` (and optionally roles) before forwarding. Downstream controllers read them via `@RequestHeader`.
3. **How can the same filter be enabled per-route?** Via `AbstractGatewayFilterFactory<Config>` with a `Config` class exposing flags (e.g., `enabled`). Routes declare `args: { enabled: true/false }` in YAML — the login/register routes can opt out while protected routes opt in.
4. **Authentication vs authorization at the gateway?** Authentication proves **who** the user is (valid token, extract userId). Authorization checks **what** they can do (role/permission claims vs the route's required roles) — usually a second filter returning 403 on mismatch.
5. **Note on Lombok and the `Config` class boolean field?** Name the field `enabled`, not `isEnabled`. Lombok's `@Data` generates `isEnabled()` for the `enabled` field; naming the field `isEnabled` breaks Spring's property binding from YAML `args`.
