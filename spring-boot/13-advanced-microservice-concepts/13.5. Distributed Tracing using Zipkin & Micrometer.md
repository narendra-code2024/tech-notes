## The Problem

In a microservice system, a single request flows through multiple services. When something goes wrong or is slow, you have to check logs of each service individually to find the issue — this is impractical with 10+ services.

```
User Request → API Gateway → Order Service → Inventory Service
                                           → Payment Service
```

Distributed tracing lets you visualize this entire flow, see how long each service took, and pinpoint bottlenecks.

---

## Key Concepts

- **Trace**: one complete request journey from start to finish across all microservices. Identified by a unique **Trace ID** shared across every service involved.
- **Span**: a single unit of work within a trace (one service call). Each span has its own **Span ID** but shares the parent's Trace ID.

```
Trace (one request, one Trace ID)
  │
  ├── Span A  → API Gateway          (800ms)
  │     │
  │     ├── Span B  → Order Service  (709ms)
  │     │     │
  │     │     ├── Span C → Inventory Service  (some ms)
  │     │     └── Span D → Payment Service    (some ms)
  │     │
  │     └── Span E  → another call   (some ms)
```

> Same Trace ID flows through all spans. Each new request gets a new Trace ID.

---

## Tools

- **Micrometer**: vendor-neutral application metrics façade that instruments your Spring Boot app. Records start time, end time, Trace ID, and Span ID for every request. Can feed multiple backends: Zipkin (tracing), Prometheus (APM), Elastic (dashboarding), CloudWatch (alerting). Replaces the older **Spring Cloud Sleuth**.
- **Zipkin**: dedicated distributed tracing UI that receives data from Micrometer and visualizes it: full request tree, time per service, dependency graph. Runs standalone on port `9411` ([install guide](https://zipkin.io/pages/quickstart)).

> **Alternatives to Zipkin:** **Jaeger** (CNCF, similar feature set, native OpenTelemetry support) and **Grafana Tempo** (cheap object-storage backend, integrates with Loki + Prometheus for trace/log/metric correlation). Micrometer Tracing supports any of them — swap the bridge dependency (`micrometer-tracing-bridge-otel` for OpenTelemetry exporters).

> **Header propagation:** Brave defaults to the **B3** headers (`X-B3-TraceId`, `X-B3-SpanId`). The OpenTelemetry bridge defaults to W3C **`traceparent`** (the modern interop standard). Pick one consistently across all services or configure both — mixed propagation breaks the trace.

## Setup

### Step 1 — Run Zipkin Server

```bash
# Download and run the JAR
java -jar zipkin-server.jar

# Or via Docker
docker run -d -p 9411:9411 openzipkin/zipkin
```

Access dashboard at: `http://localhost:9411`

### Step 2 — Add Dependencies to Each Microservice

```xml
<!-- Micrometer observation -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-observation</artifactId>
</dependency>

<!-- Micrometer tracing bridge (Brave) — sends data to Zipkin -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-brave</artifactId>
</dependency>

<!-- Zipkin reporter -->
<dependency>
    <groupId>io.zipkin.reporter2</groupId>
    <artifactId>zipkin-reporter-brave</artifactId>
</dependency>

<!-- OpenFeign + Micrometer integration (for tracing Feign calls) -->
<dependency>
    <groupId>io.github.openfeign</groupId>
    <artifactId>feign-micrometer</artifactId>
</dependency>

<!-- Actuator (if not already added) -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

### Step 3 — Configure in Global `application.yml` (GitHub config repo)

```yaml
management:
  tracing:
    sampling:
      probability: 1.0      # trace 100% of requests (dev only, use 0.1 in prod)

spring:
  zipkin:
    base-url: http://localhost:9411
```

> Default sampling probability is `0.1` (10%). In production, keep it low to save resources.

### Step 4 — Register `ObservationRegistry` Bean

Add this to `AppConfig` (or any `@Configuration` class) in each microservice:

```java
@Configuration
public class AppConfig {

    @Bean
    ObservationRegistryCustomizer<ObservationRegistry> skipActuatorEndpoints() {
        return registry -> registry.observationConfig()
            .observationPredicate(ObservationPredicate.ALWAYS);
    }
}
```

## How It Works at Runtime

```
API call made
      │
      ▼
Micrometer assigns Trace ID + Span ID to the request
      │
      ▼
Trace ID is passed along to every downstream service call
      │
      ▼
Each service records its own Span (start time, end time, Span ID)
      │
      ▼
All spans are sent to Zipkin server (port 9411)
      │
      ▼
Zipkin UI shows the full trace tree with timings
```

## Zipkin Dashboard

### Traces View

Shows all recorded traces. Each row is one request. Click **Show** to expand the full span tree:

```
API Gateway        [800ms] ──────────────────────────────
  Order Service    [709ms] ─────────────────────────
    Inventory Svc  [some ms] ──────────
```

- Clicking a span shows its individual timing and metadata

### Dependencies View

Visual graph of how services call each other:

```
API Gateway → Order Service → Inventory Service
```

> With hundreds of microservices this becomes a complex graph. You can filter by service name to focus on one service and its connections.

### Filtering & Sorting

- Filter by **service name**, **time range**, **duration**: Sort by **start time** or **duration** to find slowest requests
- Eureka heartbeats also appear as traces (every 30s) — these can be ignored

|---|---|---|

|Shows|Which services were called & how long|What happened inside each service|
|Tool|Zipkin + Micrometer|ELK Stack|

## Q&A

1. **What is distributed tracing and why do you need it?** It tracks a single request across multiple microservices using a shared Trace ID, so you can visualize the call graph and timings instead of grepping each service's logs separately.
2. **Trace vs Span?** A **Trace** is the whole request journey (one Trace ID). A **Span** is one unit of work within it (e.g., one service call), with its own Span ID but the same Trace ID.
3. **What replaced Spring Cloud Sleuth?** Micrometer (specifically `micrometer-tracing` with the Brave bridge for Zipkin). Sleuth is no longer maintained for current Spring Boot versions.
4. **What does sampling probability control?** The fraction of requests Micrometer records. `1.0` traces everything (dev), `0.1` (10%) is typical for production to limit overhead and storage.
5. **How does the Trace ID propagate to downstream services?** Micrometer adds it to outgoing HTTP headers (e.g., `b3` / `traceparent`). With `feign-micrometer`, OpenFeign clients carry it automatically; Spring MVC/WebFlux interceptors do the same for incoming calls.
6. **Tracing vs centralized logging — when do you reach for each?** Tracing (Zipkin) shows **where time is spent**. Logging (ELK) shows **what happened inside each service**. They share the Trace ID, so you pivot between them when debugging.
