## Why Centralize Exception Handling?

Without a global handler, every controller has to translate its own exceptions into HTTP responses ŌĆö leading to duplicated try/catch blocks, inconsistent error formats, and forgotten edge cases. Centralized handling:

- Returns meaningful HTTP status codes consistently
- Produces a uniform error body across all endpoints
- Removes try/catch noise from controllers
- Gives a single place to evolve error format

---

## `@ResponseStatus` (Simple Cases)

The simplest way to map an exception to an HTTP status ŌĆö annotate the exception class itself. **No handler code, no custom body**: just the right status.

```java
@ResponseStatus(HttpStatus.NOT_FOUND)
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String message) {
        super(message);
    }
}
```

When this exception propagates uncaught, the body shape depends on a Spring Boot configuration toggle:

**Default ŌĆö legacy `DefaultErrorAttributes` format:**

```json
{
    "timestamp": "2026-02-09T15:30:00Z",
    "status": 404,
    "error": "Not Found",
    "path": "/api/products/123"
}
```

**With `ProblemDetail` enabled** (recommended for new projects on Spring 6+):

```yaml
# application.yml
spring:
  mvc:
    problemdetails:
      enabled: true
```

```json
{
    "type": "about:blank",
    "title": "Not Found",
    "status": 404,
    "detail": null,
    "instance": "/api/products/123"
}
```

Either way, you have **no control over body content** with just `@ResponseStatus` ŌĆö the framework decides. Useful for trivial cases or quick prototypes; production codebases mostly use `@ExceptionHandler` for full body control.

> **Precedence:** if a matching `@ExceptionHandler` exists in your `@RestControllerAdvice`, that wins ŌĆö `@ResponseStatus` is the fallback. In production codebases with a global advice, `@ResponseStatus` is mostly redundant; teams keep it as a defensive default in case a handler is forgotten for a new exception type.

> Enable `spring.mvc.problemdetails.enabled=true` for new projects so uncaught exceptions also produce RFC 7807 bodies, aligning with explicit `ProblemDetail` returns from your handlers (next section).

## Modern Default ŌĆö `ProblemDetail` (RFC 7807)

Spring 6+ ships `ProblemDetail`, an implementation of [RFC 7807](https://datatracker.ietf.org/doc/html/rfc7807) ŌĆö a standardized error response format. **For new projects, this is the idiomatic Spring choice.** No custom error class needed.

Standard fields (all defined by the RFC):

| Field | Purpose |
|---|---|
| `type` | URI identifying the error type (defaults to `about:blank`) |
| `title` | Short human-readable summary |
| `status` | HTTP status code |
| `detail` | Specific explanation of the error |
| `instance` | URI identifying the specific occurrence |

You can also attach **custom properties** via `setProperty(...)` for fields outside the spec.

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ProblemDetail handleNotFound(ResourceNotFoundException ex) {
        return ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
    }

    @ExceptionHandler(OutOfStockException.class)
    public ProblemDetail handleOutOfStock(OutOfStockException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.CONFLICT, ex.getMessage());
        pd.setProperty("productId", ex.getProductId());
        pd.setProperty("requestedQty", ex.getRequestedQty());
        pd.setProperty("availableQty", ex.getAvailableQty());
        return pd;
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ProblemDetail handleValidation(MethodArgumentNotValidException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.BAD_REQUEST, "Validation failed");
        Map<String, String> fieldErrors = ex.getBindingResult().getFieldErrors().stream()
            .collect(Collectors.toMap(FieldError::getField, FieldError::getDefaultMessage));
        pd.setProperty("errors", fieldErrors);
        return pd;
    }

    @ExceptionHandler(Exception.class)
    public ProblemDetail handleAll(Exception ex) {
        log.error("Unhandled exception", ex);
        return ProblemDetail.forStatusAndDetail(HttpStatus.INTERNAL_SERVER_ERROR, "Something went wrong");
    }
}
```

> `FieldError` is `org.springframework.validation.FieldError` ŌĆö Spring's built-in class for field-level validation errors. Useful methods: `getField()`, `getDefaultMessage()`, `getRejectedValue()`, `getCode()`. Returned by `BindingResult.getFieldErrors()` whenever a `@Valid` payload fails.

Resulting JSON for `OutOfStockException`:

```json
{
    "type": "about:blank",
    "title": "Conflict",
    "status": 409,
    "detail": "Product 123 is out of stock",
    "productId": 123,
    "requestedQty": 5,
    "availableQty": 0
}
```

**When to pick `ProblemDetail`**: Greenfield project with no specific envelope requirements
- Public API where RFC compliance matters
- You want zero maintenance overhead on the error shape

## Custom `ApiError` (When You Need a Specific Shape)

If you need fields outside RFC 7807 ŌĆö typically a **trace/correlation ID** for support log lookup, or a specific shape your team has standardized ŌĆö define a custom `ApiError` class. This is also the path if you're aligning with an existing team convention from another stack.

```java
public class ApiError {
    private String id;                             // trace/correlation ID
    private String message;
    private LocalDateTime timestamp;
    private Map<String, String> errors;            // field-level validation errors

    public ApiError(String message) {
        this.id = UUID.randomUUID().toString();
        this.message = message;
        this.timestamp = LocalDateTime.now();
    }

    // getters, setters
}
```

Resulting JSON shapes:

```json
// Generic error
{
    "id": "5f8a3c9b-2e1f-4a8c-9b1a-d4e7f8a9b1c2",
    "message": "Product not found: 123",
    "timestamp": "2026-02-09T15:30:00Z"
}
```

```json
// Validation error
{
    "id": "5f8a3c9b-2e1f-4a8c-9b1a-d4e7f8a9b1c2",
    "message": "Validation failed",
    "timestamp": "2026-02-09T15:30:00Z",
    "errors": {
        "email": "must be a valid email",
        "firstName": "must not be blank"
    }
}
```

The `id` field gives clients something to quote in support tickets ŌĆö server logs are correlated by the same UUID, so engineers can find the exact failure instantly.

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ApiError> handleNotFound(ResourceNotFoundException ex) {
        ApiError error = new ApiError(ex.getMessage());
        log.warn("Resource not found [id={}]: {}", error.getId(), ex.getMessage());
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiError> handleValidation(MethodArgumentNotValidException ex) {
        ApiError error = new ApiError("Validation failed");
        Map<String, String> fieldErrors = ex.getBindingResult().getFieldErrors().stream()
            .collect(Collectors.toMap(FieldError::getField, FieldError::getDefaultMessage));
        error.setErrors(fieldErrors);
        return ResponseEntity.badRequest().body(error);
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiError> handleAll(Exception ex) {
        ApiError error = new ApiError("Something went wrong");
        log.error("Unhandled exception [id={}]", error.getId(), ex);
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body(error);
    }
}
```

> **Why log with the `id`?** Whatever shows up in the client's response also shows up in your logs ŌĆö one grep finds the exact failure path. This is the pattern most prod teams use, regardless of `ProblemDetail` vs `ApiError`.

## Real-World Usage

**Don't return `ex.getMessage()` in 500s**: internal exception messages leak stack frames, internal hostnames (`db-prod-01.internal`), file paths, or sensitive query details. Always return a generic message; log the real exception server-side.

**Always log unhandled exceptions in the catch-all**: without it the stack trace is lost forever. Use `log.error("...", ex)` to capture the full trace. SLF4J/Logback setup, log levels, and structured logging are covered in [4.4 Logging](<../04-production-ready-features/4.4. Logging.md>).

**Domain exception hierarchy**: production apps usually have a base `BusinessException` with domain-specific subclasses ŌĆö for an e-commerce app: `OutOfStockException`, `PaymentFailedException`. The advice maps each type to the right HTTP status (409, 422, etc.) and clients get a precise error without leaking internals. Use a single generic `ResourceNotFoundException("Product", id)` for not-found cases.

**Exception precedence**: when multiple `@ExceptionHandler`s could match, Spring picks the **most specific** match by exception type hierarchy. Handler for `ResourceNotFoundException` wins over `RuntimeException`, which wins over `Exception` (catch-all).

## Q&A

1. **`ProblemDetail` vs custom `ApiError` ŌĆö when to pick which?**
   `ProblemDetail` (RFC 7807) is the modern Spring default ŌĆö zero custom code, standardized fields, full tooling support. Use a custom `ApiError` when you need fields the RFC doesn't standardize ŌĆö most commonly a **trace/correlation ID** for support log lookup, or to match an existing team-wide error shape.

2. **Difference between `@ControllerAdvice` and `@RestControllerAdvice`?**
   `@RestControllerAdvice` = `@ControllerAdvice` + `@ResponseBody`. The REST variant serializes return values to JSON automatically ŌĆö the standard choice for REST APIs.

3. **`@ResponseStatus` vs `@ExceptionHandler` ŌĆö when to use each?**
   `@ResponseStatus` on the exception class is for **simple status mapping** with no body customization. `@ExceptionHandler` lets you build a **custom response body**, log, or branch on the exception. Real APIs use the advice approach for body uniformity; `@ResponseStatus` only for trivial cases.

4. **What happens if multiple `@ExceptionHandler`s could match?**
   Spring picks the **most specific** match by exception type hierarchy. `ResourceNotFoundException` wins over `RuntimeException`, which wins over `Exception`.

5. **How do you handle validation errors globally?**
   `@ExceptionHandler(MethodArgumentNotValidException.class)` in the advice, walk `ex.getBindingResult().getFieldErrors()`, attach a field-message map to the response (`setProperty("errors", ...)` for `ProblemDetail`, or a typed field for custom `ApiError`), and return `400 Bad Request`. See [2.6 Request Validation](<2.6. Request Validation.md>).

6. **Why include a trace ID (`id`) in error responses?**
   Same UUID appears in the client's response and the server logs ŌĆö clients quote it in support tickets, engineers grep logs and find the exact failure instantly. Cheapest observability win in any prod API.

7. **Why not return `ex.getMessage()` in a 500 response?**
   Security. Internal messages can leak stack frames, internal hostnames, file paths, or sensitive query data. Return a generic message; log the real exception server-side.

## Quick Reference

| Concept | Remember |
|---|---|
| `@RestControllerAdvice` | Global handler; replies serialized to JSON |
| `@ExceptionHandler(X.class)` | Maps exception type to handler method |
| `@ResponseStatus` on exception | Simple status mapping; no body customization |
| `ProblemDetail` | RFC 7807 ŌĆö modern default for new Spring 6+ projects |
| `ProblemDetail.setProperty(...)` | Attach fields outside the RFC spec |
| Custom `ApiError` | When you need extra fields (e.g., `id` trace) or a fixed team shape |
| `id` (UUID) on errors | Correlation between client response and server logs ŌĆö quote in support tickets |
| Catch-all 500 message | Generic ŌĆö never `ex.getMessage()`; log the real exception |
| Handler precedence | Most specific exception type wins |
