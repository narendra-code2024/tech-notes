Centralized exception handling translates exceptions into HTTP responses in one place, instead of scattering try/catch across controllers. It gives consistent status codes, a uniform error body, no try/catch noise in controllers, and a single place to evolve the error format.

---

## `@ResponseStatus` (Simple Cases)

The simplest mapping: annotate the exception class with its HTTP status. No handler code, no custom body.

```java
@ResponseStatus(HttpStatus.NOT_FOUND)
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String message) {
        super(message);
    }
}
```

When the exception propagates uncaught, the body shape depends on a toggle. By default Spring Boot emits the `DefaultErrorAttributes` shape (`timestamp`, `status`, `error`, `path`); with `spring.mvc.problemdetails.enabled=true` it emits a `ProblemDetail` body instead. Either way you get no control over the body content; `@ResponseStatus` suits trivial cases and prototypes.

> **Note**: a matching `@ExceptionHandler` in a `@RestControllerAdvice` always wins over `@ResponseStatus`. In codebases with a global advice, `@ResponseStatus` is mostly a defensive default for exception types nobody wrote a handler for.

---

## Modern Default: `ProblemDetail` (RFC 9457)

Spring 6+ ships `ProblemDetail`, an implementation of [RFC 9457](https://datatracker.ietf.org/doc/html/rfc9457) (which obsoletes the older RFC 7807), a standardized error format requiring no custom error class. For new projects this is the idiomatic choice.

| Field      | Purpose                                          |
| ---------- | ------------------------------------------------ |
| `type`     | URI identifying the error type (default `about:blank`) |
| `title`    | Short human-readable summary                      |
| `status`   | HTTP status code                                  |
| `detail`   | Specific explanation of this error                |
| `instance` | URI identifying the specific occurrence           |

Attach fields outside the spec with `setProperty(...)`:

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ProblemDetail handleNotFound(ResourceNotFoundException ex) {
        return ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
    }

    @ExceptionHandler(OutOfStockException.class)
    public ProblemDetail handleOutOfStock(OutOfStockException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.CONFLICT, ex.getMessage());
        pd.setProperty("productId", ex.getProductId());      // custom property
        pd.setProperty("availableQty", ex.getAvailableQty());
        return pd;
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ProblemDetail handleValidation(MethodArgumentNotValidException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.BAD_REQUEST, "Validation failed");
        Map<String, String> fieldErrors = ex.getBindingResult().getFieldErrors().stream()
            .collect(Collectors.toMap(FieldError::getField, FieldError::getDefaultMessage));
        pd.setProperty("errors", fieldErrors);
        return pd;
    }

    @ExceptionHandler(Exception.class)
    public ProblemDetail handleAll(Exception ex) {
        log.error("Unhandled exception", ex);
        return ProblemDetail.forStatusAndDetail(HttpStatus.INTERNAL_SERVER_ERROR, "Something went wrong");
    }
}
```

```json
{
    "type": "about:blank",
    "title": "Conflict",
    "status": 409,
    "detail": "Product 123 is out of stock",
    "productId": 123,
    "availableQty": 0
}
```

> **Note**: `FieldError` is `org.springframework.validation.FieldError`, returned by `BindingResult.getFieldErrors()` for each failed `@Valid` field; useful methods are `getField()`, `getDefaultMessage()`, and `getRejectedValue()`.

Use `ProblemDetail` for greenfield projects, public APIs where RFC compliance matters, and whenever you want zero maintenance on the error shape.

## Custom `ApiError` (Specific Shape)

When you need fields the RFC does not standardize, typically a **trace/correlation ID** for support log lookup, or a team-standard shape, define a custom `ApiError`. The same UUID appears in the response and the logs, so one grep finds the exact failure.

```java
public class ApiError {
    private String id;                  // trace/correlation ID
    private String message;
    private LocalDateTime timestamp;
    private Map<String, String> errors; // field-level validation errors
    public ApiError(String message) {
        this.id = UUID.randomUUID().toString();
        this.message = message;
        this.timestamp = LocalDateTime.now();
    }
    // getters, setters
}
```

The advice mirrors the `ProblemDetail` one but returns `ResponseEntity<ApiError>`, logging the `id` so client and server correlate:

```java
@ExceptionHandler(ResourceNotFoundException.class)
public ResponseEntity<ApiError> handleNotFound(ResourceNotFoundException ex) {
    ApiError error = new ApiError(ex.getMessage());
    log.warn("Resource not found [id={}]: {}", error.getId(), ex.getMessage());
    return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
}
```

```json
{
    "id": "5f8a3c9b-2e1f-4a8c-9b1a-d4e7f8a9b1c2",
    "message": "Validation failed",
    "timestamp": "2026-02-09T15:30:00Z",
    "errors": { "email": "must be a valid email", "firstName": "must not be blank" }
}
```

## Production Practices

- **Never return `ex.getMessage()` in a 500**: internal messages leak stack frames, internal hostnames, file paths, or query details. Return a generic message; log the real exception server-side.
- **Always log in the catch-all**: use `log.error("...", ex)` to capture the full trace, or it is lost. SLF4J/Logback setup is in [4.4. Logging](<../04-production-ready-features/4.4. Logging.md>).
- **Domain exception hierarchy**: a base `BusinessException` with subclasses (`OutOfStockException`, `PaymentFailedException`) lets the advice map each type to a precise status (`409`, `422`) without leaking internals.
- **Handler precedence**: when several `@ExceptionHandler`s match, Spring picks the most specific by type, so `ResourceNotFoundException` beats `RuntimeException` beats `Exception`.

## Q&A

1. **`ProblemDetail` vs custom `ApiError`, when to pick which?**
`ProblemDetail` (RFC 9457) is the modern default: zero custom code, standardized fields, tooling support. Use a custom `ApiError` when you need fields the RFC does not standardize, most often a trace/correlation ID, or to match a team-wide shape.

2. **`@ControllerAdvice` vs `@RestControllerAdvice`?**
`@RestControllerAdvice` is `@ControllerAdvice` + `@ResponseBody`; it serializes handler return values to JSON, the standard choice for REST APIs.

3. **`@ResponseStatus` vs `@ExceptionHandler`?**
`@ResponseStatus` on the exception class is a simple status mapping with no body control. `@ExceptionHandler` builds a custom body, logs, and branches on the exception; real APIs use the advice for body uniformity.

4. **What happens if multiple `@ExceptionHandler`s match?**
Spring picks the most specific by exception type hierarchy: `ResourceNotFoundException` over `RuntimeException` over `Exception`.

5. **Why include a trace `id` in error responses?**
The same UUID appears in the client response and the server logs, so clients quote it in tickets and engineers grep straight to the failure. The cheapest observability win in a prod API.

6. **Why not return `ex.getMessage()` in a 500?**
Security: internal messages can leak stack frames, hostnames, file paths, or query data. Return a generic message and log the real exception server-side.
