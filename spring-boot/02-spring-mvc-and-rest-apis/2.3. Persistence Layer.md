The persistence layer maps Java objects to database tables and performs CRUD over JDBC underneath. In Spring Boot it is provided by **Spring Data JPA** (`spring-boot-starter-data-jpa`), which bundles the JPA API, **Hibernate** as the default provider, and the Spring Data repository abstractions.

> **Reference**: full coverage of the JPA stack, entity annotations, repository hierarchy, derived queries, and `@Query` lives in [3.1. Hibernate and JPA](<../03-hibernate-and-spring-data-jpa/3.1. Hibernate and JPA.md>). This note is a thin orientation for what controllers and services touch.

---

## Entity and Repository

An `@Entity` maps a class to a table, with `@Id` as the primary key; declaring a `JpaRepository` interface is enough for Spring Data to generate the implementation at runtime, giving full CRUD plus pagination, sorting, and batch ops:

```java
@Entity
@Table(name = "employees")
public class Employee {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private String department;
    private Double salary;
    // getters, setters, constructors
}

public interface EmployeeRepository extends JpaRepository<Employee, Long> { }
```

`findById` returns `Optional<T>` to force explicit not-found handling at compile time, avoiding the `null` to `NullPointerException` trap of the older `findOne(id)`:

```java
Employee emp = repository.findById(id)
    .orElseThrow(() -> new ResourceNotFoundException("Employee not found: " + id));
```

## Lombok

Lombok generates getters, setters, constructors, `equals`/`hashCode`, and builders at compile time via annotations, keeping entities and DTOs clean.

| Annotation          | Generates                                                              |
| ------------------- | --------------------------------------------------------------------- |
| `@Getter` `@Setter` | Accessor methods                                                      |
| `@NoArgsConstructor`| No-argument constructor                                               |
| `@AllArgsConstructor`| Constructor with all fields                                         |
| `@Data`             | `@Getter` + `@Setter` + `@ToString` + `@EqualsAndHashCode` + `@RequiredArgsConstructor` |
| `@Builder`          | Fluent builder                                                       |

> **Note**: avoid `@Data` on JPA entities. Its `@EqualsAndHashCode` spans all fields, which loads lazy associations during equality checks and behaves unpredictably for generated IDs (the hash differs before vs after `save()`). Use explicit `@Getter`/`@Setter` on entities; reserve `@Data` for DTOs.

## H2 In-Memory Database

H2 is a lightweight in-memory database for development and tests: no external setup, data discarded on shutdown. With it on the classpath, Spring Boot auto-configures an in-memory `DataSource`.

```xml
<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <scope>runtime</scope>
</dependency>
```

> **Note**: switch to a real database (and Flyway/Liquibase migrations) before anything that must persist beyond a JVM run.

## Production DataSource Configuration

For real databases, configure the `DataSource` explicitly and let migrations own the schema; Hibernate only validates it. The high-yield production settings:

```yaml
spring:
  datasource:
    url: jdbc:postgresql://${DB_HOST}:5432/${DB_NAME}
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
    hikari:
      maximum-pool-size: 10           # start here; profile before raising
      minimum-idle: 2                 # a few warm connections
      max-lifetime: 1800000           # 30 min; must be < DB wait_timeout
  jpa:
    hibernate:
      ddl-auto: validate              # never update/create with real data
    properties:
      hibernate:
        jdbc.time_zone: UTC           # consistent Instant/Timestamp across regions
    open-in-view: false               # always disable
```

| Setting             | Why it matters                                                                                  |
| ------------------- | ----------------------------------------------------------------------------------------------- |
| `ddl-auto: validate`| Hibernate verifies entities match the schema, never alters it. `update`/`create` corrupt or wipe data; Flyway/Liquibase owns schema changes. |
| `open-in-view`      | Defaults to `true`, which keeps the JPA session open through serialization and hides lazy-loading N+1s. Set `false` to surface them at dev time. |
| `jdbc.time_zone: UTC`| Without it the JDBC driver uses the JVM zone, so servers in different regions store the same `Instant` differently. |
| HikariCP pool size  | Spring Boot's default pool. Size by `(cores * 2) + disk_spindles` as a start; bigger rarely helps because the DB saturates first, and a pool larger than the DB's max connections causes errors. |
| `max-lifetime`      | Must be shorter than the database's `wait_timeout`, or Hikari hands out stale connections.       |

Common production failures:

- **`ddl-auto: update` with real data**: silent `ALTER TABLE` from entity drift drops constraints and breaks indexes. Keep `validate` or `none`.
- **`open-in-view: true`**: lazy collections trigger queries during JSON serialization, producing silent N+1s in production.
- **Pool exhaustion** (`Connection is not available, request timed out`): usually long transactions or slow queries holding connections, not too small a pool. Add `@Transactional(readOnly = true)` on read paths and profile query times before raising the size.
- **Driver mismatch**: the JDBC URL prefix (`jdbc:postgresql://`) must match the driver dependency, or auto-detection fails cryptically.

> **Reference**: profile-specific config (`application-prod.yml`, `SPRING_PROFILES_ACTIVE`) is covered in [8.1. Spring Profiles](<../08-spring-boot-deployment-cicd/8.1. Spring Profiles.md>); migrations in [8.2. Database Migration with Flyway](<../08-spring-boot-deployment-cicd/8.2. Database Migration with Flyway.md>).

## Q&A

1. **Why does `findById` return `Optional`?**
To force explicit handling of the not-found case at compile time instead of returning `null`. Chain `.orElseThrow(...)` or `.orElse(default)`.

2. **Why avoid `@Data` on JPA entities?**
`@Data` includes an all-fields `@EqualsAndHashCode`, which loads lazy associations during equality checks and gives a different hash before vs after `save()` for generated IDs. Use explicit `@Getter`/`@Setter` on entities.

3. **`ddl-auto`: which value in production?**
`validate` (or `none`). It verifies entities match the schema without modifying it; `update` silently alters tables and `create`/`create-drop` wipe data. Schema is migration-owned.

4. **Why set `open-in-view: false`?**
The default `true` keeps the JPA session open through serialization, so lazy collections trigger queries there and hide N+1s until production. `false` forces explicit fetching or DTO projections, surfacing the bug at dev time.

5. **How do you size the HikariCP pool?**
Start near `(cores * 2) + disk_spindles` and profile under load. Larger pools rarely raise throughput because the database is the bottleneck, and a pool exceeding the DB's max connections causes errors.
