The persistence layer is responsible for:

- Mapping Java objects to database tables
- Performing CRUD operations
- Managing database interactions (via JDBC underneath)

In Spring Boot this layer is provided by **Spring Data JPA** (starter: `spring-boot-starter-data-jpa`), which pulls in JPA APIs, **Hibernate** (the default JPA provider), and the Spring Data abstractions.

> Full coverage of the JPA stack, entity annotations, repository hierarchy, derived queries, `@Query`, and Hibernate config lives in [3.1 Hibernate and JPA](<../03-hibernate-and-spring-data-jpa/3.1. Hibernate and JPA.md>) and the rest of the Hibernate / Spring Data JPA files. This section is a thin orientation for what controllers and services interact with — see those files for the deep dive.

## Entity and Repository

An `@Entity` — class mapped to a database table, with `@Id` as the primary key:

```java
@Entity
@Table(name = "employees")
public class Employee {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private String department;
    private Double salary;

    // getters, setters, constructors
}
```

A repository — declare the interface; Spring Data generates the implementation at runtime:

```java
public interface EmployeeRepository extends JpaRepository<Employee, Long> { }
```

That's enough to get full CRUD plus pagination, sorting, and batch ops. Derived query methods (`findByName(...)`), custom `@Query`, and the full repository hierarchy are covered in [3.2 Spring Data JPA](<../03-hibernate-and-spring-data-jpa/3.2. Spring Data JPA.md>).

`findById` returns `Optional<T>` to force explicit null handling at compile time:

```java
Employee emp = repository.findById(id)
    .orElseThrow(() -> new ResourceNotFoundException("Employee not found: " + id));
```

Avoids the `null` → `NullPointerException` issue that older `findOne(id)` was famous for.

## Lombok (Reducing Boilerplate)

**Lombok** generates getters, setters, constructors, `equals`/`hashCode`, builders, etc. at **compile time** via annotations — keeping entities and DTOs clean.

```java
@Entity
@Getter
@Setter
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class Employee {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private String department;
    private Double salary;
}
```

Common annotations:

| Annotation | Generates |
|---|---|
| `@Getter` / `@Setter` | Getter / setter methods |
| `@NoArgsConstructor` | No-argument constructor |
| `@AllArgsConstructor` | Constructor with all fields |
| `@Data` | `@Getter` + `@Setter` + `@ToString` + `@EqualsAndHashCode` + `@RequiredArgsConstructor` |
| `@Builder` | Fluent builder for object creation |

> **Caution:** `@Data` on JPA entities can cause performance and logic issues. `@EqualsAndHashCode` using all fields interacts badly with lazy associations and generated IDs. Prefer explicit `@Getter`/`@Setter` on entities and reserve `@Data` for DTOs.

## H2 In-Memory Database

**H2** is a lightweight in-memory relational database — handy for development and tests since it needs no external setup and data is discarded on shutdown.

```xml
<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <scope>runtime</scope>
</dependency>
```

With H2 on the classpath, Spring Boot auto-configures a `DataSource` pointing at an in-memory DB — useful for quick experiments, but switch to a real database (and Flyway) before anything that persists beyond a JVM run.

## Production DataSource Configuration

For real databases (PostgreSQL, MySQL, etc.), Spring Boot needs explicit configuration. Schema is owned by **Flyway / Liquibase migrations**: Hibernate is configured to validate, never modify, the schema. See [8.2 Database Migration with Flyway](<../08-spring-boot-deployment-cicd/8.2. Database Migration with Flyway.md>).

### `application.yml` structure

Minimum production-grade configuration:

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/myapp
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
    hikari:
      maximum-pool-size: 10
      minimum-idle: 2
      connection-timeout: 30000      # 30s — fail fast if pool exhausted
      idle-timeout: 600000            # 10 min — release idle conns
      max-lifetime: 1800000           # 30 min — recycle conns to avoid stale state

  jpa:
    hibernate:
      ddl-auto: validate              # never `update` in any environment
    properties:
      hibernate:
        dialect: org.hibernate.dialect.PostgreSQLDialect
        jdbc.time_zone: UTC           # align with `Instant` fields
    open-in-view: false               # always disable
```

> **SQL query logging** (`org.hibernate.SQL`, `format_sql`, `jdbc.bind`) is a debugging concern — covered in [3.9 N+1 Query Problem](<../03-hibernate-and-spring-data-jpa/3.9. N+1 Query Problem.md>) where you actually use it.

### HikariCP pool tuning

Spring Boot uses **HikariCP** by default — a high-performance connection pool. Sizing depends on traffic, not arbitrary defaults.

| Setting | Default | Production guidance |
|---|---|---|
| `maximum-pool-size` | 10 | Start at 10. Match to `(cores × 2) + effective_disk_spindles` rule. Going higher rarely helps — DB usually saturates first |
| `minimum-idle` | same as max | Set to 2–5 — keeps a few warm connections without hoarding |
| `connection-timeout` | 30000 (30s) | Fine for most apps. Lower (5000) if you want fast-fail behavior |
| `idle-timeout` | 600000 (10 min) | Leave default |
| `max-lifetime` | 1800000 (30 min) | Critical — must be **less than** the DB's `wait_timeout` to avoid stale connection errors |

> **Common mistake:** setting `maximum-pool-size: 100` thinking it scales throughput. It causes **DB contention**, not speed-up. Pool size > DB's max connections = errors. Profile before tuning.

### Hibernate properties

| Property | Purpose |
|---|---|
| `ddl-auto: validate` | Hibernate verifies entities match schema on startup; doesn't modify. **Never** use `update` or `create` in any environment — Flyway owns schema |
| `dialect` | Tells Hibernate the DB-specific SQL syntax. Hibernate auto-detects in most cases, but explicit is safer |
| `jdbc.time_zone: UTC` | JDBC reads/writes timestamps in UTC. **Required** for `Instant` audit fields to behave consistently across server regions |
| `open-in-view: false` | Disables Spring's "Open Session In View" behavior (see below) |

### Dev vs Prod profile differences

`application-dev.yml`:
```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/myapp_dev
    username: dev_user
    password: dev_pass
```

`application-prod.yml`:
```yaml
spring:
  datasource:
    url: jdbc:postgresql://${DB_HOST}:5432/${DB_NAME}
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
    hikari:
      maximum-pool-size: 30           # tuned per traffic
```

Profile switching: `--spring.profiles.active=prod` or `SPRING_PROFILES_ACTIVE=prod`. See [8.1 Spring Profiles](<../08-spring-boot-deployment-cicd/8.1. Spring Profiles.md>) for full coverage.

### Common issues

**1. `ddl-auto: update` in production**
Hibernate will silently `ALTER TABLE` based on entity changes. Drops constraints, breaks indexes, corrupts data. **Always `validate` (or `none`) in any environment that owns real data.** Schema changes go through migrations, never Hibernate.

**2. `open-in-view: true` (Spring's default — change it)**
Keeps the JPA session open through view rendering, hiding lazy-loading bugs. Lazy collections accidentally trigger queries during JSON serialization → silent N+1 in production. Setting `false` forces you to fetch eagerly or use DTO projections in the service layer — surfaces bugs at dev time. **Always `open-in-view: false`.**

**3. Wrong `jdbc.time_zone`**
Without `jdbc.time_zone: UTC`, the JDBC driver uses the JVM's default zone for `Instant`/`Timestamp` conversion. A server in IST and another in UTC will read/write the same moment differently. Always set UTC explicitly.

**4. Pool exhaustion under load**
`HikariPool-1 - Connection is not available, request timed out` means all connections are checked out. Causes: long-running transactions holding connections, missing `@Transactional(readOnly = true)` on read-heavy endpoints, slow DB queries. Check query times before bumping pool size — usually a query problem, not a pool size problem.

**5. Driver mismatch**
JDBC URL prefix (`jdbc:postgresql://...`) must match the driver dependency (`postgresql`). Mixing `jdbc:mysql://` with the Postgres driver fails with cryptic auto-detection errors.

## Q&A

1. **Why does `findById` return `Optional`?**
   To force explicit handling of the "not found" case at compile time, instead of returning `null` and inviting `NullPointerException`. Chain with `.orElseThrow(...)` or `.orElse(default)`.

2. **Why avoid `@Data` on JPA entities?**
   `@Data` includes `@EqualsAndHashCode` based on all fields — which loads lazy associations during equality checks and behaves unpredictably for entities with generated IDs (different hash before vs after `save()`). Use explicit `@Getter`/`@Setter` on entities; reserve `@Data` for DTOs.

3. **`ddl-auto: update` vs `validate` vs `none` — which and when?**
   `validate` for any environment with real data — verifies entities match schema, doesn't modify. `none` is fine too. `update` is dangerous (silent ALTER TABLE) and `create`/`create-drop` wipe data on startup — both dev-only at most. In any project using Flyway/Liquibase, schema is migration-owned; Hibernate stays in `validate` or `none`.

4. **Why is `open-in-view: false` important?**
   Spring's default `true` keeps the JPA session open through view rendering. Lazy-loaded collections trigger queries during JSON serialization, producing silent N+1s in production. Setting `false` forces explicit fetching or DTO projections in the service layer — bugs surface at dev time instead of production.

5. **How do you size the HikariCP connection pool?**
   Not by arbitrary defaults. Match to `(cores × 2) + effective_disk_spindles` as a starting point, then profile under load. Larger pools rarely improve throughput because the **database**, not the pool, is the bottleneck. Pool size > DB's max connections = errors. Most apps run fine on 10–30.

6. **Why set `jdbc.time_zone: UTC`?**
   Without it, the JDBC driver uses the JVM's default timezone for timestamp conversion. A server in IST and another in UTC writing `Instant.now()` produce different stored values for the same moment. Setting UTC at the JDBC layer makes timestamps server-region-independent.
