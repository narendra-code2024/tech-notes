The service layer sits between the controller (presentation) and the repository (persistence). It holds the application's business logic and keeps both neighbours thin: controllers stay focused on HTTP, repositories on data access.

It implements business rules, coordinates across repositories, manages transactions, performs non-HTTP validation, and triggers domain events.

---

## `@Service` Annotation

`@Service` is a specialization of `@Component`; it registers the class as a Spring-managed bean and marks its role as business logic. Being a bean enables dependency injection, transaction management, and AOP cross-cutting concerns (logging, security, metrics).

```java
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final PaymentService paymentService;

    public OrderService(OrderRepository orderRepository, PaymentService paymentService) {
        this.orderRepository = orderRepository;
        this.paymentService = paymentService;
    }

    public Order placeOrder(Order order) {
        paymentService.charge(order);
        return orderRepository.save(order);
    }
}
```

> **Note**: one service, one responsibility. Keep controllers thin; they should only translate HTTP to service calls and back.

---

## Transactions in the Service Layer

The service layer is where transactions naturally live: it is the unit that knows when a piece of business logic must be atomic. Annotate the method with `@Transactional` so a failure in any step rolls back the whole operation.

```java
@Service
public class OrderService {

    @Transactional
    public void placeOrder(Order order) {
        orderRepository.save(order);
        paymentService.charge(order);   // if this throws, the save rolls back
    }
}
```

> **Note**: `@Transactional` is enforced by an AOP proxy, so calling a `@Transactional` method from within the same class (`this.method()`) bypasses the proxy and no transaction starts.

> **Reference**: propagation, isolation, rollback rules, `readOnly`, the proxy mechanism, and self-invocation fixes are covered in [11.6. Transactional Annotation in Spring Boot](<../11-caching-and-concurrent-transaction-management/11.6. Transactional Annotation in Spring Boot.md>).

## Q&A

1. **Why have a service layer at all?**
Separation of concerns: controllers handle HTTP, repositories handle persistence, services hold business logic. This keeps each layer testable and independently evolvable, and gives transactions and cross-cutting concerns one clean home.

2. **Where do transactions belong, controller, service, or repository?**
The service layer. Controllers should not know about transactions (an HTTP-vs-DB concern leak), and repositories operate on single statements. Services coordinate the multi-step business operations a transaction wraps.
