The **service layer** sits between the controller (presentation) and the repository (persistence). It holds the application's **business logic** and keeps both neighbours thin — controllers stay focused on HTTP, repositories stay focused on data access.

Responsibilities:

- Implement business rules
- Coordinate across multiple repositories
- Manage transactions
- Perform non-HTTP validation
- Trigger domain events

---

## `@Service` Annotation

`@Service` is a specialization of `@Component` — it registers the class as a Spring-managed bean and signals its role as the business-logic layer. Being a bean enables **DI**, transaction management, and AOP cross-cutting concerns (logging, security, metrics).

```java
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final PaymentService paymentService;

    public OrderService(OrderRepository orderRepository, PaymentService paymentService) {
        this.orderRepository = orderRepository;
        this.paymentService = paymentService;
    }

    public Order placeOrder(Order order) {
        paymentService.charge(order);
        return orderRepository.save(order);
    }
}
```

> **Best practice:** one service = one responsibility. Keep controllers thin — they should only translate HTTP to service calls and back.

---

## Transactions in the Service Layer

The service layer is where transactions naturally live — it's the unit that knows when a piece of business logic should be atomic. Annotate the method with `@Transactional`:

```java
@Service
public class OrderService {

    @Transactional
    public void placeOrder(Order order) {
        orderRepository.save(order);
        paymentService.charge(order);   // if this throws, the save is rolled back
    }
}
```

> Full coverage of `@Transactional` (propagation, isolation, rollback rules, `readOnly`, the AOP proxy mechanism, all attributes) lives in [11.6 `@Transactional` Annotation in Spring Boot](<../11-caching-and-concurrent-transaction-management/11.6. Transactional Annotation in Spring Boot.md>).

> **Caution:** `@Transactional` is enforced by an **AOP proxy**, so calling a `@Transactional` method **from within the same class** (`this.method()`) bypasses the proxy and the transaction never starts. Classic service-layer behavior. Full explanation and fixes: [11.6 `@Transactional` Annotation in Spring Boot](<../11-caching-and-concurrent-transaction-management/11.6. ## Q&A

1. **Why have a service layer at all?**
   Separation of concerns. Controllers handle HTTP, repositories handle persistence, services hold business logic. This keeps each layer testable, reusable, and independently evolvable — and gives transactions and cross-cutting concerns a single clean home.

2. **Where do transactions belong — controller, service, or repository?**
   The **service layer**. Controllers shouldn't know about transactions (HTTP-vs-DB concern leak), and repositories operate on single statements. Services coordinate multi-step business operations, which is exactly what a transaction wraps.
