`ResponseEntity<T>` represents the **entire HTTP response**: body, status code, and headers — and gives a controller full control over what gets sent back. `T` is the body type (DTO, `String`, `List`, etc.).

Without it, a plain return value always produces `200 OK` with no header control:

```java
@GetMapping("/hello")
public String hello() {
    return "Hello";   // always 200 OK, no control over headers
}
```

With `ResponseEntity`, you set the status, body, and headers explicitly:

```java
@GetMapping("/hello")
public ResponseEntity<String> hello() {
    return ResponseEntity.ok("Hello");
}
```

---

## Creating a `ResponseEntity`

**Builder methods (preferred)**: fluent and readable:

```java
return ResponseEntity.ok(body);                               // 200 OK
return ResponseEntity.status(HttpStatus.CREATED).body(body);  // 201 Created
return ResponseEntity.noContent().build();                    // 204 No Content
return ResponseEntity.notFound().build();                     // 404 Not Found
return ResponseEntity.badRequest().body(errors);              // 400 Bad Request
```

**Constructor form**: still works, more verbose:

```java
return new ResponseEntity<>(body, HttpStatus.OK);
```

### Full example — POST returning 201 with a body

```java
@RestController
@RequestMapping("/api/products")
public class ProductController {

    private final ProductService productService;

    public ProductController(ProductService productService) {
        this.productService = productService;
    }

    @PostMapping
    public ResponseEntity<ProductDto> createProduct(@RequestBody ProductDto dto) {
        ProductDto saved = productService.create(dto);
        return ResponseEntity
            .status(HttpStatus.CREATED)
            .header("Location", "/api/products/" + saved.getId())
            .body(saved);
    }
}
```

## Q&A

1. **When should you use `ResponseEntity` vs returning the object directly?**
   Use `ResponseEntity` when you need control over the **status code** (201 Created, 204 No Content, 404, etc.) or **response headers** (`Location`, `ETag`, cache control). For straightforward 200-OK success cases, returning the DTO directly is cleaner — especially combined with a global response wrapper (see [2.8 API Response Patterns](<2.8. API Response Patterns.md>)).

2. **How do you return a 201 Created with the new resource's URL?**
   Use `ResponseEntity.status(HttpStatus.CREATED)` and a `Location` header pointing to the new resource — either manually or via `ResponseEntity.created(uri).body(dto)`.

3. **Difference between `ResponseEntity.noContent()` and `ResponseEntity.ok().build()`?**
   `noContent()` produces **204 No Content**; `ok().build()` produces **200 OK** with an empty body. Use 204 for successful operations that intentionally return no body (DELETEs, PUTs where the client doesn't need the result).
