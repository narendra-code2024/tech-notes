Spring MVC is Spring's framework for handling HTTP requests, built on the **front controller** pattern: one servlet receives every request and dispatches it to the matching handler. That servlet is the `DispatcherServlet`, the single entry point for both server-rendered views and REST/JSON APIs.

Spring Boot auto-registers `DispatcherServlet` and an embedded **Tomcat** (the default container); traditional Spring wired it manually via `web.xml` or a servlet initializer.

---

## MVC Pattern

Separation of concerns into three roles:

- **Model**: application data and domain state (entities, DTOs).
- **View**: the client-facing representation. Server-rendered HTML via Thymeleaf/JSP in classic MVC; for a REST API the "view" is the JSON body Jackson produces, so no template engine is involved.
- **Controller**: receives the request, delegates to the service layer, and returns a view name (MVC) or an object/DTO (REST).

---

## Request Processing Flow

Every request passes through `DispatcherServlet`, which orchestrates the supporting components rather than doing the work itself:

1. **Client to `DispatcherServlet`**: e.g. `GET /api/products`. The servlet container (Tomcat) creates the `HttpServletRequest` / `HttpServletResponse` and hands them over. These live in `jakarta.servlet.http` (renamed from `javax.servlet.http` in Spring Boot 3.0).
2. **`HandlerMapping`**: resolves which handler (controller method) serves the request, matching on URL path, HTTP method, headers, and media types. Default is `RequestMappingHandlerMapping`, which reads `@RequestMapping` / `@GetMapping` and friends.
3. **`HandlerAdapter`**: invokes the matched handler. `DispatcherServlet` never calls the controller directly; `RequestMappingHandlerAdapter` resolves method arguments (`@PathVariable`, `@RequestParam`, `@RequestBody`) and binds the return value. Any registered `HandlerInterceptor` runs `preHandle` / `postHandle` around this step.
4. **Controller executes**: typically calls the service layer and returns a result (view name or `ModelAndView` for MVC, object/DTO for REST).
5. **Result rendering**, which diverges by application type:
    - **MVC**: a `ViewResolver` maps the view name to a template (JSP/Thymeleaf), which renders HTML.
    - **REST**: an `HttpMessageConverter` (Jackson for JSON, by default) serializes the object straight to the response body; no `ViewResolver` is involved.
6. **Response to client**, written back through the container.

> **Note**: a handler that throws is routed to a `HandlerExceptionResolver` instead of step 5; see [2.7. Exception Handling](https://claude.ai/chat/2.7.%20Exception%20Handling.md).

## Traditional MVC vs REST

The annotation chosen determines the response path: `@RestController` is `@Controller` + `@ResponseBody`, so its return value skips view resolution.

```java
@Controller
public class HomeController {
    @GetMapping("/home")
    public String home(Model model) {
        model.addAttribute("msg", "Hello");
        return "home";              // ViewResolver -> home.html / home.jsp
    }
}

@RestController
@RequestMapping("/api")
public class ProductController {
    private final ProductService service;

    public ProductController(ProductService service) {
        this.service = service;
    }

    @GetMapping("/products")
    public List<Product> getProducts() {
        return service.getAll();    // Jackson -> JSON, no ViewResolver
    }
}
```

|Type|Return value|Rendered by|Output|
|---|---|---|---|
|Traditional MVC|view name / `ModelAndView`|`ViewResolver` -> JSP/Thymeleaf|HTML|
|REST API|object / DTO|`HttpMessageConverter` (Jackson)|JSON|

> **Note**: `ModelAndView` bundles the view name and model into one object, useful when the view is chosen conditionally. Add `@ResponseBody` to a single `@Controller` method to bypass the `ViewResolver` for just that method.

## Q&A

1. **What is the role of `DispatcherServlet`?** The front controller: every HTTP request passes through it. It consults `HandlerMapping` to find the handler, invokes it through a `HandlerAdapter`, then routes the result via a `ViewResolver` (MVC) or an `HttpMessageConverter` (REST).
    
2. **`HandlerMapping` vs `HandlerAdapter`?**
    

- **`HandlerMapping`**: decides _which_ handler serves the request (URL + method + headers + media type).
- **`HandlerAdapter`**: knows _how_ to invoke that handler, resolving its arguments and return value. The split lets one dispatcher support different handler styles (annotated methods, functional endpoints).

3. **How does Spring choose between returning a view and returning JSON?** By the handler's return path. `@Controller` returning a `String` or `ModelAndView` goes through the `ViewResolver`; `@RestController` (or a `@ResponseBody` method) skips view resolution and uses an `HttpMessageConverter`, Jackson for JSON.
    
4. **Who creates `HttpServletRequest` / `HttpServletResponse`?** The servlet container (Tomcat by default), not Spring; `DispatcherServlet` receives them from the container. Since Spring Boot 3.0 they live in the `jakarta.servlet.http` package, formerly `javax.servlet.http`.

## Quick Revision Cheat Sheet

|Concept|Remember|
|---|---|
|`DispatcherServlet`|Front controller; auto-registered by Spring Boot|
|`HandlerMapping`|Picks which controller method handles the request|
|`HandlerAdapter`|Invokes the handler; binds arguments and return value|
|`ViewResolver`|View name to template; MVC only|
|`HttpMessageConverter`|Object to JSON/XML; Jackson is the REST default|
|`@Controller`|Returns a view name or `ModelAndView`|
|`@RestController`|`@Controller` + `@ResponseBody`; returns JSON|
|jakarta namespace|Servlet API is `jakarta.*` since Spring Boot 3.0|
