An **API** (Application Programming Interface) is a contract that lets software applications communicate — exchanging data and invoking functionality across process or network boundaries. In Spring Boot the dominant style is **REST over HTTP**, returning JSON.

---

## MVC: Model-View-Controller

**MVC** is a design pattern built on **separation of concerns**: three roles, each with a single responsibility:

- **Model**: application data; interacts with the database (entities, domain objects)
- **View**: renders data for the client. Thymeleaf/JSP for server-rendered HTML; for REST APIs the "view" is the JSON body produced by Jackson
- **Controller**: receives HTTP requests, delegates to the service layer, returns a response (view name for MVC, object/DTO for REST)

---

## Spring MVC Request Flow

1. **Client sends HTTP request**: e.g. `GET /api/hello`
2. **DispatcherServlet receives it**: the front controller that every request passes through. Spring Boot auto-registers it; in traditional Spring it was wired via `web.xml` or `AbstractDispatcherServletInitializer`. The `HttpServletRequest` / `HttpServletResponse` objects themselves are created by the **servlet container** (Tomcat by default), not by `DispatcherServlet`.
3. **HandlerMapping lookup**: `DispatcherServlet` asks: *which controller method handles this?* Matching uses URL path, HTTP method, headers, and media types.
4. **Controller method invoked**: Spring resolves method parameters automatically via `@PathVariable`, `@RequestParam`, `@RequestBody`, etc.
5. **Controller processes the request**: typically calls the service layer and returns a result.
6. **Response handling**: diverges by application type:
    - **Traditional MVC**: controller returns a view name or `ModelAndView`; a `ViewResolver` locates the template (JSP/Thymeleaf); the rendered HTML is sent to the client.
    - **REST API**: controller returns an object/DTO; an `HttpMessageConverter` (Jackson, by default) serializes it to JSON. `ViewResolver` is **not** involved.

---

## Traditional MVC vs REST API

**Traditional MVC**: controller returns a view name; `ViewResolver` finds the template. Two equivalent forms:

```java
@Controller
public class HomeController {

    // Form 1 — return view name as String, pass data via Model
    @GetMapping("/home")
    public String home(Model model) {
        model.addAttribute("message", "Hello");
        return "home";   // resolves to home.jsp or home.html
    }

    // Form 2 — return ModelAndView (view name + data in one object)
    @GetMapping("/about")
    public ModelAndView about() {
        ModelAndView mav = new ModelAndView("about");
        mav.addObject("message", "About us");
        return mav;
    }
}
```

> Both forms hit the `ViewResolver`. `String` + `Model` is more common; `ModelAndView` is useful when you want to set the view name conditionally or build the response in one place.

**REST API**: controller returns an object; Jackson serializes to JSON:

```java
@RestController
@RequestMapping("/api")
public class ProductController {

    private final ProductService productService;

    public ProductController(ProductService productService) {
        this.productService = productService;
    }

    @GetMapping("/products")
    public List<Product> getProducts() {
        return productService.getAll();
    }
}
```

**Flow comparison:**

| Path | Flow |
|---|---|
| Traditional MVC | Controller → view name / `ModelAndView` → `ViewResolver` → JSP/Thymeleaf → HTML response |
| REST API | Controller → Object/DTO → `HttpMessageConverter` (Jackson) → JSON response |

## Q&A

1. **What is the role of `DispatcherServlet`?**
   It's the **front controller** in Spring MVC — every HTTP request passes through it. It consults `HandlerMapping` to find the matching controller, invokes it, and routes the result through either a `ViewResolver` (MVC) or an `HttpMessageConverter` (REST).

2. **Who creates `HttpServletRequest` and `HttpServletResponse`?**
   The **servlet container** (Tomcat by default), not Spring. `DispatcherServlet` receives them from the container and dispatches to the matched handler.

3. **How does Spring decide between returning a view and returning JSON?**
   By controller type and return signature. `@Controller` returning a `String` or `ModelAndView` goes through `ViewResolver`. `@RestController` (or `@ResponseBody` on the method) skips view resolution and uses `HttpMessageConverter` to serialize the return value — Jackson for JSON.

4. **What is `HandlerMapping`?**
   The component that maps an incoming request (URL + method + headers + media types) to a specific controller method. `RequestMappingHandlerMapping` is the default — it reads `@RequestMapping` / `@GetMapping` / etc. on your controllers.

## Quick Reference

|Concept|Remember|
|---|---|
| `DispatcherServlet` | Front controller — auto-registered in Spring Boot |
| `HandlerMapping` | Finds the controller method for a request |
| `ViewResolver` | Resolves view names to templates (MVC only) |
| `HttpMessageConverter` | Serializes objects to JSON/XML (REST) — Jackson is default |
| `@Controller` | Returns view name or `ModelAndView` |
| `@RestController` | `@Controller` + `@ResponseBody` — returns JSON |
