## Choosing a Response Pattern

Spring's default is **plain**: controllers return DTOs directly and Jackson serializes them to JSON. Works for most APIs. You move to a **custom envelope** when you need:

- **A separate `meta` slot** for pagination, totals, or other non-resource metadata about the resource itself
- **Team-wide consistency**: uniform shape clients can always parse the same way

Trace IDs / request IDs / server timing belong in **HTTP response headers** (`X-Request-ID`, `traceparent`), not the response body — handled by a filter/interceptor, not the envelope.

> For paginated response format, Spring's built-in `Page<T>` already returns `content + pagination metadata` automatically — see [3.3 Sorting and Pagination](<../03-hibernate-and-spring-data-jpa/3.3. Sorting and Pagination.md>). No custom envelope needed.

## Custom Envelope — `ApiResponse<T>` with `data` + `meta`

When you want a uniform envelope across **all** responses (not just paginated ones) — typically to match a team-wide shape clients can always parse — define a custom `ApiResponse<T>`.

```java
public class ApiResponse<T> {

    private T data;
    private Meta meta;

    private ApiResponse(T data, Meta meta) {
        this.data = data;
        this.meta = meta;
    }

    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(data, null);
    }

    public static <T> ApiResponse<T> success(T data, Meta meta) {
        return new ApiResponse<>(data, meta);
    }

    // getters, setters
}
```

A `Meta` class for pagination + extensibility:

```java
public class Meta {
    private Pagination pagination;
    // extensible — add fields specific to the resource (e.g., totals, summaries)

    public static class Pagination {
        private int page;
        private int size;
        private long totalElements;
        private int totalPages;
        // getters, setters
    }

    // getters, setters
}
```

Resulting JSON shapes:

```json
// Single resource
{
    "data": { "id": "uuid", "firstName": "John" }
}
```

```json
// Paginated list
{
    "data": [
        { "id": "uuid-1", "firstName": "John" },
        { "id": "uuid-2", "firstName": "Jane" }
    ],
    "meta": {
        "pagination": {
            "page": 0,
            "size": 20,
            "totalElements": 100,
            "totalPages": 5
        }
    }
}
```

## Auto-Wrapping with `ResponseBodyAdvice`

Without auto-wrapping, every controller method has to call `ApiResponse.success(...)` manually. `ResponseBodyAdvice` is a Spring extension point that **intercepts every controller response body** before serialization — wrap once globally, controllers stay clean.

```java
@RestControllerAdvice
public class GlobalResponseHandler implements ResponseBodyAdvice<Object> {

    @Override
    public boolean supports(MethodParameter returnType,
                            Class<? extends HttpMessageConverter<?>> converterType) {
        return true;   // intercept every controller response
    }

    @Override
    public Object beforeBodyWrite(Object body, MethodParameter returnType,
                                  MediaType selectedContentType,
                                  Class<? extends HttpMessageConverter<?>> selectedConverterType,
                                  ServerHttpRequest request, ServerHttpResponse response) {

        // Already wrapped — pass through (avoid double-wrapping)
        if (body instanceof ApiResponse) {
            return body;
        }

        // Errors from @RestControllerAdvice — pass through, do NOT wrap
        if (body instanceof ProblemDetail || body instanceof ApiError) {
            return body;
        }

        // Page<T> — extract content + pagination meta into the envelope
        if (body instanceof Page<?> page) {
            Meta meta = new Meta();
            meta.setPagination(toPagination(page));
            return ApiResponse.success(page.getContent(), meta);
        }

        // Plain DTO — wrap in success envelope
        return ApiResponse.success(body);
    }

    private Meta.Pagination toPagination(Page<?> page) {
        Meta.Pagination p = new Meta.Pagination();
        p.setPage(page.getNumber());
        p.setSize(page.getSize());
        p.setTotalElements(page.getTotalElements());
        p.setTotalPages(page.getTotalPages());
        return p;
    }
}
```

Now controllers return raw types and the envelope is applied automatically:

```java
@GetMapping("/products/{id}")
public ProductDto getProduct(@PathVariable Long id) {
    return productService.findById(id);
}

@GetMapping("/products")
public Page<ProductDto> listProducts(Pageable pageable) {
    return productService.findAll(pageable);
}
```

The `getProduct` response becomes `{ "data": { ... } }`. The `listProducts` response becomes `{ "data": [...], "meta": { "pagination": { ... } } }`.

## Trade-offs

- You give up direct control over **HTTP status codes** from a plain return type — most `200 OK` cases don't care, but for `201 Created` or `204 No Content` you still need `ResponseEntity` (or `@ResponseStatus` on the method)
- For success responses you can still use `ResponseEntity` — the advice unwraps the body, applies the envelope, then re-wraps in `ResponseEntity` with the right status
- The envelope adds one layer of indirection on the client; worth it when the team values uniformity over modern REST idioms
- Forgetting the "already wrapped" guard causes **double-wrapping** on errors — the advice must skip `ProblemDetail` / `ApiError` explicitly

## Industry Patterns

| Pattern | Success shape | Error shape | Real-world users | Best for |
|---|---|---|---|---|
| **Plain REST** | DTO directly; `Page<T>` for lists | `ProblemDetail` (RFC 7807) | Stripe, GitHub, Twilio, AWS | New Spring 6+ projects, public APIs, modern idiom |
| **Wrapped envelope** | `{ data, meta }` via `ResponseBodyAdvice` | Custom `ApiError` with trace `id` (unwrapped) | Many internal Java/Spring + Node/Laravel teams | Team consistency, uniform client parsing |
| **JSON:API spec** | `{ data, meta, links, included }` (formal) | `{ errors: [...] }` (formal array of error objects) | Some GitHub APIs, formal at [jsonapi.org](https://jsonapi.org) | APIs needing strict standard with relationships/sideloading |

Plain REST and wrapped envelope are both defensible production choices for typical Spring Boot apps. Pick one and apply consistently across the API.

## Q&A

1. **What is `ResponseBodyAdvice` used for?**
   Global hook that intercepts every controller response body before serialization. Most common use: wrapping all responses in a uniform envelope so success responses share one shape — without forcing every controller to call the wrapper manually.

2. **How do you avoid double-wrapping?**
   Check `body instanceof ApiResponse` (or `ProblemDetail` / `ApiError`) inside `beforeBodyWrite` and return the body unchanged if it's already in target shape. Only wrap raw DTOs.

3. **What's the trade-off of returning plain DTOs instead of `ResponseEntity`?**
   You lose direct control over status codes and headers. For `200 OK` success this is fine; for `201 Created`, `204 No Content`, or custom headers, use `ResponseEntity` (the advice still applies the envelope inside it).

4. **Why use `Page<T>` instead of a custom envelope for pagination?**
   `Page<T>` is built into Spring — serializes with full pagination metadata (`totalElements`, `totalPages`, `pageable`) automatically, no custom code. If pagination is the only reason you'd wrap, `Page<T>` alone is enough.
