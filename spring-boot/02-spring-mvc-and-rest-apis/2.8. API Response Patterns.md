## Choosing a Response Pattern

Spring's default is **plain**: controllers return DTOs and Jackson serializes them, which works for most APIs. Move to a **custom envelope** only when you need a dedicated `meta` slot (pagination, totals) or team-wide uniformity so clients always parse the same shape.

Trace/request IDs and server timing belong in HTTP response headers (`X-Request-ID`, `traceparent`), set by a filter or interceptor, not in the body envelope.

> **Note**: for paginated lists, Spring's `Page<T>` already carries content plus pagination metadata, so no custom envelope is needed just for pagination; see [3.3. Sorting and Pagination](<../03-hibernate-and-spring-data-jpa/3.3. Sorting and Pagination.md>).

## Custom Envelope: `ApiResponse<T>`

For a uniform envelope across all responses, define a generic `ApiResponse<T>` with a `data` payload and an optional `meta`:

```java
public class ApiResponse<T> {

    private T data;
    private Meta meta;

    private ApiResponse(T data, Meta meta) {
        this.data = data;
        this.meta = meta;
    }

    public static <T> ApiResponse<T> success(T data)            { return new ApiResponse<>(data, null); }
    public static <T> ApiResponse<T> success(T data, Meta meta) { return new ApiResponse<>(data, meta); }
    // getters
}

public class Meta {
    private Pagination pagination;          // extend with totals, summaries, etc.
    public static class Pagination {
        private int page;
        private int size;
        private long totalElements;
        private int totalPages;
        // getters, setters
    }
    // getters, setters
}
```

```json
{
    "data": [ { "id": "uuid-1", "firstName": "John" } ],
    "meta": { "pagination": { "page": 0, "size": 20, "totalElements": 100, "totalPages": 5 } }
}
```

## Auto-Wrapping with `ResponseBodyAdvice`

`ResponseBodyAdvice` intercepts every controller response body before serialization, so the envelope is applied once globally and controllers return raw types. The key correctness concern is the skip-wrap guards, which prevent double-wrapping and corrupting error bodies.

```java
@RestControllerAdvice
public class GlobalResponseHandler implements ResponseBodyAdvice<Object> {

    @Override
    public boolean supports(MethodParameter returnType,
                            Class<? extends HttpMessageConverter<?>> converterType) {
        return true;   // intercept every controller response
    }

    @Override
    public Object beforeBodyWrite(Object body, MethodParameter returnType,
                                  MediaType contentType,
                                  Class<? extends HttpMessageConverter<?>> converterType,
                                  ServerHttpRequest request, ServerHttpResponse response) {

        // Skip already-wrapped bodies and error bodies from the exception advice
        if (body instanceof ApiResponse || body instanceof ProblemDetail || body instanceof ApiError) {
            return body;
        }
        // Page<T>: lift content + pagination into the envelope
        if (body instanceof Page<?> page) {
            Meta meta = new Meta();
            meta.setPagination(toPagination(page));
            return ApiResponse.success(page.getContent(), meta);
        }
        return ApiResponse.success(body);   // plain DTO
    }

    private Meta.Pagination toPagination(Page<?> page) {
        Meta.Pagination p = new Meta.Pagination();
        p.setPage(page.getNumber());
        p.setSize(page.getSize());
        p.setTotalElements(page.getTotalElements());
        p.setTotalPages(page.getTotalPages());
        return p;
    }
}
```

Controllers then return raw types, and the advice wraps automatically: a `ProductDto` becomes `{ "data": { ... } }` and a `Page<ProductDto>` becomes `{ "data": [...], "meta": { "pagination": { ... } } }`.

## Trade-offs

- A plain return type cedes direct control over the status code; for `201 Created` or `204 No Content` you still need `ResponseEntity` or `@ResponseStatus` on the method.
- The envelope adds a layer of indirection for clients; worth it when the team values uniformity over modern REST idioms.
- Forgetting the skip-wrap guard double-wraps errors, so the advice must pass `ProblemDetail` / `ApiError` through untouched.

> **Note**: serializing `Page<T>` (`PageImpl`) directly produces an unstable JSON structure, and Spring Boot 3.3+ warns about it. For a guaranteed-stable shape, return `PagedModel` or set `spring.data.web.pageable.serialization-mode=via_dto`. A custom envelope that maps `page.getContent()` itself (as above) sidesteps the issue.

## Industry Patterns

| Pattern          | Success shape                              | Error shape                          | Best for                                      |
| ---------------- | ------------------------------------------ | ------------------------------------ | --------------------------------------------- |
| Plain REST       | DTO directly; `Page<T>` for lists          | `ProblemDetail` (RFC 9457)           | New Spring 6+ projects, public APIs, modern idiom |
| Wrapped envelope | `{ data, meta }` via `ResponseBodyAdvice`  | Custom `ApiError` with trace `id`    | Team consistency, uniform client parsing      |
| JSON:API spec    | `{ data, meta, links, included }`          | `{ errors: [...] }`                  | Strict standard with relationships/sideloading |

Plain REST and the wrapped envelope are both defensible. Pick one and apply it consistently across the API.

## Q&A

1. **What is `ResponseBodyAdvice` used for?**
A global hook that intercepts every controller response body before serialization. The common use is wrapping all responses in a uniform envelope without each controller calling the wrapper manually.

2. **How do you avoid double-wrapping?**
Inside `beforeBodyWrite`, return the body unchanged if it is already an `ApiResponse` (or a `ProblemDetail` / `ApiError` error body); only wrap raw DTOs.

3. **What's the trade-off of returning plain DTOs instead of `ResponseEntity`?**
You lose direct control over status codes and headers. Fine for `200 OK`; for `201`, `204`, or custom headers use `ResponseEntity` (the advice still applies the envelope inside it).

4. **Why use `Page<T>` instead of a custom envelope for pagination?**
`Page<T>` carries pagination metadata with no custom code. Note that serializing it directly is unstable (Boot 3.3+); use `PagedModel`, the `via_dto` serialization mode, or map its content into your own envelope.
