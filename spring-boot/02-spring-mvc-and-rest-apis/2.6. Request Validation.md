## Why Validate?

**Input validation** ensures incoming request data is correct, complete, and safe — blocking invalid data from reaching business logic or persistence. It prevents:

- Bad data polluting the database
- Unexpected runtime errors downstream
- A class of security issues (overflow, injection-adjacent bugs)

Spring Boot uses **Bean Validation** (Jakarta Validation API) via the Hibernate Validator implementation.

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
```

## How Validation Works

```
Client Request
     ↓
DTO with validation annotations
     ↓
@Valid on controller parameter
     ↓
Spring validation engine
     ↓
MethodArgumentNotValidException on failure (before controller body runs)
```

`@Valid` on a `@RequestBody` parameter triggers validation. If any constraint fails, Spring throws `MethodArgumentNotValidException` **before** the controller method executes.

```java
public class UserDto {

    @NotBlank(message = "Name is required")
    private String name;

    @Email(message = "Invalid email format")
    private String email;

    @Positive(message = "Age must be positive")
    private Integer age;

    // getters, setters
}

@RestController
@RequestMapping("/api/users")
public class UserController {

    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    @PostMapping
    public ResponseEntity<UserDto> createUser(@Valid @RequestBody UserDto dto) {
        return ResponseEntity.ok(userService.create(dto));
    }
}
```

## Validation Annotations

**Null & empty checks:**

| Annotation | Description |
|---|---|
| `@NotNull` | Must not be null |
| `@NotEmpty` | Not null + size > 0 (String, Collection, Array) |
| `@NotBlank` | Not null + trimmed length > 0 (String only) |

**Size & length:**

| Annotation | Description |
|---|---|
| `@Size(min, max)` | Validates size/length of String, Collection, Array |

**Numeric:**

| Annotation | Description |
|---|---|
| `@Min` / `@Max` | Minimum / maximum integer value |
| `@DecimalMin` / `@DecimalMax` | Min/max supporting decimals |
| `@Positive` / `@PositiveOrZero` | > 0 / ≥ 0 |
| `@Negative` / `@NegativeOrZero` | < 0 / ≤ 0 |
| `@Digits(integer, fraction)` | Limits digit counts |

**Pattern & format:**

| Annotation | Description |
|---|---|
| `@Email` | Valid email format |
| `@Pattern(regexp = "...")` | Regex-based validation |

**Date & time:**

| Annotation | Description |
|---|---|
| `@Past` / `@PastOrPresent` | Date must be in the past (or today) |
| `@Future` / `@FutureOrPresent` | Date must be in the future (or today) |

> Boolean checks: `@AssertTrue` / `@AssertFalse` — rare, used when a field must be exactly `true` or `false`.

## Nested Object Validation

To validate fields inside a nested object, place `@Valid` on the nested field — validation then cascades:

```java
public class OrderDto {

    @NotBlank
    private String orderId;

    @Valid                          // ← cascades validation into AddressDto
    private AddressDto shippingAddress;
}
```

Without `@Valid` on the nested field, annotations inside `AddressDto` are ignored.

## Handling Validation Errors

Spring raises `MethodArgumentNotValidException` when constraints fail. Handle it centrally with `@RestControllerAdvice` (see [2.7 Exception Handling](<2.7. Exception Handling.md>)). The binding result exposes each field error:

```java
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<Map<String, String>> handleValidation(MethodArgumentNotValidException ex) {
    Map<String, String> errors = new HashMap<>();
    ex.getBindingResult().getFieldErrors().forEach(err ->
        errors.put(err.getField(), err.getDefaultMessage())
    );
    return ResponseEntity.badRequest().body(errors);
}
```

Typical response shape:

```json
{
    "email": "Invalid email format",
    "age": "Must be positive"
}
```

## Q&A

1. **What does `@Valid` do on a controller parameter?**
   Triggers **Bean Validation** on the incoming object before the controller body runs. If any constraint annotation on the DTO fails, Spring throws `MethodArgumentNotValidException` — which you handle centrally in a `@RestControllerAdvice`.

2. **Difference between `@NotNull`, `@NotEmpty`, and `@NotBlank`?**
   `@NotNull` — just not null (empty string OK). `@NotEmpty` — not null and size > 0 (applies to String/Collection/Array). `@NotBlank` — not null and trimmed length > 0 (String only; rejects `"   "`).

3. **How do you validate a nested object inside a DTO?**
   Annotate the nested field with `@Valid`. Without it, constraints on the nested object's fields are skipped.

4. **Why validate DTOs instead of entities?**
   Validation is an API concern, not a persistence concern. Entity validation runs on save (too late — you've already done work) and entities may need different rules than the API accepts. DTO validation stops bad data at the boundary.

## Quick Reference

|Concept|Remember|
|---|---|
| `@Valid` | Triggers Bean Validation on the annotated parameter/field |
| `@NotBlank` | String — not null + trimmed length > 0 |
| `@NotEmpty` | String/Collection/Array — not null + size > 0 |
