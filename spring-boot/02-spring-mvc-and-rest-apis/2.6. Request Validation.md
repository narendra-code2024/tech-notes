Input validation blocks malformed or unsafe request data before it reaches business logic or persistence, stopping bad data at the API boundary. Spring Boot uses **Bean Validation** (the Jakarta Validation API) via the Hibernate Validator implementation, added with `spring-boot-starter-validation`.

Constraint annotations live in `jakarta.validation.constraints` (renamed from `javax.validation.*` in Spring Boot 3.0).

---

## How Validation Works

`@Valid` on a `@RequestBody` parameter triggers validation; if any constraint fails, Spring throws `MethodArgumentNotValidException` before the controller body runs.

```
Request -> DTO with constraint annotations -> @Valid on parameter
        -> validation engine -> MethodArgumentNotValidException on failure
```

```java
public class UserDto {

    @NotBlank(message = "Name is required")
    private String name;

    @Email(message = "Invalid email format")
    private String email;

    @Positive(message = "Age must be positive")
    private Integer age;
    // getters, setters
}

@PostMapping("/api/users")
public ResponseEntity<UserDto> createUser(@Valid @RequestBody UserDto dto) {
    return ResponseEntity.ok(userService.create(dto));
}
```

## Constraint Annotations

| Annotation                         | Validates                                                  |
| ---------------------------------- | ---------------------------------------------------------- |
| `@NotNull`                         | Not null (empty string still passes)                       |
| `@NotEmpty`                        | Not null and size > 0 (String, Collection, Array)          |
| `@NotBlank`                        | Not null and trimmed length > 0 (String only)              |
| `@Size(min, max)`                  | Length/size bounds (String, Collection, Array)             |
| `@Min` / `@Max`                    | Minimum / maximum numeric value                            |
| `@DecimalMin` / `@DecimalMax`      | Decimal-aware min/max                                      |
| `@Positive` / `@PositiveOrZero`    | > 0 / >= 0                                                 |
| `@Negative` / `@NegativeOrZero`    | < 0 / <= 0                                                 |
| `@Digits(integer, fraction)`       | Caps integer and fraction digit counts                     |
| `@Email`                           | Valid email format                                         |
| `@Pattern(regexp = "...")`         | Matches a regex                                            |
| `@Past` / `@PastOrPresent`         | Date in the past (or today)                                |
| `@Future` / `@FutureOrPresent`     | Date in the future (or today)                              |
| `@AssertTrue` / `@AssertFalse`     | Field must be exactly `true` / `false`                     |

## Nested Object Validation

To validate fields inside a nested object, place `@Valid` on the nested field so validation cascades into it. Without it, the inner constraints are skipped.

```java
public class OrderDto {

    @NotBlank
    private String orderId;

    @Valid                          // cascades validation into AddressDto
    private AddressDto shippingAddress;
}
```

## Handling Validation Errors

Handle `MethodArgumentNotValidException` centrally in a `@RestControllerAdvice`, walking the binding result for per-field messages. The full advice pattern (with `ProblemDetail`) is in [2.7. Exception Handling](<2.7. Exception Handling.md>); a minimal handler:

```java
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<Map<String, String>> handleValidation(MethodArgumentNotValidException ex) {
    Map<String, String> errors = new HashMap<>();
    ex.getBindingResult().getFieldErrors().forEach(err ->
        errors.put(err.getField(), err.getDefaultMessage()));
    return ResponseEntity.badRequest().body(errors);
}
```

```json
{ "email": "Invalid email format", "age": "Age must be positive" }
```

## Q&A

1. **What does `@Valid` do on a controller parameter?**
Triggers Bean Validation on the incoming object before the controller body runs. If a constraint fails, Spring throws `MethodArgumentNotValidException`, handled centrally in a `@RestControllerAdvice`.

2. **`@NotNull` vs `@NotEmpty` vs `@NotBlank`?**
- **`@NotNull`**: only rejects null; `""` passes.
- **`@NotEmpty`**: not null and size > 0 (String, Collection, Array).
- **`@NotBlank`**: not null and trimmed length > 0 (String only); rejects `"   "`.

3. **How do you validate a nested object inside a DTO?**
Annotate the nested field with `@Valid` so validation cascades; without it, the nested object's constraints are skipped.

4. **Why validate DTOs rather than entities?**
Validation is an API concern enforced at the boundary. Entity validation runs on save (too late, after work is done) and may need different rules than the API accepts.
