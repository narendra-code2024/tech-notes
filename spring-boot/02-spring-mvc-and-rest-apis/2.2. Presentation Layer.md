The presentation layer is the web boundary: it handles HTTP requests and responses, binds and validates input, and serializes results (JSON/XML for REST, views for MVC). In Spring Boot it is the `@RestController` layer, kept thin and delegating to services.

Use `@RestController` for REST endpoints; it serializes return values to JSON and skips view resolution. See [2.1. Spring MVC Architecture](<2.1. Spring MVC Architecture.md>) for the controller-vs-REST distinction.

## Request Mapping

`@RequestMapping` is the general-purpose mapping annotation; it matches on URL path, HTTP method, params, headers, or media types. Prefer the method-specific shortcuts (`@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`, `@PatchMapping`) for clarity:

```java
@RequestMapping(value = "/products", method = RequestMethod.GET)   // verbose
@GetMapping("/products")                                           // preferred
```

The `consumes` / `produces` attributes constrain accepted/returned media types, e.g. `@PostMapping(value = "/products", consumes = APPLICATION_JSON_VALUE)`. Defaults are sane (JSON via Jackson), so most APIs omit them; they matter for strict contracts, multipart uploads, and clean `415 Unsupported Media Type` rejection.

## Binding Request Data

Spring resolves controller method parameters from the request via binding annotations, resolved by the `HandlerAdapter` before the method runs:

```java
@RestController
@RequestMapping("/api/employees")
public class EmployeeController {

    private final EmployeeService service;

    public EmployeeController(EmployeeService service) {
        this.service = service;
    }

    @PostMapping("/{deptId}")
    public EmployeeDto create(@PathVariable Long deptId,        // path identity
                              @RequestBody EmployeeDto dto) {    // JSON body via Jackson
        return service.create(deptId, dto);
    }

    @GetMapping
    public List<EmployeeDto> list(@RequestParam(defaultValue = "true") boolean active,  // query string
                                  @RequestHeader("X-Tenant-ID") String tenant,          // HTTP header
                                  @CookieValue(value = "theme", defaultValue = "light") String theme) {
        return service.find(active, tenant);
    }
}
```

| Annotation       | Binds from       | Typical use                              |
| ---------------- | ---------------- | ---------------------------------------- |
| `@PathVariable`  | URL path segment | Resource identity (`/employees/{id}`)    |
| `@RequestParam`  | Query string     | Filtering, sorting, pagination, search   |
| `@RequestBody`   | HTTP body (JSON) | POST/PUT/PATCH payloads                  |
| `@RequestHeader` | HTTP header      | Tenant/correlation IDs, locale           |
| `@CookieValue`   | Cookie           | Theme, session-adjacent flags            |

> **Note**:
> - `@RequestParam`, `@RequestHeader`, and `@CookieValue` are required by default; use `required = false` or `defaultValue` to make them optional. `@PathVariable` is always mandatory (part of the URL).
> - `@RequestBody` is required by default; a missing body throws `HttpMessageNotReadableException`. Returning a raw DTO yields `200 OK`; for `201 Created` + `Location`, return `ResponseEntity`, see [2.5. ResponseEntity](<2.5. ResponseEntity.md>).

## DTO vs Entity

Do not expose JPA entities directly in your API:

- Entities are coupled to the database schema, so any schema change breaks the API contract.
- They leak internal fields (audit columns, relationships, lazy proxies) onto the wire.
- Lazy associations can trigger unintended queries during serialization.

Use DTOs for API request/response shapes: a clean, versionable contract decoupled from the schema, with explicit control over what is exposed and accepted. Input DTOs also carry validation annotations (see [2.6. Request Validation](<2.6. Request Validation.md>)).

```java
public class EmployeeDto {
    private Long id;
    private String name;
    private String department;
    // getters, setters, constructors; or Lombok @Data on DTOs
}
```

Map between DTO and entity in the service layer (manually, via MapStruct, or a dedicated mapper).

## Q&A

1. **`@RequestParam` vs `@PathVariable`, when to use which?**
`@PathVariable` identifies a specific resource (`/employees/{id}`) and is always mandatory. `@RequestParam` carries optional/filter data in the query string (`/employees?dept=sales&active=true`).

2. **What happens if the body is missing on a `@RequestBody` parameter?**
Spring throws `HttpMessageNotReadableException`; `@RequestBody` is required by default. Set `required = false` to allow a null body.

3. **Why use DTOs instead of exposing entities?**
Entities are coupled to the DB schema, so exposing them leaks internal structure, breaks on schema changes, and risks lazy-proxy serialization errors. DTOs give a stable, intentional contract.
