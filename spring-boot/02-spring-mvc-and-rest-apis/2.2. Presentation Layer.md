## Presentation Layer

The presentation layer is responsible for:

- Handling HTTP requests and responses
- Binding and validating input
- Returning responses (JSON/XML for REST, views for MVC)

A typical Spring Boot web project layout:

| Package | Responsibility |
|---|---|
| `controller` | Request/response handling (presentation layer) |
| `service` | Business logic |
| `repository` | Database access (JPA/Hibernate) |
| `dto` | Data Transfer Objects — API request/response shapes |
| `entity` | JPA entities — database table mappings |
| `config` | Security, Swagger, and other cross-cutting config |

---

## `@Controller` vs `@RestController`

- **`@Controller`**: used for MVC with views; methods return a view name (`String`) or `ModelAndView`. Add `@ResponseBody` on individual methods to return data directly.
- **`@RestController`**: meta-annotation combining `@Controller` + `@ResponseBody`. Return values are automatically serialized to JSON/XML by `HttpMessageConverter`; `ViewResolver` is not involved. The standard choice for REST APIs.

## Request Mapping Annotations

`@RequestMapping` is the general-purpose annotation — it can match by URL path, HTTP method, request parameters, headers, or media types (`consumes`/`produces`):

```java
@RequestMapping(value = "/products", method = RequestMethod.GET)
```

Prefer the **HTTP-method-specific shortcuts** for clarity: `@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`, `@PatchMapping`.

Optional attributes `consumes` / `produces` constrain accepted/returned media types — e.g., `@PostMapping(value = "/products", consumes = APPLICATION_JSON_VALUE)`. Defaults are sane (JSON via Jackson), so most APIs skip them. Useful for strict contracts, multipart uploads, and clean **415 Unsupported Media Type** rejection.

```java
@RestController
@RequestMapping("/api/products")
public class ProductController {

    private final ProductService productService;

    public ProductController(ProductService productService) {
        this.productService = productService;
    }

    @GetMapping
    public List<ProductDto> getProducts() {
        return productService.getAll();
    }
}
```

## Binding Request Data

Spring resolves controller method parameters from the HTTP request using these annotations:

### `@RequestParam` — query parameters

For URLs like `/employees?id=123`:

```java
@GetMapping("/employees")
public EmployeeDto getEmployee(@RequestParam Long id) {
    return employeeService.findById(id);
}
```

- **Required by default**: use `@RequestParam(required = false)` to make optional, or supply a `defaultValue`
- Typical for filtering, sorting, pagination, search

### `@PathVariable` — path parameters

For URLs like `/employees/123`:

```java
@GetMapping("/employees/{id}")
public EmployeeDto getEmployee(@PathVariable Long id) {
    return employeeService.findById(id);
}
```

- Identifies a specific resource
- Always mandatory — it's part of the URL
- The RESTful choice for resource identification

### `@RequestBody` — request body

Binds the HTTP body to a Java object via `HttpMessageConverter` (Jackson deserializes JSON → Java):

```java
@PostMapping("/employees")
public EmployeeDto createEmployee(@RequestBody EmployeeDto dto) {
    return employeeService.create(dto);
}
```

- Used with POST/PUT/PATCH when the client sends a structured payload
- **Required by default**: missing body throws `HttpMessageNotReadableException`

> Returning a raw DTO from a `POST` sends `200 OK`. For RESTful `201 Created` + `Location` header, return `ResponseEntity` — see [2.5 ResponseEntity](<2.5. ResponseEntity.md>).

### `@RequestHeader` — HTTP headers

Bind any request header to a parameter:

```java
@GetMapping("/products")
public List<ProductDto> getProducts(
    @RequestHeader("X-Tenant-ID") String tenantId,
    @RequestHeader(value = "Accept-Language", defaultValue = "en") String lang
) {
    return productService.findForTenant(tenantId, lang);
}
```

- **Required by default**: use `required = false` or `defaultValue` for optional headers
- Common uses: tenant IDs, correlation IDs, locale, custom client metadata

### `@CookieValue` — cookies

Bind a cookie value to a parameter:

```java
@GetMapping("/dashboard")
public DashboardDto getDashboard(
    @CookieValue(value = "theme", defaultValue = "light") String theme
) {
    return dashboardService.render(theme);
}
```

- Same `required` / `defaultValue` semantics as `@RequestHeader`

## DTO vs Entity

**Don't expose JPA entities directly in your API.**: Entities are tightly coupled to the database schema — any DB change breaks the API contract
- They may expose internal fields (audit columns, relationships, lazy proxies) you don't want on the wire
- Lazy-loaded associations can trigger unintended queries during serialization

**Use DTOs** for API request/response shapes:

- Clean, versionable API contract decoupled from the schema
- Explicit control over what's exposed and what's accepted
- Input DTOs can carry validation annotations (see [2.6 Request Validation](<2.6. Request Validation.md>))

```java
public class EmployeeDto {
    private Long id;
    private String name;
    private String department;
    // getters, setters, constructors — or use Lombok @Data
}
```

Map between DTO and entity in the service layer (manually, via MapStruct, or via a dedicated mapper component).

## Q&A

1. **Difference between `@Controller` and `@RestController`?**
   `@RestController` = `@Controller` + `@ResponseBody`. `@Controller` returns view names (resolved by `ViewResolver`); `@RestController` skips views and serializes return values to JSON/XML via `HttpMessageConverter`.

2. **`@RequestParam` vs `@PathVariable` — when to use which?**
   `@PathVariable` identifies a specific resource (`/employees/{id}`) and is always mandatory. `@RequestParam` carries optional/filter data in the query string (`/employees?dept=sales&active=true`).

3. **Why use DTOs instead of exposing entities?**
   Entities are coupled to the DB schema — exposing them leaks internal structure, causes serialization issues with lazy proxies, and forces API changes whenever the schema changes. DTOs give you a stable, intentional API contract.

4. **What happens if the request body is missing on a `@RequestBody` parameter?**
   Spring throws `HttpMessageNotReadableException` — `@RequestBody` is required by default. Set `@RequestBody(required = false)` to allow null bodies.

## Quick Reference

|Concept|Remember|
|---|---|
| `@RestController` | `@Controller` + `@ResponseBody`; returns JSON |
| `@GetMapping` / `@PostMapping` / ... | HTTP-method-specific shortcuts — preferred |
| `@RequestParam` | Query param; required by default |
| `@PathVariable` | URL path segment; always mandatory |
| `@RequestBody` | Binds request body to object via Jackson |
| `@RequestHeader` | Binds an HTTP header (e.g., `X-Tenant-ID`) |
| `@CookieValue` | Binds a cookie value |
| `consumes` / `produces` | Constrain accepted/returned media types |
| DTO | API-facing shape; decouples API from DB schema |
