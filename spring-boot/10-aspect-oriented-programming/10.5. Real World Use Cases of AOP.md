## Do You Really Need AOP?

Spring already provides built-in annotations for most cross-cutting concerns. Before writing a custom aspect, check if a built-in solution exists:

| Concern                    | Built-in Solution                                | Custom AOP Needed?           |
|----------------------------|--------------------------------------------------|------------------------------|
| Security                   | `@Secured`, `@PreAuthorize`                      | No (Usually not)             |
| Transactions               | `@Transactional`                                 | No                           |
| Caching                    | `@Cacheable`, `@CacheEvict`                      | No (Usually not)             |
| Validation                 | `@Valid`, `@Validated`                            | No                           |
| Auditing                   | Spring Data JPA `@CreatedBy`, `@LastModifiedDate` | No (Usually not)             |
| Logging / Monitoring       | No built-in                                      | Yes — best use case for AOP  |
| Performance profiling      | No built-in                                      | Yes                          |
| Custom exception handling  | `@RestControllerAdvice` handles most             | Warning (Sometimes)          |

> Most companies don't use custom AOP for security, caching, or transactions — the built-in annotations are well-optimized and easier to maintain. Custom **AOP** is most commonly used for logging, performance monitoring, custom auditing beyond JPA, and retry / rate-limiting decorators.

## Use Case Examples

These are **for understanding AOP**: in production, prefer the built-in alternatives mentioned above unless they don't meet your needs.

### Security Aspect

```java
@Aspect
@Component
public class SecurityAspect {

    @Before("execution(* com.app.service.*.*(..)) " +
            "&& @annotation(RequiresAdmin)")
    public void checkAdminAccess() {
        Authentication auth = SecurityContextHolder.getContext()
            .getAuthentication();
        if (!auth.getAuthorities().contains(
                new SimpleGrantedAuthority("ROLE_ADMIN"))) {
            throw new SecurityException(
                "Only admins can access this resource.");
        }
    }
}
```

> **In practice:** use `@PreAuthorize("hasRole('ADMIN')")` instead — simpler, well-tested, and integrated with Spring Security.

### Caching Aspect

```java
@Aspect
@Component
public class CachingAspect {

    private Map<String, Object> cache = new HashMap<>();

    @Around("execution(* com.app.service.*.*(..))")
    public Object cacheMethodResults(ProceedingJoinPoint joinPoint)
            throws Throwable {
        String key = joinPoint.getSignature().getName();
        if (cache.containsKey(key)) {
            return cache.get(key);
        }
        Object result = joinPoint.proceed();
        cache.put(key, result);
        return result;
    }
}
```

> **In practice:** use `@Cacheable("cacheName")` — handles cache keys, eviction, TTL, and integrates with Redis, Caffeine, etc.

### Auditing Aspect

```java
@Aspect
@Component
public class AuditAspect {

    private static final Logger log =
        LoggerFactory.getLogger(AuditAspect.class);

    @AfterReturning("execution(* com.app.service.*.get*(..))")
    public void auditAccess(JoinPoint joinPoint) {
        String user = SecurityContextHolder.getContext()
            .getAuthentication().getName();
        log.info("Data accessed by {} at {} — method: {}",
            user, LocalDateTime.now(),
            joinPoint.getSignature().getName());
    }
}
```

> **In practice:** Spring Data JPA's `@CreatedBy`, `@LastModifiedDate`, `@CreatedDate` handle most auditing. Use AOP only for custom audit logging beyond what JPA provides.

### Exception Handling Aspect

```java
@Aspect
@Component
@Slf4j
public class ExceptionHandlingAspect {

    @AfterThrowing(
        pointcut = "execution(* com.app.service.*.*(..))",
        throwing = "ex")
    public void handleException(JoinPoint joinPoint, Exception ex) {
        log.error("Exception in {}.{}: {}",
            joinPoint.getTarget().getClass().getSimpleName(),
            joinPoint.getSignature().getName(),
            ex.getMessage());
    }
}
```

> **In practice:** `@RestControllerAdvice` handles most exception scenarios. AOP-based exception handling is useful for logging exceptions in the service layer that don't propagate to controllers.

## Ordering Multiple Aspects with `@Order`

When two or more aspects target the same join point (e.g., a `LoggingAspect` and a `SecurityAspect` both advising service methods), the execution order is undefined unless you set it explicitly with `@Order` — lower value runs first on the way in, last on the way out.

```java
@Aspect
@Component
@Order(1)   // outermost — runs first before, last after
public class SecurityAspect { ... }

@Aspect
@Component
@Order(2)   // inner — runs after security on the way in
public class LoggingAspect { ... }
```

> Always order security-style aspects (which can throw and short-circuit) **outside** logging/metrics aspects — otherwise you log calls that were never authorised to run.

## Q&A

1. **What are the most common real-world uses of custom AOP?**
   Logging (method entry/exit, args, return values), performance monitoring (execution time, slow-method alerts), and custom metrics. Security/transactions/caching are typically handled by built-in annotations rather than custom aspects.

2. **Why prefer `@Transactional` / `@Cacheable` / `@PreAuthorize` over hand-written aspects?**
   They're well-tested, integrate with Spring's broader infrastructure (transaction managers, cache providers, security context), and handle edge cases like propagation, eviction, and SpEL expressions. Hand-rolled aspects rarely match the maturity.

3. **When is custom AOP the right tool?**
   When no built-in annotation covers the concern — cross-layer audit logging, custom metrics, retry/rate-limiting decorators, or specialized profiling. Also when you want to centralize a concern that's enabled per-method via a custom annotation (e.g., `@LogExecutionTime`).

4. **Are `@Transactional` and `@Cacheable` built on AOP?**
   Yes — both rely on Spring's proxy-based AOP. That's why the **self-invocation behavior** (see [10.4 Spring Proxy and Internal Working of AOP](<10.4. Spring Proxy and Internal Working of AOP.md>)) applies to them too.

5. **How do you control the order of multiple aspects?**
   Annotate each aspect with `@Order(n)` — lower values wrap outer (run first on entry, last on exit). Critical when one aspect can short-circuit (security throwing) and another logs — put security outside logging so unauthorised calls aren't logged as if they ran.

## Quick Reference

| Concern                | Prefer                            | Custom AOP When                       |
|------------------------|-----------------------------------|---------------------------------------|
| Transactions           | `@Transactional`                  | Almost never                          |
| Security               | `@PreAuthorize` / `@Secured`      | Custom roles outside Spring Security  |
| Caching                | `@Cacheable` / `@CacheEvict`      | Almost never                          |
| Auditing               | Spring Data JPA `@CreatedBy` etc. | Cross-layer audit logging             |
| Logging                | — (no built-in)                   | Always                                |
| Performance monitoring | — (no built-in)                   | Always                                |
| Exception handling     | `@RestControllerAdvice`           | Service-layer logging only            |
