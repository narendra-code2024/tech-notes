A database migration tool that manages schema changes through **versioned SQL scripts**. You write the SQL, Flyway executes it in order on startup, and tracks what's been applied — the same migrations run in dev, stage, and prod with no manual `psql` sessions or "did you remember to apply that ALTER on staging?" surprises.

---

## Why Not `ddl-auto=update`?

Hibernate's `ddl-auto=update` is convenient for development, but dangerous in production:

**Renaming a column:**

```java
// Before
private String name;

// After (renamed)
private String fullName;
```

```
What you expect: ALTER TABLE users RENAME COLUMN name TO full_name
What Hibernate does: ALTER TABLE users ADD COLUMN full_name (keeps old name column)
→ Old data stuck in name column, new column is empty
```

**Removing a field:**

```java
// Removed phoneNumber from entity
```

```
Hibernate may drop the column → production data lost
```

**Other risks**: No version history — you can't track what changed when
- No rollback — can't undo a bad migration
- No review — schema changes happen automatically, no one approves them
- Different behavior across databases — Hibernate generates different SQL for PostgreSQL vs MySQL

> `ddl-auto=update` is fine for dev. In production, always use `ddl-auto=validate` (checks schema matches entities) or `ddl-auto=none` (does nothing) — and manage schema changes with **Flyway**.

## Setup

**Dependencies:**

Modern Flyway (v10+) requires both the core engine and a database-specific module.

```xml
<!-- 1. The Core Engine -->
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>

<!-- 2. Database Specific Module (Required for Flyway 10+) -->
<!-- For PostgreSQL -->
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-database-postgresql</artifactId>
</dependency>

<!-- For MySQL -->
<!-- <dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-database-mysql</artifactId>
</dependency> -->
```

**Maven Plugin (Optional but Recommended):**

While Spring Boot runs migrations automatically on startup, the Maven plugin allows you to run manual tasks like `repair`, `info`, or `clean` from the CLI.

```xml
<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <configuration>
        <url>jdbc:postgresql://localhost:5432/mydb</url>
        <user>myuser</user>
        <password>mypassword</password>
    </configuration>
</plugin>
```
> Common commands: `mvn flyway:info` (check status), `mvn flyway:repair` (fix checksum failures).

**Configuration:**

```yaml
# Production Configuration
spring:
  jpa:
    hibernate:
      ddl-auto: validate
  flyway:
    enabled: true
    baseline-on-migrate: true
    locations: classpath:db/migration
    # Allow merging PRs with older version numbers
    out-of-order: true 
```

|Property|Purpose|Default|
|---|---|---|
|`ddl-auto: validate`|Hibernate only checks schema matches entities, never modifies|—|
|`flyway.enabled: true`|Enable Flyway|`true`|
|`flyway.baseline-on-migrate`|Adopt an existing DB that wasn't managed by Flyway|`false`|
|`flyway.out-of-order`|Allows running a lower-version migration even if a higher one exists|`false`|

## Migration Files

Place SQL files in `src/main/resources/db/migration`:

### 1. Versioned Migrations (V)
Used for schema changes (CREATE, ALTER, DROP) and reference data.

```
V1__create_users_table.sql
V2__add_phone_to_users.sql
```

**Naming convention:** `V{Version}__{Description}.sql` (Double underscore is mandatory).

### 2. Repeatable Migrations (R)
Used for objects that can be safely dropped and recreated, like **Views, Stored Procedures, and Triggers**.

```
R__create_user_summary_view.sql
```

- **No version number:** Just `R__` and description.
- **Re-runs on change:** Flyway calculates the checksum. If you edit the file, Flyway re-runs it on next startup.
- **Order:** Always runs *after* all pending versioned migrations are applied.

> **Rule:** Never use `R` for tables. Only for code-like DB objects where `CREATE OR REPLACE` logic works.

## Migration Examples

**V1 — initial schema:**

```sql
-- V1__initial_schema.sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    first_name VARCHAR(255) NOT NULL,
    last_name VARCHAR(255) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    password VARCHAR(255)
);

CREATE TABLE user_role (
    user_id BIGINT NOT NULL,
    role VARCHAR(50) NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id),
    PRIMARY KEY (user_id, role)
);

CREATE TABLE sessions (
    id BIGSERIAL PRIMARY KEY,
    refresh_token VARCHAR(500) NOT NULL UNIQUE,
    expires_at TIMESTAMP NOT NULL,
    provider VARCHAR(50) NOT NULL,
    user_id BIGINT NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

> Initial setup can group related tables in one file. After that, one change per file.

**V2 — add a column later:**

```sql
-- V2__add_phone_to_users.sql
ALTER TABLE users ADD COLUMN phone VARCHAR(20);
```

> Migrations should focus on **schema changes** only (CREATE, ALTER, DROP). For seeding data, see the section below.

## Handling Migration Failures

Flyway wraps each migration in a **Database Transaction** (on databases that support DDL transactions like PostgreSQL). 

**What happens when a script fails?**
1. The transaction rolls back (schema changes aren't applied).
2. Flyway adds an entry to `flyway_schema_history` with `success = false`.
3. The application **fails to start** until the issue is fixed.

**How to fix it:**
1. Fix the error in your SQL file.
2. Run `mvn flyway:repair`. This removes the failed migration entry from the history table.
3. Restart the app. Flyway will try running the fixed script again.

> **Warning:** On databases like MySQL that do *not* support DDL transactions, a partial failure can leave your DB in a "dirty" state where some tables were created and others weren't. You may need to manually clean up before running `repair`.

## Seeding Data

Not all data belongs in the same place.

|Data Type|Strategy|Location|Example|
|---|---|---|---|
|**Reference Data**|Flyway `V` scripts|`db/migration`|User Roles, Countries, Currencies|
|**Dev/Test Data**|`CommandLineRunner`|`@Profile("dev")`|Dummy Users, Mock Products|

**Why?** Reference data is part of the application logic. The app cannot function without `ROLE_ADMIN` in the database. Dummy users are only for development and would pollute production.

```java
@Component
@Profile("dev")
@RequiredArgsConstructor
public class DevDataSeeder implements CommandLineRunner {
    private final UserRepository userRepository;

    @Override
    public void run(String... args) {
        if (userRepository.count() == 0) {
            // Seed only for development
        }
    }
}
```

## Flyway vs Liquibase

|Feature|Flyway|Liquibase|
|---|---|---|
|**Format**|Plain SQL (mostly)|XML, YAML, JSON, SQL|
|**Philosophy**|Developer-centric (write SQL)|State-centric (define changes)|
|**Learning Curve**|Low (if you know SQL)|Moderate|
|**Rollbacks**|Manual (write a new script)|Automatic (state-based undo)|
|**Control**|Full control over SQL|Abstracted (can be limiting)|

## How Flyway Works

```
App starts
  │
  ▼
Flyway checks flyway_schema_history table in DB
  │
  ▼
Which migrations have already been applied?
  ├── V1 - applied
  ├── V2 - applied
  ├── V3 - applied
  ├── V4 - new
  ├── V5 - new
  └── R__view - check checksum
  │
  ▼
Runs V4, V5, then R__ (if changed) in order
  │
  ▼
Records changes in flyway_schema_history
  │
  ▼
App continues startup
```

## Migration Rules

**One migration per logical change.** Each file should do one thing.

**Never edit an already-applied migration.** Flyway checksums each file — if you change a migration that's already been applied, Flyway fails on startup with a checksum mismatch.

```
Need to change something already applied?
  → Write a NEW migration (V6__fix_whatever.sql)
  → Use 'mvn flyway:repair' only if the original script failed
```

**Roll back forward.** Flyway Community (free) **does not support automatic rollback**. If a migration goes wrong, write a new forward migration to undo it.

## Q&A

1. **Why not just use `ddl-auto=update` in production?** It silently produces incorrect DDL (renames become add-column, deletes can drop data), has no version history, and no review step. 

2. **What is `flyway_schema_history`?** A table Flyway creates to track applied migrations (version, checksum, success/fail status).

3. **How do you handle a failed migration?** Fix the SQL script and run `mvn flyway:repair` to clear the failed record from the history table, allowing a retry on the next startup.

4. **What are Repeatable Migrations (R__)?** Scripts without a version that run whenever their checksum changes. Ideal for Views, Procedures, and Triggers. They always run after all Versioned (V) migrations.

5. **What does `out-of-order=true` do?** Allows Flyway to apply a lower-version migration (e.g., V5) even if a higher version (e.g., V6) has already been applied. Critical for teams merging PRs at different times.

6. **Flyway vs Liquibase?** Flyway is simple and SQL-based. Liquibase is state-based (XML/YAML), supports automated rollbacks, and is better for complex multi-database schema synchronization.

## Quick Reference

|Concept|Remember|
|---|---|
| Prod `ddl-auto` | `validate` — never `update` |
| Dependencies | `flyway-core` + `flyway-database-<db>` |
| Filename pattern | `V1__init.sql` (Versioned), `R__view.sql` (Repeatable) |
| Manual Fixes | `mvn flyway:repair` (after fixing SQL) |
| Multi-Dev | `spring.flyway.out-of-order=true` |
| Reference Data | Use Flyway `V` scripts |
| Rollback | Write a new forward `V` migration |
