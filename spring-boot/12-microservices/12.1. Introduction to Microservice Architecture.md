## Monolith vs Microservices

- **Monolith**: everything in **one application**: one codebase, one deployment, one database.
- **Microservices**: application split into **small, independent services**, each with its own codebase, deployment, and database.

## Characteristics of Microservices

- **Small and focused**: each service does one thing well
- **Independently deployable**: deploy payment service without touching inventory
- **Own database**: each service manages its own data (decentralized)
- **Organized around business capabilities**: payment team owns payment service
- **Lightweight communication**: services talk via HTTP/REST or messaging

---

## Why Use Microservices?

|Benefit|How|
|---|---|
|**Independent scaling**|Scale only the payment service during high traffic, not the entire app|
|**Tech flexibility**|Payment in Java, notification in Python — each service picks what's best|
|**Faster development**|Small teams work on different services simultaneously|
|**Fault isolation**|Payment service crashes → orders and inventory still work|
|**Independent deployment**|Update one service without redeploying everything|
|**Team autonomy**|Teams own their service end-to-end without affecting others|

---

## Challenges of Microservices

|Challenge|What It Means|
|---|---|
|**Increased complexity**|Managing multiple services, deployments, databases instead of one|
|**Network reliability**|Services communicate over network — latency, failures, timeouts|
|**Distributed transactions**|A single business operation spans multiple services — hard to keep consistent|
|**Communication overhead**|Need to choose and manage protocols (REST, gRPC, messaging)|
|**Monitoring and logging**|Logs are spread across services — need centralized monitoring (ELK, Grafana)|
|**Load balancing**|Need to distribute traffic across multiple instances of a service|

---

## Monolith vs Microservices — When to Use

|                            | Monolith              | Microservices                |
|----------------------------|-----------------------|------------------------------|
| Team size                  | Small team            | Multiple teams               |
| App complexity             | Simple/medium         | Large, complex               |
| Scaling needs              | Scale entire app      | Scale individual services    |
| Deployment                 | One deployment        | Many independent deployments |
| Database                   | One shared DB         | DB per service               |
| Startup phase              | Yes (Start here)      | Migrate when needed          |
| Development speed (early)  | Faster                | Slower (more setup)          |
| Development speed (later)  | Slower (tangled code) | Faster (independent teams)   |

> **Start with a monolith.** Migrate to microservices when you have clear scaling needs, multiple teams, or features that need independent deployment. Most apps don't need microservices from day one.

## Production Considerations

- **Centralized observability**: distributed tracing (Zipkin / Jaeger / OpenTelemetry) with correlation IDs flowing through every hop is non-optional. Without it, debugging cross-service failures is impossible.
- **Centralized logging**: ship structured JSON logs from every service to ELK / Loki / Datadog so you can query across services by correlation ID.
- **Eventual consistency, not ACID**: distributed transactions across services don't work well; design with **Saga** patterns or outbox + messaging instead of trying to coordinate two-phase commits.
- **Service mesh or library**: once you have many services, cross-cutting concerns (mTLS, retries, circuit breaking, traffic shifting) live either in a sidecar mesh (Istio, Linkerd) or in shared libraries (Spring Cloud + Resilience4J — see [12.6 Circuit Breaker, Retry and Rate Limiter with Resilience4J](<12.6. Circuit Breaker, Retry and Rate Limiter with Resilience4J.md>)).
- **Versioning the contract**: `/api/v1/...` paths plus tolerant JSON deserialization (ignore unknown fields) so producers and consumers can roll out independently.
- **Secrets and config**: never bake into images; use a centralized config server ([13.3 Centralized Configuration Server](<../13-advanced-microservice-concepts/13.3. Centralized Configuration Server.md>)) plus a secret manager (Vault, AWS Secrets Manager, k8s Secrets).

> **Production warning:** "microservices" without observability, automated deploys, and a clear ownership model is just a distributed monolith — slower to ship and harder to debug than the monolith you replaced. Get the platform basics in place before splitting.

## Q&A

1. **What is the difference between a monolith and microservices?** A monolith is a single deployable application with a shared database; microservices split the application into small, independently deployable services, each with its own database and codebase.
2. **When should you NOT use microservices?** For small teams, simple/medium apps, or early-stage products — the operational overhead (service discovery, distributed tracing, multiple DBs) usually outweighs the benefits until scaling or team-size pressures appear.
3. **What is "database per service" and why does it matter?** Each microservice owns its own database; no other service reads or writes it directly. This enforces loose coupling but means cross-service joins aren't possible — you fetch via APIs instead.
4. **How do microservices communicate?** Lightweight protocols — synchronous (HTTP/REST, gRPC, OpenFeign) or asynchronous (Kafka, RabbitMQ). Choice depends on whether the caller needs an immediate response.
5. **What are the main challenges of microservices?** Distributed transactions, network reliability, debugging across services (needs distributed tracing), operational complexity (many deployments, configs), and eventual consistency instead of ACID.

## Quick Reference

| Concept | Remember |
|---|---|
| Monolith | One codebase, one deployment, one DB — start here |
| Microservices | Many small services, each with its own DB and deployment |
| Database per service | No shared tables — cross-service access via APIs |
| Communication | Sync (REST, gRPC, OpenFeign) or async (Kafka, RabbitMQ) |
| Tradeoff | Independent scaling/deploys, at the cost of operational complexity |
| When to migrate | Multiple teams, scaling pressure, or features needing independent deploy |
