## The Problem — RestClient Is Verbose

In [12.3 Service Registration and Discovery with Eureka](<12.3. Service Registration and Discovery with Eureka.md>), we used `DiscoveryClient` + `RestClient` to call another service:

```java
// Inventory Service calling Order Service — too much boilerplate
ServiceInstance orderService = discoveryClient
    .getInstances("order-service")
    .getFirst();

URI uri = orderService.getUri();

return restClient.get()
    .uri(uri + "/orders/core/helloOrders")
    .retrieve()
    .body(String.class);
```

For every service call: get instance, get URI, build request, retrieve, deserialize. Repeated everywhere.

---

## The Solution — OpenFeign

A **declarative HTTP client** originally created by Netflix (like Eureka). You write an interface that looks like a controller, and Spring handles all the HTTP, service discovery, and load balancing behind the scenes.

```java
// Same call with OpenFeign — one line
ordersFeignClient.helloOrders();
```

> OpenFeign integrates with Eureka automatically — you refer to services by **name**, not IP:port.

**Who handles load balancing?**

|Traffic Type|Handled By|
|---|---|
|Client → Service|API Gateway (distributes across service instances)|
|Service → Service|OpenFeign + Spring Cloud LoadBalancer (distributes across target instances)|

---

## Setup

### Step 1: Add Dependency

Add to any service that needs to **call** another service:

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-openfeign</artifactId>
</dependency>
```

### Step 2: Enable Feign Clients

```java
@SpringBootApplication
@EnableFeignClients
public class InventoryServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(InventoryServiceApplication.class, args);
    }
}
```

## Creating a Feign Client

A Feign Client is an **interface** that mirrors the target service's API:

```java
@FeignClient(name = "order-service", path = "/orders")
public interface OrderFeignClient {

    @GetMapping("/core/helloOrders")
    String helloOrders();
}
```

|Part|What It Does|
|---|---|
|`@FeignClient`|Marks this interface as a Feign client|
|`name = "order-service"`|The `spring.application.name` of the target service (registered in Eureka)|
|`path = "/orders"`|Base path prefix for all methods in this client|
|`@GetMapping("/core/helloOrders")`|The endpoint path — same as the target controller's mapping|

> Feign uses the `name` to ask Eureka for the service address. No hardcoded URLs.

### Using It

```java
@Service
@RequiredArgsConstructor
public class ProductService {

    private final OrderFeignClient orderFeignClient;

    public String fetchFromOrderService() {
        return orderFeignClient.helloOrders(); // one line — that's it
    }
}
```

## Real Example — Order Service Calls Inventory Service

When a user creates an order, the Order Service needs to call the Inventory Service (Inventory service setup from [12.2 Setting Up the Order and Inventory Services](<12.2. Setting Up the Order and Inventory Services.md>)) to reduce stock.

### Feign Client (in Order Service)

```java
@FeignClient(name = "inventory-service", path = "/inventory")
public interface InventoryFeignClient {

    @PutMapping("/products/reduceStocks")
    Double reduceStocks(@RequestBody OrderRequestDto orderRequest);
}
```

> This mirrors the Inventory Service's controller exactly — same HTTP method, same path, same request body.

### Target Controller (in Inventory Service)

```java
@RestController
@RequestMapping("/products")
@RequiredArgsConstructor
public class ProductController {

    private final ProductService productService;

    @PutMapping("/reduceStocks")
    public ResponseEntity<Double> reduceStocks(
            @RequestBody OrderRequestDto orderRequest) {
        Double totalPrice = productService.reduceStocks(orderRequest);
        return ResponseEntity.ok(totalPrice);
    }
}
```

### Service Logic (in Inventory Service)

```java
@Service
@RequiredArgsConstructor
public class ProductService {

    private final ProductRepository productRepository;

    @Transactional
    public Double reduceStocks(OrderRequestDto orderRequest) {
        double totalPrice = 0;

        for (OrderItemRequestDto item : orderRequest.getItems()) {
            Product product = productRepository.findById(item.getProductId())
                .orElseThrow(() -> new RuntimeException(
                    "Product not found with id: " + item.getProductId()));

            if (product.getStock() < item.getQuantity()) {
                throw new RuntimeException(
                    "Insufficient stock for product: " + product.getTitle());
            }

            product.setStock(product.getStock() - item.getQuantity());
            totalPrice += item.getQuantity() * product.getPrice();
            productRepository.save(product);
        }

        return totalPrice;
    }
}
```

```java
// In Order Service — DTOs

@Data
public class OrderItemRequestDto {
    private Long productId;
    private Integer quantity;
}

@Data
public class OrderRequestDto {
    private List<OrderItemRequestDto> items;
}

@Data
public class OrderResponseDto {
    private Long id;
    private List<OrderItemRequestDto> items;
    private Double totalPrice;
    private OrderStatus status;
}
```

> Inventory Service needs its own copy of `OrderRequestDto` and `OrderItemRequestDto` with the **same field names**. `OrderResponseDto` is only in Order Service. See "DTOs Must Exist in Both Services" section below for details.

### Order Controller

```java
@PostMapping("/core/create-order")
public ResponseEntity<OrderResponseDto> createOrder(
        @RequestBody OrderRequestDto orderRequest) {
    return ResponseEntity.ok(orderService.createOrder(orderRequest));
}
```

### Order Service Uses the Feign Client

```java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final InventoryFeignClient inventoryFeignClient;
    private final OrderRepository orderRepository;
    private final ModelMapper modelMapper;

    @Transactional
    public OrderResponseDto createOrder(OrderRequestDto orderRequest) {
        // 1. Call Inventory Service to reduce stock and get total price
        Double totalPrice = inventoryFeignClient
            .reduceStocks(orderRequest);

        // 2. Create the order
        Orders order = modelMapper.map(orderRequest, Orders.class);
        order.setTotalPrice(totalPrice);
        order.setOrderStatus(OrderStatus.CONFIRMED);

        // Set parent reference for each order item
        for (OrderItem item : order.getItems()) {
            item.setOrder(order);
        }

        // 3. Save and return
        Orders savedOrder = orderRepository.save(order);
        return modelMapper.map(savedOrder, OrderResponseDto.class);
    }
}
```

```
Client: POST /api/v1/orders/core/create-order
       │
       ▼
API Gateway (port 8080)
  → strips /api/v1, forwards to Order Service
       │
       ▼
Order Service (port 8081)
  → receives order request
  → calls inventoryFeignClient.reduceStocks(orderRequestDto)
       │
       ▼
Feign Client asks Eureka: "Where is inventory-service?"
  → Eureka returns: 10.4.5.6:8082
       │
       ▼
Inventory Service (port 8082)
  → reduces stock for each product
  → returns total price
       │
       ▼
Order Service
  → creates order with total price
  → saves to order DB
  → returns order response to client
```

## DTOs Must Exist in Both Services

Each microservice is a **separate application** with its own codebase — they can't share classes directly. So DTOs used in Feign communication must be created in both services with matching field names:

```
order-service/dto/
  ├── OrderRequestDto.java           ← client sends (items only)
  ├── OrderResponseDto.java          ← client receives (only in Order Service)
  └── OrderItemRequestDto.java       ← has productId, quantity

inventory-service/dto/
  ├── OrderRequestDto.java           ← same field names as Order Service
  └── OrderItemRequestDto.java       ← same field names as Order Service
```

> Field names **must match** for JSON serialization to work. If they differ, use `@JsonProperty` to map them. The Inventory Service DTO can have fewer fields — extra fields from Order Service are simply ignored during deserialization.

> Some teams create a **shared Maven module** (common library) to avoid duplicating DTOs. But most teams keep services fully independent with their own copies — this is the standard microservices approach.

## Feign Client Structure

Create a `clients` package in any service that calls other services:

```
order-service/src/main/java/com/example/ecommerce/order_service/
  ├── clients/
  │     └── InventoryFeignClient.java
  ├── controller/
  ├── service/
  ├── entity/
  └── dto/
```

> One Feign Client interface per target service. If Order Service calls both Inventory and Payment, create `InventoryFeignClient` and `PaymentFeignClient`.

## Production Considerations

**Set realistic timeouts.** The Feign defaults (10s connect, 60s read on the underlying client) are too generous — a slow downstream will block caller threads. Configure per-client:

```yaml
spring:
  cloud:
    openfeign:
      client:
        config:
          inventory-service:
            connect-timeout: 2000      # ms — fail fast if can't reach
            read-timeout: 5000         # ms — cap how long we wait for response
          default:
            connect-timeout: 2000
            read-timeout: 5000
```

**Propagate auth between services.** Service-to-service calls shouldn't be unauthenticated. Forward the caller's JWT via a `RequestInterceptor`:

```java
@Configuration
public class FeignAuthConfig {

    @Bean
    public RequestInterceptor authForwardingInterceptor() {
        return template -> {
            ServletRequestAttributes attrs =
                (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
            if (attrs != null) {
                String authHeader = attrs.getRequest().getHeader("Authorization");
                if (authHeader != null) {
                    template.header("Authorization", authHeader);
                }
            }
        };
    }
}
```

> Downstream services validate the JWT the same way the gateway does — every hop is authenticated.

**Errors aren't wrapped automatically.** A 4xx/5xx from the downstream service surfaces as `FeignException`. Handle it in your `@RestControllerAdvice` (see [2.7 Exception Handling](<../02-spring-mvc-and-rest-apis/2.7. Exception Handling.md>)) so the caller gets a clean error response instead of a stacktrace.

**Idempotency for retries.** If you combine Feign with retry (next file), only retry idempotent calls — `GET`, `PUT`, `DELETE`. For `POST` (e.g., create-order), require an `Idempotency-Key` header so the downstream can dedupe.

**Sync vs async.** Feign is synchronous — the caller blocks. For fire-and-forget or eventual consistency, prefer messaging (Kafka, covered in 14).

## RestClient vs OpenFeign

||RestClient|OpenFeign|
|---|---|---|
|Style|Programmatic — build request manually|Declarative — write an interface|
|Service discovery|Manual (`DiscoveryClient`)|Automatic (uses Eureka by name)|
|Load balancing|Manual|Automatic|
|Boilerplate|High (URI, headers, retrieve, deserialize)|Minimal (one method call)|
|Feels like|HTTP client code|Calling a local method|
|Use when|External third-party APIs|Service-to-service within your microservices|

## Q&A

1. **What is OpenFeign?** A declarative HTTP client — you define an interface annotated with `@FeignClient` and Spring generates the implementation at runtime. Removes boilerplate around `RestClient`/`RestTemplate` for service-to-service calls.
2. **What does `@EnableFeignClients` do?** Triggers a scan for `@FeignClient` interfaces and registers them as beans (proxy implementations) in the application context, so they can be `@Autowired`.
3. **How does Feign know which host to call?** The `name` attribute on `@FeignClient` matches a `spring.application.name` registered in Eureka. Spring Cloud LoadBalancer resolves `name` to a live instance and load-balances across them.
4. **Why must DTOs be duplicated across services?** Each microservice is its own deployable — they don't share a JAR. JSON serialization works as long as field names match; teams either duplicate or extract a shared `-common` Maven module (with the trade-off of a coupling point).
5. **What's the difference between an `@FeignClient` and a `@RestController`?** They look almost identical (same HTTP-mapping annotations) but mean opposite things: `@RestController` *implements* an endpoint; `@FeignClient` *consumes* one. The annotations on a Feign interface describe the remote endpoint's shape.
6. **Should you retry every Feign call?** No — only idempotent ones (`GET`, `PUT`, `DELETE`). Retrying a `POST` like create-order can charge a user twice. For non-idempotent calls, require an `Idempotency-Key` header so the downstream can dedupe, or skip retry entirely.

## Quick Reference

| Concept | Remember |
|---|---|
| `@FeignClient(name=...)` | `name` matches a `spring.application.name` in Eureka |
| `@EnableFeignClients` | Add on the main app class to scan for Feign interfaces |
| Discovery + LB | Automatic — Eureka + Spring Cloud LoadBalancer under the hood |
| DTO sharing | Each service has its own copy with matching field names; or extract a `-common` module |
| Default timeouts | Tighten — 2s connect / 5s read are realistic prod values |
| Auth propagation | Forward `Authorization` header via a `RequestInterceptor` |
| Errors | 4xx/5xx surface as `FeignException` — handle in `@RestControllerAdvice` |
| When to use | Service-to-service calls inside your microservices; use `RestClient` for external APIs |
