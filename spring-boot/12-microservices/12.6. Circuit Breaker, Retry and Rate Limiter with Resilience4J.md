## What is Resilience4J?

A lightweight library for making microservices **resilient to failures**. Instead of writing retry logic, rate limiting, or circuit breaking from scratch, you add annotations and config.

> Resilience4J replaced **Hystrix**, which is no longer maintained. Don't reach for Hystrix in new code.

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-circuitbreaker-resilience4j</artifactId>
</dependency>
```

---

## Three Key Patterns

|Pattern|Purpose|When to Use|
|---|---|---|
|**Retry**|Try again if a call fails|Temporary failures (network blip, service restarting)|
|**Rate Limiter**|Limit how many calls can be made in a time window|Prevent DOS attacks, protect downstream services|
|**Circuit Breaker**|Stop calling a service that's down|Prolonged failures, prevent cascade/avalanche|

---

## Retry

If a service call fails, **try again** a configured number of times before giving up.

```java
@Retry(name = "inventoryRetry", fallbackMethod = "createOrderFallback")
@Transactional
public OrderResponseDto createOrder(OrderRequestDto orderRequest) {
    log.info("Calling the create order method");
    Double totalPrice = inventoryFeignClient.reduceStocks(orderRequest);
    // ... create and save order
}

// Fallback — called after all retries fail
// Must have same return type + same params + Throwable
public OrderResponseDto createOrderFallback(
        OrderRequestDto orderRequest, Throwable throwable) {
    log.error("Fallback occurred due to: {}", throwable.getMessage());
    return new OrderResponseDto(); // return empty/default response
}
```

```
Call fails → wait → retry 1 → wait → retry 2 → wait → retry 3 → all failed → fallback
```

### Configuration

```yaml
resilience4j:
  retry:
    instances:
      inventoryRetry:
        maxRetryAttempts: 3
        waitDuration: 500ms    # wait between retries (use ms in production)
```

|Property|What It Does|Production Value|
|---|---|---|
|`maxRetryAttempts`|How many times to retry|3-5|
|`waitDuration`|Wait time between retries|100-500ms|

> **Only retry idempotent operations.** `GET`, `PUT`, `DELETE` are safe. Retrying a `POST` (e.g., create-order) can charge the user twice — require an `Idempotency-Key` header on the downstream so it can dedupe, or skip retry on `POST` entirely.

> **Use exponential backoff in production** (`enableExponentialBackoff: true`, `exponentialBackoffMultiplier: 2`) so retry storms don't pile on a recovering service.

## Rate Limiter

Limit how many calls can be made to a method within a time window. Excess calls go to fallback.

```java
@RateLimiter(name = "inventoryRateLimiter", fallbackMethod = "createOrderFallback")
@Transactional
public OrderResponseDto createOrder(OrderRequestDto orderRequest) {
    // ...
}
```

```
Window: 1 second, limit: 100 calls
  Call 1-100   → method executes normally
  Call 101+    → fallback ("rate limiter does not permit further calls")
  Next window  → counter resets, 100 calls allowed again
```

### Configuration

```yaml
resilience4j:
  rateLimiter:
    instances:
      inventoryRateLimiter:
        limitForPeriod: 100          # max calls in one window
        limitRefreshPeriod: 1s       # window duration
        timeoutDuration: 100ms       # wait for permission before failing
```

|Property|What It Does|Production Value|
|---|---|---|
|`limitForPeriod`|Max calls allowed per window|10-1000 (depends on your capacity)|
|`limitRefreshPeriod`|How often the limit resets|1s|
|`timeoutDuration`|How long to wait for permission before going to fallback|100-500ms|

## Circuit Breaker

If a service is **consistently failing**, stop calling it entirely. Prevents wasting resources on a dead service and avoids cascade failures.

**Opposite of retry**: retry says "try again", circuit breaker says "stop trying."

### Three States

```
CLOSED ──────────────────→ OPEN
(normal, calls go through)    (failures hit threshold)
  ▲                              │
  │                              │ wait duration (e.g., 20s)
  │                              ▼
  └─── success ──── HALF OPEN ───┘
                   (allow few test calls)
                       │
                       └── still failing → back to OPEN
```

**Closed**: everything works normally. Calls go through. Failures are counted.

**Open**: too many failures (threshold crossed). All calls go directly to fallback. No calls reach the actual method. Saves resources.

**Half Open**: after a wait period, allow a few test calls through. If they succeed → back to Closed. If they fail → back to Open.

```java
@CircuitBreaker(name = "inventoryCircuitBreaker",
    fallbackMethod = "createOrderFallback")
@Transactional
public OrderResponseDto createOrder(OrderRequestDto orderRequest) {
    // ...
}
```

### Configuration

```yaml
resilience4j:
  circuitbreaker:
    instances:
      inventoryCircuitBreaker:
        registerHealthIndicator: true
        slidingWindowSize: 10
        slidingWindowType: COUNT_BASED
        minimumNumberOfCalls: 10
        failureRateThreshold: 50
        waitDurationInOpenState: 10000    # 10 seconds in ms
        permittedNumberOfCallsInHalfOpenState: 3
        automaticTransitionFromOpenToHalfOpenEnabled: true
```

|Property|What It Does|
|---|---|
|`registerHealthIndicator`|Exposes circuit breaker state in actuator health endpoint|
|`slidingWindowSize`|Number of recent calls to track|
|`minimumNumberOfCalls`|Min calls before calculating failure rate|
|`failureRateThreshold`|Percentage of failures to trigger open state (e.g., 50%)|
|`waitDurationInOpenState`|How long to stay in open before moving to half-open|
|`permittedNumberOfCallsInHalfOpenState`|Test calls allowed in half-open|
|`automaticTransitionFromOpenToHalfOpenEnabled`|Auto-transition after wait duration|

### Example Walkthrough

```
10 calls made, 6 fail (60% > 50% threshold)
       │
       ▼
State: CLOSED → OPEN (all calls go to fallback)
       │
       ▼
Wait 10 seconds
       │
       ▼
State: OPEN → HALF OPEN (allow 3 test calls)
       │
       ├── 3 calls succeed → HALF OPEN → CLOSED (back to normal)
       └── 2+ calls fail   → HALF OPEN → OPEN (still broken, wait again)
```

## Retry vs Circuit Breaker

||Retry|Circuit Breaker|
|---|---|---|
|Philosophy|"Try again, it might work"|"Stop trying, it's broken"|
|When|Temporary/intermittent failures|Prolonged/consistent failures|
|Action on failure|Retry N times|Stop all calls, go to fallback|
|Protects|Current request (give it more chances)|System (prevent cascade/resource waste)|

> **Combining Retry and CircuitBreaker**: they work together, but the **order matters**. Resilience4J's recommended composition is `Retry(CircuitBreaker(call))` — retries are the **outer** layer, circuit breaker is the **inner** layer. This way:
>
> - Each retry attempt goes through the circuit breaker
> - If the circuit is **OPEN**, retries fail fast against the open breaker (no actual remote calls)
> - Avoids retry storms when the downstream is already known-failing
>
> With Spring Boot annotations, declare both on the method — Resilience4J applies the right order automatically:
>
> ```java
> @CircuitBreaker(name = "inventoryService", fallbackMethod = "fallback")
> @Retry(name = "inventoryService")
> public Product getProduct(Long id) { ... }
> ```
>
> Configure `retry.maxAttempts` modestly (3-5) and ensure circuit breaker `slidingWindowSize` and `failureRateThreshold` are tuned so the breaker opens before retries amplify load.

## Fallback Method Rules

The fallback method must have:

- **Same return type** as the original method
- **Same parameters** as the original method
- **Plus** a `Throwable` parameter at the end

```java
// Original method
public OrderResponseDto createOrder(OrderRequestDto orderRequest) { }

// Fallback — same signature + Throwable
public OrderResponseDto createOrderFallback(
        OrderRequestDto orderRequest, Throwable throwable) { }
```

## Shared Default Config

Instead of repeating config for every instance, define a default and override per instance:

```yaml
resilience4j:
  retry:
    configs:
      default:
        maxRetryAttempts: 3
        waitDuration: 500ms
    instances:
      inventoryRetry:
        baseConfig: default
        waitDuration: 200ms    # override just this property
```

> `baseConfig: default` inherits all default properties. You only override what's different for this instance.

## Monitoring with Actuator

Add actuator to see circuit breaker state in real-time:

```yaml
management:
  health:
    circuitbreakers:
      enabled: true
  endpoints:
    web:
      exposure:
        include: health,circuitbreakers,retries,ratelimiters
  endpoint:
    health:
      show-details: always
```

```
GET /actuator/circuitbreakers

{
  "inventoryCircuitBreaker": {
    "state": "CLOSED",
    "failureRate": -1,
    "bufferedCalls": 5,
    "failedCalls": 0
  }
}
```

> Monitor state changes: CLOSED → OPEN → HALF_OPEN → CLOSED. Useful for debugging in production. Pair with distributed tracing (correlation IDs flowing through Feign headers) — covered in 13.5.

## Q&A

1. **What problem does the Circuit Breaker pattern solve?** Cascading failure. If a downstream service is down, every caller blocks waiting for timeouts and exhausts threads/connections. The breaker "opens" after a failure threshold, short-circuits to fallback immediately, and protects upstream resources.
2. **Difference between Retry and Circuit Breaker?** Retry assumes the failure is transient and tries again N times. Circuit Breaker assumes the failure is sustained and stops calling entirely. They **can** be combined — Resilience4J applies the order `Retry(CircuitBreaker(call))` so retries fail fast against an open breaker instead of hammering a known-failing downstream.
3. **What are the three Circuit Breaker states?** **Closed** (calls pass through, failures counted), **Open** (calls short-circuit to fallback, no traffic reaches the dependency), **Half-Open** (a few probe calls allowed; success → Closed, failure → Open).
4. **How is the failure rate calculated?** Over a sliding window (count- or time-based) of recent calls — once `minimumNumberOfCalls` is reached, if `failureRateThreshold` (e.g., 50%) is exceeded, the breaker opens.
5. **What are the rules for a Resilience4J fallback method?** Same name as configured `fallbackMethod`, same return type as the original, same parameters, plus a `Throwable` (or specific exception type) appended at the end. Must live on the same bean.
6. **Where does the `@Retry` / `@CircuitBreaker` annotation actually take effect?** Spring AOP wraps the bean in a proxy — the annotations only fire on **external** calls into the bean. Calling from within the same class (self-invocation) bypasses the proxy and the annotations will not trigger.

## Quick Reference

| Concept | Remember |
|---|---|
| Retry | "Try again" for transient failures; idempotent ops only |
| Rate Limiter | Cap calls per window; protects downstream from spikes |
| Circuit Breaker | "Stop trying" for sustained failures; prevents cascade |
| States | CLOSED (normal) → OPEN (short-circuit) → HALF_OPEN (probe) |
| Composition order | `Retry(CircuitBreaker(call))` — retries fail fast against open breaker |
| Fallback signature | Same return type, same params, plus `Throwable` last |
| Backoff | Use exponential backoff in production to avoid retry storms |
| Proxy behavior | AOP-based — won't fire on self-invocation (same as `@Transactional`) |
| Monitoring | Actuator exposes `/actuator/circuitbreakers`, `/retries`, `/ratelimiters` |
