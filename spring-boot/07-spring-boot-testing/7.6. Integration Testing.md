Unit tests verify individual components in isolation. But many bugs only appear when components **interact**: configuration errors, missing data, incorrect queries, broken wiring. A unit test where the service is real but the repository is mocked passes happily even when the SQL query is wrong; production hits the real DB and fails. Integration tests catch these gaps and double as **regression tests** when you change code.

## `@WebMvcTest` ‚Äî Controller Slice (When You Need It)

Most thin controllers delegate to services with no logic worth isolating ‚Äî cover them via the end-to-end integration tests below. **Reach for `@WebMvcTest` only when controllers do meaningful work**: custom `HandlerMethodArgumentResolver`s, complex `@Valid` validation, security-rule overrides, exception-handler advice tied to the controller, or content-negotiation logic.

`@WebMvcTest` loads only the web slice (controllers, filters, `MockMvc`) ‚Äî services and repositories are NOT loaded, so dependencies must be supplied as `@MockBean`:

```java
@WebMvcTest(EmployeeController.class)
class EmployeeControllerSliceTest {

    @Autowired MockMvc mockMvc;

    @MockBean EmployeeService employeeService;  // service NOT loaded ‚Äî must mock

    @Test
    @WithMockUser(roles = "USER")
    void getEmployee_returnsOk() throws Exception {
        when(employeeService.findById(1L))
            .thenReturn(new EmployeeDto(1L, "Alice"));

        mockMvc.perform(get("/api/employees/1"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.name").value("Alice"));
    }

    @Test
    @WithMockUser(roles = "USER")
    void createEmployee_invalidPayload_returns400() throws Exception {
        mockMvc.perform(post("/api/employees")
                .with(csrf())
                .contentType(APPLICATION_JSON)
                .content("""
                    {"name":""}
                    """))  // empty name violates @NotBlank
            .andExpect(status().isBadRequest());
    }
}
```

**Why `MockMvc` over `WebTestClient` here:** `@WebMvcTest` doesn't start a server; `MockMvc` short-circuits the servlet container and dispatches directly to controllers. Faster than `WebTestClient` for slice tests, but doesn't exercise filters, content negotiation, or HTTP serialization the same way a real server does.

> **Trade-off vs end-to-end**: `@WebMvcTest` is fast and focused (web layer only) but skips the service+DB integration. Use it when isolating controller logic; use `@SpringBootTest` + `WebTestClient` (below) when you want to verify the whole flow.

## Test Class Naming

To distinguish unit tests from integration tests, suffix integration test classes with `IT`:

```
src/test/java/com/app/
  ‚îú‚îÄ‚îÄ repository/EmployeeRepositoryTest.java        ‚Üê unit test
  ‚îú‚îÄ‚îÄ service/EmployeeServiceTest.java              ‚Üê unit test
  ‚îî‚îÄ‚îÄ controller/EmployeeControllerIT.java          ‚Üê integration test
```

> The `IT` suffix is a common convention. Build tools like Maven Failsafe plugin use it to identify and run integration tests separately from unit tests.

## `WebTestClient`

`WebTestClient` provides a fluent API for making **real HTTP requests** to your application during tests. It's primarily associated with WebFlux but works with Spring MVC too.

**Dependency**: added with `test` scope so it's not included in the production build:

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-webflux</artifactId>
    <scope>test</scope>
</dependency>
```

**Setup:**

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@AutoConfigureWebTestClient(timeout = "10000")
@Import(TestContainersConfiguration.class)
class EmployeeControllerIT {

    @Autowired
    private WebTestClient webTestClient;

    @Autowired
    private EmployeeRepository employeeRepository;

    @BeforeEach
    void setUp() {
        employeeRepository.deleteAll();
    }
}
```

|Configuration|Purpose|
|---|---|
|`@SpringBootTest`|Loads full application context (all layers)|
|`webEnvironment = RANDOM_PORT`|Starts the actual server on a random port|
|`@AutoConfigureWebTestClient`|Auto-configures `WebTestClient` to point to the running server|
|`timeout = "10000"`|10 second timeout for requests|
|`@Import(TestContainersConfiguration.class)`|Uses real PostgreSQL via Docker (defined in [7.4 Persistence Layer Testing and TestContainers](<7.4. Persistence Layer Testing and TestContainers.md>))|
|`@BeforeEach deleteAll()`|Clean state before each test|

**`WebTestClient` methods:**

|Method|Purpose|
|---|---|
|`.get()`, `.post()`, `.put()`, `.delete()`|HTTP method|
|`.uri("/api/employees")`|Endpoint path|
|`.bodyValue(request)`|Request body|
|`.exchange()`|Execute the request|
|`.expectStatus()`|Assert HTTP status|
|`.expectBody()`|Assert response body|
|`.expectHeader()`|Assert response headers|
|`.jsonPath("$.field")`|Access specific field in JSON response|

> **Pick one HTTP test client and stick.** This handbook uses `WebTestClient` for integration tests because it makes real HTTP calls against `RANDOM_PORT`. `MockMvc` is a fine alternative for `@WebMvcTest` slices but short-circuits the servlet container ‚Äî don't mix both styles in the same project.

## Test Examples

**Create employee ‚Äî success:**

```java
@Test
@DisplayName("Should create employee and return 201")
void createEmployee_WhenValidInput_ThenReturn201() {
    // Given
    EmployeeDto request = EmployeeDto.builder()
        .firstName("Jane")
        .lastName("Doe")
        .email("jane@example.com")
        .build();

    // When / Then
    webTestClient.post()
        .uri("/api/employees")
        .bodyValue(request)
        .exchange()
        .expectStatus().isCreated()
        .expectBody()
        .jsonPath("$.data.id").isNotEmpty()
        .jsonPath("$.data.firstName").isEqualTo("Jane")
        .jsonPath("$.data.email").isEqualTo("jane@example.com");
}
```

**Create employee ‚Äî duplicate email:**

```java
@Test
@DisplayName("Should return 400 when email already exists")
void createEmployee_WhenDuplicateEmail_ThenReturn400() {
    // Given ‚Äî save one employee first
    Employee existing = Employee.builder()
        .firstName("Jane")
        .lastName("Doe")
        .email("jane@example.com")
        .build();
    employeeRepository.save(existing);

    EmployeeDto request = EmployeeDto.builder()
        .firstName("John")
        .lastName("Smith")
        .email("jane@example.com") // same email
        .build();

    // When / Then ‚Äî assert against the `ApiError` (or `ProblemDetail`) shape,
    // not just status, so the body contract is part of the test
    webTestClient.post()
        .uri("/api/employees")
        .bodyValue(request)
        .exchange()
        .expectStatus().isBadRequest()
        .expectBody()
        .jsonPath("$.id").isNotEmpty()
        .jsonPath("$.message").value(msg ->
            assertThat((String) msg).contains("already exists"));
}
```

**Find employee by email ‚Äî success:**

```java
@Test
@DisplayName("Should return employee when email is valid")
void findByEmail_WhenEmailExists_ThenReturnEmployee() {
    // Given
    Employee employee = Employee.builder()
        .firstName("Jane")
        .lastName("Doe")
        .email("jane@example.com")
        .build();
    employeeRepository.save(employee);

    // When / Then
    webTestClient.get()
        .uri("/api/employees/email/jane@example.com")
        .exchange()
        .expectStatus().isOk()
        .expectBody()
        .jsonPath("$.data.firstName").isEqualTo("Jane")
        .jsonPath("$.data.email").isEqualTo("jane@example.com");
}
```

**Find employee by email ‚Äî not found:**

```java

        .uri("/api/employees/email/nonexistent@example.com")
```

## Q&A

1. **What does `@SpringBootTest` load?**
   The full application context ‚Äî every `@Component`, `@Service`, `@Repository`, `@Controller`, security filters, the datasource, and any auto-configured beans. Slowest slice but the most realistic.

2. **Why `webEnvironment = RANDOM_PORT`?**
   It starts the actual embedded server (Tomcat) on a random port so `WebTestClient` makes real HTTP calls. Random avoids port collisions when tests run in parallel or alongside a running app.

3. **Why suffix integration tests with `IT` instead of `Test`?**
   Maven's Surefire plugin runs `*Test` classes during the `test` phase; Failsafe runs `*IT` classes during `integration-test`. The split lets fast unit tests gate the build early and slower integration tests run later.

4. **Why is `WebTestClient` preferred over `MockMvc` for integration tests?**
   `WebTestClient` makes real HTTP calls against the running server (`RANDOM_PORT`), so it exercises the full stack including filters, serialization, and content negotiation. `MockMvc` short-circuits the servlet container ‚Äî fine for `@WebMvcTest` slices, less realistic end-to-end.

5. **Should you write `@WebMvcTest` controller tests?**
   Usually not for thin controllers that just delegate ‚Äî there's no logic worth isolating. Cover them via end-to-end `@SpringBootTest` + `WebTestClient` instead. Reach for `@WebMvcTest` only when controllers do meaningful work (custom argument resolvers, complex validation, security rules).

6. **How do you keep integration tests isolated from each other?**
   Reset state in `@BeforeEach` (e.g., `repository.deleteAll()`), or use `@Transactional` on the test class so each test rolls back. With TestContainers, the container persists across tests but the data is wiped between them. Note `@Transactional` rollback **doesn't** apply to a real running server (`RANDOM_PORT`) ‚Äî the controller commits in its own thread ‚Äî so for `WebTestClient` tests, prefer explicit `deleteAll()` cleanup.

7. **How should error responses be asserted in integration tests?**
   Don't stop at status code ‚Äî assert against the `ApiError` (or `ProblemDetail`) body shape your `@RestControllerAdvice` returns. That way the error contract is part of the test, not just an unverified side effect. See [2.7 Exception Handling](<../02-spring-mvc-and-rest-apis/2.7. Exception Handling.md>).
