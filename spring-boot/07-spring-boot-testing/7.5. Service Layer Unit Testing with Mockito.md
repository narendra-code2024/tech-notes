When unit testing a service, you don't want to use the real repository (that needs a DB) or real external APIs. **Mockito** creates fake (mock) objects that simulate the behavior of real dependencies â€” so you test **only the service logic** in isolation. Without it, your "unit test" hits the real DB or HTTP layer and turns into a slow, fragile integration test.

---

## Three Core Concepts

|Concept|What It Does|Example|
|---|---|---|
|**Mocking**|Creates fake objects that look like real ones|Fake `EmployeeRepository`|
|**Stubbing**|Defines what the mock returns when a method is called|"When `findById(1L)` is called, return this employee"|
|**Verification**|Checks that certain methods were actually called|"Verify `save()` was called exactly once"|

---

## Setting Up Mockito

```java
@ExtendWith(MockitoExtension.class)
class EmployeeServiceTest {

    @Mock
    private EmployeeRepository employeeRepository;

    @Spy
    private EmployeeMapper employeeMapper = new EmployeeMapperImpl();

    @InjectMocks
    private EmployeeServiceImpl employeeService;
}
```

|Annotation|What It Does|
|---|---|
|`@ExtendWith(MockitoExtension.class)`|Enables Mockito annotations in the test class|
|`@Mock`|Creates a fake object â€” all methods return null/default by default|
|`@Spy`|Wraps a **real object**: calls real methods unless explicitly stubbed|
|`@InjectMocks`|Creates the real service and injects `@Mock` and `@Spy` dependencies into it|

**`@Mock` vs `@Spy`:**

```java
@Mock
private EmployeeRepository repository;
// repository.findById(1L) â†’ returns null (fake, no real logic)

@Spy
private EmployeeMapper mapper = new EmployeeMapperImpl();
// mapper.toDto(employee) â†’ calls REAL mapping logic
// But you CAN override: when(mapper.toDto(any())).thenReturn(...)
```

Use `@Mock` for dependencies you want to **fully control** (repositories, external services). Use `@Spy` for dependencies whose **real logic you want to keep** (mappers, utility classes).

> **Prefer `@Mock` by default.** `@Spy` is a smell in most service tests â€” if a mapper has no logic worth keeping, mock it; if it does, prefer instantiating it directly (mappers are usually pure functions and don't need to be a dependency at all). Reach for `@Spy` only when you must wrap a real collaborator and override one method on it.

You can also create mocks programmatically with `Mockito.mock(EmployeeRepository.class)` instead of `@Mock` â€” both do the same thing, annotations are cleaner.

## `@MockBean` / `@SpyBean` â€” Spring-Context Mocks

`@Mock` and `@Spy` are pure Mockito â€” they don't touch Spring's `ApplicationContext`. For tests that load a Spring context (`@SpringBootTest`, `@WebMvcTest`, `@DataJpaTest`), you need the Spring-aware variants:

|Annotation|Where it lives|What it does|
|---|---|---|
|`@Mock`|Pure Mockito|Creates a mock â€” invisible to Spring|
|`@Spy`|Pure Mockito|Wraps a real object â€” invisible to Spring|
|`@MockBean`|Spring Boot Test|**Replaces** the real bean in `ApplicationContext` with a mock|
|`@SpyBean`|Spring Boot Test|**Wraps** the real bean in `ApplicationContext` with a spy|

```java
@SpringBootTest
class OrderServiceIntegrationTest {

    @Autowired
    private OrderService orderService;       // real, Spring-managed

    @MockBean
    private PaymentClient paymentClient;     // mock REPLACES the real bean in context

    @SpyBean
    private InventoryService inventoryService; // spy WRAPS the real bean

    @Test
    void placeOrder_chargesPaymentAndDecrementsInventory() {
        when(paymentClient.charge(any())).thenReturn(PaymentResult.success());

        orderService.placeOrder(new Order(...));

        verify(paymentClient).charge(any());
        verify(inventoryService).decrement(anyLong(), anyInt());
    }
}
```

**When to use which**: **`@Mock` / `@Spy`**: pure unit tests with `@ExtendWith(MockitoExtension.class)` + `@InjectMocks`. No Spring context, fastest.
- **`@MockBean` / `@SpyBean`**: when the test runs inside a Spring context (e.g., `@WebMvcTest`, `@SpringBootTest`) and the SUT pulls dependencies via Spring DI rather than direct injection. The mock has to BE a Spring bean for the wiring to find it.

> **Performance trade-off.** Each unique combination of `@MockBean` declarations creates a new Spring `ApplicationContext` (no caching across tests with different mock sets). Heavy use of `@MockBean` with varying mocks slows the suite â€” Spring rebuilds the context. For pure unit tests, prefer `@Mock` + `@InjectMocks`; reach for `@MockBean` only when the slice annotation requires it.

## Stubbing

**`when(...).thenReturn(...)`**: "When this method is called, return this value."

```java
Employee employee = new Employee(1L, "John", "john@example.com");

when(employeeRepository.findById(1L))
    .thenReturn(Optional.of(employee));

when(employeeRepository.findByEmail("john@example.com"))
    .thenReturn(Optional.of(employee));

when(employeeRepository.save(any(Employee.class)))
    .thenReturn(employee);
```

**`when(...).thenThrow(...)`**: "When this method is called, throw this exception."

```java
when(employeeRepository.findById(999L))
    .thenThrow(new ResourceNotFoundException("Employee not found"));
```

**Stubbing reference:**

|Method|Purpose|
|---|---|
|`when(mock.method()).thenReturn(value)`|Return a specific value|
|`when(mock.method()).thenThrow(exception)`|Throw an exception|
|`when(mock.method()).thenReturn(first, second)`|Return different values on consecutive calls|

## Verification

After executing your service method, verify that the mock's methods were called as expected.

```java
// Verify save was called exactly once
verify(employeeRepository).save(any(Employee.class));

// Verify findById was called with argument 1L
verify(employeeRepository).findById(1L);
```

**Verification modes:**

```java
// Called exactly 2 times
verify(employeeRepository, times(2)).findById(anyLong());

// Never called
verify(employeeRepository, never()).delete(any());

// Called at least once
verify(employeeRepository, atLeastOnce()).save(any());

// Called at least 3 times
verify(employeeRepository, atLeast(3)).findById(anyLong());

// Called at most 5 times
verify(employeeRepository, atMost(5)).findById(anyLong());

// This was the ONLY method called on the mock
verify(employeeRepository, only()).save(any());
```

|Mode|Checks|
|---|---|
|`times(n)`|Exactly n times|
|`never()`|Zero times|
|`atLeastOnce()`|At least 1 time|
|`atLeast(n)`|At least n times|
|`atMost(n)`|At most n times|
|`only()`|This method and no other method was called|

## Argument Captor

Sometimes you need to check **what arguments** were passed to a mock method â€” not just that it was called.

```java
@Captor
private ArgumentCaptor<Employee> employeeCaptor;

@Test
void shouldSaveEmployeeWithCorrectData() {
    EmployeeDto request = new EmployeeDto("John", "john@example.com");

    when(employeeRepository.save(any(Employee.class)))
        .thenReturn(new Employee(1L, "John", "john@example.com"));

    employeeService.createEmployee(request);

    // Capture what was passed to save()
    verify(employeeRepository).save(employeeCaptor.capture());

    Employee captured = employeeCaptor.getValue();
    assertThat(captured.getFirstName()).isEqualTo("John");
    assertThat(captured.getEmail()).isEqualTo("john@example.com");
}
```

> Useful when the service **transforms** data before saving â€” you want to verify the transformation was correct.

## Argument Matchers

Instead of exact values, use matchers for flexible stubbing and verification:

```java
// any() â€” matches any value
when(employeeRepository.save(any(Employee.class))).thenReturn(employee);

// anyLong(), anyString() â€” type-specific
when(employeeRepository.findById(anyLong())).thenReturn(Optional.of(employee));

// eq() â€” exact match (use when mixing matchers with exact values)
when(service.findByNameAndCity(eq("John"), anyString())).thenReturn(result);
```

> **Important:** If you use a matcher for one argument, you must use matchers for **all** arguments. Can't mix `eq("John")` with a raw `"Delhi"` â€” use `eq("Delhi")` instead.

## Full Example

```java
@ExtendWith(MockitoExtension.class)
class EmployeeServiceTest {

    @Mock
    private EmployeeRepository employeeRepository;

    @Spy
    private EmployeeMapper employeeMapper = new EmployeeMapperImpl();

    @InjectMocks
    private EmployeeServiceImpl employeeService;

    private Employee employee;

    @BeforeEach
    void setUp() {
        employee = Employee.builder()
            .id(1L)
            .firstName("John")
            .lastName("Doe")
            .email("john@example.com")
            .build();
    }

    @Test
    @DisplayName("Should return employee when email is valid")
    void shouldReturnEmployeeByEmail() {
        // Given
        when(employeeRepository.findByEmail("john@example.com"))
            .thenReturn(Optional.of(employee));

        // When
        EmployeeDto result = employeeService
            .getEmployeeByEmail("john@example.com");

        // Then
        assertThat(result).isNotNull();
        assertThat(result.getEmail()).isEqualTo("john@example.com");
        verify(employeeRepository).findByEmail("john@example.com");
    }

    @Test
    @DisplayName("Should throw exception when employee not found")
    void shouldThrowWhenEmployeeNotFound() {
        // Given
        when(employeeRepository.findById(999L))
            .thenReturn(Optional.empty());

        // When / Then
        assertThatThrownBy(() -> employeeService.getEmployeeById(999L))
            .isInstanceOf(ResourceNotFoundException.class)
            .hasMessageContaining("not found");

        verify(employeeRepository).findById(999L);
    }

    @Test
    @DisplayName("Should save employee with valid input")
    void shouldCreateEmployee() {
        // Given
        when(employeeRepository.save(any(Employee.class)))
            .thenReturn(employee);

        EmployeeDto request = new EmployeeDto("John", "Doe",
            "john@example.com");

        // When
        EmployeeDto result = employeeService.createEmployee(request);

        // Then
        assertThat(result).isNotNull();
        assertThat(result.getEmail()).isEqualTo("john@example.com");
        verify(employeeRepository, times(1)).save(any(Employee.class));
    }
}
```

> **Test naming convention:** drop the `test` prefix â€” `@Test` already marks it. Widely-followed pattern: `[method]_[scenario]_[expectedResult]` (e.g., `findByEmail_WhenEmailIsValid_ThenReturnEmployee`). Other common styles you'll see in the wild: `shouldReturnEmployee_WhenEmailIsValid`, `givenValidEmail_whenFind_thenReturnEmployee`. Pick one and stay consistent â€” the goal is a name that reads as a sentence in the CI log.

> **Mocking external HTTP calls**: for outbound `RestTemplate`/`WebClient` calls, prefer **WireMock** (records real HTTP fixtures, exercises serialization) over `@MockBean` on the client. `@MockBean` works for unit-style tests but skips the wire layer entirely; WireMock catches header/query-param/serialization bugs that mocks hide.

## Q&A

1. **`@Mock` vs `@Spy` â€” when to use which?**
   `@Mock` creates a fully fake object â€” every method returns null/default unless stubbed. `@Spy` wraps a real object â€” real methods run unless you explicitly override them. Use `@Mock` for dependencies you want full control over (repositories, external clients), `@Spy` when you want to keep real logic (mappers, utility classes) but selectively stub a method.

2. **What does `@InjectMocks` do?**
   It instantiates the class under test and injects `@Mock` and `@Spy` fields into it via constructor, setter, or field injection (in that priority order). The injected object is the real class â€” only its dependencies are mocks.

3. **Why use `ArgumentCaptor`?**
   To assert on **what** was passed to a mock, not just that it was called. Useful when the service transforms input before delegating (e.g., DTO â†’ entity mapping) â€” you capture the entity passed to `save()` and verify the transformation.

4. **What's the rule when mixing argument matchers with raw values?**
   You can't mix â€” if any argument uses a matcher (`any()`, `anyLong()`), all arguments must. Wrap raw values in `eq("value")` to make them matchers too.

5. **Why does Mockito not let you stub `final` or `static` methods by default?**
   It uses runtime proxies/subclasses which can't override `final`/`static`. Mockito-inline (the default since Mockito 5) supports them via bytecode manipulation, but it's slower and traditionally signaled a code smell â€” final/static methods are hard to test.

6. **What's the difference between `verify(mock).method()` and `when(mock.method()).thenReturn(...)`?**
   `when` configures behavior **before** the test acts (input). `verify` checks behavior **after** the test acts (output â€” was the method called?). Together they form the "stub-act-verify" pattern.

## Quick Reference

|Concept|Remember|
|---|---|
| `@Mock` | Fully fake â€” methods return null/default until stubbed |
| `@Spy` | Wraps real object â€” real methods run unless overridden |
| `@InjectMocks` | Builds real SUT and injects mocks/spies into it |
| `when(...).thenReturn(...)` | Stub return value |
| `when(...).thenThrow(...)` | Stub thrown exception |
| `verify(mock, times(n))` | Assert method called n times |
| `ArgumentCaptor` | Capture and assert on the argument passed to a mock |
| Matchers rule | If one arg uses a matcher, all must â€” wrap literals with `eq()` |
