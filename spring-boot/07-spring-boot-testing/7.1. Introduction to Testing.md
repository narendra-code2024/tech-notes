Testing isn't a one-time phase tacked on before deployment — in modern development, you write tests **alongside** code, and tests run automatically before every deploy. The payoff:

- **Identifies bugs early**: cheaper to fix during development than after deployment
- **Mitigates deployment risks**: tests catch breaking changes before they reach production
- **Increases productivity**: refactor with confidence, tests tell you if something broke
- **Builds trust**: developers and teams trust code that has tests backing it

---

## Test Writing Approaches

**TDD — Test-Driven Development.** Write tests **before** the code:

```
1. Write a failing test (Red)
       │
       ▼
2. Write minimum code to pass the test (Green)
       │
       ▼
3. Refactor the code for better design (Refactor)
       │
       └──→ repeat
```

> The "Red-Green-Refactor" cycle. You start with what the code **should do**, then make it work.

**BDD — Behavior-Driven Development.** An extension of TDD — tests are written in human-readable language describing **behavior**:

```
Given: a user with email "john@example.com" exists
When:  I call GET /api/users/1
Then:  I should receive the user's details with status 200
```

> BDD focuses on **what the system does** from the user's perspective, not how it works internally.

**Comparison:**

|Approach|How It Works|
|---|---|
|**TDD**|Write test first → then code|
|**BDD**|Define behavior first → write scenarios → then code|
|**Test-After Development**|Write code first → then add tests|
|**Simultaneous Development**|Write code and tests together|

> TDD and BDD produce the most reliable tests, but Test-After is the most common in practice — most teams write tests after implementing features.

## Q&A

1. **Why write tests at all?**
   Tests catch bugs early (cheaper to fix), guard against regressions during refactors, document expected behavior, and give teams confidence to deploy.

2. **TDD vs BDD vs Test-After — what's the difference?**
   **TDD** writes a failing test first, then just enough code to pass it (Red-Green-Refactor). **BDD** extends TDD with human-readable Given/When/Then scenarios that describe behavior from the user's perspective. **Test-After** writes tests after the code — most common in practice, least rigorous.

3. **What's the testing pyramid?**
   Many fast unit tests at the base, fewer integration tests in the middle, very few end-to-end tests at the top. Inverting this (lots of slow E2E, few unit tests) is an anti-pattern called the "ice-cream cone".

4. **Where does testing fit in the SDLC?**
   Not a separate phase at the end — tests are written alongside code and run automatically in CI before every deployment.
