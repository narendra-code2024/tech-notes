**Java Code Coverage**: a free library that measures how much of your code is covered by tests. It shows which lines, branches, and methods were executed during testing and which were missed.

**Why use it**: **Visibility into test coverage**: see exactly what percentage of your code is tested
- **Identify dead code**: find code that's never executed
- **Focus on critical areas**: uncovered parts highlight where to prioritize testing
- **Maintainability**: coverage reports help maintain quality standards over time

---

## Setup — Maven Plugin

Add the JaCoCo Maven plugin to your `pom.xml`:

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.jacoco</groupId>
            <artifactId>jacoco-maven-plugin</artifactId>
            <version>0.8.12</version>
            <executions>
                <execution>
                    <goals>
                        <goal>prepare-agent</goal>
                    </goals>
                </execution>
                <execution>
                    <id>report</id>
                    <phase>test</phase>
                    <goals>
                        <goal>report</goal>
                    </goals>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

> Running tests with JUnit automatically activates the JaCoCo agent — no extra configuration needed.

---

## Running and Generating Reports

**Option 1: From IDE.** Right-click test package → **Run All Tests with Coverage**. The IDE shows a coverage table (lines, branches, methods) and offers to export an HTML report.

**Option 2: From terminal (Maven).**

```bash
./mvnw clean package
```

This does everything:

```
clean     → clears target folder
compile   → compiles source code
test      → runs all tests
package   → builds the JAR
       │
       ▼
JaCoCo agent runs during test phase
       │
       ▼
Generates:
  target/jacoco.exec              ← binary coverage data
  target/site/jacoco/index.html   ← HTML report
```

> Open `target/site/jacoco/index.html` in a browser to view the report.

## Reading the Report

The HTML report shows coverage at multiple levels:

```
Package level → Class level → Method level → Line level
```

|Metric|What It Measures|
|---|---|
|**Line coverage**|Percentage of code lines executed|
|**Branch coverage**|Percentage of `if`/`else` branches taken|
|**Method coverage**|Percentage of methods called|
|**Class coverage**|Percentage of classes touched|

Color coding in the report:

- **Green**: fully covered
- **Yellow**: partially covered (e.g., only one branch of an `if`/`else`)
- **Red**: not covered at all

## What Coverage to Aim For

|Metric|Reasonable target|Why|
|---|---|---|
|Line coverage|80%+|Industry standard for service/repository code|
|Branch coverage|70%+|Stricter than line — forces you to test both sides of conditionals|
|Method coverage|90%+|Easy to hit; uncovered methods usually mean dead code|

You can fail the build below a threshold via the JaCoCo `check` goal — useful as a safety net in CI:

```xml
<execution>
    <id>jacoco-check</id>
    <goals><goal>check</goal></goals>
    <configuration>
        <rules>
            <rule>
                <element>BUNDLE</element>
                <limits>
                    <limit><counter>LINE</counter>
                        <value>COVEREDRATIO</value>
                        <minimum>0.80</minimum></limit>
                    <limit><counter>BRANCH</counter>
                        <value>COVEREDRATIO</value>
                        <minimum>0.70</minimum></limit>
                </limits>
            </rule>
        </rules>
    </configuration>
</execution>
```

> **Coverage is not a quality proxy.** 100% coverage with `assertThat(result).isNotNull()` everywhere catches nothing. A failing test you can read and fix is worth more than a passing test that just inflates the percentage. Use the threshold to flag regression, not to chase a number — exclude trivial getters, generated code, DTOs, and `Application.main` from the report.

## Q&A

1. **What does JaCoCo actually measure?**
   It instruments compiled bytecode at runtime and records which lines, branches, and methods executed during tests. It produces both a binary `.exec` file and an HTML report.

2. **Line coverage vs branch coverage — what's the difference?**
   Line coverage counts how many source lines ran. Branch coverage counts how many `if`/`else` (and ternary, switch) outcomes ran. A line with `if (a && b)` can hit 100% line coverage but only 50% branch coverage if you never test the false case.

3. **Is 100% coverage the goal?**
   No — it's diminishing returns. 100% coverage with weak assertions doesn't catch bugs. 80% with meaningful assertions on critical business logic is more valuable. Aim for 80%+ on services/repositories and skip trivial getters/DTOs.

4. **How does the JaCoCo agent attach during a Maven build?**
   The `prepare-agent` goal injects the agent JAR into Surefire's JVM args. When tests run, the agent records coverage to `target/jacoco.exec`. The `report` goal then converts that binary file into the HTML report.

5. **Why might JaCoCo coverage report show 0% even though tests passed?**
   Common causes: `prepare-agent` goal wasn't bound, Surefire's `argLine` was overridden and dropped the agent, or the build skipped the `test` phase. Check that `mvn test` runs and that `target/jacoco.exec` is non-empty.

|Concept|Remember|
|---|---|
| JaCoCo | Bytecode-instrumenting code coverage tool |
