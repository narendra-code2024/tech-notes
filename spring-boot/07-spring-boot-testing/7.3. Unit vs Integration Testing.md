## Unit Testing

Tests a single **unit** in isolation — usually one class or method. Dependencies are **mocked** (faked), so you're only testing the logic of that one unit.

|||
|---|---|
|**Focus**|Individual components — one method, one class|
|**Dependencies**|Mocked (no real DB, no real API calls)|
|**Speed**|Very fast (no Spring context, no DB)|
|**Tools**|JUnit, Mockito, AssertJ|
|**Example**|Testing a service method that performs business logic|

**Example**: service tested with a mocked repository, no Spring context, runs in milliseconds:

```java
@ExtendWith(MockitoExtension.class)
class EmployeeServiceTest {

    @Mock
    private EmployeeRepository employeeRepository;

    @InjectMocks
    private EmployeeServiceImpl employeeService;

    @Test
    @DisplayName("Should return employee by id")
    void shouldReturnEmployeeById() {
        Employee employee = new Employee(1L, "John Doe",
            "john@example.com");
        when(employeeRepository.findById(1L))
            .thenReturn(Optional.of(employee));

        EmployeeDto result = employeeService.getEmployeeById(1L);

        assertThat(result.getName()).isEqualTo("John Doe");
        assertThat(result.getEmail()).isEqualTo("john@example.com");
    }
}
```

## Integration Testing

Tests **multiple layers working together**: the real interaction between service, repository, database, controllers, etc.

|||
|---|---|
|**Focus**|Interaction between multiple components|
|**Dependencies**|Real (actual DB, actual Spring context)|
|**Speed**|Slower (loads Spring context, starts DB)|
|**Tools**|Spring Test, `@SpringBootTest`, `@DataJpaTest`|
|**Example**|Testing that a service correctly saves and retrieves from DB|

**Example**: full Spring context, real database, asserts the actual flow from service through repository to DB:

```java
@SpringBootTest
class EmployeeIntegrationTest {

    @Autowired
    private EmployeeService employeeService;

    @Autowired
    private EmployeeRepository employeeRepository;

    @Test
    @DisplayName("Should save and retrieve employee")
    void shouldSaveAndRetrieveEmployee() {
        EmployeeDto request = new EmployeeDto("Jane Doe",
            "jane@example.com");

        EmployeeDto saved = employeeService.createEmployee(request);

        Employee fromDb = employeeRepository.findById(saved.getId())
            .orElseThrow();

        assertThat(fromDb.getName()).isEqualTo("Jane Doe");
        assertThat(fromDb.getEmail()).isEqualTo("jane@example.com");
    }
}
```

## Unit vs Integration

||Unit Test|Integration Test|
|---|---|---|
|Scope|Single class/method|Multiple layers together|
|Dependencies|Mocked|Real|
|Database|No|Yes (H2, TestContainers)|
|Spring context|Not loaded|Loaded (`@SpringBootTest`)|
|Speed|Fast (ms)|Slow (seconds)|
|What it catches|Logic bugs in one unit|Wiring bugs, query bugs, config issues|
|When it fails|"This method has a bug"|"These components don't work together"|
|How many to write|Many|Fewer|

```
Testing Pyramid:
       /\
      /  \        Fewer — slow, expensive
     / IT \       Integration Tests
    /------\
   /  Unit  \     Many — fast, cheap
  /----------\
```

> Write **many unit tests** (fast, cheap, catch logic bugs) and **fewer integration tests** (slower, but catch wiring/config bugs). Both are necessary.

## Test Annotations by Layer

| Layer          | Test Type          | Annotation                            | What It Loads                          | Tools                          |
| -------------- | ------------------ | ------------------------------------- | -------------------------------------- | ------------------------------ |
| **Repository** | Unit / Integration | `@DataJpaTest`                        | JPA only (repositories, EntityManager) | JUnit, AssertJ, TestContainers |
| **Service**    | Unit               | `@ExtendWith(MockitoExtension.class)` | Nothing — no Spring context            | JUnit, Mockito, AssertJ        |
| **Controller** | Unit               | `@WebMvcTest`                         | Web layer only (controllers, filters)  | JUnit, MockMvc, Mockito        |
| **Full API**   | Integration        | `@SpringBootTest`                     | Full application context               | JUnit, AssertJ, TestContainers |

Other useful annotations:

|Annotation|Purpose|
|---|---|
|`@TestConfiguration`|Define extra beans or config for tests only|
|`@AutoConfigureTestDatabase`|Replace actual DB with embedded DB for tests|

> Each annotation loads only what's needed — `@DataJpaTest` and `@WebMvcTest` are faster because they skip unrelated layers. `@SpringBootTest` loads everything (slowest).

## Q&A

1. **What's the difference between a unit test and an integration test?**
   A unit test isolates one class — all dependencies are mocked, no Spring context, no DB. An integration test wires real components together (service + repository + DB, often via `@SpringBootTest`) to catch wiring, query, and config bugs that mocks hide.

2. **What is the testing pyramid?**
   Many fast unit tests at the base, fewer integration tests in the middle, very few slow end-to-end tests at the top. Keeps the suite fast while still catching wiring bugs where it matters.

3. **Why not just write integration tests everywhere?**
   They're slow (seconds each vs milliseconds), flaky (depend on DB/network), and give vague failure signals ("something broke"). Unit tests pinpoint the exact method with the bug.

4. **Which slice annotation should I use for a repository test?**
   `@DataJpaTest` — loads only JPA (entities, repositories, `EntityManager`) and skips the web/service layer. Transactional by default so each test rolls back.

5. **When does `@SpringBootTest` make sense?**
   When you need the full context wired together — typically for end-to-end API tests (request → controller → service → repository → DB → response) via `MockMvc` or `WebTestClient`.

## Quick Reference

|Concept|Remember|
|---|---|
| Unit test | One class, dependencies mocked, no Spring context |
| Integration test | Multiple layers, real DB, Spring context loaded |
| `@DataJpaTest` | JPA slice — repositories + `EntityManager` only |
| `@WebMvcTest` | Web slice — controllers + filters only |
| `@SpringBootTest` | Full application context |
| Testing pyramid | Many unit > fewer integration > fewest E2E |
