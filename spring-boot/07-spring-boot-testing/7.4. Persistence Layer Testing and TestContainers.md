Repository methods — custom queries like `findByEmail`, derived queries, native queries — need a real database to verify (not mocks), because we're testing that the **queries actually work**.

## Creating Test Classes

In your IDE, go to the repository class → `Cmd+Shift+T` (Mac) or `Ctrl+Shift+T` (Windows) → create test class. Test files follow the **same package structure** as main:

```
src/main/java/com/app/repository/EmployeeRepository.java
src/test/java/com/app/repository/EmployeeRepositoryTest.java
```

Name tests to describe **what they test and what they expect**: widely-followed pattern: `[method]_[scenario]_[expectedResult]`:

```java
void findByEmail_WhenEmailIsValid_ThenReturnEmployee()
void findByEmail_WhenEmailNotFound_ThenReturnEmpty()
```

> No `test` prefix — `@Test` already marks the method (the prefix was a JUnit 3/4 leftover when the runner discovered tests by name). Good test names act as documentation — you can understand what the code does just by reading them.

## `@DataJpaTest`

Loads **only JPA components**: repositories, `EntityManager`, and an in-memory database. Does not load controllers, services, or security filters.

```java
@DataJpaTest
class EmployeeRepositoryTest {

    @Autowired
    private EmployeeRepository employeeRepository;

    @Test
    void findByEmail_WhenEmailIsValid_ThenReturnEmployee() {
        // Given (Arrange)
        Employee employee = Employee.builder()
            .firstName("John")
            .lastName("Doe")
            .email("john@example.com")
            .build();
        employeeRepository.save(employee);

        // When (Act)
        Optional<Employee> result = employeeRepository
            .findByEmail("john@example.com");

        // Then (Assert)
        assertThat(result).isPresent();
        assertThat(result.get().getFirstName()).isEqualTo("John");
        assertThat(result.get().getEmail()).isEqualTo("john@example.com");
    }

    @Test
    void findByEmail_WhenEmailNotFound_ThenReturnEmpty() {
        // When
        Optional<Employee> result = employeeRepository
            .findByEmail("nonexistent@example.com");

        // Then
        assertThat(result).isEmpty();
    }
}
```

**Test case pattern — Given / When / Then:**

|Step|Also Called|Purpose|
|---|---|---|
|**Given**|Arrange|Set up test data|
|**When**|Act|Execute the method being tested|
|**Then**|Assert|Verify the result|

**Key behaviors**: **Transactional by default**: each test runs in a transaction that is **rolled back** after the test. Tests don't affect each other, clean state every time. (Override with `@Rollback(false)` or `@Commit` if you need to assert against committed state, but that's rare.)
- **Auto-configures in-memory DB**: by default uses H2 if it's on the classpath.
- **Only loads JPA layer**: faster than `@SpringBootTest` because it skips web, service, security layers.
- **Application context caching**: Spring caches the context after the first test method loads it; other test methods in the same class (and other classes with identical config) reuse the cached context, so subsequent tests start instantly.

## TestContainers

By default, `@DataJpaTest` uses **H2** (in-memory database). H2 is fast and zero-setup, but its SQL dialect doesn't match production — features like PostgreSQL `JSONB`, arrays, and window functions either don't exist or behave differently. Tests pass on H2 and fail on real PostgreSQL.

Tests should run against a database as close to production as possible. Creating a dedicated permanent test database is overkill. **TestContainers** spins up a **real database in a Docker container** during tests — your tests run against actual PostgreSQL (or MySQL, MongoDB, etc.), the container starts before tests, runs during tests, and is destroyed after.

**Prerequisites:** Docker must be installed and running. Download from <https://docker.com>.

**Add dependencies:**

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-testcontainers</artifactId>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>postgresql</artifactId>
    <scope>test</scope>
</dependency>
```

**Test configuration**: `@ServiceConnection` tells Spring Boot to auto-configure the datasource from this container, no manual URL/username/password setup:

```java
@TestConfiguration(proxyBeanMethods = false)
public class TestContainersConfiguration {

    @Bean
    @ServiceConnection
    PostgreSQLContainer<?> postgresContainer() {
        return new PostgreSQLContainer<>(
            DockerImageName.parse("postgres:latest"));
    }
}
```

> **Speed tip — container reuse.** Container startup dominates suite time. Enable Testcontainers reuse by setting `withReuse(true)` on the container and adding `testcontainers.reuse.enable=true` to `~/.testcontainers.properties` on dev machines (and CI runners that persist Docker state). The container survives across test runs instead of being torn down each time, cutting per-run cost from seconds to near-zero. Don't enable in ephemeral CI where the Docker daemon is wiped — there's nothing to reuse.

**Use in tests:**

```java
@DataJpaTest
@Import(TestContainersConfiguration.class)
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
class EmployeeRepositoryTest {

    @Autowired
    private EmployeeRepository employeeRepository;

    @Test
    void findByEmail_WhenEmailIsValid_ThenReturnEmployee() {
        // Given
        Employee employee = Employee.builder()
            .firstName("John")
            .lastName("Doe")
            .email("john@example.com")
            .build();
        employeeRepository.save(employee);

        // When
        Optional<Employee> result = employeeRepository
            .findByEmail("john@example.com");

        // Then
        assertThat(result).isPresent();
        assertThat(result.get().getEmail()).isEqualTo("john@example.com");
    }
}
```

**Why `@AutoConfigureTestDatabase(replace = NONE)`?** `@DataJpaTest` tries to replace your datasource with H2 by default. Since we want to use the TestContainers PostgreSQL instead, we tell Spring **don't replace** the datasource.

|Value|Behavior|
|---|---|
|`ANY` (default)|Replace the datasource — uses H2|
|`NONE`|Don't replace — use whatever is configured (TestContainers)|
|`AUTO_CONFIGURED`|Only replace if it was auto-configured|

> Either remove H2 from `pom.xml` entirely, or set `replace = NONE`. If both H2 and TestContainers are present without this setting, Spring will use H2 and ignore your container.

||H2|TestContainers|
|---|---|---|
|Setup|Zero — just add dependency|Needs Docker installed|
|Speed|Very fast|Slower (container startup)|
|Matches production|Different SQL dialect|Same database as production|
|Catches real bugs|May miss DB-specific issues|Catches them|

## Q&A

1. **What does `@DataJpaTest` do?**
   Loads only the JPA slice — entities, repositories, `EntityManager` — and skips controllers, services, and security. It also wraps each test in a transaction that's rolled back at the end, and auto-configures an in-memory H2 database.

2. **Why use TestContainers instead of H2?**
   H2 is fast but has a different SQL dialect — features like PostgreSQL JSONB, arrays, or window functions either fail or behave differently. TestContainers spins up a real PostgreSQL in Docker so tests run against the same database as production.

3. **What does `@ServiceConnection` do?**
   It tells Spring Boot to automatically wire the datasource (URL, username, password, driver) from the container bean — no manual `application-test.properties` overrides needed.

4. **Why `@AutoConfigureTestDatabase(replace = NONE)`?**
   `@DataJpaTest` defaults to swapping in H2. `replace = NONE` keeps the datasource configured by your TestContainers bean instead of replacing it.

5. **How does Spring speed up tests when many use `@DataJpaTest`?**
   It caches the `ApplicationContext` per unique test configuration — the first test pays the startup cost, subsequent tests with identical config reuse the cached context.
