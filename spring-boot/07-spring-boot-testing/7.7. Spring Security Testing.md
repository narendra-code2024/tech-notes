Testing secured endpoints needs Spring Security's test helpers â€” without them, every protected endpoint returns 401 and your tests fail before they even reach the controller logic. The `spring-security-test` module provides annotations and configurers that inject an authenticated `SecurityContext` into your tests.

---

## Dependency

```xml
<dependency>
    <groupId>org.springframework.security</groupId>
    <artifactId>spring-security-test</artifactId>
    <scope>test</scope>
</dependency>
```

> Spring Boot's `spring-boot-starter-test` doesn't pull this in automatically. Without it, the security annotations below silently no-op â€” a common "passes locally, fails in CI" footgun when one developer has it cached in their local Maven repo.

---

## `@WithMockUser` â€” Inject a Synthetic User

The most common helper. Builds a fake user with the given username/roles, populates `SecurityContext`, runs the test, then auto-clears.

```java
@SpringBootTest
@AutoConfigureMockMvc
class UserControllerTest {

    @Autowired MockMvc mockMvc;

    @Test
    @WithMockUser(username = "alice", roles = "USER")
    void getProfile_authenticatedUser_returnsOk() throws Exception {
        mockMvc.perform(get("/api/users/me"))
            .andExpect(status().isOk());
    }

    @Test
    @WithMockUser(roles = "ADMIN")
    void deleteUser_asAdmin_succeeds() throws Exception {
        mockMvc.perform(delete("/api/users/1").with(csrf()))
            .andExpect(status().isNoContent());
    }

    @Test
    @WithMockUser(roles = "USER")
    void deleteUser_asUser_returns403() throws Exception {
        mockMvc.perform(delete("/api/users/1").with(csrf()))
            .andExpect(status().isForbidden());
    }
}
```

> Pass roles **without** the `ROLE_` prefix â€” `@WithMockUser` adds it automatically (matching `hasRole()` semantics). Pass `"ADMIN"`, not `"ROLE_ADMIN"`.

## `@WithUserDetails` â€” Load Real User from `UserDetailsService`

Use when your code casts `Authentication.getPrincipal()` to your domain `User` class, or when test logic needs the real DB-loaded entity.

```java
@Test
@WithUserDetails(
    value = "alice@example.com",
    userDetailsServiceBeanName = "customUserDetailsService"
)
void protectedEndpoint_loadsRealUser() throws Exception {
    mockMvc.perform(get("/api/posts"))
        .andExpect(status().isOk());
}
```

Calls your `UserDetailsService.loadUserByUsername(value)` to populate `SecurityContext` with the real user â€” including roles, account flags, and any custom fields on your `User` entity.

> The user row must exist before the test runs. Pre-load via `@Sql` or create in `@BeforeEach`.

## Testing JWT-Protected Endpoints

Two patterns depending on whether you want to exercise the real `JwtAuthFilter` chain.

**Pattern 1 â€” Skip the filter (slice tests, fast):**

```java
@WebMvcTest(PostController.class)
@AutoConfigureMockMvc(addFilters = false)  // disable security filter chain
class PostControllerSliceTest {

    @Autowired MockMvc mockMvc;
    @MockBean PostService postService;

    @Test
    @WithMockUser(roles = "USER")
    void getPosts_authenticated_returnsList() throws Exception {
        when(postService.findAll()).thenReturn(List.of(samplePost()));

        mockMvc.perform(get("/api/posts"))
            .andExpect(status().isOk());
    }
}
```

**Pattern 2 â€” Exercise the real filter (full integration):**

```java
@SpringBootTest(webEnvironment = RANDOM_PORT)
@AutoConfigureWebTestClient
class PostControllerIntegrationTest {

    @Autowired WebTestClient webTestClient;
    @Autowired JwtService jwtService;

    @Test
    void getPosts_withValidJwt_returnsList() {
        String token = jwtService.generateAccessToken("alice@example.com");

        webTestClient.get().uri("/api/posts")
            .header("Authorization", "Bearer " + token)
            .exchange()
            .expectStatus().isOk();
    }

    @Test
    void getPosts_withExpiredJwt_returns401() {
        String expiredToken = "eyJ...";  // pre-built expired token

        webTestClient.get().uri("/api/posts")
            .header("Authorization", "Bearer " + expiredToken)
            .exchange()
            .expectStatus().isUnauthorized();
    }
}
```

> Pattern 1 isolates controller logic from auth â€” fast, focused. Pattern 2 verifies the actual filter chain (signature validation, expired-token handling, `SecurityContextHolder` population). Use both at different levels of the test pyramid.

## Testing Method-Level Security (`@PreAuthorize`)

For methods annotated with `@PreAuthorize` / `@PostAuthorize`, the `@EnableMethodSecurity` config must be active in the test context.

```java
@SpringBootTest
class PostServiceMethodSecurityTest {

    @Autowired PostService postService;

    @Test
    @WithMockUser(roles = "ADMIN")
    void deletePost_asAdmin_succeeds() {
        assertThatCode(() -> postService.delete(1L))
            .doesNotThrowAnyException();
    }

    @Test
    @WithMockUser(roles = "USER")
    void deletePost_asUser_throwsAccessDenied() {
        assertThatThrownBy(() -> postService.delete(1L))
            .isInstanceOf(AccessDeniedException.class);
    }

    @Test
    void deletePost_anonymous_throwsAuthentication() {
        assertThatThrownBy(() -> postService.delete(1L))
            .isInstanceOf(AuthenticationCredentialsNotFoundException.class);
    }
}
```

> `@PreAuthorize` failures throw `AccessDeniedException` (not `AuthenticationException`). Over HTTP this surfaces as 403; in unit-style service tests assert the exception type directly.

## Testing 401 vs 403

Two distinct rejection states â€” tests should cover both since they reflect different bugs:

```java
@Test
void protectedEndpoint_noAuth_returns401() throws Exception {
    // No @WithMockUser â€” anonymous request
    mockMvc.perform(get("/api/posts"))
        .andExpect(status().isUnauthorized());
}

@Test
@WithMockUser(roles = "USER")
void adminEndpoint_asUser_returns403() throws Exception {
    mockMvc.perform(delete("/api/admin/users/1").with(csrf()))
        .andExpect(status().isForbidden());
}
```

| Status | Meaning | Cause |
|---|---|---|
| **401 Unauthorized** | No valid authentication | Missing/invalid token, anonymous request |
| **403 Forbidden** | Authenticated but lacks required role/authority | Wrong role, missing scope, denied by `@PreAuthorize` |

> Conflating these is a common bug â€” returning 403 for "no auth" leaks the existence of the endpoint to anonymous attackers; returning 401 for "lacks permission" forces re-login when the user just needs higher privilege.

## CSRF in Tests â€” `.with(csrf())`

Spring Security enables CSRF by default for stateful (cookie-based) auth. MockMvc tests of `POST` / `PUT` / `DELETE` endpoints fail with 403 if no CSRF token is supplied.

```java
import static org.springframework.security.test.web.servlet.request.SecurityMockMvcRequestPostProcessors.csrf;

@Test
@WithMockUser
void createPost_withCsrfToken_succeeds() throws Exception {
    mockMvc.perform(post("/api/posts")
            .with(csrf())  // injects valid CSRF token
            .contentType(MediaType.APPLICATION_JSON)
            .content("""
                {"title":"Hello","body":"World"}
                """))
        .andExpect(status().isCreated());
}
```

For **stateless JWT auth** (where CSRF is disabled in `SecurityConfig`), `.with(csrf())` is a no-op â€” but include it anyway for portability between session-based and JWT-based test setups.

## Production Considerations

- **`@WithMockUser` vs `@WithUserDetails` trade-off**: Mock is fast (no DB) but synthetic; UserDetails is slower but loads the real entity. If your code does `((User) auth.getPrincipal()).getEmail()`, mock-user gives a Spring `User`, not your domain class â€” choose `@WithUserDetails` or write a custom `@WithSecurityContext`.
- **Test password encoding**: when creating test users, hash passwords with the same `PasswordEncoder` your production code uses. Mismatched encoders cause silent BCrypt comparison failures that look like wrong-password bugs.
- **Don't disable CSRF globally in tests**: disable per-test (`@AutoConfigureMockMvc(addFilters = false)`) or per-request (`.with(csrf())`). Globally disabling hides CSRF misconfigurations until the bug ships to prod.
- **`SecurityContextHolder` leakage between tests**: `SecurityContextHolder` uses `ThreadLocal`. JUnit thread reuse can leak auth from one test to the next. `@WithMockUser` auto-clears via `WithSecurityContextTestExecutionListener`; manually-set contexts (`SecurityContextHolder.getContext().setAuthentication(...)`) need `@AfterEach { SecurityContextHolder.clearContext(); }`.
- **Don't test Spring Security itself**: assume the framework's filter chain works. Your tests should verify *your* rules: which roles access which endpoints, your custom `JwtAuthFilter` behavior, your `@PreAuthorize` expressions. Don't write tests that re-verify `hasRole()` matches the configured filter â€” that's Spring's job.

## Q&A

1. **Why doesn't `@Mock` work for testing security-protected endpoints?**
   `@Mock` is pure Mockito â€” the mock object never enters the Spring `ApplicationContext`, so the security filter chain doesn't see it. You need `@WithMockUser` (populates `SecurityContext` directly) or `@MockBean` (registers a mock as a Spring bean) instead.

2. **What's the difference between `@WithMockUser` and `@WithUserDetails`?**
   `@WithMockUser` synthesizes a `User` with the given username/roles â€” fast, no DB. `@WithUserDetails` calls your `UserDetailsService.loadUserByUsername(...)` to load the real user from your store â€” slower but exercises the lookup path. Use the latter when your code reads the real principal.

3. **Why does a POST/PUT/DELETE test fail with 403 even when authenticated?**
   CSRF protection is on by default for non-GET requests under stateful auth. MockMvc tests need `.with(csrf())` from `SecurityMockMvcRequestPostProcessors`, or `@AutoConfigureMockMvc(addFilters = false)` to skip the filter chain entirely.

4. **How do you test a JWT-secured endpoint end-to-end without mocking the filter?**
   `@SpringBootTest(webEnvironment = RANDOM_PORT)` + generate a real token via your `JwtService` + set `Authorization: Bearer <token>` on the request. Exercises the actual `JwtAuthFilter` chain rather than bypassing it.

5. **What does `@AutoConfigureMockMvc(addFilters = false)` do, and when should you avoid it?**
   It disables Spring Security's filter chain in MockMvc â€” useful for slice tests focused purely on controller logic. Avoid it when you actually want to verify auth behavior (401 vs 403, JWT validation, CSRF) â€” those need the filter chain enabled.

6. **A test passes locally with `@WithMockUser` but fails in CI with 401 â€” what's the likely cause?**
   The `spring-security-test` dependency is missing or wrong scope in `pom.xml`. The annotation silently no-ops without that module on the classpath; a clean CI build exposes it. Fix: add `spring-security-test` with `<scope>test</scope>`.

7. **Why does `@PreAuthorize` throw `AccessDeniedException` instead of returning 403 directly?**
   `@PreAuthorize` runs at the method-call level, before HTTP serialization. The exception bubbles up to `ExceptionTranslationFilter` (in the security filter chain), which translates `AccessDeniedException` to a 403 response. In unit tests of services (no HTTP), you assert the exception directly.
