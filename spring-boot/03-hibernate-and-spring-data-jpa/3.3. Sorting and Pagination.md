## Sorting

### Via method name (`OrderBy`)

```java
public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    List<Employee> findAllByOrderByNameAsc();
    List<Employee> findByDepartmentOrderByNameDesc(String department);
}
```

Rules:

- `OrderBy` comes after the `By` clause and any query conditions
- Direction (`Asc` / `Desc`) is optional — default is `ASC`
- **Limitation**: field and direction are hardcoded; you can't change them at runtime

### Via `Sort` parameter (dynamic)

Accept a `Sort` object at the call site — fully dynamic, supports multiple fields.

```java
public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    List<Employee> findByDepartment(String department, Sort sort);
}

// caller
Sort sort = Sort.by(Sort.Direction.ASC, "name");
List<Employee> employees = employeeRepository.findByDepartment("IT", sort);

// multiple fields
Sort multi = Sort.by(
    Sort.Order.asc("name"),
    Sort.Order.desc("salary")
);
```

`JpaRepository` already exposes `findAll(Sort)` — no extra declaration needed for sorting the full table.

```java
List<Employee> employees = employeeRepository.findAll(Sort.by("name").descending());
```

## Pagination

Splits large result sets into smaller pages. Core types:

- **`Pageable`**: input interface carrying page number, size, and optional sort
- **`PageRequest`**: concrete `Pageable` implementation you construct via `PageRequest.of(...)`
- **`Page<T>`**: the returned result; carries the items on this page plus pagination metadata (totals, position):
    - `getContent()` — items on the current page
    - `getTotalElements()` — total row count
    - `getTotalPages()` — total page count
    - `getNumber()` — current page index (**zero-based**)
    - `isFirst()`, `isLast()` — boundary checks

### Using `Page`, `Pageable`, `PageRequest`

```java
public interface UserRepository extends JpaRepository<User, Long> {
    Page<User> findByLastName(String lastName, Pageable pageable);
}

// caller — page 0 (first page), 10 records
Pageable pageable = PageRequest.of(0, 10);
Page<User> page = userRepository.findByLastName("Smith", pageable);

// combining pagination + sorting
Pageable sortedPage = PageRequest.of(0, 5, Sort.by("salary").descending());
```

Spring Data JPA fires **two queries** for a `Page<T>` result — the count query mirrors the main query's `WHERE` clause:

```sql
-- main query
SELECT * FROM employee WHERE last_name = ? LIMIT ? OFFSET ?;

-- count query
SELECT COUNT(*) FROM employee WHERE last_name = ?;
```

> **Deep pagination is expensive.** `OFFSET 100000` makes the database scan and discard 100k rows before returning the next page — cost grows linearly with depth. `Page<T>` + `OFFSET` is fine for admin UIs and shallow pages; for infinite scroll or millions of rows, prefer **cursor (keyset) pagination** (covered next).

### Cursor (Keyset) Pagination

Pages are anchored by the **last seen sort value** instead of an offset — constant query cost regardless of how deep you go.

The underlying SQL form:

```sql
SELECT * FROM employee
WHERE id > :lastSeenId
ORDER BY id
LIMIT 20;
```

Each page returns the items + the last `id` seen; the client passes that back as `lastSeenId` for the next page. No `OFFSET` — the DB seeks directly to the cursor via the index.

In Spring Data — two repository styles:

```java
public interface EmployeeRepository extends JpaRepository<Employee, Long> {

    // Naive cursor — derived method, caller passes lastSeenId
    List<Employee> findByIdGreaterThanOrderByIdAsc(Long lastSeenId, Pageable limit);

    // Spring Data 3.1+ — built-in keyset support via Window<T>
    Window<Employee> findByOrderByIdAsc(ScrollPosition position, Limit limit);
}
```

Usage across two HTTP requests:

- **Page 1** (`GET /employees?limit=20`) → query + extract cursor → respond with items + `nextCursor`
- **Page 2** (`GET /employees?cursor=xyz&limit=20`) → decode cursor → query → respond

```java
@GetMapping("/employees")
public CursorPageResponse<EmployeeDto> list(
    @RequestParam(required = false) String cursor,
    @RequestParam(defaultValue = "20") int limit
) {
    // Missing cursor → first page; present → decoded keyset position
    ScrollPosition position = (cursor == null || cursor.isBlank())
        ? ScrollPosition.keyset()
        : cursorCodec.decode(cursor);

    Window<Employee> window = employeeRepository.findByOrderByIdAsc(position, Limit.of(limit));

    // Include nextCursor only if more pages exist
    String nextCursor = window.hasNext()
        ? cursorCodec.encode(window.positionAt(window.size() - 1))
        : null;

    return new CursorPageResponse<>(toDto(window.getContent()), nextCursor);
}
```

> `cursorCodec` is a custom helper you write — typically Base64-encoded JSON of the cursor's keys map. Spring doesn't ship one because the encoding choice is yours (Base64-JSON, JWT-signed, etc.).

Requirements and trade-offs:

- **Sort is mandatory**: the cursor encodes "where you left off" in the sort column; no sort = no cursor
- **Cursor must encode the actual sort columns**, not just `id`. If user sorts by `created_at DESC`, cursor = last `created_at`. For non-unique sort columns, include a unique tiebreaker (`id`)
- **Dynamic sort + cursor is hard**: when an API exposes user-driven sort, especially multi-column (`?sort=createdAt:desc,price:asc`), the cursor must encode every sort column as a tuple, and its structure changes per sort combination. Most production APIs use cursor only for *fixed* sort (feeds, audit logs) and fall back to `OFFSET` for *dynamic* sort (search/filter dashboards)

## Return Types — `Page<T>` vs `Slice<T>` vs `List<T>`

| | `Page<T>` | `Slice<T>` | `List<T>` |
|---|---|---|---|
| COUNT query fired | Yes | No | No |
| Knows total pages | Yes | `hasNext()` only | N/A |
| Internal trick | Separate `COUNT(*)` | Fetches `LIMIT + 1` rows | Plain query |
| Performance | Slightly heavier | Lighter | Lightest |
| Use case | UI needs total count / page count | Infinite-scroll style UIs | Just items, no pagination metadata |

## Q&A

1. **`Page<T>` vs `Slice<T>` — when to use which?**
   `Page` fires an extra `COUNT(*)` so it knows `getTotalElements()` and `getTotalPages()`. `Slice` skips the count and fetches `LIMIT + 1` rows to learn if another page exists. Use `Slice` for infinite-scroll UIs; use `Page` when you need total counts.

2. **Why is sorting recommended with pagination even though it's optional?**
   Without `ORDER BY`, row order is undefined — results can shift between pages as data mutates or even between executions. A stable sort (ideally including the primary key as a tiebreaker) guarantees consistent paging.

3. **Is `Pageable.pageNumber` zero-based or one-based?**
   Zero-based. `PageRequest.of(0, 10)` returns the first 10 rows.

4. **Method-name sorting (`findAllByOrderByNameAsc`) vs `Sort` parameter — which is better?**
   `Sort` is almost always better: it's dynamic, supports multiple fields, and keeps method names short. Use `OrderBy` in the method name only when a single repository method always needs the same fixed sort.

5. **What's the problem with `OFFSET`-based pagination on large tables?**
   `OFFSET N` forces the database to scan and discard `N` rows before returning the page — cost grows linearly with page depth. For deep pagination use **keyset (cursor) pagination**: `WHERE id > :lastSeenId ORDER BY id LIMIT N`. Constant cost, plus stable across inserts.

6. **When would you choose cursor over OFFSET pagination, and what does the cursor encode?**
   Cursor (keyset) pagination has constant cost regardless of depth — use it for deep paging, infinite scroll, feeds, audit timelines. The cursor encodes the **last seen sort column value(s)**, plus a unique tiebreaker (`id`) when the sort field isn't unique. Trade-off: cursor pairs cleanly with a *fixed* sort; with user-driven *dynamic* sort, the cursor structure changes per query, so most production APIs reserve cursor for fixed-sort endpoints and fall back to `OFFSET` for search/filter dashboards. Spring Data 3.1+ provides `KeysetScrollPosition` and `Window<T>` for built-in keyset support.

7. **If a user binds `Sort` directly from a request parameter, what's the security risk?**
   `Sort.by(field)` accepts any property name, including ones the API didn't intend to expose (e.g., `passwordHash`). Always validate the sort field against an allow-list before constructing the `Sort`.
