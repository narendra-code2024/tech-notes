A performance issue where loading a collection of `N` parent entities triggers `1 + N` queries:
 one to fetch the parents, then one extra query per parent to load its children.

```java
@Entity
public class Order {

    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items;
}

// caller
List<Order> orders = orderRepository.findAll();
for (Order order : orders) {
    System.out.println(order.getItems().size());   // lazy load triggers per order
}
```

What Hibernate executes:

```sql
-- 1: parent query
SELECT * FROM orders;

-- N: child queries, one per order
SELECT * FROM order_item WHERE order_id = 1;
SELECT * FROM order_item WHERE order_id = 2;
SELECT * FROM order_item WHERE order_id = 3;
-- ... and so on
```

5 orders → 6 queries. 100 orders → 101 queries. Pages of 1000 rows magnify this into a real outage.

---

## Why It Happens

- `@OneToMany` is `LAZY` by default
- Hibernate loads each child collection only when first accessed
- Touching it inside a loop means N separate `SELECT`s

> **Important:** N+1 happens with `EAGER` too — Hibernate may still issue one query per association rather than a single join. Switching `LAZY` → `EAGER` is **not** a fix; it just changes when the queries fire.

---

## Solutions

### 1. Fetch Join (most common)

Force a single `JOIN` in JPQL with `JOIN FETCH`:

```java
public interface OrderRepository extends JpaRepository<Order, Long> {

    @Query("SELECT o FROM Order o JOIN FETCH o.items")
    List<Order> findAllWithItems();
}
```

Resulting SQL — one query:

```sql
SELECT o.*, i.*
FROM orders o
JOIN order_item i ON o.id = i.order_id;
```

> **`JOIN` vs `JOIN FETCH`**: `JOIN` only joins for the `WHERE` clause (filtering); Hibernate may still fire a separate query to populate the association. `JOIN FETCH` populates the association as part of the same query. Use `JOIN` when you only need filtering; `JOIN FETCH` when you actually want the related data on the returned entities.

**Limitations of fetch join**: **Multiple bag collections**: fetching two `List`-typed collections in one query throws `MultipleBagFetchException`. Use `Set` or split into separate queries.
- **Cartesian product explosion**: fetching multiple collections produces `parents × children1 × children2` rows. Distinct or batched queries are usually saner.
- **Pagination + fetch join is broken**: Hibernate fetches all rows into memory and paginates in-app, defeating `LIMIT / OFFSET`. Use `@EntityGraph` with pagination only on `@ManyToOne` / `@OneToOne` associations, or load IDs first then fetch in a second query.

### 2. `@EntityGraph` (cleaner with Spring Data)

Declarative — no custom JPQL:

```java
public interface OrderRepository extends JpaRepository<Order, Long> {

    @EntityGraph(attributePaths = "items")
    List<Order> findAll();
}
```

Spring rewrites the query to use a fetch join.

### 3. Batch Fetching (Hibernate-specific)

Hibernate groups child loads into batches of `N` IDs:

```properties
spring.jpa.properties.hibernate.default_batch_fetch_size=20
```

100 orders with batch size 20 → 1 parent query + 5 batched child queries instead of 1 + 100. Better, but still not as good as a fetch join for read-heavy paths.

### 4. DTO Projection (best for read-heavy APIs)

Project only the columns you need — no entity loading at all:

```java
public interface OrderRepository extends JpaRepository<Order, Long> {

    @Query("""
        SELECT new com.example.OrderDTO(o.id, i.name)
        FROM Order o
        JOIN o.items i
    """)
    List<OrderDTO> findOrderData();
}
```

See [3.4 Projections](<3.4. Projections.md>).

## How to Detect N+1

### SQL query logging

The first tool — see what Hibernate actually sends to the DB. In `application.yml`:

```yaml
spring:
  jpa:
    properties:
      hibernate:
        format_sql: true              # pretty-print generated SQL

logging:
  level:
    org.hibernate.SQL: DEBUG          # logs every query
    org.hibernate.orm.jdbc.bind: TRACE  # logs bound parameter values
```

| Property | What it shows |
|---|---|
| `org.hibernate.SQL: DEBUG` | Every SQL statement Hibernate executes |
| `org.hibernate.orm.jdbc.bind: TRACE` | The actual parameter values bound to `?` placeholders |
| `format_sql: true` | Multi-line, indented SQL — much easier to read than the default single-line dump |

> `spring.jpa.show-sql=true` is an older alias that prints SQL without log-level control. Prefer `org.hibernate.SQL: DEBUG` — it goes through SLF4J/Logback, respects log appenders, and can be toggled per environment.

### Spotting N+1 in the logs

Look for **repeated SELECTs on the same table with different IDs** firing right after a parent query:

```sql
-- the offending parent query
select * from author where id = 1

-- N child queries (the N+1)
select * from book where author_id = 1
select * from book where author_id = 2
select * from book where author_id = 3
...
```

If you see this pattern in dev for a single endpoint hit, that's N+1 — fix with `JOIN FETCH`, `@EntityGraph`, or DTO projection (above).

### Don't enable in production

`org.hibernate.SQL: DEBUG` produces one log line per query. A page serving 50 queries × thousands of requests/sec = log volume + I/O cost. Use **`WARN`** in prod and rely on observability (APM, slow-query logs) instead. Toggle to `DEBUG` per-instance only for active debugging.

### Other detection tools

- **Hibernate statistics**: `hibernate.generate_statistics=true` exposes counters via JMX/Actuator (query count, fetch count, second-level cache hits)
- **p6spy**: wraps the JDBC driver; logs every SQL with timings + stacktrace
- **Datasource-Proxy**: assert query count in integration tests (`assertThat(queryCount).isLessThan(5)`)
- **APM tools** (New Relic, Datadog, Honeycomb) — surface N+1 patterns automatically in production traces

## Q&A

1. **What is the N+1 problem?**
   Loading `N` parent entities and then firing one extra query per parent to load its associations — `1 + N` total queries instead of one. Common with lazy `@OneToMany` collections accessed in a loop.

2. **Does switching `LAZY` to `EAGER` fix N+1?**
   No. EAGER just moves the extra queries to load time — Hibernate may still emit one SELECT per association. Real fixes are `JOIN FETCH`, `@EntityGraph`, batch fetching, or DTO projections.

3. **`JOIN` vs `JOIN FETCH` — what's the difference?**
   `JOIN` is for filtering (it joins for the `WHERE` clause but doesn't populate the association on the entity). `JOIN FETCH` additionally hydrates the related entities so accessing them later doesn't trigger another query.

4. **Why does pagination break with `JOIN FETCH`?**
   The join produces `parent × child` rows, so `LIMIT N` would cut children off mid-parent. Hibernate detects this and fetches all rows into memory, paginating in-app — typically a warning in logs and a memory hazard. Use `@EntityGraph` only on single-valued associations when paginating, or load parent IDs first and fetch the children in a second query.

5. **How do you actually find N+1 in a real app?**
   Enable `org.hibernate.SQL: DEBUG` + `format_sql: true` in dev, look for repeated SELECTs on the same table with different IDs. In tests, assert query count with a tool like Datasource-Proxy. p6spy and Hibernate statistics work well in staging. APM tools (New Relic, Datadog) surface N+1 in production traces. Never run with `DEBUG` in prod — log volume kills performance.
