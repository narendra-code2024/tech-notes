## The Persistence Stack

From your code down to the database:

```
Application
    ↓
Spring Data JPA        (repository abstraction)
    ↓
JPA                    (specification — interfaces & annotations)
    ↓
Hibernate              (JPA provider — implementation)
    ↓
JDBC                   (low-level SQL driver)
    ↓
Database
```

- **JDBC** — lowest-level API; executes raw SQL, manages connections / statements / result sets. Hibernate uses JDBC under the hood.
- **JPA** (**Java Persistence API**) — a **specification**. Defines annotations (`@Entity`, `@Id`), interfaces (`EntityManager`), JPQL, transaction rules, and relationship mapping. Provides no implementation. Popular providers: Hibernate (most common), EclipseLink, Apache OpenJPA.
- **Hibernate** — the most popular JPA provider. Full-featured ORM: converts entity objects to SQL, maps result sets back to Java objects, supports caching (L1 / L2), lazy/eager loading, and query optimization.
- **Spring Data JPA** — sits on top of JPA. Eliminates DAO boilerplate by generating repository implementations at runtime. See [3.2 Spring Data JPA](<3.2. Spring Data JPA.md>).

## JPA Is a Specification

JPA is a **specification** — it defines the rules but ships no concrete persistence code. JPA providers (like Hibernate) supply the implementation.

Take `EntityManager` as an example:

- JPA **defines** the `EntityManager` interface
- Hibernate **implements** it
- Spring **injects** the Hibernate-backed implementation

When you call `userRepository.save(user)`, Spring Data JPA invokes `entityManager.persist(user)` under the hood. The flow from there:

1. Spring Data JPA calls the JPA interface (`EntityManager.persist`)
2. Hibernate's implementation receives the call and runs its persistence logic
3. Hibernate generates the SQL `INSERT` statement
4. SQL is executed via JDBC
5. JDBC sends it to the database

## Entity Annotations

### `@Entity` and `@Table`

`@Entity` marks a class as a JPA entity. `@Table` customizes the table mapping (optional — defaults to the class name).

```java
@Entity
@Table(name = "employees")
public class Employee {
    // ...
}
```

`@Table` attributes: `name`, `schema`, `catalog`, `uniqueConstraints`, `indexes`.

> In production with Flyway/Liquibase migrations, typically only `name` is used — schema, unique constraints, and indexes live in migration scripts. The other annotation attributes mainly drive Hibernate's DDL generation (`ddl-auto=create/update`), which you don't use in prod with `ddl-auto=validate`.

> If you do want them in the annotation (e.g., dev with `ddl-auto=create` or a JPA-only prototype):
>
> ```java
> @Table(
>     name = "employees",
>     uniqueConstraints = @UniqueConstraint(columnNames = "email"),
>     indexes = @Index(name = "idx_name", columnList = "name")
> )
> ```

> Schema is typically configured globally via `spring.jpa.properties.hibernate.default_schema` or the JDBC URL (`?currentSchema=hr`). Only specify `schema = "..."` per-entity for multi-schema apps where different entities live in different schemas.

### `@Id` and `@GeneratedValue`

`@Id` marks the primary key. `@GeneratedValue` delegates ID generation to the provider.

```java
@Entity
public class Employee {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
}
```

| Strategy | Behavior |
|---|---|
| `IDENTITY` | DB auto-increment column (MySQL, PostgreSQL) — most common |
| `SEQUENCE` | DB sequence (preferred for PostgreSQL/Oracle) |
| `AUTO` | Hibernate picks based on dialect — not always auto-increment |
| `TABLE` | Uses a separate table to simulate sequences — least efficient |

### Composite Uniqueness — Surrogate PK + Unique Constraint

In most apps, the primary key is a surrogate `@GeneratedValue Long id` for every entity. When you need uniqueness across multiple columns (e.g., "one review per user per product"), declare a composite **unique constraint** instead of a composite primary key:

```java
@Entity
@Table(
    name = "reviews",
    uniqueConstraints = @UniqueConstraint(
        name = "uk_review_per_user_product",
        columnNames = { "user_id", "product_id" }
    )
)
public class Review {

    @Id @GeneratedValue
    private Long id;                  // surrogate PK

    @ManyToOne
    @JoinColumn(name = "user_id")
    private User user;

    @ManyToOne
    @JoinColumn(name = "product_id")
    private Product product;

    private Integer rating;
    private String comment;
}
```

This enforces "a user can leave only one review per product" at the DB level, while keeping the surrogate PK for clean FK references and stable entity identity. **This is the modern default for nearly all entities.**

### Composite Primary Keys (`@EmbeddedId` vs `@IdClass`)

When the primary key itself spans multiple columns — uncommon in modern app design, but useful for legacy schemas or join tables modeled as entities (e.g., `Review(userId, productId)` enforcing "one review per user per product") — use a composite primary key. Two approaches: `@EmbeddedId` (preferred) and `@IdClass` (legacy).

**`@EmbeddedId` — preferred**

Wrap the key fields in an `@Embeddable` class, then reference it as a single ID field:

```java
@Embeddable
public class ReviewId implements Serializable {
    private Long userId;
    private Long productId;
    // equals, hashCode, no-args + all-args constructors
}

@Entity
public class Review {
    @EmbeddedId
    private ReviewId id;
    private Integer rating;
    private String comment;
}
```

**`@IdClass` — legacy alternative**

Define a separate "ID class" with the same field names, then mark each PK field on the entity with `@Id`:

```java
public class ReviewId implements Serializable {
    private Long userId;
    private Long productId;
    // equals, hashCode
}

@Entity
@IdClass(ReviewId.class)
public class Review {
    @Id private Long userId;
    @Id private Long productId;
    private Integer rating;
    private String comment;
}
```

| | `@EmbeddedId` | `@IdClass` |
|---|---|---|
| ID accessed via | `review.getId().getUserId()` | `review.getUserId()` |
| Boilerplate | Less — single class for the key | More — duplicate fields on entity + ID class |
| Modern preference | ✅ Yes | Legacy / interop with older code |

**Requirements (both):**

- ID class must implement `Serializable`
- Override `equals()` and `hashCode()` based on all key fields
- Provide a no-args constructor

> **Prefer surrogate PK + composite UNIQUE.** A *surrogate* PK is an auto-generated ID with no business meaning — typically `@GeneratedValue Long id`. Stable (never changes if business fields are renamed), compact (single `BIGINT` indexes/joins faster than composite), and simpler to map. For "one review per user per product", the modern shape is `id` PK + `UNIQUE (user_id, product_id)` — not a composite PK on `(user_id, product_id)`. Reach for composite PKs only when you have a true compound natural key with business meaning, a pure join table with no extras, or legacy DB interop.

### `@MapsId` — composite PK + relationship sharing FK columns

When a composite-key entity also has `@ManyToOne` relationships pointing to the same columns that form the PK (typical for join entities with extras), use `@MapsId` to tell Hibernate the FK column doubles as part of the embedded PK.

Take "one review per user per product" — the `Review` entity has a composite PK `(user_id, product_id)` and also needs `@ManyToOne` relationships to `User` and `Product`.

**Without `@MapsId` — duplicate column mapping error:**

```java
@Embeddable
public class ReviewId implements Serializable {
    private Long userId;        // maps to user_id
    private Long productId;     // maps to product_id
}

@Entity
public class Review {
    @EmbeddedId
    private ReviewId id;

    @ManyToOne
    @JoinColumn(name = "user_id")    // ❌ user_id already mapped via id.userId
    private User user;
}
```

Hibernate complains: *"Repeated column in mapping for entity: Review column: user_id"*

**With `@MapsId`:**

```java
@Entity
public class Review {

    @EmbeddedId
    private ReviewId id;

    @ManyToOne(fetch = FetchType.LAZY)
    @MapsId("userId")                    // ← shares id.userId
    @JoinColumn(name = "user_id")
    private User user;

    @ManyToOne(fetch = FetchType.LAZY)
    @MapsId("productId")                 // ← shares id.productId
    @JoinColumn(name = "product_id")
    private Product product;

    private Integer rating;
    private String comment;
}
```

**What `@MapsId("userId")` does:**
- Tells Hibernate: "the FK column for this relationship IS the `userId` field of the embedded ID"
- Avoids declaring the same column twice (once in the ID, once via `@JoinColumn`)
- Setting `review.setUser(user)` automatically populates `id.userId` — no manual sync

**When you need it:** join entities with extra columns (`rating`, `comment`, timestamps) where the PK is composed of FKs to two parent entities. Without `@MapsId`, you'd either need a surrogate ID column or hit the duplicate-mapping error above.

**Without composite PK** (using surrogate `id` + composite UNIQUE), `@MapsId` isn't needed — relationships are independent of the PK.

### `@Column`

Customizes the column mapping.

```java
@Column(name = "full_name", nullable = false, length = 50, updatable = false)
private String name;
```

Common options: `name`, `nullable`, `unique`, `length`, `updatable`, `insertable`.

### `@CreationTimestamp` and `@UpdateTimestamp`

Hibernate-specific (not pure JPA) — automatically populated on insert / update.

```java
@CreationTimestamp
private LocalDateTime createdAt;

@UpdateTimestamp
private LocalDateTime updatedAt;
```

## Hibernate Configuration

| Property | Description |
|---|---|
| `spring.jpa.hibernate.ddl-auto` | Controls schema generation |
| `spring.jpa.show-sql` | Prints SQL to console |
| `spring.jpa.properties.hibernate.format_sql` | Pretty-prints the SQL |
| `spring.jpa.properties.hibernate.dialect` | SQL dialect for the target DB |

### `ddl-auto` values

| Value | Meaning |
|---|---|
| `none` | No action (default for non-embedded DBs) |
| `validate` | Validates schema against entities at startup |
| `update` | Updates schema if entities changed |
| `create` | Drops and recreates schema |
| `create-drop` | Same as `create`, plus drops at shutdown |

**Production recommendation:** `validate` or `none`. Use Flyway / Liquibase for real schema changes.

### SQL Init Properties

| Property | Default | Possible Values |
|---|---|---|
| `spring.sql.init.mode` | `embedded` | `always`, `embedded`, `never` |
| `spring.sql.init.schema-locations` | `classpath:schema.sql` | Any valid path |
| `spring.sql.init.data-locations` | `classpath:data.sql` | Any valid path |
| `spring.sql.init.platform` | `all` | Any platform name |
| `spring.sql.init.continue-on-error` | `false` | `true` / `false` |
| `spring.sql.init.encoding` | `UTF-8` | Any valid encoding |

**Defer DataSource initialization** — when using both Hibernate (for schema) and `data.sql` (for seed data), order matters:

```properties
spring.jpa.defer-datasource-initialization=true
```

- With this: Hibernate creates tables first, then `data.sql` runs
- Without this: `data.sql` may execute before tables exist → errors

## Important Interview Questions

1. **JPA vs Hibernate vs Spring Data JPA?**
   **JPA** is a specification (interfaces + annotations, no code). **Hibernate** is the most common JPA provider — it implements the spec and generates SQL. **Spring Data JPA** sits on top of JPA and generates repository implementations at runtime to eliminate DAO boilerplate.

2. **Is JPA an implementation or a specification?**
   A specification. JPA defines `EntityManager`, `@Entity`, JPQL, and transaction rules, but ships no persistence code. Providers like Hibernate supply the implementation.

3. **What's JPQL and how does it differ from SQL?**
   JPQL queries entity classes and fields, not tables and columns. Hibernate translates JPQL to SQL using the configured dialect. It's portable across databases because the dialect abstracts vendor-specific SQL.

4. **What `ddl-auto` value should you use in production?**
   `validate` or `none`. `update` looks convenient but can silently corrupt schemas on rename / type change. Use Flyway or Liquibase for real migrations.

5. **Where does JDBC fit in the JPA stack?**
   JDBC is the lowest layer — Hibernate uses JDBC to send SQL to the database. You rarely interact with JDBC directly when using Spring Data JPA.

6. **`@EmbeddedId` vs `@IdClass` — which and why?**
   Both define composite primary keys. **`@EmbeddedId`** wraps the key fields in a separate `@Embeddable` class — modern preference; less boilerplate, the ID is one cohesive field. **`@IdClass`** keeps the PK fields directly on the entity but requires a parallel ID class with the same fields. Use `@EmbeddedId` for new code; `@IdClass` mainly for legacy interop. Both require the ID class to implement `Serializable`, override `equals` / `hashCode`, and provide a no-args constructor.
