Spring Data JPA simplifies database access by providing a **repository abstraction** over JPA. Built on top of JPA — with Hibernate as the default implementation underneath.

Provides:

- Repository abstraction — no manual DAO implementation
- Automatic query generation from method names
- Pagination and sorting out of the box
- Reduced boilerplate

Instead of writing `entityManager.persist(...)` / `entityManager.createQuery(...)`, you just declare an interface:

```java
public interface UserRepository extends JpaRepository<User, Long> { }
```

That's it — full CRUD (`save`, `findById`, `findAll`, `delete`, `count`, etc.) is inherited automatically. Spring generates the implementation at runtime via a proxy. Custom query methods are added on top — covered below.

---

## Repository Hierarchy

```
Repository                     (marker, no methods)
    ↓
CrudRepository                 (basic CRUD)
    ↓
PagingAndSortingRepository     (pagination & sorting)
    ↓
JpaRepository                  (batch ops, flush, JPA-specific)
```

Most applications extend `JpaRepository<EntityType, IdType>` to get the full set of features.

**Built-in methods** (inherited):

- **Read**: `findAll()`, `findById(id)`, `findAll(Pageable)`, `findAll(Sort)`
- **Existence / count**: `count()`, `existsById(id)`
- **Save**: `save(entity)`, `saveAll(entities)`, `saveAndFlush(entity)`, `saveAllAndFlush(entities)`
- **Delete**: `deleteById(id)`, `delete(entity)`, `deleteAll()`, `deleteAllInBatch()`

## CRUD Operations

Putting the repository to work in a service:

```java
@Service
@RequiredArgsConstructor
public class UserService {

    private final UserRepository userRepository;

    // Create
    public User createUser(String email) {
        User user = new User(email);                 // id is null
        return userRepository.save(user);            // → INSERT
    }

    // Read — by ID
    public User getUser(Long id) {
        return userRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("User", id));
    }

    // Read — all
    public List<User> getAllUsers() {
        return userRepository.findAll();
    }

    // Update
    public User updateEmail(Long id, String newEmail) {
        User user = userRepository.findById(id).orElseThrow();
        user.setEmail(newEmail);
        return userRepository.save(user);            // → UPDATE
    }

    // Delete
    public void deleteUser(Long id) {
        userRepository.deleteById(id);               // → DELETE
    }
}
```

How operations map to SQL:

- **Create**: `save(entity)` with no ID → `INSERT`
- **Read**: `findById(id)` returns `Optional`; `findAll()` returns `List`
- **Update**: `save(entity)` with existing ID → `UPDATE`
- **Delete**: `deleteById(id)` or `delete(entity)` → `DELETE`

## Derived Query Methods

Spring parses repository method names and builds queries automatically. A derived query method has four parts:

```
ReturnType   QuerySubject   By   QueryPredicate(InputParams)
```

```java
public interface UserRepository extends JpaRepository<User, Long> {
    List<User> findByEmail(String email);
    List<User> findByNameAndStatus(String name, String status);
}
```

Spring converts `findByEmail` to JPQL `SELECT u FROM User u WHERE u.email = :email`, which Hibernate then translates to SQL.

### Query subject prefixes

Common prefixes: `find…By`, `read…By`, `get…By`, `query…By`, `count…By`, `exists…By`, `delete…By`, `remove…By`.

### Limiting results

```java
Employee findFirstByName(String name);
List<Employee> findTop5ByStatus(String status);
List<Employee> findFirst3ByOrderByCreatedAtDesc();
```

### Distinct results

```java
List<Employee> findDistinctByName(String name);
```

### Combining conditions

```java
List<Employee> findByNameAndStatus(String name, String status);
List<Employee> findByEmailOrPhone(String email, String phone);
```

> **Parameter order matters.** Derived methods bind parameters **positionally** to the predicates in the method name. `findByNameAndStatus(String name, String status)` binds the 1st arg to `Name`, 2nd to `Status`. Swapping them silently runs the wrong query — no compile error. Parameter *names* don't matter; only order. For named binding, use `@Query` with `@Param`.

### Query keywords

Spring supports many operators — `Between`, `LessThan`, `GreaterThanEqual`, `Like`, `StartingWith`, `In`, `IsNull`, etc.

```java
List<Employee> findByCreatedAtBetween(LocalDateTime start, LocalDateTime end);
List<Employee> findBySalaryGreaterThanEqual(Double salary);
List<Employee> findByNameLike(String pattern);              // caller passes wildcards: "%john%"
List<Employee> findByEmailStartingWith(String prefix);
List<Employee> findByStatusIn(List<String> statuses);
List<Employee> findByManagerIsNull();                       // top-level employees (no manager)
```

Full list: [official keywords reference](https://docs.spring.io/spring-data/jpa/reference/repositories/query-keywords-reference.html).

### Nested property queries

If the entity has a relationship, traverse it by property name — Spring generates the JOIN automatically.

```java
@Entity
public class Order {
    @ManyToOne
    private User user;
}

public interface OrderRepository extends JpaRepository<Order, Long> {
    List<Order> findByUserName(String name);   // JOIN on order.user.name
}
```

## When Derived Queries Aren't Enough

Use `@Query` for cases derived methods can't express — long names, JOINs, subqueries, aggregations, or native SQL. Same query, three styles:

```java
public interface UserRepository extends JpaRepository<User, Long> {

    // 1. Derived — getting unwieldy
    List<User> findByNameAndStatusAndTypeAndCreatedAtBetween(
        String name, String status, String type, LocalDateTime start, LocalDateTime end
    );

    // 2. JPQL @Query — entity-aware, portable across databases
    @Query("""
        SELECT u FROM User u
        WHERE u.name = :name
          AND u.status = :status
          AND u.type = :type
          AND u.createdAt BETWEEN :start AND :end
        """)
    List<User> findFilteredUsers(
        @Param("name") String name,
        @Param("status") String status,
        @Param("type") String type,
        @Param("start") LocalDateTime start,
        @Param("end") LocalDateTime end
    );

    // 3. Native SQL — when JPQL can't express what you need
    @Query(value = """
        SELECT * FROM users
        WHERE name = :name AND status = :status
          AND type = :type AND created_at BETWEEN :start AND :end
        """, nativeQuery = true)
    List<User> findFilteredUsersNative(
        @Param("name") String name,
        @Param("status") String status,
        @Param("type") String type,
        @Param("start") LocalDateTime start,
        @Param("end") LocalDateTime end
    );
}
```

### JPQL vs Native SQL

| Concern                                                            | JPQL `@Query`            | Native SQL                                            |
|--------------------------------------------------------------------|--------------------------|-------------------------------------------------------|
| Simple filtering on entity fields                                  | Works                    | Works, slightly shorter                               |
| JOINs across entity associations (`JOIN u.orders`)                 | Yes (Entity-aware)       | No (Manual `JOIN orders ON ...`)                      |
| `JOIN FETCH` to avoid N+1                                          | Yes (Built-in)           | No (Doesn't exist)                                    |
| `SELECT new DTO(...)` projection                                    | Yes (Direct)             | No (Manual mapping)                                   |
| DB-specific features (window functions, CTEs, JSON ops, `UPSERT`)   | No (Limited)             | Yes (Required)                                        |
| Refactor safety (renaming a field)                                 | Yes (Catches at startup) | No (Silent breakage - column names hidden in string)  |
| DB portability (MySQL ↔ PostgreSQL)                                | Yes (Mostly portable)    | No (Tied to dialect)                                  |

> **Recommended:** prefer JPQL `@Query` by default — entity-aware, refactor-safe, DB-portable. Reach for native SQL only when you genuinely need DB-specific features (window functions, CTEs, JSON operators, vendor hints, complex `UPSERT`).

For extremely dynamic queries (many optional filters), use **Criteria API** or **QueryDSL** / `Specification` instead.

## Bulk Updates and Deletes

For high-volume operations where entity-by-entity `save()` would be too slow (thousands of rows in N round-trips), use `@Modifying` `@Query` to run a single SQL statement:

```java
public interface ProductRepository extends JpaRepository<Product, Long> {

    @Modifying
    @Query("UPDATE Product p SET p.price = p.price * 1.1 WHERE p.category = :cat")
    int bulkRaisePrice(@Param("cat") String category);

    @Modifying
    @Query("DELETE FROM Token t WHERE t.expiresAt < CURRENT_TIMESTAMP")
    int deleteExpiredTokens();
}
```

- `@Modifying` is **required** for any update/delete through `@Query` — otherwise Spring tries to run it as a `SELECT` and fails
- Return type is `int` — number of rows affected
- Must be called inside a `@Transactional` method

> **Note:** Bulk queries go straight to the database, so any entities you've already loaded in the same transaction still hold the old values (stale data):

```java
Product p = productRepository.findById(1L).orElseThrow();    // p.price = 100
productRepository.bulkRaisePrice("electronics");             // DB: price → 110
log.info(p.getPrice());                                      // prints 100 — stale!
```

Two flags fix this:

- `clearAutomatically = true` — clears the in-memory cache **after** the bulk query, so the next `findById` fetches fresh from the DB
- `flushAutomatically = true` — pushes pending unsaved changes to the DB **before** the bulk query runs, so the bulk query sees them

```java
@Modifying(clearAutomatically = true, flushAutomatically = true)
@Query("UPDATE Product p SET p.price = p.price * 1.1 WHERE p.category = :cat")
int bulkRaisePrice(@Param("cat") String category);
```

> Bulk queries also skip lifecycle callbacks (`@PrePersist`/`@PreUpdate`), auditing fields, and `@SQLDelete` soft-delete. Prefer `save()` / `delete()` for normal writes.

## Calling Stored Procedures

For databases that wrap business logic in stored procedures, Spring Data offers `@Procedure` for direct invocation:

```java
public interface OrderRepository extends JpaRepository<Order, Long> {

    @Procedure(name = "calculate_discount")
    BigDecimal calculateDiscount(@Param("orderId") Long orderId);
}
```

The procedure must be declared on the entity via `@NamedStoredProcedureQuery`:

```java
@Entity
@NamedStoredProcedureQuery(
    name = "calculate_discount",
    procedureName = "sp_calculate_discount",
    parameters = {
        @StoredProcedureParameter(mode = ParameterMode.IN,  name = "orderId",  type = Long.class),
        @StoredProcedureParameter(mode = ParameterMode.OUT, name = "discount", type = BigDecimal.class)
    }
)
public class Order { ... }
```

**When to use**: Legacy databases with business logic already in stored procedures
- Multi-step computations or bulk operations where running close to the data reduces network round-trips (e.g., scan millions of rows, return one aggregate)
- Compliance / audit requirements that mandate database-side logic

**When to avoid**: New apps where logic can live in the service layer (more testable, version-controlled)
- Cross-database portability is needed (procedures are dialect-specific)
- Most CRUD scenarios — `@Query` or derived methods are simpler

## How It Works Internally

On application startup:

1. Spring scans repository interfaces
2. Creates a proxy implementation for each
3. Parses method names into query semantics
4. Generates JPQL at first call
5. Uses `EntityManager` to execute
6. Hibernate translates JPQL to SQL via JDBC

You never implement the DAO yourself — Spring Data JPA still uses JPA, `EntityManager`, Hibernate, and JDBC underneath; it just hides the complexity.

## Q&A

1. **What does Spring Data JPA add on top of JPA?**
   Repository abstraction, query derivation from method names, pagination / sorting helpers, and runtime proxy implementations — all of which remove the manual DAO layer. Under the hood it still uses JPA, `EntityManager`, and Hibernate.

2. **How does Spring generate a query from a method name like `findByEmailAndStatus`?**
   At startup it scans repository interfaces, parses each method name into a query subject (`findBy`) and predicate (`EmailAndStatus`), and builds a JPQL query. Method parameters bind to predicates in order.

3. **`JpaRepository` vs `CrudRepository` vs `PagingAndSortingRepository`?**
   `CrudRepository` → basic CRUD. `PagingAndSortingRepository` adds `findAll(Pageable)` / `findAll(Sort)`. `JpaRepository` adds batch operations, `flush()`, and JPA-specific methods. Extend `JpaRepository` unless you have a reason not to.

4. **When would you use `@Query` over a derived method?**
   When the method name would become unreadable, when you need JPQL constructs not supported by derivation (e.g., `JOIN FETCH`, subqueries, `SELECT new ...` DTO), or when you need a native query.

5. **What's the difference between JPQL and native queries?**
   JPQL queries entity classes / fields and is database-portable. Native queries use raw SQL — needed for database-specific features (window functions, CTEs) but tied to that vendor's dialect.

6. **Why does an `UPDATE`/`DELETE` `@Query` need `@Modifying`, and what about `clearAutomatically` / `flushAutomatically`?**
   Without `@Modifying`, Spring tries to execute the JPQL as a `SELECT` and fails. Bulk modifying queries bypass the persistence context, so loaded entities still hold stale state — `clearAutomatically = true` evicts them, `flushAutomatically = true` pushes pending changes first. Required when the same transaction reads or saves affected entities afterwards.

7. **When would you use `@Procedure` over `@Query`?**
   For databases with existing business logic in stored procedures, or when performance-critical computations benefit from running in-database. The procedure must be declared on the entity via `@NamedStoredProcedureQuery`. For new apps, prefer service-layer logic with `@Query` — more testable, version-controlled, and DB-portable.
