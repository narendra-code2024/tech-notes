## EntityManager and Persistence Context

**`EntityManager`** is the main JPA interface for interacting with the database. It handles `persist`, `find`, `remove`, query creation, and manages the **persistence context**.

The **persistence context** is the in-memory cache of managed entity instances associated with an `EntityManager`. Also known as the **first-level cache (L1)**:

- One persistence context per `EntityManager`
- Stores all entities that are currently *managed*
- Guarantees one instance per primary key within its scope (entity identity)
- Automatically tracks changes (**dirty checking**) and synchronizes them on flush

```java
Employee emp  = entityManager.find(Employee.class, 1L);
Employee emp2 = entityManager.find(Employee.class, 1L);
// emp == emp2 → true (same reference, returned from L1 cache)
```

> In Spring Boot, you rarely call `EntityManager` directly — `repository.findById(id)` internally calls `EntityManager.find(...)`, with the same identity guarantee and L1 cache behavior.

### The factory and EntityManager hierarchy

Spring Boot manages a single application-wide `EntityManagerFactory` (per data source). It produces short-lived `EntityManager` instances on demand — typically one per `@Transactional` method.

| | Count | Scope |
|---|---|---|
| `EntityManagerFactory` | 1 per data source (singleton bean) | Application lifetime |
| `EntityManager` | Many — one per `@Transactional` method / unit of work | Lives only during the transaction |
| Persistence context | 1 per `EntityManager` | Same as the `EntityManager` |

Two concurrent requests running `@Transactional` methods get **two separate `EntityManager`s** with **two separate L1 caches**: they don't share state.

## Entity Lifecycle States

A JPA entity moves through four states.

### Transient (New)

Just created with `new`, not yet associated with an `EntityManager`, no row in the database.

```java
Employee emp = new Employee();   // transient
```

### Persistent (Managed)

Associated with the persistence context and tracked. Field changes synchronize to the database automatically on flush.

```java
entityManager.persist(emp);                                  // transient → persistent
Employee found = entityManager.find(Employee.class, 1L);     // also persistent
```

### Detached

Was persistent, but the persistence context is no longer tracking it (transaction ended, `clear()`, or explicit `detach()`). Field changes are **not** propagated.

```java
entityManager.detach(emp);   // persistent → detached
```

### Removed

Marked for deletion — actual `DELETE` runs on commit / flush.

```java
entityManager.remove(emp);   // persistent → removed
```

> The examples above use `EntityManager` to show state transitions explicitly. In day-to-day Spring Boot code you'd see `userRepository.save(emp)` (≈ `persist` for new entities, `merge` for detached) and `userRepository.delete(emp)` (≈ `remove`) — these call the `EntityManager` methods underneath. `detach` happens automatically when the `@Transactional` method returns.

## Dirty Checking

While an entity is in the **persistent** state you do not need to call `update()` or `save()`. At commit time Hibernate compares the original snapshot to the current state and emits an `UPDATE` for any changed fields.

```java
@Transactional
public void renameEmployee(Long id, String newName) {
    Employee emp = entityManager.find(Employee.class, id);
    emp.setName(newName);
    // no save / update call — dirty checking emits UPDATE on commit
}
```

---

## Flush vs Commit

- **`flush()`**: synchronizes the persistence context with the database by executing pending SQL. Does **not** commit; changes can still be rolled back.
- **`commit()`**: permanently persists the changes and ends the transaction.

Spring calls `flush()` automatically before commit; you rarely call it yourself (occasionally needed to surface constraint violations early in tests).

## `persist` vs `merge` vs `save`

All three "save" an entity to the database, but they handle different starting states. Getting them wrong silently drops your changes.

### `persist(entity)` — for NEW entities

Use when the entity has no ID yet (just created with `new`).

```java
Employee emp = new Employee("Alice", 50000);   // transient — no id
entityManager.persist(emp);
// emp is now managed; INSERT runs on flush; emp.getId() gets populated
// You can keep using `emp` directly — it's the same object, now managed
```

- Throws if the entity already has a persisted ID
- Returns `void`, but the argument itself is mutated in place (ID populated, becomes managed)

### `merge(entity)` — for DETACHED entities

Use when an entity was previously managed but the transaction has ended (detached). E.g., the same entity loaded in one transaction and updated in another.

```java
Employee emp = ...;                       // detached
emp.setName("Bob");
Employee managed = entityManager.merge(emp);
// merge COPIES emp's state into a NEW managed copy
// `managed` is now in the persistence context
// `emp` stays detached — never becomes managed
```

**Common bug**: ignoring the return value:

```java
entityManager.merge(emp);                 // returned value not captured
emp.setName("Charlie");                   // invalid - won't persist — emp is still detached
```

The fix is to assign and use the return value:

```java
Employee managed = entityManager.merge(emp);
managed.setName("Charlie");               // OK - persists — managed is in the context
```

### `save(entity)` — Spring Data's smart choice

`JpaRepository.save()` picks automatically:

- If the entity is new (`id == null`, or `@Version` is null when present) → calls `persist`
- Otherwise → calls `merge`

```java
User saved = userRepository.save(user);
```

**Always assign and use the return value.** For new entities, `saved == user` (same object). For detached entities, `saved` is the managed copy and `user` stays detached — using `saved` avoids the behavior described above.

## Entity `equals` / `hashCode`

Entities are stored in `Set`s and `Map`s (especially in `@ManyToMany` and bidirectional `@OneToMany` collections), so their identity contract matters.

- **Don't use Lombok `@Data` on entities**: it generates `equals` / `hashCode` over **all fields** including lazy collections. Touching that during a `Set.add` triggers lazy loads and can `StackOverflowError` on bidirectional graphs.
- **Don't use the auto-generated ID** in `equals` directly — two transient instances both have `id` as `null` and would be considered equal.
- Safest pattern: a **business key** (natural unique identifier) when one exists; otherwise an `equals` that returns `true` only when both IDs are non-null and equal, and a constant `hashCode` (e.g., `getClass().hashCode()`).

## First-Level vs Second-Level Cache

| | First-Level (L1) | Second-Level (L2) |
|---|---|---|
| Layer | JPA persistence context | JPA, plugged in via a cache provider |
| Scope | Per `EntityManager` / transaction | Shared across sessions / requests |
| Stores | Managed entities (1 instance per primary key) | Entity rows (deconstructed) |
| Enabled | Always on (built into the persistence context) | Opt-in — must be configured |
| Backends | N/A (built-in) | Ehcache, Caffeine, Hazelcast, Infinispan, Redis |
| Use case | Identity guarantee within a transaction | Cross-request entity caching |
| Modern usage | Universal | Rare in microservices (cache coherence headaches across instances) |

> **Don't confuse JPA L2 with Redis-via-`@Cacheable`.** Most teams use Redis as an **application-level cache** for method results (DTOs, API responses, computations) via Spring Cache annotations — that's a different layer than JPA L2 (which caches entity rows). See [11.3 Redis Cache in Spring Boot](<../11-caching-and-concurrent-transaction-management/11.3. Redis Cache in Spring Boot.md>).

## `LazyInitializationException`

Thrown when code accesses a lazy-loaded relationship on a **detached** entity — the persistence context is closed, so Hibernate has no session to fire the SELECT.

Common causes:

- Returning an entity from the service layer and accessing lazy fields in the controller / view
- Calling `entityManager.clear()` mid-flow
- Missing `@Transactional` on the calling method (see [3.8 Transactions](<3.8. Transactions.md>))

Fixes — load the data before the context closes:

- `JOIN FETCH` in JPQL
- `@EntityGraph` on the repository method
- DTO projection
- Keep the read inside an `@Transactional` method

See [3.9 N+1 Query Problem](<3.9. N+1 Query Problem.md>) for the related fetch-strategy discussion.

## Q&A

1. **What is the persistence context?**
   An in-memory cache of managed entities tied to an `EntityManager`. It guarantees a single instance per primary key, tracks field changes via dirty checking, and synchronizes them to the database on flush. Also called the first-level cache.

2. **Walk through the four entity lifecycle states.**
   **Transient**: just `new`'d, not managed. **Persistent**: tracked by the persistence context, changes flushed automatically. **Detached**: was persistent but the context is closed; changes ignored. **Removed**: marked for deletion, actual `DELETE` fires on commit.

3. **What is dirty checking?**
   Hibernate's mechanism of comparing each managed entity's current field values to the snapshot taken when it was loaded. Any differences produce an `UPDATE` at flush time, so you don't call `save()` for already-managed entities.

4. **Why does `LazyInitializationException` happen?**
   The entity is detached (persistence context closed) and the code touches a lazy relationship — Hibernate has no session to fire the SELECT. Fix by loading the data eagerly via `JOIN FETCH` / `@EntityGraph` / DTO projection, or by keeping the access inside the transaction.

5. **`flush()` vs `commit()`?**
   `flush()` pushes pending SQL to the database but stays inside the transaction (changes can still be rolled back). `commit()` makes the changes permanent and closes the transaction. Spring calls `flush()` automatically before commit.

6. **First-level vs second-level cache?**
   L1 is the persistence context — per `EntityManager`, on by default, can't be disabled. L2 is an optional shared cache across sessions (Ehcache, Caffeine, etc.) used to avoid repeated reads of the same entity from disk.

7. **`persist` vs `merge` vs `save`?**
   `persist` transitions transient → managed (`void` return). `merge` copies a detached entity's state into a managed copy and returns it — the argument stays detached. Spring Data's `save` calls `persist` for new entities and `merge` otherwise; always use the returned value.

8. **Why is Lombok `@Data` dangerous on JPA entities?**
   `@Data` generates `equals` / `hashCode` over **all fields** including lazy collections — when you `Set.add` an entity Hibernate lazily loads the collections, which can `StackOverflowError` on bidirectional graphs. Use `@Getter` / `@Setter` only and write `equals` / `hashCode` deliberately (business key or ID-with-non-null guard).
