A projection fetches **only selected fields** from an entity instead of the whole row.

- Instead of `SELECT * FROM employee`
- You fetch `SELECT name, salary FROM employee`

Without projection, `List<Employee> findAll()` loads all columns plus any eagerly-fetched relationships — wasteful when you only need a few fields. Projections reduce query size, memory, and network overhead, and help mitigate over-fetching.

---

## Interface-Based Projection

Declare an interface with getters matching entity field names. Spring creates a proxy and fetches only the requested columns.

```java
@Entity
public class Employee {
    @Id private Long id;
    private String name;
    private String email;
    private Double salary;
}

public interface EmployeeNameEmail {
    String getName();
    String getEmail();
}

public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    List<EmployeeNameEmail> findBySalaryGreaterThan(Double salary);
}
```

Rules:

- Getter names must match entity field names
- Only declared getters are fetched
- Works with derived queries and `@Query`

> Internally, Spring generates a proxy implementing the interface and maps the SELECT'd columns to the getters.

---

## Closed vs Open Projections

**Closed**: only simple getters mapping to real columns. Spring fetches just those columns.

```java
public interface EmployeeClosed {
    String getName();
}

public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    List<EmployeeClosed> findClosedByDepartment(String department);
}
// → SQL: SELECT name FROM employee WHERE department = ?
```

**Open**: uses SpEL expressions via `@Value`. Spring has to load the full entity to evaluate the expression, so it **loses the performance benefit**. Avoid on hot paths.

```java
public interface EmployeeOpen {
    @Value("#{target.name + ' - ' + target.email}")
    String getDisplayInfo();
}

public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    List<EmployeeOpen> findOpenByDepartment(String department);
}
// → SQL: SELECT * FROM employee WHERE department = ?  (full row loaded for SpEL eval)
```

## DTO Projection (Record or Class)

Define a target type with a constructor matching the selected fields, then use `SELECT new <full.package.path.ClassName>(...)` in JPQL — the full package path is required.

### Java `record` (modern default)

In Java 17+ (handbook baseline), use a `record` — concise, immutable, no boilerplate. Spring Data JPA supports records natively as projection targets.

```java
public record EmployeeDto(String name, String email) {}

public interface EmployeeRepository extends JpaRepository<Employee, Long> {

    @Query("SELECT new com.example.EmployeeDto(e.name, e.email) FROM Employee e")
    List<EmployeeDto> fetchEmployeeSummaries();
}
```

The record's canonical constructor (auto-generated from the component list) must match the selected columns exactly — same types, same order.

### Class form (when you need more)

Use a regular class with an explicit constructor when records don't fit — e.g., you need mutability, custom validation in the constructor, or you're stuck on pre-Java 17:

```java
public class EmployeeDto {
    private final String name;
    private final String email;

    public EmployeeDto(String name, String email) {
        this.name = name;
        this.email = email;
    }
    // getters...
}
```

Same `@Query("SELECT new com.example.EmployeeDto(e.name, e.email) FROM Employee e")` — JPQL doesn't care whether the target is a record or a class.

## Dynamic Projection

Choose the projection type at runtime by passing a `Class<T>` argument.

```java
public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    <T> List<T> findBySalaryGreaterThan(Double salary, Class<T> type);
}

// callers
employeeRepository.findBySalaryGreaterThan(50_000.0, Employee.class);         // full entity
employeeRepository.findBySalaryGreaterThan(50_000.0, EmployeeNameEmail.class); // projection
```

Useful for reusable APIs where different callers need different views of the same data.

## Nested Projections

Projections compose across relationships — Spring generates the JOIN automatically and fetches only the requested fields from both sides.

Given an entity relationship:

```java
@Entity
public class Employee {
    @Id private Long id;
    private String name;
    private Double salary;

    @ManyToOne
    private Department department;
}

@Entity
public class Department {
    @Id private Long id;
    private String name;
    private String location;
}
```

Define nested projections — outer interface declares the relationship, inner interface declares the related fields you want:

```java
public interface EmployeeView {
    String getName();
    DepartmentView getDepartment();          // nested projection
}

public interface DepartmentView {
    String getName();
}

public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    List<EmployeeView> findAllBy();
}
// → SQL: SELECT e.name, d.name
//        FROM employee e
//        JOIN department d ON e.department_id = d.id
```

Spring sees the nested `DepartmentView` return type and emits a JOIN, selecting only the requested columns. `salary` and `location` are skipped.

> The inherited `findAll()` returns `List<Employee>` (the entity) — you can't override its return type, so declare a custom method like `findAllBy()` whose return type is the projection.

## Entity vs Interface Projection vs DTO Projection

| Feature | Entity | Interface Projection | DTO Projection |
|---|---|---|---|
| Fetches selected fields only | No | Yes | Yes |
| Requires explicit constructor | No | No | Yes (auto for `record`) |
| Mutability | Mutable (JPA-managed) | Read-only (proxy) | Typically immutable |
| Type at runtime | Concrete entity class | Spring-generated proxy | Concrete object (record / class) |
| Best for | Full CRUD operations | Quick read views | Cross-layer data flow (API responses, services) |

### Key constraints

- Projections apply to `SELECT` queries only — cannot be used for save / update
- Reduce over-fetching but don't always fix N+1 (see [3.9 N+1 Query Problem](<3.9. N+1 Query Problem.md>))

## Q&A

1. **Why use projections instead of loading full entities?**
   To avoid over-fetching — fewer columns loaded means smaller SQL payloads, less memory use, and fewer joins triggered by eager relationships. Especially valuable on list / search endpoints that only need a handful of fields.

2. **Interface projection vs DTO projection — which is better?**
   Interface projections are the simplest: declare an interface and Spring builds the query. DTO projections need an explicit `@Query` with `SELECT new ...` but give you a concrete object you can pass around freely (not a proxy). On Java 17+, prefer a **`record`** as the DTO target — same JPQL, zero boilerplate. Use interfaces for quick read models, DTOs (records) when the object needs to flow outside the repository layer.

3. **What is an issue with open projections?**
   The SpEL expression (`@Value("#{target.name}")`) forces Spring to materialize the full entity first, then evaluate the expression — you lose the "fetch fewer columns" benefit. Use closed projections for performance.

4. **Can projections help with N+1?**
   Sometimes — a DTO projection with an explicit `JOIN` can flatten parent + child fields into one query and avoid lazy loading. But projections don't replace fetch joins or `@EntityGraph` for relationship loading in general.
