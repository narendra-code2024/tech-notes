## Entity Relationships

In relational databases, tables are linked through foreign keys. JPA represents those links with annotations on the entity classes.

| Relationship | Annotation |
|---|---|
| One-to-One | `@OneToOne` |
| One-to-Many | `@OneToMany` |
| Many-to-One | `@ManyToOne` |
| Many-to-Many | `@ManyToMany` |

**Owning side vs inverse side**: the **owning side** is the entity that holds (and updates) the foreign-key column. The **inverse side** uses `mappedBy` to point back to the owning side's field. Only changes on the owning side are persisted to the FK column.

---

## `@ManyToOne`

Many employees belong to one department. DB: `employee(id, name, department_id)` and `department(id, name)`.

```java
@Entity
public class Employee {
    @Id
    private Long id;
    private String name;

    @ManyToOne
    @JoinColumn(name = "department_id")   // owning side: this entity manages the FK column
    private Department department;
}

@Entity
public class Department {
    @Id
    private Long id;
    private String name;
}
```

- `@ManyToOne` is **always the owning side**: `@JoinColumn` lives on the owning side and configures the FK column (`name`, `nullable`, etc.)

---

## `@OneToMany`

The inverse side of `@ManyToOne`. One department has many employees.

```java
@Entity
public class Department {
    @Id
    private Long id;
    private String name;

    @OneToMany(mappedBy = "department")   // inverse side: refers to Employee.department
    private List<Employee> employees;
}
```

`mappedBy` tells JPA: "the FK is managed by `Employee.department`, don't create a separate join table." Without `mappedBy`, Hibernate would create an unwanted join table.

---

## `@OneToOne`

One user has one profile. DB: `profile(id, user_id)`.

```java
@Entity
public class Profile {
    @Id
    private Long id;

    @OneToOne
    @JoinColumn(name = "user_id")     // owning side
    private User user;
}

@Entity
public class User {
    @Id
    private Long id;

    @OneToOne(mappedBy = "user")      // inverse side
    private Profile profile;
}
```

---

## `@ManyToMany`

Students enroll in courses. DB: `student`, `course`, plus a join table `student_course(student_id, course_id)`.

```java
@Entity
public class Student {
    @Id
    private Long id;

    @ManyToMany
    @JoinTable(
        name = "student_course",
        joinColumns = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id")
    )
    private Set<Course> courses;
}

@Entity
public class Course {
    @Id
    private Long id;

    @ManyToMany(mappedBy = "courses")   // inverse side
    private Set<Student> students;
}
```

- The owning side declares `@JoinTable`; the inverse side uses `mappedBy`
- Prefer `Set` over `List` — Hibernate has known duplicate-row issues with `List`-backed `@ManyToMany`
- Many teams replace `@ManyToMany` with an explicit join entity (`StudentCourse`) to attach extra columns like `enrolledAt`

## Fetch Types

| Relationship | Default Fetch |
|---|---|
| `@ManyToOne` | `EAGER` |
| `@OneToOne` | `EAGER` |
| `@OneToMany` | `LAZY` |
| `@ManyToMany` | `LAZY` |

Override:

```java
@ManyToOne(fetch = FetchType.LAZY)
private Department department;
```

- **EAGER**: related entity loaded immediately with the parent (additional joins / queries at load time)
- **LAZY**: related entity loaded only on first access; triggers a query at that moment

```java
Department dept = departmentRepository.findById(1L).orElseThrow();
dept.getEmployees();   // LAZY → query fires here
```

> **Best practice:** make every relationship `LAZY` and pull related data in via `JOIN FETCH` or `@EntityGraph` when you actually need it. `EAGER` collections cause hidden N+1 problems and load far more than the use case requires. See [3.9 N+1 Query Problem](<3.9. N+1 Query Problem.md>).

## Keeping Both Sides in Sync (Bidirectional)

In a bidirectional relationship, **only changes on the owning side hit the database**. The inverse side (the collection) is just an in-memory mirror. So updating the collection alone doesn't change anything in the DB.

The bug:

```java
Department dept = departmentRepo.findById(1L).orElseThrow();
Employee emp = new Employee("Alice");

dept.getEmployees().add(emp);     // invalid - only the collection changes — in memory
departmentRepo.save(dept);        // saves dept (no changes to dept's columns)
// emp is NEVER persisted; emp's department_id stays null
```

Why? `Employee.department` is the owning side — it holds the FK column. Adding `emp` to `dept.employees` doesn't tell JPA "set Employee's department_id". JPA only watches the owning side for FK changes.

The fix — set the owning side:

```java
emp.setDepartment(dept);          // OK - owning side updated — FK gets set
employeeRepo.save(emp);
```

But now the in-memory `dept.employees` list doesn't include `emp` until the next reload. Code that reads `dept.getEmployees()` in the same transaction would see stale state.

The clean solution is **a helper method on the parent that updates both sides at once**:

```java
@Entity
public class Department {

    @OneToMany(mappedBy = "department", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Employee> employees = new ArrayList<>();

    public void addEmployee(Employee e) {
        employees.add(e);            // inverse side — keeps in-memory graph consistent
        e.setDepartment(this);       // owning side — what JPA actually persists
    }

    public void removeEmployee(Employee e) {
        employees.remove(e);
        e.setDepartment(null);
    }
}
```

Now `dept.addEmployee(emp)` does both — DB gets the FK update **and** the in-memory graph reflects reality.

## Q&A

1. **Owning side vs inverse side — why does it matter?**
   The owning side holds the foreign key; only changes on the owning side are persisted. The inverse side declares `mappedBy` and exists for object navigation. Updating only the inverse side leaves the FK unchanged — a classic source of "my save didn't persist" bugs.

2. **Why is `@ManyToOne` always the owning side?**
   Because the FK column lives on the "many" side's table — the row belongs there. The "one" side has no column to update, so it can only ever be the inverse side via `mappedBy`.

3. **`Set` vs `List` for `@ManyToMany` — which and why?**
   `Set`. Hibernate's `List` handling for `@ManyToMany` triggers `MultipleBagFetchException` when combined with other collection fetch joins, and produces duplicate rows. `Set` enforces uniqueness and avoids those issues.

4. **What's the default fetch type for each relationship?**
   `@ManyToOne` and `@OneToOne` default to `EAGER`; `@OneToMany` and `@ManyToMany` default to `LAZY`. Note: `EAGER` collections fire extra joins / queries on every load. Best practice is `LAZY` everywhere with explicit `JOIN FETCH` / `@EntityGraph` where needed.

5. **What's the difference between unidirectional and bidirectional mappings?**
   Unidirectional defines the relationship on one side only. Bidirectional defines both sides, requiring `mappedBy` on the inverse to avoid a duplicate FK / join table. Bidirectional is convenient for navigation but adds the burden of keeping both sides in sync in code.
