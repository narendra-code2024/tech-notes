## What is Cascade?

Cascade defines which JPA operations on a parent entity automatically propagate to its related entities.

```java
@Entity
public class Order {

    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL)
    private List<OrderItem> items;
}
```

Saving / deleting / updating the `Order` now propagates to its `OrderItem`s automatically — no manual loop required.

### Cascade types

| Type | Behavior |
|---|---|
| `CascadeType.PERSIST` | Save parent → save children |
| `CascadeType.MERGE` | Merge parent → merge children |
| `CascadeType.REMOVE` | Delete parent → delete children (use carefully) |
| `CascadeType.REFRESH` | Reload parent → reload children |
| `CascadeType.DETACH` | Detach parent → detach children |
| `CascadeType.ALL` | All of the above |

### JPA cascade vs database cascade

- **JPA cascade** — handled by Hibernate at the application layer; respects entity lifecycle
- **Database cascade** (`ON DELETE CASCADE`) — handled by the database engine; Hibernate has no visibility into it

Best practice: use JPA cascade for object-level control, and only add DB-level cascade when you need the database to enforce it independently of the application.

## Orphan Removal

`orphanRemoval = true` means: when a child is removed from the parent's collection (or the parent reference is set to `null`), Hibernate **deletes** that child from the database.

```java
@OneToMany(
    mappedBy = "order",
    cascade = CascadeType.ALL,
    orphanRemoval = true
)
private List<OrderItem> items;
```

What happens:

- Save order → items saved (`CascadeType.PERSIST`)
- Delete order → items deleted (`CascadeType.REMOVE`)
- Remove an item from `items` list → that item's row is deleted (`orphanRemoval`)

Orphan removal works only with `@OneToMany` and `@OneToOne`.

### Why "orphan"?

A child entity removed from the parent's collection no longer has a parent — it becomes an orphan. With `orphanRemoval = true`, Hibernate deletes orphans; without it, they stay in the database with a (possibly null) foreign key.

```
Order ── Item1, Item2

// remove Item1 from order.items
Order ── Item2

// Item1 is now an orphan
//   orphanRemoval = true  → DELETE FROM order_item WHERE id = Item1
//   orphanRemoval = false → row remains, FK may go null
```

### `CascadeType.REMOVE` vs `orphanRemoval`

| Action | `cascade = REMOVE` | `orphanRemoval = true` |
|---|---|---|
| Delete parent | Children deleted | Children deleted |
| Remove child from collection | Child stays | Child deleted |

### Requirements and when to use it

- Must run within an **active transaction** (persistence context must be open)
- Both sides of a bidirectional relationship must be kept in sync in code

**Use when** the child cannot exist without the parent — `Order` → `OrderItem`, `Blog` → `Comment`, `Invoice` → `InvoiceLine`.

**Avoid when** the child can exist independently — `Student` ↔ `Course` (`@ManyToMany`), `Employee` → `Department`.

## Soft Delete Pattern

Sometimes physical deletion is wrong — regulatory requirements, audit trails, recoverability ("oops, restore that order"). Soft delete keeps the row but marks it deleted via a timestamp column.

Spring Data JPA has no built-in soft delete (unlike Laravel's `SoftDeletes` trait). Use Hibernate's `@SQLDelete` + `@SQLRestriction`:

```java
@Entity
@SQLDelete(sql = "UPDATE employees SET deleted_at = NOW() WHERE id = ?")
@SQLRestriction("deleted_at IS NULL")
public class Employee {

    @Id
    @GeneratedValue
    private Long id;

    private String name;
    private Instant deletedAt;
}
```

- `@SQLDelete` — overrides the DELETE statement; `repository.delete()` (and `entityManager.remove()`) now runs your UPDATE instead
- `@SQLRestriction` — adds `WHERE deleted_at IS NULL` to every query automatically

### What does and doesn't soft delete

| Operation | Soft delete? |
|---|---|
| `repository.delete(entity)` / `deleteById(id)` / `entityManager.remove()` | ✅ Auto (via `@SQLDelete`) |
| `findAll()` / `findById()` and other queries | ✅ Auto-filtered (via `@SQLRestriction`) |
| Cascade delete to child entities | ❌ Children physically deleted unless they also have `@SQLDelete` |
| Native query | ❌ Bypasses `@SQLRestriction` — write `WHERE deleted_at IS NULL` explicitly |
| Bulk `@Modifying` `DELETE` query | ❌ Bypasses `@SQLDelete` — write `UPDATE ... SET deleted_at = NOW()` instead |
| Restoring a soft-deleted row | ❌ No `JpaRepository` method does this — needs native `UPDATE ... SET deleted_at = NULL` |
| Unique constraint on a soft-deleted column | ❌ Deleted row still occupies the unique slot — include `deleted_at` in the unique constraint, or use partial indexes (PostgreSQL) |

### Alternative: archive table

For high-volume tables, soft-delete bloats the main table and slows queries (every query carries the `deleted_at IS NULL` filter). Move deleted rows to a separate `archive_employees` table via a scheduled job — keeps the live table fast and history is preserved separately.

### Hibernate 6.4+ `@SoftDelete` (cleaner alternative)

Hibernate 6.4 (Spring Boot 3.3+) added a native `@SoftDelete` annotation that replaces the `@SQLDelete` + `@SQLRestriction` combo:

```java
@Entity
@SoftDelete(columnName = "deleted_at",
            strategy = SoftDeleteType.TIMESTAMP)
public class Employee {

    @Id
    @GeneratedValue
    private Long id;

    private String name;
    // no need to declare deletedAt field — Hibernate manages the column
}
```

| Aspect | `@SQLDelete` + `@SQLRestriction` | `@SoftDelete` |
|---|---|---|
| Hibernate version | 5.x onwards | 6.4+ only |
| DELETE override | Manual SQL string | Auto-generated |
| Query filter | Manual `@SQLRestriction` | Auto-applied |
| Field declaration | Must declare `deletedAt` | Optional — Hibernate manages the column |
| Strategy options | Timestamp only (custom SQL) | `TIMESTAMP`, `ACTIVE`, `DELETED` (boolean flag) |
| Cascade behavior gotchas | Same — children still need their own annotation | Same — apply on each entity that should soft-delete |

**Strategy options:**
- `SoftDeleteType.TIMESTAMP` — column stores deletion timestamp (NULL when active)
- `SoftDeleteType.ACTIVE` — boolean flag, `true` when active
- `SoftDeleteType.DELETED` — boolean flag, `true` when deleted

For new projects on Spring Boot 3.3+, prefer `@SoftDelete` — less manual SQL, native Hibernate support, same gotchas table above still applies.

## Important Interview Questions

1. **`CascadeType.REMOVE` vs `orphanRemoval = true` — what's the difference?**
   `REMOVE` propagates parent deletion to children. `orphanRemoval` additionally deletes a child the moment it's removed from the parent's collection (or its parent reference is nulled). Use `orphanRemoval` for parent-owned children whose lifecycle is bound to the parent.

2. **Why is `CascadeType.ALL` considered risky?**
   It includes `REMOVE`, so deleting a parent silently deletes everything it references — easy to wipe out related data unintentionally, especially across long entity graphs. Prefer the minimal subset (`PERSIST, MERGE`) and add `REMOVE` deliberately.

3. **JPA cascade vs database `ON DELETE CASCADE` — which should you use?**
   They operate at different layers. JPA cascade keeps the persistence context consistent and fires entity lifecycle callbacks. DB-level cascade enforces the rule even when rows are deleted outside the application. Use JPA cascade as the primary mechanism; add DB-level cascade only when you need the database to enforce integrity on its own.

4. **Why doesn't `orphanRemoval` work on `@ManyToMany`?**
   Because in a many-to-many relationship the child has multiple potential parents — removing one association doesn't make the child an orphan. JPA only allows `orphanRemoval` on `@OneToMany` and `@OneToOne` where ownership is clear.
